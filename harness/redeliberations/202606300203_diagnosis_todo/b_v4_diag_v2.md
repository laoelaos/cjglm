## 发现问题

### 1. [一般] G1 被错误归类为"测试正确性组"

- **问题描述**：修改流程建议中将 S3、S4、G1 归入同一组，组标题为"测试正确性组"（第 474 行）。但 G1 的修改方向是修改生产代码（`common.cj:173-179` roundEven 分支逻辑交换，见第 93 行），而非修正测试代码。S3 和 S4 确实属于测试修正，而 G1 属于库行为修正。将 G1 归入"测试正确性组"会误导执行者认为只需改测试文件。
- **所在位置**：第 474-477 行
- **严重程度**：一般
- **改进建议**：将 G1 移至"库行为正确性组"（S1/S2+G7/G10 组），或将组标题改为"测试正确性 + 库行为修正"并明确区分子分类。

### 2. [一般] G24 定性为"真实问题"与正文描述矛盾

- **问题描述**：G24 的"真实性"判定为"真实问题"（第 342 行），但正文描述为"设计争议（未明确零轴是否应保留原 q 或返回单位四元数）"（第 343 行）。这是一个尚未解决的设计分歧，而不是已确认的缺陷。当前定性会给执行者错误的紧迫信号——他们会认为这是一个需要立即修复的问题，但实际需要先决策再行动。G24 的根因"设计争议"本身就暗示尚未确定是否存在真实问题。
- **所在位置**：第 342 行（真实性判断）vs 第 343 行（根因描述）
- **严重程度**：一般
- **改进建议**：将 G24 的真实性改为"设计待决"或"设计争议"，修改方向相应调整为"先完成 OOD 决策，再对齐测试和实现"。与 G4（按设计）的定位方式对齐，避免执行者将设计争议当成已知缺陷盲目修复。

### 3. [轻微] G7 根因分类在正文与汇总表之间不一致

- **问题描述**：正文根因标注为"实现编码遗漏"（第 172 行），理由是"OOD 文档 §3.3 gtc/ulp.cj 未明确要求 NaN/Inf 检查，但同函数的其他重载保持一致的设计原则要求 float_distance 也有此保护"（第 180 行）——这意味着 OOD 未设计此功能，编码者本应主动补充。但汇总表（第 446 行）将 G7 归入"实现与设计偏差"，修改方向为"修改实现代码对齐设计"。然而 OOD 文档根本未设计 NaN/Inf 检查，不存在"对齐设计"的前提。"实现与设计偏差"暗示 OOD 有设计但编码偏离，与真相不符。
- **所在位置**：第 172 行（正文）vs 第 446 行（汇总表）
- **严重程度**：轻微
- **改进建议**：有两种修复选择：(a) 将汇总表的 G7 移至新增类别"实现编码遗漏"，或 (b) 将正文根因改为"实现与设计偏差"并相应修改 OOD 覆盖注释以说明一致性设计原则等价于设计要求。建议选择 (a)，更符合实际情况。

### 4. [轻微] G16/G34 等条目修改建议缺乏可操作细节

- **问题描述**：G16（第 290-295 行）修改方向为"补充缺失的 Float32 版本独立测试"，G34（第 414 行）为"补充所有 18+ 打包函数的 round-trip 测试"。这些条目缺乏具体缺失函数清单、测试文件路径或至少数量统计，执行者必须自行 grep 搜索才能确定"缺失的"具体指哪些。与此对比，G31（第 388-397 行）给出了完整函数清单和分类，G14（第 274-279 行）给出了精确的测试用例名称和异常情况。部分条目的精细度不均匀。
- **所在位置**：第 290-295 行（G16），第 414 行（G34）
- **严重程度**：轻微
- **改进建议**：至少补充以下信息：(a) G16：当前已覆盖 Float32 的函数清单和缺失的函数清单；(b) G34：列出 18+ 打包函数的具体名称，区分已有测试和缺失测试的函数。

### 5. [轻微] S3 副作用评估"无"过于绝对

- **问题描述**：S3（第 64 行）副作用评估仅写明"无。仅修改测试文件...不影响生产代码或其他测试"。但修正测试期望值后，如果测试通过则验证了现有生产代码行为是"错误"的——这个"验证"过程本身有潜在风险：生产代码中的 quaternion rotate 实现可能正好返回了被错误期望值验证的值，而正确期望值会导致测试失败，暴露更深层的生产代码问题。副作用评估应考虑"测试修复可能暴露生产代码缺陷"的连锁效应。
- **所在位置**：第 64 行
- **严重程度**：轻微
- **改进建议**：增加"测试期望值修正后，若测试失败应考虑生产代码 rotate 实现是否存在对应缺陷"的说明，建议执行者在修正 S3 后同步验证生产代码 `ext/quaternion_transform.cj` 中 rotate 实现的正确性。
