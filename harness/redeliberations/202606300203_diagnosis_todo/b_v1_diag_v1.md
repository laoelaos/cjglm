## 发现问题

### Q1（严重）：G10 分类矛盾——同一问题在诊断正文和汇总表中归类不一致

**位置**：第 188-196 行（正文）vs 第 247 行（汇总表）

**问题描述**：正文将 G10 描述为"潜在问题，公式实现可能与 GLM 不一致，需交叉验证"，并明确结论是"潜在问题"和"实现需要验证"。但汇总表「按根因分类」中将 G10 归类为"实现编码错误"。两者语义矛盾——一个"潜在/待验证"的问题不能被归类为已确认的"实现编码错误"。

**严重程度**：严重——分类矛盾会使执行者对 G10 的实际状态产生歧义理解：到底该当作代码错误去修复，还是先验证再定？

**改进建议**：统一分类。建议在验证完成前将 G10 单独列为"待验证"类别，或在汇总表中增加一行"待验证"分类。验证确认后再移入"实现编码错误"或"非问题"。

- G15（Float16 覆盖不足）、G16（Float32 独立测试缺失）——未分析这些是否为已记录的偏差（deviations.md 中 DV-04 等涉及 Float16 限制）

**改进建议**：对 G14-G37 每个 issue 至少逐条回答三个问题。多数可以精简（1-2 行一个 issue），但必须独立呈现结论。

**位置**：通篇缺失

**问题描述**：用户需求列出了 `docs/deviations.md` 作为参考文档。报告中从未引用该文件。部分 issue 的诊断可能因此遗漏重要上下文中已经记录为已知偏差的信息。例如：

**改进建议**：对每个涉及 API 行为差异或类型限制的 issue（G11、G15-G16、G24 等），查阅 deviations.md 确认是否有已记录的偏差影响诊断结论。

### Q4（中等）：G3 修改建议方向不明确

**位置**：第 99-102 行

**问题描述**：报告给出两个相反的修改方向——"建议修改 OOD 文档中的措辞匹配代码（`<=` 更保守），或修改代码匹配设计"。诊断报告的功能是为执行者提供可行动的方向，而非列出所有可能性。

**严重程度**：中等——执行者收到报告后不知道应该改哪一边，需要额外决策。

**改进建议**：基于 IEEE 754 中 `-0.0 == +0.0` 为 true，且 `dot(v,v) ≥ 0` 的数学保证，`<=` 和 `==` 行为一致。建议选定一个方向。推荐修改 OOD 文档匹配代码（代码中的 `<=` 更稳健，捕获理论上的负零等边缘情况），并说明理由。

### Q5（中等）：G2 根因分析未区分阶段责任

**位置**：第 78-88 行

**问题描述**：报告将 G2（`detail/vector_relational.cj` 缺失 5 函数）的根因归类为"OOD 设计遗漏"，但 OOD 文档 §2 中 `vector_relational.cj` 标记为"沿用阶段三"。这意味着阶段四的设计有意继承了阶段三的范围。根本问题可能是：
- 阶段三 OOD 设计时未涵盖这 5 个函数（阶段三的设计遗漏）
- 或阶段四的 OOD 设计没有审查并补救阶段三的历史遗漏（阶段四的设计审查遗漏）

当前诊断未区分这两种可能性，仅笼统归类为"OOD 设计遗漏"，执行者无法确定该找谁追责或从哪个阶段开始修改。

**严重程度**：中等——影响修复方案的精准度。

**改进建议**：明确关联阶段三的设计文件（`docs/05_ood_phase3.md`），确认阶段三是否已遗漏，并据此给出修改路径。
