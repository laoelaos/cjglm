### Q4（中等）：G3 修改建议方向不明确

**位置**：第 99-102 行

**问题描述**：报告给出两个相反的修改方向——"建议修改 OOD 文档中的措辞匹配代码（`<=` 更保守），或修改代码匹配设计"。诊断报告的功能是为执行者提供可行动的方向，而非列出所有可能性。

**严重程度**：中等——执行者收到报告后不知道应该改哪一边，需要额外决策。

**改进建议**：基于 IEEE 754 中 `-0.0 == +0.0` 为 true，且 `dot(v,v) ≥ 0` 的数学保证，`<=` 和 `==` 行为一致。建议选定一个方向。推荐修改 OOD 文档匹配代码（代码中的 `<=` 更稳健，捕获理论上的负零等边缘情况），并说明理由。

### Q11（严重）：缺少关键修复方案的潜在副作用评估

**位置**：第 257-289 行（修改流程建议部分通篇缺失）

**问题描述**：用户需求明确要求评估"修复方案是否有潜在副作用"。报告在修改流程建议中列出了优先级排序和修改方向，但对每项修改的潜在副作用几乎未作分析。具体来说：

**1. S1（rotate/shear 乘法顺序反转）的副作用未评估**
- 修改 `Rot * m` → `m * Rot` 将改变所有调用 `ext.matrix_transform.rotate()` 和 `ext.matrix_transform.shear()` 的代码结果
- 受影响的测试文件：`tests/glm/ext/matrix_transform_test.cj`（第 57、68、82 行使用 rotate/shear）、`tests/glm/lib_test.cj`（第 515 行使用 rotate）
- 如果该库已被实际项目使用，此修改会改变所有已有调用者的行为——需评估向后兼容性策略
- 报告仅简单提及"需同步更新测试用例"，但未评估是否有内部代码（如 lookAt 等函数）调用了 rotate/shear 作为内部实现

**2. S2（float_distance Int32 溢出）的副作用未评估**
- 从 `(y.toBits() as Int32).getOrThrow()` 改为位重解释后，NaN/Inf 输入的行为从抛异常变为返回一个位模式距离值
- 这与 G7（NaN/Inf 前置检查）联动：如果先修复 S2 再修复 G7，中间状态的 float_distance 对 NaN/Inf 输入会返回无意义值而非抛异常
- 需评估修复顺序对行为的影响，或建议 S2 和 G7 在同一个改动中一起修复

**3. G5（补齐 16 个 fmin/fmax 向量函数）的副作用未评估**
- 新增的函数与已有的 `fmin(a, b)` 同属 ext/vector_common.cj 包，需验证仓颉编译器在重载决议时能否正确区分 2、3、4 参数版本
- 新增函数可能触发与已有测试的重叠覆盖——原有同维 `fmin(a,b)` 的测试用例是否会与新函数产生冲突？
- OOD 文档中这些函数的设计约束条件（`T <: FloatingPoint<T> & Comparable<T>`）是否在已有代码中得到验证？

**4. G8（删除 ext/quaternion_double.cj 中的 dquat）的副作用未评估**
- `lib.cj:18` 明确 `public import glm.ext.{..., dquat, ...}`——同步修改必不可少
- 需确认代码库中是否有任何内部代码使用 `glm.ext.dquat` 限定路径访问
- `cjglm/src/ext/quaternion_double_precision.cj` 中定义了 `highp_dquat`/`mediump_dquat`/`lowp_dquat`——这些类型别名是否也受影响？

**严重程度**：严重——执行者在不清楚副作用的情况下直接按报告建议修改，可能导致未预料的回归故障或中间状态的编译错误。

**改进建议**：为第一优先级的每个修复方案（至少 S1、S2、G5、G8）补充副作用分析。分析应覆盖：
1. 受影响的测试文件清单
2. 修复顺序的依赖关系（如 S2 与 G7 应同时修复）
3. 向后兼容性影响
4. 需要同步修改的辅助代码（如 lib.cj 的 import 语句）

### Q12（中等）：优先级排序缺少合理性论证

**位置**：第 257-285 行

**问题描述**：用户需求要求评估"优先级排序是否合理"。报告给出了三优先 + 持续改进的排序，但未论证排序依据，且存在以下可商榷之处：

**1. S3/S4（测试期望值错误）与 S1/S2（库行为正确性）同为第一优先**
- S1（rotate 乘法顺序）影响所有使用 rotate/shear 的库用户——行为错误直接暴露给最终用户
- S3（quaternion_transform_test 期望值错误）仅为测试断言错误，不影响库行为——库函数本身的 rotate 可能正确，只是测试期望值算错
- 将功能正确性 bug 与测试 bug 并列第一优先，稀释了"第一优先"的严重性信号。建议 S3/S4 降为第二优先，或至少在第一优先中分组标注（S1/S2 为"库行为修复"，S3/S4 为"测试修复"）

**2. G7（NaN/Inf 前置检查）在第二优先，S2（float_distance 溢出修复）在第一优先**
- G7 和 S2 针对的是同一个函数 `float_distance` 的不同部分
- 仅修复 S2 而不修复 G7 后，float_distance 对 NaN/Inf 输入从"抛异常"变为"返回无意义值"——行为实际上变差了（静默错误而非显式失败）
- 建议将 G7 与 S2 并列第一优先，或者至少注明"G7 应在 S2 之前或同时修复"

**3. G13-G37（24 个测试覆盖问题）被压缩为"持续改进"一个条目**
- 这些 issue 占 todo.md 的 65%，统一标记为"持续改进"意味着它们都没有紧迫性
- 但有些测试覆盖问题与其他修复联动：G36/G37 的 ulp 测试受 S2/G7 影响，G27（fmin/fmax 3/4 输入测试不足）受 G5（补齐函数）影响
- 建议至少按影响面分组：(a) 与已有 bug 修复联动需同步更新（G23/G36/G37）(b) 缺失边界测试（G14/G17/G21/G22）(c) 维度覆盖不足（G15/G16/G25/G33）

**严重程度**：中等——当前排序粒度不够细，部分 issue 优先级与其实际影响面不匹配，可能导致执行者按错误顺序修复。

**改进建议**：
1. 在第一优先内部进一步分组，区分"库行为正确性"和"测试正确性"
2. 将 G7 移至与 S2 同优先级组（或明确标注同时修复）
3. 对 G13-G37 至少按修复紧迫性和联动态势做二级分组，而非简单归为"持续改进"
