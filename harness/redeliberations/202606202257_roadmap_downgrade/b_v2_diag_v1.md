## 3. Vec1 纳入理由与 CangJie 实际实现方式矛盾

- **问题描述**：§3B（line 371-374）将 Vec1 纳入迁移范围的理由之一是"与 vec2-4 共享相同的模板机制和部分操作，添加边际成本极低"。但 §0.1（line 17）已明确指出 CangJie 不支持 C++ 式偏特化，Vec1~Vec4 是**四个独立的泛型结构体**，不共享同一模板主定义。在 CangJie 实现中，Vec1 作为独立结构体需独自实现全部运算符、构造函数、辅助接口等，不存在"共享模板机制"和"边际成本极低"的前提条件。论证基础在 CangJie 上下文中不成立。
- **所在位置**：line 371-374
- **严重程度**：一般
- **改进建议**：修正理由描述——从"完备性验证"或"与 GLM API 兼容性"角度论证 Vec1 的纳入必要性，删除或改写基于共享模板的表述

## 5. `@OverflowWrapping` 标注量估算可能高估

- **问题描述**：§0.5.1（line 100-103）估算标注量为 4×(4+1+2)=28 处，其中包含 `%=`（取模赋值）和 `++`/`--`（自增自减）。但 GLM 核心向量类型 `type_vecN` 通常**不定义** `%=` 运算符和自增自减运算符——这些是标量类型规范操作，向量类型未提供对应语义。若 Vecl~Vec4 不实现这些运算符，标注量应为 4×4=16 处（仅算术复合赋值）。估算结果可能偏高 75%。
- **所在位置**：line 100-103
- **严重程度**：低
- **改进建议**：核实 Vec1~Vec4 实际实现的运算符集合后调整标注量估算，避免影响后续工作量评估的准确性
