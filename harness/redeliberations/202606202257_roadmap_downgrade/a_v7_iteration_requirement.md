### P1. §0.5.1 残留规划性语言（"后续优化方向"）
- **所在位置**：§0.5.1，第 98 行
- **严重程度**：一般
- **问题描述**：原文"直接逐分量实现（每个 Vec 类型独立实现二元运算，消除委托拷贝）可作为后续优化方向——在性能剖析确认拷贝开销成为热点时采用，不影响外部接口语义。"其中"可作为后续优化方向"与"在……时采用"均为面向未来的规划/建议性措辞，与文档定位声明"非路线图，不包含未来执行计划"直接矛盾。该问题在迭代 v3 中已作为 Issue 5 被标出（"可能""若后续"等推测性措辞），虽将原文"若后续性能剖析确认"改写为"在性能剖析确认……时"，但规划性语义未根本消除。

### P2. "迁移顺序"措辞传递规划语义
- **所在位置**：§3E 第 398 行、§3G 第 457 行
- **严重程度**：一般
- **问题描述**：两处使用"**迁移顺序**：按子范围编号增序逐层迁移验证。子范围 1 + S → 子范围 2a,2b → 子范围 4-7 → 子范围 8。"的表述。"迁移顺序"（migration order）及箭头记号为典型的执行计划措辞，与"已完成的基线分析记录"定位不符。虽然该顺序来源于依赖分析（是分析结论，可归为回顾性记录），但"迁移顺序"一词本身传达了未来的执行语义，读者会视为"应按照此顺序进行下一步操作"。
- **改进建议**：改为"依赖编译次序"或"依赖约束"等回顾性措辞，例如："各子范围的编译依赖约束为：S → 1 → 2a,2b → 4-7 → 8，按此依赖次序编译可满足各子范围的依赖闭合性。"

### P3. 附录 A "当前状态"列系统性残留规划性语言
- **所在位置**：附录 A，第 637-652 行，"当前状态"列
- **严重程度**：一般
- **问题描述**："当前状态"列系统性使用了面向未来的能力描述："可独立迁移"、"依赖 Matrix 类型迁移完成后可启动"、"可随 Vec 类型同步迁移"、"依赖核心类型迁移完成"。附录头部声明（第 635 行）称"不包含各组件的详细依赖分析和迁移规划"，但列内容与声明矛盾——大量"可……"句式描述的是"未来可以做什么"，而非"已完成的依赖关系分析"。此问题在迭代 v2（Issue 2）和 v4（Issue 2）中已被反复提出，当前虽增加了头部声明来调和，但列内容本身未做去规划化改造，属于以声明覆盖实质问题的不彻底处理。
- **改进建议**：将"当前状态"列重构为"依赖关系"列，仅陈述事实性的依赖关系。例如：将"依赖已迁移类型，可独立迁移"改为"依赖 Vec1~Vec4 完整定义，无额外编译逻辑"；将"依赖 Vec 类型迁移完成后可启动"改为"内部数据由向量列组成，向量类型为其编译前置依赖"。

### P4. §4.4 验收准则的验证主张缺乏可追溯证据
