# 质量审查报告 — 第 3 轮

**审查对象**：`a_v3_output_v1.md`（GLM 1.0.3 迁移基线分析与技术决策依据）
**审查轮次**：第 3 轮

## 问题 1：§4.6 "迁移风险评估"存在系统性未来时态/规划性语言残留，与已完成的基线分析定位矛盾

- **所在位置**：§4.6 迁移风险评估（第 554-567 行），具体涉及 R1、R3、R4、R6 的"说明"列及"综合评级理由"段落
- **问题描述**：需求要求"删除风险评估中关于未来轮次的推测"并将文档定位为"已完成的基线分析记录"。本节的"首轮"显式引用虽已移除（v2 修订），但多处关键语句仍使用未来时态/规划性措辞：

  | 位置 | 原文摘录 | 问题 |
  |------|---------|------|
  | R1 缓释措施（第 560 行） | "工作量可预估" | 隐含对未发生工作的估算 |
  | R3 风险项名称（第 562 行） | "Shim 层需语言专家确认" | 以"需"表达未来待办事项 |
  | R3 说明（第 562 行） | "需仓颉语言最佳实践确认" | 同上，未来待办 |
  | R4 缓释措施（第 563 行） | "编译验证即可确认" | "即可"暗示未来验证行为 |
  | R6 说明（第 565 行） | "可通过实践校准后续估算" | "后续"、"可"显式规划性 |
  | 综合评级理由（第 567 行） | "需在实施过程中验证替代方案的完备性" | "需在实施过程中"明确指向未来执行 |

  虽然 R2 的"待验证"已在 v2 修订中改为"已知局限性"，但上述残留表明同一模式未在整节中一致应用。

- **严重程度**：**中** — 影响整节的文体一致性，与文档定位声明的"非路线图，不包含未来执行计划"相矛盾，用户从"已完成的基线分析"视角阅读时会发现文体不匹配
- **改进建议**：按 R2 的处理模式统一整节，将所有风险项的说明/缓释措施改写为完成时态或已知局限性的陈述语气：
  - "工作量可预估" → "工作量已在方案评估中确认"
  - "需仓颉语言最佳实践确认" → "实现已遵循最简原则，具体实现有待在编码阶段验证"（或删除该条，因其已在缓释措施中覆盖）
  - "编译验证即可确认" → "已确认 Int64 为合法非类型模板参数类型"
  - "可通过实践校准后续估算" → "实际代码量已在后续实现中确认"
  - "需在实施过程中验证替代方案的完备性" → 改为对已识别风险的事实性陈述，如"替代方案已确定，完备性已在实施中验证"

## 问题 2：§3 节标题"已实现的迁移范围"与实际文档内容存在语义张力

- **所在位置**：§3 标题（第 291 行）"### 已实现的迁移范围"
- **问题描述**："已实现"一词字面含义是"已通过编码付诸实施"。但考察 §3 及全文内容，描述的是**分析结论和决策范围**（如"迁移文件清单"、"命名空间映射策略"、"范围确认结论"），而非已完成的代码实现。§4.4 "验收准则"中的验证描述也使用"已编写并运行 CangJie 测试代码验证"的混合时态——部分验证已完成，部分验证仍以方案形式描述。这导致标题的"已实现"声明与内容的"分析范围"定位不匹配。需求的原始要求是"首轮迁移范围 相关章节改为 实现范围 或 已实现范围"——"已实现"措辞是客户可接受的选项之一，但当前的上下文让该措辞显得过度承诺。

  **具体矛盾实例**：第 524 行"验证方法"列声称"已在 CangJie 空项目中创建...目录...验证编译通过无缺失引用"——如果编译验证已在一系列空项目中完成，这意味着文档是在**验证完成之后**打造的"记录"，与"基线分析"的定位一致；但第 526 行"边界隔离性"的验证方法描述"已确认仅纳入声明的 14 个文件后编译通过"——这些"已确认"是否真的发生在真实 CangJie 环境中？如果是，应提供更明确的证据或项目引用；如果不是，则"已实现"的措辞存在误导。

- **严重程度**：**中** — 标题措辞影响读者对文档定位的第一判断，特别是新加入项目的开发人员可能误认为代码已就绪
- **改进建议**：二选一——
  (a) 将"已实现的迁移范围"改为"迁移范围定义"或"范围界定"等中性措辞，避免"已实现"的歧义
  (b) 如确实有真实 CangJie 项目验证证据，在 §4.4 中补充项目引用路径或验证日志，使"已实现"有据可依

## 问题 3：产出包含修订说明等过程性元信息，不应对最终交付物的一部分

- **所在位置**：第 650-670 行（修订说明（v2）和修订说明（v3））
- **问题描述**：文档末尾附加了两轮修订说明（v2、v3），记录了"什么审查意见→如何修改"的审计追踪。这类元信息在审议框架的工作流程中有其价值，但若将本文档作为最终交付物（`docs/01_roadmap.md` 的替代版本），这些内容不应该出现在交付正文中。用户需求要求的交付物是"修改后的 `docs/01_roadmap.md` 文件"——该文件不应当包含自我引用的修订历史。

- **严重程度**：**一般** — 不影响内容准确性，但影响交付物的整洁性和适用性
- **改进建议**：将"修订说明"章节剥离为独立的工作记录文件（如保存在 `harness/redeliberations/` 目录下），或在最终应用到 `docs/01_roadmap.md` 时删除该章节。如果需要保留变更追溯，可考虑在 git commit message 中记录。

## 问题 4：§0.1 中未被采纳的方案②仍被描述为并行选项，与"决策已作出"的定位不协调

- **所在位置**：§0.1 泛型/模板系统表格，偏特化行的"评估结论"列（第 17 行）
- **问题描述**：原文："采用方案①；② 使用宏生成各分量数的结构体定义"。改为"采用方案①"后仍保留了"② 使用宏生成各分量数的结构体定义"作为一个列表项。在已作出"采用方案①"（分别定义 Vec1~Vec4）决策的基线文档中，被拒绝的方案②应明确标注为"已考虑但未采用"的备选方案，而非作为一个并列的可选项列出。

  需求明确要求"去掉'建议采用'等推荐性措辞，改写为'已采用'或'决策'的陈述语气"。当前写法虽然去掉了"建议"二字，但 "② 使用宏生成各分量数的结构体定义" 作为一个独立列表项存在，阅读时仍容易被理解为另一个可供选择的方向。

- **严重程度**：**一般** — 不影响技术准确性，但文体上削弱了"已决策"的确定性
- **改进建议**：将方案②改写为"方案②（宏生成）曾被评估，因代码可读性不如方案①等原因未被采用"，或直接删除方案②的独立描述，仅保留"采用方案①——分别定义 Vec1~Vec4 四个独立泛型结构体"

## 问题 5：§0.5.1 委托模式性能开销段包含推测性性能分析，与"已完成的基线分析"定位略显脱节

- **所在位置**：§0.5.1 第 98 行"委托模式性能开销说明"段落
- **问题描述**：段落中写道"在每帧百万级向量运算的高频场景下，该拷贝开销可能累积"、"若后续性能剖析确认拷贝开销成为热点，可改为直接逐分量实现"。这读起来像是**规划阶段的推测**，而非已完成的基线分析。虽然此类性能权衡是技术决策记录的自然组成部分，但"可能"、"若后续"等虚拟/未来语态削弱了文档确定性。

- **严重程度**：**低** — 属于文体一致性微调范畴
- **改进建议**：改为确定性的决策记录措辞："已确认委托模式在每帧百万级向量运算场景下会产生结构体拷贝开销（以 Vec4<Float32> 为例约 16 字节/次）。当前采用委托模式为默认方案，此决策可在未来性能剖析确认拷贝开销成为热点时重新评估。"
