## 发现的问题

### 问题1：`@OverflowWrapping` 标注范围与计数不一致（二元算术运算符的 `%` 存在矛盾）

| 属性 | 内容 |
|------|------|
| **问题描述** | §0.5.1 的"二元算术运算符分类表"（第83行）列出需标注 `@OverflowWrapping` 的二元算术运算符为 5 个：`+`、`-`、`*`、`/`、`%`。但同节的"标注工作量"段落（第98行）计算标注总量时仅统计其中 4 个运算符（`+`、`-`、`*`、`/`），得出 4 类型 × 4 运算符 = 16 处标注，排除了 `%` 运算符。两处对同一范围（"二元算术运算符"）的声称相互矛盾：若 `%` 需标注，标注量应为 4 × 5 = 20 处而非 16 处；若 `%` 已被排除（同 `%=` 一样），则第83行的表格不应将其列入"二元算术"类别并要求标注。第98行的源文件核查仅提及 `%=`（取模赋值）的检查结果，未明确说明二进制 `%`（取模）运算符的处理方式。 |
| **所在位置** | 第83行（分类表）vs 第98行（标注量计算），§0.5.1 |
| **严重程度** | **严重** |
| **改进建议** | 二者取其一：(a) 若二进制 `%` 运算符也在迁移范围内且需 `@OverflowWrapping`，在第98行将标注量更正为 4 × 5 = 20 处，并补充说明 `%` 的 `@OverflowWrapping` 标注位置；(b) 若二进制 `%` 运算符被排除，在第83行表格中将其移出"二元算术"类别并标注排除原因（同 `%=` 的处理方式），确保分类表与标注量计算一致。 |

### 问题2：§4.4 "范围定义标准"的结构与其前提声明存在语义张力

| 属性 | 内容 |
|------|------|
| **问题描述** | 第541行明确声明本节"非未来验证计划"（即本节内容是范围界定标准，而非验证报告）。但该节的表格（第543-551行）同时包含以下具有矛盾指向的元素：(a) "验证方法"列——描述具体的验证操作步骤（如"在 CangJie 空项目中创建 `src/glm/` 目录，将 14 个文件...放入"），这本质上是一份验证方案描述，与"非验证计划"的声明直接冲突；(b) "范围约束条件"列——所有约束条件均使用完成时态/事实性措辞（"0 编译错误"、"测试用例全部通过"、"核查通过"），而非条件性措辞（"应达到 0 编译错误"、"须全部通过"）。即使将"范围约束条件"列解读为"定义了须满足的标准"，完成时态的措辞仍使读者无法区分"这是已完成的验证结果"还是"这是需要达成的条件"。该结构导致读者无法确定 §4.4 的准确用途——是范围定义、验证方案还是验证报告。 |
| **所在位置** | 第539-551行，§4.4 |
| **严重程度** | **严重** |
| **改进建议** | 两种重构方案择一：(a) 若本节的定位确定为**范围定义标准**（"非未来验证计划"），则删除"验证方法"列（或将其降级为注释而非表格列），并将"范围约束条件"列的措辞改为条件性/义务性表达（如"应满足：14 文件按依赖拓扑顺序编译，0 编译错误且 0 文件缺失"）；(b) 若本节需要保留验证方法，则删除"非未来验证计划"的声明，改为"范围定义与验证方案"，并补充实际的验证证据（文件路径、输出日志、测试代码锚点）。注意：此问题自内部审议第6轮起反复标记（对应原 §4.4 验收准则的证据缺失问题），历次通过改换标题和措辞尝试修复但未触及根本结构矛盾。 |

### 问题3：§4.5 备选方案表格未对已排除方案做明确标记

| 属性 | 内容 |
|------|------|
| **问题描述** | §4.5 的备选方案对比表（第567-571行）并列列出三个方案 A/B/C，仅方案 A 标注了"采用此方案"。方案 B（重新设计 swizzle 机制）和方案 C（使用 CFFI unsafe 绕过）未作任何标记来表明它们已被评估并排除。"决策结论"段落（第573行）独立于表格之外声明"采用方案 A"，但表格本身缺乏对 B/C 的"已排除"标记。当读者快速浏览表格时（尤其当该文档作为决策依据被引用时），无法在表格内直接看到 B/C 的处置状态，必须结合外部段落判断。此问题与第3轮内部审议发现的 §0.1 中未采纳方案缺乏标记的问题属于同一模式。 |
| **所在位置** | 第567-573行，§4.5 |
| **严重程度** | **一般** |
| **改进建议** | 在表格中为方案 B 和 C 的"影响"列末尾添加明确的"**已排除**"或"**已考虑但未采纳**"标记，使读者在表格内即可获知各方案的最终处置状态。 |

## 整体评价

产出在**需求响应层面**已实质性满足用户要求：文档定位声明、措辞去规划化（首轮→迁移实施、建议→决策）、轮次讨论删除、冗余合并、标题调整等核心需求均已落实。前10轮内部审议反复标记的修订说明嵌入交付物正文的问题已根本解决（v11 已无修订说明章节），`GLM_CONFIG_SWIZZLE` 枚举值错误也已修正。

剩余质量问题集中在**内部一致性**（问题1：`%` 运算符标注范围与计数矛盾）和**结构定性**（问题2：§4.4 的定位与结构矛盾导致读者无法确定其用途），这两个问题阻塞直接交付。问题3 和 4 为可修复的完善性缺口。建议在 v12 轮次中优先解决问题1（事实性错误）和问题2（结构矛盾），其次修复问题3 和 4。
