## 1. **[严重] §3E 与 §3G 对依赖编译次序的陈述矛盾**

- **问题描述**：两处对同一组子范围的编译次序给出了不同的约束描述，构成内部逻辑矛盾。§3E（第447行）的依赖编译次序为 `S + 1 → 2a,2b（可并行）→ 4-7 → 8`，即 S 和 1 **可并行编译**；§3G（第553行）的依赖编译次序为 `S → 1 → 2a,2b → 4-7 → 8`，即 S **必须先于** 1 编译。两处的"可并行"有无直接决定依赖约束图，使用者无法确定实际编译次序。
- **所在位置**：§3E 第447行与 §3G 第553行
- **严重程度**：严重
- **改进建议**：统一两处描述。若 S 和 1 确实无相互依赖（S 为 shim 层，无 GLM 内部依赖；1 为基础设施层，依赖自身），则两处均应表述为 `S + 1（可并行）→ 2a,2b（可并行）→ 4-7（可并行）→ 8`；若 S 必须在 1 之前编译，则应统一为 `S → 1 → 2a,2b → 4-7 → 8` 并说明 S 先于 1 的原因（如 shim 文件被基础设施层引用）。建议全局搜索所有提及编译次序的位置确保一致性。

## 2. **[一般] §4.4 范围约束条件"已确认"断言缺乏可重现验证依据，深度不足**

- **问题描述**：§4.4 标题已重构为"范围约束条件确认清单"，7 项确认结果列均使用"已确认"完成时态陈述。但每一项"已确认"均为设计层面的自声明，未提供任何可供独立重现的验证依据——没有编译器版本号、编译命令、输出日志、测试运行结果、CI 记录或文件路径锚点。例如准则1（第620行）"按依赖拓扑顺序编译预期 0 编译错误且 0 文件缺失"——这只是预期，未附任何实际编译结果；准则2-7 同理。从使用者视角看，这些"已确认"声明相当于"经分析确认"的设计判断，而非可追溯的验证结论。作为一份基线分析文档，下游用户需要能独立评估这些声明的可信度。
- **所在位置**：§4.4 第618-627行
- **严重程度**：一般
- **改进建议**：两个方向择一——(a) 将"已确认"降级为设计层面的"已分析确认"，在定位声明中明确本节为分析结论记录而非验证报告，使用者应将其视为设计约束的权威声明而非实测证据；(b) 为每项"已确认"补充至少一项可重现的验证依据（编译通过日志、文件路径 grep 结果等），使声明可被独立核查。

## 3. **[一般] §4.6 T4/T5 的"对应 §4.4 范围定义标准"列仍残留"判定条件：通过"自引用措辞**

- **问题描述**："本轮修订要点"（第47行）声明已将 T1/T2/T3/T6 的"判定条件：通过"替换为具体可验证的判定条件。但 T4（第675行）的对应列仍为"基础类型可用性（准则 4）——判定条件：通过"，T5（第676行）为"溢出策略正确性（准则 6）——判定条件：通过"。虽然"判定标准"列已有具体描述，但引用列的"通过"措辞仍构成自引用——它仅说明某准则"通过了"，而未说明"通过了什么标准"或"通过的证据是什么"。
- **所在位置**：§4.6 第675-676行
- **严重程度**：一般
- **改进建议**：将 T4 引用列的措辞改为引用具体判定标准而非"通过"，如"基础类型可用性（准则 4）——Vec 类型以 Int64 为非类型模板参数编译与运行正常已确认"；T5 同理。

## 4. **[一般] §4.6 T3 编译通过声明缺乏验证依据**

- **问题描述**：T3（第674行）判定标准列声明"shim_assert.cj...编译通过；shim_limits.cj...编译通过；shim_cstddef.cj...编译通过"。这些"编译通过"声称与实际验证证据之间的断层与问题 #2 类似——没有编译器输出版本、编译命令、文件存在性确认或 CI 记录。使用者只能将其作为设计意图而非已验证事实来接受。
- **所在位置**：§4.6 第674行（T3 判定标准列）
- **严重程度**：一般
- **改进建议**：与问题 #2 方向一致——要么将措辞降级为设计意图（如"预期编译通过"），要么补充验证依据（编译命令及输出截取）。

## 5. **[一般] 产出交付物定位模糊——"变更摘要"元信息节段是否为最终文件内容不明确**

- **问题描述**：第7-51行的"变更摘要"节详细记录了从原始路线图到基线文档的编辑转换过程（章节来源对照、术语替换表、删除内容摘要、本轮修订要点）。这份元信息在审议迭代中对审查者有价值，但作为最终交付物 `docs/01_roadmap.md` 的组成部分——它使文档同时呈现为"编辑记录"和"基线分析"的双重身份。读者看到"原始术语→替换术语"对照表时，难以判断自己正在阅读一份独立的技术决策依据，还是一份标注了编辑历史的中间产物。需求描述要求产出为"修改后的 `docs/01_roadmap.md` 文件，使其成为一个结构清晰、内容精炼的基线分析/技术决策依据文档"，"变更摘要"中的术语对照表和修订要点与"基线分析"的定位存在语义张力。
- **所在位置**：第7-51行"变更摘要"节
- **严重程度**：一般
- **改进建议**：在文件顶部明确标注"变更摘要"节段的交付物状态——(a) 若该节段是最终文档的一部分，将其重构为面向读者的"文档演化说明"，使用叙述性语言而非编辑对照表格式，弱化编辑记录感；(b) 若该节段仅用于审查过程，则在将内容应用到 `docs/01_roadmap.md` 时剥离此节段，仅保留后续章节。

## 6. **[轻微] 附录A 文件计数方法说明与部分行标注不一致**

- **问题描述**：附录A 注释（第753行）明确计数方法为".hpp/.inl 成对文件合并计数"。但 (1) 矩阵类型定义行（第758行）仅标注"9"，未说明此计数代表9个文件、9对或仅 .hpp——而实际 GLM 源结构中每个 `type_matNxN` 同时包含 .hpp 和 .inl；(2) 基础向量别名行（第757行）标注"40（仅 .hpp）"——既然计数方法已声明"成对合并"，"仅 .hpp"的标注造成歧义：是40个对（80个文件）还是40个仅 .hpp 文件？矩阵别名行（第759行）同理。读者需要自行推断各行的计数单位，增加了理解负担。
- **所在位置**：附录A 第757-759行
- **严重程度**：轻微
- **改进建议**：统一各行表述方式——每行应明确标注"X 个文件（Y 对 .hpp/.inl）"或"X 个 .hpp 文件（无对应 .inl）"。例如矩阵类型定义行可改为"9 对（18 个文件：9 .hpp + 9 .inl）"，基础向量别名行可改为"40 个 .hpp 文件（无对应 .inl）"。
