# 质量审查报告：a_v2_copy_from_v1.md（OOD 阶段二迁移 v35）

## 审查范围

以需求文档（requirement.md）为基准，从需求响应充分度、事实正确性、逻辑一致性、落地可用性四个维度审查 OOD 产出。审查排除内部审议已覆盖的技术可行性维度，侧重内部审议未充分覆盖的维度。

## 问题 1

**问题描述**：`fromMat` 6a/6b 的 144 个签名中，`fromMat` 6a（同类型不同尺寸）的 `one: T` 参数在"纯收缩方向下被忽略"的设计抉择存在 API 可用性缺陷——调用方必须为 `one` 传入一个无意义的 dummy 值，这在 API 语义上构成误导，且易导致调用方误认为 `one` 值对收缩结果有影响。文档仅在注释和段落说明中标注"被忽略"，编码者容易遗漏此约束，下游消费者也无法从签名本身获知此行为。

**所在位置**：§3.3 item 6 纯收缩方向 `one` 参数行为说明（L286 附近）

**严重程度**：一般

**改进建议**：考虑两种改进方案之一：(1) 为纯收缩方向提供不含 `one` 参数的独立重载（72 个收缩方向 × 2 变体 = 144 个额外签名，量虽大但可通过代码生成脚本解决，且 API 语义正确）；(2) 至少设计文档中将 `one` 在纯收缩方向的 dummy 语义通过函数名显式表达（如 `fromMatShrink`），或在 §5 错误处理策略中将此列为已知 API 设计缺陷并标注迁移注意事项。若坚持保留 `one` 参数的统一签名，须在 §9 差异声明中新增一条，明确声明此为仓颉语言泛型限制导致的接口偏差。

## 问题 2

**问题描述**：`geometric.cj` stub 签名中 `dot` 函数提供了 Vec1 的重载（`func dot(x: Vec1<T, Q>, y: Vec1<T, Q>): T`），但 GLM 1.0.3 的 `geometric.hpp` 中 `dot` 仅声明了 Vec2~Vec4 的版本，不含 Vec1。对标 GLM 1.0.3 源码 `geometric.hpp`（第 36-44 行），其模板约束为 `length_t L >= 2`，Vec1 的 dot 未在 GLM 中提供。此为 OOD 新增的非 GLM 对齐功能，但设计文档中未标注此偏差，也未说明添加 Vec1 dot 的理由。

**所在位置**：§3.7 geometric.cj 完整函数签名清单（L645）

**严重程度**：一般

**改进建议**：(1) 若决定保留 Vec1 dot，应在 §9 差异声明中新增条目说明此为仓颉版扩展（非 GLM 对齐）；(2) 若决定严格对齐 GLM，移除 Vec1 dot 重载，保持与 GLM 一致。

## 问题 3

**问题描述**：`diagonal(scalar: T)` 工厂函数仅对非方阵不可用（D34 声称"非方阵因无 diagonal(scalar) 语义（非方阵对角矩阵构造不通用），仅提供 identity(one)"），但在数学上，非方阵的对角矩阵构造是良定义的——对角线位置 `min(C,R)` 内填入 scalar，其余填零。GLM 中非方阵的 `mat(T s)` 构造函数（如 `type_mat2x3.inl:34-45` 的 `mat(T s)` 构造为对角线填 s、其余填零）正是此语义。设计文档混淆了"一般对角矩阵"与"标量对角构造"：前者确实不通用（不同行数列数有不同的对角元素数），但 `diagonal(scalar)` 作为"对角线=scalar、其余零"的特定语义是通用且与 GLM 的 `mat(T s)` 行为完全一致的。当前 `identity(one)` 实际上就是 `diagonal(one)` 的特例，将它们拆分为两个函数但仅在方阵上同时提供，在非方阵上只提供 `identity` 而不提供 `diagonal`，造成 API 不对称——非方阵用户若需"对角线=s（s≠1）"的构造，无直接 API 可用。

**所在位置**：§3.3 item 3 diagonal 构造函数说明（L231-236）；§7 D34（L753）；§9 差异声明 diagonal 条目（L868）

**严重程度**：中等

**改进建议**：重新评估 `diagonal(scalar: T)` 对非方阵的适用性。数学上"对角线元素=scalar、其余=零"对非方阵是良定义的（如 GLM 的 `mat(T s)` 构造）。建议对所有 9 个矩阵类型均提供 `diagonal(scalar)`，与 `identity(one)` 形成完整配对。若保留当前设计（仅方阵有 diagonal），应在 §9 差异声明中明确声明此为仓颉设计与 GLM `mat(T s)` 构造的行为偏差——GLM 对非方阵也提供标量对角构造。

## 问题 4

**问题描述**：矩阵-矩阵除法 `Mat / Mat` 的设计声称"仅方阵"且"内部实现为 `this * inverse(rhs)`"，但该实现方式在 `rhs` 不可逆（奇异矩阵）时的行为未定义。当前设计依赖 `inverse` stub（本阶段抛异常），阶段三完整实现后，`inverse` 对奇异矩阵的行为是什么？GLM 中 `inverse` 对奇异矩阵返回未定义值（C++ 不检查），仓颉版需决定：是否在 `inverse` 内部做可逆性检测并抛异常？还是在 `/` 运算符内部做防御性检查？这直接影响下游消费者对 `Mat / Mat` 的错误处理策略，但设计文档对此场景完全沉默。

**所在位置**：§3.5 矩阵-矩阵除法签名（L515-521）；§5 错误处理策略（L713）

**严重程度**：中等

**改进建议**：(1) 在 §5 错误处理策略中新增"矩阵除法奇异矩阵行为"条目，明确 `inverse` 的奇异矩阵行为策略（抛异常 / 返回未定义值 / 返回 Option），以及 `Mat / Mat` 运算符是否需要防御性检查；(2) 若 `inverse` 在奇异矩阵上抛异常，应在 §5 同步声明 `Mat / Mat` 在该场景下会传播异常；(3) 若 `inverse` 返回未定义值（与 C++ GLM 一致），应在 §9 差异声明中标注"矩阵除法对奇异矩阵行为未定义，与 C++ GLM 一致"，供迁移者参考。

## 问题 5

**问题描述**：`scalar_mat_ops.cj` 的签名模板中将 `T <: Number<T>` 约束应用于全部 4 种运算（add/sub/mul/div），但 `add` 和 `sub` 在语义上仅需要 `+`/`-` 运算符，`Number<T>` 虽包含它们但约束过强——Bool 类型无法满足 `Number<T>` 约束，因此 Bool 矩阵无法使用 `add(s: Bool, m: Mat)` 进行布尔标量加法。然而在 GLM 中，`mat + scalar` 对 Bool 矩阵也是不提供的（GLM 的 `mat<C,R,T,Q>::operator+` 对 Bool 元素类型同样不可用，因 GLM 内部 static_assert 限制为浮点/整型），所以当前约束在结果上与 GLM 一致。但设计文档未显式说明此约束排除决策，编码者可能误认为这是仓颉限制导致的偏差而非有意设计。

**所在位置**：§3.5 scalar_mat_ops.cj 签名模板（L528-549）

**严重程度**：提示

**改进建议**：在 `scalar_mat_ops.cj` 段落补充注释：标量-矩阵运算的 `Number<T>` 约束有意排除了 Bool 矩阵（与 GLM 对 `mat + scalar` 仅支持浮点/整型元素的约束一致），此非仓颉语言偏差。

## 问题 6

**问题描述**：`geometric.cj` stub 签名中 `cross` 仅有 Vec3 版本，但 GLM 1.0.3 的 `geometric.hpp` 中 `cross` 还提供了 Vec2 版本（2D 叉积返回标量）和 Vec4 版本（支持 Vec4 上的叉积，GLM 实现 `cross(Vec4, Vec4) -> Vec4`）。设计文档中 `cross` 仅有 `func cross<T, Q>(x: Vec3<T, Q>, y: Vec3<T, Q>): Vec3<T, Q>` 一条签名，遗漏了 Vec2 和 Vec4 的叉积版本。阶段二虽以 stub 占位，但 stub 签名应完整覆盖方阵 `.inl` 实际引用的函数——`type_mat4x4.inl` 中 `lookAt` 等函数可能间接依赖 `cross(Vec3)`，但 `geometric.hpp` 中 Vec2/Vec4 的 `cross` 是 GLM 标准 API 的一部分。

**所在位置**：§3.7 geometric.cj 完整函数签名清单（L649）

**严重程度**：一般

**改进建议**：(1) 查验 GLM `geometric.hpp` 确认 `cross` 的完整重载列表（Vec2/Vec3/Vec4 各一版），在 `geometric.cj` 签名清单中补齐 Vec2/Vec4 的 `cross` stub；(2) 在 §10 API 覆盖矩阵中同步更新 `cross` 的覆盖维度标注。

## 问题 7

**问题描述**：`fromParts` 签名中 `where Q <: Qualifier` 约束仅出现在部分签名的声明中。§3.3 item 4 的 9 个 `fromParts` 签名示例（L241-249）全部显式写出 `where Q <: Qualifier`，但仔细审查可知 `fromParts` 不使用 T 的任何运算符——参数类型 `U` 可通过 `conv: (U) -> T` 转换，构造仅需赋值操作——实际上 `fromParts` 对 T 不需要 `Number<T>` 约束，仅需 T 可赋值即可。文档中 `fromParts` 仅声明 `where Q <: Qualifier` 是正确的，但缺少显式说明"为何 `fromParts` 不需要 `Number<T>` 约束"的理由，编码者可能困惑为什么不加约束。与 `diagonal/identity`（需要 Number<T>）的对比未在本处出现。

**所在位置**：§3.3 item 4 fromParts 签名（L237-251）

**严重程度**：提示

**改进建议**：在 fromParts 签名段落补充一句约束说明："`fromParts` 仅需赋值和 conv 闭包，无需 `Number<T>` 约束，因此 Bool 矩阵也可使用此构造函数"。此补充可帮助编码者理解约束选择的意图。

## 问题 8

**问题描述**：`[]` 赋值版本的签名 `public mut operator func [](i: Int64, value!: VecR<T,Q>): Unit` 采用命名参数传值模式来绕过值语义副本问题，但这意味着调用方必须使用 `m[0] = value: newCol` 语法而非 `m[0] = newCol`。经核实仓颉函数文档 §8.3 规范，赋值形式 `operator func [](args..., value!: T): Unit` 的调用语法为 `a[i] = v`（无需使用 `value:` 前缀），命名参数 `value!` 仅在定义时声明，调用时不需要带前缀。设计文档中未给出 `[]` 赋值版本的调用示例，编码者可能从签名推断需要 `m[0, value: newCol]` 的调用形式而非直观的 `m[0] = newCol`。

**所在位置**：§3.4 行列访问 `[]` 赋值版本签名说明（L474）

**严重程度**：一般

**改进建议**：在 §3.4 或 §4 行为契约中补充 `[]` 赋值版本的调用示例，明确调用形式为 `m[0] = newCol`（与阶段一 Vec 类型的 `v[0] = value` 一致），消除签名中命名参数带来的调用语法歧义。

## 问题 9

**问题描述**：编码模板示例 1（Mat2x3←Mat2x2，B 类简洁方向）中使用了 `m.c0[0]`、`m.c0[1]` 等下标访问模式来获取列向量的分量。但在阶段一的 Vec 类型实现中，列向量类型 `Vec2<T,Q>` 的 `[]` 取值版本返回的是副本而非引用，且包含 `assert + throw`，非 const。此编码路径在运行时是正确的（值正确），但效率上不如直接使用属性访问 `m.c0.x, m.c0.y`。更重要的是，编码模板示例作为编码者的参考实现，其风格会直接影响 144 个 fromMat 重载的实现质量——若编码者照搬 `[0]`/`[1]` 模式，144 个函数将全部带来不必要的 assert 开销和副本创建。

**所在位置**：§3.3 编码模板示例 1（L406-423）

**严重程度**：一般

**改进建议**：将编码模板示例中的 `m.c0[0], m.c0[1]` 改为 `m.c0.x, m.c0.y` 等属性访问模式，并注明"优先使用列向量属性访问（`.x/.y/.z/.w`）而非下标访问（`[i]`），前者无 assert 开销且可 const"。

**所在位置**：§3.3 item 6 末尾（L284）；规则化编码策略段落（L386-398）

**严重程度**：提示

## 问题 11

**问题描述**：`common.cj` 中 `mod` 函数签名标注为"暂定"，且注释指出"Number<T> 不直接提供 % 运算符，最终实现需拆分为整数版本和浮点具体类型重载"。但此"暂定"标注仅出现在 §3.7 签名清单和 §10 覆盖矩阵中，§5 错误处理策略中未提及 `mod` 的暂定状态——若编码者按当前签名实现 `mod` stub，使用 `Number<T>` 约束，则在阶段四完整实现时会发现浮点类型无法实例化（编译错误），需要返工拆分签名。设计文档应在编译顺序或编码启动注意事项中标注此风险为阻断性项。

**所在位置**：§3.7 common.cj mod 签名（L578-579）；§8 编码启动前验证项（L771）

**严重程度**：中等

**改进建议**：将 `common.cj` 的 `mod` 签名暂定性从"注释"提升为"编码阻断项"——在 §8 编码启动前验证项中新增：验证 `mod<T>(a: T, b: T): T where T <: Number<T>` 对 Float32/Float64 是否可编译通过；若不能，应在编码首日修正签名（拆分为 `Integer<T>` 约束版 + Float32/Float64 具体类型重载），避免阶段四返工。

## 问题 12

**问题描述**：`lib.cj` 现有的 `mod` 导出（L5：`public import glm.detail.{ add, sub, mul, div, mod }`）来自 `scalar_vec_ops.cj`，但阶段二新增的 `scalar_mat_ops.cj` 中不包含 `mod`（矩阵取模无数学意义）。然而阶段二完成后，`lib.cj` 的 `mod` 导出将同时覆盖向量版本和（未来可能的）额外版本。设计文档 §8 正确说明"现有 mod 导出仅需覆盖向量版，阶段二无新增修改"，但未考虑一个潜在的重载歧义风险：当阶段二新增的 `scalar_mat_ops.cj` 的 `add/sub/mul/div` 全局函数与现有的 `scalar_vec_ops.cj` 的同名函数共存于 `glm.detail` 包时，调用 `add(s, m)` 的重载解析是否能正确区分标量+矩阵与标量+向量？两者参数模式相同（`s: T, m/v: Mat/Vec<T,Q>`），重载解析按参数类型区分应能工作，但设计文档未显式确认此交互。

**所在位置**：§2 scalar_mat_ops.cj 条目（L74-76）；§8 lib.cj 新增导出清单（L766-770）

**严重程度**：提示

**改进建议**：在 §8 编码启动前验证项中增加一项：验证 `add(s: T, m: Mat2x2<T,Q>)` 的重载调用不与现有 `add(s: T, v: Vec2<T,Q>)` 产生歧义（因参数类型 `Mat2x2<T,Q>` vs `Vec2<T,Q>` 不同，理论上是安全的，但应编码验证）。

## 问题 13

**问题描述**：`geometric.cj` stub 签名中未提供 `dot(Vec1<T,Q>, Vec1<T,Q>): T` 以外的标量点积替代。GLM 中 `dot` 的标准签名实际上接受任意长度 `L >= 1` 的向量，GLM 内部通过模板 `vec<L, T, Q>` 实现。仓颉无法模板化长度维度，需要手动列举 Vec1~Vec4。OOD 中已有 Vec1~Vec4 的 dot 重载，但根据 GLM `geometric.hpp` 源码，`dot` 的模板约束实际为 `length_t L >= 1`，理论上对所有长度皆可用。OOD 提供 Vec1~Vec4 的 dot 是合理的（覆盖 C=1~4），但此处的核心问题是——**阶段一的 Vec 实现中是否已存在 `dot` 函数？** 如果阶段一已在 `type_vecN.cj` 的 extend 块或独立文件中实现了 Vec 的 `dot` 成员函数或全局函数，阶段二的 `geometric.cj` 中的 `dot` stub 将与已有实现产生重复定义冲突。设计文档未说明阶段二 `geometric.cj` 中的 dot 与阶段一已有 dot 实现的关系。

**所在位置**：§3.7 geometric.cj stub 签名清单（L645-648）

**严重程度**：一般

**改进建议**：(1) 检查阶段一 `cjglm/src/detail/` 中是否已存在 `dot` 的自由函数或成员函数实现；(2) 若存在，在 `geometric.cj` stub 中注明"与阶段一已有 dot 的关系：阶段二 stub 作为同名函数的更外层 API 重新导出"或"需确保不产生重复定义"；(3) 若不存在，确认 dot 是首次在阶段二作为 stub 引入。

## 问题 14

**问题描述**：`fwd.cj` 新增别名的对照汇总表中，"f32 PascalCase" 的"4 精度 × 9 尺寸 = 36"中的"4 精度"计数似乎包含了 `defaultp`。但仓颉版中不存在 GLM 的 `defaultp` 概念（仓颉不使用 GLM 的默认精度机制，阶段一用显式 `PackedHighp` 等替代），因此实际精度级别应为 3（lowp/mediump/highp），而非 4。文档中"4 精度"的计数与仓颉版的实际精度实现不一致。虽然后续 v33 修订说明 Q3 已将排除统计改为定性描述，但 §8 别名对照汇总表中的"4 精度"字样仍存在。

**所在位置**：§8 fwd.cj 别名对照汇总表（L835）

**严重程度**：提示

**改进建议**：将"4 精度"修改为"3 精度（lowp/mediump/highp）"，或注明"GLM 中含 defaultp 共 4 精度，仓颉版不使用 defaultp，实际 3 精度"。
