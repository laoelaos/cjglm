## 审查发现

### 问题 1：fromMat 6b 中 T(0) 演算方式存在隐含缺陷

**问题描述**：
文档 §3.3 编码模板说明（第 469-471 行）声称"6a（同类型不同尺寸）与 6b（跨类型不同尺寸）函数体结构一致，仅 6b 签名新增 conv 参数"。但此断言忽略了 T(0) 演算在 6b 中的关键差异：

- 6a 模板中 `let zero = m.c0.x - m.c0.x` 正确计算 T(0)，因为 `m.c0.x` 类型为 T，且 `T <: Number<T>` 已由约束保证。
- 6b 中 `m.c0.x` 类型为 U（源元素类型），`m.c0.x - m.c0.x` 计算的是 U(0) 而非 T(0)，且 6b 约束仅包含 `T <: Number<T>`、未声明 `U <: Number<U>`。若 U 未实现 Number<U>（如 Bool），此代码无法编译。

编码者盲目沿袭 6a 模板中的零值演算路径，将在 6b 实现中引入编译错误或语义错误。

**所在位置**：§3.3 编码模板说明（第 469-471 行）、§3.3 item 6b 签名约束声明（第 280-282 行）

**严重程度**：中等

**改进建议**：
1. 在 6b 的实现说明中显式声明 T(0) 的获取方式：由于 U 不保证实现 Number<U>，应使用 `one - one`（基于 `T <: Number<T>` 约束，语义上 1-1=0）或 `conv(sourceValue) - conv(sourceValue)` 计算 T(0)。
2. 为 6b 单独提供一个编码模板示例（或至少一段零值计算代码），与 6a 示例并列展示。
3. 在 §3.3 规则化编码策略的"维度信息获取方式"段落中，增加 T(0) 演算在 6a/6b 下的分支说明。

### 问题 2：scalar_mat_ops.cj 中 div(s, m) 的语义歧义未在设计层面解决

**问题描述**：
文档 §8 编码启动前验证项（第 844 行末尾）将 `div(s: T, m: Mat)` 的语义标注为"需验证此语义等价性"，称其"实际展开为 m / s 的等价变换"。但此语义不确定性源于设计层面的未决矛盾——`add/sub/mul` 的语义明确（标量 ±/* 矩阵各分量），而 `div(s, m)` 若实现为 `m / s`（矩阵各分量除以标量）则与函数名和参数顺序暗示的 `s / m`（标量除以各分量）相反。

将此不确定性推迟至编码阶段验证，意味着 API 设计在编码启动时仍可能发生变化（若验证发现语义不可接受，需重命名或重排参数），这将导致连锁的文档/测试/使用示例修改。设计层面应提前锁定此决策。

**所在位置**：§8 编码启动前验证项（第 844 行末尾）、§3.5 scalar_mat_ops.cj（第 591-613 行）

**严重程度**：中等

**改进建议**：
在 §7 设计决策表中新增决策项，明确锁定 `div(s, m)` 的最终语义和实现方式：
- 若设计意图为 `s / m`（标量逐元素除以矩阵），应直接给出展开公式 `result[j][i] = s / m.c_j[i]`，并移除"待验证"标注。
- 若实际需要 `m / s`（矩阵逐元素除以标量），应将函数按正确语义签名（如交换参数顺序为 `div(m, s)`，或使用不同函数名 `rdiv` 等）以消除参数顺序与语义的错位。

### 问题 3：§10 API 覆盖矩阵缺少 fromMat 系列转换函数的追踪条目

**问题描述**：
§10 "GLM 1.0.3 API 阶段覆盖矩阵"按头文件组织，覆盖了 matrix.hpp、common.hpp、geometric.hpp、mat2x2~4x4.hpp、fwd.hpp、ext/ 等。但 fromMat 6a（同类型跨尺寸，72 个方向）、6b（跨类型跨尺寸，72 个方向）、7（跨类型同尺寸，9 个）作为阶段二的重要构造函数体系，在 GLM 中以 `mat2x3(mat2x2 const& m)` 等形式存在于各 matNxM.hpp 的构造函数中。当前 §10 mat2x2~mat4x4.hpp 覆盖矩阵未单独列出行级条目追踪这些转换函数的覆盖状态，编码者无法从 §10 直观获知哪些跨类型转换已被覆盖。

**所在位置**：§10 mat2x2~mat4x4.hpp 覆盖矩阵（第 1004-1025 行）

**严重程度**：中等

**改进建议**：
在 §10 mat2x2~mat4x4.hpp 覆盖矩阵中新增以下行级追踪条目：
- `fromMat 6a（同类型跨尺寸转换，72 个方向）` → 本阶段实现
- `fromMat 6b（跨类型跨尺寸转换，72 个方向）` → 本阶段实现
- `fromMat 7（跨类型同尺寸转换，9 个）` → 本阶段实现

或至少新增一条汇总条目"跨矩阵类型转换工厂函数"标注覆盖状态和对应章节。

### 问题 4：fromMat 6a/6b 的测试覆盖策略缺少量化指导

**问题描述**：
文档正确统计了 fromMat 6a/6b 合计 144 个签名（第 284 行），并建议使用代码生成脚本。但文档对 144 个方向的测试覆盖仅提到"编码阶段验证要求"（第 476-478 行）列出了一些方向，未给出覆盖率的量化目标。编码者缺乏判断"多少测试算够"的基准，可能导致测试不足或过度测试。

**所在位置**：§3.3 item 6（第 284 行）、§3.3 编码阶段验证要求（第 476-478 行）

**严重程度**：一般

**改进建议**：
在 §8 产出物风险项中给出每个组合方向类别（B 类、双扩展、列缩行扩、列扩行缩、双收缩）的最低测试用例数指导，例如"至少每类组合方向 1 个代表性尺寸的逐元素与 GLM 对照验证"。

## 整体质量评价

文档经过 4 轮内部审议迭代，在技术可行性、签名正确性、约束精确性等维度已高度成熟。核心的架构决策（9 独立结构体、列主序、双层包结构、T(0)/T(1) 统一策略）合理且文档化充分。§10 API 覆盖矩阵和 §9 差异声明为 GLM 对齐提供了系统性参考。

尚存的质量问题主要集中在 **跨类型转换（fromMat 6b）的零值演算遗漏**（问题 1）和 **设计层面未决的 API 语义决策**（问题 2），这两项若在编码阶段暴露将导致返工。建议在设计阶段优先解决。

### 补充评估 B：API 消费者视角

从下游调用方视角评估 API 质量和使用风险：

1. **fromMat 6a/6b one 参数的使用困惑**：纯收缩方向下 `one` 参数在函数体中被忽略但签名保留（§3.3 第 286 行），虽然文档标注了"纯收缩方向下 one 参数被忽略"，但调用方仍需判断当前方向是否为纯收缩方向——调用方必须理解 fromMat 的方向分类才能确定自己传入的 `one` 是否被使用。若调用方在纯收缩方向下传入具有副作用的表达式（如函数调用），可能产生"表达式被计算但结果被忽略"的意外行为。建议在函数签名注释中为纯收缩方向专用标注（如 `// NOTE: one is ignored in pure-shrink directions`）。

2. **div(s, m) 参数顺序与语义错位**：问题 2 已覆盖。这是消费者视角最明显的 API 设计缺陷。

3. **编码模板降低门槛**：§3.3 和 §3.5 的编码模板示例（Mat2x3←Mat2x2、Mat4x4←Mat4x2、Mat2x3×Mat3x2→Mat2x2）为调用方提供了可直接参照的使用模式，降低了 API 使用门槛。

4. **命名一致性和可搜索性**：工厂函数命名为 `fromMat`/`fromParts`/`fromColumns`，与阶段一的 `fromBoolVec` 风格一致，有利于调用方记忆和搜索。

### 补充评估 C：异常场景和边界条件

检查文档对异常/边界场景的处理策略：

| 场景 | 状态 | 处理策略 |
|------|------|---------|
| `[]` 下标越界 | ✅ 已覆盖 | 抛出 Exception（§3.4, §5） |
| `col(i)` 越界 | ✅ 已覆盖 | 抛出 Exception（§3.4） |
| Mat/Mat 除法奇异矩阵 | ✅ 已覆盖 | inverse 推荐抛异常（§5） |
| NaN 比较行为（==/!=/equalExact/equalEpsilon） | ✅ 已覆盖 | 精确标注 NaN 语义差异（§3.6, §5） |
| IEEE 754 NaN 传播（Mat×Vec/Vec×Mat） | ✅ 已覆盖 | 测试用例策略给出（§5） |
| 整数溢出策略 | ✅ 已覆盖 | 统一 @OverflowWrapping（§5） |
| fromMat 6a/6b 中 one 为 NaN 时行为 | ❌ 未覆盖 | 文档未定义 one=NaN 时对角填充行为 |
| fromMat 纯收缩方向 one 被忽略（但被计算） | ✅ 已覆盖 | 注释说明忽略行为（§3.3） |
| Bool 矩阵操作限制 | ✅ 已覆盖 | D33 完整列出不可用操作清单 |
| common.cj mod 对 Float32/Float64 编译失败 | ✅ 已覆盖 | 标注为阻断性验证项（§8） |
| 同包前向类型引用延迟解析失败 | ✅ 已覆盖 | 三层递进 fallback 方案（§8） |
| Mat4x4←Mat4x2 偏差方向 | ✅ 已覆盖 | 独立编码路径标注（§3.3） |

**补充发现**：fromMat 6a/6b 的 `one` 参数文档未定义当 `one` 的值为 NaN 时的行为。理论上 T(0) 通过 `scalar - scalar` 演算不会产生 NaN（除非输入 `scalar` 本身为 NaN 这种不可预知的场景），但 `one` 参数由调用方传入，若调用方误传 NaN 作为 `one`，则对角填充结果全为 NaN。建议在 §3.3 item 6a/6b 的 `one: T` 参数语义说明中标注"调用方应确保 `one` 为非 NaN 值，否则填充行为未定义"。
