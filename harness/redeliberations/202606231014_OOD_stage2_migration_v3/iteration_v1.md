## 需要解决的问题

- **问题描述**：`[]` 赋值版本签名存在双重语法违规——参数名未使用 `value!` 命名参数，且缺少 `mut` 修饰符，将直接导致编译失败。此为历史迭代已修正问题的回归
- **所在位置**：§3.4 行列访问（第 474 行）
- **严重程度**：严重
- **改进建议**：将签名修正为 `public mut operator func [](i: Int64, value!: VecR<T,Q>): Unit`，全文检查所有 `[]` 赋值签名统一修正

- **问题描述**：复合赋值运算符自动生成被标注为"待验证假设"，但仓颉语言规范已明确声明为确定性保证，导致文档包含不必要的手动回退方案且可能误导编码者
- **所在位置**：§3.5 复合赋值运算符段落；§8 编码启动前验证项
- **严重程度**：一般
- **改进建议**：明确声明自动生成为语言规范保证行为，将手动回退方案移至独立附录标注为编译器 bug 防御方案，从 §8 验证项中删除或降级为非阻断性可选测试

- **问题描述**：路线图可选的 `type_float.cj` / `type_half.cj` 在设计文档中完全未提及，需求覆盖存在遗漏
- **所在位置**：全文——§1 概述、§2 模块划分、§7 设计决策表、§8 产出物清单
- **严重程度**：一般
- **改进建议**：在 §7 设计决策表中新增决策，明确 `type_float.cj` / `type_half.cj` 的阶段归属及取舍理由

- **问题描述**：`lib.cj` 新增导出清单中 `mod` 函数与矩阵标量运算导出的交互未显式说明
- **所在位置**：§2 scalar_mat_ops.cj 条目；§8 lib.cj 新增导出清单
- **严重程度**：一般
- **改进建议**：在 §8 lib.cj 导出清单中显式说明 `mod` 不在矩阵标量运算中提供，现有 `mod` 导出仅覆盖向量版且无需修改

- **问题描述**：`abs` 和 `sign` 的泛型约束选用 `Equatable<T>` 而非 `Comparable<T>`，无法满足实现所需的 `<` 比较运算接口
- **所在位置**：§3.7 common.cj — `abs` 和 `sign` 签名（第 585-586 行）
- **严重程度**：一般
- **改进建议**：将约束从 `where T <: Number<T> & Equatable<T>` 修正为 `where T <: Number<T> & Comparable<T>`

- **问题描述**：矩阵-矩阵除法 `Mat / Mat` 的运算符签名完全缺失，编码者无法确定方阵除法运算符的归属和签名
- **所在位置**：§3.5 算术运算符；§10 API 覆盖矩阵
- **严重程度**：一般
- **改进建议**：在 §3.5 补充方阵矩阵-矩阵除法的 3 个签名，标注内部实现为 `this * inverse(rhs)`（依赖 inverse stub），同步更新 §10
