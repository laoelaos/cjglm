## 需要解决的问题

- **问题描述**：`colExtend` 定义中"已包含行扩展/收缩"的注解与先列后行组合规则直接矛盾，可能导致编码者将行扩展逻辑错误混入 colExtend 步骤
- **所在位置**：§3.3 fromMat 四项基本操作定义 — `colExtend` 条目（第 343-344 行）
- **严重程度**：中等
- **改进建议**：将"列向量复制（已包含行扩展/收缩）"修正为"列向量复制（行维度不变，保持源行数 R_src）"，明确 colExtend 结果矩阵行数等于 R_src

- **问题描述**：fromMat 6a/6b 编码策略中 R_dst 获取方式描述先暗示可通过函数调用获取、后又否定，前后矛盾导致编码者困惑
- **所在位置**：§3.3 "规则化的编码策略"段落 — "维度信息的获取方式"子段（第 389-392 行）
- **严重程度**：中等
- **改进建议**：删除暗示函数调用获取的前半句，直接陈述"R_src 和 R_dst 无法通过仓颉静态查询函数获取，在每个 fromMat 重载的函数体中硬编码为字面量"

- **问题描述**：27 个 Mat×Mat 跨尺寸乘法运算符仅提供签名表和结果类型推导规则，缺少编码模板或实现说明，编码者面对 24 个方向的高工作量且易出错的展开逻辑缺乏指导
- **所在位置**：§3.5 算术运算符 — 27 个乘法重载签名表（第 485-513 行）
- **严重程度**：中等
- **改进建议**：补充 1~2 个代表性方向的编码模板示例，注明统一展开定义 `result[j][i] = sum_{k=0}^{C_left-1} left[k][i] * right[j][k]`

- **问题描述**：同包前向类型引用延迟解析被标记为待验证但未列入 §8 编码启动前阻断性验证项，若验证失败且 fallback 均不可行则 Vec×Mat 运算符 API 形态需重新决策，属设计层面风险
- **所在位置**：§8 编码启动前验证项（第 771 行附近），§8 compile order 说明（第 783 行）
- **严重程度**：一般
- **改进建议**：将该原型验证提升为 §8 编码启动前阻断性验证项，明确阻断条件及 API 形态退化时的决策路径

- **问题描述**：`colExtend` 定义中"VecR(...)"的 R 未明确标注是 R_src 还是 R_dst，编码者可能误解
- **所在位置**：§3.3 fromMat 四项基本操作定义 — `colExtend` 条目（第 344 行）
- **严重程度**：一般
- **改进建议**：将"VecR(...)"改为"VecRdst(...)"或标注"VecR 此处 R = R_dst"

- **问题描述**：`matrix.cj` 包含 27 个可直接实现函数和 6 个 stub，被归类为"部分实现文件"但在 §8 编码顺序中被归入纯 stub 批次，可能误导编码优先级判断
- **所在位置**：§1 核心抽象表（第 19 行）及 §2/§8 相关位置
- **严重程度**：一般
- **改进建议**：将 matrix.cj 分类明确标注为"混合型文件"，在 §8 编码顺序中将其从纯 stub 批次独立或添加注释说明
