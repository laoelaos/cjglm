# 质量审查诊断报告（v3 轮次）

## 审查发现

### 问题 1（中等）：`colExtend` 定义中存在自相矛盾的表述，将误导编码者

**所在位置**：§3.3 fromMat 四项基本操作定义 — `colExtend` 条目（第 343-344 行）

**问题描述**：`colExtend` 的定义中写道"N[i] = M[i]（列向量复制，**已包含行扩展/收缩**）"，但 `colExtend` 发生在组合规则的第一步（先列后行），此时行维度尚未变化，M[i] 就是源矩阵列向量，不可能"已包含行扩展/收缩"。行扩展/收缩是独立的第二步 `rowExtend`/`rowShrink` 负责的。括号中的注解与四项基本操作的组合执行顺序（先列后行）直接矛盾——如果 `colExtend` 结果已经包含了行扩展，就不需要第二步 `rowExtend` 了。此误导性表述可能导致编码者错误地将行扩展逻辑混入 `colExtend` 步骤。

**严重程度**：中等

**改进建议**：将"列向量复制（已包含行扩展/收缩）"修正为"列向量复制（行维度不变，保持源行数 R_src）"，明确 `colExtend` 结果矩阵 M' 的行数等于源矩阵行数 R_src。同步确认编码模板示例与修正后的定义一致（当前示例代码是正确的——示例 1 中 `Vec3(m.c0.x, m.c0.y, zero)` 是行扩展结果，出现在 colExtend 之后的 rowExtend 步骤中，但 `colExtend` 定义却声称行扩展已包含在 colExtend 中）。

### 问题 2（中等）：fromMat 6a/6b 编码策略中 R_dst 获取方式描述自相矛盾

**所在位置**：§3.3 "规则化的编码策略"段落 — "维度信息的获取方式"子段（第 389-392 行）

**问题描述**：文档写道"R_dst（目标矩阵类型的 length() 返回的列数对应的行数决定）"，暗示 R_dst 可通过某种函数调用链获取。但紧接着又说"仓颉无行数静态查询函数，R_src 和 R_dst 须在每个 fromMat 重载实现的函数体中硬编码为字面量"。前半句给出了一种推测性的获取方式（且表述含糊——"length() 返回的列数对应的行数"是什么意思？），后半句又否定了该方式。这种先暗示可行再否定的写法容易让编码者困惑：到底能否通过函数调用获取 R_dst？实际上，R_dst 在每个重载中是编译期已知常量，仅能硬编码。

**严重程度**：中等

**改进建议**：删除前半句中暗示 R_dst 可通过函数调用的表述。直接陈述："R_src 和 R_dst 无法通过仓颉静态查询函数获取——`length()` 仅返回列数 C，仓颉无行数静态查询函数。R_src 和 R_dst 在每个 fromMat 重载的函数体中硬编码为字面量（如 Mat2x3 目标中 `let R_dst = 3`），可由代码生成脚本按矩阵类型自动填入。"

---

### 问题 3（中等）：Mat×Mat 跨尺寸乘法运算符缺少编码模板，编码者缺乏实现指导

**所在位置**：§3.5 算术运算符 — 27 个乘法重载签名表（第 485-513 行）

**问题描述**：27 个 Mat×Mat 乘法运算符是本阶段新增的最复杂的算术运算符（24 个跨尺寸方向，每个方向的行/列内积展开不同），但 §3.5 仅给出了签名表和结果类型推导规则，没有提供任何编码模板或实现说明。与之对比，fromMat 6a/6b 有详细的编码模板（两个示例），transpose/matrixCompMult/outerProduct 也各有实现说明。Mat×Mat 乘法反而缺少实现指导，编码者需要自行从 GLM .inl 源码推导跨尺寸乘法的列向量内积展开逻辑——这对 24 个方向而言是高工作量且易出错的。

**严重程度**：中等

**改进建议**：在 §3.5 乘法签名表后补充 1~2 个代表性方向的编码模板示例（如 Mat2x3×Mat3x2→Mat2x2 和 Mat4x4×Mat2x4→Mat4x4），展示跨尺寸矩阵乘法的列向量内积展开模式。注明所有 27 个重载的展开逻辑遵循统一定义：`result[j][i] = sum_{k=0}^{C_left-1} left[k][i] * right[j][k]`，编码者可按此公式对每个方向硬编码展开。

### 问题 4（一般）："同包前向类型引用延迟解析"标记为"待验证"但未提升为阻断性验证项

**所在位置**：§8 编码启动前验证项（第 771 行附近），§8 compile order 说明（第 783 行）

**问题描述**：Vec 类型 extend 块中行向量×矩阵运算符对"仓颉编译器同包内前向类型引用延迟解析"的依赖被标记为【待验证】，且提供了详细的 3 级 fallback 方案。但该验证项未被列入 §8 编码启动前阻断性验证项（仅cjpm子包构建、Mat4x4←Mat4x2偏差验证、@Derive[Hashable]验证、mod签名验证、重载解析验证被列为阻断项）。如果不支持同包前向引用且所有 fallback 方案均失败，Vec×Mat 运算符将完全无法实现（最差情况下 API 从 `v * m` 退化为全局函数 `mulRowVecMat(v, m)`），这是显著的设计风险，应作为阻断项优先验证。

**严重程度**：一般

**改进建议**：将"同包前向类型引用延迟解析"原型验证提升为 §8 编码启动前阻断性验证项，与 cjpm 子包构建验证并列。明确阻断条件：若原型验证失败且方案二、方案三均不可行，则 Vec×Mat 运算符的 API 形态需重新决策（可能退化为全局函数），须在设计层面（而非编码阶段）提前确认。

---

### 问题 5（一般）：`fromMat` 中 `colExtend` 新列构造用的是目标行数 R_dst 但定义措辞模糊

**所在位置**：§3.3 fromMat 四项基本操作定义 — `colExtend` 条目（第 344 行）

**问题描述**：`colExtend` 定义中构造新增列时使用"VecR(...)"，R 在此处语义不明确——是指源行数 R_src 还是目标行数 R_dst？在组合规则（先列后行）的执行顺序下，`colExtend` 发生在 `rowExtend`/`rowShrink` 之前，因此新增列的直接结果应该是 R_dst 维度的向量（因为最终结果需要是 R_dst 维）。结合编码模板示例 2（Mat4x4←Mat4x2），代码 `Vec4<T,Q>(zero, zero, one, zero)` 构造的是 R_dst=4 维向量，确认 `colExtend` 新列应为 R_dst 维。但定义中使用的"VecR"未明确标注 R=R_dst，编码者可能误解为 R=R_src。

**严重程度**：一般

**改进建议**：在 `colExtend` 定义中将"VecR(...)"改为"VecRdst(...)"或直接标注"VecR 此处 R = R_dst（目标行数）"，消除歧义。同理检查 `rowExtend` 定义中"VecR_dst"标记是否已足够明确。

### 问题 6（一般）：§1 核心抽象表中 `matrix.cj` 描述"部分实现文件"的分类不完整

**所在位置**：§1 核心抽象表（第 19 行）

**问题描述**：`matrix.cj` 被描述为"矩阵运算函数库，含可直接实现的函数和 stub 空壳"，类型形态标注为"部分实现文件"。但在 §3.7 和 §8 产出物清单中，`matrix.cj` 包含的 `transpose`、`matrixCompMult`、`outerProduct` 共 27 个可直接实现的函数，其代码量远大于 6 个 stub 空壳（determinant/inverse 仅 6 行 throw 语句）。将 `matrix.cj` 与 `common.cj`/`geometric.cj`（纯 stub，无任何实现）并列分类为"stub 文件"不够准确——§2 包组织图和 §8 编码顺序说明都将此文件与纯 stub 文件混在"stub 文件"批次中，但 `matrix.cj` 的大量可编译实现代码在编译顺序上对"stub 占位"无依赖需求。将其与纯 stub 文件放在同一编译批次可能误导编码者对其实现优先级的判断。

**严重程度**：一般

**改进建议**：在 §1 核心抽象表和 §2 包组织图中，将 `matrix.cj` 的分类从"部分实现 + 部分 stub"明确标注为区别于纯 stub 文件的"混合型文件"，在 §8 编码顺序说明中将其从"stub 文件"批次独立出来或添加注释说明其 27 个函数可直接实现、6 个函数为 stub、整体可在第一阶段编码。

### 问题 7（提示）：非方阵 identity() 与 diagonal() 的示例输出格式存在维度标注歧义

**所在位置**：§3.3 item 8 — identity(one) 在 Mat2x4 上的示例（第 296-307 行）

**问题描述**：示例用"4×2 矩阵形式"展示了 Mat2x4 的 identity 输出，但矩阵标注使用行×列格式（4 行 × 2 列的传统数学表示法），而本文档其他所有位置统一使用列×行（C×R）格式描述矩阵类型（`Mat2x4` 即 C=2, R=4）。这里突然切换到行×列的视觉展示格式，虽然对理解矩阵形状有帮助，但未显式标注"以下为传统行×列视觉展示"与"上方列×行格式"的区别，可能造成维度方向混淆。

**严重程度**：提示

**改进建议**：在矩阵视觉展示前添加格式说明注释，如"以下按传统行×列格式展示矩阵形状（注意：与类型命名 Mat{C}x{R} 的列×行格式方向不同）"，或标注"行标/列标"使方向明确。

### 问题 8（提示）：附录 A 复合赋值手动定义回退方案表中 "add(Mat)" / "sub(Mat)" 条目在 GLM 中无对应运算

**所在位置**：附录 A — 复合赋值运算符手动定义回退方案表（第 1128-1142 行）

**问题描述**：附录 A 表格中非方阵文件的运算符列表为 `+=/-=/*=/ =(T), +=/-=(Mat{N}x{M})`。其中 `+=(Mat)` 和 `-=(Mat)`（同尺寸矩阵加减复合赋值）对应 GLM 的 `operator+=(mat)` / `operator-=(mat)`，属于 GLM 已有运算。但表格标题和注释未区分"此表是编译器自动生成失败时的手动回退方案"与"这些运算符是否在 GLM 中有对应"——编码者可能误以为 `+=(Mat)` 中的 `Mat` 指代跨尺寸矩阵，但实际上仅同尺寸矩阵支持 `+=`/`-=`（跨尺寸矩阵加法结果类型不变，本来就是同尺寸才能 `+=`）。虽不影响设计正确性，但表格中 `+=/=-=(Mat2x3)` 的标注方式可能让编码者对 `Mat` 参数的尺寸约束产生疑问。

**严重程度**：提示

**改进建议**：在附录 A 表格的运算符列中，将 `+=/=-=(Mat2x3)` 明确标注为同尺寸约束（如 `+=/=-=(Mat2x3 同尺寸)`），或添加脚注说明 `+=(Mat)` 中的矩阵参数仅为同尺寸类型。

## 整体质量评价

本产出经过 3 轮再审议迭代，已达到较高的成熟度和完整性。历史迭代的严重问题（`[]` 赋值签名语法违规、Mat×Vec 签名方向错误、乘法结果类型推导规则错误、9×8 转换矩阵表多处标签错误等）均已修正，核心设计规则（fromMat 四项基本操作组合规则、行扩展规则的 on-diagonal one 填充修正、Mat4x4←Mat4x2 DEVIATION 偏差标注）已通过 GLM 1.0.3 源码逐项对照验证。文档的结构完整性良好，§1~§10 和附录 A 覆盖了设计目标、模块划分、核心抽象、行为契约、错误处理、并发设计、设计决策、产出物清单、差异声明、API 覆盖矩阵等完整维度。本次审查发现的 2 个中等问题均为表述歧义类（`colExtend` 自相矛盾、R_dst 获取方式矛盾），不影响核心设计正确性，但可能导致编码实现偏差，建议在编码启动前修正。
