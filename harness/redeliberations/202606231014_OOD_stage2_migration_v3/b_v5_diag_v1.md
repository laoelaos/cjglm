## 审查发现

### 问题 1：fromMat 6b 中 T(0) 演算方式存在隐含缺陷

**问题描述**：
文档 §3.3 编码模板说明（第 469-471 行）声称"6a（同类型不同尺寸）与 6b（跨类型不同尺寸）函数体结构一致，仅 6b 签名新增 conv 参数"。但此断言忽略了 T(0) 演算在 6b 中的关键差异：

- 6a 模板中 `let zero = m.c0.x - m.c0.x` 正确计算 T(0)，因为 `m.c0.x` 类型为 T，且 `T <: Number<T>` 已由约束保证。
- 6b 中 `m.c0.x` 类型为 U（源元素类型），`m.c0.x - m.c0.x` 计算的是 U(0) 而非 T(0)，且 6b 约束仅包含 `T <: Number<T>`、未声明 `U <: Number<U>`。若 U 未实现 Number<U>（如 Bool），此代码无法编译。

编码者盲目沿袭 6a 模板中的零值演算路径，将在 6b 实现中引入编译错误或语义错误。

**所在位置**：§3.3 编码模板说明（第 469-471 行）、§3.3 item 6b 签名约束声明（第 280-282 行）

**严重程度**：中等

**改进建议**：
1. 在 6b 的实现说明中显式声明 T(0) 的获取方式：由于 U 不保证实现 Number<U>，应使用 `one - one`（基于 `T <: Number<T>` 约束，语义上 1-1=0）或 `conv(sourceValue) - conv(sourceValue)` 计算 T(0)。
2. 为 6b 单独提供一个编码模板示例（或至少一段零值计算代码），与 6a 示例并列展示。
3. 在 §3.3 规则化编码策略的"维度信息获取方式"段落中，增加 T(0) 演算在 6a/6b 下的分支说明。

### 问题 2：scalar_mat_ops.cj 中 div(s, m) 的语义歧义未在设计层面解决

**问题描述**：
文档 §8 编码启动前验证项（第 844 行末尾）将 `div(s: T, m: Mat)` 的语义标注为"需验证此语义等价性"，称其"实际展开为 m / s 的等价变换"。但此语义不确定性源于设计层面的未决矛盾——`add/sub/mul` 的语义明确（标量 ±/* 矩阵各分量），而 `div(s, m)` 若实现为 `m / s`（矩阵各分量除以标量）则与函数名和参数顺序暗示的 `s / m`（标量除以各分量）相反。

将此不确定性推迟至编码阶段验证，意味着 API 设计在编码启动时仍可能发生变化（若验证发现语义不可接受，需重命名或重排参数），这将导致连锁的文档/测试/使用示例修改。设计层面应提前锁定此决策。

**所在位置**：§8 编码启动前验证项（第 844 行末尾）、§3.5 scalar_mat_ops.cj（第 591-613 行）

**严重程度**：中等

**改进建议**：
在 §7 设计决策表中新增决策项，明确锁定 `div(s, m)` 的最终语义和实现方式：
- 若设计意图为 `s / m`（标量逐元素除以矩阵），应直接给出展开公式 `result[j][i] = s / m.c_j[i]`，并移除"待验证"标注。
- 若实际需要 `m / s`（矩阵逐元素除以标量），应将函数按正确语义签名（如交换参数顺序为 `div(m, s)`，或使用不同函数名 `rdiv` 等）以消除参数顺序与语义的错位。

### 问题 3：§10 API 覆盖矩阵缺少 fromMat 系列转换函数的追踪条目

**问题描述**：
§10 "GLM 1.0.3 API 阶段覆盖矩阵"按头文件组织，覆盖了 matrix.hpp、common.hpp、geometric.hpp、mat2x2~4x4.hpp、fwd.hpp、ext/ 等。但 fromMat 6a（同类型跨尺寸，72 个方向）、6b（跨类型跨尺寸，72 个方向）、7（跨类型同尺寸，9 个）作为阶段二的重要构造函数体系，在 GLM GLM 中以 `mat2x3(mat2x2 const& m)` 等形式存在于各 matNxM.hpp 的构造函数中。当前 §10 mat2x2~mat4x4.hpp 覆盖矩阵未单独列出行级条目追踪这些转换函数的覆盖状态，编码者无法从 §10 直观获知哪些跨类型转换已被覆盖。

**所在位置**：§10 mat2x2~mat4x4.hpp 覆盖矩阵（第 1004-1025 行）

**严重程度**：中等

**改进建议**：
在 §10 mat2x2~mat4x4.hpp 覆盖矩阵中新增以下行级追踪条目：
- `fromMat 6a（同类型跨尺寸转换，72 个方向）` → 本阶段实现
- `fromMat 6b（跨类型跨尺寸转换，72 个方向）` → 本阶段实现
- `fromMat 7（跨类型同尺寸转换，9 个）` → 本阶段实现

或至少新增一条汇总条目"跨矩阵类型转换工厂函数"标注覆盖状态和对应章节。

### 问题 4：fromMat 144 个签名的实现风险缺少评估

**问题描述**：
文档正确统计了 fromMat 6a/6b 合计 144 个签名（第 284 行），并建议使用代码生成脚本。但从 OOD 设计的完整性角度，以下风险未进行评估：
1. **编译时间影响**：144 个泛型重载签名（每个包含 where 约束链）在 cjc 编译器上的编译时间未评估。仓颉编译器的泛型重载解析对签名数量是否敏感？
2. **二进制体积膨胀**：每个泛型重载在具现化时可能产生独立的代码副本，对库体积的影响未讨论。
3. **测试覆盖要求**：144 个方向中需要多少个测试用例才能达到与 GLM 1.0.3 行为的一致验证？文档仅提到"编码阶段验证要求"（第 476-478 行）列出了一些方向，但未给出覆盖率的量化目标。

**所在位置**：§3.3 item 6（第 284 行）、§3.3 编码阶段验证要求（第 476-478 行）

**严重程度**：一般

**改进建议**：
在 §8 产出物风险项中新增：
1. 评估 144 个泛型重载在 cjc 编译器上的编译时间，并标注此项是否需纳入编码启动前验证。
2. 给出每个组合方向类别（B 类、双扩展、列缩行扩、列扩行缩、双收缩）的最低测试用例数指导，例如"至少每类组合方向 1 个代表性尺寸的逐元素与 GLM 对照验证"。

### 问题 5：9×9 转换矩阵缺少 R_src/R_dst 字面量辅助查找表

**问题描述**：
9×9 fromMat 转换矩阵表（第 383-393 行）使用图例标注每个方向的维度变化模式（如 `colExt(3)→rowExt`），但编码者仍需手动从源/目标矩阵尺寸推导 R_src 和 R_dst 字面量。对于非 B 类方向（63 个混合方向），每次推导均增加出错概率。文档虽已声明 R_src/R_dst 须硬编码（第 400-403 行），但未提供从方向标注到具体字面量的转换辅助。

**所在位置**：§3.3 9×9 转换矩阵表（第 383-393 行）

**严重程度**：提示

**改进建议**：
在 9×9 转换矩阵表下方新增辅助小节，为每个方向直接列出 `(C_src, R_src, C_dst, R_dst)` 四元组字面量，可用精简表格形式呈现，供代码生成脚本直接消费。例如：

| 方向 | C_src | R_src | C_dst | R_dst |
|------|-------|-------|-------|-------|
| Mat2x2→Mat2x3 | 2 | 2 | 2 | 3 |
| Mat2x2→Mat4x4 | 2 | 2 | 4 | 4 |
| ... | ... | ... | ... | ... |

## 整体质量评价

文档经过 4 轮内部审议迭代，在技术可行性、签名正确性、约束精确性等维度已高度成熟。核心的架构决策（9 独立结构体、列主序、双层包结构、T(0)/T(1) 统一策略）合理且文档化充分。§10 API 覆盖矩阵和 §9 差异声明为 GLM 对齐提供了系统性参考。

尚存的质量问题主要集中在 **跨类型转换（fromMat 6b）的零值演算遗漏**（问题 1）和 **设计层面未决的 API 语义决策**（问题 2），这两项若在编码阶段暴露将导致返工。建议在设计阶段优先解决。
