## 问题清单

### 问题 1（严重）：`[]` 赋值版本签名存在双重语法违规——参数名和 `mut` 修饰符均缺失

**问题描述**：§3.4 中 `[]` 赋值版本签名写为 `operator func [](i: Int64, v: VecR<T,Q>): Unit`，违反仓颉语法规范的两项强制要求：

1. **参数名违规**：仓颉函数文档 §8.3 明确规定赋值形式 `operator func [](args..., value!: T): Unit` 的最后一个参数必须为命名参数 `value!`（无默认值），当前签名使用了位置参数 `v`，编译器将报错。
2. **`mut` 修饰符缺失**：`[]` 赋值版本需要修改 struct 内部数据成员，根据仓颉 struct 的 `mut` 规则（struct 文档 §3.5："mut 也可修饰运算符函数"），赋值运算符必须声明为 `mut`。阶段一 Vec 类型的实际实现证实了此规范（如 `type_vec4.cj:47`：`public mut operator func [](i: Int64, value!: T): Unit`）。

此为**回归问题**——v1 迭代第 6 轮问题 4（iteration_history.md 第 51-55 行）已将 `componentAt` 的赋值版本签名修正为含 `value!` 和 `mut` 的形式；v2 迭代第 22 轮也修正了 `col()` 赋值版本的值语义矛盾。当前版本中 `[]` 赋值签名仍缺少 `mut` 和 `value!`，说明前序修正未传导至此签名。

若编码者按当前签名直接编码，将产生编译错误。

**所在位置**：§3.4 行列访问（第 474 行）

**严重程度**：严重

**改进建议**：将签名修正为 `public mut operator func [](i: Int64, value!: VecR<T,Q>): Unit`，与仓颉语法规范和阶段一 Vec 实现风格一致。同时全文检查所有出现 `[]` 赋值签名之处，确保统一修正。

### 问题 2（中等）：复合赋值运算符自动生成被标记为"待验证假设"，但仓颉语言规范已将其声明为确定性保证

**问题描述**：§3.5 和 §8 将复合赋值运算符自动生成标注为需要编码首日验证的假设（"若自动生成验证失败，启用 §3.5 中的手动定义回退方案"），并为此准备了 54+6 个手动定义的回退方案表。但仓颉函数文档 §8.5 明确声明："重载二元运算符（关系运算除外）时**自动**启用对应复合赋值版本（`+=`、`-=` 等），**前提**是返回类型与左操作数类型匹配或为其子类型。"此为已文档化的语言规范特性，不是待验证假设。

当前设计为此"假设"准备的回退方案表占据了显著的文档篇幅（§3.5 手动定义回退方案文件分布表 + stub 行为说明段落），且因其"待验证"标注，编码者可能实际编写全部手动定义代码，造成不必要的维护负担。

非方阵 `*=(Mat)` 因返回类型与左操作数类型不匹配而不可自动生成，这是正确且需保留的说明。

**所在位置**：§3.5 复合赋值运算符段落（手动定义回退方案文件分布表及其前后说明）；§8 编码启动前验证项

**严重程度**：中等

**改进建议**：(1) 明确声明复合赋值自动生成为仓颉语言规范保证的行为，而非待验证假设；(2) 将手动定义回退方案从设计主体中移除，移至独立附录并标注"仅用于防御编译器版本回归 bug，非正常编码路径"；(3) 从 §8 编码启动前验证项中删除复合赋值验证项，或将该项降级为可选的防御性测试（非阻断性验证）。

### 问题 4（中等）：`lib.cj` 新增导出清单包含 `mod` 函数，但矩阵不提供 `mod` 且向量 `mod` 已有独立约束体系

**问题描述**：§8 lib.cj 新增 public import 清单第 3 条建议导出 `import glm.detail.{ add, sub, mul, div }`（矩阵标量运算），但当前 `lib.cj`（第 5 行）已导出 `glm.detail.{ add, sub, mul, div, mod }`。阶段一 `scalar_vec_ops.cj` 中的 `mod` 函数使用了 `Integer<T>` 约束（整数版 4 个重载）+ 具体 Float32/Float64/Float16 类型重载（浮点版 12 个重载），与 `add/sub/mul/div` 的统一 `Number<T>` 约束模式不同。设计文档未说明 `mod` 函数的矩阵版本是否应新增（§2 明确标注"矩阵取模无数学意义，GLM 也未为矩阵类型定义 % 运算符"），也未说明 `lib.cj` 现有 `mod` 导出与新增矩阵标量运算导出的交互——向量 `mod` 已在 `lib.cj` 中导出，矩阵标量 `mod` 不新增，两者在 `lib.cj` 层面不应产生新增改动，但设计文档未显式确认此理解。

**所在位置**：§2 scalar_mat_ops.cj 条目（第 74-76 行）；§8 lib.cj 新增导出清单

**严重程度**：中等

**改进建议**：在 §8 lib.cj 导出清单中显式说明：(1) `mod` 不在 `scalar_mat_ops.cj` 中提供（与 §2 已声明一致）；(2) `lib.cj` 当前 `mod` 导出仅覆盖向量版 `scalar_vec_ops.cj` 的 `mod` 重载，阶段二无新增矩阵 `mod` 重载需要导出，因此 `lib.cj` 的 `mod` 导出行无需修改。

### 问题 5（中等）：`common.cj` stub 中 `abs` 和 `sign` 的泛型约束选用 `Equatable<T>` 而非 `Comparable<T>`，不满足实现所需接口

**问题描述**：§3.7 common.cj 签名清单中，`abs` 和 `sign` 的约束为 `where T <: Number<T> & Equatable<T>`。但仓颉标准库中 `Number<T>` 接口定义（`public interface Number<T>`）不继承 `Comparable<T>` 或 `Equatable<T>`——`Number<T>` 仅提供 `+`、`-`、`*`、`/` 和一元取负运算符，不提供比较运算符。

- `abs` 的典型实现 `a < T(0) ? -a : a` 需要 `<` 比较运算符，属于 `Comparable<T>` 接口。仅约束 `Equatable<T>`（提供 `==`）**无法满足 `<` 运算的接口需求**。
- `sign` 的实现涉及 `a == T(0)` 和 `a > T(0)` / `a < T(0)`，需要 `==` 和 `<`/`>` 运算，即同时需要 `Equatable<T>` 和比较运算。

而仓颉标准库中 `Comparable<T>` 的定义为 `public interface Comparable<T> <: Equatable<T> & Less<T> & Greater<T> & LessOrEqual<T> & GreaterOrEqual<T>`，即 `Comparable<T>` 已继承 `Equatable<T>`。因此，将约束改为 `where T <: Number<T> & Comparable<T>` 可同时满足 `abs` 和 `sign` 的全部运算需求，且不引入冗余约束。

**所在位置**：§3.7 common.cj — `abs` 和 `sign` 签名（第 585-586 行）

**严重程度**：中等

**改进建议**：将 `abs` 和 `sign` 的泛型约束从 `where T <: Number<T> & Equatable<T>` 修正为 `where T <: Number<T> & Comparable<T>`。同步检查其余函数（`min`、`max`、`clamp`、`step`、`smoothstep`）的约束是否也需要类似修正——它们同样涉及 `<`/`>`/`<=`/`>=` 比较，当前标注为 `Comparable<T>` 是正确的。

### 问题 6（中等）：矩阵-矩阵除法 `Mat / Mat` 的运算符签名完全缺失——仅有 stub 函数库的 `inverse` 间接描述

**问题描述**：设计文档中 `inverse` 和 `determinant` 作为 `matrix.cj` 的 stub 函数列出，但矩阵-矩阵除法 `Mat2x2 / Mat2x2`、`Mat3x3 / Mat3x3`、`Mat4x4 / Mat4x4`（仅方阵）的**运算符签名**从未在任何章节给出。§3.5 算术运算符签名表仅覆盖 `+`、`-`、`*`，未列出 `/` 的同尺寸矩阵除法签名。§5 和 §9 仅用文字描述"mat/mat 依赖 inverse stub"。但编码者需要知道：矩阵除法运算符是否定义为方阵类型的成员运算符？如果是，其签名模式、泛型约束、与 `inverse` stub 的调用关系是什么？如果不是成员运算符而是包级函数，定义在哪个文件？

阶段一 Vec 类型中 `vec / vec` 是成员运算符，矩阵的方阵除法若也设计为成员运算符，则应在 §3.5 中列出签名，且须标注 `@OverflowWrapping` 和 stub 异常行为。

**所在位置**：§3.5 算术运算符（缺少矩阵-矩阵除法签名）；§10 API 覆盖矩阵（未列出 Mat/Mat 除法运算符）

**严重程度**：中等

**改进建议**：在 §3.5 中补充方阵矩阵-矩阵除法 `operator func /(rhs: Mat{N}x{N}<T,Q>): Mat{N}x{N}<T,Q>` 的 3 个签名（Mat2x2/Mat3x3/Mat4x4），标注内部实现为 `this * inverse(rhs)`（依赖 inverse stub，本阶段抛出异常），泛型约束和 `@OverflowWrapping` 标注与乘法运算符一致。同步更新 §10 API 覆盖矩阵。

### 问题 7（一般）：`diagonal(scalar)` 与 `identity(one)` 在方阵上行为重叠，设计决策未记录两者关系

**问题描述**：§3.3 item 3 定义 `diagonal(scalar: T)` 为对角矩阵工厂函数，§3.3 item 8 定义 `identity(one: T)` 为单位矩阵工厂函数。当 `scalar = one` 时，方阵上 `diagonal(one)` 与 `identity(one)` 产生完全相同的结果。两个 API 共存增加了用户选择负担——用户需要理解何时使用 `diagonal`、何时使用 `identity`。v2 迭代第 2 轮问题 4（iteration_history.md 第 53-57 行）曾提出"非方阵 identity(one) 内部委托 filled(one)"的建议并被采纳为"非方阵 identity(one) 内部委托 filled(one)，方阵保持独立实现"，但当前 v34 文档中此决策未被显式记录在 §7 决策表中。编码者可能将两者实现为互相调用或完全独立实现，使用者也不知在方阵场景下应优先选用哪个。

**所在位置**：§3.3 items 3, 8；§7 设计决策表

**严重程度**：一般

**改进建议**：(1) 在 §7 决策表中补充一条决策，明确方阵 `identity(one)` 内部委托 `diagonal(one)` 实现（延续 v2 迭代已采纳方案），非方阵仅提供 `identity(one)`；(2) 在 §4 行为契约的 `identity` 示例旁注明"方阵 `identity(1.0f)` 等价于 `diagonal(1.0f)`"；(3) 在 API 注释中推荐：需单位矩阵时优先使用 `identity`，需一般对角矩阵时使用 `diagonal`。

### 问题 8（一般）：`outerProduct` 实现说明中公式 `c * r[j]` 的语义表达存在歧义

**问题描述**：§3.7 matrix.cj 中 outerProduct 的实现说明写道"外积矩阵的第 j 列 = `c * r[j]`"——此处 `c` 表示列向量模板，逐分量乘以标量 `r[j]`。但 `c` 的分量数等于行数 R（而非列数 C），`c` 语义为"每列的模板"而非"列向量"，命名对不熟悉 GLM 的编码者可能产生"c 的分量数=列数"的直觉误解。GLM 源码中使用相同命名（`c`/`r`），设计文档此前迭代已补充参数语义说明，但公式表述的歧义仍不能完全消除。

**所在位置**：§3.7 matrix.cj — outerProduct 实现说明（"外积矩阵的第 j 列 = `c * r[j]`"）

**严重程度**：一般

**改进建议**：将公式改写为更精确的元素公式：`M[j][i] = c[i] * r[j]`（i 为行索引，j 为列索引），或 `result.column(j) = colVec * r[j] where colVec[i] = c[i] * r[j]`，消除"列向量"与"分量数=行数"之间的命名歧义。

### 问题 9（提示）：精度前缀方阵短别名的大小写风格可能产生可读性歧义

**问题描述**：§8 fwd.cj 别名清单中方阵短别名 `Highpmat2`、`Mediumpmat2`、`Lowpmat2` 中 `p` 与 `mat` 拼接处可能被读作"p-mat"而非"p 加 Mat"。此命名风格虽不会导致编译错误，但影响 API 可读性。此为 API 命名风格偏好问题。

**所在位置**：§8 fwd.cj 矩阵类型别名新增清单（方阵短别名行）

**严重程度**：提示

**改进建议**：若确认保留当前命名，应在别名定义处添加注释说明命名规则（精度前缀 + `mat` + 方阵尺寸），消除用户误解。

## 整体质量评价

本设计文档经过 34 轮迭代修订，在技术可行性、GLM 对照验证、签名完整性等内部审议维度上已达到较高水准。核心架构决策（9 个独立结构体、列主序存储、T(0)/T(1) 获取策略、fromMat 四项操作组合规则、Mat×Vec/Vec×Mat 运算符归属等）均已充分验证。

本次审查发现的问题主要集中在三个维度：

1. **与仓颉语言规范的精确对齐**（问题 1、2、5）：`[]` 赋值签名的双重语法违规和 `abs`/`sign` 约束选用错误是最可能影响编码效率的风险点——问题 1 将直接导致编译失败，问题 5 的 `Equatable<T>` 约束无法支撑 `<` 比较运算的实现。复合赋值自动生成的"待验证"标注则可能导致编码者执行不必要的回退方案。
2. **需求响应充分度**（问题 3、4、6）：路线图的可选项 `type_float.cj`/`type_half.cj` 未做取舍决策；矩阵-矩阵除法运算符签名在 §3.5 和 §10 中均缺失；`lib.cj` 的 `mod` 导出与矩阵标量运算导出的交互未显式说明。
3. **API 设计的内聚性与消费者体验**（问题 7、8）：`diagonal`/`identity` 关系缺失决策记录；`outerProduct` 公式语义仍存在歧义。

从"设计可直接指导编码实现"的标准来看，问题 1（编译必然失败）和问题 6（关键运算符签名缺失）是最需优先修正的两个问题——前者会阻断编码启动，后者会令编码者对方阵除法运算符的归属和签名无所适从。
