# 质量质询报告（v1）

## 质询结果

CHALLENGED

## 逐维度审查

### 1. 证据充分性

**[问题-严重]** 问题 2 关于 `dot` 的 Vec1 偏差声明存在事实错误。审查报告称"GLM 1.0.3 的 `geometric.hpp` 中 `dot` 仅声明了 Vec2~Vec4 的版本，不含 Vec1"，并声称"GLM 的模板约束为 `length_t L >= 2`"。但经核实 GLM 1.0.3 源码 `geometric.hpp:49`，`dot` 的声明为 `template<length_t L, typename T, qualifier Q>`，模板参数 `L` 无 `>= 2` 的约束——其文档注释（L44）明确声明 "L An integer between 1 and 4 included"。同时 `func_geometric.inl:28-35` 提供了 `compute_dot` 的 Vec1 特化实现，`func_geometric.inl:182-187` 的公开 `dot` 函数使用 `template<length_t L>` 且内部无 `GLM_STATIC_ASSERT` 限制 L 的最小值。因此 OOD 提供 Vec1 的 `dot` 完全对齐 GLM 1.0.3，**不存在偏差**，审查报告的核心论据"Vec1 dot 是 OOD 新增的非 GLM 对齐功能"不成立，由此衍生的"应在 §9 差异声明中新增条目"改进建议也不成立。

**[通过]** 问题 3 关于 `diagonal(scalar)` 对非方阵的适用性分析有充分的 GLM 源码证据支撑（`type_mat2x3.inl:33-42` 的 `mat(T scalar)` 构造确实为"对角线=scalar、其余=零"语义）。

**[通过]** 问题 4 关于 `Mat / Mat` 奇异矩阵行为未定义的分析有充分依据——OOD §5 确实未涉及此场景，且 `inverse` 的最终行为策略确实影响下游消费者。

**[通过]** 问题 9 关于编码模板示例使用下标访问而非属性访问的分析，经核实 OOD L419-420 确实使用了 `m.c0[0], m.c0[1]` 模式。

**[问题-严重]** 问题 6 关于 `cross` 的 Vec2/Vec4 遗漏声称存在部分事实错误。审查报告称"GLM 1.0.3 的 `geometric.hpp` 中 `cross` 还提供了 Vec2 版本（2D 叉积返回标量）和 Vec4 版本"。经核实 `geometric.hpp:58-59`，公开 API 仅声明 `cross(vec<3>, vec<3>)` 一个签名，无 Vec2 版本。Vec4 的 `cross` 仅出现在 `func_geometric.inl` 中 `compute_cross` 结构体的内部 `call` 方法（L85-94），但该 Vec4 版本**未通过公开 API `cross()` 导出**——`func_geometric.inl:190-194` 的 `cross()` 函数仅接受 Vec3 参数并转发到 `compute_cross`，而 `compute_cross` 内部的 Vec4 `call` 方法是冗余代码（有调用路径吗？需验证）。因此"GLM 提供 Vec2 版本的 cross"是事实错误的，"GLM 提供 Vec4 版本的 cross"的公开 API 层面也不准确。审查报告建议补齐 Vec2 cross 缺乏 GLM 对齐依据。

**[通过]** 问题 11 关于 `mod` 签名暂定状态未升级为编码阻断项的分析准确——§5 和 §8 确实未将此列为阻断性风险。

### 2. 逻辑完整性

**[问题-严重]** 问题 2 和问题 13 之间存在逻辑矛盾。问题 2 声称 Vec1 的 `dot` 是 OOD 新增的"非 GLM 对齐功能"，建议在 §9 差异声明中标注；问题 13 则从另一个角度质询 Vec1~Vec4 的 `dot` 是否与阶段一已有实现产生冲突。这两个问题对 Vec1 `dot` 的定性互相矛盾——问题 2 将其定性为"非 GLM 对齐的扩展"（应声明偏差），问题 13 将其定性为"可能重复定义"（应说明与阶段一关系），但两者均未首先确认 GLM 1.0.3 是否确实提供 Vec1 `dot`。事实上 GLM 提供了，因此问题 2 的核心论据不成立，问题 13 的质疑方向虽合理但应合并与问题 2 统一考虑。

**[通过]** 其余各问题之间逻辑自洽，改进建议与问题一致且可行。

### 3. 覆盖完备性

**[问题-严重]** 任务描述要求审查"需求响应充分度、整体深度和完整性"维度，特别是"设计是否可直接指导编码实现、接口定义是否足以支持下游消费者、异常场景和边界条件是否已考虑"。以下需求响应维度未被覆盖：

- **阶段二路线图的完整性覆盖验证**：用户需求引用了 `docs/02_roadmap.md` 作为实现路线图，审查报告未验证 OOD 产出是否完整覆盖了路线图中阶段二的全部范围（特别是路线图中列出但 OOD 中未涉及的文件/功能/模块），也未验证 `type_float.cj`/`type_half.cj` 的排除是否与路线图定义一致（仅依赖 OOD 自身 D35 决策，未交叉验证路线图原文）。

- **阶段一到阶段二的迁移连续性**：用户需求指出"已经实现了阶段一，项目在：cjglm"，审查报告未评估 OOD 中阶段一成果物的引用完整性和一致性（例如 Vec 类型的现有 API 是否与矩阵类型设计中的引用一致、阶段一未定义但阶段二需要的 Vec 功能是否存在缺口）。问题 13 触及了此维度但不够深入。

**[通过]** 事实错误/逻辑矛盾维度已通过问题 2-6、8-14 覆盖。深度和完整性维度通过问题 1、3-4、9-11 部分覆盖。

### 4. 报告必要性

**[问题-严重]** 问题 7 关于 `fromParts` "为何不需要 `Number<T>` 约束"的设计理由缺失，属于过度挖掘。`fromParts` 签名中未声明 `Number<T>` 约束是一个约束更少的正确选择（加法约束不会带来功能收益），这与任务描述要求的"需求响应充分度、落地可用性"等维度无关，编码者完全可以按签名实现而非必须理解"为什么不加约束"。此类问题分散了对真正重要问题的关注力度。

**[通过]** 其余问题均与需求响应或落地可用性相关，无无关紧要的细节问题。

## 质询要点

- **问题**：问题 2 关于 Vec1 `dot` 的核心论据存在事实错误——GLM 1.0.3 `geometric.hpp` 的 `dot` 声明模板参数 `L` 范围为 1-4，且 `func_geometric.inl` 提供了 Vec1~Vec4 的完整特化，因此 OOD 提供 Vec1 `dot` 不构成与 GLM 的偏差，"应在 §9 差异声明中新增条目"的建议不成立
  - **原因**：审查报告将 GLM 不提供 Vec1 `dot` 作为核心证据，但这些证据与 GLM 源码事实不符，导致审查结论错误，误导产出作者在 §9 增加不必要的偏差声明
  - **建议方向**：撤回问题 2 中的偏差声明建议；若仍认为 Vec1 `dot` 值得讨论，应重新定性为"OOD 的 Vec1 dot 完全对齐 GLM，无需偏差声明"，仅保留 Vec1 dot 与阶段一可能重复的质询方向（与问题 13 合并处理）

- **问题**：问题 6 声称 GLM 提供 Vec2 `cross`（2D 叉积），但 `geometric.hpp` 公开 API 仅声明 Vec3 版本，无 Vec2 版本；Vec4 `cross` 仅存在于内部 `compute_cross` 结构体，未通过公开 API 导出。审查报告建议"补齐 Vec2/Vec4 的 cross stub"缺乏 GLM 对齐依据
  - **原因**：基于错误事实给出的改进建议（补齐 Vec2 cross）会误导产出作者增加 GLM 中不存在的 API，与项目"对齐 GLM 实现迁移"的核心需求相悖
  - **建议方向**：修正问题 6 的事实基础——GLM 公开 API 仅提供 Vec3 `cross`；如果 Vec4 内部 `compute_cross` 无外部调用路径，则 OOD 仅保留 Vec3 版本是正确的对齐选择。若确需补充 Vec4 内部版本，应重新定性为"内部实现依赖"而非"GLM 标准 API"

- **问题**：问题 2 和问题 13 对 Vec1 `dot` 存在逻辑矛盾——前者认定其为"非 GLM 对齐扩展"，后者质疑其与阶段一实现冲突，二者基于互相排斥的前提且均未基于正确的 GLM 事实
  - **原因**：逻辑矛盾导致产出作者无法确定应执行哪个改进建议
  - **建议方向**：合并问题 2 和问题 13，以"GLM 提供 Vec1~Vec4 的 dot（事实确认）"为基准，统一评估：(1) OOD Vec1 dot 无偏差需声明；(2) 与阶段一已有 dot 实现的关系需确认

- **问题**：需求响应维度的覆盖不完备——未验证 OOD 与路线图的覆盖完整性，未评估阶段一到阶段二的迁移连续性（Vec 现有 API 与矩阵设计引用的一致性）
  - **原因**：任务描述明确要求审查"需求响应充分度"，路线图完整性和阶段间迁移连续性是该维度的关键子项，遗漏意味着审查未完成对任务描述要求的全面响应
  - **建议方向**：增加对 `docs/02_roadmap.md` 阶段二范围与 OOD 产出的交叉覆盖验证；增加对阶段一 Vec 已有 API 与矩阵设计中 Vec 引用的完整性评估

- **问题**：问题 7 过度挖掘约束设计理由，与任务描述核心审查维度无关
  - **原因**：此类问题增加审查报告的噪音比，分散对真正影响落地可用性的问题的注意力
  - **建议方向**：移除问题 7 或降低为信息性备注
