# 质量审查报告：a_v2_copy_from_v1.md（OOD 阶段二迁移 v35）

## 审查范围

以需求文档（requirement.md）为基准，从需求响应充分度、事实正确性、逻辑一致性、落地可用性四个维度审查 OOD 产出。审查排除内部审议已覆盖的技术可行性维度，侧重内部审议未充分覆盖的维度。

## 问题 1

**问题描述**：`diagonal(scalar: T)` 工厂函数仅对方阵提供，非方阵不提供。但 GLM 1.0.3 中 `type_mat2x3.inl:33-42` 的 `mat(T scalar)` 构造函数对所有矩阵类型（含非方阵）均提供"对角线=scalar、其余=零"语义，这是 GLM 标准 API 的一部分。OOD 的 D34 声称"非方阵因无 diagonal(scalar) 语义（非方阵对角矩阵构造不通用），仅提供 identity(one)"，但此理由不成立——"对角线元素=scalar、其余=零"对非方阵是确定的数学定义（对角线长度 `min(C,R)` 内填 scalar），且 GLM 对非方阵已经实现了此语义。当前设计导致非方阵用户无法通过 `diagonal(s)` 构造"对角线=s（s≠1）"的矩阵，API 存在对称性缺陷。

**所在位置**：§3.3 item 3 diagonal 构造函数说明（L231-236）；§7 D34（L753）；§9 差异声明 diagonal 条目（L868）

**严重程度**：中等

**改进建议**：对所有 9 个矩阵类型均提供 `diagonal(scalar)`，与 `identity(one)` 形成完整配对。若保留当前设计（仅方阵有 diagonal），应在 §9 差异声明中明确声明此为仓颉设计与 GLM `mat(T s)` 的行为偏差——GLM 对非方阵也提供标量对角构造。

## 问题 2

**问题描述**：矩阵-矩阵除法 `Mat / Mat` 在 `rhs` 不可逆（奇异矩阵）时的行为未定义。设计声称"内部实现为 `this * inverse(rhs)`"，但 `inverse` 的奇异矩阵行为策略完全缺失——§5 错误处理策略仅笼统写出"依赖 inverse stub，本阶段 stub 异常"，未对阶段三完整实现后的行为做决策预定义。GLM 中 `inverse` 对奇异矩阵返回未定义值（C++ 不检查），仓颉版是选择抛异常、返回 Option、还是返回未定义值？此决策直接影响 `Mat / Mat` 运算符的下游消费者错误处理策略，应在 OOD 阶段明确，而非推迟到编码阶段临场决定。

**所在位置**：§3.5 矩阵-矩阵除法签名（L515-521）；§5 错误处理策略（L713）

**严重程度**：中等

**改进建议**：在 §5 错误处理策略中新增"矩阵除法奇异矩阵行为"条目，明确策略选择：(1) `inverse` 在奇异矩阵上抛异常——则 `Mat / Mat` 传播异常，在 §5 显式声明；(2) `inverse` 返回未定义值（与 C++ GLM 一致）——则在 §9 差异声明中标注"矩阵除法对奇异矩阵行为未定义"。

## 问题 3

**问题描述**：`geometric.cj` 中 `cross` 仅有 Vec3 版本，但 GLM 的 `gtx/exterior_product.hpp` 中还提供了 Vec2 版本的 `cross`（2D 叉积返回标量）。虽然 Vec2 `cross` 属于 `gtx/` 实验性扩展而非 `core`，且阶段二 `geometric.cj` 仅需覆盖方阵 `.inl` 的依赖闭合，但设计文档未说明为何 Vec2 `cross` 不纳入 `geometric.cj` stub，也未在 §10 中追踪此 API。阶段六（gtx 实验性扩展）将需要 Vec2 `cross`，但当前 §10 追踪表中 `cross` 仅标注"Vec3"，可能导致阶段六实施者遗漏此 API。

**所在位置**：§3.7 geometric.cj 完整函数签名清单（L649）；§10 geometric 函数覆盖矩阵

**严重程度**：一般

**改进建议**：(1) 在 `geometric.cj` 签名清单附近注明"Vec2 `cross`（2D 叉积）属 gtx/exterior_product 范围，不在本阶段 core geometric stub 内"；2) 在 §10 中为 `cross` 追加一行备注"Vec2 `cross` 属 gtx/exterior_product，后续阶段六覆盖"。

## 问题 4

**问题描述**：`common.cj` 中 `mod` 函数签名标注为"暂定"——注释指出 `Number<T>` 不直接提供 `%` 运算符，最终实现需拆分为整数版本和浮点具体类型重载。但此"暂定"状态仅在 §3.7 签名清单注释和 §10 覆盖矩阵中体现，§8 编码启动前验证项未将其列为阻断性风险项。若编码者按当前签名 `func mod<T>(a: T, b: T): T where T <: Number<T>` 实现 stub，Float32/Float64 将在阶段四完整实现时无法实例化（编译错误），需要返工拆分签名。deviations.md DV-02 已记录 `%` 仅在 `Integer<T>` 约束下可用的事实，`Number<T>` 约束的 `mod` 对浮点类型无法编译是确定性的编译期错误而非可能的风险。

**所在位置**：§3.7 common.cj mod 签名（L578-579）；§8 编码启动前验证项（L771）

**严重程度**：中等

**改进建议**：将 `common.cj` 的 `mod` 签名暂定性从"注释"提升为"编码阻断项"——在 §8 编码启动前验证项中新增：验证 `mod<T>(a: T, b: T): T where T <: Number<T>` 对 Float32/Float64 是否可编译通过；若不能，应在编码首日修正签名（拆分为 `Integer<T>` 约束版 + Float32/Float64 具体类型重载），避免阶段四返工。引用 deviations.md DV-02 作为已验证的证据。

## 问题 5

**问题描述**：编码模板示例 1（Mat2x3←Mat2x2，B 类简洁方向）中使用了 `m.c0[0]`、`m.c0[1]` 等下标访问模式来获取列向量的分量。但阶段一 Vec 类型的 `[]` 取值版本包含 `assert + throw`，非 `const`。属性访问 `m.c0.x, m.c0.y` 无 assert 开销且可 const。编码模板示例的样式将直接影响 144 个 fromMat 重载的实现质量——若编码者照搬 `[0]`/`[1]` 模式，所有函数将带来不必要的 assert 开销。

**所在位置**：§3.3 编码模板示例 1（L406-423）

**严重程度**：一般

**改进建议**：将编码模板示例中的 `m.c0[0], m.c0[1]` 改为 `m.c0.x, m.c0.y` 等属性访问模式，并注明"优先使用列向量属性访问（`.x/.y/.z/.w`）而非下标访问（`[i]`），前者无 assert 开销且可 const"。

## 问题 6

**问题描述**：`scalar_mat_ops.cj` 新增的 36 个 `add/sub/mul/div` 全局函数与现有 `scalar_vec_ops.cj` 中同名的全局函数共存于 `glm.detail` 包。两者参数模式相同（`s: T, m/v: Mat/Vec<T,Q>`），重载解析按参数类型区分理论可行，但设计文档未在 §8 编码启动前验证项中显式要求验证同包同名函数重载解析无歧义。这是一个编译期风险——若仓颉编译器在泛型重载解析上有限制（如类型推断歧义），可能导致调用方 `add(1.0f, m22)` 编译失败。

**所在位置**：§8 编码启动前验证项（L771）

**严重程度**：一般

**改进建议**：在 §8 编码启动前验证项中增加一项：验证 `add(s: T, m: Mat2x2<T,Q>)` 的重载调用不与现有 `add(s: T, v: Vec2<T,Q>)` 产生歧义（因参数类型 `Mat2x2<T,Q>` vs `Vec2<T,Q>` 不同，理论上是安全的，但应编码验证）。

## 问题 7

**问题描述**：`geometric.cj` stub 中 `dot` 包含 Vec1 版本，且 `length`/`normalize`/`distance`/`reflect`/`refract` 仅提供 Vec2~Vec4 版本（不含 Vec1）。经核实 GLM 1.0.3 `geometric.hpp`，`dot` 声明模板参数 `L` 范围为 1-4，GLM 确实提供 Vec1 的 `dot`，因此 OOD 包含 Vec1 `dot` 完全对齐 GLM，无需偏差声明。但 `length`/`normalize`/`distance`/`reflect`/`refract` 在 GLM 中的模板参数 `L` 范围也为 1-4（GLSL 规范中这些函数对 `L=1` 也有效——长度为 1 的向量的 normalize 为 +1/-1，length 为绝对值等），OOD 中缺少 Vec1 版本与 GLM 存在覆盖差异。设计文档未说明省略 Vec1 `length`/`normalize` 等的理由，也未标注为有意的范围裁剪。

**所在位置**：§3.7 geometric.cj 完整函数签名清单（L650-664）；§10 geometric 函数覆盖矩阵

**严重程度**：一般

**改进建议**：(1) 查验 GLM `func_geometric.inl` 确认 `length`/`normalize`/`distance`/`reflect`/`refract` 是否对 Vec1 提供特化（若 GLM 在内部 `compute_length` 等结构体中支持 L=1，则 OOD 遗漏了 Vec1 版本）；(2) 若确认 GLM 提供 Vec1 版本，在 `geometric.cj` stub 中补齐；若确为有意省略，在 §9 差异声明或 §3.7 注释中说明省略理由。

## 问题 8

**问题描述**：`fromMat` 6a/6b 的 144 个签名中，`one: T` 参数在纯收缩方向下被忽略。设计文档已标注此行为并给出调用示例，但此 API 设计对下游消费者构成实际可用性隐患——调用方必须在纯收缩方向传入无意义的 dummy 值，但签名和调用形式与扩展方向完全一致，无法从调用语法上区分"忽略 one"与"需要 one"的方向。更关键的是，若调用方偶然在收缩方向传入了错误值（如 `one: 0.0f`），代码仍能编译运行，但语义上令人困惑。此为仓颉泛型限制导致的 API 统一性与语义精确性之间的权衡，应在 §9 差异声明中显式声明。

**所在位置**：§3.3 item 6 纯收缩方向 `one` 参数行为说明（L286 附近）；§9 差异声明

**严重程度**：一般

**改进建议**：在 §9 差异声明中新增条目"fromMat 6a/6b 纯收缩方向 one 参数被忽略"，声明此为仓颉泛型限制（无法按方向拆分签名族）导致的 API 设计偏差，标注调用方注意此语义。

## 问题 9

**问题描述**：§10 API 覆盖矩阵中 `fwd.hpp` 别名族统计使用了"4 精度 × 9 尺寸 = 36"的表述。经核实，GLM 的 `fwd.hpp` 确实有 4 个精度级别（lowp/mediump/highp/defaultp），"4 精度"是对 GLM 源码的准确描述。但仓颉版中 `Defaultp` 是 `PackedHighp` 的类型别名（见 `qualifier.cj:13`），不构成独立精度级别。因此"4 精度 × 9 尺寸 = 36"在 GLM 侧是准确的，但容易误导编码者认为仓颉版也需 4 个精度族。v33 修订说明 Q3 已将统计改为定性描述，但 §10 第 947-948 行（f32/f64 PascalCase 条目）仍保留了"4 精度 × 9 尺寸 = 36"原文。

**所在位置**：§10 fwd.hpp 别名覆盖矩阵（L947-948）

**严重程度**：提示

**改进建议**：将 §10 中"4 精度 × 9 尺寸 = 36"加注说明"GLM 含 defaultp 共 4 精度级别，仓颉版 defaultp = PackedHighp，实际 3 独立精度族"。

## 问题 10

**所在位置**：§8 产出物清单（L760-764）

**严重程度**：提示

**改进建议**：在 §8 产出物清单中新增"矩阵测试文件"条目，说明测试文件组织和命名约定（如沿用阶段一模式：`detail/type_mat{C}x{R}_test.cj` 各一），并引用路线图验证标准中的"矩阵创建、行列访问、基本算术运算"等验证要点。

## 问题 11

**问题描述**：阶段一到阶段二的迁移连续性存在一个潜在缺口：阶段一 `fwd.cj` 使用 `import glm.detail` 命名空间导入方式（见 `fwd.cj:5`），所有别名通过 `detail.MatNxN<...>` 限定访问。但 OOD §8 新增别名清单中未检查此导入模式是否会导致矩阵别名与现有向量别名产生名称冲突——例如 `fwd.cj` 中已存在 `public type Mat` 类别行吗？经检查当前 `fwd.cj`，阶段一仅含标量和向量别名，不存在矩阵相关别名名，新增矩阵别名不会冲突。但 OOD 未显式确认此点，也未说明新增矩阵别名的 `import` 声明是否需修改（当前 `fwd.cj` 的 `import glm.detail` 和 `import glm.detail.{ PackedHighp, PackedMediump, PackedLowp }` 已足够支撑矩阵别名定义）。

**所在位置**：§8 fwd.cj 矩阵类型别名新增清单（L798-843）

**严重程度**：提示

**改进建议**：在 §8 别名清单前新增一行说明：阶段一 `fwd.cj` 已包含 `import glm.detail` 命名空间导入，矩阵别名可直接通过 `detail.MatNxN<...>` 限定访问，无需额外 import 语句；同时确认矩阵别名命名（PascalCase/camelCase）与现有向量别名无命名冲突。

## 整体质量评价

该 OOD 文档经过多轮审议迭代（v27~v35），在核心设计维度上已达到较高质量：9×9 转换矩阵表与 GLM 源码逐项对齐、Mat×Vec/Vec×Mat 签名与矩阵乘法规则已修正、Bool 矩阵操作限制已在 D33 完整列举、复合赋值运算符规范保证已正确声明。

主要的待改进方向集中在：(1) API 设计上与 GLM 的语义对齐缺口（`diagonal` 对非方阵的排除、`geometric.cj` stub 中 Vec1 `length`/`normalize` 等的遗漏）；(2) 边界场景行为预定义缺失（Mat/Mat 除法奇异矩阵行为）；(3) 编码阻断性风险标注不足（`mod` 签名暂定性未升级为阻断项）；(4) 跨阶段迁移连续性确认不够充分（`fwd.cj` 别名无冲突确认、测试文件规划缺失）。
