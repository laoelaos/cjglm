## 需要解决的问题（仅 RETRY 时存在）

- **问题描述**：fromMat 6b T(0) 演算路径存在类型系统缺陷。6b 中 `m.c0.x` 类型为 U（源元素类型），`m.c0.x - m.c0.x` 计算 U(0) 而非 T(0)，且 6b 约束未声明 `U <: Number<U>`，直接沿用 6a 模板中的零值演算路径将引发编译错误或类型不匹配。
- **所在位置**：§3.3 编码模板说明（第467-472行）、§3.3 item 6b 模板示例区（第280-282行附近）
- **严重程度**：一般
- **改进建议**：将 6b 的 T(0) 演算路径从 `m.c0.x - m.c0.x` 修正为 `one - one`；为 6b 单独提供编码模板示例；若 U 需参与运算，增加 `U <: Number<U>` 约束。

- **问题描述**：div(s, m) 语义歧义仍未在设计层面锁定。`add`/`sub`/`mul` 语义一致但 `div(s, m)` 若实现为 `m / s` 则与参数顺序暗示的 `s / m` 语义相反，将不确定性推迟至编码阶段可能导致连锁修改。
- **所在位置**：§8 编码启动前验证项（第844行末尾）、§3.5 scalar_mat_ops.cj（第591-613行）
- **严重程度**：一般
- **改进建议**：在 §7 设计决策表中新增决策项锁定 `div(s, m)` 最终语义，选择 `s / m` 或 `m / s` 并给出展开公式，移除"待验证"标注。

- **问题描述**：§10 API 覆盖矩阵缺少 fromParts/fromColumns/fromMat 系列工厂函数追踪条目。fromParts（item 4）、fromColumns（item 5）、fromMat 6a/6b（item 6）、fromMat 7（item 7）共 5 个函数组完全未在覆盖矩阵中追踪。
- **所在位置**：§10 mat2x2~mat4x4.hpp 覆盖矩阵（第1004-1025行）
- **严重程度**：一般
- **改进建议**：在 §10 覆盖矩阵中新增上述 5 个函数组的行级追踪条目，标注实现阶段归属。

- **问题描述**：fromMat 6a/6b 的 `one` 参数 NaN 值行为未在设计层面定义。`one` 完全由调用方传入，若误传 NaN 则对角填充结果全为 NaN，identity(one) 也存在同样问题。
- **所在位置**：§3.3 item 6a/6b one 参数语义说明（第276-282行）、§3.3 item 8 identity(one) 行为定义（第293-318行）
- **严重程度**：一般
- **改进建议**：在 one 参数语义说明和 identity 行为定义中补充 NaN 值行为约束："调用方应确保 one 为非 NaN 值，否则行为未定义"；或定义为"当 one 为 NaN 时对角填充自动退化为全 NaN 填充"。
