## 问题清单

### 问题 1（中等）：`[]` 赋值版本签名违反仓颉语法规范

**问题描述**：§3.4 中 `[]` 赋值版本签名写为 `operator func [](i: Int64, v: VecR<T,Q>): Unit`，使用了位置参数 `v`。但仓颉语言规范明确要求 `[]` 赋值形式的最后一个参数必须是**命名参数** `value!`，无默认值，返回 `Unit`（见仓颉函数文档 §8.3："赋值形式：`operator func [](args..., value!: T): Unit` — 须有唯一命名参数 `value!`，无默认值，返回 `Unit`"）。阶段一 Vec 类型的实际实现也遵循此规范（如 `type_vec4.cj:47`：`public mut operator func [](i: Int64, value!: T): Unit`）。当前签名若按文档直接编码，编译器将报错。

**所在位置**：§3.4 行列访问（"下标运算符 `[]`…赋值版本 `operator func [](i: Int64, v: VecR<T,Q>): Unit`"）

**严重程度**：中等

**改进建议**：将签名修正为 `mut operator func [](i: Int64, value!: VecR<T,Q>): Unit`，与仓颉语法规范和阶段一 Vec 实现风格一致。同步检查全文所有出现 `[]` 赋值签名处，确保统一修正。

### 问题 2（中等）：复合赋值运算符自动生成机制被标记为"待验证"，但仓颉语言规范已明确声明其为标准行为

**问题描述**：§3.5 和 §8 多处将复合赋值运算符自动生成标注为需要编码首日验证的假设（"若自动生成验证失败，启用 §3.5 中的手动定义回退方案"），并为此准备了 54+6 个手动定义的回退方案表。但仓颉函数文档 §8.5 明确声明："重载二元运算符（关系运算除外）时**自动**启用对应复合赋值版本（`+=`、`-=` 等），**前提**是返回类型与左操作数类型匹配或为其子类型。"此为已文档化的语言特性，无需作为假设验证。当前设计为此假设准备的回退方案表占据了显著的文档篇幅（§3.5 复合赋值运算符手动定义回退方案文件分布表 + 下方的 stub 行为说明段落），增加了文档复杂度但对实际编码无确定性贡献——编码者可能因"待验证"标注而实际编写全部手动定义代码，造成不必要的维护负担。

**所在位置**：§3.5 复合赋值运算符段落（手动定义回退方案文件分布表及其前后说明）；§8 编码启动前验证项

**严重程度**：中等

**改进建议**：(1) 明确声明复合赋值自动生成为仓颉语言规范保证的行为，而非待验证假设；(2) 保留手动定义回退方案表但将其降级为"参考附录"或"极端防御方案"，注明仅在编译器版本存在回归 bug 时启用；(3) 从 §8 编码启动前验证项中删除或降级复合赋值验证项（仅保留为可选的防御性测试，而非阻断性验证）。非方阵 `*=(Mat)` 因返回类型与左操作数类型不匹配而不可自动生成，这是正确且需保留的说明。

### 问题 3（一般）：`scalar_mat_ops.cj` 与 `scalar_vec_ops.cj` 同包同名函数重载——下游消费者使用 `import glm.detail.*` 时是否会产生歧义问题未充分评估

**问题描述**：§2 包组织中将 `scalar_mat_ops.cj` 置于 `glm.detail` 包，其 `add/sub/mul/div` 函数名与阶段一 `scalar_vec_ops.cj` 完全相同。设计声称"通过参数类型重载区分"，但这仅对**同包内**调用成立。§8 中 `lib.cj` 通过 `public import glm.detail.{ add, sub, mul, div, mod }` 导出，下游消费者 `import glm.*` 时获得的 `add/sub/mul/div` 将同时包含向量版和矩阵版（共 4×9+4×4=52 个重载）。关键问题在于：当外部包使用 `import glm.*` 导入后，若本地代码也定义了同名函数 `add(s: T, v: Vec2<T,Q>)` 等，根据仓颉遮蔽规则（包文档 §4.3），本地同签名函数将遮蔽导入版本，但参数类型不同时构成重载。设计未评估以下场景：下游项目同时 `import glm.*` 和自定义 `add` 函数时的编译器行为。此风险虽低，但作为库设计应提供消费指引。

**所在位置**：§2 包组织（scalar_mat_ops.cj 条目）；§8 lib.cj 新增 public import 清单

**严重程度**：一般

**改进建议**：在 §8 lib.cj 导出说明中补充下游消费指引：当下游项目同时 import glm 和本地自定义 add/sub/mul/div 函数时，需注意仓颉重载解析规则——参数类型不同构成重载、同签名本地函数遮蔽导入版。若产生歧义，可使用限定导入 `import glm.detail.{ add as matAdd }` 等方式消歧。

### 问题 4（一般）：`diagonal(scalar: T)` 工厂函数命名与 `identity(one: T)` 的关系在设计决策中未记录

**问题描述**：§3.3 item 3 定义 `diagonal(scalar: T)` 为对角矩阵工厂函数，`identity(one: T)` 为单位矩阵工厂函数。当 `scalar = one` 时，方阵上 `diagonal(one)` 与 `identity(one)` 产生完全相同的结果。两个 API 共存增加了用户选择负担——用户需要理解何时使用 `diagonal`、何时使用 `identity`。设计文档中仅有 v26 更名历史（`filled` → `diagonal`）和 D31（非方阵 identity 有意设计），但未记录"两者在方阵上功能重叠是否有意保留"的显式决策。编码者可能误将两者实现为互相调用或独立实现，使用者在方阵场景下也不知该优先选用哪个。

**所在位置**：§3.3 items 3, 8；§7 设计决策表

**严重程度**：一般

**改进建议**：(1) 在 §7 决策表中补充一条决策：方阵 `identity(one)` 内部委托 `diagonal(one)` 实现（或反之），非方阵仅提供 `identity(one)`（等价于 `diagonal(one)` 语义扩展）；(2) 在 §4 行为契约的 `identity` 示例旁注明"方阵 `identity(1.0f)` 等价于 `diagonal(1.0f)`"；(3) 考虑在 API 注释中推荐：需单位矩阵时优先使用 `identity`，需一般对角矩阵时使用 `diagonal`。

### 问题 5（一般）：fwd.cj 矩阵别名的精度前缀方阵短别名大小写不一致

**问题描述**：§8 fwd.cj 矩阵类型别名新增清单中，方阵短别名的大小写风格与 GLM 原始约定不一致。清单中列出 `Highpmat2`（P 小写开头）、`Mediumpmat2`、`Lowpmat2`，但 GLM 1.0.3 `fwd.hpp` 中对应别名的高精度版本为 `highp_mat2`（snake_case）。设计选择 PascalCase/camelCase 混合风格，但 `Highpmat2` 中 `Highp` 为 PascalCase 首部而 `mat2` 为 camelCase 首部，两者拼接处 `pmat` 可能被读作"p-mat"而非"p 加 Mat"——即用户可能误解 `Highpmat2` 为 `Highp + mat2` 而非 `Highp + Mat2`。此命名歧义虽不会导致编译错误，但影响 API 可读性。

**所在位置**：§8 fwd.cj 矩阵类型别名新增清单（HighpMat 家族、MediumpMat 家族、LowpMat 家族的方阵短别名）

**严重程度**：一般

**改进建议**：考虑在精度前缀与方阵短别名之间增加下划线分隔（如 `Highp_mat2`）或使用一致性更高的驼峰形式（如 `HighpMat2`，与 PascalCase 全尺寸别名 `HighpMat2x2` 保持前缀一致）。当前方案若确认保留，应在别名定义处添加注释说明命名规则，避免用户误解。

### 问题 6（一般）：ext/ 别名文件的向量别名文件数量与 §2 包组织图不一致

**问题描述**：§2 包组织图中 ext 下列出 `vector_float1.cj ~ vector_float4.cj ★` 和 `vector_double1.cj ~ vector_double4.cj ★`，共 8 个向量别名文件。§3.8 和 §8 产出物清单均称"26 个 ext/ 别名文件（18 矩阵 + 8 向量）"——此处 18+8=26。但阶段一的 cjglm 项目当前无 ext/ 目录和 ext/ 别名文件（已在 `cjglm/src/` 下验证），也未在 fwd.cj 中发现 ext/ 向量别名的定义。设计文档未明确说明"8 个向量别名文件"是属于阶段二新增还是阶段一遗留——若是新增，则路线图 (`docs/02_roadmap.md` §阶段2范围) 列出"ext/ 下的向量/矩阵别名文件"与此一致；若是阶段一遗留，则阶段一的实际代码中并不存在。此模糊性可能导致编码者对工作量估算产生偏差。

**所在位置**：§2 包组织图（ext/ 目录下的向量别名文件）；§8 产出物清单

**严重程度**：一般

**改进建议**：明确标注 ext/ 下 8 个向量别名文件为阶段二新增创建（与路线图一致），并说明向量别名文件的内容仅为 Vec1~Vec4 的 Float32/Float64 具现化 type alias（参照 GLM `ext/vector_float.hpp` 等头文件），消除"是否属于阶段一遗留"的歧义。

### 问题 7（一般）：`outerProduct` 参数 `c` 与 `r` 的语义描述与签名表列的方向标注存在不直观之处

**问题描述**：§3.7 matrix.cj 中 outerProduct 的实现说明写道"外积矩阵的第 j 列 = `c * r[j]`"——此处 `c` 逐分量乘以标量 `r[j]`。但此公式隐含 `c` 为列向量、`r` 为行向量，结果矩阵尺寸为 len(c)×len(r)。设计中签名表列出 `outerProduct(c: Vec2, r: Vec3): Mat3x2`，即 c 分量数=2=R（行数），r 分量数=3=C（列数），结果为 Mat3x2（3列×2行）。但 `c * r[j]` 公式中的 `c` 语义为"列向量"（用于构建矩阵的每一列），而 c 的分量数为 R（行数），此命名对 GLM 习惯用户虽合理（GLM 源码中也使用 `col`/`row` 或 `c`/`r` 命名），但对不熟悉 GLM 的编码者可能产生"c 的分量数=列数"的直觉误解。设计文档此前的迭代已补充参数语义说明，但公式 `c * r[j]` 中 `c` 语义为"每列的模板"而非"列向量"，此表述仍存在歧义。

**所在位置**：§3.7 matrix.cj — outerProduct 实现说明（"外积矩阵的第 j 列 = `c * r[j]`"）

**严重程度**：一般

**改进建议**：将公式改写为更精确的表述：`result.column(j) = colVec * r[j]`，其中 `colVec[i] = c[i] * r[j]`（标量乘），或使用矩阵元素公式 `M[j][i] = c[i] * r[j]`，消除"列向量"与"分量数=行数"之间的命名歧义。

### 问题 8（一般）：`common.cj` stub 中 `abs` 和 `sign` 的泛型约束包含 `Equatable<T>` 但实际运算不需要等值比较

**问题描述**：§3.7 common.cj 完整函数签名清单中，`abs` 和 `sign` 的约束为 `where T <: Number<T> & Equatable<T>`。阶段一 `shim_limits.cj` 中 `abs<T>(a: T): T where T <: Number<T>` 不要求 `Equatable<T>`。`abs` 的实现仅涉及 `a < T(0) ? -a : a` 形式的比较，需 `Comparable<T>` 而非 `Equatable<T>`。`sign` 的实现涉及 `a == T(0)` 判断才需 `Equatable<T>`，但也可通过 `a < T(0)` 和 `a > T(0)` 的 `Comparable<T>` 约束实现。当前约束标注为 `Equatable<T>` 而非 `Comparable<T>`，若编码者按此约束实现 `abs`，会多引入不必要的 `Equatable` 约束；若实现 `sign` 仅用 `Comparable` 则 `Equatable` 也可省略。

**所在位置**：§3.7 common.cj — `abs` 和 `sign` 签名

**严重程度**：一般

**改进建议**：核对 GLM 1.0.3 `common.hpp` 中 `abs`/`sign` 的模板约束，评估是否应改为 `where T <: Number<T> & Comparable<T>`（而非 `Equatable<T>`）。若 GLM 原始实现使用 `std::enable_if` 等机制对 `abs` 和 `sign` 分别约束，应在设计中区分标注。

## 整体质量评价

本设计文档经过 34 轮迭代修订，在技术可行性、GLM 对照验证、签名完整性等内部审议维度上已达到较高水准。核心质量问题（如 Mat×Vec 签名方向修正、矩阵乘法推导规则修正、fromMat 转换矩阵表修正等）均已在前序迭代中解决。本次审查发现的问题主要集中在：

1. **与仓颉语言规范的精确对齐**（问题 1、2）：`[]` 赋值签名的语法规范和复合赋值自动生成的确定性声明，属于编码落地时容易产生编译错误或返工的风险点。
2. **API 设计的消费者体验**（问题 3、4、5）：同名函数重载的消费指引、重叠 API 的关系说明、命名一致性，属于影响库使用者体验的维度。
3. **文档内部一致性**（问题 6、8、9）：属于编码者可能产生理解偏差的较小风险。

从"设计可直接指导编码实现"的标准来看，当前设计已具备充分的落地指导性——9×9 转换矩阵表、编码模板示例、完整签名清单等均为编码者提供了足够的信息。问题 1 和问题 2 是最可能影响编码效率的两个风险点，建议优先修正。
