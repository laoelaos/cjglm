## 当前审查结果

### 问题 1（严重）：`identity()` 静态工厂函数中 T(1) 演算公式依赖 `!` 运算符，但 `Number<T>` 不提供该运算符

§3.3 第 8 条和 D20 决策提出在 `Number<T>` 约束的 extend 块中通过 `let one = -(!(zero))` 演算 T(1)，但 `!`（按位取反）运算符属于 `Integer<T>` 接口而非 `Number<T>` 接口。在 `Number<T>` 约束上下文中调用 `!zero` 会导致编译错误，特别是对 Float32/Float64 类型不可用。此外，与阶段一 `increment()` 的类比前提错误——`increment()` 定义在 `Integer<T>` 约束的 extend 块中，而非 `Number<T>`。

**改进建议**：`identity()` 需补充 T(1) 参数（如 `identity(one: T)`），或为 Float32/Float64 等具体类型写特化版本。需在设计中明确决策并更新 D20。

### 问题 2（一般）：构造函数第 1 条 `init(scalar: T)` 标量填充构造的 T(0) 获取方式未给出具体实现策略

§3.3 第 1 条标注需要利用 `scalar - scalar` 得到 T(0)，但 init 构造函数必须在 struct 体内定义（不能在 extend 块中定义），struct 体内无约束泛型 `T` 不支持 `-` 运算符。设计中"编码时须确保此构造函数仅在带约束的 extend 块中可用"与此矛盾。

**改进建议**：增加 `zero: T` 参数（如 `init(scalar: T, zero: T)`），需考虑与问题 3 的统一解决策略。注意阶段一 `Vec2.init(scalar: T)` 的语义是全分量赋同一标量值（无需 T(0)），矩阵的 `init(scalar: T)` 语义是对角矩阵（需 T(0) 填充非对角线），两者不可类比。

### 问题 3（一般）：跨矩阵类型转换构造（第 6 条）72 个构造函数需 T(0)/T(1) 但未分析如何在 struct 体中获得

72 个构造函数作为 init 必须定义在 struct 体内，而 struct 体内的 `T` 是无约束泛型，无法执行 T(0)/T(1) 构造。设计中未给出在无约束泛型上下文中获得 T(0) 和 T(1) 的实现策略。

**改进建议**：将 72 个跨尺寸转换构造改为 extend 块中的命名工厂函数（如 `fromMat2x2(m, zero, one)`），zero 通过 `someValue - someValue` 在 `Number<T>` 上下文中获得，one 通过参数传入。需考虑与问题 2 的统一解决策略——质询报告指出两者根因相同但改进策略不一致，应考虑统一的解决方向。

### 问题 4（轻微）：fwd.cj 精度变体别名的 DMat Float64 族精度变体命名是否覆盖方阵短别名未明确

**改进建议**：在命名规则中补充精度变体是否覆盖方阵短别名（如 `HighpMat4` 对应 `Mat4x4<Float32, PackedHighp>` 是否也有 Highp 前缀变体）。

### 问题 5（一般）：`[]` 运算符赋值版本签名缺少 `mut` 修饰符

§3.4 中赋值版本签名写为 `[](i: Int64, value!: VecR<T,Q>): Unit`，但阶段一 Vec2 实际实现为 `mut operator func [](i: Int64, value!: T): Unit`。矩阵版本同样需要 `mut`。

**改进建议**：将 `[]` 运算符赋值版本签名补全为 `mut operator func [](i: Int64, value!: VecR<T,Q>): Unit`。

### 问题 6（一般）：跨类型逐分量构造（第 4 条）和跨类型列向量构造（第 5 条）在 struct 体内的实现可行性未分析

两个构造函数涉及泛型类型参数 `U`/`V1, V2...` 到 `T` 的转换，但仓颉泛型上下文下隐式类型转换不可用。设计未说明类型约束条件，也未参考阶段一 `castVecN(v, conv)` 模式。

**改进建议**：分析跨类型构造在仓颉中的可行性约束，若不可行则降级为 `fromParts(v1, v2, ..., conv)` 工厂函数模式。

### 问题 7（轻微）：`col()` 与 `[]` 并存但未说明编码级使用场景差异指导

**改进建议**：补充简要编码指导：① 两者应在同一 extend 块中定义；② 推荐编码规范（如内部实现用 `[]`、对外 API 用 `col()`）。

### 问题 8（一般）：lib.cj 中导出 add/sub/mul/div 时下游消费者同时 `import glm.*` 和 `import glm.detail.*` 是否产生歧义未评估

**改进建议**：补充说明下游消费者使用不同导入路径时重载解析不冲突的依据，或确认会产生歧义并提供规避方案。

### 问题 9（一般）：`const init` 条件性描述未为各构造函数分类标注是否可 const init

**改进建议**：为每个构造函数标注 const init 可用性：① 逐分量同类型构造 → const init 可用（纯赋值）；② 跨类型逐分量构造 → 非 const（涉及类型转换）；③ 列向量构造 → 取决于 VecR 是否有 const init；④ 标量填充和跨矩阵转换 → 非 const（涉及运算）。

### 问题 10（轻微）：跨尺寸矩阵乘法运算符的 import 策略描述冗余/误导

在 glm.detail 同包中 9 个矩阵类型互相直接可见，无需跨文件 import。

**改进建议**：明确说明同包内无需 import，移除"在对应文件顶部 import 所需右操作数矩阵类型"的误导性描述。

### 问题 11（轻微）：测试文件中缺少别名类型的构造函数可用性验证

**改进建议**：补充别名类型上的构造函数和运算符可用性验证（如 `mat4(scalar)`、`mat4.identity()`、`mat4 + mat4` 等）。

### 问题 12（一般）：`identity()` 未分析 `const func` 可行性，也未在 §9 差异声明中记录与 C++ GLM constexpr 行为差异

**改进建议**：在 T(1) 获取方案确定后补充 `identity()` 的 const func 可行性分析；在 §9 差异声明中补充：若 `identity()` 不可为 const func，则 `const val m = Mat4x4<Float32, PackedHighp>.identity()` 不可用。
