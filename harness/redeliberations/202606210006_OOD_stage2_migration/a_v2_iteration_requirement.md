## 当前审查结果

### P1（严重）：行向量×矩阵乘法签名错误
**位置**: §3.5「二元运算符（矩阵-向量）」
当前设计方案中签名写为 `Vec{C}<T,Q> * Mat{C}x{R}<T,Q> → Vec{R}<T,Q>`，与 GLM 参考实现不符。正确签名应为 `Vec{R}<T,Q> * Mat{C}x{R}<T,Q> → Vec{C}<T,Q>`。GLM 证据：`type_mat2x3.inl:410-419` 定义 `operator*(col_type const& v, mat<2,3,T,Q> const& m)`，其中 `col_type=vec3`（R=3），返回 `row_type=vec2`（C=2），即 `Vec3 * Mat2x3 → Vec2`，对应 `Vec{R} * Mat{C}x{R} → Vec{C}`。
**严重程度**: 严重
**改进建议**: 修正为 `Vec{R}<T,Q> * Mat{C}x{R}<T,Q> → Vec{C}<T,Q>`。同时在 §2 矩阵类型概览表中补充示例说明以帮助编码阶段快速核验（如 Mat2x3: `Vec3 * Mat2x3 → Vec2`）。

### P2（中等）：跨尺寸矩阵乘法运算符未完整定义
**位置**: §3.5「二元运算符（矩阵-矩阵，同尺寸）」— 矩阵乘法子节
设计方案仅列举了 `Mat2x3 * Mat2x2 → Mat2x3` 一个跨类型示例，未给出完整的跨尺寸矩阵乘法组合清单或系统性的匹配规则。GLM 参考实现在各 `type_matNxM.inl` 中定义了丰富的跨尺寸乘法（如 `Mat4x4 * Mat2x4 → Mat2x4`、`Mat2x3 * Mat3x2 → Mat3x3` 等）。
**严重程度**: 中等
**改进建议**: 补充跨尺寸矩阵乘法完整的组合规则说明。建议按"左矩阵列数 = 右矩阵行数则形态兼容，结果尺寸为 (左矩阵行数 × 右矩阵列数)"的规则对所有 9 个矩阵类型列出其乘法伙伴集。

### P3（中等）：跨矩阵类型转换构造的填充规则不精确
**位置**: §3.3「构造函数体系」第 8 条（跨矩阵类型转换构造）
当前描述为"额外值填默认值（如 0 或 1）"，过于模糊。对照 GLM 1.0.3 参考实现（`type_mat4x4.inl:175-285`），填充规则为单位矩阵扩展语义——缺失列/行补齐为单位矩阵的相应列/行，而非简单的"所有额外位置填 0，对角填 1"。
**严重程度**: 中等
**改进建议**: 补充明确的填充规则描述："当目标矩阵尺寸大于源矩阵时，源矩阵的元素复制到目标矩阵的对应位置；目标矩阵中源矩阵未覆盖的行列从单位矩阵的对应行/列取值（即缺失元素中，`(i,i)` 填 `T(1)`，其余填 `T(0)`）"。

### P4（中等）：ext/ 别名文件数量描述前后矛盾
**位置**: §2 模块清单（列 18 个文件）vs §3.8（称"16 个矩阵别名文件"）
§2 模块清单列举了 18 个矩阵别名文件（9 尺寸 × 2 精度），但 §3.8 描述为"其他 16 个矩阵别名文件同理（9 种尺寸 × 2 种精度 - 精度不重复的若干组合）"，前后矛盾。9 × 2 = 18，不存在"精度不重复的若干组合"。
**严重程度**: 中等
**改进建议**: 统一为 18 个文件（9 尺寸 × 2 精度），删除"精度不重复的若干组合"的模糊描述。同时修正 §3.8 中与之矛盾的文件计数。

### P5（中等）：矩阵除法 `/` 的 stub 依赖表述不准确
**位置**: §3.5「二元运算符（矩阵-矩阵，同尺寸）」— 除法子节
当前描述"矩阵除法等价于乘逆矩阵（this * inverse(rhs)）"存在两个问题：① `mat / scalar` 不依赖 inverse（逐元素操作），不应与 `mat / mat` 一并推迟；② 仓颉 `extend` 块中不能直接使用 `this`，且 GLM 实际实现为 `mat *= inverse(rhs)`（赋值语义而非乘法语义）。
**严重程度**: 中等
**改进建议**: 明确区分① `mat / scalar`（本阶段可实现，逐元素操作）和② `mat / mat`（依赖 inverse，推迟至阶段四）。将 `this * inverse(rhs)` 的描述改为 `mat *= inverse(rhs)` 以匹配 GLM 的实际语义。

### P6（轻微）：标量-矩阵除法的 stub 策略未说明
**位置**: §5「错误处理策略」
当前 §5 仅提及"本阶段矩阵除法（/）因依赖 inverse 函数（stub），运算符内部调用时抛出 stub 异常"，未区分 `mat / mat`（依赖 inverse）与 `mat / scalar`（不依赖 inverse），可能误导开发者将所有除法形式标记为 stub。
**严重程度**: 轻微
**改进建议**: 在 §5 中明确注明哪些除法形式受 stub 影响（仅 `mat / mat`），哪些可在本阶段完整实现（`mat / scalar`、`scalar / mat`）。
