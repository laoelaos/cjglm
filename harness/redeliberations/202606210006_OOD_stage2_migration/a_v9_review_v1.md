# OOD 设计方案审查报告（v1）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** 9 个矩阵类型各自独立定义为泛型结构体 `Mat{C}x{R}<T, Q>`，与阶段一 Vec 类型的设计策略一致。仓颉不支持模板偏特化（D01），独立类型是正确选择。每个矩阵类型的列向量/行向量类型在映射表中明确指定为具体 VecN<T,Q>，不依赖内部类型别名（D11），符合仓颉类型别名只能在文件顶层定义的限制。

**[通过]** extend 块中运算符使用 `where T <: Number<T>, Q <: Qualifier` 约束，符合仓颉泛型约束语法。跨类型工厂函数（fromParts/fromColumns/fromMat 6b）使用额外类型参数 U/SrcT + conv 闭包，在仓颉泛型系统能力范围内。

**[通过]** 运算符重载以成员运算符形式（mat op scalar）和全局函数形式（scalar op mat）分别处理，符合仓颉 `operator func` 的 this 绑定左操作数的语义。复合赋值运算符自动生成规则（返回类型与左操作数匹配时自动生成）与仓颉文档 §8.5 一致。分类表格（4 类复合赋值）清晰区分了方阵/非方阵的自动生成行为。

**[通过]** identity/filled/fromMat 工厂函数在 `Number<T>` 约束的 extend 块中通过 `someValue - someValue` 演算 T(0) 并要求调用方显式传入 `one: T` 获得 T(1)，此策略与仓颉无约束泛型下无法调用 T(0)/T(1) 构造的限制完全匹配（DV-01）。

**[通过]** `@Derive[Hashable]` 用于矩阵类型——设计中确认所有成员（Vec 列向量）已实现 Hashable，符合 `@Derive[Hashable]` 要求所有成员类型可哈希的前提约束。struct 上使用 @Derive 在仓颉自动派生规则中可行。

**[轻微]** §3.3 第 6a 条 fromMat 中的 SrcQ 类型参数：当同一个目标类型 T 的 fromMat<SrcQ> 被多次调用但 SrcQ 不同时，会生成不同的泛型实例化。文档已说明"SrcQ 并不显著膨胀运行时代码量"，此评估合理，但编码者需注意 SrcQ 的不同实参实际上是不同的静态函数——若编译器对 Qualifier 参数的泛型实例化策略有特殊优化则无影响，否则可能产生多个近乎相同的代码实例。此为轻微问题，不阻塞通过。

### 2. 标准库与生态覆盖

**[通过]** 设计依赖 `std.math.Number<T>` 接口进行泛型约束，用于 extend 块中算术运算符和工厂函数的类型约束——此接口已在阶段一实现中使用并验证可用。

**[通过]** `@Derive[Hashable]` 来自 `std.deriving`，阶段一已有使用先例（`@Derive[Hashable]` on Vec4）。

**[通过]** `@OverflowWrapping` 注解用于 scalar_mat_ops.cj 全局函数和复合赋值运算符，与阶段一 scalar_vec_ops.cj 的做法一致。标注在函数声明上，作用域为函数内的整数运算和整型转换——符合仓颉文档中溢出注解的使用规则。

**[通过]** stub 文件（common.cj/matrix.cj/geometric.cj）仅提供空壳签名，不依赖标准库中尚未验证的能力。各 stub 函数签名中使用的约束（`Number<T>`、`Comparable<T>`、`Equatable<T>` 等）均为已知可用的标准库接口。

**[通过]** lib.cj 中 `public import glm.detail.{ add, sub, mul, div, mod }` 的重导出已覆盖 scalar_mat_ops.cj 中同名的 add/sub/mul/div 函数——两者同属 glm.detail 包，同名函数通过参数类型（Vec 型 vs 矩阵型）重载自然区分，无需额外 import。mod 函数仅覆盖 scalar_vec_ops.cj，文档已显式说明 scalar_mat_ops.cj 不提供 mod。

### 3. 语言特性可行性

**[通过]** 错误处理策略：下标越界使用 assert + fallback throw Exception，与阶段一 Vec 的 `componentAt` 和 `[]` 一致。方阵 mat/mat 除法运算符内部调用 stub inverse 后 throw Exception，推迟至阶段四。这些策略在仓颉异常体系内可行。

**[通过]** 并发设计：矩阵为值类型（struct），运算符返回新实例，天然线程安全——与阶段一一致，无需额外同步机制。此分析正确。

**[通过]** 资源管理：矩阵类型无资源管理需求，struct 值语义确保栈分配和自动复制，最大 4×4=16 标量的内存占用对栈分配友好。

**[通过]** 模块/包结构设计符合 cjpm 项目组织方式：
- src/detail/ 下文件声明 package glm.detail，与目录路径匹配
- src/ext/ 下文件声明 package glm.ext，与目录路径匹配
- src/ 根目录下 fwd.cj/lib.cj 声明 package glm，与目录路径匹配
- 包依赖方向为单向 DAG：glm.ext → glm.detail，glm → glm.detail

**[通过]** const init 可用性分析精确：逐分量构造和列向量构造为 const init（纯赋值），filled/identity/fromMat/fromParts/fromColumns 非 const（涉及运算或闭包调用），与仓颉 const init 规则一致——const init 内仅允许对实例成员变量赋值，不允许其他赋值表达式（如 `+=`、`-=`），不允许非 const 函数调用。

**[通过]** 运算符不能为 static、不能为泛型——设计中所有运算符重载符合此限制。跨尺寸矩阵乘法运算符定义在左操作数矩阵类型的 extend 块中，右操作数为不同矩阵类型（`operator func *(rhs: Mat{C2}x{R2}<T,Q>)`），这不是运算符本身的泛型参数，而是 extend 块级别的泛型上下文中的成员函数签名，符合仓颉语法。

**[通过]** 仓颉 struct 中同一类型的多个 extend 块允许存在，且跨扩展可见性规则允许非 private 成员互相访问。72 个 fromMat 重载分布在单个 extend 块中在语法上可行，文档建议可拆分为多个 extend 块——此方案在仓颉中完全可行。

### 4. 设计一致性

**[通过]** 各抽象的职责描述清晰无歧义：9 个矩阵结构体为值对象承载列向量数据，3 个 stub 文件为依赖闭合占位，scalar_mat_ops.cj 为标量左侧全局函数。

**[通过]** 协作关系形成闭环：方阵类型的 .inl 编排 → stub 文件（common.cj/matrix.cj/geometric.cj）→ 编译通过；scalar_mat_ops.cj → 矩阵类型（为 9 种类型提供全局函数）；glm → glm.detail（重导出）；glm.ext → glm.detail（类型别名）。

**[通过]** 行为契约足够指导后续实现：
- 构造函数体系（8 种）全部给出完整签名框架和泛型参数展开
- 算术运算符（一元/矩阵-标量/标量-矩阵/矩阵-矩阵/矩阵-向量）全部给出签名的泛型约束框架
- 矩阵乘法展开规则提供通用公式和 2 个具体示例
- stub 文件提供完整函数签名清单
- scalar_mat_ops.cj 明确列出 36 个全局函数的覆盖范围

**[通过]** 模块间依赖方向合理，无循环依赖：glm.detail 为核心层无外部依赖；glm 和 glm.ext 均单向依赖 glm.detail，相互无依赖。同包内类型互相直接可见，无需跨文件 import。

**[通过]** 上一轮审查的 14 个问题已全部在本版设计中得到响应和修改。修订说明（v10）逐条对应了每条审查意见的采纳情况和修改措施。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：每个矩阵类型文件只包含一个矩阵结构体定义+对应 extend 块；scalar_mat_ops.cj 专门处理标量左侧运算；3 个 stub 文件各自对应一方阵依赖闭合；别名文件各自包含 1-2 个类型别名。

**[通过]** 抽象层次恰当：架构级设计提供了类型形态、接口签名、行为契约和依赖关系，不涉及具体字段值或方法体实现细节。T(0)/T(1) 获取策略作为系统性设计约束统一描述，避免了各处重复分析。

**[通过]** 设计便于后续的详细设计和实现：
- 逐分量构造、列向量构造可直接在 struct 体内编码
- extend 块中工厂函数和运算符有明确的约束签名
- 矩阵乘法展开规则提供了编码参照公式
- stub 函数签名清单可直接复制为空壳实现
- cjpm 子包构建预验证步骤为编码启动提供了前置验证项

**[通过]** 设计便于单元测试（可 mock、可隔离）：
- 矩阵类型为值类型，运算符返回新实例，天然可隔离测试
- stub 函数不影响矩阵类型本身的测试（非方阵文件无 stub 依赖）
- 标量-矩阵运算为全局函数，可独立测试
- 别名类型的构造函数和运算符可用性验证已在测试文件清单中列出

**[轻微]** §3.5 矩阵-向量乘法运算符中 `Mat{C}x{R}<T,Q> * Vec{C}<T,Q> → Vec{R}<T,Q>` 和 `Vec{R}<T,Q> * Mat{C}x{R}<T,Q> → Vec{C}<T,Q>` 的签名为设计文档中的模板记法矩阵，编码时需逐一展开为 9 个具体矩阵类型各自对应的向量乘法签名。文档第 2 节映射表和示例已足够编码者推导，但若提供完整的 27 个乘法重载和 9+9 个向量乘法的具体签名列表会更便于编码参照。此为轻微问题，设计文档层次上不要求此级别细节。
