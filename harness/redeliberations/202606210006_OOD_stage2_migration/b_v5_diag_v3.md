## 发现问题

### P1（严重）：默认单位矩阵构造函数 `init()` 在无约束泛型 `T` 下依赖 `T(0)/T(1)` 不可行

**问题描述**：§3.3 第 1 条设计默认单位矩阵构造 `init()`，需要使用 `T(0)` 和 `T(1)` 表达式来构造各列向量的分量值。但矩阵结构体定义中泛型 `T` 无约束（与阶段一 Vec 类型一致，运算符定义在有 `Number<T>` 约束的 extend 块中），而仓颉无约束泛型参数**不支持 `T(n)` 构造调用**。

- **阶段一 Vec 类型如何回避此问题**：阶段一 Vec 类型**没有默认构造函数 `init()`**（实际实现 `type_vec2.cj:11` 起始构造函数为 `init(scalar: T)`，仅执行参数赋值 `this.x = scalar`，无需 `T(0)/T(1)`）。Vec 的 `const init(x: T, y: T)` 仅传递参数，不涉及 `T(n)` 调用
- **矩阵 `init()` 的独特困难**：矩阵的 `init()` 需要在无约束环境下构造 `T(0)` 和 `T(1)`，这在仓颉中不可行。这不是 `const init` vs 普通 `init` 的问题——**两种形式同样受此限制**。`const init` 不可用并非因为"const 上下文更严格"，而是根本原因在于无约束泛型 `T` 上无法调用 `T(0)/T(1)` 构造
- **降级方案也无效**：§3.3 第 1 条称"若 const init 不可用，退化为普通 init 构造函数"，但普通 `init()` 同样无法在无约束下构造 `T(0)/T(1)`。降级方案的承诺是虚假的

**所在位置**：§3.3 构造函数体系第 1 条

**严重程度**：严重

**改进建议**：将 §3.3 第 1 条重写为以下方案之一（推荐方案 c）：
- **(a)** 单位矩阵构造推迟到 `Number<T>` 约束的 extend 块中，作为 `public static func identity(): Mat4x4<T,Q>` 工厂函数
- **(b)** 参考阶段一 `fromBoolVec` 的模式，`init()` 增加 `zero: T, one: T` 参数：`init(zero: T, one: T)`
- **(c)** `init()` 的 `T(0)/T(1)` 标记为具现化时编译错误条件——仅当 `T` 实际类型支持无参 `T()` 及 `T(Int64)` 构造时方可通过编译，其余类型实例化时报编译错误。此方案最简洁且与阶段一处理 `T(0)/T(1)` 限制的方式一致（参见 deviations.md DV-01 的处理策略：不回避限制，而是声明约束条件）

无论选何种方案，需同步删除当前设计中的"const init 降级为普通 init"描述，改为对 `T(0)/T(1)` 约束条件的明确声明。需显式引用 deviations.md DV-01 约束。

### P2（严重）：§3.3 第 8 条跨矩阵类型转换构造的填充规则坐标复制范围定义与 GLM 实际行为不一致——当源矩阵列数充足但行数不足时，GLM 不保留源矩阵行不足列的部分数据

**问题描述**：§3.3 第 8 条的填充规则声明"源矩阵 M_src[j][i] 复制到目标矩阵的相同坐标位置，范围 0 ≤ j < min(C_src, C_dst)、0 ≤ i < min(R_src, R_dst)"。当 C_dst > C_src 时，额外列从单位矩阵取值；当 R_dst > R_src 时，额外行从单位矩阵取值。

此规则在**源矩阵列数充足（C_src ≥ C_dst）但行数不足（R_src < R_dst）**的场景下与 GLM 行为不一致。经直接核对 GLM 源码：

**Mat4x4 ← Mat4x2**（`type_mat4x4.inl:246-257`）：
```cpp
value{col_type(m[0], 0, 0), col_type(m[1], 0, 0), col_type(0, 0, 1, 0), col_type(0, 0, 0, 1)}
```

设计规则预测：min(C_src,C_dst)=min(4,4)=4, min(R_src,R_dst)=min(2,4)=2 → 对于第 2、3 列，应复制 `src[2][0]`、`src[2][1]`、`src[3][0]`、`src[3][1]`。但 GLM 实际行为是第 2、3 列**完全从单位矩阵构造**——`col_type(0, 0, 1, 0)` 和 `col_type(0, 0, 0, 1)`，不含任何源矩阵第 2、3 列的数据。

**Mat4x4 ← Mat3x2**（`type_mat4x4.inl:218-229`）：
```cpp
value{col_type(m[0], 0, 0), col_type(m[1], 0, 0), col_type(m[2], 1, 0), col_type(0, 0, 0, 1)}
```

同理，源 Mat3x2 的第 2 列（m[2]）是 Vec2(x,y)，GLM 将其扩展为 `Vec4(x, y, 1, 0)` 而非 `Vec4(x, y, 0, 0)`——其中 `(2,2)` 位置的 `1` 来自单位矩阵对角线，而非设计规则定义的"源矩阵复制范围"。

**GLM 的实际策略为逐列构造**：每一目标列向量由源矩阵对应列向量构造——若源矩阵有该列（j < C_src），则取该列的前 min(R_src, R_dst) 个分量直接复制，剩余分量的填充规则为：仅当列索引 j 恰好等于该分量的行索引（即对角线位置 (j,j) 且 j ≥ R_src）时填 `T(1)`，其余填 `T(0)`。若源矩阵无该列（j ≥ C_src），则整列从单位矩阵取值。

这与设计规则的坐标复制范围 `0 ≤ i < min(R_src, R_dst)` 关键差异在于：**当列索引 j < C_src 但该列中行索引 i ≥ R_src 时**，GLM 并不填 0，而是按"单位矩阵对角线延伸"规则处理（`(j,j)` 填 1、其余填 0）。设计规则的"复制范围限制在 min(R_src, R_dst) 以内"和"额外行统一从单位矩阵取值"两个条款组合后，**对列索引 j < C_src 的列的处理与 GLM 的逐列构造行为不一致**——设计规则未区分"源矩阵有该列"的情况下的行不足填充。

**所在位置**：§3.3 构造函数体系第 8 条

**严重程度**：严重（72 个转换构造函数中所有涉及"源矩阵行数 < 目标矩阵行数"的转换均受影响；编码者若严格按设计规则实现，将无法正确实现 Mat4x2→Mat4x4 等常见转换）

**改进建议**：将填充规则重写为逐列构造规则：

> 目标矩阵的第 j 列构造规则：
> - 若 j < C_src：取源矩阵第 j 列的前 min(R_src, R_dst) 个分量复制到目标矩阵第 j 列的对应位置；剩余行中，列索引等于行索引（即对角线 (j,j) 且 j ≥ R_src）时填 `T(1)`，其余填 `T(0)`
> - 若 j ≥ C_src：整列从单位矩阵取值——(j,j) 填 `T(1)`，其余填 `T(0)`

同时在 §9 差异声明中注明：仓颉版本的跨矩阵转换行为以 GLM `.inl` 逐列构造逻辑为权威来源，编码阶段须逐一对照 `.inl` 文件实现。

### P3（一般）：第 2 条与第 9 条构造函数存在重载歧义风险——当 `U=T` 时参数类型完全重叠

**问题描述**：§3.3 第 2 条定义 `init<Q2>(m: Mat4x4<T, Q2>)`（跨 Qualifier 复制），第 9 条定义 `init<U, P>(m: Mat4x4<U, P>)`（跨类型同尺寸转换）。当第 9 条实例化为 `U=T, P≠Q` 时，参数类型为 `Mat4x4<T, P>`，与第 2 条实例化为 `Q2=P` 时的参数类型 `Mat4x4<T, P>` 完全相同。

仓颉函数重载解析规则基于参数类型（含泛型参数）进行匹配。当两个构造函数的参数类型在特定实例化下相同时，编译器需要根据"更具体的签名"规则进行选择。第 2 条的 `Q2` 是自由泛型参数，第 9 条的 `U, P` 也是自由泛型参数——两者具体程度无差别，编译器很可能报歧义错误。

**所在位置**：§3.3 构造函数体系第 2 条 vs 第 9 条

**严重程度**：一般

**改进建议**：选择以下方案之一消除歧义：
- **(a) 推荐**：删除第 2 条，将其功能完全由第 9 条覆盖（当 `U=T` 时即跨 Qualifier 复制）——此方案最简洁
- **(b)** 保留第 2 条，但在第 9 条中明确约束 `U ≠ T`（需仓颉支持 where 约束中的类型不等式，当前版本可能不支持）
- **(c)** 若第 2 条保留，在设计文档中标注此歧义风险并提供编码阶段的处理指引（如：若编译器报歧义，删除第 2 条）

### P5（一般）：ext/ 向量别名与 fwd.cj 向量别名的定位差异和使用指引缺失

**问题描述**：§3.8 ext/ 别名文件中定义了向量别名（如 `vector_float4.cj` 中的 `vec4 = Vec4<Float32, PackedHighp>`），而 fwd.cj 中已定义了 `Vec4 = detail.Vec4<Float32, detail.PackedHighp>`（PascalCase）。两处对 Float32 highp 的 Vec4 提供了不同名称的冗余别名。

设计文档未说明：① 为什么需要两套别名；② 用户应优先使用哪个；③ 两套别名之间的差异仅在于命名风格（PascalCase vs camelCase）还是包路径不同（`import glm.Vec4` vs `import glm.ext.vector_float4.vec4`）。

此问题首次由第 3 轮审查 P11 提出，经第 4 轮审查保留，第 5 轮审查 b_v5_diag_v2 P5 再次指出，至今已历 3 轮未获回应。

**所在位置**：§3.8 ext/ 别名文件 — 向量别名文件

**严重程度**：一般

**改进建议**：在 §3.8 中补充说明：ext/ 下的 camelCase 别名（vec1~vec4/dvec1~dvec4）与 fwd.cj 中的 PascalCase 别名（Vec1~Vec4/DVec1~DVec4）是同一类型的冗余别名，提供两套访问风格（与 GLM ext/vector_float*.hpp vs fwd.hpp 的二元体系一致）。推荐用户优先使用 fwd.cj 中的别名（import 路径更短 `import glm.{Vec4}`），ext/ 别名主要用于与 GLM C++ 代码做逐文件对应迁移场景。

### P6（一般）：`col()` 和 `[]` 的越界保护机制使其无法声明为 `const func`，但设计文档未提及此限制

**问题描述**：§3.4 和 §5 规定下标越界时"抛出 Exception（含 assert + fallback throw）"。但阶段一的实际实现已表明 `assert` 不是 `const` 函数（deviations.md IF-03 记录了 `componentAt` 因 assert/throw 无法声明为 const 的限制）。

矩阵的 `[]` 运算符是否需要 `const` 版本？设计文档未讨论。如果用户需要在 `const` 上下文中访问矩阵列（如 `const val m = Mat4x4(...); const val col = m[0]`），当前设计中的 assert + throw 模式将阻止 `[]` 声明为 `const func`。这与 §3.3 第 1 条中对 `const init` 的关注一致——设计文档在构造函数中对 const 上下文表现出了前瞻性考量，但在列访问运算符中缺少相同的关注。

**所在位置**：§3.4 行列访问；§5 错误处理策略

**严重程度**：轻微

**改进建议**：在 §3.4 或 §5 中标注：`[]` 和 `col()` 的越界保护机制（assert + throw）使其无法声明为 `const func`，与阶段一 `componentAt` 面临的限制一致（IF-03）。若后续阶段有 `const` 上下文矩阵列访问需求，需提供不含 assert/throw 的替代访问路径。

## 整体评价

经 5 轮迭代审查，设计方案在架构层面已高度成熟，核心决策（9 个独立结构体、列主序、extend 块运算符、stub 函数库）合理且可行。

**上一轮审查报告核心更正**：上一轮审查 b_v5_diag_v2 中 P1 正确论证了"当前设计的单位矩阵扩展填充规则与 GLM 一致"——此结论经源码核验仍然成立。上一轮 P4 声称 Mat4x4 ← Mat4x2 的第 3 列为 `col_type(m[3], 0)` 系事实错误（实际为 `col_type(0, 0, 0, 1)`），质询报告已正确指出此问题。但上一轮 P4 触及的设计规则真实缺陷（坐标复制范围与 GLM 逐列构造行为不一致）仍然存在——本报告 P2 以正确的 GLM 源码引用重新定位了此问题。

**当前最关键的两个遗留问题**：
1. **P1（严重）**：`init()` 默认构造函数在无约束泛型 `T` 下不可行——此为编码落地的实际阻塞项，必须在编码前给出明确方案
2. **P2（严重）**：填充规则的坐标复制范围定义与 GLM 逐列构造行为不一致——所有涉及源矩阵行数 < 目标矩阵行数的转换均受影响，编码者若按当前规则实现将产出与 GLM 不同的结果
