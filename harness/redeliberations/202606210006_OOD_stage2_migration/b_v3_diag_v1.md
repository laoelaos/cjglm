## 发现的问题

### P1（一般）§2 关键行为验证示例注解错误

**问题描述**：跨尺寸矩阵乘法核验示例的括号注解中写 `C1=R2=3`，但 Mat2x3 的列数 C1=2，Mat3x2 的行数 R2=2，正确值应为 `C1=R2=2`。乘法结果 `Mat3x3` 是正确的，仅注解有事实错误。

**所在位置**：§2「关键行为验证示例」第二行：
```
（Mat{C1}x{R1} * Mat{C2}x{R2} → Mat{C2}x{R1}，C1=R2=3）
```

**严重程度**：一般

**改进建议**：将 `C1=R2=3` 修正为 `C1=R2=2`。

---

### P2（一般）§3.1 默认单位矩阵构造的 const init 条件性未配套降级方案

**问题描述**：第 1 条默认构造函数描述为"如果仓颉允许 const init 中包含列向量构造表达式链"时采用 `const init` 形式。但未给出该条件不满足时的明确降级方案。若编译器不允许，则默认构造函数降为运行时非 const，影响编译期矩阵构造场景。

**所在位置**：§3.3 构造函数体系第 1 条

**严重程度**：一般

**改进建议**：补充 `const init` 不可用时的降级方案（如退化为普通 `init` + 编译期不可用于 const 上下文的约束说明），与阶段一 OOD 中 `const init` 条件性策略的处理方式一致。

---

### P3（一般）§3.4 `col()` 与 `[]` 行为差异化描述不清晰

**问题描述**：§3.4 描述 `col(i)` 为"提供带边界检查的安全访问"，但 §5 明确 `[](i)` 下标越界同样"抛出 Exception"——两者行为似乎完全一致。如果 `col()` 与 `[]` 行为无区别，则 `col()` 是冗余 API。如果存在区别（如 `col()` 提供更详细的错误信息），应明确说明。

**所在位置**：§3.4 行列访问

**严重程度**：一般

**改进建议**：明确 `col()` 与 `[]` 的行为差异（例如 `col()` 是否有不同的错误消息？是否提供列名级描述？）或合并描述。若两者确实行为等价，考虑是否移除 `col()` 以简化 API 面，或保留但阐明设计意图（如作为同阶段一 `componentAt` 对齐的具名替代）。

---

### P4（一般）scalar_mat_ops.cj 与阶段一命名约定冲突未处理

**问题描述**：阶段一 OOD（`docs/03_ood_phase1.md:108`）明确约定：
> `glm.detail` 包中约定**仅 `scalar_vec_ops.cj`** 定义名为 `add`、`sub`、`mul`、`div`、`mod` 的包级函数。

本设计新增 `scalar_mat_ops.cj` 在同一个包 `glm.detail` 中定义同名函数 `add/sub/mul/div`（36 个重载），直接违反了上述命名约定。该约定是为了避免 `public import glm.detail.{ add, sub, mul, div, mod }` 产生导入歧义。虽然重载解析（参数类型不同：Vec vs Mat）很可能能区分，但此约定未在设计中讨论或更新，可能造成编码阶段混淆。

**所在位置**：§3.5 标量-矩阵运算段落，§8 产出物清单

**严重程度**：一般

**改进建议**：在 §7 设计决策中新增条目，说明命名约定已更新为"scalar_vec_ops.cj 和 scalar_mat_ops.cj 共同定义 add/sub/mul/div 包级函数，通过重载解析（参数类型不同）区分"，或选择在 scalar_mat_ops.cj 中使用不同函数名（如 `matAdd`/`matSub`/`matMul`/`matDiv`）以完全避免冲突。

---

### P5（一般）§8 `fwd.cj` 矩阵别名命名规范未定义

**问题描述**：§8 产出物清单中仅写"新增矩阵类型别名（Mat2x2f、Mat4x4 等）"，但未定义 `fwd.cj` 中使用的具体命名模式。ext/ 目录的别名文件已有明确的命名模式（如 `public type mat4x4 = ...`），但 `fwd.cj` 的矩阵别名命名规则（如 `Mat2x2f`？`Mat2x2`？`Mat4`？精度变体如何对应？）未说明，实现阶段可能导致风格不一致。

**所在位置**：§8 产出物清单 — 修改文件 — `src/fwd.cj`

**严重程度**：一般

**改进建议**：明确 `fwd.cj` 中矩阵别名的命名规则，例如：
- 是使用 `type Mat2x2 = detail.Mat2x2<Float32, PackedHighp>` 还是 `type Mat2x2f = ...`？
- 是否有精度变体（`HighpMat2x2`、`LowpMat2x2` 等）？与阶段一向量的命名规范（如 16 家族 × 4 精度变体）是否保持一致？
- 方阵是否提供不带尺寸的短别名（如 `type Mat4 = Mat4x4<Float32, PackedHighp>`）？

---

### P6（轻微）§3.1/§3.6 未讨论 `@Derive[Hashable]` 在矩阵类型上的一致性

**问题描述**：阶段一 `Vec4` 等类型使用 `@Derive[Hashable]` 自动派生。矩阵类型继承此模式的考虑未在设计中讨论。若矩阵类型无法 Hashable，则无法用于 `HashSet`/`HashMap`。

**所在位置**：§3.1、§3.6

**严重程度**：轻微

**改进建议**：在 §7 设计决策中记录矩阵类型的 Hashable 支持策略（如：支持/不支持/条件性支持），与阶段一保持一致。如选择支持，明确是否使用 `@Derive[Hashable]` 或手动实现。

---

### P7（轻微）路线图中可选的 `type_float.cj`/`type_half.cj` 未做取舍说明

**问题描述**：`docs/02_roadmap.md` §2 阶段二范围中列出了"可选：type_float.cj、type_half.cj（浮点类型标签，仅依赖 setup.cj）"。设计文档未说明这两个文件是否纳入本阶段范围。若纳入应列出文件路径和内容，若不纳入应注明决定和理由。

**所在位置**：§2 包组织清单 / §8 产出物清单

**严重程度**：轻微

**改进建议**：在产出物清单中明确注明 `type_float.cj` 和 `type_half.cj` 的处理决定（纳入/延迟至后续阶段/不实现），保持与路线图的可追溯性。
