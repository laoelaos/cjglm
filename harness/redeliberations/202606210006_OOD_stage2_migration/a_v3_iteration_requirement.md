## 当前审查结果

以下问题来自上一轮组件B的诊断报告（LOCATED，所有诊断结论已确认）：

### 未充分响应需求的方面

**Q01（严重）：ext/ 向量别名文件未纳入设计范围**
路线图将 `ext/` 下的向量/矩阵别名文件一并列为阶段二范围。当前设计只规划了18个矩阵别名文件，未包含任何向量别名文件。阶段一现有实现也未在 `src/ext/` 下提供这些文件（向量别名集中在 `fwd.cj` 中）。设计未讨论是否将现有向量别名从 `fwd.cj` 拆分/迁移到 `ext/`，也未说明不迁移的理由。
- **所在位置**：§2「包组织」— `package glm.ext` 文件清单
- **改进建议**：明确阶段二是否包含 ext/ 向量别名文件的创建。若包含，补充对应的向量别名文件；若不包含，说明理由并在 §9 偏差声明中记录此范围调整。

### 事实错误

**Q02（严重）：D08 等效替代建议数学错误——`+= identity` ≠ `++m`**
决策 D08 称矩阵自增"可通过 += identity 等价替代"。但 `++m` 在 C++ GLM 中将每个矩阵元素 +1，而 `+= identity` 仅在对角线元素加 1、非对角线元素不变。两者数学语义完全不同。
- **所在位置**：§7 D08（决策表第 8 行）
- **改进建议**：修正为"可通过 `+= T(1)`（逐元素加 1）等价替代" 或 "可通过 `+= identity`（仅对角线加 1，与 C++ GLM 的 `++m` 语义不同）"——前者保持行为一致，后者需标注行为差异。

**Q03（严重）：跨矩阵类型转换构造——72 个独立重载未分析**
§3.3 第 8 条描述为一条泛化"跨矩阵类型转换构造"，但实际上在仓颉中 9 个矩阵类型是独立结构体，无法用单一泛型构造函数处理所有跨尺寸转换。每个目标矩阵类型需要为每种源矩阵类型定义独立构造函数重载——约 72 个构造函数重载。设计未评估此实现量级的可行性、未讨论是否可以缩减、未讨论交叉重复的填充逻辑如何复用。
- **所在位置**：§3.3「构造函数体系」第 8 条
- **改进建议**：①明确 72 个构造函数重载是否可以接受，若不能接受提出缩减方案；②为每个转换构造函数明确填充规则中的边界情况；③评估仓颉重载解析在 72 个构造函数 + 其他构造函数共存时是否存在歧义。

### 逻辑矛盾/设计不一致

**Q04（中等）：`componentAt` 返回类型与 Vec 模式不一致**
§3.4 定义 `componentAt(i: Int64): VecR<T,Q>` 返回列向量。但在阶段一 Vec 类型中，`componentAt` 返回的是标量分量。矩阵的 `componentAt` 返回整个列向量，函数命名语义与 Vec 版本矛盾。
- **所在位置**：§3.4「行列访问」
- **改进建议**：改名以避免歧义，如 `col(i: Int64): VecR<T,Q>` 或直接移除（下标访问已覆盖此功能）。或保留 `componentAt` 但语义明确返回标量元素：`componentAt(col: Int64, row: Int64): T`。

**Q05（中等）：`row()` 成员函数在本阶段无实际作用且与 C++ GLM 不一致**
§3.4 定义 `row(i: Int64): VecC<T,Q>` 为可选的成员函数。在 C++ GLM 中，行访问是自由函数 `glm::row()`（属于函数库，阶段四才完整实现）。若本阶段实现为成员函数，则在阶段四替换为自由函数时产生 API 不兼容。
- **所在位置**：§3.4「行访问函数」
- **改进建议**：明确决策——要么定义为成员函数作为阶段二实现的一部分（消除"可选"模糊性），要么标记为延期至阶段四随 `matrix.cj` 一起提供自由函数版本。更新 §7 决策表补充此决策。

**Q06（中等）：跨类型构造函数（不同 T）未明确覆盖跨类型同尺寸矩阵转换**
C++ GLM 支持不同类型的同矩阵尺寸构造（如 `Mat4x4<Float32,Q>` 从 `Mat4x4<Int32,P>` 构造）。当前设计的构造函数清单中：第 2 条只覆盖了 T 相同的跨 Qualifier 复制；第 8 条只覆盖了不同尺寸。缺少 `init<U, P>(m: Mat4x4<U, P>)` 的明确定义。
- **所在位置**：§3.3「构造函数体系」
- **改进建议**：在构造函数清单中补充跨类型同尺寸转换构造函数，或说明其已包含在第 6/7 条的泛化描述中并明确签名。

### 深度/完整性不足

**Q07（中等）：标量-矩阵运算全局函数的溢出策略未说明**
§5 描述了矩阵算术运算的 `@OverflowWrapping` 溢出策略，但仅提及"算术复合赋值运算符标注 @OverflowWrapping"。`scalar_mat_ops.cj` 中的 36 个全局函数的溢出策略未说明——它们使用 `@OverflowWrapping` 还是继承调用方的溢出上下文？与阶段一 `scalar_vec_ops.cj` 的做法是否一致？
- **所在位置**：§5「错误处理策略」
- **改进建议**：明确 `scalar_mat_ops.cj` 各函数的 `@OverflowWrapping` 标注策略，与阶段一 `scalar_vec_ops.cj` 保持一致。

**Q08（中等）：矩阵除法 `/` 的 mat/vec 运算未讨论**
§3.5 区分了 `mat / mat`（依赖 inverse，推迟）和 `mat / scalar`（完整实现）。但 C++ GLM 还定义了 `mat / vec` 和 `vec / mat` 除法，它们依赖 `inverse(m) * v` 或 `v * inverse(m)` 实现。设计未提及这两种除法运算。
- **所在位置**：§3.5「二元运算符（矩阵-矩阵）」除法子节
- **改进建议**：明确 `mat / col_vec` 和 `row_vec / mat` 是否在本阶段支持。若依赖 inverse 则也推迟，应在 §3.5 中注明。

**Q09（轻微）：跨矩阵转换填充规则对非数值类型适用性未讨论**
§3.3 第 8 条的填充规则规定缺失行列中 `(i,i)` 填 `T(1)`、其余填 `T(0)`。当矩阵实例化为非数值类型（如 `Bool`、自定义类型）时，`T(0)` 和 `T(1)` 构造是否合法？这与阶段一 Vec 构造函数面临的 `T(n)` 构造限制（见 `docs/deviations.md` DV-01）是同一类问题。
- **所在位置**：§3.3「构造函数体系」第 8 条
- **改进建议**：标注此约束——跨矩阵转换构造函数要求 `T` 支持 `T(0)`/`T(1)` 构造，否则实例化时报错，与阶段一的延迟检查策略一致。
