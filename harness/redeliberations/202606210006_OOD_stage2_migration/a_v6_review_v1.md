# OOD 设计方案审查报告（v1）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** 9 个矩阵类型各自独立定义为泛型 struct，与阶段一 Vec 类型策略一致。仓颉泛型 struct 支持多类型参数和 where 约束（以 `struct Mat4x4<T, Q> where Q <: Qualifier` 形式），类型形态选择合理。

**[通过]** 矩阵类型以列向量（VecN<T,Q>）为数据成员，列向量类型由矩阵尺寸唯一确定，使用处直接引用具体 Vec 类型（如 Vec4<T,Q>），避开仓颉类型别名不能在结构体内部定义的限制。此模式与阶段一 Vec 实现（不在 struct 内部定义类型别名）一致，经编译验证可行。

**[通过]** 运算符重载定义在带约束的 extend 块中（如 `extend<T, Q> Mat4x4<T, Q> where T <: Number<T>, Q <: Qualifier`），与阶段一 Vec 类型模式一致。仓颉扩展可添加运算符重载、成员函数和接口实现（extend 文档 §2.2），且支持带 where 约束的泛型扩展（extend 文档 §2.3 + §2.4），此设计在仓颉类型系统中可行。

**[通过]** `identity()` 静态工厂函数定义在 `Number<T>` 约束的 extend 块中。仓颉扩展支持 static 成员函数（extend 文档 §4.2 允许 `static` 修饰符），且泛型约束的扩展可行。T(0) 和 T(1) 的演算方案引用阶段一 `increment()` 中已验证的 `-(!(this.x - this.x))` 模式（`type_vec2.cj:126`），在 `Number<T>` 约束下 `a - a` 得零值、`!zero` 得 -1、`-(-1)` 得 1，运算链由 `Number<T>` 和 `Integer<T>` 的运算符支撑，可行。设计文档已正确识别了静态函数无 `this` 的问题，并提出通过构造全零矩阵后再设置对角线的编码方案。

**[通过]** D20 决策明确 `init()` 无参构造函数在仓颉不可行，已正确理解仓颉泛型约束规则（§7.1："无约束时只能传递/返回泛型类型的值，不能进行其他操作"），且 `Number<T>` 不提供 `T(n)` 构造声明（deviations.md DV-01 同根因），删除原 `init()` 改为 `identity()` 工厂函数的方案可行。

**[通过]** D21 决策删除原 §3.3 第 2 条 `init<Q2>(m: Mat4x4<T, Q2>)` 以消除与第 7 条 `init<U, P>(m: Mat4x4<U, P>)` 在 `U=T` 时的重载歧义，方案合理。剩余的第 7 条在 `U=T, P≠Q` 时自然覆盖跨 Qualifier 复制，功能无损。

**[通过]** 72 个跨矩阵类型转换构造函数的可行性：每个重载的参数类型为不同的矩阵结构体类型（Mat2x2、Mat2x3 等），类型分集度极高，仓颉重载解析（函数文档 §7.3："最具体匹配"）可区分。与阶段一已实现的 64 个向量跨类型构造函数可行性前提一致。

**[轻微]** `identity()` 的 T(0) 演算方案中 `let zero = this.c0[0] - this.c0[0]` 使用了 `[]` 运算符，但 `[]` 在无约束泛型 struct 定义中声明（不限 `Number<T>`），而 `identity()` 需在 `Number<T>` 约束的 extend 块中，此处访问列向量分量可用 `this.c0.x` 等属性访问替代 `[]` 运算符，编码更简洁且避免 const 限制问题。不过使用 `[]` 也无语法障碍，此为偏好性建议。

### 2. 标准库与生态覆盖

**[通过]** 矩阵运算所需的核心数学能力（算术运算、比较运算）均由 `Number<T>`、`Integer<T>`、`Equatable<T>`、`Comparable<T>` 等标准库接口提供，与阶段一 Vec 类型完全一致。无需引入额外的标准库能力。

**[通过]** `@Derive[Hashable]` 由 `std.deriving` 提供（阶段一已使用），矩阵 struct 的所有列向量成员均为已实现 Hashable 的 Vec 类型，派生条件满足。`@OverflowWrapping` 注解由仓颉内置支持（反射与注解文档中记录），阶段一已广泛使用。

**[通过]** stub 函数库（common.cj、matrix.cj、geometric.cj）仅提供空壳签名，不依赖标准库高级功能。异常处理使用 `Exception`（std.core 提供），与阶段一一致。

**[通过]** `assert` + fallback `throw` 的越界保护模式与阶段一 `componentAt`（`type_vec2.cj:28-34`）一致，标准库 `Exception` 和内置 `assert` 均可用。

**[轻微]** devicions.md 中记录了 `std.math.fmod` 用于浮点取模场景（DV-02），矩阵-矩阵除法的 stub 实现后续可能用到 `std.math` 中的其他函数（如 determinant/inverse 算法中的 sqrt/abs 等），但这些属于阶段四完整实现范畴，当前设计不涉及。

### 3. 语言特性可行性

**[通过]** 错误处理策略与阶段一一致：下标越界使用 `assert` + fallback `throw Exception`，与 `type_vec2.cj:28-34` 的模式完全一致。矩阵 `[]` 运算符和 `col()` 因 assert/throw 不可声明为 `const func`，此限制已在 §5 和 §9 中正确记录，与 deviations.md IF-03 同类限制一致。

**[通过]** 并发设计不引入额外同步机制。矩阵为值类型（struct），运算符返回新实例，天然线程安全。此分析与仓颉并发模型兼容。

**[通过]** 资源管理无特殊需求。矩阵 struct 自动栈分配，值语义复制，无需手动资源管理或 `Resource` 接口。

**[通过]** 模块/包结构设计符合仓颉 cjpm 项目组织方式：`glm.detail` 包位于 `src/detail/`，`glm.ext` 包位于 `src/ext/`，`glm` 包位于 `src/`。包声明与目录路径匹配规则（package 文档 §2.1："须与相对于 src/ 的目录路径匹配"）满足。`glm.ext` 作为 `glm` 的子包，与 `glm.detail` 模式一致。

**[通过]** lib.cj 中 `public import glm.detail.{ add, sub, mul, div, mod }` 自动覆盖 scalar_mat_ops.cj 中同名函数——两者同属 `glm.detail` 包，同名函数通过参数类型重载自然覆盖，无需额外 import。

**[通过]** `init(scalar: T)` 标量填充构造函数声明了需要 T 支持零值构造。设计明确指出需在 `Number<T>` 约束的 extend 块中实现或提供零值参数，与 DV-01 同类限制处理策略一致。

**[通过]** 跨矩阵类型转换构造函数中 `T(0)/T(1)` 构造限制已正确识别，并添加了约束声明："对于不支持的数值类型在实例化时报编译错误"。这与阶段一 Vec 构造函数面临的设计约束一致。

**[轻微]** 跨矩阵类型转换构造函数的 T(0)/T(1) 依赖，在无约束泛型 struct 定义中同样不可行（与 `init(scalar: T)` 的零值构造面临同类限制）。编码时这些构造函数也需在带约束的 extend 块中实现，或在 struct 定义中以 `init<U>(m: MatCxR<U, P>)` 形式声明并在函数体内通过 `U` 到 `T` 的隐式转换赋值（仓颉数值类型 `T(e)` 转换）。设计文档已标注此约束（§3.3 第 6 条），此为编码细节层面备注，不阻塞设计可行性。

### 4. 设计一致性

**[通过]** 各抽象职责描述清晰：矩阵 struct 为值对象承载列向量数据和运算；stub 文件为依赖闭合的空壳；ext/ 文件为具现化别名。职责边界明确无歧义。

**[通过]** 协作关系闭环：矩阵类型依赖 Vec 列向量（阶段一已实现）→ 矩阵构造/运算使用 Vec 类型 → fwd.cj 和 ext/ 导出别名 → lib.cj 重新导出。所有依赖路径完整，无缺失环节。

**[通过]** 行为契约描述完整：§4 提供了矩阵构造、乘法、跨尺寸转换、跨类型转换、ext/ 别名使用等关键场景的行为示例，足以指导后续实现。

**[通过]** 模块间依赖方向合理：`glm.detail` 为核心无外部依赖（仅依赖标准库和内部 Vec 类型）；`glm` 和 `glm.ext` 同级单向依赖 `glm.detail`；两者相互无依赖。无循环依赖。

**[通过]** ext/ 向量别名与 fwd.cj 向量别名的定位差异已在本轮充分说明（§3.8 新增段落）：两套别名指向同一类型，差异仅在命名风格和包路径，推荐优先使用 fwd.cj PascalCase 别名，ext/ 别名用于 GLM C++ 逐文件迁移场景。此问题已彻底解决。

**[通过]** 跨矩阵类型转换填充规则已提供精确定义（§3.3 第 6 条：源矩阵元素复制到目标矩阵相同坐标位置，额外行列从单位矩阵取值），且已添加编码阶段注意事项：某些转换构造函数与主规则存在局部差异，编码时须逐一对照 GLM .inl 源码。此处理策略务实合理。

**[轻微]** `identity()` 静态工厂函数的 T(0)/T(1) 演算描述中提到"构造一个全零矩阵（列向量构造 + 标量运算），再设置对角线元素"，但"构造全零矩阵"本身也需要在 `Number<T>` 约束下获得零值。具体编码方案需在实现阶段细化（如利用 `Vec2(scalar - scalar, scalar - scalar)` 构造零列向量），此为编码细节，不影响设计一致性。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责：矩阵 struct 只承载数据和运算；scalar_mat_ops.cj 专门处理标量左侧运算（与 scalar_vec_ops.cj 平行）；stub 文件仅满足依赖闭合；ext/ 文件仅包含类型别名。各模块职责单一且内聚。

**[通过]** 抽象层次恰当：9 个独立矩阵类型而非统一泛型避免模板偏特化问题；运算符分布在 extend 块中按约束分层；identity() 工厂函数解决了泛型构造限制。不过度设计（无冗余抽象层）也不设计不足（所有必要功能均已覆盖）。

**[通过]** 设计便于后续实现：§3.3 构造函数体系按优先级排列且给出 Mat4x4 示例并注明其他矩阵按列数/行数适配；§3.5 运算符列表完整（含 27 个矩阵乘法重载）并给出文件归属策略；编码注意事项已标注了填充规则特例和 GLM .inl 对照需求。

**[通过]** 设计便于单元测试：矩阵为值类型（struct），运算符返回新实例，无副作用，天然可隔离测试。@Derive[Hashable] 支持矩阵用于 HashSet/HashMap 测试场景。stub 函数允许方阵文件编译通过，独立于函数库实现。

**[通过]** 设计迭代充分：P1（identity() 工厂函数替代 init()）、P2（消除构造函数重载歧义）、P4（ext/ vs fwd.cj 别名定位）、P5（填充规则编码注意事项）、P6（const func 限制记录）等关键问题均已在 v9 修订中彻底解决。历史遗留问题（init() 泛型约束、ext/ 别名定位、构造函数歧义）已全部闭环。

**[轻微]** `init(scalar: T)` 标量填充构造函数的逻辑等价性与 `identity()` 存在功能重叠：`Mat4x4<Float32, PackedHighp>(1.0f)` 与 `Mat4x4<Float32, PackedHighp>.identity()` 对方阵产生相同结果。设计文档已区分两者（`init(scalar)` 对非方阵产生 min(C,R) 对角线长度的对角矩阵，`identity()` 对非方阵无定义/无意义），此重叠不影响设计质量但编码时可考虑在 `identity()` 内部委托 `init(scalar: T)` 以减少代码重复——此为实现建议，不阻塞。
