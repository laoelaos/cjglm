## 需要解决的问题

- **问题描述**：`identity()` 静态工厂函数中通过 `-(!(zero))` 演算 T(1) 依赖 `!` 运算符，但 `!` 属于 `Integer<T>` 接口而非 `Number<T>`，在 `Number<T>` 约束上下文中对浮点类型编译不通过
- **所在位置**：§3.3 第 8 条（第 222~228 行），D20 决策（第 464 行）
- **严重程度**：严重
- **改进建议**：`identity()` 需补充 T(1) 参数（如 `identity(one: T)`），或将约束改为 `Integer<T>`（但排除浮点矩阵），需在设计中明确决策并更新 D20

- **问题描述**：`init(scalar: T)` 标量填充构造需要 T(0) 但构造函数必须在 struct 体内定义（无 `Number<T>` 约束），`scalar - scalar` 不可用
- **所在位置**：§3.3 第 1 条（第 191~196 行）
- **严重程度**：一般
- **改进建议**：增加 `zero: T` 参数（如 `init(scalar: T, zero: T)`），需考虑与问题 3 的统一解决策略

- **问题描述**：72 个跨尺寸转换构造函数需要 T(0)/T(1) 但作为 init 只能在 struct 体内定义，无约束泛型上下文中 T(0)/T(1) 不可获得
- **所在位置**：§3.3 第 6 条（第 211~216 行）
- **严重程度**：一般
- **改进建议**：改为 extend 块中的命名工厂函数（如 `fromMat2x2(m, zero, one)`），需考虑与问题 2 的统一解决策略（质询报告指出两者根因相同但改进策略不一致）

- **问题描述**：`[]` 运算符赋值版本签名缺少 `mut` 修饰符
- **所在位置**：§3.4 行列访问（第 237~241 行）
- **严重程度**：一般
- **改进建议**：将签名补全为 `mut operator func [](i: Int64, value!: VecR<T,Q>): Unit`

- **问题描述**：跨类型逐分量构造（第 4 条）和跨类型列向量构造（第 5 条）的泛型类型转换在仓颉中实现可行性未分析，未参考阶段一 `castVecN` 模式
- **所在位置**：§3.3 第 4 条和第 5 条（第 206~210 行）
- **严重程度**：一般
- **改进建议**：分析跨类型构造在仓颉中的可行性约束，若不可行则降级为 `fromParts(v1, v2, ..., conv)` 工厂函数模式

- **问题描述**：lib.cj 中 `public import glm.detail.{ add, sub, mul, div, mod }` 与阶段一向量版本的同名函数导出，下游消费者同时 `import glm.*` 和 `import glm.detail.*` 时是否存在歧义未评估
- **所在位置**：§8 修改文件 — lib.cj 条目（第 498 行）
- **严重程度**：一般
- **改进建议**：补充说明下游消费者使用不同导入路径时重载解析不冲突的依据，或确认会产生歧义并提供规避方案

- **问题描述**：`const init` 条件性描述未为各构造函数分类标注是否可 const init，与 D20 中"构造函数不能在 extend 块中定义"的约束存在遗留矛盾
- **所在位置**：§3.3 第 2 条（第 197~200 行）
- **严重程度**：一般
- **改进建议**：为每个构造函数标注 const init 可用性：纯赋值构造可 const、涉及类型转换或运算的不可 const

- **问题描述**：`identity()` 未分析 `const func` 可行性；若不可为 const func 则 `const val m = Mat4x4<Float32, PackedHighp>.identity()` 不可用，与 C++ GLM constexpr 行为差异未在 §9 差异声明中记录
- **所在位置**：§3.3 第 8 条，§9 差异声明，D20
- **严重程度**：一般
- **改进建议**：在 T(1) 获取方案确定后补充 const func 可行性分析；在 §9 差异声明中补充对应差异项
