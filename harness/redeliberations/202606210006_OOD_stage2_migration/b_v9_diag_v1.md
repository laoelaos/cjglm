> 审查日期：2026-06-21
> 审查对象：a_v9_design_v1.md
> 审查轮次：第 9 轮（首轮审查）

## 审查发现

### 问题 1：非方阵 `operator/=(scalar)` 复合赋值运算符的存在性遗漏

**问题描述**：设计文档 §3.5 复合赋值运算符表格第④类指出方阵 `operator/=(mat/mat)` 由编译器自动生成并继承 stub 行为，但遗漏了对非方阵 `operator/=(scalar)` 复合赋值的说明。6 种非方阵类型虽有 `mat / scalar` 成员运算符（返回同类型），编译器也应自动生成 `mat /= scalar` 复合赋值，但设计文档未在任何分类中明确非方阵 `operator/=(scalar)` 的存在——第②类"标量运算的复合赋值"笼统写了"所有 9 种类型"，但未逐项列出 `/=` 对非方阵的适用性，与方阵 `/=(mat/mat)`（第④类，stub）形成区分模糊。此外，方阵同时拥有 `/=(scalar)` 和 `/=(mat/mat)` 两类 `/=`，编码者可能困惑于方阵的 `/=` 究竟是 stub 还是可用。

**所在位置**：§3.5 复合赋值运算符表格第②类和第④类

**严重程度**：一般

**改进建议**：在第②类中明确 `/=(scalar)` 对所有 9 种矩阵类型均可用（含非方阵），与方阵的第④类 `/=(mat/mat)` stub 状态形成对照；补充说明方阵同时存在两版 `/=` 重载（标量版可用，矩阵版 stub），编译器按右操作数类型自动选择重载

---

### 问题 2：`mat / scalar` 标量除法运算符返回类型非同类型时复合赋值不可自动生成——但实际返回同类型，表格描述可能产生误导

**问题描述**：§3.5 复合赋值运算符表格第②类"标量运算的复合赋值 +=scalar/-=scalar/*=scalar//=scalar"的"//=scalar"措辞可能使编码者误解为 mat /= scalar 也涉及 stub（因 §5 中 mat/mat 除法为 stub）。虽然 `mat / scalar` 为逐元素运算、不依赖 inverse，但设计文档未在同一表格中显式标注 `/=(scalar)` 与 `/=(mat/mat)` 的 stub 差异，可能导致编码者误将 `/=(scalar)` 也标记为 stub。

**所在位置**：§3.5 复合赋值运算符表格；§5 错误处理策略

**严重程度**：轻微

**改进建议**：在§3.5 复合赋值表格第②类中明确注明"//=scalar 逐元素操作，不依赖 inverse，本阶段完整实现（非 stub）"，与第④类方阵"/=(mat/mat) 为 stub"形成显式区分；在§5 中补充确认 mat /= scalar 不受 stub 影响

---

### 问题 3：跨尺寸乘法兼容性表缺少两个合法乘法组合

**问题描述**：§3.5 跨尺寸乘法兼容性表遗漏了以下 2 个 C1 = R2 的合法乘法组合：
- Mat3x3 * Mat2x3 → Mat2x3（C1=3, R1=3, C2=2, R2=3, C1=R2=3 ✓）
- Mat4x3 * Mat2x4 → Mat2x3（C1=4, R1=3, C2=2, R2=4, C1=R2=4 ✓）

验证方法：遍历全部 9×9=81 种矩阵类型对，筛选 C1 = R2 的组合：
- 按现有表中每行已有条目计算：Mat2x2 有 2 个、Mat2x3 有 3 个、Mat2x4 有 3 个、Mat3x2 有 3 个、Mat3x3 有 2 个、Mat3x4 有 3 个、Mat4x2 有 3 个、Mat4x3 有 3 个、Mat4x4 有 2 个——共计 24 个跨尺寸，加上 3 个同尺寸共 27 个。但实际穷举 81 种中 C1=R2 的组合为 29 个（含同尺寸 3 个），即跨尺寸应为 26 个而非 24 个。遗漏的 2 个即 Mat3x3*Mat2x3 和 Mat4x3*Mat2x4。

**所在位置**：§3.5 跨尺寸乘法兼容性表

**严重程度**：严重

**改进建议**：补充遗漏的 2 个乘法组合：
- Mat3x3 可与 Mat2x3 → Mat2x3 相乘
- Mat4x3 可与 Mat2x4 → Mat2x3 相乘

同时更新跨尺寸乘法总数为 26 个（非 24 个），总有效重载为 29 个（非 27 个）；同步更新 D04 和 §8 产出物清单中相关数量的描述

---

### 问题 4：`fromMat` 6a/6b 中 `one: T` 参数在目标尺寸小于源尺寸时的必要性未分析

**问题描述**：§3.3 第 6a/6b 条的 `one: T` 参数用于单位矩阵扩展填充（当 C_dst > C_src 或 R_dst > R_src 时）。但当目标矩阵尺寸**小于**源矩阵时（如 Mat4x4 → Mat2x2），不存在额外行列，不需要单位矩阵填充，`one` 参数实际未被使用。设计文档未分析此场景——对于 72 个 fromMat 重载中尺寸缩减的方向（如 Mat4x4→Mat2x2、Mat4x4→Mat3x3 等），`one` 参数冗余但又不可省略（因为同一签名需统一）。此冗余对下游消费者造成不必要的调用负担。

**所在位置**：§3.3 第 6a/6b 条

**严重程度**：一般

**改进建议**：在 §3.3 中明确说明 `one` 参数在尺寸缩减方向上的冗余性——当 C_dst ≤ C_src 且 R_dst ≤ R_src 时 `one` 参数虽签名中存在但不参与运算；评估是否可额外提供无 `one` 参数的缩减方向工厂函数重载（如 `fromMatTruncate`），或至少在文档中为编码者标注此冗余的简化路径

---

### 问题 5：第 7 条（跨类型同尺寸转换）与第 6a 条功能重叠未显式区分

**问题描述**：§3.3 第 7 条 `fromMat<U, P>(m: Mat4x4<U, P>, conv: (U) -> T)` 与第 6a 条 `fromMat<SrcQ>(m: Mat{C_src}x{R_src}<T, SrcQ>, one: T)` 在 U=T、尺寸相同、SrcQ=P 时功能完全重叠——两者都执行同类型同尺寸跨 Qualifier 复制。调用方会选择哪个签名？第 7 条因需要 conv 闭包而更冗余，但功能上不互斥。设计文档未说明两者的调用优先级或冗余关系。

**所在位置**：§3.3 第 7 条 vs 第 6a 条

**严重程度**：一般

**改进建议**：明确说明当源矩阵与目标矩阵元素类型相同且尺寸相同时，第 7 条的 fromMat 也可使用但冗余（因需提供不必要的 conv 闭包如 `{x: T => x}`）；推荐调用方优先使用第 6a 条（无需 conv），与已有的"6a/6b 调用指导"风格一致

---

### 问题 6：outerProduct 签名的参数语义与 C++ GLM 不一致

**问题描述**：§3.7 matrix.cj 中 outerProduct 的签名为 `outerProduct(c: VecN<T,Q>, r: VecM<T,Q>)`，参数名 `c` 表示列向量、`r` 表示行向量。C++ GLM 中 `outerProduct` 的参数定义为 `outerProduct(const vec<C, T, Q>& c, const vec<R, T, Q>& r)`（见 matrix.hpp:382-398），其语义是 c 的每个分量作为结果矩阵一列的缩放系数、r 的每个分量作为结果矩阵一行的缩放系数——结果矩阵 M 的第 j 列 = c * r[j]。但如果按设计文档中给出的 `outerProduct(c: Vec2, r: Vec3)` 返回类型为 Mat3x2（3 列 2 行），这意味着 c 为 Vec2（2 分量 = 列数=3？，行数=2）——这与 GLM 定义不一致。GLM 中 `outerProduct(vec2, vec3)` 返回 `mat<3, 2, T, Q>`（3 列 2 行），其中 vec2 的分量数 = 行数 = 2，vec3 的分量数 = 列数 = 3。设计文档标注 `c: Vec2, r: Vec3 -> Mat3x2`，c 分量数 = 2 = 行数，r 分量数 = 3 = 列数，语义实际与 GLM 一致（c 为列缩放向量 = 行数维向量，r 为行缩放向量 = 列数维向量）。但文档注释中 `c` 被命名为"列向量"可能误导编码者认为 c 分量数 = 列数，实际 c 分量数 = 行数。此命名歧义直接影响编码正确性。

**所在位置**：§3.7 matrix.cj — outerProduct 签名列表

**严重程度**：一般

**改进建议**：在设计文档中明确 outerProduct 的参数语义：第一个参数 c 的分量数 = 结果矩阵行数 R，第二个参数 r 的分量数 = 结果矩阵列数 C；结果 M[j][i] = c[i] * r[j]。补充说明 GLM 的 outerProduct 参数命名中 c（column-like）指的是"用于构建列"的向量分量数等于行数，而非"列维度"

---

### 问题 7：C++ GLM 中非方阵 `++`/`--` 运算符未在差异声明中记录其覆盖范围

**问题描述**：设计文档 §9 差异声明和 D08 仅笼统说明"排除 ++/-- 运算符"，但 C++ GLM 在全部 9 种矩阵类型（含 6 种非方阵）上均定义了 `++`/`--` 运算符（已通过查阅 .hpp 和 .inl 源码确认——如 type_mat2x3.hpp:93-96 和 type_mat2x3.inl:301-330）。当前差异声明仅提及"仓颉不支持重载自增/自减运算符"及等价替代 `m += T(1)`，但未显式说明此缺失影响全部 9 种矩阵类型（包括非方阵），也未说明非方阵 `++`/`--` 在 C++ GLM 中的语义同样是逐元素加/减 1。差异声明的覆盖完整性不足。

**所在位置**：§9 差异声明"无 ++/-- 运算符"条目；D08

**严重程度**：轻微

**改进建议**：在 §9 差异声明中明确标注"全部 9 种矩阵类型（含 6 种非方阵）在 C++ GLM 中均提供 ++/-- 运算符，仓颉中均不可用"；在 D08 中补充非方阵 `++`/`--` 也排除的说明

---

### 问题 8：`common.cj` stub 中的 `mix`/`step`/`smoothstep` 签名仅支持标量，但 C++ GLM 中这些函数同时提供标量和向量版本

**问题描述**：§3.7 common.cj 的完整函数签名清单仅列出标量版本（如 `func mix<T>(a: T, b: T, t: T): T`），但 C++ GLM 中 `common.hpp` 的 `mix`、`step`、`smoothstep` 等函数同时提供标量版本和向量版本（如 `mix(vec3, vec3, vec3)` 等）。虽然这些向量版本可能在后续阶段随 common.cj 完整实现时添加，但设计文档未说明当前 stub 范围是否仅含标量版本，也未说明向量版本推迟至阶段四的决策。编码者无法确定 common.cj stub 的完整边界。

**所在位置**：§3.7 common.cj 完整函数签名清单

**严重程度**：一般

**改进建议**：明确说明 common.cj stub 仅包含标量版本函数，向量版本的 mix/step/smoothstep 等推迟至阶段四随 common.cj 完整实现时添加；或在签名清单中补充向量版本 stub 签名（若方阵 .inl 依赖闭合需要向量版 mix 等函数）

---

### 问题 9：`geometric.cj` stub 中的 `dot` 仅有 Vec2/Vec3/Vec4 版本，缺少 Vec1 版本

**问题描述**：§3.7 geometric.cj 完整函数签名清单中 `dot` 函数仅有 Vec2、Vec3、Vec4 三个版本，但阶段一的 `type_vec1.cj` 中 Vec1 类型也存在 `.inl` 中可能调用 `dot(vec1, vec1)` 的场景。此外 C++ GLM 中 `func_geometric.hpp` 确实为所有分量数（1~4）均提供了 `dot` 函数。遗漏 Vec1 版本的 dot 可能导致编译依赖闭合不完整（若后续有类型依赖 Vec1 的几何运算）。

**所在位置**：§3.7 geometric.cj — dot 函数签名清单

**严重程度**：轻微

**改进建议**：在 geometric.cj stub 中补充 `func dot<T, Q>(x: Vec1<T, Q>, y: Vec1<T, Q>): T` 签名；或在文档中显式说明阶段二的 geometric.cj stub 仅覆盖方阵 .inl 实际引用的函数签名（type_mat4x4.inl 仅调用 Vec2/Vec3/Vec4 版本的 dot），Vec1 版本推迟至阶段四

---

### 问题 10：fwd.cj 矩阵别名范围（Mat{C}x{R} 系列 + DMat{C}x{R} 系列 + 精度变体 + 方阵短别名）与 C++ GLM fwd.hpp 的命名对应完整性未通过详尽对照确认

**问题描述**：设计文档 §8 给出的 fwd.cj 矩阵别名命名规则包含 Mat2x2~Mat4x4、DMat2x2~DMat4x4、HighpMat{C}x{R} 等精度变体、mat2~mat4 方阵短别名及精度变体、mat2x3~mat4x3 非方阵短别名及 dmat2x3~dmat4x3 等列出。但 C++ GLM 的 fwd.hpp 中还包含 `fmat`/`f32mat`/`f64mat` 系列的**非方阵全尺寸精度变体**（如 `lowp_fmat2x3`、`mediump_f32mat4x2`、`highp_f64mat3x4` 等），设计文档虽通过 D19 排除了 fmat/f32mat/f64mat 系列，但排除的理由为"显式前缀冗余别名"——实际上 fmat 系列的精度变体（lowp_fmat2x3 等）虽与 lowp_Mat2x3 指向相同类型，但在 C++ GLM 中的排列模式为独立别名族。当前排除决策是合理的（避免别名爆炸），但排除决策中未说明 fmat/f32mat/f64mat 系列的**非方阵精度变体**数量（比仅方阵精度变体多出 6×3=18 个/族），导致排除理由的"爆炸"程度论据不够充分。

**所在位置**：D19；§8 fwd.cj 关于此排除决策的说明

**严重程度**：轻微

**改进建议**：在 D19 中补充 fmat/f32mat/f64mat 系列非方阵精度变体的具体数量（每族 9×3=27 个别名，3 族共 81 个），以强化"冗余别名导致爆炸"的排除理由数值佐证

## 整体评价

本设计方案经过 9 轮迭代，在技术可行性方面已相当成熟：T(0)/T(1) 获取策略、构造函数降级方案、extend 块约束设计等关键仓颉语言限制均有系统性解决。构造函数 const init/const func 可用性汇总表、复合赋值运算符分类表格、stub 函数完整签名清单等细节已达到可指导编码实现的粒度。

本次审查发现的核心严重问题为跨尺寸乘法兼容性表遗漏了 2 个合法组合（Mat3x3*Mat2x3 和 Mat4x3*Mat2x4），将直接影响编码阶段乘法运算符重载的完整性。其余问题主要为设计细节的边界补充和文档逻辑精确性，不影响整体架构正确性。
