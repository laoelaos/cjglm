# OOD 设计方案审查报告（v10）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** 9 个矩阵类型采用独立泛型结构体 `Mat{C}x{R}<T, Q> where Q <: Qualifier`，与阶段一 Vec 类型的设计策略一致。仓颉不支持模板偏特化，独立类型各自声明列向量成员和运算符签名是唯一可行路径（D01）。

**[通过]** 矩阵类型以 `Vec{R}<T,Q>` 为列向量数据成员（值类型嵌套值类型），列向量类型由矩阵尺寸唯一确定。仓颉 struct 成员变量可为泛型类型实例，此模式与阶段一 `Vec2<T,Q>` 含 `T` 成员一致。

**[通过]** 运算符重载定义在带 `Number<T>` 约束的 extend 块中——仓颉的 extend 机制支持带 `where` 约束的泛型扩展（已查阅 extend 文档 §2.4），运算符重载可在 extend 块中添加（extend 文档 §2.2 确认 `operator func` 可在扩展中定义）。

**[通过]** 复合赋值运算符的自动生成规则：仓颉编译器在二元运算符返回类型与左操作数类型匹配时自动生成复合赋值版本（function 文档 §8.5）。设计按 4 类表格严谨分析自动生成条件（① 同尺寸 ±= ② 标量复合赋值 ③ 方阵 *= ④ 方阵 /=），与仓颉规则匹配。

**[通过]** `mut operator func [](i: Int64, value!: VecR<T,Q>): Unit` 用于列赋值——仓颉 struct 的 `mut` 函数可在 extend 块中定义（extend 文档 §4.3 确认），`[]` 赋值形式需 `mut` 修饰符（设计文档明确标注，与阶段一 Vec2 实际实现匹配：`type_vec2.cj:37` 已有 `mut operator func []`）。

**[通过]** 跨尺寸矩阵乘法运算符定义在左操作数矩阵类型的 extend 块中，9 个矩阵类型同属 `glm.detail` 包，同包类型互相直接可见无需跨文件 import——与仓颉包可见性规则一致（package 文档 §5.1 确认 `internal` 对包及子包可见）。

**[通过]** `fromMat` 工厂函数拆分为 6a（同类型，无 conv 闭包）和 6b（跨类型，带 conv 闭包）两个独立签名——消除了 SrcT≠T 时类型不匹配的编译难题，且 6a 签名利用 `SrcQ` 类型参数支持跨 Qualifier 转换，签名推导与仓颉泛型约束系统匹配。

**[通过]** `identity(one: T)`, `filled(scalar: T)`, `fromMat(m, one: T)` 等工厂函数需要 `Number<T>` 约束才能访问减法运算获得 T(0)。这些函数定义在 `extend<T, Q> Mat{C}x{R}<T, Q> where T <: Number<T>, Q <: Qualifier` 中——仓颉 extend 支持 `where` 约束附加（已确认），且 `Number<T>` 提供减法运算符。

**[通过]** `const init` 可用性分析：逐分量构造和列向量构造仅涉及纯赋值，满足 `const init` 要求（const 文档 §4.2 确认：const init 内可使用赋值表达式）；涉及运算的工厂函数（identity/filled/fromMat）不可为 const——与仓颉 const 表达式规则一致（const 文档 §2 第 7 条仅允许数值类型算术表达式为 const 表达式，泛型约束运算不在其列）。

**[通过]** `@Derive[Hashable]` 自动派生——矩阵类型所有成员为 Vec 列向量（阶段一已通过 @Derive[Hashable] 实现哈希），满足仓颉 struct 的 @Derive[Hashable] 要求（所有成员可哈希）。

**[通过]** `scalar_mat_ops.cj` 与 `scalar_vec_ops.cj` 同包同名函数通过参数类型重载解析区分——仓颉函数重载规则确认参数类型不同即构成有效重载（function 文档 §7.1），且导入时的遮蔽规则中同签名才遮蔽、不同签名构成重载（package 文档 §4.3）。

**[通过]** `fwd.cj` 使用限定访问 `detail.Mat4x4<...>` 定义别名——仓颉类型别名（type 文档 §5.2 规则）仅限顶层定义，且 `fwd.cj` 中别名名（如 `Mat4x4`）与 `detail.Mat4x4` 同名，需限定访问消除歧义。`ext/` 文件中别名名（如 `mat4`）与 `detail.Mat4x4` 不同名，可用命名导入。

### 2. 标准库与生态覆盖

**[通过]** 矩阵类型需要的核心能力为算术运算（`Number<T>` 接口）、相等比较（`Equatable<T>`）、序比较（`Comparable<T>`）——均在仓颉 `std.math` 标准库中定义，与阶段一 Vec 类型使用相同接口。

**[通过]** stub 文件（common.cj / matrix.cj / geometric.cj）所需的函数均为标准数学/几何函数（abs/sign/floor/ceil/mix/step/transpose/determinant/inverse/dot/cross/normalize 等），这些函数的签名参数类型均为基本数值类型或矩阵/向量类型——不依赖外部生态。

**[通过]** `@OverflowWrapping` 注解来自仓颉内置注解体系（reflect_and_annotation 文档 §1.1 确认三种溢出策略注解均为内置），与阶段一 scalar_vec_ops.cj 的使用方式一致。

**[通过]** `Assert` 和 `Exception` 为仓颉核心包（core）提供的内置功能，下标越界处理使用 `assert + fallback throw Exception` 与阶段一 Vec 实现一致（type_vec2.cj:29 确认）。

**[通过]** `cjpm.toml` 中 `src-dir` 配置及子包发现机制——设计中已包含编码启动前验证步骤，若 cjpm 不支持 `src/ext/` + `package glm.ext` 子包结构则需调整，此风险已识别并有备选方案。

### 3. 语言特性可行性

**[通过]** 错误处理策略：下标越界使用 `assert` + fallback `throw Exception`——仓颉 error_handle 文档确认 Exception 为运行时异常层次，assert 和 throw 均为合法语法。`[]` 和 `col()` 因含 assert/throw 不可声明为 const func——已在 §9 差异声明中记录。

**[通过]** 矩阵-矩阵除法（mat/mat）依赖 inverse（stub），本阶段运算符内部调用 stub 时抛出异常。矩阵-标量除法（mat/scalar）为逐元素操作不依赖 inverse，对所有 9 种矩阵类型均可用——已在 §3.5 第②类备注和 §5 错误处理策略中显式区分 /=scalar 和 /=(mat/mat) 的 stub 边界。

**[通过]** 并发设计：矩阵类型为值类型 struct，所有运算符返回新实例，天然线程安全，无需额外同步机制。

**[通过]** 资源管理：矩阵类型仅含 Vec 列向量值类型成员，无堆分配或资源持有，无需显式资源管理。

**[通过]** 包组织结构：`glm.detail` 包含全部 9 个矩阵泛型结构体定义和运算符重载，`glm` 包通过类型别名暴露常用具现化，`glm.ext` 包存放具现化别名文件——三种包声明与目录路径匹配（package 文档 §2.1 确认包名须与相对于 src/ 的目录路径匹配）。

**[通过]** `fwd.cj` 使用 `import glm.detail` 限定访问 + `detail.Mat4x4<...>` 别名定义，`ext/` 文件使用 `import glm.detail.{ Mat2x2, ... }` 命名导入——两种风格均符合仓颉 import 规则，设计文档已分析两者命名风格的原因和差异。

**[通过]** 运算符重载限制确认：仓颉不支持重载 `++`/`--`（function 文档 §8.2 可重载运算符列表中不包含 ++/--），设计以 `m += T(1)` 等价替代并在 §9 差异声明中记录——对所有 9 种矩阵类型均适用。

### 4. 设计一致性

**[通过]** 各抽象职责描述清晰无歧义。矩阵结构体承载列向量数据、构造函数/工厂函数体系、下标/列访问、算术运算符——职责边界分明。

**[通过]** 协作关系形成闭环：9 个矩阵类型 → Vec 列/行向量类型，方阵 .inl 编排 → matrix.cj / common.cj / geometric.cj stub 依赖闭合，scalar_mat_ops.cj → 9 个矩阵类型，fwd.cj / lib.cj → glm.detail 类型，ext/ 别名文件 → glm.detail 类型。所有依赖均有明确归属文件。

**[通过]** 行为契约描述完整：§4 提供了 8 个关键行为契约示例覆盖 identity/filled 构造、矩阵乘法、跨尺寸转换、跨类型转换、分量级修改、ext/ 别名使用、标量-矩阵除法，足以指导后续实现。

**[通过]** 模块间依赖方向合理无循环：`glm.detail` ← `glm` ← `glm.ext`，三个包形成单向依赖链，无循环。glm 与 glm.ext 同级且均依赖 glm.detail 相互无依赖。

**[通过]** 构造函数/工厂函数体系的 const init 可用性汇总表（§3.3 末尾表格）完整覆盖全部 8 类构造方式的 const 可用性判断——每类均有明确理由，与仓颉 const 规则一致。

**[通过]** 跨矩阵类型转换 6a 与第 7 条的功能重叠已显式说明：当 U=T 且尺寸相同时推荐优先使用 6a 版本，第 7 条仅用于 U≠T 场景。调用指导清晰可行。

**[通过]** fromMat 6a 中 `one: T` 在尺寸全面缩减方向的冗余性已分析：保留统一签名避免重载膨胀（72→144），并在文档中标注此冗余场景。评估合理。

**[通过]** 复合赋值运算符按 4 类表格拆分（① 同尺寸矩阵 ±= ② 标量复合赋值 ③ 方阵 *= ④ 方阵 /=），每类标注适用矩阵范围、自动生成条件和 stub 状态——第②类明确 /=scalar 逐元素操作非 stub，第④类明确 /=(mat/mat) 自动继承 stub 行为，两者边界清晰。

**[通过]** outerProduct 参数语义说明完整：第一个参数 c 的分量数=结果矩阵行数 R，第二个参数 r 的分量数=结果矩阵列数 C；结果 M[j][i]=c[i]*r[j]；c 来自 GLM outerProduct(col, row) 命名中 column-like 指"用于构建列"而非"列向量"——与 GLM 参考实现一致（func_matrix.inl:35 确认 `m[i] = c * r[i]` 中 c 是列缩放因子、r[i] 是列权重）。

**[通过]** common.cj stub 仅包含标量版本的决策已说明：当前方阵 .inl 仅引用标量版本，向量版本推迟至阶段四——合理且依赖闭合需求已满足。

**[通过]** geometric.cj stub 包含 Vec1 版本 dot 签名的决策已说明：虽当前方阵 .inl 未直接引用，但为保持签名字母序完整性一并提供——合理且不增加运行时代码量（stub 均为空壳）。

**[通过]** D19 排除 fmat/f32mat/f64mat 及整型矩阵别名的数值佐证已补充：每族 9 个矩阵尺寸 × 3 种精度前缀 = 27 个别名/族，3 族共 81 个；加上 i8mat~u64mat 等 12 种整型前缀 × 9 种矩阵尺寸 = 108 个；合计约 189 个冗余别名——排除理由充分。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：每个矩阵类型仅承载自身尺寸的列向量数据和运算符定义；scalar_mat_ops.cj 专门处理标量左侧运算；3 个 stub 文件各自对应一个函数库的依赖闭合。无职责越界。

**[通过]** 抽象层次恰当：设计为架构级 OOD，不包含具体字段偏移、完整方法实现体等实现细节，但提供了足够的行为契约、类型关系和接口签名供后续详细设计使用。构造函数体系的 8 类构造方式覆盖了 C++ GLM 中所有构造路径的仓颉映射方案。

**[通过]** 设计便于后续详细设计和实现：每个 extend 块中的静态函数签名、运算符签名均已明确参数类型和约束条件；72 个 fromMat 的编译可行性已评估并建议可拆分为多个 extend 块；编码阶段注意事项已标注须逐一对照 GLM .inl 源码实现各转换工厂函数。

**[通过]** 设计便于单元测试：矩阵类型为值类型 struct，所有运算符返回新实例，天然可隔离测试；工厂函数签名清晰可验证输入输出；stub 函数的错误行为（throw Exception）可测试；别名类型的透明性可通过构造函数和运算符可用性验证；行访问推迟至阶段四避免 API 不兼容。

**[通过]** 关键行为验证示例提供了跨尺寸乘法兼容性的穷举验证（9×9 = 81 种配对，C1=R2 合法组合 29 个含同尺寸 3 个，跨尺寸 26 个），按左操作数逐项列举跨尺寸数并求和验证（2+3+3+3+2+3+3+3+2=26）——穷举方法严谨，结论可复核。

**[通过]** 差异声明覆盖完整：§9 列出 18 项差异声明，覆盖无 C 数组成员、无条件编译、无 ++/--、ext/ 包名差异、矩阵除法推迟、标量左侧全局函数、mat/scalar 成员运算符形态差异、无类型别名体系、无 row() 成员函数、col() 替代 componentAt、[] 取值副本不做链式修改、fwd.cj 排除冗余别名、非方阵无 *=、复合赋值不支持跨类型、方阵 /= 自动继承 stub、无 init() 默认构造、identity 非方阵语义、filled 工厂函数、fromMat 拆分、fromParts/fromColumns 单一源类型、fromMat 同尺寸转换、[]/col() 非 const、identity/filled 非 const——每项差异均有对应设计决策（D01~D31）支撑。
