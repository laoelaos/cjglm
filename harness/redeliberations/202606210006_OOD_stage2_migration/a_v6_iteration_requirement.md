## 当前审查结果

### P1（严重）：`init()` 默认单位矩阵构造函数在无约束泛型 `T` 下依赖 `T(0)/T(1)` 不可行

**问题描述**：§3.3 第 1 条设计默认单位矩阵构造 `init()`，需要在无约束泛型 `T` 上构造 `T(0)` 和 `T(1)`。但仓颉泛型约束规则明确："无约束时只能传递/返回泛型类型的值，不能进行其他操作"——`T(0)/T(1)` 属于构造调用，无约束泛型不支持。阶段一 Vec 类型没有默认 `init()` 构造函数（`type_vec2.cj:11` 起始构造函数为 `init(scalar: T)`），回避了此问题。矩阵的 `init()` 是全新需求——C++ GLM 的 `mat4()` 默认构造为单位矩阵，但此构造函数在无约束泛型 struct 定义中不可行。§3.3 第 1 条的降级方案"若 const init 不可用，退化为普通 init 构造函数"误导性更强——降级原因是 `T(0)/T(1)` 在无约束泛型下不可用，与 `const` 上下文无关。普通 `init()` 同样无法在无约束下构造 `T(0)/T(1)`。

**所在位置**：§3.3 构造函数体系第 1 条

**严重程度**：严重（编码落地的直接阻塞项）

**改进建议**：将 `init()` 默认构造函数重写为以下方案之一（推荐方案 a）：
- (a) 将单位矩阵构造推迟到 `Number<T>` 约束的 extend 块中，作为 `public static func identity(): Mat4x4<T,Q>` 工厂函数。代价：`Mat4x4<Float32, PackedHighp>()` 语法不可用，改为 `Mat4x4<Float32, PackedHighp>.identity()`——但可在 extend 块中利用 `Number<T>` 约束构造 `T(0)/T(1)`（实际实现可能需要利用整数运算 `T(1)` 演算技巧，与阶段一 `increment()` 使用 `-( !(this.x - this.x) )` 演算出 `T(1)` 的模式一致）
- (b) `init()` 增加 `zero: T, one: T` 参数：`init(zero: T, one: T)`——与 `fromBoolVec` 的 `zero: T, one: T` 模式一致（DV-01），但破坏了 `Mat4x4()` 无参构造的惯用法
- (c) 将默认构造改为零矩阵（`init()` 全填零）而不是单位矩阵：这可在 extend 块中用 `Mat4x4(this - this)` 技巧实现零值——但与 C++ GLM 的 `mat4()` 默认构造为单位矩阵的语义矛盾

无论选何种方案，需：① 删除"const init 降级为普通 init"的描述，改为对 `T(0)/T(1)` 约束条件的明确声明；② 在 §9 差异声明中记录此偏差；③ 显式引用 deviations.md DV-01 的同类限制

### P2（一般）：§3.3 第 2 条与第 9 条构造函数在 `U=T` 时存在重载歧义

**问题描述**：§3.3 第 2 条定义 `init<Q2>(m: Mat4x4<T, Q2>)`（跨 Qualifier 复制），第 9 条定义 `init<U, P>(m: Mat4x4<U, P>)`（跨类型同尺寸转换）。当第 9 条实例化为 `U=T, P≠Q` 时，参数类型为 `Mat4x4<T, P>`，与第 2 条实例化为 `Q2=P` 时的参数类型 `Mat4x4<T, P>` 完全相同。两个构造函数的泛型参数具体程度无差别，仓颉编译器重载解析极可能报歧义错误。

**所在位置**：§3.3 构造函数体系第 2 条 vs 第 9 条

**严重程度**：一般

**改进建议**：删除第 2 条，将其功能完全由第 9 条覆盖——当 `U=T` 时即自然包含跨 Qualifier 复制。此方案最简洁，且第 4 轮审查 P3 已提出此建议，产出方尚未回应

### P3（轻微）：§2 模块清单中 `type_mat4x3.cj` 条目标注的类型错误

**问题描述**：§2 包组织清单第 43 行写 `type_mat4x3.cj ★ — Mat4x4<T,Q>`，应为 `Mat4x3<T,Q>`。上下文中第 42 行 `type_mat4x2.cj — Mat4x2<T,Q>` 和第 44 行 `type_mat4x4.cj — Mat4x4<T,Q>` 均正确，此为笔误。

**所在位置**：§2 包组织清单第 43 行

**严重程度**：轻微

**改进建议**：将 `Mat4x4<T,Q>` 修正为 `Mat4x3<T,Q>`

### P4（一般）：ext/ 向量别名与 fwd.cj 向量别名的定位差异和使用指引历经 3 轮未获回应

**问题描述**：§3.8 ext/ 别名文件中定义了向量别名（如 `vector_float4.cj` 中的 `vec4 = Vec4<Float32, PackedHighp>`），而 fwd.cj 中已定义了 `Vec4 = detail.Vec4<Float32, detail.PackedHighp>`（PascalCase，参见 `cjglm/src/fwd.cj:107`）。两处对 Float32 highp 的 Vec4 提供了不同命名风格的冗余别名。设计文档未说明：① 为什么需要两套别名；② 用户应优先使用哪个；③ 两套别名的差异仅在于命名风格（PascalCase vs camelCase）还是包路径不同。此问题从第 3 轮审查起连续 3 轮被指出，产出方始终未回应。

**所在位置**：§3.8 ext/ 别名文件 — 向量别名文件

**严重程度**：一般

**改进建议**：在 §3.8 中补充说明：
- ext/ 下的 camelCase 别名（vec1~vec4/dvec1~dvec4）与 fwd.cj 中的 PascalCase 别名（Vec1~Vec4/DVec1~DVec4）是同一类型的冗余别名，对应 GLM 中 ext/vector_float*.hpp vs fwd.hpp 的二元体系
- 推荐用户优先使用 fwd.cj 中的别名（import 路径更短：`import glm.{Vec4}`），ext/ 别名主要用于与 GLM C++ 代码做逐文件对应迁移场景

### P5（一般）：跨矩阵类型转换构造的填充规则与 GLM Mat4x4 ← Mat4x2 行为不一致

**问题描述**：§3.3 第 8 条定义填充规则为"源矩阵 M_src[j][i] 复制到目标矩阵的相同坐标位置，范围 0 ≤ j < min(C_src, C_dst)、0 ≤ i < min(R_src, R_dst)，额外行列从单位矩阵取值"。此规则在 Mat4x4 ← Mat4x2 场景下与 GLM 行为不一致：GLM 在 Mat4x4 ← Mat4x2 中第 2、3 列完全丢弃源数据替换为单位矩阵列，而设计规则会复制 src 第 2、3 列的部分数据后再补齐。但 Mat4x4 ← Mat3x2 与设计规则一致（经上一轮质询确认）。GLM 在不同目标矩阵类型中行为不一致（Mat4x3 ← Mat4x2 保留源数据 vs Mat4x4 ← Mat4x2 丢弃源数据），说明 GLM 的转换语义是逐类型在 .inl 中硬编码的，而非按统一规则实现。

**所在位置**：§3.3 构造函数体系第 8 条

**严重程度**：一般

**改进建议**：
1. 在 §3.3 第 8 条中添加注释：编码阶段须逐一对照 GLM `.inl` 源码实现各转换构造函数，当前设计规则作为主规则描述，但存在特例（特别是当 C_src > C_dst 且 R_src < R_dst 时，部分 `.inl` 的逐列构造行为可能偏离主规则）
2. 若 GLM 参考源码在本项目可访问，建议在编码阶段将 72 个转换构造函数逐一与 `.inl` 对照验证；若参考源码不可直接访问，将此作为编码阶段的风险项记录
3. 避免提出过于笼统的替代规则，改为承认 GLM 行为因目标类型而异的复杂性，由编码阶段按 `.inl` 逐一实现

### P6（轻微）：`[]` 和 `col()` 的越界保护机制不支持 `const func`，但未在设计文档和差异声明中记录

**问题描述**：§3.4 和 §5 规定下标越界时"抛出 Exception（含 assert + fallback throw）"。阶段一实际实现已表明 `assert` 和 `throw` 不是 `const` 函数（deviations.md IF-03 记录了 `componentAt` 因 assert/throw 无法声明为 const 的限制）。矩阵的 `[]` 运算符是否需要 `const` 版本？C++ GLM 中 `operator[]` 在 constexpr 上下文可用。deviations.md IF-03 仅为 Vec 的 `componentAt` 记录了此限制，矩阵的 `[]` 同类限制尚未记录。

**所在位置**：§3.4 行列访问；§9 差异声明

**严重程度**：轻微

**改进建议**：在 §9 差异声明新增条目：矩阵 `[]` 和 `col()` 不可在 `const` 上下文使用，与阶段一 `componentAt` 面临的限制一致（IF-03）。C++ GLM 中 `m[i]` 在 constexpr 上下文可用，仓颉版本因 assert/throw 机制无法声明为 `const func`
