## 逐维度审查

### 1. 类型系统可行性

**[严重]** 类型别名无法定义在结构体内部 — 设计第 3.2 节提出"每个矩阵结构体内部定义以下 `public type` 别名"，但仓颉语言明确规定"只能在源文件顶层定义类型别名"（参见 `typealias.md`：`type aliases can only be defined at the top level of the source file`）。阶段一 Vec 类型的现有实现（`type_vec2.cj`、`type_vec4.cj` 等）也全部未在结构体内部定义类型别名，侧面印证此限制。此问题影响全部 9 个矩阵类型的设计可行性。

**[通过]** 设计其余类型形态选择正确：泛型 struct（矩阵为值对象，值语义）、泛型参数 `<T, Q>` + `where Q <: Qualifier` 约束、9 个独立类型替代不支持的模板偏特化，均与仓颉类型系统能力匹配。

**[通过]** 运算符重载通过 `extend` 块实现，使用 `Number<T>`、`Equatable<T>`、`Comparable<T>` 等约束条件，模式与阶段一 Vec 类型一致，可行。

**[通过]** `const init` 构造函数的设计采用了正确的语法形式。设计已标注实现风险（列向量构造表达式链在 `const init` 中的可行性），态度恰当。

### 2. 标准库与生态覆盖

**[通过]** 依赖的 `Number<T>`、`Integer<T>`、`Equatable<T>`、`Comparable<T>` 来自 `std.math` 和 std core 核心包，均为标准库提供。

**[通过]** `@OverflowWrapping` 注解是仓颉内置溢出策略注解，可用。

**[通过]** 涉及的基础数学函数（`min`、`max`、`abs`、`fmod` 等）在 `std.math` 中可用。

**[通过]** `ComputeEqual<T>`、`ComputeEqualNumeric<T>` 的引用方式与阶段一 Vec 实现一致，合理。

### 3. 语言特性可行性

**[通过]** 错误处理策略（下标越界抛 `Exception`、`assert` + fallback throw）与仓颉错误处理能力匹配。

**[通过]** 并发设计：结构体值类型天然线程安全，无需额外同步机制。

**[通过]** 包组织符合 cjpm 规范：`glm.detail`（`src/detail/` 子目录）、`glm`（`src/` 根目录）、`glm.ext`（`src/ext/` 子目录），目录路径与包名匹配。

**[通过]** 运算符重载规则利用正确：`operator func` 的 `this` 固定为左操作数，`T + Mat` 无法通过运算符表达，设计采用全局函数（`scalar_mat_ops.cj`）处理标量左侧运算，与阶段一 `scalar_vec_ops.cj` 模式一致。

**[通过]** 复合赋值运算符由编译器自动生成（条件：二元运算符返回类型与左操作数类型匹配），文档已确认此行为。

**[轻微]** 若将类型别名移至文件顶层，泛型别名不能使用 `where` 约束。但别名仅做类型简写，实际约束由被别名类型（如 `Vec4<T,Q>`）的声明处保证，不影响功能。

### 4. 设计一致性

**[严重]** 第 3.2 节"类型别名体系"与第 1 节"核心抽象"中描述的类型别名在结构体内部的存放方式无法实现。此问题涉及设计中所有矩阵类型的静态结构定义，属核心设计缺陷。

**[通过]** 模块依赖方向合理：`glm.ext → glm.detail`、`glm → glm.detail`，形成有向无环图，无循环依赖。

**[通过]** 职责划分清晰，每个矩阵类型独立负责自身的列向量成员和运算，协作关系形成闭环。

**[通过]** 构造函数体系、运算符体系、行列访问等行为契约描述完整，足以指导后续实现。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：每个矩阵类型只表示一种尺寸的数学矩阵，运算逻辑通过 `extend` 块按约束条件分组。

**[通过]** 抽象层次恰当，不包含实现细节（如字段偏移、内联展开策略等），属于架构级设计范畴。

**[通过]** 使用了与阶段一 Vec 类型一致的设计模式（独立泛型结构体、extend 块运算符、全局标量运算函数），降低了学习成本和维护负担。

**[通过]** 矩阵为值类型且运算返回新实例，便于单元测试和 mock 隔离。

## 修改要求（REJECTED 时存在）

### 问题 1：类型别名无法定义在结构体内部

- **问题**：设计第 3.2 节要求在矩阵结构体内部定义 `public type col_type`、`public type row_type` 等类型别名。仓颉语言规定类型别名只能在源文件顶层定义，不能在结构体/类/函数体等嵌套作用域内定义。
- **原因**：此限制直接影响全部 9 个矩阵类型的类型成员体系，设计不可直接实现。
- **建议方向**：
  - **Option A**：将类型别名移至文件顶层（`struct` 定义之前）。例如：`public type Mat4x4ColType<T, Q> = Vec4<T, Q> where Q <: Qualifier`。注意泛型类型别名不支持 `where` 约束，但约束会在使用别名时代替 `Vec4<T,Q>` 时由 `Vec4` 自身的声明处强制执行。
  - **Option B**：移除全部类型别名，在使用处直接引用具体的 Vec 类型（如 `Vec4<T, Q>`、`Vec3<T, Q>`）。这是阶段一 Vec 类型的做法，最简单直接。
  - **Option C**：使用 `public import` 在包层面重新导出 Vec 类型组合，但此方案不如前两者清晰。
  - 推荐 **Option B**，与阶段一实现风格一致，避免维护额外别名。若保留 `col_type` 等别名对 GLM 兼容性重要，选 **Option A**。
