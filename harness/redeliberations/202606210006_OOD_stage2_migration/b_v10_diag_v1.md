## 审查发现

### 问题 1：`fromMat` 6a 与 6b 在 `SrcT=T` 且同尺寸时存在重载歧义，文档未给出编译器层面的消歧方案

**所在位置**：§3.3 第 6a 条 vs 第 6b 条（第 264 行附近）

**问题描述**：
文档在第 6a 条末尾提供了"调用指导"：当 `SrcT=T` 时应优先使用 6a 版本。但 6a 签名为 `fromMat<SrcQ>(m: Mat2x2<T, SrcQ>, one: T)`，6b 签名为 `fromMat<SrcT, SrcQ>(conv: (SrcT)->T, m: Mat2x2<SrcT, SrcQ>, one: T)`。当 `SrcT=T` 时，两个签名参数个数不同（6a 有 2 个参数，6b 有 3 个参数），按仓颉重载解析规则参数个数不同即可区分，不存在歧义。然而文档关于此问题的表述方式——"调用指导"而非"编译器自动选择"——可能误导编码者认为编译器无法区分两者。实际上编译器按参数数量即可消歧，无需调用方干预。

**严重程度**：轻微

**改进建议**：将"调用指导"重写为"编译器消歧说明"——明确指出 6a 和 6b 参数个数不同，编译器按参数数量自动选择，无需调用方显式指定优先级；"调用指导"仅作为风格建议（6a 更简洁）而非编译必要性。

### 问题 2：`fwd.cj` 中 `Mat{C}x{R}` 与 `detail.Mat{C}x{R}<T, Q>` 别名遮蔽问题未评估

**所在位置**：§8 修改文件 — fwd.cj 条目（第 765~772 行）

**问题描述**：
fwd.cj 中的矩阵别名命名规则为 `public type Mat2x2 = detail.Mat2x2<Float32, detail.PackedHighp>`。与阶段一向量别名一致（如 `public type Vec4 = detail.Vec4<Float32, detail.PackedHighp>`），别名名 `Mat2x2` 与 detail 包中的泛型类型 `Mat2x2<T,Q>` 同名。阶段一 fwd.cj 使用 `detail.Mat4x4<...>` 限定访问而非命名导入来避免遮蔽（见 deviations.md INT-03）。本设计方案 §8 也提到使用 `detail.Mat4x4<...>` 限定访问，但**未评估矩阵别名数量（9 个全尺寸 × 4 精度族 = 约 60 个）对于 fw.cj 文件大小的影响**——阶段一 fwd.cj 已有 324 行纯向量别名，矩阵别名将使该文件膨胀至约 500+ 行。虽然这不是功能性问题，但大规模别名文件的维护性和可读性值得注意。

**严重程度**：轻微

**改进建议**：注明矩阵别名将使 fwd.cj 膨胀至约 500+ 行，建议编码阶段考虑别名生成脚本自动生成（阶段一已有此注释），和/或评估是否需要拆分 fwd.cj 为多个文件。

---

### 问题 3：`fromMat` 6b 中 `conv` 闭包与 `one: T` 参数在跨类型同尺寸转换中的冗余共存未分析

**所在位置**：§3.3 第 6b 条 vs 第 7 条（第 264 行附近 vs 第 287 行附近）

**问题描述**：
第 7 条 `fromMat<U, P>(m: Mat4x4<U, P>, conv: (U) -> T)` 用于跨类型同尺寸转换（无 `one` 参数）。当用户使用 6b 版本做跨类型跨尺寸转换且 `C_dst = C_src, R_dst = R_src` 时，`one` 参数不再需要（无扩展行列需填充），但仍存在于签名中。这与 6a 中 `one` 参数在尺寸缩减方向的冗余性是同一类问题，但文档仅分析了 6a 的此冗余，未分析 6b 中的类似场景。编码者可能对"6b 的 `one` 何时冗余"产生困惑。

**严重程度**：轻微

**改进建议**：在第 6b 条补充 `one` 参数冗余性分析：与 6a 相同——当 C_dst ≤ C_src 且 R_dst ≤ R_src 时 `one` 冗余（仅涉及 conv 转换源元素，无扩展填充），设计选择同样保留统一签名。

### 问题 4：`identity(one: T)` 的泛型推断可行性未充分验证——当 Q 未显式指定时编译器能否正确推断

**所在位置**：§3.3 第 8 条（第 295~303 行）

**问题描述**：
文档声称 `identity(one: T)` 中 "Q 从实例上下文推断或显式指定，T 从 `one` 参数推断"。以 `Mat4x4<Float32, PackedHighp>.identity(1.0f)` 为例：T=Float32 从 `one` 推断，Q=PackedHighp 从 `Mat4x4<Float32, PackedHighp>` 推断。这看起来可行。但考虑别名场景：`mat4.identity(1.0f)` — `mat4` 是 `type mat4 = Mat4x4<Float32, PackedHighp>`，类型别名在仓颉中是透明的，编译器应将 `mat4` 解析为 `Mat4x4<Float32, PackedHighp>`，推断行为一致。然而 **当 extend 块中 Q 是 extend 级别类型参数时**（`extend<T, Q> Mat4x4<T, Q> where T <: Number<T>, Q <: Qualifier`），静态函数 `identity(one: T)` 的 Q 是 extend 块的上下文类型参数，不是函数签名的类型参数——编译器需要从 `Mat4x4<Float32, PackedHighp>` 这个接收类型推断出 Q=PackedHighp。这是仓颉泛型 extend 块中静态函数的常规推断模式，与阶段一 Vec 的 `fromBoolVec(v, zero, one)` 中 Q 的推断方式相同（Vec 的 extend 块也声明了 `extend<T, Q> Vec2<T, Q> where...`）。鉴于阶段一已有成功编译的先例，此推断应可行。但文档中缺少对别名场景下推断的说明。

**严重程度**：轻微

**改进建议**：在 §3.3 第 8 条的完整泛型参数展开后补充一句：别名类型（如 mat4 = Mat4x4<Float32, PackedHighp>）上的 `identity(1.0f)` 调用推断行为与原始泛型类型一致（仓颉类型别名为透明别名）。

### 问题 5：`@Derive[Hashable]` 对矩阵类型的可行性未验证——列向量成员包含 Float32/Float64 时 Hashable 语义的正确性

**所在位置**：§3.1 矩阵结构体（第 181 行）；D17

**问题描述**：
D17 声称"矩阵类型包含 Vec 列向量成员，而 Vec 已通过 @Derive[Hashable] 自动派生哈希实现……仓颉中 struct 的 @Derive[Hashable] 要求所有成员可哈希，矩阵全部满足此条件"。这在语法上可行（Vec 已可哈希）。但**语义上**，Float32/Float64 的默认哈希实现将 NaN 映射为零值哈希（或特定值），而矩阵中可能包含 NaN 分量。C++ GLM 中矩阵的 `operator==` 对 NaN 矩阵返回 false，但哈希可能将不同 NaN 矩阵映射到同一桶——这是浮点哈希的通用问题。然而文档中未提及此语义局限，也未在 §9 差异声明中记录。这不是一个阻止编码的问题，但对下游消费者（如将矩阵存入 HashSet）有潜在影响。

**严重程度**：轻微

**改进建议**：在 D17 中补充说明：Float32/Float64 矩阵的 Hashable 实现对 NaN 分量的行为与 Vec 的 Hashable 一致——NaN 在哈希中可能产生冲突。若下游消费者需区分不同 NaN 模式的矩阵，应使用 `equalEpsilon` 而非 HashSet/HashMap。

### 问题 6：`scalar_mat_ops.cj` 中 `div` 函数对整型矩阵的除零行为未充分说明

**所在位置**：§3.5「二元运算符（标量-矩阵）」（第 368~375 行）；§5 错误处理策略

**问题描述**：
§5 错误处理策略中提到"整型矩阵除零风险"与阶段一 Vec 类型一致——仓颉整数除法由编译器/运行时处理。但设计文档中 `scalar_mat_ops.cj` 的 `div(s: T, m: Mat{C}x{R}<T,Q>)` 执行 `s / each_element`——当 s 是整型且某分量为零时，会产生整型除零异常。此行为与 C++ GLM 中 `operator/(T, mat)` 对整型矩阵的除零行为一致（C++ 整数除零为未定义行为）。仓颉的整型除零会抛出运行时异常而非未定义行为，此差异未在 §9 差异声明中记录。

**严重程度**：轻微

**改进建议**：在 §9 差异声明中补充：仓颉整型矩阵的标量-矩阵除法（`div(s, m)`）在除零时抛出运行时异常，而 C++ GLM 中整型除零为未定义行为。这是仓颉运行时安全性与 C++ 未定义行为的规范化差异。

### 问题 7：`fromColumns` 签名中 `Q2` 参数未被使用，与 6a 中 `SrcQ` 的分析不一致

**所在位置**：§3.3 第 5 条（第 243 行）

**问题描述**：
第 5 条 `fromColumns<V1, Q2>(conv: (V1) -> T, v1: Vec4<V1, Q2>, ...)` 中，Q2 是列向量的 Qualifier 类型参数。在 fromColumns 的实现中，Q2 仅出现在列向量参数的类型声明中，不参与运算逻辑——这与 6a 中 SrcQ 的情况完全相同。迭代第 7 轮已分析 6a 中 SrcQ 的作用和影响，但第 5 条的 Q2 未做同等分析。编码者可能对 Q2 是否导致不必要的重载膨胀产生疑问。

**严重程度**：轻微

**改进建议**：补充 Q2 参数的作用说明（与 6a 中 SrcQ 一致：允许接受不同 Qualifier 的列向量），并说明 Q2 不影响运行时行为，仅影响编译器泛型实例化。

---

### 问题 8：非方阵的 `filled(scalar: T)` 行为示例仅描述了 Mat2x3 一种情况，其他 5 种非方阵未给出示例

**所在位置**：§3.3 第 3 条（第 224~233 行）；§4.1 行为契约

**问题描述**：
文档中 `filled(scalar: T)` 的非方阵示例仅覆盖 Mat2x3（2×3 对角矩阵），其余 5 种非方阵（Mat2x4/Mat3x2/Mat3x4/Mat4x2/Mat4x3）的填充行为未给出具体示例。虽然主对角线规则（min(C,R)）已足够推导，但 6 种非方阵的主对角线长度和填充模式各不相同——特别是当 C > R 时（如 Mat3x2：3 列 2 行，主对角线长度 min(3,2)=2，意味着第 2 列的对角元素 (2,2) 超出行范围 R=2），编码者需要对每种非方阵逐一确认。提供一个 C > R 的示例（如 Mat3x2）即可消除歧义。

**严重程度**：轻微

**改进建议**：在 §4.1 补充一个 C > R 的非方阵 `filled` 示例（如 Mat3x2），与现有的 Mat2x3（C < R）示例形成互补覆盖。

---

### 问题 9：跨尺寸矩阵乘法编码时 `operator func *(rhs: Mat{C2}x{R2})` 的返回类型推断可行性未分析

**所在位置**：§3.5 跨尺寸乘法兼容性表及归属策略（第 380~395 行）

**问题描述**：
文档称跨尺寸矩阵乘法定义在左操作数的 extend 块中。以 `extend<T, Q> Mat2x3<T, Q> where T <: Number<T>, Q <: Qualifier` 为例，需要定义 `operator func *(rhs: Mat3x2<T, Q>): Mat3x3<T, Q>` 和 `operator func *(rhs: Mat4x2<T, Q>): Mat4x3<T, Q>` 等。这些运算符的右操作数类型（Mat3x2、Mat4x2）和返回类型（Mat3x3、Mat4x3）不在当前 extend 块的类型参数中——它们是同一包（glm.detail）中的独立类型。**同包类型互相直接可见，无需 import**——文档已正确指出此点。但文档未说明一个关键编码细节：右操作数类型中的 Q 与左操作数的 Q 是同一个 Q——跨尺寸乘法要求左右操作数的 Q 相同（同 C++ GLM 的 `mat<C1,R1,T,Q> * mat<C2,R2,T,Q>`）。文档中跨尺寸乘法兼容性表隐含了此约束（所有示例中左右 Q 一致），但未显式声明"跨尺寸乘法仅支持同 Q 操作数"。

**严重程度**：轻微

**改进建议**：在 §3.5 跨尺寸乘法兼容性表中补充说明：所有跨尺寸乘法要求左右操作数具有相同的 T 和 Q——不同 T 或不同 Q 的矩阵不可直接相乘（与 C++ GLM 中 `mat<C,R,T,Q>` 的 T 和 Q 必须匹配一致）。

### 问题 10：`outerProduct` 签名中参数 `c` 和 `r` 的 Qualifier 与结果矩阵的 Qualifier 关系未说明

**所在位置**：§3.7 matrix.cj — outerProduct 签名清单（第 496~508 行）

**问题描述**：
outerProduct 的所有 9 个重载签名中，参数 `c: VecN<T, Q>` 和 `r: VecN<T, Q>` 以及返回值 `Mat{C}x{R}<T, Q>` 使用同一个 Q。这隐含要求 c 和 r 的 Qualifier 必须相同。在 C++ GLM 中，`outerProduct(col, row)` 的两个向量参数也可有不同的 Q（此时结果矩阵的 Q 由编译器隐式选择）。本设计中统一 Q 的做法与 C++ GLM 有微小差异——当用户持有 `Vec3<Float32, PackedHighp>` 的 c 和 `Vec2<Float32, AlignedHighp>` 的 r 时，无法直接调用 `outerProduct(c, r)`。

**严重程度**：轻微

**改进建议**：明确声明 outerProduct 要求两个向量参数具有相同 Q，与 C++ GLM 允许跨 Q 的能力有微小差异。若此差异对用户有影响，可考虑提供一个 Q2 版本 `outerProduct<T, Q, Q2>(c: VecR<T,Q>, r: VecC<T,Q2>): Mat{C}x{R}<T,Q>`。

## 整体质量评价

经过 10 轮迭代审查后，当前设计方案的质量已达到较高水平。前序迭代中发现的所有严重问题（行向量×矩阵乘法签名错误、identity 中 T(1) 演算依赖不可用的 `!` 运算符、fromMat 跨类型转换缺少 conv 闭包、scalar/mat 对非方阵的遗漏、跨尺寸乘法兼容性表遗漏等）均已在本次版本中修正。当前版本的核心类型定义、运算符体系、构造函数/工厂函数设计、错误处理策略、差异声明等关键技术维度均坐落在可靠的基础上。

本轮审查发现的 10 个问题均为轻微级别——主要涉及文档完整性和编码指导精度（如参数冗余性分析的一致性、特定场景的示例补充、语义局限的说明等），不存在事实错误或逻辑矛盾。这些问题不影响设计方案的总体可行性，可在编码阶段的早期迭代中逐步补齐。

**无严重或一般级别问题**。设计方案可直接指导编码实现。
