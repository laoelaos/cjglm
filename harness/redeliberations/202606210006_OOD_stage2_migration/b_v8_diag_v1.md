## 问题清单

### 问题 1（严重）：`scalar / mat` 运算符对非方阵类型的遗漏与事实错误

**问题描述**：§3.5 和 §5 中声称 `scalar / mat` 除法运算"不依赖 inverse，可本阶段完整实现"但仅提及以 scalar_mat_ops.cj 全局函数形式提供，未明确该运算对所有 9 种矩阵类型均适用。然而更严重的问题在于：C++ GLM 参考实现中**所有 9 种矩阵类型**（含 6 种非方阵）均定义了 `operator/(T scalar, mat)`，例如 `type_mat2x3.hpp:146` 和 `type_mat2x3.inl:474-479` 明确声明并实现了 `T/mat` 除法运算符。而 §3.5 的 `scalar / mat` 描述仅笼统提及"通过 scalar_mat_ops.cj 中的全局具名函数处理"，未列出非方阵的 `scalar / mat` 也属于 36 个全局函数的覆盖范围。编码者可能误认为非方阵不需要 `scalar / mat` 运算符。

此外，§3.5 在"标量-矩阵除法语义"段落（第 362 行）仅描述了 div 的逐元素语义，但未清晰声明 36 个全局函数涵盖全部 9 种矩阵类型（包括 6 种非方阵）的标量除法。由于 §5 的分类中将 `scalar / mat` 与 `mat / scalar` 合并描述为"逐元素操作"，但 §3.5 中仅 `mat / scalar` 作为成员运算符出现，`scalar / mat` 仅作为全局函数提及——编码者需要明确知道 `scalar / mat` 的 9 个 div 重载已包含在 36 个全局函数中。

**所在位置**：§3.5「二元运算符（标量-矩阵）」；§5「错误处理策略」

**严重程度**：严重

**改进建议**：① 在 §3.5「标量-矩阵运算」段落中明确列出 scalar_mat_ops.cj 的 36 个函数涵盖范围：add/sub/mul/div 各为 9 种矩阵类型提供重载（包含全部 6 种非方阵）；② 在 §5 中补充确认 `scalar / mat` 对所有 9 种矩阵类型（含非方阵）均可本阶段完整实现；③ 对照 GLM 参考实现（如 type_mat2x3.inl:474-479）确认非方阵的 `T/mat` 除法在 C++ GLM 中确实存在

### 问题 2（一般）：`+=mat`/`-=mat` 复合赋值运算符的自动生成条件描述不精确

**问题描述**：§3.5 末尾"复合赋值运算符"段落声明"所有矩阵类型的逐元素运算（+=、-=、+=scalar、-=scalar、*=scalar、/=scalar）均可自动生成复合赋值"，但未区分 `+=(mat)` 和 `+=(U)`（泛型标量/矩阵参数）的情况。C++ GLM 中 `operator+=(mat)` 和 `operator*=(scalar)` 的签名接受泛型参数 `U`（如 `template<typename U> mat& operator+=(mat<U,Q> const& m)`），这与 `T` 类型的同尺寸矩阵不同。在仓颉的 `Number<T>` 约束 extend 块中，`operator func +(rhs: Mat4x4<T, Q>)` 的右操作数类型固定为 `Mat4x4<T, Q>`，不可能有 `U ≠ T` 的泛型矩阵类型参数。编译器自动生成的 `+=` 运算符仅覆盖 `T` 类型的同尺寸矩阵（与二元运算符签名一致），而非 C++ GLM 中 U 类型的跨类型矩阵。此差异在 §9 差异声明和 D08 中均未提及。

**所在位置**：§3.5「复合赋值运算符」段落；§9 差异声明；D08

**严重程度**：一般

**改进建议**：① 在 §3.5 复合赋值段落明确说明自动生成的 `+=(mat)` 仅覆盖 `Mat{C}x{R}<T, Q>` 同 T 同尺寸类型，不覆盖跨类型矩阵复合赋值（C++ GLM 的 `U` 类型参数版本）；② 在 §9 差异声明中补充：仓颉复合赋值运算符不支持跨类型矩阵参数（C++ GLM 中 `m1 += m2_float` 的 U=T2 形式不可用）；③ 更新 D08 补充此差异说明

### 问题 3（一般）：geometric.cj stub 函数签名中存在输入错误

**问题描述**：§3.7 中 `geometric.cj` 的 `distance` 函数签名列表存在一处输入错误。第 463 行：`func distance<T, Q>(p0: Vec3<T, Q>, p1: Vec3<T, Q>): T where T <: Number<T>, Q <| Qualifier`。其中 `Q <| Qualifier` 应为 `Q <: Qualifier`，`<|` 不是仓颉语言的合法子类型关系运算符。编码者直接复制此签名将导致编译失败。

**所在位置**：§3.7「geometric.cj」完整函数签名清单第 3 条 distance（Vec3 版本）

**严重程度**：一般

**改进建议**：将 `Q <| Qualifier` 修正为 `Q <: Qualifier`

---

### 问题 4（一般）：`matrix.cj` stub 中 `transpose` 函数签名使用了非仓颉语法的伪签名

**问题描述**：§3.7 中 `matrix.cj` 的 `transpose` 函数签名写为：`func transpose<C, R, T, Q>(m: Mat{C}x{R}<T, Q>): Mat{R}x{C}<T, Q>`。其中 `Mat{C}x{R}` 和 `Mat{R}x{C}` 是设计文档中的模板记法，不是合法的仓颉类型语法。在仓颉中不存在 `Mat{C}x{R}` 这样的类型——9 个矩阵类型各自独立命名（如 Mat2x2、Mat2x3 等），不能用泛型参数 C/R 参数化。因此 `transpose` 不可能是一个接受泛型 C/R 参数的全局函数，而必须为每个矩阵尺寸各提供一个独立重载（共 9 个）。设计文档中虽然注释了"每个矩阵尺寸各一版重载，共 9 个"，但主签名使用了不合法的伪语法，编码者无法直接使用。

**所在位置**：§3.7「matrix.cj」完整函数签名清单

**严重程度**：一般

**改进建议**：将 `transpose` 的伪签名替换为 9 个具体重载签名的列举，或至少改为明确说明每个尺寸有独立签名的规则（如 `func transpose<T, Q>(m: Mat2x2<T, Q>): Mat2x2<T, Q> / ... / func transpose<T, Q>(m: Mat4x4<T, Q>): Mat4x4<T, Q>`），与同清单中 `determinant` 和 `inverse` 的具体签名风格一致

---

### 问题 5（一般）：`mat / scalar` 运算符在当前设计下作为成员函数定义，但 C++ GLM 中它是独立自由函数

**问题描述**：§3.5「二元运算符（矩阵-标量）」列出了 `+、-、*、/（右操作数为 T）：逐元素操作`，这些运算符作为矩阵 extend 块中的成员运算符函数定义（如 `operator func /(rhs: T): Mat{C}x{R}<T,Q>`）。但在 C++ GLM 参考实现中，`mat / scalar` 是全局自由函数（非成员函数），如 `type_mat4x4.inl:710` `operator/(mat<4, 4, T, Q> const& m, T scalar)`。仓颉中 `mat / scalar` 可以自然地表示为成员运算符 `operator func /(rhs: T)`（`this` 即左操作数），语义等价——这不影响运算结果。但此差异点（成员运算符 vs 自由函数）的 API 形态差异未在 §9 差异声明中记录。当用户需要取 `m / scalar` 的函数引用时（仓颉中无法取成员运算符的函数引用），可能遇到不便。

**所在位置**：§3.5「二元运算符（矩阵-标量）」；§9 差异声明

**严重程度**：一般

**改进建议**：在 §9 差异声明中补充说明：C++ GLM 中 `mat / scalar` 为全局自由运算符，仓颉版本为成员运算符函数——两者运算语义等价，但 API 形态差异导致无法作为高阶函数参数传递（如无法写 `matList.map({ m => m / 2.0f })` 以外的方式引用此运算符）

---

### 问题 6（一般）：`operator/=`（mat/mat）的 stub 策略与 `operator/`（mat/mat）的 stub 策略不一致，未明确声明

**问题描述**：§3.5 声明 `mat / mat` 除法运算符"因 inverse 属于 matrix.cj 函数库（本阶段为 stub），矩阵-矩阵除法推迟至阶段四"，且 §5 仅说明了 `operator/`（mat/mat）的 stub 处理。但对于方阵类型的 `operator/=`（mat/mat），C++ GLM 参考实现中它是显式定义的（如 type_mat4x4.inl:393-396：`mat& operator/=(mat const& m) { return *this *= inverse(m); }`）。仓颉编译器自动生成 `operator/=` 的条件是存在 `operator/`（同类型参数），但当前设计中方阵的 `operator/`（mat/mat）本身就是 stub 调用，因此自动生成的 `/=` 也是 stub。此行为在设计文档中未明确说明——编码者可能困惑于 `/=` 是否也需要显式声明为 stub。

**所在位置**：§3.5「二元运算符（矩阵-矩阵）」除法子节；§5「错误处理策略」

**严重程度**：一般

**改进建议**：在 §3.5 或 §5 中明确说明：方阵类型的 `operator/=`（mat/mat）由编译器基于 `operator/`（mat/mat）自动生成，因此自动继承 stub 行为；无需单独声明 `/=` 的 stub 状态；当阶段四 inverse 完整实现后，`/=` 也会自动成为可用运算符

---

### 问题 7（轻微）：`filled(scalar: T)` 工厂函数的 `Number<T>` 约束下 `one - one` 获取 T(0) 的合理性存疑

**问题描述**：§3.3 第 8 条 identity 工厂函数中说明 T(0) 通过 `one - one` 在 `Number<T>` 上下文中演算获得。但在 §3.3 第 3 条 filled 工厂函数中，T(0) 的获取方式描述为 `scalar - scalar`。问题在于：当 `scalar` 是用户传入的任意值时，`scalar - scalar` 应正确产生 T(0)——这在数学上是正确的。但文档未分析 `filled(0.0f)` 的边界场景：此时 `scalar = T(0)`，`scalar - scalar = T(0) - T(0) = T(0)` 仍然正确，不会产生问题。此边界行为虽正确但文档未显式确认，编码者可能产生疑虑。

**所在位置**：§3.3 第 3 条 filled 工厂函数

**严重程度**：轻微

**改进建议**：补充一条注释或说明：即使 scalar=T(0)，`scalar - scalar` 仍正确产生 T(0)，filled(0.0f) 产生零矩阵（全部元素为 0），主对角线也为 0——语义上与 C++ GLM 的 `mat(0.0f)` 一致

---

### 问题 8（一般）：ext/ 别名文件中的 `vec1`/`dvec1` 别名命名风格与 GLM 参考实现不一致

**问题描述**：§3.8 将 vector_float1.cj 中的别名定义为 `public type vec1 = Vec1<Float32, PackedHighp>` 和 `public type dvec1 = Vec1<Float64, PackedHighp>`。然而在 C++ GLM 参考实现中，`ext/vector_float1.hpp` 定义的别名是 `vec1`（小写），`ext/vector_double1.hpp` 定义的是 `dvec1`（小写 d）。设计文档中的命名与 C++ GLM 一致，此处不存在问题。

但文档中对 `vector_float1.cj` 和 `vector_double1.cj` 的描述缺少文件内容示例：其他尺寸的向量别名文件（如 vector_float4.cj）的别名定义格式可以在 ext/ 别名文件段落中推断出，但 vec1/dvec1 的别名定义未显式列出。这是完整性问题——编码者无法从文档直接得知 vec1/dvec1 别名是否需要额外的 GLM 扩展头文件引用说明。

**所在位置**：§3.8 ext/ 别名文件部分

**严重程度**：轻微

**改进建议**：在 §3.8 向量别名文件列表中，为 vec1/dvec1 补充完整别名定义行（如 `public type vec1 = Vec1<Float32, PackedHighp>`)，与 mat4x4 的示例格式保持一致

---

### 问题 9（一般）：`identity(one: T)` 函数在非方阵类型上的语义未明确

**问题描述**：§3.3 第 8 条声明 `identity(one: T)` 作为所有矩阵类型的工厂函数，返回"主对角线为 T(1)、其余为 T(0) 的单位矩阵"。但严格来说，"单位矩阵"（identity matrix）仅对方阵有定义。非方阵的"单位矩阵"在数学上不成立——其"主对角线"长度为 min(C,R)，产生的不是方阵的单位矩阵而是对角矩阵。例如 Mat2x3 的 identity(1.0f) 产生 2×3 对角矩阵（与 §3.3 第 3 条 `filled(1.0f)` 对 Mat2x3 产生的结果完全相同）。设计中未明确 `identity` 在非方阵上的语义是否等同于 `filled(1.0f)`，也未说明是否应限制 `identity` 仅对方阵类型提供。

**所在位置**：§3.3 第 8 条 identity 工厂函数

**严重程度**：一般

**改进建议**：① 明确声明 `identity(one: T)` 对所有 9 种矩阵类型均提供，但在非方阵上等价于 `filled(one)`（两者产生相同的对角矩阵），记录为有意设计；② 或限制 `identity` 仅对方阵提供（语义更严格），非方阵使用 `filled` 构造对角矩阵——此方案更符合数学定义但减少了 API 统一性；③ 无论选择哪种方案，在 §9 差异声明中补充：非方阵的 identity 语义与 C++ GLM 不一致（C++ GLM 中非方阵不提供默认构造为单位矩阵的语义）

---

### 问题 10（一般）：`matrix.cj` stub 中 `matrixCompMult` 和 `outerProduct` 函数不属于方阵 .inl 的依赖闭合

**问题描述**：§3.7 和 §8 产出物清单中声明 `matrix.cj` 为 stub，且 §2 模块间依赖中声明 `type_mat2x2/type_mat3x3/type_mat4x4` 的 .inl 编排依赖 `matrix.cj`。但 `matrixCompMult` 和 `outerProduct` 这两个函数在 C++ GLM 中定义于 `matrix.hpp`/`matrix.inl` 中，但它们**不被** type_mat2x2.inl、type_mat3x3.inl、type_mat4x4.inl 直接调用——它们是独立的全局函数，供用户代码直接调用。将它们纳入 matrix.cj stub 本身是正确的（它们确实属于 matrix 函数库），但当前文档的措辞暗示这些函数是因为方阵 .inl 的依赖闭合而需要 stub，这可能误导编码者认为 stub 内容的选择是纯依赖驱动的。

**所在位置**：§3.7「matrix.cj」描述；§2 模块间依赖

**严重程度**：轻微

**改进建议**：在 §3.7 matrix.cj 条目中区分"方阵 .inl 依赖闭合所需的函数"（transpose、determinant、inverse）和"随 matrix.cj 附带提供的额外函数"（matrixCompMult、outerProduct），后者虽然在同一文件但目的不同——它们被纳入是因为同一函数库文件的组织约定，而非 .inl 编排依赖

### 问题 11（一般）：跨矩阵类型转换工厂函数 6a 中 SrcQ 参数在 T 相同场景下可能引发重载歧义

**问题描述**：§3.3 第 6a 条 `fromMat<SrcQ>(m: Mat{C_src}x{R_src}<T, SrcQ>, one: T)` 声明了 SrcQ 类型参数。当同一个目标矩阵类型同时有 `fromMat<SrcQ>(m, one)` 的 6a 版本和 `fromMat<SrcT, SrcQ>(conv, m, one)` 的 6b 版本时，如果调用方传入同类型源矩阵（SrcT = T），两个签名均可匹配——6a 要求 SrcT=T 隐含，6b 可以显式 T 作为 SrcT。此时编译器可能无法自动选择 6a 版本。虽然 SrcT 在 6b 中是显式类型参数，仓颉的泛型推断可能偏向 6b（因为 6b 有 conv 参数可以区分），但当省略 conv 的场景不存在时（6b 总是需要 conv），两个签名的参数个数不同，自然可以区分。然而，编码者需要了解：当 SrcT=T 时应优先使用 6a（无需 conv），文档未给出此指导。

**所在位置**：§3.3 第 6a 条 vs 第 6b 条

**严重程度**：轻微

**改进建议**：在 §3.3 第 6 条"6a 和 6b 共享的设计属性"中补充调用指导：当源矩阵与目标矩阵元素类型相同（SrcT=T）时，应使用 6a 版本（无需 conv 闭包）；仅在 SrcT≠T 时使用 6b 版本

---

### 问题 12（一般）：C++ GLM 中非方阵类型也提供 `operator*=(mat)` 复合赋值（语义为逐列乘标量），与当前设计声称不符

**问题描述**：§3.5 复合赋值运算符段落声称"非方阵类型的矩阵乘法结果类型不同于左操作数，编译器不会为非方阵自动生成矩阵乘复合赋值 `*=`"。这在仓颉自动生成规则下是正确的。但该段落同时声称"方阵 `*=`(mat) 由编译器自动生成，语义与 C++ GLM 的显式定义一致"。然而 C++ GLM 中**非方阵**也显式定义了 `operator*=(U s)`（标量乘复合赋值），见 type_mat2x3.inl:282-287。设计文档确实提到了非方阵的 `*=scalar` 可自动生成，但遗漏了 C++ GLM 中非方阵还定义了 `operator/=(U s)`（标量除复合赋值）——这与 `*=scalar` 同属逐元素操作。当前设计文档中仅笼统提及"*=scalar、/=scalar 可自动生成"，但未区分方阵和非方阵在这两个运算符上的行为一致性。

此问题较轻微，因为仓颉编译器自动生成规则对全部 9 种矩阵一致适用（逐元素运算均自动生成复合赋值），但文档措辞可能让编码者误以为只有方阵能获得某些复合赋值运算符。

**所在位置**：§3.5 复合赋值运算符段落

**严重程度**：轻微

**改进建议**：将当前"所有矩阵类型的逐元素运算（+=、-=、+=scalar、-=scalar、*=scalar、/=scalar）均可自动生成复合赋值"的表述拆分为更清晰的结构：① 同尺寸矩阵 ± 的复合赋值 `+=`/`-=`（所有 9 种类型，逐元素操作）；② 标量运算的复合赋值 `+=scalar`/`-=scalar`/`*=scalar`/`/=scalar`（所有 9 种类型）；③ 方阵乘法的复合赋值 `*=`(same_mat)（仅 3 种方阵）；④ 方阵除法的复合赋值 `/=`(same_mat)（仅 3 种方阵，stub 依赖）

### 问题 13（一般）：从需求响应充分度角度，阶段二验证标准中"矩阵-向量乘法语义正确性"未在设计文档中充分展开

**问题描述**：路线图（02_roadmap.md）阶段二验证标准包括"矩阵-向量乘法语义正确性验证"。当前设计文档的 §3.5 中定义了矩阵-向量乘法的签名（`Mat{C}x{R} * Vec{C} → Vec{R}` 和 `Vec{R} * Mat{C}x{R} → Vec{C}`），但未提供具体的语义验证用例或计算规则。编码者在实现乘法运算符时需要知道：对于 `Mat2x3 * Vec2 → Vec3`，具体如何按列主序展开计算（`result[i] = Σ col[k][i] * v[k]` for k=0..C-1）。当前设计假设"矩阵乘法语义是标准线性代数定义，无需展开"，但对于跨尺寸乘法（27 种重载），每个乘法的具体展开公式的正确性对编码者至关重要——尤其是文档 D04 选择"直接内联展开而非委托函数库"的决策意味着编码者需要手动编写每种乘法展开。

**所在位置**：§3.5「二元运算符（矩阵-向量）」；§4.2 关键行为契约代码示例；D04

**严重程度**：一般

### 问题 14（轻微）：lib.cj 现有的 `public import glm.detail.{ add, sub, mul, div, mod }` 已包含 scalar_vec_ops.cj 的同名函数，新增 scalar_mat_ops.cj 的同名函数后 mod 函数的重载不一致

**问题描述**：§8 中 lib.cj 条目说明现有 `public import glm.detail.{ add, sub, mul, div, mod }` 自动覆盖 scalar_mat_ops.cj 中 add/sub/mul/div。但 scalar_vec_ops.cj 中有 `mod` 函数（整数 mod + Float32/Float64/Float16 具体 mod），而 scalar_mat_ops.cj 不包含 `mod` 函数（矩阵取模无数学意义）。当前 lib.cj 的 `public import glm.detail.{ ..., mod }` 仅覆盖 scalar_vec_ops.cj 的 mod 函数，不含矩阵版本（因矩阵不提供 mod）。此情况无歧义但文档中的解释逻辑可能让编码者误解为 `mod` 也覆盖了 scalar_mat_ops.cj——文档应明确说明 `mod` 仅从 scalar_vec_ops.cj 导出，scalar_mat_ops.cj 不提供 mod。

**所在位置**：§8 修改文件 — lib.cj 条目

**严重程度**：轻微

**改进建议**：在 lib.cj 条目中补充说明：`public import glm.detail.{ add, sub, mul, div, mod }` 中的 mod 仅覆盖 scalar_vec_ops.cj 提供的标量-向量取模函数，scalar_mat_ops.cj 不提供 mod（矩阵取模无数学意义）
