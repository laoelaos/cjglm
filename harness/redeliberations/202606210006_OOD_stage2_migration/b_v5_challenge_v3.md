## 逐维度审查

### 1. 证据充分性

**[问题-严重]** P2 声称设计规则 §3.3 第 8 条的坐标复制范围定义与 GLM 逐列构造行为不一致，并在多处引用 Mat4x4 ← Mat3x2 作为差异证据。但经逐一核对 GLM `type_mat4x4.inl` 源码，**Mat4x4 ← Mat3x2 与设计规则完全一致，P2 将其列为差异证据属于事实错误**：

P2 在第 49 行声称"源 Mat3x2 的第 2 列（m[2]）是 Vec2(x,y)，GLM 将其扩展为 Vec4(x, y, 1, 0) 而非 Vec4(x, y, 0, 0)"。但设计规则 §3.3 第 8 条明确定义了两条填充规则：① 源矩阵 M_src[j][i] 复制到目标矩阵的相同坐标位置，范围 0 ≤ j < min(C_src, C_dst)、0 ≤ i < min(R_src, R_dst)；② 当 R_dst > R_src 时，额外行（i ≥ R_src）从单位矩阵取值——即 (i,i) 填 T(1)，其余填 T(0)。对于 Mat4x4 ← Mat3x2（C_src=3, R_src=2, C_dst=4, R_dst=4），列 j=2 的构造过程为：(2,0) 和 (2,1) 按①复制源数据，(2,2) 属于额外行 i=2 ≥ R_src=2 且在 (i,i)=(2,2) 对角线上按②填 T(1)，(2,3) 属于额外行但非对角线按②填 T(0)。设计规则预测结果 Vec4(x, y, 1, 0) 与 GLM 实际实现 `col_type(m[2], 1, 0)` **完全一致**。P2 误认为"设计规则将 (2,2) 填 0"，这是对设计规则第②条（单位矩阵对角线延伸）的曲解。

P2 据此错误分析进一步推导出"设计规则的坐标复制范围定义与 GLM 逐列构造行为不一致"的通用结论，并声称"所有涉及源矩阵行数 < 目标矩阵行数的转换均受影响"。但经逐一核验 Mat4x4 的全部 8 个转换构造函数，**真实差异仅存在于 Mat4x4 ← Mat4x2 一种场景**：

- Mat4x4 ← Mat4x2（`type_mat4x4.inl:246-257`）：GLM 第 2 列为 `col_type(0, 0, 1, 0)`，第 3 列为 `col_type(0, 0, 0, 1)`——**完全丢弃源矩阵第 2、3 列的数据**，替换为单位矩阵列。设计规则预测复制 src[2][0], src[2][1]（因 j=2 < min(C_src=4, C_dst=4)=4, i=0,1 < min(R_src=2, R_dst=4)=2），与 GLM 不一致
- Mat4x4 ← Mat3x2（`type_mat4x4.inl:218-229`）：GLM 第 2 列为 `col_type(m[2], 1, 0)`——保留源数据并按单位矩阵对角线扩展。**与设计规则一致**
- Mat4x4 ← Mat2x2（`type_mat4x4.inl:176-187`）：列 0,1 扩展，列 2,3 为单位矩阵列。与设计规则一致
- Mat4x4 ← Mat3x3 / Mat2x3 / Mat2x4 / Mat3x4 / Mat4x3：均与设计规则一致

此外，经核查 `type_mat4x3.inl`，Mat4x3 ← Mat4x2（`type_mat4x3.inl:228-239`）中 GLM 第 2 列为 `col_type(m[2], 1)`、第 3 列为 `col_type(m[3], 0)`——**保留源矩阵全部 4 列数据并扩展**，与 Mat4x4 ← Mat4x2 的行为（丢弃列 2,3）不同。这说明 GLM 的行为因目标矩阵类型而异，而非遵循 P2 提出的统一"逐列构造规则"。P2 对差异的范围和根本原因分析均不准确。

**[通过]** P1 对 `init()` 默认构造函数在无约束泛型 T 下不可行的分析证据充分、逻辑自洽。阶段一 Vec 类型回避此问题的机制（无默认 init）经源码验证准确，降级方案无效的论证有效。

**[通过]** P3 对第 2 条与第 9 条构造函数重载歧义风险的分析合理，改进建议可行。

**[通过]** P4 笔误识别准确。

**[通过]** P5 对 ext/ 与 fwd.cj 别名定位差异的分析有据且指出了持续 3 轮未回应的问题。

**[通过]** P6 对 `col()` 和 `[]` 在 const 上下文中的限制补充合理。

### 2. 逻辑完整性

**[问题-严重]** P2 提出的"逐列构造规则"改进建议在 Mat4x4 ← Mat4x2 这一关键差异场景下**自身存在逻辑缺陷**。P2 的改进方案对 `j < C_src` 的列保留源矩阵数据复制（"取源矩阵第 j 列的前 min(R_src, R_dst) 个分量复制到目标矩阵第 j 列的对应位置"），而 GLM 在 Mat4x4 ← Mat4x2 中对 j=2,3（j < C_src=4）**完全丢弃源数据**、替换为单位矩阵列。即 P2 的改进方案无法解决它所识别的核心差异——产出作者若按此方案修改，Mat4x4 ← Mat4x2 的列 2,3 仍保留源数据，与 GLM 行为不一致。

产生此缺陷的原因是 P2 将差异错误归因于"逐列构造 vs 坐标复制范围定义"的模式差异，而真实差异在于 GLM 在 Mat4x4 ← Mat4x2 这一特定场景下采用了与设计规则不同的语义：GLM 对 j ≥ R_src 的列（尽管 j < C_src）放弃源数据而回退为单位矩阵列，但对于 Mat4x3 ← Mat4x2 等场景则保留源数据。P2 未能精确识别此条件的复杂性。

**[通过]** P1、P3~P6 各问题之间无逻辑矛盾。P2 的修订说明对上一轮质询中 P1/P4 矛盾的统一处理合理。

### 3. 覆盖完备性

**[问题-一般]** P2 声称"72 个转换构造函数中所有涉及'源矩阵行数 < 目标矩阵行数'的转换均受影响"，此影响范围评估严重过度泛化。经核验，Mat4x4 的 8 个转换中仅 Mat4x4 ← Mat4x2 存在差异，其他 R_src < R_dst 的转换（如 Mat4x4 ← Mat3x2、Mat4x4 ← Mat2x2、Mat4x4 ← Mat2x3）设计规则与 GLM 均一致。在其他目标矩阵类型中仍需逐案核验——Mat4x3 ← Mat4x2 与设计规则一致。报告应精确识别受影响的转换子集，而非泛化声称"所有"涉及行数不足的转换。

**[问题-一般]** P6 提及 `[]` 和 `col()` 的 const 限制时，仅建议"标注"但未评估此限制是否构成新的用户可见差异应记录在 §9——C++ GLM 中 `operator[]` 在 constexpr 上下文可用（deviations.md IF-03 已为 `componentAt` 记录此限制），矩阵 `[]` 的同类限制是否也应在 §9 差异声明中新增条目，P6 未给出判断。

**[通过]** 报告覆盖了需求响应充分度（P1）、事实正确性（P2）、逻辑一致性（P3）、完整性（P5、P6）等维度。历史迭代反馈中 ext/ 别名定位差异（P5）经 3 轮追踪已保留。

### 4. 报告必要性

**[问题-一般]** P4（`type_mat4x3.cj ★ — Mat4x4<T,Q>` 笔误）属文档校对类细节，与任务审查视角（需求响应充分度、整体深度和完整性）关联度极低。报告虽将严重程度降级为"轻微"但仍保留为正式问题条目，建议删除或降为附注。

**[通过]** P6 对 const 上下文可行性的分析与 §3.3 第 1 条对 `const init` 的关注形成对称，具有设计完整性价值，必要性可接受。

## 质询要点

- **问题1**：P2 将 Mat4x4 ← Mat3x2 列为设计规则与 GLM 不一致的差异证据，但经源码核验两者完全一致——设计规则第②条（额外行从单位矩阵取值，(i,i) 填 T(1)）正确预测了 (2,2)=T(1)，P2 对设计规则的解读存在根本性错误。P2 据此推导的"通用不一致"结论和"所有涉及行数不足的转换均受影响"的范围评估均不可信
  - **原因**：产出作者无法根据包含事实错误的审查意见准确修改设计规则——按 P2 建议修改已在 Mat4x4 ← Mat3x2 场景下正确的设计规则，可能引入不必要的偏离
  - **建议方向**：重新核验全部转换场景，区分"与设计规则一致"和"与设计规则不一致"的场景。当前经核验仅 Mat4x4 ← Mat4x2 一种场景存在真实差异。需进一步核查其他目标矩阵类型（Mat4x3、Mat3x3 等）的转换以确认完整的影响范围。特别需注意 GLM 在 Mat4x3 ← Mat4x2 中保留了源数据（与 Mat4x4 ← Mat4x2 行为不同），差异的条件比 P2 所述更复杂

- **问题2**：P2 提出的"逐列构造规则"改进方案在 Mat4x4 ← Mat4x2 这一真实差异场景下无法复现 GLM 行为——P2 的方案仍对 j < C_src 的列保留源数据复制，而 GLM 在该场景中对 j=2,3 丢弃源数据
  - **原因**：改进方案自身不完整，产出作者无法据此产出与 GLM 一致的实现
  - **建议方向**：改进方案需完整覆盖真实差异场景，特别是需解释 GLM 在 Mat4x4 ← Mat4x2 与 Mat4x3 ← Mat4x2 中采用不同策略的条件（前者丢弃 j≥R_src 的源列数据，后者保留所有源列数据），并给出可指导编码的精确规则

- **问题3**：P2 影响范围评估过度泛化——"72 个构造函数中所有涉及'源矩阵行数 < 目标矩阵行数'的转换均受影响"不符合事实
  - **原因**：过宽的影响范围评估导致问题严重度虚高
  - **建议方向**：逐一核验受影响的转换子集，给出精确的组合清单而非笼统声称
