## 1. 事实错误：`throw` 被错误定性为非 const 表达式

**问题描述**：
报告在 #12 分析（第 131 行）、#17 分析（第 167 行）及多处下游依赖段落（第 345 行优先级建议、第 360 行副作用矩阵）中反复声称"`throw` 表达式在仓颉中同样不是 const 表达式"。然而，仓颉语言 const 表达式规范明确将 `throw` 列为合法的 const 表达式——"`if`、`match`、`try`、`throw`、`return`、`is`、`as` — 这些表达式内的表达式必须都是 const 表达式"（cangjie-lang-features/const/README.md §2 规则 8）。

`componentAt` 中阻塞 `const` 的真实原因需要重新审视：`throw` 本身不阻塞 const，需要验证的是 `Exception("...")` 构造函数调用是否为 const 表达式（即 Exception 类是否提供 `const init`）。`assert` 调用确实可能为非 const（需检查 `assert` 是否有 `const` 重载），但 `throw` 的定性是错误的。

**所在位置**：第 131 行（#12 修复前提）、第 167 行（#17 修复前提）、第 345 行（优先级建议）、第 360 行（副作用矩阵）

**严重程度**：严重

**改进建议**：
(a) 删除"`throw` 表达式不是 const 表达式"的错误断言；
(b) 重新验证 `Exception("...")` 是否为 const 表达式（查阅标准库文档确认 Exception 是否有 `const init`）；
(c) 根据验证结果修正 #12/#17 的修复前提分析——如果 `Exception(...)` 不是 const 表达式，则症结在 `Exception` 构造函数而非 `throw` 关键字；如果 `Exception(...)` 是 const 表达式，则 `throw` 分支不阻塞 const，唯一阻塞因素是 `assert`；
(d) 根据修正后的前提重新评估 #12/#17 的修复路径和优先级。

## 3. 可行性未经验证：#7 浮点 mod 修复方案中的泛型 `trunc` 调用

**问题描述**：
报告第 82 行建议新增 4 个浮点 mod 重载，使用约束 `where T <: Number<T> & Equatable<T> & Comparable<T>`，函数体使用 `trunc(x / y)` 实现 `x - y * trunc(x / y)` 算术恒等式。但 `trunc` 是 **重载自由函数**（非泛型函数，也非任何接口的方法），其三个签名 `trunc(Float16): Float16`、`trunc(Float32): Float32`、`trunc(Float64): Float64` 分别针对具体浮点类型。FloatingPoint<T> 接口（查阅原版文档确认）不提供 `trunc` 方法。

在泛型函数体中调用 `trunc(expr)` 其中 `expr` 类型为泛型参数 `T`，编译器能否在定义时解析 `trunc` 的重载，还是需推迟到单态化（monomorphization）时刻——此行为在报告中未经验证。如果仓颉不在单态化时刻做重载解析，该修复方案无法通过编译。

**所在位置**：第 82 行（#7 修复提示及副作用分析）

**严重程度**：一般

**改进建议**：
(a) 通过实际编译实验验证泛型 `trunc` 调用在仓颉中是否可行；
(b) 如果不可行，需修正修复方案为：为 Float32/Float64 各编写独立的具体类型重载（如 `mod(s: Float32, v: Vec1<Float32, Q>)`，函数体使用 `Float32(trunc(Float64(x) / Float64(y)))` 或 `trunc(x / y)` 直接调用）；
(c) 如果可行，在报告中补充验证说明以供执行者确认。
