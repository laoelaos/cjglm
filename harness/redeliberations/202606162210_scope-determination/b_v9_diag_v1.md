## 发现问题

### 问题1：Task 3"范围确认结论"缺乏统一呈现，结论要素分散在三处

- **问题描述**：用户需求 Task 3 明确要求"给出范围确认结论，包括核心文件清单、类型数量、验证标准"作为统一的结论性输出。当前文档将文件清单置于 3E 节、类型数量置于 3D 节、验证标准置于 4.4 节，三要素散布在各处，缺少一个独立的「范围确认结论」章节统一汇总并明确确认"上述范围即为首轮迁移的最终结论"。实施者需要跨章节拼读才能获取完整的结论视图。
- **所在位置**：第 3 节（3D、3E）、第 4.4 节
- **严重程度**：中
- **改进建议**：在 3E 节之后（或 4.4 节之后）新增一个「范围确认结论」小节，以精简清单形式汇总：(1) 确认首轮迁移的核心文件清单（文件名、数量 11 个）；(2) 确认类型数量（4 个 Vec 结构体 + 256 个别名 + 10 个标量类型 + bool）；(3) 确认验证标准摘要；(4) 明确声明该范围满足依赖闭合性和独立可编译性。使读者能在单节中获得完整的范围结论。

### 问题2：文件清单(3E)依赖列使用 C++ 头文件符号，未与仓颉 shim 层建立显式关联

- **问题描述**：3E 节文件清单的"依赖"列以 C++ 标准库头文件符号（`<cstddef>`、`<limits>`、`<functional>`、`<cassert>`）标注依赖关系。虽然 0.7.2 节给出了 C++ 标准库 → 仓颉 shim 的映射总表，但文件清单中未显式标注每个文件对应的仓颉 shim 层文件名。实施者拿到文件清单后，需要手动交叉索引到 0.7.2 节才能确定每个文件的实际仓颉 import 依赖。这对文档的"直接可用性"构成障碍——实施者无法仅凭文件清单完成编译配置。
- **所在位置**：3E 节文件清单表（第 1-11 行及 S1-S3 行），0.7.2 节 Shim 策略总表
- **严重程度**：中
- **改进建议**：在 3E 节文件清单的依赖列中，为每个 C++ 头文件依赖标注对应的仓颉 shim 文件名。例如：`<cstddef>` 标注为 "+ `<cstddef>` → `shim_cstddef.cj`"，`<limits>` 标注为 "+ `<limits>` → `shim_limits.cj`"。或者新增一个"shim 依赖列"，直接列出每个文件的 shim 依赖列表。

### 问题3：`compute_vector_relational.hpp` 的 `<limits>` 依赖描述存在语义上的误导风险

- **问题描述**：文档在多个位置（第 1 节、3E 行 3、4.6 子范围 2a、第 5 节图注）反复强调 `compute_vector_relational.hpp` "自身活跃代码仅执行 `a == b`，未直接使用 `std::numeric_limits`"，称 `<limits>` 依赖由"调用方通过 `compute_equal` 模板实参活跃使用"。然而，`compute_equal` 模板中引用 `std::numeric_limits<T>::is_iec559` 的**代码本身写在该文件内**，是该文件模板定义的一部分。从仓颉编译模型看，若该文件定义了引用 `shim_limits` 中类型的泛型函数，则**该文件自身的 import 语句必须包含 shim_limits**，与调用方无关。文档的"未直接使用"描述在 C++ 模板实例化语义下勉强成立，但在仓颉 import 模型下产生误导——实施者可能误以为该文件不需要导入 shim_limits。
- **所在位置**：第 1 节层次 1 描述（第 179 行）、3E 节文件清单第 3 行（第 411 行）、4.6 节子范围 2a 说明（第 658 行）、第 5 节图注（第 703 行）
- **严重程度**：中
- **改进建议**：统一修正相关描述，将"自身未直接使用 `std::numeric_limits`"改为"该文件模板定义中引用了 `std::numeric_limits<T>::is_iec559`（函数体内），因此在仓颉等效实现中该文件需要直接 import shim_limits"。在 3E 节文件清单的依赖列，将该文件的 `<limits>` 依赖从注释性质升级为标准依赖项。

### 问题4：二元算术运算符委托模式的性能开销未讨论

- **问题描述**：0.5.1 节推荐二元算术运算符通过"拷贝 + 复合赋值"委托模式实现，即每次 `a + b` 产生一次 Vec 结构体的完整拷贝。对于图形学数学库，向量算术运算是核心高频操作（每帧可能出现数万至数百万次），额外的拷贝开销与 GLM C++ 版本的性能预期可能存在显著差异。文档将此方案作为推荐策略，但未讨论其性能影响，也未提及其他可能性（如直接实现二元运算符以避免拷贝）。对于一份"可投入使用"的范围分析文档，这是一个遗漏。
- **所在位置**：0.5.1 节"委托模式说明"（第 81-87 行）
- **严重程度**：低
- **改进建议**：在 0.5.1 节补充一段性能影响讨论：(1) 量化单次委托引入的拷贝代价（对于 Vec4<Float32> 约为 16 字节拷贝）；(2) 说明首轮以功能正确性优先，性能优化留待后续轮次；(3) 标注此方案为"首轮默认"，后续可改为直接实现二元运算符以消除拷贝。将此影响也列入 4.6 节风险评估中（作为补充风险项 R7）。

### 问题5：「最简可行范围」描述缺乏独立文件清单，不可直接对比

- **问题描述**：3E 节末尾的"首轮范围性质说明"提及"最简可行范围约含 5 个核心文件及约 48 个别名"，但未给出这 5 个核心文件的具体清单。读者无法将"5 个核心文件"与"推荐范围的 9 个核心文件"做逐项对比，无法准确评估降级的影响。这在范围决策场景中降低了文档的可用性——如果项目需要压缩范围，读者不知道哪些文件可以移除、哪些保留。
- **所在位置**：3E 节末尾"首轮范围性质说明"（第 431 行）
- **严重程度**：低
- **改进建议**：补充"最简可行范围文件清单"，以表格形式列出 5 个核心文件路径、类型说明和与推荐范围的差异。同时补充说明移除 `vec1` 和 `compute_vector_decl.hpp`/`_vectorize.hpp` 对依赖闭合性的具体影响（如 `type_vec4.inl` 中的调用需 stub 处理）。

### 问题6：文档行文冗余，相同依赖关系在 5+ 处重复描述

- **问题描述**：`compute_vector_relational.hpp` 的 `<limits>` 依赖关系、`std::size_t` → `Int64` 映射的语义差异等核心事实，在文档中重复出现在第 1 节、3E 节、4.6 节和 5 节等多个位置。每次描述措辞略有差异，虽无实质性错误，但增加了文档保持一致性的人力成本（过去 9 轮迭代中已因此产生多次修正）。冗长篇幅也降低了读者定位关键信息的效率。这虽不影响正确性，但影响深度质量和使用体验。
- **所在位置**：第 1 节（第 179 行）、3E 节（第 411-412 行）、4.6 节（第 658 行、第 667 行）、5 节（第 703-704 行）
- **严重程度**：低
- **改进建议**：将核心依赖事实统一在 0.7 节的`<limits>`行和 3E 节文件清单中一次完整陈述，其他引述位置统一改为"详见 0.7 节"或"详见 3E 节文件清单第 N 行"的交叉引用，消除重复内容。

## 整体评价

产出整体质量高，9 轮迭代使文档在技术准确性和细节完整性上达到较高水准。需求三要素（基础类型识别、范围边界、验证标准）均已覆盖，语言适应性分析、文件清单和风险控制措施详实可用。上述 6 个问题属于**可用性和深度完善层面**，不涉及根本性的事实错误或逻辑矛盾。其中最需优先处理的是问题 1（结论统一呈现）和问题 2（shim 依赖关联），直接影响实施者能否直接将文档投入下一阶段工作。
