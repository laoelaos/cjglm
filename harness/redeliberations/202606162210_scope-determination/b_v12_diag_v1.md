## 问题清单

### 问题1（严重）：`Vec<T, Q>` 的泛型参数 Q 设计存在根本性语法错误

**问题描述**：文档将 C++ GLM 中 `qualifier` 枚举值（如 `packed_highp`）直接作为 CangJie 泛型结构体 `Vec<T, Q>` 的类型实参使用，例如 `Vec1<Float32, qualifier.packed_highp>`。但 CangJie 的泛型类型参数仅接受**类型**作为实参，不接受值（包括枚举构造器值）。`qualifier.packed_highp` 是一个枚举值（enum constructor value），不是类型，不能作为泛型实参。CangJie 不支持 C++ 式的非类型模板参数（non-type template parameter）。

此问题贯穿全文所有涉及 `Vec<T, Q>` 类型参数 Q 的描述，包括但不限于：

- **所在位置**：
  - 0.1 节（第15行）：泛型 struct 定义 `Vec<T, Q>` 示例
  - 0.3 节（第36行）：泛型类型别名说明 `type Vec2<T> = Vec<2, T>` 及评估结论
  - 3F 节（第478行）：`type Vec1f32 = Vec1<Float32, qualifier.packed_highp>`
  - 4.4 节（第699行）：验证代码 `Vec4<Float32, qualifier.packed_highp>(...)`

- **严重程度**：严重

- **改进建议**：重新设计 `qualifier` 在 CangJie 中的映射方案。两个可行方向：
  - **方案A（接口多态）**：将各 `qualifier` 变体定义为独立类型（如空结构体 `struct PackedHighp <>: Qualifier {}`、`struct AlignedHighp <>: Qualifier {}`），借助 `where Q <: Qualifier` 约束实现类型级区分。
  - **方案B（运行时字段）**：去掉 Q 类型参数，将 `qualifier` 作为 Vec 结构体的运行时枚举字段存储（`let q: qualifier`），牺牲编译期类型区分以简化设计。
  
  无论采用哪种方案，需全文统一修正所有 `Vec<T, Q>` 的类型签名、代码示例和别名定义。

### 问题4（低）：`@OverflowWrapping` 示例代码中的注解语法错误

**问题描述**：0.5.1 节（第83-86行）的代码示例中，`@OverflowWrapping` 被放置在行尾注释位置（`// @OverflowWrapping`），而非作为属性注解放在函数声明之前。CangJie 中属性注解的正确语法是：

```cangjie
@OverflowWrapping
public operator func +=(rhs: Vec2<T>): Vec2<T>
```

当前写法可能误导实现者。

- **所在位置**：0.5.1 节第83-86行（代码示例块）
- **严重程度**：低
- **改进建议**：将 `@OverflowWrapping` 从行尾注释中移至函数声明前作为独立属性行
