**审查视角**：使用者视角 — 需求响应充分度、事实正确性、深度/完整性

## 问题清单

### 问题 1（严重）：编译期条件包含 `_swizzle.hpp` 替代方案存在事实盲区

**问题描述**：
0.4 节将"`const` 配置常量 + `if` 分支"作为替换 C++ `#ifdef`/`#include` 模式的**首选**策略，但该策略无法处理"条件包含独立文件"的场景（如 `type_vecN.hpp` 中的 `#if GLM_CONFIG_SWIZZLE == GLM_SWIZZLE_OPERATOR` / `#include "_swizzle.hpp"`）。

经查阅 CangJie 语言文档确认：
- `import` 语句必须是文件顶层声明（`package` 之后、其他声明之前），不能出现在 `if` 分支内
- CangJie 宏（`@MacroName`）可用于编译时代码生成，但其展开代码"不能包含包声明或导入语句"（`macro/README.md:52`）
- 因此 CangJie 中不存在与 C++ `#if` + `#include` 等价的"条件性选择加载不同文件"的机制

当前首轮禁用 swizzle（方案 A）不会立即暴露此问题，但 "`const` + `if`" 被表述为通用策略而未标注此关键限制，会给后续 swizzle 功能的迁移路径规划造成严重误导——后续若要启用 swizzle，仅靠 `const` + `if` 无法实现按配置条件选择加载不同实现文件的模式。

**所在位置**：0.4 节"推荐首轮策略"段落（第 46 行）

**严重程度**：严重

**改进建议**：将"`const` 配置常量 + `if` 分支"的表述从"首选方案"降级为"适用于**内联条件代码**（函数体中的条件分支）的有限替代方案"；明确标注其不适用于条件包含独立文件的场景；补充说明 CangJie 宏在条件代码生成方面的可行性约束；为后续 swizzle 功能的启用提供更具体的迁移路径（例如：所有 4 个 Vec 类型定义中内联 swizzle 成员方法，通过 `if (GLM_CONFIG_SWIZZLE != GLM_SWIZZLE_DISABLED) { ... }` 条件编译，并验证 const 死分支消除对最终二进制的影响）。

### 问题 2（中）：3D 节"合计可实例化类型 = 256"表述未按 v9 修订计划修正

**问题描述**：
v9 修订说明（第 895 行）明确声称已将 3D 节"合计可实例化类型"改为"合计类型别名定义数"，并注明实际泛型结构体实例化数约 44 个。但实际文档 3D 节（第 402 行）仍保留"合计可实例化类型 | 256 个具现化类型"的旧表述。

256 是 type alias 定义数（`type Vec2f = Vec2<Float32>` 等语句数），而非实际可实例化的结构体类型数（最大约 4 个 Vec 结构体 × 11 种基础类型 ≈ 44 个）。"可实例化类型"易被误解为"需要单独编写的结构体数量"，严重误导工作量评估。

**所在位置**：第 3D 节首轮范围规模总结表（第 402 行）

**严重程度**：中

**改进建议**：按 v9 修订方案将"合计可实例化类型"改为"合计类型别名定义数"，并在该行或下方同步补充"泛型结构体实际实例化数：≤44 个"。同时排查 3G 节类型数量汇总表（第 493-495 行，当前已正确标注别名数 256 与实例化数 ≤44），确保两处表述口径一致。

### 问题 3（中）：3E → 3G → 3F 章节顺序逻辑颠倒

**问题描述**：
"## 3. 建议的首轮迁移范围"中子章节顺序为：3E（文件清单）→ 3G（范围确认结论）→ 3F（命名空间/包映射策略）。3G 节中已使用 `package glm.detail`、`package glm`、`import glm.detail.*` 等概念（第 484、534 行），但读者此时尚未读到 3F 节对这些映射规则的定义。

正确的逻辑顺序应为：3E（给出文件清单）→ 3F（定义包映射规则，解释清单中引用的包名）→ 3G（基于前两者汇总范围结论）。当前顺序导致 3G 节出现超前概念引用。

**所在位置**：第 3 节内子章节（3G 始于第 454 行，3F 始于第 522 行）

**严重程度**：中

**改进建议**：将 3F 节前移至 3E 节之后、3G 节之前，使读者先了解包映射规则，再阅读使用这些规则的汇总结论。也可在 3G 节首次出现 `package glm.detail` 等概念处添加交叉引用说明"详见 3F 节"。

### 问题 4（中）：3E 文件清单中 `<functional>` 适配的 Shim 覆盖说明缺失

**问题描述**：
3E 节文件清单第 4 行（`compute_vector_decl.hpp`）的 C++ 依赖列明确标注了 `<functional>`（`std::plus<T>` 等函数对象），但 Shim 依赖列仅列出 `shim_cstddef.cj` 和 `shim_limits.cj`（传递），未对 `<functional>` 的适配策略作任何标注。后文 0.7.2 节将 `<functional>` 的策略列为"非独立文件"，说明其采用内联替代而非独立 shim 文件。

作为"可直接作为实施依据"的文件清单（文档自称），缺失此项标注会导致实施者无法从清单中获知 `<functional>` 需要适配，只有通读 0.7.2 节的深层描述才能发现。

**所在位置**：3E 节文件清单第 4 行（第 414 行）

**严重程度**：中

**改进建议**：在 3E 节文件清单第 4 行 Shim 依赖列增加注释（如"`<functional>` → 内联替代，无独立 shim 文件"），或在列末添加脚注说明"未独立列出的 C++ 标准库依赖详见 0.7.2 节"。同步检查其他条目是否有类似遗漏。

### 问题 5（低）：3D 节与 3G 节合计文件数口径不统一

**问题描述**：
- 3D 节（第 401 行）"合计核心文件 = 9 个文件"：统计范围不含 `fwd.hpp`（setup + qualifier + 4 vec + 3 辅助）
- 3G 节（第 460-474 行）"核心文件清单"列出 11 个 GLM 源文件（上述 9 个 + `fwd.hpp` 标量别名部分 + `fwd.hpp` 向量别名部分）

3D 节将 `fwd.hpp` 完全排除在"核心文件"计数之外，而 3G 节将其列为两个条目。两节未说明计数口径的差异原因。实施者查阅 3D 节时会认为需要迁移 9 个文件，查阅 3G 节时看到 11 个文件的清单，造成困惑。

**所在位置**：3D 节（第 401 行）、3G 节（第 460-474 行）

**严重程度**：低

**改进建议**：统一文件计数口径。建议在 3D 节表格中将"合计核心文件"拆分为"核心实现文件（9 个）+ 别名/映射文件（fwd.hpp 的 2 个子部分）"或采用与 3G 节一致的 11 个文件计数，消除歧义。

### 问题 6（低）：3E 文件清单将 `fwd.hpp` 拆为两个条目未注明文件同一性

**问题描述**：
第 10-11 行将同一物理文件 `fwd.hpp`（`glm/fwd.hpp`）的两个不连续部分标记为独立条目（"标量别名部分第 1-178 行"和"向量别名部分第 180-507 行"），但未以任何方式注明两者对应于同一物理文件。这可能导致实施者误以为需要创建两个独立的 CangJie 文件。

**所在位置**：3E 节文件清单第 10-11 行（第 421-422 行）

**严重程度**：低

**改进建议**：在清单表下方或通过注脚注明"条目 10 和 11 均对应同一 C++ 源文件 `fwd.hpp` 的不同节段，CangJie 中应根据包结构拆分为两个文件（`package glm` 包下分别定义标量别名和向量别名）"。
