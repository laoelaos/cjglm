## 逐维度审查

### 1. 类型系统可行性

**[通过]** 9 个独立矩阵泛型结构体（Mat2x2<T,Q> ~ Mat4x4<T,Q>）的类型形态选择正确——仓颉不支持模板偏特化，无法实现 Mat<C,R,T,Q> 单一泛型后按 C/R 分别特化。

**[通过]** 矩阵类型以列向量为数据成员（var c0: VecR<T,Q> 等），结构体值语义选择合理：数学计算中的值对象应使用值类型，与阶段一 Vec 类型一致。

**[通过]** 泛型约束使用正确：`where T <: Number<T>`（自引用接口约束）、`where Q <: Qualifier` 均在仓颉泛型系统能力范围内。Number<T> 接口提供 `+-*/` 和一元负号，不提供 T(0)/T(1) 构造——设计中的 T(0) 演算策略（`someValue - someValue`）和 T(1) 参数化策略（`one: T`）准确反映了此限制。

**[通过]** extend 块中添加运算符重载和工厂函数的模式是仓颉标准实践。运算符函数不能为泛型（不能引入额外类型参数），设计中的运算符使用封闭类型的类型参数 T/Q，不引入额外泛型参数，符合语言规则。

**[通过]** `@Derive[Hashable]` 自动派生在 struct 上受支持，参与派生的字段需为 public。设计未显式标注字段访问修饰符，但架构级设计中可隐含要求 `public var` 以实现派生——编码阶段实现时落实即可。

### 2. 标准库与生态覆盖

**[通过]** 设计中使用的语言/库能力均在仓颉标准库覆盖范围内：
- `Exception`（std.core 自动导入）
- `@OverflowWrapping`（内置溢出注解，作用于函数声明）
- `@Derive[Hashable]`（std.deriving）
- `Number<T>`、`Integer<T>`（std.math）
- `!`（按位取反）仅位于 `Integer<T>` 接口——设计明确区分 T(1) 在 Number<T> 上下文中不可演算，策略正确

**[通过]** `mod` 函数签名标注为暂定并计划拆分为整数版本（`Integer<T>` 约束）和浮点重载，符合仓颉 `%` 仅存在于 `Integer<T>` 中的实际情况。

**[通过]** transpose/matrixCompMult/outerProduct 共 27 个重载的约束已从 `where T <: Number<T>` 统一放松为 `where Q <: Qualifier`——这些运算仅涉及索引重排或按位运算，无需算术约束，也兼容 Bool 矩阵，与 GLM 行为一致。

### 3. 语言特性可行性

**[通过]** 错误处理采用 `throw Exception("stub")` 标准方式，下标越界抛出 Exception——均在仓颉异常体系范围内。

**[通过]** 并发设计合理：矩阵为值类型，所有运算符返回新实例，天然线程安全，无需额外同步机制。

**[通过]** 包结构设计遵循 cjpm 规范：
- `package glm.detail` — 对应 `src/detail/` 目录
- `package glm` — 对应 `src/` 根目录
- `package glm.ext` — 对应 `src/ext/` 目录

设计已识别 `package glm.ext` 的子包发现依赖原型验证（cjpm 对 `src/` 下子目录的自动扫描策略），并提供三套递进方案，风险可控。

**[通过]** `const init` 用于逐分量构造和列向量构造（纯赋值），符合仓颉 const 表达式规则。使用 `var` 字段的 struct 中允许定义 `const init`（class 有 `var` 限制，struct 无此限制）。

**[通过]** `@OverflowWrapping` 已标注于所有算术运算符签名表上方（§3.5），编码阶段需在对应函数声明前添加此注解——注解仅能作用于函数声明，设计已正确指明这一点。

**[通过]** 编译顺序问题（行向量×矩阵 Vec extend 块中引用矩阵类型）已充分分析，提供三套 fallback 方案并评估各自影响，编码前将通过原型验证确定具体方案。

### 4. 设计一致性

**[通过]** 各抽象职责清晰：每个矩阵类型承载一个 C×R 尺寸的列向量成员和对应运算符；stub 函数库（common.cj、matrix.cj、geometric.cj）职责明确区分为纯 stub 和部分实现+部分 stub。

**[通过]** 模块间依赖关系明确且无循环依赖：type_mat 依赖 type_vec，scalar_mat_ops 依赖 type_mat，fwd.cj/lib.cj 依赖 detail 包。Vec extend 块引用矩阵类型是唯一的逆向引用，已通过独立文件方案（方案二）或矩阵文件末尾 extend（方案三）解决。

**[通过]** 144 个 fromMat 6a/6b 签名构造量的风险已标注，建议代码生成工具——这是合理的实施风险管控。

**[通过]** Bool 矩阵不支持的操作清单（D33）已完整更新包括 diagonal() 和 identity()，与 F9 修复一致。

**[通过]** fromMat 转换矩阵表的 72 个方向标注已修正，图例改用 `rowExt`/`rowSh`/`colExt`/`colSh` 精确符号，消除歧义。执行顺序统一为"先列后行"并体现在所有标注中。

**[通过]** Mat4x4←Mat4x2 偏差已文档化，包含专用编码处理和单元测试要求。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：每个矩阵类型只处理一个 C×R 尺寸；stub 文件按功能领域拆分（数学函数 common.cj、矩阵运算 matrix.cj、几何运算 geometric.cj）。

**[通过]** 抽象层次恰当：不要求完整实现细节（如具体方法体），仅在 §3.3 item 6 中给出四项基本操作的定义和组合规则，足以指导编码阶段实现全部 72 个转换方向。

**[通过]** 模块依赖自洽，编码顺序有明确建议（基础类型 → 矩阵类型 → Vec extend → 辅助文件）。

**[通过]** 设计考虑了可测试性：B 类方向单元测试验证要求、Mat4x4←Mat4x2 偏差专用测试、非 B 类方向验证策略、编码启动前验证项均有明确定义。

**[通过]** 迁移影响评估完整：`v * m` → `mulRowVecMat(v, m)` 的 API 破坏性和升降级判定标准已在 F10 修复中补充。
