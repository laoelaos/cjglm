# OOD 设计方案质量审查报告（第 5 轮）

## 审查范围

- **审查视角**：需求响应充分度、整体深度和完整性、下游消费者可用性、异常/边界条件覆盖
- **避免复验的维度**：技术可行性（已在组件A内部审议中确认，输出 APPROVED）
- **落地视角**：设计是否可直接指导编码实现、接口定义是否足以支持下游消费者

## 发现的问题

### 问题 1（中等）：cjpm 子包构建风险缺乏充分应对方案

**问题描述**：
§2"cjpm 子包构建预验证"段落（第160-167行）将 `src/ext/` + `package glm.ext` 的构建可行性列为"编码启动首日"验证项，但仅提供了一个粗略的备选方案（"调整为将 ext/ 文件置于 src/ 根目录并声明 package glm，需在项目级评估别名冲突风险"）。此备选方案未做任何冲突风险评估——若将 ext/ 文件与 lib.cj/fwd.cj 置于同一包（package glm）中，矩阵别名（mat4、mat4x4 等）与 detail 包泛型类型（Mat4x4）是否会产生名称冲突？fwd.cj 中已定义的别名（Mat4x4 = detail.Mat4x4<Float32, detail.PackedHighp>）与 ext/ 中的别名（mat4 = Mat4x4<Float32, PackedHighp>）在同一包中是否会产生重定义错误？此备选方案的可行性本身未经评估。

此外，若 cjpm 不支持内置目录相关的子包发现（即 `src/ext/` 不被自动识别为子包源目录），此问题将直接阻碍全部 26 个 ext/ 别名文件的编码。此风险影响范围是整个 Stage 2 产出物的一半文件数量（26/43 个新增文件）。

**所在位置**：§2 "cjpm 子包构建预验证"（第160-167行）

**严重程度**：中等

**改进建议**：
1. 在编码前的设计阶段主动用小规模原型验证 cjpm 的子包发现机制，而非留在编码首日——若验证失败，设计阶段就有充裕时间调整包组织策略
2. 备选方案（ext/ 文件并入 package glm）需提前做冲突风险评估：在 `package glm` 中同时存在 `public type mat4 = Mat4x4<Float32, PackedHighp>` 和 `public type Mat4x4 = detail.Mat4x4<Float32, detail.PackedHighp>` 是否合法（仓颉在同一包内是否允许大小写不同的同名类型别名）？
3. 若备选方案也不可行，需有第三方案（如在 src/ 根目录下新增 ext_float4x4.cj 等命名变体的文件）

### 问题 2（轻微）：复合赋值运算符自动生成假设的 fallback 工作量被低估

**问题描述**：
§3.5 复合赋值运算符段落（第449-451行）依赖仓颉编译器自动生成 `+=`, `-=`, `*=`, `/=`，并以"编码启动前验证项"的方式提议验证此假设。若验证失败（编译器不自动生成），则需要"在本设计中显式列出全部 36 个复合赋值运算符定义"。但"36 个"这个计数与实际需要量存在偏差：

- 复合赋值运算符需要为 9 种矩阵类型 × 4 种运算（+=, -=, *=, /=）定义，这是 36 个
- 但 *= 仅对 3 种方阵有效（非方阵乘法返回不同类型），方阵还需额外处理 /=(mat/mat)（依赖 inverse stub）
- 更关键的是，这些运算符分布在 9 个不同的 .cj 文件中（type_mat2x2.cj ~ type_mat4x4.cj），每个文件需要添加 4~6 个运算符重载，若手动定义将显著增加编码量

此问题的实质是：OOD 对语言特性的假设尚未验证，若假设不成立，编码阶段需要补充大量重复性工作。设计文档在风险量化上不够充分——仅给出了笼统的"36个"计数，未按文件分布给出具体影响范围。

**所在位置**：§3.5 "复合赋值运算符"段落（第449-451行）

**严重程度**：轻微（仅影响编码工作量估算，不影响设计正确性）

**改进建议**：
1. 在设计阶段主动验证此假设（而非留给编码阶段），可直接在项目现有测试框架中添加最小化验证用例
2. 无论验证结果如何，在 §8 产出物清单中补充复合赋值运算符的手动定义备选方案的文件分布表，使编码人员提前了解工作量

---

### 问题 3（中等）：B 类 fromMat 转换方向的行为契约不完整

**问题描述**：
§3.3 第6条提供了 72 个 fromMat 转换对的完整映射表（第298-310行），标注了 9 个"需核对 GLM .inl"的 B 类方向。但只有 Mat4x4 ← Mat4x2 这一个方向给出了具体的 GLM 偏差描述（第280-281行）。其余 8 个 B 类方向（Mat2x3 ← Mat2x2、Mat2x4 ← Mat2x2/Mat2x3、Mat3x3 ← Mat3x2、Mat3x4 ← Mat3x2/Mat3x3、Mat4x3 ← Mat4x2）仅标注了"需核对 GLM .inl"，未提供任何行为描述或偏差分析。

编码阶段需要逐一对照 GLM .inl 源文件，但不同方向的偏差模式可能不同（例如 Mat2x4 ← Mat2x2 和 Mat4x4 ← Mat4x2 的零初始化行为可能不同）。设计文档未给出这些场景的 GLM 实际行为分析，编码人员可能在实现 A 类方向（63 个）后才发现 B 类方向存在未预见的复杂行为，导致重新审视填充规则策略。

**所在位置**：§3.3 第6条 "fromMat 目标→源矩阵类型映射表"（第292-310行）及"已知偏离示例"（第280-281行）

**严重程度**：中等（影响编码效率，但不导致设计层面的错误）

**改进建议**：
1. 在映射表下方为每个 B 类方向补充 GLM .inl 的实际行为摘要（或至少列出对应 .inl 文件的行号范围），编码人员无需重复搜索定位
2. 如果时间不允许逐一验证，至少应补充说明：B 类方向的 GLM 行为是否与 Mat4x4 ← Mat4x2 的偏差模式一致（都是"零初始化列向量后设(j,j)=1"模式），还是各有不同模式
3. 将"先按 A 规则实现，再逐一修正 B 类方向"的策略中，为 B 类方向增加单元测试的验证要求，确保每个 B 类方向都有对应的 GLM 参考实现对照测试

### 问题 4（轻微）：Vec 类型 extend 块中矩阵乘法运算符编译顺序未明确

**问题描述**：
§7 D32（第797行）和 §3.5（第447行）决定将 `Vec{R} * Mat{C}x{R} → Vec{C}` 定义为 Vec 类型的成员运算符，即在 type_vecN.cj 中的 extend 块定义。但矩阵类型（Mat2x2~Mat4x4）是 Stage 2 新增的类型，type_vecN.cj 是 Stage 1 已有的文件。

在仓颉的包编译模型中，同包（glm.detail）内的跨文件类型引用不需要 import 语句。但 Vec extend 块中对 Mat2x2 等类型的引用是否要求 Vec 文件在矩阵类型文件之后编译（或仓颉编译器是否支持前向引用）？设计文档声明"同包类型直接可见，无需额外 import 声明"，但未讨论文件编译顺序依赖——如果 type_vecN.cj 文件的 extend 块引用了尚未编译的矩阵类型，仓颉编译器是延迟解析还是报错？

此问题不影响设计正确性，但编码阶段可能因此遇到编译错误，需要对文件编译顺序有明确了解。

**所在位置**：§3.5 "二元运算符（矩阵-向量）"（第440-448行）及 §7 D32（第797行）

**严重程度**：轻微（不影响设计，仅影响编码阶段的编译配置）

**改进建议**：
1. 在 D32 或 §3.5 中补充说明此实现方式的编译顺序要求：矩阵类型文件（type_mat*.cj）应在 Vec 类型的 extend 块之前编译，或在同一编译单元中同时编译
2. 若仓颉编译器在同包内支持前向引用（不依赖文件顺序），则需确认并在文档中注明

### 问题 5（轻微）：@Derive[Hashable] 对 Bool 的推导结论缺少对矩阵 T=Bool 的全面验证

**问题描述**：
§3.1（第183行）和 D17（第782行）论证矩阵类型支持 `@Derive[Hashable]` 时，依赖 Vec 类型已通过 T=Bool 的 Hashable 验证来证明"矩阵类型仅包含 VecN<T,Q> 成员，当 T=Bool 时安全可用"。

但这仅证明 `VecN<Bool, Q> <: Hashable`，未证明 `Mat{C}x{R}<Bool, Q>` 在 `@Derive[Hashable]` 下的编译正确性。仓颉中 `@Derive[Hashable]` 要求所有**成员字段类型**实现 Hashable。矩阵的成员字段类型是 `Vec{R}<Bool, Q>`（列向量类型）。问题在于：`@Derive[Hashable]` 的约束检查是在 struct 定义的泛型参数层面执行，还是在使用处的实例化时检查？

如果是 struct 定义时检查（即编译器检查 `Mat4x4<T, Q>` 中 `T` 的约束是否满足 Hashable 的要求），那么 `@Derive[Hashable]` 对于 `Mat4x4<T, Q>` 中的无约束泛型 T 会失败——因为编译器不知道 T 是否实现了 Hashable。如果是在实例化时（`Mat4x4<Bool, PackedHighp>`）检查，则 Vec<Bool,Q> 的 Hashable 足以满足。

设计文档的推理是假设"实例化时检查"，但未明确说明仓颉编译器对 `@Derive[Hashable]` 的检查时机。若仓颉采用"定义时检查"策略（即要求泛型类型参数有 Hashable 约束），则 `@Derive[Hashable]` 对矩阵不可行，需要降级方案。

**所在位置**：§3.1（第183行）"矩阵结构体"职责和 D17（第782行）

**严重程度**：轻微（可通过编码快速验证解决，需留意风险）

**改进建议**：
1. 将此项列入"编码启动前验证项"（§8 第839-841行），与 cjpm 子包验证并列
2. 若定义时检查失败，提供备选方案：不在矩阵类型上使用 `@Derive[Hashable]`，改为手动实现或移除 Hashable 支持（需评估下游影响）

### 问题 6（轻微）：§9 差异声明未记录 `const func length()` 与 C++ GLM 的行为差异

**问题描述**：
§3.1（第179行）将 `length()` 设计为 `const func length(): Int64` 返回列数 C。C++ GLM 的 `mat<C,R,T,Q>::length()` 返回 `C`（列数），两者行为一致。

但 C++ GLM 中 `length()` 是 `static constexpr length_type length()`——即**静态**成员函数（不需要实例），调用方式为 `mat4::length()`。仓颉版本（当前设计）为实例成员函数 `const func length(): Int64`，调用方式为 `m.length()`。

虽然这不是功能差异，但 API 调用方式不同（静态 vs 实例）可能导致迁移代码时需要修改调用方式。§9 差异声明（第851-881行）没有记录此差异。

**所在位置**：§3.1（第179行）及 §9 差异声明（第851-881行）

**严重程度**：轻微（用户可能很快习惯，但按照 §9 的编写意图——"每一项偏差均从用户视角出发"，此差异应当记录）

**改进建议**：在 §9 中新增条目："`length()` 为实例成员函数而非静态函数——C++ GLM 的 `mat4::length()` 为静态 constexpr，调用语法为 `mat4::length()`；仓颉版本 `length()` 为实例 const func，调用语法为 `m.length()`。行为等价"

### 整体质量评价

本 OOD 设计方案（v14）经过多轮审议迭代，整体质量较高。在技术可行性维度上已获组件 A 内部审议 APPROVED 结论，需求响应充分度良好——完整覆盖了 GLM 1.0.3 全部 9 个矩阵类型及其基本运算、构造体系、别名体系。深度和完整性总体满足编码启动要求，设计决策（D01~D32）条理清晰、理由充分。

以上 6 个问题中，问题 1（cjpm 子包风险）和问题 3（B 类填充规则行为不完整）对编码阶段的效率影响最大，建议在进入编码前补充验证和补充行为分析。其余 4 个问题属于完善性建议，不影响总体设计方向。
