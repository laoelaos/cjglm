## 发现的问题

### P1（严重）：复合赋值运算符自动生成假设未经验证

**问题描述**：§3.5（第423行）声明"复合赋值运算符：由仓颉编译器自动生成（当二元运算符返回类型与左操作数类型匹配时）"，并基于此假设推导出全套复合赋值运算符的可用性表（第425-432行第①~④类）。此假设在现有资料和阶段一代码中均无实证支持：阶段一 Vec 类型的实现代码中未定义任何复合赋值运算符（无 `+=`/`-=`/`*=`/ `/=`），若仓颉编译器自动生成，则 Vec 类型也应存在这些运算符——但它们实际不存在。参考的 C++ GLM 中所有复合赋值运算符均为显式定义（`type_mat2x2.inl:244-310`），不依赖编译器自动生成。

**所在位置**：§3.5 复合赋值运算符段落（第423-432行）

**严重程度**：严重

**改进建议**：在编码启动前通过最小化测试验证仓颉编译器是否自动生成复合赋值运算符：
- 创建包含 `operator func +(rhs: Self): Self` 的 struct，测试 `a += b` 是否编译通过
- 若自动生成成立，当前设计正确
- 若自动生成不成立，需在设计中显式列出全部复合赋值运算符，每矩阵类型至少 4 个（+=, -=, *=, /=），方阵额外包含 mat/mat 除法的 /=。总计影响约 9×4=36 个显式运算符定义

---

### P2（一般）：fromMat 第7条与第6条参数顺序不一致

**问题描述**：第6b条（§3.3 第266行）签名为 `fromMat<SrcT, SrcQ>(conv, m, one)`——conv 闭包在前、源矩阵在后；第7条（§3.3 第292行）签名为 `fromMat<U, P>(m, conv)`——源矩阵在前、conv 闭包在后。同一组 `fromMat` 工厂函数族内参数顺序不统一，编码时容易因混淆参数顺序产生编译错误或运行时逻辑错误。第6b条与第7条在功能上有重叠（同类型同尺寸时第6a条可替代第7条），但跨类型场景下两者参数顺序差异会增加编码者的认知负担。

**所在位置**：§3.3 第6b条（第266行）vs 第7条（第292行）

**严重程度**：一般

**改进建议**：统一参数顺序，建议统一为 `fromMat(conv, m, ...)` 模式（conv 在前，源矩阵在后），与 castVecN 的 `castVecN(v, conv)` 模式区分（向量转换是 v 在前 conv 在后，因为 castVecN 是自由函数而非成员函数；fromMat 是成员函数，conv 参数紧跟在矩阵参数前更符合"先转换后运算"的直觉）。参数顺序应在设计阶段锁定，避免编码阶段中途修改导致大范围返工。

---

### P3（一般）：fromParts 和 fromColumns 被 Number\<T\> 约束不必要地限制

**问题描述**：`fromParts(conv, ...)`（第238行）和 `fromColumns(conv, ...)`（第245行）定义在带 `Number<T>` 约束的 extend 块中。但这两个工厂函数的实现逻辑仅涉及 conv 闭包调用和赋值操作，不依赖 `Number<T>` 提供的任何运算（加、减、取负等）。`Number<T>` 约束使这两个函数对非 Number 类型（如用户自定义的结构体或泛型元组）不可用，而实际转换逻辑本身不需要此约束。

**所在位置**：§3.3 第4条 fromParts（第238行）、第5条 fromColumns（第245行）

**严重程度**：一般

**改进建议**：将 fromParts 和 fromColumns 移出 `Number<T>` 约束的 extend 块，放入单独的无约束 extend 块或 struct 体内（仅需 `Q <: Qualifier` 约束）。这样不会影响当前使用（Float32/Int32 等标准类型均有 Number 约束，仍然可用），但为后续扩展留出空间。如果仓颉不允许在 struct 体内定义 static 函数，则添加一个最小约束的 extend 块。

---

### P4（一般）：非方阵 identity(one) 与 filled(one) 的设计冗余待明确

**问题描述**：§3.3 第8条（第305行）和 D31（第768行）说明非方阵上 `identity(one)` 等价于 `filled(one)`。这是有意设计，但设计未讨论编码阶段如何避免重复实现——非方阵的 identity 调用 filled 直接返回，还是有各自的独立实现路径？若均为独立实现，则 6 种非方阵类型中 filled 和 identity 各实现一次，产生 12 个内部逻辑完全相同的函数，浪费编码工作量且产生维护隐患（修改 filled 时可能忘记同步修改 identity）。

**所在位置**：§3.3 第8条（第305行）、D31（第768行）

**严重程度**：一般

**改进建议**：在设计中明确非方阵 identity 的实现策略——推荐方案：非方阵的 `identity(one)` 内部委托 `filled(one)`（直接调用 return self.filled(one)）。或者统一为非方阵仅提供 `filled(one)` 而不提供 identity（与 D31 当前决策冲突——但需评估 API 统一性的价值是否大于实现冗余的成本）。

---

### P5（轻微）：fromMat 6a 完整泛型展开示例遗漏 SrcQ 约束

**问题描述**：第255行正式签名中包含 `where SrcQ <: Qualifier`，但第263行的完整泛型参数展开示例中仅写为 `public static func fromMat<SrcQ>(m: Mat2x2<T, SrcQ>, one: T): Mat4x4<T,Q>`，省略了 `where SrcQ <: Qualifier`。编码人员如果仅看第263行的展开示例而忽略第255行的正式签名，可能会在编码时遗漏此约束，导致编译失败。虽然在第14轮修订（P1）中已修复正式签名，但展开示例未同步更新。

**所在位置**：§3.3 第6a条完整泛型参数展开（第263行）

**严重程度**：轻微

**改进建议**：将第263行的展开示例补充 `where SrcQ <: Qualifier` 约束，与第255行正式签名保持一致。

---

### P6（轻微：需求响应充分度）：scalar_mat_ops.cj 的设计决策未与阶段一 OOD 的约定对齐

**问题描述**：D15（第752行）指出"阶段一 OOD 中约定'仅 scalar_vec_ops.cj 定义 add/sub/mul/div 包级函数'"，本阶段新增 scalar_mat_ops.cj 在同一包中定义同名函数，依靠参数类型重载区分。设计文档对此有详细说明并提供了区分理由。然而，阶段一 OOD 曾在 D15 中明确约定"仅 scalar_vec_ops.cj 定义这些函数"，本阶段的设计决策 D15 实质上推翻了阶段一 OOD 的约定。这不是逻辑错误，但反映出一个过程问题：阶段范围的界定和跨阶段约定的管理机制在 OOD 层面未明确。对于编码阶段来说，需要在实现 scalar_vec_ops.cj 时确保矩阵类型的全局函数重载不会产生歧义或意外重载（例如当某个未来新增类型恰巧可同时匹配 Vec 和 Mat 参数模板时）。

**所在位置**：§7 D15（第752行）、§3.5 scalar_mat_ops.cj 段落（第370-377行）

**严重程度**：轻微

**改进建议**：在设计中补充参数类型重载解析的验证建议——验证现有 `add<T,Q>(s: T, v: Vec2<T,Q>)` 与新增 `add<T,Q>(s: T, m: Mat2x2<T,Q>)` 在仓颉重载解析中的确定无歧义性（推荐编码启动前做最小化测试验证）。

---

### P7（轻微）：从Mat2x3实际填充行为与GLM参考实现可能存在局部差异的注意事项不够具体

**问题描述**：§3.3 第278行提到"编码阶段须逐一对照 GLM .inl 源码实现各转换工厂函数，而非机械套用主规则"。这本质上是正确的风险提示，但缺乏具体列出已知偏差场景。从实际 GLM 源码看（`type_mat2x2.inl:121-215`），所有跨矩阵转换构造实现均为逐列复制 `col_type(m[0])`、`col_type(m[1])`——GLM 的内部实现是逐列通过列向量构造函数完成类型转换和维度适配的，与设计文档中描述的"填充规则"精神一致。设计文档的警告虽正确，但过于笼统，编码人员无法预判哪些场景可能发生偏离。

**所在位置**：§3.3 第6条'编码阶段注意事项'（第278行）

**严重程度**：轻微

**改进建议**：在警告后补充一个具体的偏离示例（如设计文档自身分析的 Mat4x4 ← Mat4x2 场景）作为编码人员的参考模板，帮助其理解何种情况下会产生偏离。

---

## 综合评估

**需求响应充分度**：产出充分覆盖了阶段二路线图要求的全部范围——9 个矩阵类型、运算符重载、ext/ 别名文件、fwd.cj 矩阵别名、3 个 stub 文件。设计文档在多个版本迭代中已与阶段一现有代码风格对齐。

**深度和完整性**：设计文档工程化程度高，包含设计决策记录、偏差声明、行为契约示例、错误处理策略、并发设计和产出物清单。但存在两处深度不足：（1）复合赋值运算符的自动生成假设未验证（P1），这是编码阶段的核心阻断风险；（2）fromMat 参数顺序不一致（P2）可能在编码后期导致返工。

**编码启动前需验证项**：
1. 复合赋值运算符是否自动生成（P1）
2. `static func` 在 extend 块中的可用性（低风险，Cangjie 文档已列明为有效，但无实际示例）
3. cjpm 子包 glm.ext 的构建可行性（设计文档已列出验证步骤）
4. 包级函数重载解析（scalar_mat_ops.cj 与 scalar_vec_ops.cj 的同名函数参数区分）
