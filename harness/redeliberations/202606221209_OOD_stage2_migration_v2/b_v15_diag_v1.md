## 发现问题

### F1. fromMat 9×8 转换矩阵表存在多处标签错误

**问题描述**：§3.3 的 9×8 fromMat 转换矩阵表（L350-360）中多个单元格的标签与实际的维度变化关系不符，具体包括：

| 方向 | 实际变化 | 表中标签 | 错误说明 |
|------|---------|---------|---------|
| Mat2x2→Mat3x2 | C:2→3, R:2→2 | `C: colExt(3)→rowSh→*` | R 未变化，不存在 rowShrink 操作 |
| Mat2x2→Mat4x2 | C:2→4, R:2→2 | `C: colExt(4)→rowSh→*` | 同上 |
| Mat3x2→Mat4x2 | C:3→4, R:2→2 | `C: colExt(4)→rowSh→*` | 同上 |
| Mat2x3→Mat2x2 | C:2→2, R:3→2 | `C: rowSh→colSh` | C 未变化，"C:" 前缀错误标注；且操作顺序为 row→col 与 L341-344 规定的 col→row 顺序矛盾 |
| Mat2x4→Mat2x2 | C:2→2, R:4→2 | `C: rowSh→colSh` | 同上（C 未变化，"C:" 错误） |
| Mat2x4→Mat2x3 | C:2→2, R:4→3 | `C: rowSh` | C 未变化，"C:" 错误 |
| Mat3x3→Mat3x2 | C:3→3, R:3→2 | `C: rowSh→colSh` | C 未变化，"C:" 错误 |
| Mat3x4→Mat3x2 | C:3→3, R:4→2 | `C: rowSh→colSh` | 同上 |
| Mat3x4→Mat3x3 | C:3→3, R:4→3 | `C: rowSh` | C 未变化，"C:" 错误 |
| Mat4x3→Mat4x2 | C:4→4, R:3→2 | `C: rowSh→colSh` | C 未变化，"C:" 错误 |
| Mat2x2→Mat4x4 | C:2→4, R:2→4 | `C→R→⛔` | ⛔ 仅针对 Mat4x2→Mat4x4 偏差方向，Mat2x2→Mat4x4 不涉及特殊处理 |

**所在位置**：§3.3，L350-360（9×8 转换矩阵表）

**严重程度**：**严重**

**改进建议**：逐方向验证并修正表内标签，确保与维度变化及执行规则一致：
- 列/行同时无变化 → `EQL`
- C 不变仅 R 扩展 → `B: rowExt(R_dst)`
- C 不变仅 R 收缩 → `rowSh`（无"C:"前缀）
- C 变化 R 不变 → `colExt(C_dst)` 或 `colSh`（无冗余 rowSh）
- C 变化 R 收缩 → `colOp→rowSh`（colOp=colExt 或 colSh）
- C 变化 R 扩展 → `colOp→rowExt`
- 偏差方向单独标注 `🚨 **DEVIATION**`
- 移除 Mat2x2→Mat4x4 的 ⛔ 标注

### F2. fromMat 执行顺序规定与实际标注不一致

**问题描述**：L341-344 明确将执行顺序设定为"先列维度操作、再行维度操作"，但转换矩阵表中的多个方向（如 Mat2x3→Mat2x2、Mat2x4→Mat2x2、Mat3x3→Mat3x2、Mat3x4→Mat3x2、Mat4x3→Mat4x2）被标注为 `rowSh→colSh`（先进行收缩、再进行列收缩），与规定顺序相反。

虽然在这些特定方向上 C 实际未变化（colSh 为零操作），顺序对结果无影响，但符号约定不一致会严重损害表格的可信度和可读性，编码者无法依赖表格判断操作序列。

**所在位置**：§3.3，L341-344 vs L350-360

**严重程度**：中等

**改进建议**：统一所有方向的操作标注均遵循"先列后行"顺序。对 C 无变化的纯行收缩方向，标注为 `rowSh` 即可，无需 `rowSh→colSh`。

### F3. `transpose` 函数使用了不必要的 `Number<T>` 约束

**问题描述**：§3.7 matrix.cj 中所有 9 个 `transpose` 重载的签名约束为 `where T <: Number<T>, Q <: Qualifier`。但 `transpose` 仅是元素的索引重排（位置交换），不涉及任何算术运算——实现描述（L522）也确认"转置索引映射可在代码中硬编码展开，无需循环"。因此 `Number<T>` 约束是不必要的，且不正确地排除了 Bool 矩阵的转置操作。

GLM 1.0.3 的 `transpose` 对任意类型 T 均可用，不限于数值类型。以 `Number<T>` 约束该函数与 GLM 行为不一致。

**所在位置**：§3.7，L483-491（transpose 签名清单）

**严重程度**：中等

**改进建议**：将 `transpose` 的约束从 `where T <: Number<T>, Q <: Qualifier` 放松为 `where Q <: Qualifier`（无需 `Number<T>`），并在 §9 差异声明中备注此放宽对 Bool 矩阵的兼容性影响（如果有）。

注意：`matrixCompMult` 和 `outerProduct` 需要 `Number<T>`（涉及乘法运算），无需修改。

### F4. common.cj stub `mod` 签名约束与实际实现需求不匹配

**问题描述**：§3.7 common.cj stub 中 `mod` 函数的签名为 `where T <: Number<T>`。但根据既有偏差记录（`deviations.md` CL-04, CL-10），`Number<T>` 接口不提供 `%` 运算符（仅 `Integer<T>` 提供），同样不提供 `fmod`/`trunc`（这些是具体浮点类型的重载自由函数）。因此 `Number<T>` 约束下的 `mod` 函数体实际上无法在泛型上下文中实现——整数版本需要 `Integer<T>`，浮点版本需要具体类型重载。

虽然 stub 阶段的函数体仅为 `throw Exception("stub")`，约束不会导致编译错误，但应确保 stub 签名与预期最终实现的签名一致，否则后续阶段实现时需额外修改签名。

**所在位置**：§3.7 common.cj 函数签名清单，L471：`func mod<T>(a: T, b: T): T where T <: Number<T>`

**严重程度**：一般

**改进建议**：在 §3.7 的 `mod` 签名旁添加注释说明"此签名为暂定，最终实现需拆分为整数版本（`Integer<T>` 约束）和浮点具体类型重载（参考 scalar_vec_ops.cj 模式）"。

### F5. fromMat 6a 的 `one: T` 参数在跨 Qualifier 方向上的语义未明确

**问题描述**：L273 的 6a 签名示例中，`one: T` 参数表示 T(1) 值。当源矩阵的 Qualifier 与目标矩阵不同（如 `PackedLowp→PackedHighp`）时，`one: T` 作为显式传入的 T 类型值，其 Qualifier 语境是目标的 Qualifier。文档未明确说明此语义——编码者可能错误地以为 `one` 来自源上下文。

**所在位置**：§3.3 item 6a，L272-276

**严重程度**：一般

**改进建议**：在 6a 后补充一句话明确 `one` 的类型是目标元素类型 T（非源类型 U），其值由调用方根据目标矩阵的数值精度选取。

### F6. 矩阵乘法的 `operator *(r: RHS): ResultType` 未声明 `@OverflowWrapping`

**问题描述**：L389-422 列出了全部 27 个矩阵乘法重载的签名表，指出每个"定义为左矩阵类型的成员运算符"。§5 错误处理策略（L598-610）声明"以下所有算术运算符均标注 @OverflowWrapping"，其中包含"矩阵-矩阵同尺寸 +-*"和"矩阵-矩阵跨尺寸 *"。但 §3.5 的签名表中未在任何重载上标注 `@OverflowWrapping`，也未说明该注解将加在 extend 块级别还是函数级别（现有阶段一代码是在函数级别标注的）。

编码者无法从 §3.5 直接获知每个乘法重载是否需要 `@OverflowWrapping` 标注，需要对照 §5 才能确定。

**所在位置**：§3.5 矩阵乘法签名表 L390-422；§5 溢出策略 L598-610

**严重程度**：一般

**改进建议**：在 §3.5 乘法签名表上方或图例中明确标注"所有乘法运算符标注 @OverflowWrapping"，或每个重载后标注 `@OverflowWrapping`。

### F7. `C→R→→` 标注对行收缩方向有误导性

**问题描述**：转换矩阵表中，多个行列同时变化且行数减少的方向（如 Mat2x3→Mat3x2：R:3→2、Mat2x3→Mat4x2：R:3→2、Mat2x4→Mat3x2：R:4→2 等）被标注为 `C→R→→`。但图例中 `R→→` 的解释为"行扩展然后..."（L362），暗示行数增加。实际这些方向执行的是行收缩（rowShrink），并非行扩展。`C→R→→` 标注无法区分行扩展与行收缩。

**所在位置**：§3.3 转换矩阵表 L352-358 的 `C→R→→` 单元格，以及图例 L362-363

**严重程度**：中等

**改进建议**：
- 图例中补充符号：`C→R←←` 或 `C→rowSh` 用于行收缩的组合方向
- 表中对行收缩的组合方向使用与行扩展不同的标注，避免歧义

---

### F8. `fromMat 6a/6b` 的 144 个函数签名构造量未作为实施风险明确标注

**问题描述**：文档在 L285（04_ood_phase2.md 对应内容）提到"每个目标矩阵类型的 extend 块中将包含 8 个 fromMat（6a）和 8 个 fromMat（6b）共 16 个静态函数签名"，但未将其量化为 144 个签名（9 目标 × 8 源 × 2 变体 = 144），也未评估此大量签名的可维护性影响。

144 个签名中每个的函数体逻辑虽然相似（统一 match+length 模式），但每个都是独立的泛型重载。Cangjie 编译器按需实例化不产生代码膨胀，但 144 个签名的声明和维护对编码者构成显著的编辑负担。编码者需要编写和审查 144 个独立的函数签名声明。

**所在位置**：§3.3 item 6（L267-370）；§8 L650（少量提及"验证项"）

**严重程度**：一般

**改进建议**：在 §3.3 item 6 末尾或 §8 的验证项中新增一条明确说明：
- 统计 144 个 fromMat 6a/6b 签名总数
- 建议编码阶段考虑使用宏或代码生成辅助工具
- 评估拆分为多个 extend 块（按源矩阵分组）的可读性方案

### F9. `diagonal()` 工厂函数缺少 Bool 矩阵不支持的标注

**问题描述**：L236 标注了 "Bool 矩阵不提供此操作，见 D33"。但 D33（L633）只列出了 "Bool 矩阵不提供算术运算符、标量填充工厂函数和 equalEpsilon"，未明确提及 `diagonal()`。`diagonal()` 函数是 L231 定义的"extend 块工厂函数，Number<T> 约束"。Bool 不实现 Number<T>，所以确实不支持 `diagonal()`。但需要确保 §9 差异声明和 D33 中显式涵盖 `diagonal()`。

**所在位置**：§3.3 item 3（L236 标注）；L633 D33 声明

**严重程度**：一般

**改进建议**：在 D33 的 Bool 矩阵不支持列表中显式补充 `diagonal()` 和 `identity()`（两者均依赖 `Number<T>` 约束），使 D33 与 §3.3 的标注保持一致。

### F10. 从 `v * m` 退化为 `mulRowVecMat(v, m)` 的迁移影响未评估

**问题描述**：§8 fallback 方案一（L660-661）描述了当同包延迟解析不可用时，行向量×矩阵乘法将从运算符 `v * m` 退化为全局函数 `mulRowVecMat(v, m)`。文档注明"API 破坏性最大，因此作为第三备选"。但未评估此变更对已有阶段一代码的影响——若阶段一已有 `v * m` 用法的测试代码或 demo 代码，阶段二实施时需要全部迁移。同样也未评估方案二和方案三之间的升降级条件。

**所在位置**：§8 fallback 方案说明 L658-673

**严重程度**：一般

**改进建议**：补充方案升降级的具体判定标准（如"原型验证失败后按方案二→方案三→方案一的顺序尝试，每步验证编译通过"，并提供各方案之间的迁移路径说明）。

## 整体质量评价

本设计文档经过 14 轮迭代后，在核心内容（构造函数体系、算术运算符、stub 函数库、GLM 偏差分析）上已达到较高成熟度。主要的 B 类方向 GLM 对照分析和 Mat4x4←Mat4x2 偏差验证是高质量的。

但 **9×8 fromMat 转换矩阵表的标签准确性存在严重问题**（F1），直接影响该表作为"编码者可直接判断每个方向实现策略"的核心指引的可信度。此外，`transpose` 约束过严（F3）和 fromMat 操作顺序不一致（F2）是需要修正的实质性设计缺陷。其余问题（F4-F10）为完整性补充和风险提示性质。

建议在输出前修正 F1-F3 的问题后，再交付编码阶段。
