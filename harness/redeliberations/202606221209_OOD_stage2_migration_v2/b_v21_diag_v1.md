## 问题清单

### Q1：`@Derive[Hashable]` 与手动 `==` 的哈希一致性未纳入编码验证清单

- **所在位置**：§3.1（L187，标注 `@Derive[Hashable]`）、§3.6（L502-503，手动定义 `==`）、§8（L717，编码启动前验证项）
- **问题描述**：设计使用 `@Derive[Hashable]` 自动派生哈希函数，同时在 Equatable<T> 约束的 extend 块中手动定义 `==` 为逐列比较。仓颉语言要求哈希一致性契约：`a == b` 必须蕴含 `a.hash() == b.hash()`。若 `@Derive[Hashable]` 自动生成的 `hash()` 与手动定义的 `==` 采用不同的等价判断标准，将导致 `HashMap<K,V>`/`HashSet<K>` 等容器运行时行为异常（如 `contains` 误判）。虽然阶段一 Vec 类型已有先例，但矩阵类型因嵌套层级更深（矩阵→列向量→标量分量），哈希器链路更长，风险链路不同，不能直接假定安全。
- **严重程度**：一般
- **改进建议**：
  1. 将 `@Derive[Hashable]` 编译验证从当前仅标注"T=Bool 编译验证"扩展为覆盖所有元素类型（Float32、Float64、Int32、Int64 等），纳入§8 编码启动前验证清单
  2. 在编码验证中增加哈希一致性测试：构造相等的矩阵对（如通过构造函数链确认 `==` 返回 true），验证 `hash()` 返回值相同

### Q2：`fromMat 6a/6b` 中对 `one: T` 参数在纯收缩方向被忽略的调用方可发现性不足

- **所在位置**：§3.3（L286，纯收缩方向的 `one` 参数行为说明段落）
- **问题描述**：设计明确表示纯收缩方向下 `one` 参数被忽略但保留以保持 API 统一性。然而调用方无法通过类型系统获知这一行为——`one: T` 是必需的签名参数，所有方向均需传入。对于纯收缩方向（如 Mat4x4→Mat2x2），调用方必须提供永远不会被使用的 `one` 值。设计当前仅在签名注释中标注，而注释在 IDE 自动补全和泛型代码生成场景下不易被调用方觉察。这可能导致调用方困惑（"为什么收缩也需要 one？"）或误解（"one 传入 0 会怎样？"）。
- **严重程度**：一般
- **改进建议**：在§3.3 中补充以下两种方案供编码阶段选择：
  - 方案 A：为纯收缩方向额外提供一组无 `one` 参数的重载（如 `fromMatTruncate<SrcQ>(m: Mat4x4<T,SrcQ>): Mat2x2<T,Q>`），但需评估重载膨胀成本（约 20 个额外签名）
  - 方案 B：在文档中明确标注纯收缩方向列表（C_dst ≤ C_src 且 R_dst ≤ R_src 的所有方向组合），并以具体示例（如 `fromMat(m4x4, one: 0.0f)` — 实际 `one` 不被使用）帮助调用方理解

### Q3：fwd.cj 别名排除清单与 GLM 1.0.3 fwd.hpp 的逐项对照验证缺失

- **所在位置**：§8（L771-773，别名总量统计和排除族声明）
- **问题描述**：设计声称排除"fmat/f32mat/f64mat 系列（约 81 个）及 i8mat~u64mat 整型系列（约 108 个）"，合计约 171 个。但 GLM 1.0.3 的 fwd.hpp（约 530 行矩阵别名声明）还包含 imat/umat 族、i8mat~i64mat 族、u8mat~u64mat 族等，每个族×3 精度等级 = 27 个别名/族（含 defaultp）。实际 GLM 定义的矩阵别名总量约 540+ 个（f32/f64/double × 3 精度 + int × 3 精度 + int8/16/32/64 × 3 精度 + uint8/16/32/64 × 3 精度，每族 9 尺寸 3 精度 = 27，约 20+ 族）。设计声称排除约 171 个，但排除后的保留别名（108 个）远小于 GLM 总量（540+），两者差距（432+）表明"排除"指的是排除整数族别名，而非总差量。设计应明确排除策略的精确覆盖范围。
- **严重程度**：一般
- **改进建议**：
  1. 在§10 的 `glm/fwd.hpp` 条目下方增加排除策略说明："本阶段仅实现 float32/float64 默认精度及短别名，共计 108 个；其余浮点精度变体（lowp/mediump/highp 前缀族）以及所有整数/无符号整数矩阵别名族均推迟至后续阶段"
  2. 提供一个简化的覆盖矩阵：GLM 各别名族 vs 本阶段覆盖状态（✓ 本阶段实现 / ○ 后续阶段）

### Q4：`col()` 函数的 mut 赋值版本签名未在设计中显式列出

- **所在位置**：§3.4（L411-415）
- **问题描述**：下标运算符 `[]` 的赋值版本被描述为"赋值版本需 mut 修饰符"，但 `col()` 函数仅描述了取值版本签名 `col(i: Int64)` 与 `[]` 等价，未提及 `col()` 是否提供对应的 mut 赋值版本（如 `mut col(i: Int64, value: VecR<T,Q>): Unit`）。从对称性出发，`col()` 应同时提供取值和赋值版本以保持与 `[]` 的行为完全等价（如设计所述"与 `[]` 完全等价"），但当前描述仅覆盖取值语义。编码实现者可能据此仅实现取值版本，导致与声明的"完全等价"不一致。
- **严重程度**：一般
- **改进建议**：
  1. 在§3.4 中补充 `col()` 的赋值版本签名：`mut col(i: Int64, value!: VecR<T,Q>): Unit`
  2. 或明确声明 `col()` 仅提供取值版本，赋值操作应通过 `[]` 进行

### Q5：§10 API 覆盖矩阵中 `glm/common.hpp` 的 `mod` 函数覆盖状态标注不精确

- **所在位置**：§10（L835，`mod` 行）
- **问题描述**：覆盖矩阵将 `mod` 标注为"本阶段 stub（标量）"，引用的§3.7 common.cj 确实提供了标量版本的 `mod` stub 签名。但§3.7 common.cj 中 `mod` 的签名注释明确指出"此签名为暂定。Number<T> 不直接提供 % 运算符（仅 Integer<T> 提供），也不提供 fmod/trunc。最终实现需拆分为整数版本（Integer<T> 约束）和浮点具体类型重载。"这意味着当前 stub 签名（`where T <: Number<T>`）在实际编译时可能因泛型约束不足而无法实现——对浮点类型 `%` 不可用，对整数类型 `Integer<T>` 约束不可与 `Number<T>` 互换。当前 stub 签名仅能占位，但编码阶段需要两套独立签名。API 覆盖矩阵应将此标注为"本阶段 stub（标量，签名暂定，编码阶段需拆分为 Integer<T> 和浮点重载）"以提醒实现者。
- **严重程度**：一般
- **改进建议**：同步更新§10 覆盖矩阵中 `mod` 行的覆盖状态说明，使其与§3.7 common.cj 中的注释一致
