根据以下审查结果，迭代上一轮的产出，形成新版的文件，从而更好地满足用户需求。

## 当前审查结果

### 问题1（中等）：`fwd.cj` 矩阵别名新增的具体内容未定义
§2 模块划分中标明 `fwd.cj ★ — 新增矩阵类型别名`，§8 产出物清单提到"修改文件：src/fwd.cj, src/lib.cj"，但全文中未定义**哪些**矩阵类型别名需要新增到 `fwd.cj`。现有的 `fwd.cj`（阶段一）包含 B/I/U/I8/I16/I32/I64/U8/U16/U32/U64/F32/F64 共 10+ 个元素类型家族的向量别名（每个家族含 4 种精度变体）。设计未说明矩阵别名是否覆盖全部家族（与向量别名一致）还是仅限基本浮点类型（mat2x2/mat3x3/mat4x2/mat4x4 等 × Float32/Float64）。缺少此清单，编码者无法确定 `fwd.cj` 的新增范围。
**改进建议**：明确列出 `fwd.cj` 中需要新增的所有矩阵类型别名。建议格式参考现有的向量别名模式（如 `public type Mat2x2 = detail.Mat2x2<Float32, PackedHighp>`），按元素类型家族和精度变体分组枚举。

### 问题2（中等）：stub 文件缺少函数签名清单
§3.7 仅提到"创建 common.cj、matrix.cj、geometric.cj 三个 stub 文件"，未提供每个 stub 需要声明的函数签名。编码者仅凭此信息无法知道每个 stub 文件应包含哪些函数。前序版本（`docs/04_ood_phase2.md`）中包含了完整的 stub 签名清单，但本设计未继承此细节。
**改进建议**：在每个 stub 文件的描述后附上完整的函数签名清单，包括函数名、参数类型、泛型约束和返回值类型。可参考 `docs/04_ood_phase2.md` §3.7 中已有的详细签名。

### 问题3（中等）：Vec 类型 extend 块中行向量×矩阵乘法运算符的具体签名未定义
§2 包组织中标注 `type_vecN.cj（修改）★ — Vec1~Vec4 各新增行向量×矩阵成员运算符 Vec{R} * Mat{C}x{R} extend 块（方案A）`，但未指定每个 Vec 类型需要新增哪些具体运算重载。每个 Vec{R} 类型只需要与行向量维数匹配的矩阵类型（即 Vec{R} 只能乘 Mat{C}x{R} 其中 R 匹配）。
**改进建议**：按 Vec1/Vec2/Vec3/Vec4 逐一列出每个类型需要新增的 `operator *` 签名。例如：
- Vec4 extend 块：`operator func *(m: Mat2x4<T,Q>): Vec2<T,Q>`、`operator func *(m: Mat3x4<T,Q>): Vec3<T,Q>`、`operator func *(m: Mat4x4<T,Q>): Vec4<T,Q>`（共 3 个）
- Vec3 extend 块：`operator func *(m: Mat2x3<T,Q>): Vec2<T,Q>`、`operator func *(m: Mat3x3<T,Q>): Vec3<T,Q>`、`operator func *(m: Mat4x3<T,Q>): Vec4<T,Q>`（共 3 个）

### 问题4（轻微）：Bool 矩阵类型的支持策略未明确
设计将所有算术运算符放在 `Number<T>` 约束的 extend 块中，但 Bool 类型不实现 `Number<T>` 接口。这意味着 `Mat2x2<Bool, Q>` 等 Bool 矩阵类型将无法使用算术运算符。设计未讨论矩阵类型对 Bool 的支持范围。
**改进建议**：在 §9 差异声明或 §7 设计决策中增加一条，明确说明 Bool 矩阵的支持范围。

### 问题5（轻微）：`equalExact` 函数是否需要对矩阵类型提供未明确
§3.6 比较运算符仅提到 ==/!= 和 equalEpsilon。阶段一 Vec 类型同时提供了 `equalExact`（精确比较的显式具名版本），矩阵类型是否也应提供未说明。
**改进建议**：补充说明 equalExact 的提供状态。若排除，在 §9 记录；若包含，补充成员函数签名定义。
