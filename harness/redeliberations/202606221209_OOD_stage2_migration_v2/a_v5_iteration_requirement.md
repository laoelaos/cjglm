### 问题 1（中等）：D08 及 §9 中 ++/-- 替代方案 `m += T(1)` 与 T(1) 约束矛盾
D08（第753行）和 §9（第837行）差异声明中，++/-- 的替代写法仍为 `m += T(1)`。但 §1 第30-37行系统性设计约束已明确论证 `Number<T>` 接口不提供 `T(n)` 构造声明，在泛型上下文中 `T(1)` 不可编译。v9修订说明修复了 `identity()` 签名中的同类问题，但 D08 和 §9 中的同名引用未被同步更新。

**改进建议**：将 `m += T(1)` 修正为与 §1 约束一致的表述——在具体类型上下文中使用 `m += 1.0f`（明确写出对应类型的字面量），或在泛型上下文中通过参数传递 `one: T` 后使用 `m += one`。D08 和 §9 两处需同时修正。

### 问题 2（中等）：§9 差异声明缺失模板赋值运算符 `operator=` 的差异条目
GLM 1.0.3 全部 9 个矩阵类型的 `.hpp` 文件均声明了模板赋值运算符 `template<typename U> operator=(mat<C,R,U,Q> const& m)`，支持跨类型同尺寸矩阵赋值。仓颉不允许重载 `operator=`，此能力完全不可用。§9 差异声明（第831-858行）未包含此条目，§3.5 运算符清单也未提及。用户从 C++ GLM 迁移代码时若使用 `mat4_f32 = mat4_i32` 将遇到编译错误且原因不明。

**改进建议**：在 §9 中新增条目"无模板赋值运算符 `operator=(mat<C,R,U,Q>)`"，说明跨类型同尺寸矩阵赋值不可用，推荐通过 `fromMat(conv, m)` 或显式逐元素转换替代。

### 问题 3（中等）：fromMat 工厂函数签名使用伪代码 `Mat{C_src}x{R_src}`，缺少具体的重载清单
§3.3 第6a/6b条签名为伪代码形式 `fromMat(<SrcQ>)(m: Mat{C_src}x{R_src}<T, SrcQ>, one: T)`，其中 `Mat{C_src}x{R_src}` 是概念性简写。实际仓颉代码中不存在此类型表达式，每个目标矩阵类型需要为 8 个不同的源矩阵类型分别手写重载。设计虽正确计数为 72 个转换对、144 个函数签名，但未提供任何目标→源映射表。编码人员需自行从概念规则推导全部 72 个具体重载，存在遗漏风险和填充规则复杂度差异。

**改进建议**：提供一个完整的"目标矩阵类型 → 8 个源矩阵类型"映射表（或等价代码骨架），标注每个方向的填充规则类别（简单截断/扩展 vs 需核对 GLM .inl 的复杂路径），消除编码阶段的推导遗漏风险。

### 问题 4（轻微）：fromParts/fromColumns 的泛型类型参数约束在文档中不完整
§3.3 第4条 `fromParts` 签名（第238行）和第5条 `fromColumns` 签名（第246行）未在签名行中显式给出 `where Q <: Qualifier` 约束——该约束仅在后续文字描述中出现。第7条 `fromMat<U, P>` 的签名行（第296行）完全未提及 `P <: Qualifier` 约束，仅通过"与第5条一致"间接传达。

**改进建议**：在上述三处签名行中补充 `where` 子句，确保签名行本身自包含完整的约束信息。
