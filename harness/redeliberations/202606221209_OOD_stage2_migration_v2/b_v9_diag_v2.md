## 问题 1（中等）— 填充规则描述"扩展列/行按单位矩阵填充"存在歧义，可能误导实现

**问题描述**：§3.3 item 6 的填充规则表述为"复制重叠区域，扩展列/行按单位矩阵填充"。经对照 GLM 1.0.3 源码验证，实际行为需要区分两种场景：

- **列扩展（C_dst > C_src）**：新列按单位矩阵列填充（对角线位置填入 T(1)，其余 T(0)）
- **行扩展（B 类方向，R_dst > R_src, C 不变）**：新增行元素填入 **T(0)**（零），而非单位矩阵值

当前表述将两种场景混为一谈，开发者若机械遵循"按单位矩阵填充"的字面含义，可能在行扩展场景中用 T(1) 填充新增行元素，导致 8 个 B 类方向的 fromMat 实现错误——与 GLM 行为不一致。

**所在位置**：
- §3.3 item 6 (L225)：填充规则描述 "复制重叠区域，扩展列/行按单位矩阵填充"
- §3.3 item 6a "精确填充规则（同类型 6a 版本）"：未区分列扩展与行扩展的差异

**严重程度**：中等 — 当前描述歧义可能在编码阶段导致系统性错误

**改进建议**：将填充规则拆分为两个明确表述：
- **列扩展**：新列在 `(新列索引, 新列索引)` 位置填入 `one`（=T(1)），其余位置填入 T(0)
- **行扩展**：新增行元素全部填入 T(0)（通过 `someValue - someValue` 演算）

同时在 B 类方向对照表（L241-254）后补充此拆分的确认说明，显式注明 B 类方向行扩展填充 T(0) 与主规则的对应关系。

## 问题 2（中等）— 非方阵 identity(one) 的语义未明确定义

**问题描述**：§3.3 item 8 说"单位矩阵工厂函数 identity(one: T) — 对所有 9 种矩阵类型均提供"，§9 说"identity 对非方阵也提供 — API 统一性设计"，但均未明确定义非方阵上 identity 的具体行为和计算结果。对于非方阵（如 Mat2x3、Mat4x2），"单位矩阵"概念本身不是数学上标准的——设计需要明确其行为，否则下游使用者无法获知非方阵 identity 的预期行为，编码者也缺少明确的实现规范。

**所在位置**：
- §3.3 item 8 (L239)：仅一句"单位矩阵工厂函数 identity(one: T) — 对所有 9 种矩阵类型均提供"，无非方阵语义说明
- §9 (L552)："identity 对非方阵也提供 — API 统一性设计"，同样无语义说明

**严重程度**：中等 — 缺少语义说明可能导致非方阵 identity 的实现与用户预期不一致

**改进建议**：在 §3.3 item 8 下补充非方阵 semantics 说明段，明确：
1. 非方阵 `identity(one)` 的行为为对角矩阵：主对角线上填 `one`、其余填 T(0)
2. 注意此行为与 `filled(one)` 结果一致（因为 `filled()` 在本设计中定义为对角填充，而非全元素填充）
3. 举例说明：`Mat2x3<Float32, PackedHighp>.identity(1.0f)` → 列 0 = Vec3(1,0,0)、列 1 = Vec3(0,1,0)

## 问题 3（中等）— fromMat 非 B 类方向（C 和 R 同时变化）的行为未与 GLM 对照验证

**问题描述**：§3.3 的 GLM 逐项对照分析仅覆盖了 B 类方向（C_src=C_dst, R_src<R_dst），共 9 个方向。但 fromMat 6a 包含 72 个转换方向，其中大量方向涉及 C 和 R 同时变化（如 Mat2x2→Mat3x4、Mat4x4→Mat2x3 等）。这些方向的行为未经验证是否与 GLM 一致，且文档中"编码阶段须逐一对照 GLM .inl 源码"的建议将验证责任完全推给了编码阶段。

从已验证的 Mat4x4←Mat4x2 偏差案例可以看出，GLM 的部分转换构造函数并非遵循"复制重叠，单位矩阵填充新列/行"的统一规则——Mat4x4←Mat4x2 中 GLM 丢弃了源矩阵列 2、3 的全部数据，即使这些列在目标矩阵中"有对应位置"。这种模式偏差可能也存在于其他未验证的方向中。

**所在位置**：
- §3.3 B 类方向对照分析表 (L241-259)：仅覆盖 9 个 B 类方向
- §3.3 item 6 末尾的编码阶段提醒：将验证责任推给编码阶段

**严重程度**：中等 — 未经验证的转换方向可能包含更多 GLM 偏差，编码阶段发现后需要回溯修改设计

**改进建议**：建议至少补充以下几组验证方向（而非全部 72 个）：
1. **C 和 R 同时扩展**（如 Mat2x2→Mat3x4、Mat2x2→Mat4x3）：验证 GLM 是否遵循"复制重叠 + 新列单位矩阵"规则
2. **C 收缩 R 扩展**（如 Mat4x2→Mat2x4、Mat3x2→Mat2x3）：这是 Mat4x4←Mat4x2 偏差的"缩小版"，可能也存在类似偏差
3. **C 扩展 R 收缩**（如 Mat2x4→Mat3x2）：验证重叠区域复制后超出行为是否合理

若已验证，可在报告中补充验证编号以增加完整性；若尚未验证，则应标注为"待验证"而非"编码阶段验证"。

## 问题 4（轻微）— §8 编译顺序 fallback 方案一（全局自由函数）未讨论 API 契约变化对下游的影响

**问题描述**：§8 为"行向量×矩阵乘法"的编译顺序问题提供了三个 fallback 方案。方案一将 `operator *` 替换为全局自由函数（如 `vec4_mul_mat4x4(v, m)`），但此方案完全改变了该操作的调用语法——从 `v * m` 变为函数调用形式，破坏了「运算符重载」的 API 契约。GLM 用户习惯 `v * m` 的行向量×矩阵乘法的语法，fallback 方案一的语法变化对下游代码的影响未被评估（如需要全局搜索替换所有行向量×矩阵乘法的调用点）。

§8 仅描述方案一的实现方式，未讨论其对用户代码的破坏性影响及迁移成本。

**所在位置**：§8 "compile order 说明"节，方案一描述 (L488)

**严重程度**：轻微 — 只是 fallback 方案，且方案一非首选；但缺少影响评估可能导致编码阶段选择方案一时未充分告知下游

**改进建议**：在方案一的说明后补充一句话注明 API 变化影响：全局自由函数方式将使用 `vec_mul_mat(v, m)` 而非 `v * m` 语法，与 C++ GLM 语法不一致，用户迁移时需修改所有行向量×矩阵乘法的调用点。同时建议在方案一的函数命名上使用运算符风格的具名（如 `mul_row_vec_mat` 而非 `vec4_mul_mat4x4`），提升可读性。

## 问题 5（中等）— §2 中 matrix.cj 的"stub"标签与 §3.7 中"附带提供"的承诺自相矛盾

**问题描述**：§2 包组织中，matrix.cj 被标注为"stub（仅函数签名空壳，供 type_mat2x2/type_mat3x3/type_mat4x4 依赖闭合；以及附带提供的额外矩阵运算函数）"。但 §3.7 中详细列出了 matrix.cj 的函数签名清单，包括 transpose（9 个重载）、determinant（3 个）、inverse（3 个）、matrixCompMult（9 个）、outerProduct（9 个），共计 33 个函数签名。

其中 **transpose**、**matrixCompMult**、**outerProduct** 这三类函数（共 27 个重载）的实现完全不依赖外部库——它们可以通过简单的索引映射和算术运算直接实现。将这类有明确数学定义、可直接实现的函数列为"stub（空壳）"与 §2 的标签矛盾。如果这些函数在本阶段真的以空壳形式存在，则阶段二完成后这些基本运算不可用，下游消费者无法使用矩阵转置等核心功能。

**所在位置**：§2 包组织 matrix.cj 条目 (L57) vs §3.7 matrix.cj 函数签名清单 (L322-364)

**严重程度**：中等 — 设计自相矛盾，编码者无法判断这些函数是直接实现还是留空

**改进建议**：在 §2 matrix.cj 的描述中标明哪些函数是直接实现（transpose、matrixCompMult、outerProduct），哪些是 stub 空壳（determinant、inverse）。同时建议对可直接实现的函数补充简要的实现说明，提升编码阶段的指导性。

## 问题 6（轻微）— fromMat 6a 中同 T 不同 Q 的跨尺寸转换缺口未文档化

**问题描述**：§3.3 item 6a 的签名要求源和目标矩阵的 `T` 和 `Q` 完全相同（`fromMat(m: Mat2x2<T,Q>, one: T)`）。这意味着当用户需要跨 Qualifier 转换且尺寸不同时（如 `Mat2x2<Float32, PackedLowp> → Mat2x3<Float32, PackedHighp>`），必须绕道使用 6b 并传入恒等 conv 闭包：`fromMat({x => x}, source, one: 1.0f)`。这与直接用 6a 的自然写法 `fromMat(source, one: 1.0f)` 相比，API 体验有明显差异。

此缺口未在 §9 的差异声明中记录。虽然 6b 路径可行，但用户首次遇到时难以发现。

**所在位置**：§3.3 item 6a (L228)、§9 差异声明 (L554)

**严重程度**：轻微 — 有 workaround（用 6b + identity conv），不影响功能完整性

**改进建议**：在 §9 的 fromMat 差异条目中补充此缺口说明，或提供另一个 6c factory 支持同 T 不同 Q 的场景。
