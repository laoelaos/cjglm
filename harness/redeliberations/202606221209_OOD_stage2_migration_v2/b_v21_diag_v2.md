## 问题清单

### Q1（修订）：`@Derive[Hashable]` 跨元素类型编译验证未纳入编码清单

- **所在位置**：§3.1（L187，标注 `@Derive[Hashable]`）、§8（L717，编码启动前验证项）
- **状态**：**已降级**（回应质询：接受"推测性论证"批评，从设计质量缺陷降级为编码阶段验证建议）
- **问题描述**：v1 报告断言"矩阵嵌套层级更深导致哈希一致性风险链路不同"，但该论证基于推测而非实际证据。仓颉 `@Derive[Hashable]` 对 struct 所有字段递归求哈希，手动 `==` 逐列委托 Vec 的 `==`，两者使用字段级等价判断标准一致，理论上嵌套深度不改变数学性质。阶段一 Vec 类型已成功使用相同模式。因此不应将此标记为设计质量缺陷。
- **严重程度**：编码阶段建议（非设计质量缺陷）
- **改进建议**：在 §8 编码启动前验证清单中，将 `@Derive[Hashable]` 的编译验证从"T=Bool"扩展为覆盖所有目标元素类型（Float32、Float64、Int32、Int64），验证自动生成的 hash() 能通过编译且与手动 == 行为无冲突。此验证为常规编译器行为确认，无需预判风险。

### Q2（修订）：纯收缩方向 `one` 参数的文档强化建议

- **所在位置**：§3.3（L286，纯收缩方向的 one 参数行为说明段落）
- **状态**：**已降级**（回应质询：接受"API 文档偏好问题，20 额外重载比例失衡"批评，移除 API 改造建议）
- **问题描述**：设计明确方文档说了纯收缩方向下 `one` 参数被忽略但保留签名以保持统一。质询方指出"注释可见性不足"是 API 文档偏好而非设计质量缺陷，且建议 20 个额外重载解决问题比例失衡。此判断被接受。
- **严重程度**：编码阶段建议（非设计质量缺陷）
- **改进建议**：在现有说明段落基础上，增加一个使用示例帮助调用方理解：在纯收缩方向转换的示例代码中显式传递任意 dummy 值（如 `fromMat(m4x4, one: 0.0f)` 并注明 one 不会被使用）。无需创建额外重载。

### Q3（修订）：排除别名族的覆盖说明缺乏精确性

- **所在位置**：§8（L771-773，别名总量统计和排除族声明）、§10（L876-881，GLM 别名覆盖矩阵）
- **状态**：**已修订**（回应质询：接受"数字未经核实"批评，以下数据已对照 GLM 1.0.3 fwd.hpp 文件验证）
- **问题描述**：
  - v1 报告自身使用的数字（"约 530 行"、"每族 27 别"）经核实与 GLM 1.0.3 fwd.hpp 事实不符：矩阵别名节为 L508-L1203（约 696 行），每个命名家族有 4 精度等级（lowp/mediump/highp/defaultp）× 9 尺寸 = **36 个别名/族**（而非 27）。质询成立，v1 报告的证据充分性存在瑕疵。
  - 核心关注点不变：设计文档 L773 声称"fmat/f32mat/f64mat 系列（约 81 个）及 i8mat~u64mat 整型系列（约 108 个）"的排除计数仍明显偏离 GLM 实际：
    - fmat + f32mat + f64mat 三族 = (12+36) + (12+36) + (12+36) = **144 个**（设计称"约 81"）
    - 整数家族有符号/无符号各 5 族 × 每个家族 48 个别名（12 NxN + 36 MxN）= **480 个**（设计称"约 108"）
  - 此处数字差异不构成技术错误（排除族本身是正确的），但"约 81"和"约 108"与 GLM 实际差距过大（分别差 63 和 372），读者无法据此建立真实覆盖印象。API 覆盖矩阵（L876-881）仅用"3 精度"概括 GLM 别名层次（实际 4 精度），存在同样的精度不足问题。
- **严重程度**：一般
- **改进建议**：
  1. 将 L773 的排除统计替换为精确数字（基于 fwd.hpp 的实际 typedef 统计）或改为定性描述（"fmat/f32mat/f64mat 精度系列及所有整数矩阵系列"）
  2. §10 覆盖矩阵中 GLM 别名族的精度等级计量从"3 精度"修正为"4 精度"

### Q4（保留）：`col()` 函数的 mut 赋值版本签名未在设计中显式列出

- **所在位置**：§3.4（L411-415）
- **状态**：保留，未受质询挑战
- **问题描述**：下标运算符 `[]` 被描述为拥有取值和赋值双版本。`col()` 被描述为"与 [] 完全等价"，但仅提供了 `col(i: Int64)` 的取值签名形式，未提及对应的 mut 赋值版本签名。实现者可能据此仅实现取值版本，导致与声明的"完全等价"不一致。
- **严重程度**：一般
- **改进建议**：
  1. 在 §3.4 中补充 `col()` 的赋值版本签名
  2. 或明确声明 `col()` 仅提供取值版本，赋值操作应通过 `[]` 进行

### Q5（保留）：§10 API 覆盖矩阵中 `mod` 函数的覆盖状态标注不精确

- **所在位置**：§10（L835，`mod` 行）
- **状态**：保留，未受质询挑战
- **问题描述**：覆盖矩阵将 `mod` 标注为"本阶段 stub（标量）"。但 §3.7 common.cj 的 `mod` 签名注释明确指出"此签名为暂定，最终实现需拆分为整数版本（Integer<T> 约束）和浮点具体类型重载"。当前 stub 签名（`where T <: Number<T>`）在实际编译时可能因泛型约束不足而无法实现——对浮点类型 `%` 不可用，对整数类型 Integer<T> 约束不可与 Number<T> 互换。覆盖矩阵未反映此风险。
- **严重程度**：一般
- **改进建议**：同步更新 §10 覆盖矩阵中 `mod` 行的覆盖状态说明，使其与 §3.7 common.cj 中的注释一致，例如标注"本阶段 stub（标量，签名暂定，编码阶段需拆分为 Integer<T> 和浮点重载）"

### Q6（新增）：`equalExact` 函数的 NaN 语义在 §3.6 与 §5 之间存在未对齐风险

- **所在位置**：§3.6（L503-504，equalExact 描述）、§5（L664，equalEpsilon NaN 行为描述）
- **状态**：新增
- **问题描述**：§3.6 将 `equalExact` 描述为"与 == 行为语义等价"，而在 §3.6 的 NaN 比较行为段落中（L509），明确说明了 `==` 和 `equalExact` 在 NaN 上返回 false，`!=` 返回 true。但 §5 错误处理策略（L664）仅覆盖了 `equalEpsilon` 的 NaN 行为，未提及 `equalExact`。虽然从 §3.6 的声明可以推导出 `equalExact` 的 NaN 行为与 `==` 一致（对 NaN 返回 false），但设计文档未在 §5 中为 `equalExact` 做相同级别的 NaN 行为显式声明，在错误处理策略层面形成了一个文档缺口。
- **严重程度**：一般
- **改进建议**：在 §5 中补充"equalExact NaN 行为"条目，明确声明"equalExact 对含 NaN 分量的矩阵返回 false，与 == 的 NaN 语义一致（详见 §3.6）"，与 equalEpsilon 的 NaN 声明保持对称
