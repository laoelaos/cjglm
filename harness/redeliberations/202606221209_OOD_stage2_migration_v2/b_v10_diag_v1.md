## 发现的问题

### 问题 1：`fromParts` 家族签名模板的参数数量存在结构矛盾

- **所在位置**：§3.3 item 4，Mat4xR 族签名示例
- **严重程度**：中等
- **问题描述**：

  Mat4xR 族的签名模板为：
  ```
  public static func fromParts<U, P>(conv: (U) -> T, a00..a03: U, a10..a13: U, a20..a23: U, a30..a33: U): Mat4xR<T,Q>
  ```
  该模板固定展示 16 个分量参数，但注释说明"R=2 时 8 个分量、R=3 时 12 个、R=4 时 16 个"。这意味着 Mat4x2、Mat4x3、Mat4x4 各自需要不同参数数量的独立签名。然而仓颉不支持条件编译参数数量，`fromParts` 必须在每个矩阵类型上独立定义——Mat4x2 需要 8 参数版本、Mat4x3 需要 12 参数版本、Mat4x4 需要 16 参数版本。该模板传递给编码实现者的信息是矛盾的：无法用一个签名同时适配三种参数数量。

  Mat2xR 族的签名模板也使用了类似的条件简写语法 `[a20,a21 / a30,a31,a32,a33]`，该写法不是合法的仓颉声明语法，也无法直接映射到代码。

- **改进建议**：

  参考 §3.3 item 5 `fromColumns` 的做法（已给出 9 个矩阵类型的独立签名），为 `fromParts` 逐一列出 9 个矩阵类型的独立签名，确保每个签名中的参数数量与实际列数×行数一致。或者至少在家族模板后补充一句说明："每个矩阵类型需要独立定义签名，参数数量为 C×R。" 以避免歧义。

### 问题 2：`filled(scalar)` 与 `identity(one)` 的 Bool 矩阵排除边界未在声明位置标注

- **所在位置**：§3.3 item 3（filled 声明）和 item 8（identity 声明）
- **严重程度**：轻微
- **问题描述**：

  §7 D33 明确记录 Bool 矩阵不提供 `filled()`、`identity()`、`equalEpsilon()`、算术运算符。但 §3.3 item 3 和 item 8 的声明位置没有任何指向 D33 的交叉引用或 Bool 排除注释。代码实现者需要在阅读 §3.3 时自发意识到"某些操作在 Bool 矩阵上不可用"，并自行翻到 §7 确认排除范围。

  这导致两个问题：(1) 阅读者无法在单个章节内获得完整信息；(2) 如果后续修改移除了 D33，§3.3 的代码实现者不会收到通知。

- **改进建议**：

  在 §3.3 item 3 末尾添加注释："Bool 矩阵不支持此函数（见 D33）。" 同样在 item 8 末尾添加对应注释。或者为 Bool 不支持的函数统一添加一个脚注标记。

### 问题 3：`HighpDMat` 方阵短别名命名存在不一致且未在 §9 记录

- **所在位置**：§8 fwd.cj 别名新增清单 — HighpDMat 家族
- **严重程度**：轻微
- **问题描述**：

  HighpDMat 家族的方阵短别名为 `Highpdmat2`/`Highpdmat3`/`Highpdmat4`（首字母大写的 PascalCase 前缀 + 全小写的 `dmat`）。这与阶段一 Vec 别名的命名模式不一致——阶段一的 `HighpDVec2` 是统一 PascalCase（`Highp` + `D` + `Vec` + `2`）。同时，GLM 参考（`fwd.hpp:732-734`）使用 `highp_dmat2`（全小写+下划线）。

  该命名偏差（仓颉版本采用混搭大小写）未被 §9 差异声明记录。如果这是有意设计决策，应在 §9 增加条目说明理由。

- **改进建议**：

  方案一（推荐）：统一为 `HighpDMat2`/`HighpDMat3`/`HighpDMat4`（与 `HighpDVec2` 的 PascalCase 模式一致）。方案二：在 §9 新增差异条目记录命名偏差及理由。

### 问题 4：`fromMat` item 7（同尺寸跨类型）约束条件可能导致下游消费者误解泛型适用范围

- **所在位置**：§3.3 item 7 vs §7 D33
- **严重程度**：轻微
- **问题描述**：

  §3.3 item 7 声明 fromMat 7 "无需 `T <: Number<T>` 约束，因此 Bool 矩阵支持此操作"。但在 item 7 的签名示例中，`T` 是目标矩阵的元素类型。如果目标是 `Mat4x4<Bool, Q>` 而源是 `Mat4x4<Float32, P>`，则 conv: `(Float32) -> Bool` 可以通过 `{ x => x != 0.0f }` 实现，编译应无问题。

  但问题是：当目标类型为 Bool 时，conv 返回 Bool，但仓颉的 `Bool` 类型是否支持所有可能的 `U -> Bool` 转换路径？例如，如果源是 `Mat4x4<Int32, P>`，conv 为 `{ x => Bool(x) }`，`Bool(Int32)` 是否合法？仓颉标准库中 `Bool` 是否有接受 `Int32` 的构造函数？如果 Bool 只接受 `Int8` 等受限类型，则 `fromMat<Int32>` 的 conv 闭包需要更复杂的写法（如 `{ x => x != 0 }`），文档未就此给出示例或说明。

  此问题虽然不致命（用户总可以通过表达式闭包绕过），但文档的"支持"声明过于乐观，未提及实际使用中 conv 闭包可能需要的适配逻辑。

- **改进建议**：

  在 §3.3 item 7 末尾补充说明："当目标 T=Bool 时，conv 闭包通常需使用比较表达式（如 `{ x => x != 0 }`）而非构造函数调用，因为 Bool 类型可能不接受所有源类型的直接构造。"

### 问题 5：§8 compile order 的 fallback 方案一（全局自由函数）对 API 调用语法的破坏性变更影响范围未在 §9 记录

- **所在位置**：§8 fallback 方案一的"API 变化影响"段落 vs §9 差异声明
- **严重程度**：轻微
- **问题描述**：

  §8 已正确分析 fallback 方案一的 API 影响（将 `v * m` 改为 `mul_row_vec_mat(v, m)`，需要所有下游使用者修改代码），并说明"此变化不可自动迁移"。但该 API 形态的巨大差异——如果最终实施方案一，意味着行向量×矩阵乘法彻底从运算符重载变为具名函数调用——未被 §9 差异声明记录。

  如果编码阶段验证失败（仓颉编译器不支持同包延迟解析），最终选用了方案一，§9 需要新增一条差异来记录这个 API 形态偏差。当前 §9 最后一个与行向量×矩阵乘法相关的条目是"行向量×矩阵运算符在 Vec extend 块中"，标注为"编译顺序依赖同包延迟解析（待原型验证）"，但未提及方案一（全局自由函数）的存在。此信息碎片化：读者在 §8 看到方案一，在 §9 看到待验证标注，两者无交叉引用。

- **改进建议**：

  在 §9 对应差异条目末尾添加交叉引用："若验证失败，将退化为 global 自由函数方案（见 §8 fallback 方案一），API 形态从 `v * m` 变为 `mul_row_vec_mat(v, m)`。"

### 问题 6：§8 产出物清单中 `lib.cj` 的导出粒度未定义

- **所在位置**：§8 阶段二产出物清单 — `lib.cj` 条目
- **严重程度**：轻微
- **问题描述**：

  §8 产出物清单提到"修改文件：src/fwd.cj, src/lib.cj"，但未定义 lib.cj 具体需要新增哪些 public import。§2 也说"lib.cj — 新增矩阵类型的 public import"，但没有给出导入清单。

  相比之下，fwd.cj 的新增内容在 §8 有完整的 108 个别名清单。lib.cj 作为公共 API 面的出口文件，其导出内容直接影响下游消费者的可用性，但设计文档没有定义其导出范围。

- **改进建议**：

  在 §8 补充 lib.cj 的 public import 清单，至少明确：(1) 是否导出全部 9 个 detail 矩阵类型；(2) 是否导出 ext/ 别名文件的 public import；(3) 是否导出 scalar_mat_ops.cj 的全局函数。可参考 §8 fwd.cj 别名清单的详细程度。

### 问题 7：非方阵 `*=` 的简要提及无法指导编码实现

- **所在位置**：§9 差异声明："非方阵 *= 不可用" 与 §3.5 手动定义回退方案表
- **严重程度**：轻微
- **问题描述**：

  §9 说"非方阵 *= 不可用"，§3.5 的复合赋值回退表也仅对 3 个方阵列出 *=(Mat) 和 /=(Mat)。这个结论是正确的——非方阵矩阵乘法产生不同尺寸的结果（如 Mat2x3 * Mat3x4 → Mat2x4），因此 `mat2x3 *= mat3x4` 无法在同一变量上执行。

  但对编码实现者而言，此结论意味着非方阵类型（6 个）的 extend 块中 `operator func *=` 可以被安全地省略。文档应明确给出此实现指导，而非仅让读者从差异声明和回退表的缺失信息中推导。

- **改进建议**：

  在 §3.5 复合赋值回退方案表后添加一句话说明："非方阵不定义 *=(Mat) 和 /=(Mat) 运算符，因乘法结果类型与左操作数类型不一致，编译器不自动生成且手动定义无意义。"

## 整体评价

产出整体质量较高，经过 9 轮迭代已解决大部分严重/中等程度问题。当前文档对需求的主要方面（9 矩阵类型、工厂函数体系、算术运算符、stub 策略、别名体系）均有覆盖，技术分析扎实（T(0)/T(1) 约束分析、fromMat 行列扩展规则拆分、B 类方向 GLM 对照验证均为亮点）。

上述 7 个问题均为轻微-中等程度，集中在：签名模板的精确性（问题 1）、交叉引用完整性（问题 2、4、5、6）、命名一致性（问题 3）、实现指导明确性（问题 7）。修复成本低，但能有效降低编码阶段的歧义风险。
