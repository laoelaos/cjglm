根据以下审查结果，迭代上一轮的产出，形成新版的文件，从而更好地满足用户需求。

## 当前审查结果

### Q1（严重）：fromMat 72 个转换方向的实现策略不充分，编码者无法直接落地

fromMat 6a（同类型不同尺寸）和 6b（跨类型不同尺寸）共覆盖 72 个转换方向（9 源 × 8 目标），但仅详细分析了 9 个 B 类方向（C 不变、R 扩展）和 3 个代表性非 B 类方向。剩余 60 个方向仅按风险等级分组（高/中/低各 ~20），既无通用算法描述，也无具体行为规范。编码者拿到此 OOD 后无法直接为 60 个方向编写实现。该问题已在 v3、v4、v5 等多轮审查中反复提出，至今未被充分解决。

所在位置：§3.3 item 6a/6b（L272-341）
改进建议：提供以下至少一种完备性方案：(a) 给出完整的列扩展/行扩展/列收缩/行收缩的数学定义，证明该规则可机械地应用于所有 72 个方向；(b) 在 OOD 中补充一张 9×8 的 fromMat 转换矩阵表，标注每个方向的填充规则和 GLM 偏差情况；(c) 明确说明采用 match 运行时分支的实现策略（通过 length() 静态方法获取列数/行数做统一处理）并给出原型验证计划。

### Q2（严重）：矩阵-矩阵跨尺寸乘法（24 个重载）签名完全缺失

§3.5 声称有 24 个跨尺寸矩阵乘法重载（3 个同尺寸 + 24 个跨尺寸 = 27 个），但仅给出了 Mat×Vec 和 Vec×Mat 的完整签名表，完全未给出 Mat×Mat 跨尺寸乘法的函数签名。编码者无法确认哪些尺寸组合的乘法是合法的。同样缺失的是跨尺寸乘法的结果类型推导规则。该问题在 v7、v13 审查中已提出，尚未被解决。

所在位置：§3.5 算术运算符（L355-358）
改进建议：提供完整的 Mat×Mat 乘法重载签名表（至少以列表形式覆盖全部 24+3=27 个重载），明确标注结果类型的推导规则（左矩阵列数 = 右矩阵行数）。考虑到编码可直接推导，也可给出通用规则加少量示例。

### Q3（严重）：§3.7 transpose 直接实现说明中示例方向写反，代码示例与文本描述不一致

§3.7 matrix.cj "直接实现说明"中，transpose 举例写道"以 Mat2x3→Mat3x2 为例：`dst.c0 = Vec3(src[0][0], src[1][0], src[2][0])`、`dst.c1 = Vec3(src[0][1], src[1][1], src[2][1])`"。但 Mat2x3→Mat3x2 方向下：Mat2x3 有 2 列（c0/c1），不存在 src[2]；且 Mat3x2 的列向量是 Vec2 而非 Vec3。实际代码示例演示的是 Mat3x2→Mat2x3 的行为。文本声称的方向与示例代码严重不一致，编码者若按此实现将产生编译错误。经对照 GLM 参考文件验证，GLM 中 transpose(Mat2x3) 的正确实现为 `dst.c0 = Vec2(src[0][0], src[1][0]), dst.c1 = Vec2(src[0][1], src[1][1]), dst.c2 = Vec2(src[0][2], src[1][2])`。

所在位置：§3.7 matrix.cj 直接实现说明（L456）
改进建议：修正为 (a) 将方向标签改为"Mat3x2→Mat2x3"，保留代码示例不变；或 (b) 保留"Mat2x3→Mat3x2"方向标签，将代码修正为正确实现。

### Q4（中等）：`filled(scalar)` 的名称与语义存在误导

`filled(scalar: T)` 名称强烈暗示将矩阵所有元素设置为 scalar（即全填充矩阵），但实际行为是"对角线元素填入 scalar、非对角线填入 T(0)"（即对角矩阵/scaled identity）。使用 filled 名称会造成编码者的认知偏差——首次接触此 API 的开发者会认为 filled(0.0f) 产生零矩阵，但实际产生的是对角形式。

所在位置：§3.3 item 3（L231-236）
改进建议：考虑将函数改名为 diagonal(scalar)、scale(scalar) 或 fromScalar(scalar) 以消除歧义。若坚持保留 filled 名称，应在其文档注释中显式标注"行为是标量对角矩阵，非全填充"的警告。

### Q5（中等）：§8 compile order fallback 方案一的"首选"标签与 API 破坏性矛盾

§8 将方案一标为"首选"，但其描述明确指出"将 v * m 运算符调用改为全局具名函数调用 mul_row_vec_mat(v, m)，完全改变了调用语法"、"此变化不可自动迁移，需通知所有使用者"。方案一 API 破坏性最大，而方案二（独立文件，保留运算符语法，12 行代码，集中维护）不破坏 API。v25 修订说明将方案二与方案三互换优先级，但方案一仍保留"首选"标签。

所在位置：§8 compile order fallback 方案（L594-601），修订说明第 4 项（L683）
改进建议：将方案一降级为"第三备选"，方案二升级为"首选"，使优先级排序与 API 稳定性要求一致。

### Q6（中等）：行向量×矩阵运算符回退方案（方案一）的函数名 mul_row_vec_mat 与已有命名风格不一致

方案一建议的自由函数名 mul_row_vec_mat 采用了下划线分隔的小写命名风格，但现有代码库中的所有函数使用 camelCase（如 fromBoolVec、castVec4、logicalAnd）或全小写单词（add、sub、mul、div）。mul_row_vec_mat 是 snake_case 命名，与整个项目的命名风格不统一。

所在位置：§8 方案一描述（L594）
改进建议：将函数名改为 mulRowVecMat 以匹配项目 camelCase 风格，或采用 matRowMul/rowMul 等更简洁的命名。

### Q7（较低）：复合赋值运算符手动定义 60 个的验证计划缺失

OOD 列出了手动定义 60 个复合赋值运算符的分布表（§3.5），并标注"若自动生成验证失败"作为回退。但对于"自动生成"是否成功，未定义验证标准、验证时间和验证方法。这与 §8 中"编码启动首日"的验证计划形成对比——复合赋值运算符的自动生成验证同样应纳入编码启动首日的预验证项。

所在位置：§3.5 复合赋值运算符回退方案（L360-376），§8 编码启动前验证项（L584）
改进建议：在 §8 "编码启动前验证项"中补充复合赋值运算符自动生成验证。

### Q8（较低）：非方阵 identity() 的 GLM 行为声明缺少行级对照

§3.3 item 8 声明非方阵 identity() 的行为"与 GLM 1.0.3 中 identity 的实现方式一致"，但未提供与 GLM 源码的行级对照（与 B 类方向分析中逐项对照签名行号的做法形成对比）。虽然参考文件已存在，但 OOD 中未引用具体行号来支撑该声明。

所在位置：§3.3 item 8（L286-299）
改进建议：补充非方阵 identity() 与 GLM 1.0.3 的逐项对照分析，至少覆盖 2-3 个代表性非方阵类型。
