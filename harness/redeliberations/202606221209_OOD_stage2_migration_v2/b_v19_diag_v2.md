## 新增发现问题（本轮审查扩展覆盖 §3.5, §3.7, §5, §8 后确认）

### 问题 3：矩阵乘法运算符签名缺少泛型约束声明（§3.5）

| 项目 | 内容 |
|------|------|
| **问题描述** | §3.5 列出了 27 个矩阵×矩阵乘法运算符（L418-448），但仅以"左矩阵类型 \| 右矩阵类型 \| 结果类型"的类型映射表形式呈现，**未提供任何 `where` 泛型约束**。与其他章节形成对比：§3.7 stub 函数库中每个重载均明确写出 `where T <: Number<T>, Q <: Qualifier`；§3.5 scalar_mat_ops.cj 签名模板也列出了完整约束。矩阵乘法涉及 `*` 算术运算，需要对 `T` 施加 `Number<T>` 约束，但表中完全未体现。实现者无法直接从该表获知 `operator func *(r: Mat2x3<T,Q>): Mat2x4<T,Q>` 应使用什么约束。 |
| **所在位置** | §3.5，L418-448（27 个运算符的签名表格） |
| **严重程度** | 严重 — 缺少约束将导致编译错误或在文档阅读阶段引入不确定性 |
| **改进建议** | 在签名表上方或图例中补充一条泛型签名模板，明确列出 `where T <: Number<T>, Q <: Qualifier`。例如："所有矩阵×矩阵乘法运算符的完整签名为 `public operator func *(r: 右矩阵类型<T,Q>): 结果矩阵类型<T,Q> where T <: Number<T>, Q <: Qualifier`"。同时确认该约束与 §5 的 `@OverflowWrapping` 标注一致。 |

### 问题 4：fromMat 转换规则的四项基本操作缺少对 Bool 矩阵的兼容性评估（§3.3）

| 项目 | 内容 |
|------|------|
| **问题描述** | §3.3 fromMat 6a/6b 的统一编码策略（L339-402）定义了四项基本操作（colExtend/colShrink/rowExtend/rowShrink）的组合规则。但文档在各处说明中仅提及"6a/6b 需要 `Number<T>` 约束，Bool 矩阵不支持"，未进一步分析：当 fromMat 7（跨类型同尺寸转换）用于 Bool 矩阵时，这些基本操作中哪些仍能工作？例如，`colShrink` 仅截断列向量、`rowShrink` 仅截断行向量——这些操作不涉及算术，理论上 Bool 矩阵也应支持。然而当前文档将全部 72 个方向的填充模式统一归属到 fromMat 6a/6b 的 `Number<T>` 约束下讨论，未区分非算术操作（colSh/rowSh 等纯截断）在 Bool 矩阵上的可用性。与 §9 D33 的声明（"Bool 矩阵不提供 fromMat 6a/6b"）间存在策略真空：若用户需要"跨尺寸但非算术"的 Bool 矩阵转换，当前设计不支持。 |
| **所在位置** | §3.3 fromMat 6a/6b（L267-284, L339-402）与 §9 D33（L788） |
| **严重程度** | 中等 — 当前设计不可谓错误，但未明确说明 Bool 转换的策略边界，可能导致后续需求遗漏 |
| **改进建议** | 在 D33 的 Bool 矩阵操作清单中补充：明确说明 Bool 矩阵是否支持缩减转换（colShrink/rowShrink 等纯截断路径）。若支持，需在 fromMat 6a/6b 中为 Bool 矩阵提供无 `Number<T>` 约束的专门重载；若不支持，需说明理由。 |

### 问题 5：§5 错误处理策略未覆盖 `equalEpsilon` 的 NaN 行为对下游的影响（§3.6, §5）

| 项目 | 内容 |
|------|------|
| **问题描述** | §3.6（L497）描述 equalEpsilon 的 NaN 行为时称"由逐层委托的 Vec equalEpsilon 具体实现决定"。通过与阶段一现有代码 `compute_vector_relational.cj`（已验证）交叉验证，`ComputeEqualNumeric.callConst` 在 NaN 参与的任何比较中都返回 false。但文档未将这一行为显式记录到 §5 错误处理策略中。对于依赖矩阵精确 NaN 语义的下游消费者（如科学计算），此信息缺失可能产生误判。 |
| **所在位置** | §3.6 L497（NaN 比较行为段）；§5 L647-658（错误处理策略，仅覆盖溢出和 stub 异常） |
| **严重程度** | 一般 — 行为本身与 Vec 一致且正确，但缺乏在错误处理章节的显式记载 |
| **改进建议** | 在 §5 中补充一行：注明"equalEpsilon 在含 NaN 分量时返回 false，与 == 的 NaN 语义一致"。或直接在 §3.6 标注中明确 equalEpsilon 对 NaN 的行为而非模糊委托。 |

### 问题 6：§10 API 覆盖矩阵存在一个遗漏（§10）

| 项目 | 内容 |
|------|------|
| **问题描述** | §10 API 覆盖矩阵列出 `glm/geometric.hpp` 时标注 `dot` 覆盖 Vec1~Vec4（L825），但 `cross` 仅列出 Vec3 版本（L826）。GLM 1.0.3 的 `glm/geometric.hpp` 中 `cross` 仅对 Vec3 有定义。对照 `geometric.cj` 的完整签名清单（L586-601），`cross` 确实只提供了 `Vec3<T,Q>` 版本。矩阵覆盖正确。但 `geometric.cj` 的签名清单（L582-601）中提供了 `dot` 的 Vec1/Vec2/Vec3/Vec4 四个版本（L583-585），而 §10 覆盖矩阵中 `dot` 行标注为"本阶段 stub"，未列出覆盖哪些 Vec 维度，与 `geometric.cj` 清单的对应关系不够透明。同为 stub 函数的 `normalize`、`length`、`distance` 等也仅标注"Vec2~Vec4"或未标注覆盖维度。 |
| **所在位置** | §10，L821-831（glm/geometric.hpp 覆盖矩阵） |
| **严重程度** | 一般 — 不影响可用性，仅透明度不足 |
| **改进建议** | 在 `dot` 行的"覆盖状态"列补充覆盖维度标记，如"本阶段 stub（Vec1~Vec4）"；同步为 `normalize`、`length`、`distance`、`reflect`、`refract` 补充维度信息。 |
