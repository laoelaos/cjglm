## 当前审查结果

### Q1（编码阶段建议）：`@Derive[Hashable]` 跨元素类型编译验证未纳入编码清单
- **位置**：§3.1（L187）、§8（L717）
- **问题**：`@Derive[Hashable]` 的编译验证当前仅覆盖 T=Bool，未扩展至所有目标元素类型（Float32、Float64、Int32、Int64）。虽不构成设计质量缺陷，但应在编码启动阶段验证。
- **建议**：在 §8 编码启动前验证清单中，将 `@Derive[Hashable]` 的编译验证从"T=Bool"扩展为覆盖所有目标元素类型。

### Q2（编码阶段建议）：纯收缩方向 `one` 参数的文档强化建议
- **位置**：§3.3（L286）
- **问题**：纯收缩方向下 `one` 参数被忽略但保留签名，当前仅以文本说明，缺少使用示例帮助调用方理解。
- **建议**：在现有说明段落基础上，增加一个使用示例（如在纯收缩方向示例代码中显式传递任意 dummy 值并注明 one 不会被使用）。

### Q3（一般）：排除别名族的覆盖说明缺乏精确性
- **位置**：§8（L771-773）、§10（L876-881）
- **问题**：排除统计数字（"约 81"实际 144、"约 108"实际 480）与 GLM 实际偏差过大；API 覆盖矩阵精度等级（"3 精度"实际 4 精度）不准确。
- **建议**：将排除统计替换为精确数字或定性描述；§10 覆盖矩阵中 GLM 别名族精度等级从"3 精度"修正为"4 精度"。

### Q4（一般）：`col()` 函数的 mut 赋值版本签名未在设计中显式列出
- **位置**：§3.4（L411-415）
- **问题**：下标运算符 `[]` 被描述为拥有取值和赋值双版本。`col()` 被描述为"与 [] 完全等价"，但仅提供了取值签名形式，未提及对应的 mut 赋值版本签名。
- **建议**：在 §3.4 中补充 `col()` 的赋值版本签名，或明确声明 `col()` 仅提供取值版本。

### Q5（一般）：§10 API 覆盖矩阵中 `mod` 函数的覆盖状态标注不精确
- **位置**：§10（L835）
- **问题**：覆盖矩阵将 `mod` 标注为"本阶段 stub（标量）"，但 §3.7 common.cj 的注释指出签名为暂定且需在编码阶段拆分为整数和浮点重载，矩阵未反映此风险。
- **建议**：同步更新 §10 覆盖矩阵中 `mod` 行的覆盖状态说明，使其与 §3.7 注释一致（标注签名暂定及编码阶段拆分需求）。

### Q6（一般）：`equalExact` 函数的 NaN 语义在 §3.6 与 §5 之间存在未对齐风险
- **位置**：§3.6（L503-504）、§5（L664）
- **问题**：§3.6 声明 `equalExact` 与 `==` 行为语义等价（NaN 返回 false），但 §5 错误处理策略仅覆盖了 `equalEpsilon` 的 NaN 行为，未为 `equalExact` 做相同级别的显式声明。
- **建议**：在 §5 中补充"equalExact NaN 行为"条目，声明 equalExact 对含 NaN 分量的矩阵返回 false，与 equalEpsilon 的 NaN 声明保持对称。
