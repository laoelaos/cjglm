## 问题清单

### Q1（严重）：GLM 1.0.3 参考头文件缺失，OOD 中所有关于 GLM .inl 行为的声明无法独立验证

- **问题描述**：OOD 多处引用 GLM 1.0.3 头文件的具体行号来支撑设计决策（如 "type_mat2x3.inl:122"、"type_mat4x4.inl:246" 等），并基于此声称 Mat4x4←Mat4x2 的转换行为"偏离主规则"。然而 `references/glm-1.0.3/` 目录实际为空，不存在任何 .inl 或 .hpp 文件。这些行为声明的准确性、是否存在其他未发现的 GLM 特殊行为均无法独立验证。
- **所在位置**：§3.3 B 类方向对照分析表（L305-314）、非 B 类方向对照分析（L321-328）、偏差声明 §9 "Mat4x4←Mat4x2 转换行为偏离主规则"（L670）
- **严重程度**：严重
- **改进建议**：在编码启动前补充 GLM 1.0.3 参考头文件到 `references/glm-1.0.3/` 目录，并逐项对照验证 OOD 中引用的所有 GLM 行为声明。若头文件不可得，应在 OOD 中标注所有 GLM 行为声明为"待验证"并记录信息来源。

### Q2（严重）：fromMat 72 个转换方向的实现策略不充分，编码者无法直接落地

- **问题描述**：OOD 声称从 Mat 6a（同类型不同尺寸）和 6b（跨类型不同尺寸）共覆盖 72 个转换方向（9 源 × 8 目标），但仅详细分析了 9 个 B 类方向（C 不变、R 扩展）和 3 个代表性非 B 类双变化方向。剩余 60 个方向仅按风险等级分组（高/中/低各 ~20），既无通用算法描述，也无具体的行为规范。编码者拿到此 OOD 后无法直接为 60 个方向编写实现——要么需要自行推导每个方向的填充规则，要么逐个查阅 GLM 参考头文件。这不符合 OOD 应"直接指导编码实现"的质量要求。
- **所在位置**：§3.3 item 6a/6b（L272-341）
- **严重程度**：严重
- **改进建议**：提供以下至少一种完备性方案：**(a)** 给出完整的列扩展/行扩展/列收缩/行收缩的数学定义，证明该规则可机械地应用于所有 72 个方向（即编码者可按公式编码，无需逐例查阅 GLM）；或 **(b)** 在 OOD 中补充一张 9×8 的 fromMat 转换矩阵表，标注每个方向的填充规则（和 GLM 偏差情况）；或 **(c)** 明确说明采用 match 运行时分支的实现策略（通过 `length()` 静态方法获取列数/行数做统一处理）并给出原型验证计划。

### Q3（严重）：矩阵-矩阵跨尺寸乘法（24 个重载）签名完全缺失

- **问题描述**：§3.5 声称有 24 个跨尺寸矩阵乘法重载（3 个同尺寸 + 24 个跨尺寸 = 27 个），但仅给出了 Mat×Vec 和 Vec×Mat 的完整签名表，完全未给出 Mat×Mat 跨尺寸乘法的函数签名。编码者无法确认哪些尺寸组合的乘法是合法的——例如 Mat2x3 × Mat3x4 → Mat2x4 是否支持、Mat2x3 × Mat2x4 是否非法。同样缺失的是跨尺寸乘法的结果类型推导规则。
- **所在位置**：§3.5 算术运算符（L355-358）
- **严重程度**：严重
- **改进建议**：提供完整的 Mat×Mat 乘法重载签名表（至少以列表形式覆盖全部 24+3=27 个重载），明确标注结果类型的推导规则（左矩阵列数 = 右矩阵行数）。

### Q4（中等）：`filled(scalar)` 的名称与语义存在误导

- **问题描述**：`filled(scalar: T)` 名称强烈暗示将矩阵所有元素设置为 `scalar`（即全填充矩阵），但实际行为是"对角线元素填入 scalar、非对角线填入 T(0)"（即对角矩阵/scaled identity）。在 GLM 中此行为通过单参数构造函数 `mat<C,R>(val)` 实现，非具名函数。使用 `filled` 名称会造成编码者的认知偏差：首次接触此 API 的开发者会认为 `filled(0.0f)` 产生零矩阵，但实际产生的是对角形式。OOD 虽在内部阐述了真实行为，但名称与语义的偏差在设计层面构成潜在的 API 误用风险。
- **所在位置**：§3.3 item 3（L231-236），函数名称及其行为描述
- **严重程度**：中等
- **改进建议**：考虑将函数改名为 `diagonal(scalar)`、`scale(scalar)` 或 `fromScalar(scalar)` 以消除歧义。若坚持保留 `filled` 名称，应在其文档注释中显式标注"行为是标量对角矩阵，非全填充"的警告。

### Q5（中等）：非方阵 `identity()` 的行为与 GLM 的逐项对照缺失

- **问题描述**：OOD 声明非方阵 `identity()` 的行为"与 GLM 1.0.3 中 identity 的实现方式一致"，但未提供与 GLM 源码的行级对照（与 B 类方向分析中逐项对照签名行号的做法形成对比）。考虑到 Mat4x4←Mat4x2 已发现存在偏离主规则的情况，非方阵 identity 同样可能存 GLM 未遵循主规则的场景。缺少 GLM 对照意味着该声明的可信度不足。
- **所在位置**：§3.3 item 8（L286-299）主对角线描述，§9 偏差声明中对应条目（L662）
- **严重程度**：中等
- **改进建议**：补充非方阵 `identity()` 与 GLM 1.0.3 的逐项对照分析，至少覆盖 9 个矩阵类型中尺寸差异最大的组合（如 Mat2x4、Mat4x2、Mat3x4、Mat4x3）。若参考头文件不可及，应标注为"待验证"。

### Q6（中等）：fromMat 方案一（首选）与 API 破坏性矛盾

- **问题描述**：OOD 将编译顺序 fallback 方案一标记为"首选"，但其描述明确指出"将 `v * m` 运算符调用改为全局具名函数调用 `mul_row_vec_mat(v, m)`，完全改变了调用语法"，"此变化不可自动迁移，需通知所有使用者"。首选方案的 API 破坏性最大，而方案二（独立文件，保留运算符语法，12 行代码，维护简单）和方案三（分散 9 文件，36 行代码，保留运算符语法）均不破坏 API。逻辑上方案二才是应该首选的方案——OOD 的修订说明（v25）声称已调整优先级，但当前 §8 正文中的标签"首选/备选/第三备选"与修订意图不完全一致：方案一仍被标记为"首选"，与其 API 破坏性不相称。
- **所在位置**：§8 compile order fallback 方案（L594-601），修订说明第 4 项（L683）
- **严重程度**：中等
- **改进建议**：将方案一降级为"第三备选"，方案二升级为"首选"，使优先级排序与 API 稳定性要求一致。或在方案一描述中明确标注"如编译顺序问题无法以其他方式解决的最后手段"。

### Q7（中等）：行向量×矩阵乘法重载数量声称与实际不符

- **问题描述**：§8 方案三描述中仍声称"全部 12 个 Vec extend 块"，但根据 §2 包组织中的完整签名清单，实际只有 9 个 Vec extend 块（Vec4 对应 3 个矩阵类型、Vec3 对应 3 个、Vec2 对应 3 个、Vec1 对应 0 个，合计 9 个）。此问题在第 12 轮迭代中已被指出（见 iteration_history.md），OOD 在后续轮次未完全修正——§8 方案三的描述中"12 个"的表述仍可能残留。
- **所在位置**：§8 方案三描述（L598）声称"全部 12 个 Vec extend 块"
- **严重程度**：中等
- **改进建议**：将"12"修正为"9"，或逐项列出 9 个 Vec extend 块并确认方案三中每个矩阵文件末尾实际追加的 extend 数量。

### Q8（中等）：行向量×矩阵运算符回退方案（方案一）的函数名 `mul_row_vec_mat` 与已有命名风格不一致

- **问题描述**：方案一建议的自由函数名 `mul_row_vec_mat` 采用了下划线分隔的小写命名风格，但现有代码库中的所有函数使用 camelCase（如 `fromBoolVec`、`fromBoolVecQ2`、`castVec4`、`logicalAnd`、`logicalOr`）或全小写单词（`add`、`sub`、`mul`、`div`）。`mul_row_vec_mat` 是 snake_case 命名，与整个项目的命名风格不统一。
- **所在位置**：§8 方案一描述（L594）
- **严重程度**：中等
- **改进建议**：将函数名改为 `mulRowVecMat` 以匹配项目 camelCase 风格，或采用 `matRowMul`/`rowMul` 等更简洁的命名。

### Q9（较低）：`identity()` 的 `one: T` 参数与 `filled()` 的 `scalar: T` 参数语义关联不透明

- **问题描述**：`identity(one: T)` 将参数命名为 `one`，强调必须传入"1"；`filled(scalar: T)` 将参数命名为 `scalar`，暗示可以是任意值。但两者的内部实现都使用此值作为对角线元素填入——identity 的 `one` 参数实际上可以是任意 T 值（不仅限于 1.0f），`filled` 的 `scalar` 参数同理。参数命名与函数语义存在微妙的 m 语义不一致：identity 命名为 `one` 但行为上不强制要求传入 1，这与 DV-01 中 `fromBoolVec` 的 `one` 参数语义不同（fromBoolVec 的 `one` 参数确实应该传入"1"的表示值）。
- **所在位置**：§3.3 item 3（filled 签名）、item 8（identity 签名）
- **严重程度**：较低
- **改进建议**：建议将 identity 的参数名统一为 `scalar` 或 `diagonalValue` 而非 `one`，以消除"必须传入 1"的误导。注意此变更需同步考虑与 `fromMat 6a/6b` 中 `one: T` 参数的关联——fromMat 的 `one` 参数语义正确（确实需要 1 的表示值用于对角线填充），不应一并改名。

### Q10（较低）：非方阵 `filled()` 和 `identity()` 的边界条件未说明

- **问题描述**：当 C=0 或 R=0 的极端场景（虽在正常使用中不会出现）或当 `scalar` / `one` 为 NaN 时的浮点行为未定义。对于方阵，`identity(NaN)` 产生的矩阵对角线为 NaN、非对角线为 NaN-NaN=NaN，该结果在 IEEE 754 语义下是合法的。但这些行为未被文档化。
- **所在位置**：§3.3 item 3（filled）和 item 8（identity）
- **严重程度**：较低
- **改进建议**：在 §4 "关键行为契约"中补充 NaN 输入的行为契约，明确 identity(NaN) 产生的矩阵中所有元素均为 NaN。

### Q11（较低）：复合赋值运算符手动定义 60 个的验证计划缺失

- **问题描述**：OOD 列出了手动定义 60 个复合赋值运算符的分布表（§3.5 文件分布表），并标注"若自动生成验证失败"作为回退。但对于"自动生成"是否成功，未定义验证标准、验证时间和验证方法。这与 §8 中"编码启动首日"的验证计划形成对比——复合赋值运算符的自动生成验证同样应纳入编码启动首日的预验证项。
- **所在位置**：§3.5 复合赋值运算符回退方案（L360-376），§8 编码启动前验证项（L584）
- **严重程度**：较低
- **改进建议**：在 §8 "编码启动前验证项"中补充复合赋值运算符自动生成验证，明确验证标准（如：为 Mat2x2 定义一个 `+= T` 测试代码，确认编译通过且语义正确；若失败则启用 §3.5 回退方案的手动定义）。

### Q12（严重）：transpose 直接实现说明中示例方向写反，代码示例与文本描述不一致

- **问题描述**：§3.7 matrix.cj "直接实现说明"中，transpose 举例写道"以 Mat2x3→Mat3x2 为例：`dst.c0 = Vec3(src[0][0], src[1][0], src[2][0])`、`dst.c1 = Vec3(src[0][1], src[1][1], src[2][1])`"。但 Mat2x3→Mat3x2 方向下：Mat2x3 有 2 列（c0/c1），不存在 src[2]；且 Mat3x2 的列向量是 Vec2 而非 Vec3。实际代码示例演示的是 Mat3x2→Mat2x3 的行为（从 3 个 Vec2 列向量各取首元素构成 Vec3）。文本声称的方向与示例代码严重不一致，编码者若按此实现将产生编译错误。
- **所在位置**：§3.7 matrix.cj 直接实现说明（L456）
- **严重程度**：严重
- **改进建议**：修正为 **(a)** 将方向标签改为"Mat3x2→Mat2x3"，保留代码示例不变；或 **(b)** 保留"Mat2x3→Mat3x2"方向标签，将代码修正为 `dst.c0 = Vec2(src[0][0], src[1][0])`、`dst.c1 = Vec2(src[0][1], src[1][1])`、`dst.c2 = Vec2(src[0][2], src[1][2])`。两种改法择一即可，但不能文本与代码双向矛盾。

## 整体评价

产出在仓颉语言约束的适配（T(0)/T(1) 获取策略、编译顺序分析、Qualifier 泛型设计等）方面展现了对底层限制的深刻理解，内部审议已充分覆盖技术可行性维度。

但产出在**指导编码落地的直接性和完备性**方面存在显著不足：核心问题集中在 fromMat 72 个转换方向（Q2）和矩阵乘法重载（Q3）的签名缺失，以及 transpose 示例方向写反（Q12），导致编码者无法直接开始实现；参考头文件的缺失（Q1）使得 OOD 中所有 GLM 行为声明处于"不可验证"状态；优先级矛盾（Q6）和命名一致性问题（Q4, Q8, Q9）则影响了下游决策的连贯性。

建议编码启动前优先解决 Q1-Q3 和 Q12，否则编码阶段会频繁因为"GLM 行为不确定"或"实现策略不明确"而阻塞。
