## 需要解决的问题

- **问题描述**：fromMat 72 个转换方向的实现策略不充分，编码者无法直接落地。60个方向仅按风险等级分组，既无通用算法描述也无具体行为规范。该问题已在多轮审查中反复提出，至今未被充分解决。
- **所在位置**：§3.3 item 6a/6b（L272-341）
- **严重程度**：严重
- **改进建议**：提供以下至少一种完备性方案：(a) 给出完整的列扩展/行扩展/列收缩/行收缩的数学定义，证明该规则可机械地应用于所有72个方向；(b) 在OOD中补充一张9×8的fromMat转换矩阵表，标注每个方向的填充规则和GLM偏差情况；(c) 明确说明采用match运行时分支的实现策略并给出原型验证计划。

- **问题描述**：矩阵-矩阵跨尺寸乘法（24个重载）签名完全缺失。仅给出了Mat×Vec和Vec×Mat的完整签名表，完全未给出Mat×Mat跨尺寸乘法的函数签名。同样缺失跨尺寸乘法的结果类型推导规则。该问题在v7、v13审查中已提出，尚未被解决。
- **所在位置**：§3.5 算术运算符（L355-358）
- **严重程度**：严重
- **改进建议**：提供完整的Mat×Mat乘法重载签名表（至少以列表形式覆盖全部24+3=27个重载），明确标注结果类型的推导规则（左矩阵列数=右矩阵行数）。

- **问题描述**：§3.7 transpose直接实现说明中示例方向写反，代码示例与文本描述不一致。文本声称Mat2x3→Mat3x2，但代码示例实际演示的是Mat3x2→Mat2x3的行为。编码者若按此实现将产生编译错误。
- **所在位置**：§3.7 matrix.cj 直接实现说明（L456）
- **严重程度**：严重
- **改进建议**：修正方向标签或代码示例，使二者一致。两种改法择一即可。

- **问题描述**：`filled(scalar)`名称强烈暗示全填充矩阵，但实际行为是对角矩阵。使用`filled`名称会造成编码者的认知偏差。
- **所在位置**：§3.3 item 3（L231-236）
- **严重程度**：中等
- **改进建议**：改名为`diagonal(scalar)`、`scale(scalar)`或`fromScalar(scalar)`以消除歧义。若坚持保留`filled`名称，应在其文档注释中显式标注警告。

- **问题描述**：§8将方案一（API破坏性最大）标为"首选"，但方案二（独立文件，保留运算符语法，不破坏API）明显更优。"首选"标签与其API破坏性特征矛盾。
- **所在位置**：§8 compile order fallback 方案（L594-601），修订说明第4项（L683）
- **严重程度**：中等
- **改进建议**：将方案一降级为"第三备选"，方案二升级为"首选"。

- **所在位置**：§8 方案一描述（L594）
- **严重程度**：中等
