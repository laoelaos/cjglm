# OOD 设计方案审查报告（v33 / a_v22）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** 9 个独立泛型结构体（Mat2x2~Mat4x4）而非单一 C×R 参数化泛型的策略正确——仓颉不支持模板偏特化，9 个独立类型是唯一可行的映射方式。

**[通过]** 泛型约束 `where T <: Number<T>, Q <: Qualifier` 的语法在仓颉中有效。`Number<T>` 接口可作为算术运算的泛型约束，`Qualifier` 接口作为精度标记的约束。

**[通过]** 矩阵结构体使用 `var c0: VecR<T,Q>, var c1: VecR<T,Q>, ...` 的列向量成员设计——仓颉 struct 支持多个 `var` 具名字段，字段类型可为其他泛型结构体。

**[通过]** extend 块中定义运算符和成员函数的模式（`extend<T, Q> Mat2x2<T, Q> where ... { ... }`）符合仓颉扩展语法形式 B（带新类型参数的泛型扩展），且 extend 后声明的所有类型参数均在被扩展类型中使用，规则一致。

**[通过]** `operator func [](i: Int64): VecR<T,Q>` 下标运算符定义在 extend 块中——仓颉扩展支持添加运算符重载函数。

**[通过]** `mut func col(i: Int64): VecR<T,Q>` 的 mut 函数模式正确——仓颉 extend 块允许 `mut` 修饰符用于结构体可变函数。

**[通过]** `public static func length(): Int64` 静态函数定义在 extend 块中——仓颉扩展允许 `static` 修饰符。

**[通过]** `public static func fromMat<U, P>(conv: (U) -> T, m: Mat2x2<U,P>, one: T)` 等静态泛型工厂函数——仓颉支持在 extend 块中定义静态泛型函数，多个泛型参数和 `where` 约束语法正确。

**[通过]** `@Derive[Hashable]` 自动派生在泛型 struct 上的约束传递性——`std.deriving` 包确认支持，所有要求类型参数在实例化时须实现 `Hashable`，与仓颉的编译时检查一致。

**[通过]** 跨类型转换使用 `conv: (U) -> T` 闭包参数模式——仓颉支持函数类型作为参数，Lambda 表达式语法支持此模式。

**[通过]** `Number<T>` 约束下 T(0)/T(1) 获取限制已正确识别并设计了统一策略（T(0) 通过 `someValue - someValue` 演算、T(1) 通过显式 `one: T` 参数传入），与 deviations.md DV-01 记录的限制一致。

### 2. 标准库与生态覆盖

**[通过]** `@Derive[Hashable]` 依赖 `std.deriving` 包，确认支持泛型结构体自动派生哈希接口。

**[通过]** `@OverflowWrapping` 溢出注解确认可在函数声明上使用，作用于函数内整数运算，与仓颉注解文档一致。

**[通过]** `std.math` 库确认提供 `trunc`、`abs` 等浮点函数，可用于后续阶段的完整实现。

**[通过]** `std.unittest` 测试框架确认可用，测试目录结构与 `cjpm test` 自动发现机制兼容。

**[通过]** `std.convert` 的 `Parsable` 接口可用于编码阶段的类型解析需求。

**[通过]** `NumericLimits<T>.epsilon()` 的约束处理已在设计中通过拆分 `NumericLimits` 和 `isIec559Of<T>()` 解耦，正确避免了 `Bool` 等非数值类型的约束冲突。

### 3. 语言特性可行性

**[通过]** 错误处理策略合理：下标越界 `throw Exception`、stub 函数 `throw Exception("stub")`，均在仓颉异常处理体系内。溢出策略通过 `@OverflowWrapping` 标注统一控制。

**[通过]** `const init` 构造函数的使用正确——仓颉 `const init` 可在非 const 上下文中作为运行时构造函数使用，且其存在使得 const 实例成员函数可以定义。

**[通过]** extend 块不支持 `const` 修饰符已由仓颉扩展文档确认（允许的修饰符列表中不含 `const`）。设计正确处理此限制（D32/D33），矩阵扩展成员函数声明为非 `const`。

**[通过]** 并发设计正确：矩阵为值类型（struct），跨线程复制语义确保无共享写冲突。复合赋值非原子性的声明准确。

**[通过]** 模块/包结构设计符合 cjpm 项目组织方式：`package glm.detail`、`package glm`、`package glm.ext` 三个包的划分合理，`lib.cj` 通过 `public import` 重导出。`src/ext/` 子包方案虽需原型验证，但已提供完整的降级路径（方案一到方案三）。

**[通过]** 编码前验证清单覆盖关键假设：子包构建预验证、复合赋值自动生成验证、Mat4x4←Mat4x2 偏差验证、`@Derive[Hashable]` 跨类型编译验证、同包延迟解析原型验证等——均为合理的风险管控措施。

### 4. 设计一致性

**[通过]** v33 修订完整处理了迭代要求 Q1~Q6 全部问题：
- **Q1**：`@Derive[Hashable]` 编译验证已从 T=Bool 扩展为覆盖所有目标元素类型（Float32/Float64/Int32/Int64/Bool）
- **Q2**：纯收缩方向 `one` 参数已补充使用示例（`m_shrunk = Mat2x2.fromMat(m4x4, one: 1.0f)` 并注明可传递 dummy 值）
- **Q3**：排除别名族统计已替换为定性描述（"冗余精度别名族"），精度等级已修正为"4 精度 × 9 尺寸 = 36"
- **Q4**：`col()` 赋值版本签名模式已补充（`mut func col(i: Int64): VecR<T,Q>`）并说明值语义下的整体替换行为
- **Q5**：`mod` 覆盖状态已更新为"签名暂定，编码阶段需拆分为整数和浮点重载"，与 §3.7 注释对齐
- **Q6**：§5 错误处理策略已新增"equalExact NaN 行为"条目，声明含 NaN 时返回 false

**[通过]** 各抽象职责清晰：9 个矩阵类型、3 个 stub 函数库、标量矩阵运算文件、ext/ 别名文件——职责划分明确，无重叠。

**[通过]** 协作关系形成闭环：矩阵 `.inl` 依赖 → stub 函数库（common.cj / matrix.cj / geometric.cj），标量运算 → 9 个矩阵类型，glm 包 → glm.detail 包。依赖方向单一，无循环依赖。

**[通过]** 模块间依赖方向合理：glm → glm.detail，glm.ext → glm.detail。无反向依赖。

**[通过]** §9 差异声明全面，覆盖所有已知偏差：无 C 数组成员、无 ++/--、无模板赋值运算符、ext/ 别名包名调整、Mat4x4←Mat4x2 特殊偏差等。

**[轻微]** §8 编码前验证项中，多个验证项（如子包构建、同包延迟解析、复合赋值自动生成）在编码阶段由原型验证决定最终方案，设计文档未明确标注验证不通过时的升级/降级决策时间线（如"验证首日第 X 小时内完成"），建议补充判定时间窗口以降低编码启动阶段的不确定性。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：每个文件有明确的关注点（type_mat{N}x{M}.cj 仅定义矩阵结构体 + 基本运算符、matrix.cj 仅负责矩阵运算函数、scalar_mat_ops.cj 仅处理标量左侧运算场景等）。

**[通过]** 抽象层次恰当：为 9 个矩阵类型分别定义而非过度抽象为单一泛型，避免引入仓颉不支持的偏特化复杂性。stub 函数仅提供签名空壳，不引入不必要的实现复杂度。

**[通过]** 设计便于后续实现：§8 定义增量实现路径（编译顺序优先级）、§8 编码启动前验证清单覆盖所有关键假设、§7 设计决策文档完整。

**[通过]** 风险管理和回退路径完备：编译顺序依赖提供三级 fallback 方案（方案二 → 方案三 → 方案一），子包构建提供三级降级路径，复合赋值自动生成提供手动定义回退。每个回退方案均标注触发条件和影响评估。

**[通过]** 可测试性良好：§10 API 覆盖矩阵逐函数标注覆盖状态（本阶段实现/本阶段 stub/后续阶段），§5 NaN 传播策略明确 4 类测试场景，B 类方向单元测试验证要求具体。

**[通过]** fromMat 6a/6b 的 144 个签名构造量识别为代码生成风险，建议使用代码生成脚本自动产生签名模板，措施合理。
