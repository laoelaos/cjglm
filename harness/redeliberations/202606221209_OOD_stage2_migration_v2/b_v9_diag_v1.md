## 问题 1（中等）— fromMat 6a/6b 工厂函数签名缺少 `SrcQ <: Qualifier` 显式约束

**问题描述**：§3.3 items 6a/6b 给出的 fromMat 泛型签名中，类型参数 `SrcQ` 被用于矩阵类型的 Qualifier 位置（如 `Mat2x2<T, SrcQ>`、`Mat2x2<SrcT, SrcQ>`），但 `where` 子句中未显式声明 `SrcQ <: Qualifier` 约束。现有阶段一代码模式中，函数级类型参数在用于带约束的位置时均显式声明约束（如 `scalar_vec_ops.cj` 的 `where T <: Number<T>, Q <: Qualifier`）。缺少此约束可能导致编译器报错——仓颉要求函数签名中使用带约束的类型参数时，必须显式满足该约束。

**所在位置**：
- §3.3 item 6a (L228)：`public static func fromMat<SrcQ>(m: Mat2x2<T,SrcQ>, one: T): Mat2x3<T,Q> where T <: Number<T>, Q <: Qualifier` — 缺少 `SrcQ <: Qualifier`
- §3.3 item 6b (L233)：`public static func fromMat<U, P>(conv: (U) -> T, m: Mat2x2<U,P>, one: T): Mat2x3<T,Q> where T <: Number<T>, Q <: Qualifier, P <: Qualifier` — 注意 6b 示例签名中已包含 `P <: Qualifier`（对应参数名 `P` 而非 `SrcQ`），但 6a 示例签名中 `SrcQ` 缺少对应约束，两处的约束声明不一致
- 此问题涉及时 6a 的全部 72 个签名（9 目标 × 8 源），因 6a 的泛型签名模板在所有目标类型上复用

**严重程度**：中等 — 编码阶段若直接按缺乏约束的签名实现将产生编译错误

**改进建议**：在 fromMat 6a 的 `where` 子句中补充 `SrcQ <: Qualifier`：
```
public static func fromMat<SrcQ>(m: Mat2x2<T,SrcQ>, one: T): Mat2x3<T,Q>
    where T <: Number<T>, Q <: Qualifier, SrcQ <: Qualifier
```
同步检查 §3.3 中所有 fromMat 6a 签名模板（含 Mat3xR 和 Mat4xR 族），确保 `SrcQ` 均带 `Qualifier` 约束。

## 问题 2（中等）— fromMat 填充规则描述"扩展列/行按单位矩阵填充"存在歧义，可能误导实现

**问题描述**：§3.3 item 6 的填充规则表述为"复制重叠区域，扩展列/行按单位矩阵填充"。经对照 GLM 1.0.3 源码验证，实际行为需要区分两种场景：
- **列扩展（C_dst > C_src）**：新列按单位矩阵列填充（对角线位置填入 T(1)，其余 T(0)）
- **行扩展（B 类方向，R_dst > R_src, C 不变）**：新增行元素填入 **T(0)**（零），而非单位矩阵值

当前表述将两种场景混为一谈，开发者若机械遵循"按单位矩阵填充"的字面含义，可能在行扩展场景中用 T(1) 填充新增行元素，导致 8 个 B 类方向的 fromMat 实现错误——与 GLM 行为不一致。

**所在位置**：
- §3.3 item 6 (L225-250)：填充规则描述 "复制重叠区域，扩展列/行按单位矩阵填充"
- §3.3 item 6a "精确填充规则（同类型 6a 版本）"子段 (L272-274)：未区分列扩展与行扩展的差异

**严重程度**：中等 — 当前描述歧义可能在编码阶段导致系统性错误

**改进建议**：将填充规则拆分为两个明确表述：
- **列扩展**：新列在 `(新列索引, 新列索引)` 位置填入 `one`（=T(1)），其余位置填入 T(0)
- **行扩展**：新增行元素全部填入 T(0)（通过 `someValue - someValue` 演算）

并在 B 类方向对照表（L241-254）后补充此拆分的确认说明。同时确认"主规则"在 B 类方向逐项验证结果中的"一致"判定——目前 L245-252 中 8 个 B 类方向标注为"✅ 一致"，但主规则描述模糊，建议在表中显式注明 B 类方向行扩展填充 T(0) 与主规则的对应关系。

## 问题 3（中等）— identity(one) 对非方阵的语义未明确定义

**问题描述**：§3.3 item 8 说"单位矩阵工厂函数 identity(one: T) — 对所有 9 种矩阵类型均提供"，§9 说"identity 对非方阵也提供 — API 统一性设计"，但均未明确定义非方阵上 identity 的具体行为和计算结果。对于非方阵（如 Mat2x3、Mat4x2），"单位矩阵"概念本身不是数学上标准的——设计需要明确其行为：
- `Mat2x3<Float32, PackedHighp>.identity(1.0f)` 的结果应是什么？
- 它与 `filled(1.0f)` 的结果关系如何？

此问题会影响下游使用者——用户无法从设计文档中获知非方阵 identity 的预期行为。编码者也缺少明确的实现规范。

**所在位置**：
- §3.3 item 8 (L239)：仅一句"单位矩阵工厂函数 identity(one: T) — 对所有 9 种矩阵类型均提供"，无非方阵语义说明
- §9 (L552)："identity 对非方阵也提供 — API 统一性设计"，同样无语义说明

**严重程度**：中等 — 缺少语义说明可能导致非方阵 identity 的实现与用户预期不一致

**改进建议**：在 §3.3 item 8 下补充非方阵 semantics 说明段，明确：
1. 非方阵 `identity(one)` 的结果为主对角线上填 `one`、其余填 T(0) 的对角矩阵
2. 非方阵上 `identity(one)` 与 `filled(one)` 行为等价
3. 举例说明：`Mat2x3.identity(1.0f)` → 列 0 = Vec3(1,0,0)、列 1 = Vec3(0,1,0)

也可引用前序版本 v13（`docs/04_ood_phase2.md` L303）中的详细说明作为参考。

## 问题 4（中等）— fromMat 非 B 类方向（C 和 R 同时变化）的行为未与 GLM 对照验证

**问题描述**：§3.3 的 GLM 逐项对照分析仅覆盖了 B 类方向（C_src=C_dst, R_src<R_dst），共 9 个方向。但 fromMat 6a 包含 72 个转换方向，其中大量方向涉及 C 和 R 同时变化（如 Mat2x2→Mat3x4、Mat4x4→Mat2x3 等）。这些方向的行为未经验证是否与 GLM 一致，且文档中"编码阶段须逐一对照 GLM .inl 源码"的建议将验证责任完全推给了编码阶段。

从已验证的 Mat4x4←Mat4x2 偏差案例可以看出，GLM 的部分转换构造函数并非遵循"复制重叠，单位矩阵填充新列/行"的统一规则——Mat4x4←Mat4x2 中 GLM 丢弃了源矩阵列 2、3 的全部数据，即使这些列在目标矩阵中"有对应位置"。这种模式偏差可能也存在于其他未验证的方向中。

**所在位置**：
- §3.3 B 类方向对照分析表 (L241-259)：仅覆盖 9 个 B 类方向
- §3.3 item 6 末尾的编码阶段提醒 (L276)：将验证责任推给编码阶段

**严重程度**：中等 — 未经验证的转换方向可能包含更多 GLM 偏差，编码阶段发现后需要回溯修改设计

**改进建议**：建议至少补充以下几组验证方向（而非全部 72 个）：
1. **C 和 R 同时扩展**（如 Mat2x2→Mat3x4、Mat2x2→Mat4x3）：验证 GLM 是否遵循"复制重叠 + 新列单位矩阵"规则
2. **C 收缩 R 扩展**（如 Mat4x2→Mat2x4、Mat3x2→Mat2x3）：这是 Mat4x4←Mat4x2 偏差的"缩小版"，可能也存在类似偏差
3. **C 扩展 R 收缩**（如 Mat2x4→Mat3x2）：验证重叠区域复制后超出行为是否合理

若已验证，可在报告中补充验证编号以增加完整性；若尚未验证，则应标注为"待验证"而非"编码阶段验证"。

## 问题 5（轻微）— §8 编译顺序 fallback 方案一（全局自由函数）未讨论 API 契约变化对下游的影响

**问题描述**：§8 为"行向量×矩阵乘法"的编译顺序问题提供了三个 fallback 方案。方案一将 `operator *` 替换为全局自由函数（如 `vec4_mul_mat4x4(v, m)`），但此方案完全改变了该操作的调用语法——从 `v * m` 变为函数调用形式，破坏了「运算符重载」的 API 契约。GLM 用户习惯 `v * m` 的行向量×矩阵乘法的语法，fallback 方案一的语法变化对下游代码的影响未被评估（如需要全局搜索替换所有行向量×矩阵乘法的调用点）。

§8 仅描述方案一的实现方式，未讨论其对用户代码的破坏性影响及迁移成本。

**所在位置**：§8 "compile order 说明"节，方案一 (L488)

**严重程度**：轻微 — 只是 fallback 方案，且方案一非首选；但缺少影响评估可能导致编码阶段选择方案一时未充分告知下游

**改进建议**：在方案一的说明后补充一句话注明 API 变化影响：全局自由函数方式将使用 `vec_mul_mat(v, m)` 而非 `v * m` 语法，与 C++ GLM 语法不一致，用户迁移时需修改所有行向量×矩阵乘法的调用点。同时建议在方案一的函数命名上使用运算符风格的具名（如 `mul_row_vec_mat` 而非 `vec4_mul_mat4x4`），提升可读性。

## 整体质量评价

该产出经历了多轮内部审议（v17→v20），整体设计质量较高。核心抽象（9 个独立结构体）、构造函数体系（逐分量/列向量/filled/fromParts/fromColumns/fromMat/identity）、运算符体系（标量/矩阵/向量运算）和偏差声明均经过充分推敲。
