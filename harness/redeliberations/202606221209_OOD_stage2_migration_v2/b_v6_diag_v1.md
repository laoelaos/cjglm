# 质量审查报告

> **审查对象**：`a_v6_design_v1.md`（阶段二 OOD 设计方案 v17）
> **审查视角**：需求响应充分度、事实/逻辑准确性、深度与完整性（面向编码落地的可用性）

## 问题清单

### 问题1（中等）：`fwd.cj` 矩阵别名新增的具体内容未定义

**问题描述**：§2 模块划分中标明 `fwd.cj ★ — 新增矩阵类型别名`，§8 产出物清单提到"修改文件：src/fwd.cj, src/lib.cj"，但全文中未定义**哪些**矩阵类型别名需要新增到 `fwd.cj`。现有的 `fwd.cj`（阶段一）包含 B/I/U/I8/I16/I32/I64/U8/U16/U32/U64/F32/F64 共 10+ 个元素类型家族的向量别名（每个家族含 4 种精度变体）。设计未说明矩阵别名是否覆盖全部家族（与向量别名一致）还是仅限基本浮点类型（mat2x2/mat3x3/mat4x2/mat4x4 等 × Float32/Float64）。缺少此清单，编码者无法确定 `fwd.cj` 的新增范围。

**所在位置**：§2 包组织（`fwd.cj ★ — 新增矩阵类型别名`）、§8 产出物清单

**严重程度**：中等 — 直接影响 `fwd.cj` 编码工作量估算和实现排期

**改进建议**：明确列出 `fwd.cj` 中需要新增的所有矩阵类型别名。建议格式参考现有的向量别名模式（如 `public type Mat2x2 = detail.Mat2x2<Float32, PackedHighp>`），按元素类型家族和精度变体分组枚举。

---

### 问题2（中等）：stub 文件缺少函数签名清单

**问题描述**：§3.7 仅提到"创建 common.cj、matrix.cj、geometric.cj 三个 stub 文件"，未提供每个 stub 需要声明的函数签名。编码者仅凭此信息无法知道每个 stub 文件应包含哪些函数。例如 common.cj 的 stub 需要为 type_mat3x3/type_mat4x4 的 .inl 编排提供依赖闭合，但具体需要哪些函数（min/max/abs/sign/floor/ceil/fract/mod/clamp/mix/step/smoothstep 等）不得而知。前序版本（`docs/04_ood_phase2.md`）中包含了完整的 stub 签名清单，但本设计未继承此细节。若本设计是唯一的编码参考，则此信息缺失将导致编码阶段需要额外回溯文档。

**所在位置**：§3.7 stub 函数库

**严重程度**：中等 — 编码者需要额外查阅前序文档或 GLM 参考头文件才能确定 stub 内容

**改进建议**：在每个 stub 文件的描述后附上完整的函数签名清单，包括函数名、参数类型、泛型约束和返回值类型。可参考 `docs/04_ood_phase2.md` §3.7 中已有的详细签名（common.cj 约 12 个函数，matrix.cj 含 transpose/determinant/inverse/matrixCompMult/outerProduct 的完整重载集）。

---

### 问题3（中等）：Vec 类型 extend 块中行向量×矩阵乘法运算符的具体签名未定义

**问题描述**：§2 包组织中标注 `type_vecN.cj（修改）★ — Vec1~Vec4 各新增行向量×矩阵成员运算符 Vec{R} * Mat{C}x{R} extend 块（方案A）`，但未指定每个 Vec 类型需要新增哪些具体运算重载。4 个 Vec 类型 × 9 个矩阵类型中，每个 Vec{R} 类型只需要与行向量维数匹配的矩阵类型（即 Vec{R} 只能乘 Mat{C}x{R} 其中 R 匹配）。设计应当给出每个 Vec 类型的 extend 块中需要定义的确切 `operator *` 签名列表。编码者仅凭"方案A"的描述无法确定 Vec1/Vec2/Vec3/Vec4 各自需要多少重载。

**所在位置**：§2 包组织（`type_vecN.cj（修改）`条目）、§3.5 算术运算符（行向量×矩阵）

**严重程度**：中等 — 缺少每个 Vec 类型的运算签名，编码阶段需要自行推导 29 个乘法重载中属于 Vec extend 块的部分

**改进建议**：按 Vec1/Vec2/Vec3/Vec4 逐一列出每个类型需要新增的 `operator *` 签名。例如：
- Vec4 extend 块：`operator func *(m: Mat2x4<T,Q>): Vec2<T,Q>`、`operator func *(m: Mat3x4<T,Q>): Vec3<T,Q>`、`operator func *(m: Mat4x4<T,Q>): Vec4<T,Q>`（共 3 个）
- Vec3 extend 块：`operator func *(m: Mat2x3<T,Q>): Vec2<T,Q>`、`operator func *(m: Mat3x3<T,Q>): Vec3<T,Q>`、`operator func *(m: Mat4x3<T,Q>): Vec4<T,Q>`（共 3 个）
- 以此类推。

### 问题4（轻微）：Bool 矩阵类型的支持策略未明确

**问题描述**：设计将所有算术运算符放在 `Number<T>` 约束的 extend 块中，但 Bool 类型不实现 `Number<T>` 接口。这意味着 `Mat2x2<Bool, Q>` 等 Bool 矩阵类型将无法使用 +、-、*、/ 等算术运算符，仅支持 ==/!= 比较。`fwd.cj` 阶段一已定义了 Bool 向量别名（BVec1~BVec4），说明 Bool 是受支持的标量类型。但设计未讨论矩阵类型对 Bool 的支持范围——是显式不支持 Bool 矩阵（应记录在 §9 差异声明）、还是部分支持（仅比较运算）、还是完全支持（为 Bool 提供替代约束路径如 `operator |`、`operator &` 等）。此策略缺失可能导致编码阶段后期发现 Bool 矩阵实例化时编译错误。

**所在位置**：§3.1 矩阵结构体（泛型参数 T）、§3.5 算术运算符（Number<T> 约束）

**严重程度**：轻微 — 当前阶段主要目标为 Float32/Float64 矩阵，Bool 矩阵使用场景较少，但策略不明确会在编码测试阶段产生不确定性

**改进建议**：在 §9 差异声明或 §7 设计决策中增加一条，明确说明 Bool 矩阵的支持范围。建议的两种方向：① 如果有意排除 Bool 矩阵，在 §9 记录并说明原因；② 如果支持，说明仅提供 ==/!= 和 `@Derive[Hashable]`（算术运算不可用）。

---

### 问题5（轻微）：`equalExact` 函数是否需要对矩阵类型提供未明确

**问题描述**：§3.6 比较运算符仅提到 ==/!= 和 equalEpsilon。但阶段一 Vec 类型同时提供了 `equalExact`（精确比较的显式具名版本）。作为与 Vec 类型设计一致的延续，矩阵类型是否也应提供 `equalExact` 成员函数？当前设计未说明。若不提供，建议在 §9 差异声明中记录以避免与 Vec 类型 API 风格不一致。

**所在位置**：§3.6 比较运算符

**严重程度**：轻微 — == 和 equalExact 在语义上等价（同为精确比较），不影响功能正确性

**改进建议**：补充说明 equalExact 的提供状态。若排除，在 §9 记录；若包含，补充成员函数签名定义。

## 整体评价

该设计文档在技术可行性分析（T(0)/T(1) 约束分析）、B 类方向 GLM 逐项对照验证、跨尺寸矩阵乘法完备性推导等方面质量较高。内部审议已充分覆盖技术可行性维度。

从编码落地视角看，上述问题1、2、3 属于**"知道要做什么，但不知道具体做哪些"**类型的问题——文档标明了需要修改/新增的文件和功能方向，但未给出编码者可直接执行的完整规格（签名级定义）。这可能导致编码阶段需要额外的前序文档回溯或 GLM 参考头文件分析，增加编码迭代成本。

问题4、5 属于 API 一致性和范围策略的完善，不影响当前阶段核心编码工作，但应在编码启动前明确。
