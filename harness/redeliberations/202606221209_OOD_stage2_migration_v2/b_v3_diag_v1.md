**审查轮次**：第 3 次
**审查视角**：需求响应充分度、事实准确性、深度和完整性（侧重内部审议未充分覆盖的维度）

## 问题清单

### 问题 1：§4.5 行为契约示例与 §3.3 形式化签名矛盾

- **问题描述**：§3.3 第7条跨类型同尺寸转换工厂函数的签名定义为 `fromMat<U, P>(conv: (U) -> T, m: Mat4x4<U, P>)`，`conv` 是第一个位置参数。但 §4.5 行为契约中的调用方式为 `Mat4x4<Float32, PackedHighp>.fromMat(mInt, conv: { x => Float32(x) })`——第一个位置实参传入了矩阵 `mInt`，而 `conv` 以命名参数形式出现。按照签名，第一个位置参数对应 `conv`，因此 `mInt` 会被作为 `conv` 参数传入，导致类型不匹配（矩阵值赋给函数类型参数），编译错误。
- **所在位置**：§3.3 第7条（签名声明）vs §4.5 行为契约（使用示例）
- **严重程度**：中等（事实性矛盾——实现者若按第7条签名编码，则契约示例无法编译；若按契约示例编码，则与签名不一致，产生歧义）
- **改进建议**：将 §4.5 的使用示例修正为与签名一致，即 `fromMat({ x => Float32(x) }, mInt)`（`conv` 在前，`mInt` 在后）。同时检查全文中其他使用 `fromMat` 第7条的调用示例是否也存在类似问题。

### 问题 2：fromMat 填充规则对非方阵边缘情形描述不完整

- **问题描述**：§3.3 第6a/6b条的填充规则在描述"额外行"和"额外列"的单位矩阵填充时，规则表述为"额外行从单位矩阵取值——即 (i,i) 填 `one`，其余填 T(0)"。该规则未显式处理"对角位置 (i,i) 在目标矩阵中不存在"的边缘情形——当 `R_dst > R_src` 且 `i ≥ C_dst` 时（或对称地，`C_dst > C_src` 且 `j ≥ R_dst` 时），目标矩阵的列数/行数约束使得对角位置无法写入。例如 `Mat3x2(3列2行)→Mat2x3(2列3行)` 场景：额外行 i=2 的 (2,2) 位置需要 j=2，但目标列数为2（j仅0,1），该位置不存在。当前规则缺少对此情形的明确指导，实现者需要自行推断。
- **所在位置**：§3.3 第6a/6b条"精确填充规则"段落
- **严重程度**：低（不影响正确实现，但设计文档应提供更精确的描述以减少编码阶段的推断成本）
- **改进建议**：在填充规则中补充边缘情形说明：当 `R_dst > R_src` 时，对于每个额外行 i ≥ R_src，若 i < C_dst 则 (i,i) 填 `one` 其余填 T(0)；若 i ≥ C_dst 则整行填 T(0)。对称地处理额外列情形。

### 问题 3：`length(): Int64` 的 const 属性未明确

- **问题描述**：§3.1 声明矩阵类型提供 `length(): Int64` 返回列数 C（编译期固定值），但未说明该函数是否为 `const func`。从功能性质看，`length()` 返回的是类型级常量，理论上应可声明为 `const func` 以支持编译期使用场景。该属性未被 const init/const func 可用性汇总表（§3.3 末尾表格）覆盖。
- **所在位置**：§3.1 矩阵结构体职责列表 + §3.3 const 可用性汇总表
- **严重程度**：低（不影响功能完整性，但若编码阶段发现非 const 设计不合理需要返工，会增加不必要的沟通成本）
- **改进建议**：在设计中明确 `length()` 的 const 属性。建议标注为 `const func`，并补充到 const 可用性汇总表中。

### 问题 4：fromMat 重载数量表述存在歧义

- **问题描述**：§3.3"跨矩阵类型转换工厂函数的数量推导"段落的引导句写"**72 个 fromMat 重载**的编译可行性评估"，但后续行文描述"每个目标矩阵类型的 extend 块中将包含 8 个 fromMat（6a）和 8 个 fromMat（6b）共 **16 个**静态函数签名"，9 个目标类型合计 **144 个**函数签名。"72 个"与"144 个"之间的对应关系对读者不清晰——72 指的是"转换方向"数（9×8=72 个源-目标对），每个方向有 2 个变体（6a+6b），共 144 个函数。引导句存在歧义，可能导致后续维护者对重载规模的认知偏差。
- **所在位置**：§3.3 第6条末尾"跨矩阵类型转换工厂函数的数量推导"段落
- **严重程度**：低（非事实性错误，但表述上的歧义会给后续维护带来理解成本）
- **改进建议**：将引导句修改为明确表达，如"**144 个 fromMat 函数签名**（72 个转换对 × 2 变体）的编译可行性评估"，或"**72 个转换对**（每对含 6a + 6b 双版本）的编译可行性评估"，并保持后续计数的清晰对应。

## 整体评价

该设计文档经过 15 轮修订（v1~v15），在技术方案完整性、问题覆盖度、差异声明详尽度等方面已达到较高成熟度。前两轮迭代发现的 9 个问题均已被合理处理。本轮审查发现的问题以文档一致性（问题1）和精度（问题2/3/4）为主，均为可修复的局部问题，不涉及设计方向或核心架构的调整。

## 需求响应充分度评估

需求要求对 GLM 1.0.3 阶段二迁移进行 OOD，参考原始头文件、阶段一代码和仓颉语言限制。设计完整覆盖了 9 个矩阵类型、构造函数体系、算术/比较运算符、stub 函数库、别名文件和差异声明，且对仓颉限制（如无约束泛型无法获取 T(0)/T(1)）的应对策略系统一致。与阶段一代码模式保持对齐。需求响应度满足要求。

## 编码落地就绪度评估

设计在以下方面提供了足够的编码指导：类型签名、约束条件、const 可用性、GLM 差异、模块间依赖关系、错误处理策略等。不足之处是行为契约示例（问题1）与签名不一致，以及少数边缘情形（问题2）的规则精度不足——这些问题修复后即可直接指导编码实现。
