# 质量审查报告 — a_v18_design_v1.md（v2）

## 审查范围

- **产出版本**：v29（第 18 轮迭代）
- **审查视角**：从实际落地视角出发，评估设计是否可直接指导编码、接口定义是否足以支持下游消费者、异常/边界条件是否已考虑
- **侧重维度**：需求响应充分度、整体深度和完整性、内部审议未覆盖的维度

## 审查发现

### 问题 1（严重）：fromMat 9×8 转换矩阵表缺少 Mat4x3 目标列

**问题描述**：
§3.3 fromMat 6a 完整 9×8 转换矩阵表的表头仅列出 8 个目标类型（→Mat2x2/2x3/2x4/3x2/3x3/3x4/4x2/4x4），**缺少 →Mat4x3**。GLM 1.0.3 中存在 `mat4x3` 类型，且具有从其他尺寸矩阵构造的构造函数。表头缺失导致以下 9 个方向的填充模式标注完全遗漏（包含 Mat4x3→Mat4x3 EQL）：

- Mat2x2 → Mat4x3
- Mat2x3 → Mat4x3
- Mat2x4 → Mat4x3
- Mat3x2 → Mat4x3
- Mat3x3 → Mat4x3
- Mat3x4 → Mat4x3
- Mat4x2 → Mat4x3
- Mat4x3 → Mat4x3（EQL）
- Mat4x4 → Mat4x3

此外，§3.3 item 6 正文中的算式推论有误：文档声称"9 源 × 8 目标 − 9 同尺寸 = 72"，但 `9×8-9=63` 而非 72。正确的理解是"9 源 × 8 非自身目标 = 72 方向"。去掉文本中的 `− 9 同尺寸` 即可修正。

**所在位置**：§3.3 fromMat 6a 9×8 转换矩阵表（L368-378）

**严重程度**：严重

**改进建议**：
- 将 9×8 表扩展为完整的 9×9 表（含 →Mat4x3 目标列），逐行标注 9 个源矩阵到 Mat4x3 的填充模式
- 同步修正 item 6 正文中的算式

### 问题 2（中等）：fromColumns 函数签名均省略了 where 约束子句

**问题描述**：
§3.3 item 5 中所有 9 个 `fromColumns` 签名示例均未写出 `where` 约束子句，仅在表上方以注释形式提醒。相较于 §3.3 其他条目（如 item 4 fromParts 显式写出 `where Q <: Qualifier`、item 6a/6b/7 均写出完整约束），item 5 的省略方式不一致。

**所在位置**：§3.3 item 5（L253-265）

**严重程度**：中等

**改进建议**：在 9 个签名示例中至少写出第一个（如 Mat2x2）的完整签名含 where 子句，其余 8 个可复用"（约束同前）"标注。

### 问题 3（中等）：复合赋值手动定义表格未区分"可用手动定义"与"抛异常手动定义"

**问题描述**：
§3.5 复合赋值手动定义回退方案表将方阵的 `/=(Mat)` 和 `*=(Mat)` 计入"需定义的运算符"数量（各 3 个，共 6 个），并纳入总计 60 个的统计中。但表后注释明确说明这两个运算符"在本阶段执行时抛出 stub 异常"。v29 修订说明（F3）已在表下方补充注释，但表格本身仍将二者与正常可用运算符并列计数，未做标注区分。实现者仅看表格仍会误以为这 6 个运算符也是本阶段可正常使用的。

**所在位置**：§3.5 复合赋值手动定义回退方案文件分布表（L444-460）

**严重程度**：中等

**改进建议**：
- 在表格中对 `*=(Mat)` 和 `/=(Mat)` 条目添加标注，如 `*=(Mat)†` 并在表脚注明"† 依赖 inverse stub，本阶段不可用"
- 或修改表格结构，将可用手动定义与 stub 手动定义分列展示
- 总计数字 60 应同步拆分为"54 个可用 + 6 个 stub"，或明确标注 6 个为不可用

### 问题 4（一般）：§9 差异声明 D18 未同步标注 Bool 矩阵排除

**问题描述**：
§3.3 L290 已将 "identity() 对所有 9 种矩阵类型均提供" 修正为 "对所有 9 种尺寸的矩阵类型均提供（Bool 元素类型除外，见 D33）"，消除了与 L303 注释的显式矛盾。但 §9 差异声明 D18（L766）未同步提及 Bool 矩阵排除——读者仅看 D18 会认为 identity 对所有元素类型（包括 Bool）的非方阵均提供，需结合 D33 才能获得完整信息。

**所在位置**：§9 差异声明 D18（L766）

**严重程度**：一般

**改进建议**：在 D18 条目末尾追加"(Bool 矩阵不提供此操作，见 D33)"，保持各章节间的一致性。

### 问题 5（一般）：§8 compile order fallback 方案优先级标注与编码建议可能存在时序依赖遗漏

**问题描述**：
§8 建议的文件组织顺序（基础类型 → 矩阵类型 → Vec extend 块 → 辅助文件）中，stub 文件（common.cj、geometric.cj、matrix.cj）被归入"辅助文件"类别放在最后。但 §2 模块间依赖明确指出 `type_mat4x4.cj` 的 .inl 编排依赖 `geometric.cj`（stub）。如果 stub 文件被放在最后编译，而 type_mat4x4.cj 在其 extend 块中引用了 geometric.cj 中的函数（如 dot/length），编译可能因前向引用失败。文档未讨论此时序依赖。

**所在位置**：§8 文件组织顺序建议（L706-711）、§2 模块间依赖（L130）

**严重程度**：一般

**改进建议**：在 §8 文件组织顺序中，将 stub 文件细化为"stub 文件（需在其他引用者之前编译）"并调整至矩阵类型文件之前或之间，或补充说明 stub 文件的编译顺序要求。

### 问题 6（一般）：fromMat 7（跨类型同尺寸转换）的模板化编码策略未说明 9 个重载的签名生成方式

**问题描述**：
§3.3 item 7 仅提供了一个 `Mat4x4` 的代表性签名示例，其余 8 个矩阵类型的 fromMat 7 签名未逐一列出。虽然模式是简单的类型替换，但考虑到 fromMat 7 的泛型约束与 fromMat 6a/6b 不同（无需 `T <: Number<T>`），实现者需从示例推断其余签名的约束。文档也未像 fromMat 6a/6b 那样明确统计 fromMat 7 的重载数量和推荐代码生成策略。

**所在位置**：§3.3 item 7（L286-288）

**严重程度**：一般

**改进建议**：补充说明 fromMat 7 共 9 个重载（每个矩阵类型一版），或像 item 4/5 那样以表格式逐一列出。

### 问题 7（一般）：fromMat 规则化编码策略缺少 R_src 的获取方式

**问题描述**：
§3.3 fromMat 6a 的"规则化的编码策略"段描述了统一实现模式：先获取 `C_src = typeof(src).length()`（返回列数 C）、`C_dst`、`R_dst`，然后按行列操作规则组合。但该策略**仅说明了如何获取 C_src 和 C_dst**（通过 `length()`），未说明如何获取 `R_src`（源矩阵行数）和 `R_dst`（目标矩阵行数）。`length()` 仅返回列数 C，不返回行数 R。实现者需要知道 R_src 才能判断是否需要行扩展/收缩，但设计文档未提供 R_src 的获取路径。

实际编码时，每个 fromMat 实现是一个针对具体 (C_src,R_src)→(C_dst,R_dst) 方向的特定方法，因此 R_src 和 R_dst 是编译期已知常量——文档应明确说明此依赖关系，即 R_src 和 R_dst 需要在每个 fromMat 实现中硬编码，而非通过 `length()` 或其他统一查询函数获取。

**所在位置**：§3.3 fromMat 6a 规则化的编码策略段（L382-387）

**严重程度**：一般

**改进建议**：
- 在规则化编码策略段中补充说明：R_src 和 R_dst 需在每个 fromMat 实现中硬编码（二者均为编译期已知常量，分别由源/目标矩阵类型唯一确定）
- 或考虑在矩阵类型上增加 `public static func rows(): Int64` 静态函数，保持与 `length()` 返回列数对称的接口一致性

### 问题 8（一般）：fromMat 6a/6b 的 `one: T` 参数在 shrink-only 方向上的语义冗余

**问题描述**：
§3.3 fromMat 6a/6b 的所有方向均要求调用方传入 `one: T` 参数。但在纯收缩方向（如 Mat4x4→Mat2x2，列收缩+行收缩）中，转换操作仅涉及截断，完全不需要 T(1) 值（无任何扩展操作）。要求调用方在收缩方向上额外提供 `one` 参数造成不必要的 API 负担——调用方必须传入一个不会被使用的值，且该值的选择可能引起困惑（"我传的 1.0f 会用到哪？"）。文档未讨论此场景。

**所在位置**：§3.3 fromMat 6a/6b（L267-284）

**严重程度**：一般

**改进建议**：
- 拆分为两个 API：`fromMat(m, one)`（含扩展的方向）和 `fromMat(m)`（纯收缩方向，无需 one 参数，仅需 6b 的 conv 闭包），消除无用参数
- 或在签名注释中明确标注"纯收缩方向下 one 参数被忽略"，消除调用方困惑

### 问题 9（一般）：§10 API 覆盖矩阵中 glm/common.hpp 的 trunc/round/roundEven 覆盖状态标注与 §3.7 不一致

**问题描述**：
§10 API 覆盖矩阵（L802）将 `trunc/round/roundEven` 标注为"本阶段 stub（标量部分：floor/ceil/fract）"，但括号内仅列出 floor/ceil/fract，未包含 trunc/round/roundEven。§3.7 common.cj 签名清单（L495-508）中也不包含 trunc/round/roundEven 的 stub 签名。两者的共同结论是 trunc/round/roundEven 不在本阶段 stub 范围内，但 §10 的"本阶段 stub"标识与括号内仅含 floor/ceil/fract 的限定形成矛盾——读者可能误解为这三者也是本阶段 stub 但签名被遗漏了。

修正方向：应将 "floor/ceil/fract" 后的 "trunc / round / roundEven" 移出本阶段 stub 范围（标注为"后续阶段"），或补充其 stub 签名并同步更新 §3.7。

**所在位置**：§10 GLM 1.0.3 API 阶段覆盖矩阵（L802）

**严重程度**：一般

**改进建议**：将 trunc/round/roundEven 从"本阶段 stub（标量部分）"行中移出，单独列为"后续阶段"条目，避免读者误判 stub 覆盖范围。

## 需求响应充分度评估

**结论：需求响应充分。**

经逐条对照 requirement.md 和 §10 API 覆盖矩阵，本设计文档对阶段二 OOD 需求的响应情况如下：

| 需求维度 | 覆盖情况 | 备注 |
|---------|---------|------|
| 9 个矩阵类型（Mat2x2~Mat4x4） | ✅ 全部覆盖 | §3.1-§3.3，每个矩阵有独立结构体定义 |
| 列主序存储 | ✅ 明确声明 | §3.1 与 GLM 约定一致 |
| 矩阵运算（±*/） | ✅ 全部覆盖，含跨尺寸乘法 | §3.5 共 27 个矩阵乘法重载 |
| 标量-矩阵运算 | ✅ 通过 scalar_mat_ops.cj 全局函数 | §3.5 36 个全局函数 |
| 矩阵-向量/向量-矩阵乘法 | ✅ Mat×Vec 成员运算符 + Vec extend 块 | §2 包组织矩阵 |
| 构造函数体系 | ✅ 逐分量、列向量、工厂函数 | §3.3 items 1-8 |
| 类型转换（fromMat） | ✅ 跨尺寸/跨类型/同尺寸全覆盖 | §3.3 items 6a/6b/7 |
| 下标访问 + col() | ✅ | §3.4 |
| 比较运算（==/!=/equalEpsilon/equalExact） | ✅ | §3.6 |
| transpose/matrixCompMult/outerProduct | ✅ 完整实现 | §3.7 matrix.cj |
| determinant/inverse | ✅ stub 占位 | §3.7 matrix.cj，阶段三完整实现 |
| common/geometric 函数 | ✅ stub 占位 | §3.7 common.cj/geometric.cj |
| 别名体系（fwd.cj + ext/） | ✅ 108 个新增别名 | §8 fwd.cj 清单 |
| 差异声明 | ✅ 覆盖全部已知差异 | §9 + deviations.md 交叉引用 |
| API 覆盖矩阵 | ✅ 新增 §10 | 按 GLM 头文件路径逐项标注 |

**主要缺口**：唯一未充分响应的点是 fromMat 9×8 转换矩阵表缺少 →Mat4x3 列（问题 1），导致该方向的设计未被文档覆盖。此缺口虽严重但修复成本低（扩展为 9×9 表即可），不影响 OOD 整体设计的可行性。

## 整体评价

设计在经历了 17 轮迭代修正后，核心 API 设计清晰、差异声明完整、GLM API 覆盖矩阵完备。需求响应充分，深度和完整性达到指导编码的水平。遗留问题集中在：① fromMat 转换矩阵表缺少 →Mat4x3 列（严重）；② 规则化编码策略中 R_src 获取方式未说明（一般）；③ fromMat 6a/6b `one` 参数在纯收缩方向存在语义冗余（一般）；④ 跨章节一致性细节（问题 4/5/9）。建议优先修复问题 1 后再进入编码阶段。
