## 问题 2（中等）：D08 及 §9 中 `++/--` 替代方案 `m += T(1)` 与自身结论矛盾

- **问题描述**：文档 §1 系统性设计约束和 deviations.md DV-01 明确指出：仓颉无约束泛型不支持 `T(1)` 构造调用，`Number<T>` 接口也不提供 `T(n)` 构造声明。但 D08 决策和 §9 差异声明中，建议作为 `++/--` 替代方案的写法为 `m += T(1)`——这在泛型上下文中不可编译，与文档自身反复论证的结论矛盾。在具体实例化场景（如 `Mat4x4<Float32, PackedHighp>`）中 `m += 1.0f` 虽可行，但文档未做此区分说明。
- **所在位置**：
  - D08（第 753 行）：`"可通过 m += T(1)（逐元素加 1）等价替代"`
  - §9 差异声明（第 837 行）：`"可通过 m += T(1)（逐元素加 1）等价替代"`
- **严重程度**：中等
- **改进建议**：修正替代建议表述，区分两种情况：① 在具体类型上下文（如 Float32 矩阵）中使用 `m += 1.0f`；② 在泛型上下文中的替代方案需显式传入 `one: T` 参数。

## 问题 3（中等）：缺少跨类型赋值运算符 `operator=` 的差异声明

- **问题描述**：C++ GLM 为全部矩阵类型定义了模板赋值运算符 `template<typename U> operator=(mat<C,R,U,Q> const& m)`，支持跨类型同尺寸矩阵赋值（如 `mat4(Float32) = mat4(Int32)`）。仓颉不允许重载 `operator=`，此能力完全不可用。但 §9 差异声明和 §3.5 运算符清单均未提及此差异。用户从 C++ GLM 迁移代码时，若使用了 `mat4_f32 = mat4_i32` 形式的跨类型矩阵赋值，将遇到编译错误且原因不明。
- **所在位置**：§9 差异声明（第 831-858 行）——缺少此条目；§3.5 运算符清单——未提及
- **严重程度**：中等
- **改进建议**：在 §9 差异声明中新增条目：`"无模板赋值运算符 operator=(mat<C,R,U,Q>)"`，说明跨类型同尺寸矩阵赋值不可用，建议通过 `fromMat(conv, m)` 或显式逐元素转换替代。

---

## 问题 4（中等）：`col()` 不可在 const 上下文中使用的限制未在 §9 中明确记录

- **问题描述**：§5 错误处理策略中提到 `[]` 和 `col()` 因内部包含 `assert` + fallback `throw` 而不可声明为 `const func`，§9 差异声明中仅记录了 `[]` 的 const 限制（第 857 行："`[]` 和 `col()` 不可在 `const` 上下文使用"），但该条差异描述的侧重点在 `[]` 上。由于 `col()` 是本阶段新增的 API 函数（C++ GLM 中无对应的成员函数 `col()`），用户可能不清楚其 const 属性。更关键的是，§3.3 末尾的 `const init/const func 可用性汇总表`（第 314-325 行）仅包含构造函数和工厂函数，未包含 `col()` 和 `[]` 运算符的 const 状态，导致信息查找需要跨章节定位。
- **所在位置**：§3.3 const 可用性汇总表（第 314-325 行）——缺少 `col()` 和 `[]`；§9 差异（第 857 行）——侧重点偏 `[]`
- **严重程度**：中等
- **改进建议**：在 §3.3 const 可用性汇总表中新增行：`"[] 取值版本"`（❌ 非 const func）和 `"col() 取值版本"`（❌ 非 const func）；在 §9 差异中单独突出 `col()` 的 const 限制。

## 问题 5（轻微）：`filled(scalar: T)` 的 T 类型参数来源未在行为契约中示例说明

- **问题描述**：`filled` 定义为 `Number<T>` 约束 extend 块中的静态工厂函数，Q 从外部类型推断。但 §4.1 行为契约中 `filled` 的 "完整泛型参数" 描述与 `identity` 不同——`identity` 示例明确标注了 "Q 从 Mat4x4<Float32, PackedHighp> 的第二个类型实参推断，T 从 one 参数推断"，而 `filled` 示例仅输入了类型参数 `Mat4x4<Float32, PackedHighp>.filled(1.0f)`，未说明 T 是如何推断的（从 scalar 参数类型推断，与 identity 中 one 参数的角色类似）。对于编码者来说这是一个信息缺口——尤其是当 scalar 参数类型与矩阵元素类型不同时（如 typed literal 歧义场景），缺少类型推断机制的明确说明。
- **所在位置**：§4.1（第 599-607 行）
- **严重程度**：轻微
- **改进建议**：在 §4.1 `filled` 的使用示例后补充类型推断说明，与 `identity` 的标注风格一致。

---

## 问题 6（轻微）：§9 差异声明未覆盖 `fromParts` / `fromColumns` / `fromMat` 的 const 能力差异

- **问题描述**：§9 差异声明记录了 `identity` 和 `filled` 不可在 const 上下文使用（第 858 行），但未记录 `fromParts`、`fromColumns`、`fromMat`（6a/6b/7）等工厂函数同样不可在 const 上下文使用。C++ GLM 中对应的跨类型构造函数（如 `mat4(ivec4)`）或跨尺寸转换构造（如 `mat4(mat2)`）部分可在 constexpr 上下文使用，而本设计对应的工厂函数均不可用。差异声明对 C++ GLM 中可用但仓颉不可用的能力覆盖不完整。
- **所在位置**：§9 差异声明（第 858 行仅覆盖 `identity` 和 `filled`）
- **严重程度**：轻微
- **改进建议**：扩展第 858 行差异记录的覆盖范围，说明 `fromParts`、`fromColumns`、`fromMat` 系列（共 6 个工厂函数族）同样不可在 const 上下文使用，并列出 C++ GLM 中对应可 constexpr 的构造场景。
