# OOD 设计方案审查报告（v21）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** 九种矩阵类型各自定义为独立的泛型结构体（struct），是 Cangjie 类型系统下正确的选择——Cangjie 不支持模板偏特化，无法实现单一 Mat<C,R,T,Q> 泛型后按 C/R 分别特化。

**[通过]** struct 值类型语义与矩阵的数学值对象角色匹配，所有运算符返回新实例、无副作用，与阶段一 Vec 类型策略一致。

**[通过]** 泛型约束使用 `where T <: Number<T>, Q <: Qualifier`，经查阅标准库文档 `Number<T>` 接口确实存在（std.math 文档明确列出）且支持 `-` 运算用于 T(0) 演算；`Qualifier` 作为类型参数的设计与阶段一保持一致。

**[通过]** T(1) 由调用方显式传入（`one: T` 参数）的策略正确——Cangjie 的 `Number<T>` 接口不提供 `T(1)` 构造声明，也不存在从 `Number<T>` 约束下演算 T(1) 的通用路径（`!` 属于 `Integer<T>` 接口，对 Float 不可用）。此约束已在 deviations.md DV-01 文档化。

**[通过]** `public static func length(): Int64` 在 struct 中定义正确，Cangjie 支持 struct 的静态成员函数。

**[通过]** operator func 重载（`+`, `-`, `*`, `/`, `[]`, `==`, `!=`）均在 Cangjie 可重载运算符列表内（function.md §8.2），且正确遵循"不能为 static、不能为泛型、须在 struct/enum/extend 中定义"等限制。

**[通过]** extend 块泛型约束模式——为已有类型添加带额外 `where` 约束的成员函数，经 extend.md §2.4 确认支持。

**[通过]** `@Derive[Hashable]` 可用于泛型 struct（deriving.md §1 确认支持），但 T=Bool 场景列为编码前验证项是必要的预防措施。

### 2. 标准库与生态覆盖

**[通过]** `Number<T>`、`Equatable<T>`、`Comparable<T>` 均为 Cangjie 标准库核心接口（std.core），设计中的使用方式正确。

**[通过]** `@OverflowWrapping` 是 Cangjie 内置溢出注解（reflect_and_annotation.md §1.1），可标注于函数声明上，作用于函数内整数运算和整型转换。

**[通过]** `@Derive[Hashable]` 由 `std.deriving.*` 提供，适用于泛型 struct，要求字段类型已实现 `Hashable`。`VecN<T,Q>` 在阶段一已实现，此假设合理。

**[通过]** `Exception` 是 Cangjie 标准异常基类，`throw Exception("stub")` 是合法的 stub 占位方式。

### 3. 语言特性可行性

**[通过]** 错误处理策略与 Cangjie 异常体系一致：下标越界抛出 `IndexOutOfBoundsException`（可用）或泛化 `Exception`；stub 函数抛出 `Exception` 占位；算术运算标注 `@OverflowWrapping` 合规。

**[通过]** 值类型矩阵所有运算符返回新实例，天然线程安全，与 Cangjie 的 M:N 线程模型兼容。

**[通过]** 包组织符合 cjpm 规范：`package glm.detail` 对应 `src/detail/` 目录，子包 `package glm.ext` 对应 `src/ext/` 目录（package.md §2.1 确认"目录名须与包名匹配"）。设计对 `ext/` 子包自动发现的不确定性提供了三档递进 fallback 方案和原型验证计划，风险覆盖充分。

**[通过]** 复合赋值运算符自动生成机制已查阅 function.md §8.5 确认："重载二元运算符（关系运算除外）时自动启用对应复合赋值版本，前提是返回类型与左操作数类型匹配或为其子类型"。设计同时提供了完整的手动定义回退方案（60 个运算符分布表），风险覆盖充分。

**[通过]** 编译顺序分析中的"同包延迟解析"声明标注为待验证并提供了三档 fallback 方案，设计谨慎合理。

### 4. 设计一致性

**[通过]** 各抽象职责描述清晰无歧义：矩阵结构体、stub 函数库、scalar_mat_ops.cj、Vec extend 块各行其责。

**[通过]** 模块间依赖方向合理：detail 包内类型引用遵循 type_mat → type_vec 方向；glm 包（fwd.cj/lib.cj）仅引用 detail 包；glm.ext 引用 detail 包；无循环依赖。

**[通过]** §3.3 的构造函数体系从 1 到 8 逐条展开，每个子类均提供完整泛型签名示例，行为契约（filled 对角填充、identity 主对角线填充、fromMat 列/行扩展规则等）描述精确。

**[通过]** B 类方向 GLM 逐项对照分析表验证了 9 个方向中 8 个与主规则一致，Mat4x4←Mat4x2 偏差已详细记录并给出编码阶段特殊处理要求。非 B 类方向补充了 3 组代表性方向分析，覆盖了 C/R 双扩展、C 缩 R 扩、C 扩 R 缩三种典型模式。

**[通过]** v21 版本已精确拆分列扩展规则（新列在 `(新列索引,新列索引)` 位置填入 `one`）与行扩展规则（新增行元素全部为 `T(0)`），消除了此前的歧义。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：矩阵结构体仅承载数据+基本运算符，各辅助函数库按领域分离（matrix.cj 负责矩阵运算、geometric.cj 负责几何运算、common.cj 负责基础数学）。

**[通过]** 抽象层次恰当：使用 9 个独立整编泛型结构体而非偏特化模板方案（后者在 Cangjie 中不可行），使用 stub 满足依赖闭合而非超前实现，使用 extend 块按约束分层暴露能力。

**[通过]** 设计便于后续实现：所有签名已定型，实现策略（内联展开、列主序映射等）已有方向，测试验证要求已明确（B 类方向单元测试、非 B 类方向编码验证、偏差处理模式）。

**[通过]** 设计便于单元测试：值语义确保无副作用；可通过构造非平凡源矩阵后逐元素断言来验证转换逻辑；Mat4x4←Mat4x2 偏差有专用测试要求；stub 函数可通过 mock 替换。

**[通过]** 历史问题 v21 版本已全部解决：填充规则歧义（问题1→已拆分列/行规则）、非方阵 identity 语义（问题2→已补充非方阵行为定义和示例）、非 B 类方向验证（问题3→已补充 3 组代表性方向）、fallback API 影响（问题4→已补充 API 变化影响说明）、matrix.cj stub 矛盾（问题5→已拆分为直接实现+stub 并标注各自数量）、跨 Qualifier 转换缺口文档化（问题6→已在 §9 补充）。

## 修改要求

无严重或一般问题。

## 注意事项

以下为轻微级别可改进项，供参考：

1. **（轻微）** `equalExact` 与 `==` 行为语义等价，提供具名版本是 API 设计选择。建议编码阶段确认 Cangjie 中 `operator func ==` 是否自动派生 `!=`；若是，可省略 `!=` 的显式定义。

2. **（轻微）** §9 差异声明中 `fromMat 6a 额外限制`（不支持跨 Qualifier 的同类型不同尺寸转换）已在当前版本中完整文档化，用户绕道路径明确，建议在未来的 API 文档中对用户做明确告示。
