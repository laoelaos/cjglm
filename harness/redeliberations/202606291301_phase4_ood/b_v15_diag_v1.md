## 发现的全部问题

### P1（严重）— gtc/noise.cj 周期性 Perlin 噪声仅覆盖 4D，遗漏 1D/2D/3D

**所在位置**：§3.3 gtc/noise.cj，第 657–672 行

**问题描述**：
GLM 1.0.3 `gtc/noise.hpp`（第 47–50 行）通过模板维度参数 `L` 为**所有维度**提供周期性 Perlin 噪声重载 `perlin(vec<L,T,Q>, vec<L,T,Q> rep)`，即 1D/2D/3D/4D 的周期版本一一对应。当前文档仅添加了 `perlin4D(v, period)` 一个维度，缺少 `perlin1D(x, period)`、`perlin2D(v, period)`、`perlin3D(v, period)` 三个周期重载。若按此设计编码，下游用户将无法使用 1D/2D/3D 的周期 Perlin 噪声。

**严重程度**：严重 — 功能遗漏，与 GLM 1.0.3 的 API 覆盖范围不符。

**改进建议**：
- 为 `perlin1D`/`perlin2D`/`perlin3D` 各补充一个周期参数重载，签名模式为 `func perlinND<T, Q>(v: VecN<T, Q>, period: VecN<T, Q>): T`（N=1/2/3），约束及实现路径参照 4D 版本。

### P2（严重）— gtc/random.cj 种子策略依赖未经验证的 Thread API

**所在位置**：§3.3 gtc/random.cj，第 687 行

**问题描述**：
种子生成策略使用 `Thread.currentThread().id` 与时间戳异或。H5 确定性声明仅验证了 `ThreadLocal<T>` 的可用性（来自 `core` 包，无需 `Send`/`Sync` 约束），但未验证 `Thread` 类型是否提供 `.id` 属性或等价的方法。仓颉标准库文档仅确认 `Thread.currentThread` 的存在（返回 `Thread` 实例），未记录 `Thread` 的其他 API。若 `.id` 属性不存在，种子生成策略将无法编译。

**严重程度**：严重 — 未经确认的 API 假设，若编译不通过则整个种子策略需要重设计。

**改进建议**：
- 新增验证项，确认 `Thread.currentThread().id` 的编译可行性。可替换方案包括：
  - 使用 `std.env.getProcessId()`（已验证可用）替代线程 ID，但所有线程共享进程 ID，碰撞概率略增；
  - 使用线程局部位移量（如 `ThreadLocal<AtomicInt64>` 计数器）生成唯一递增种子；
  - 删除线程 ID 混合，仅使用时间戳并接受低碰撞风险（与 GLM 1.0.3 一致）。

### P3（中等）— gtc/ulp.cj float_distance 描述与签名语义不一致

**所在位置**：§3.3 gtc/ulp.cj，第 728–729 行

**问题描述**：
描述文字为"返回 x 和 y 之间可表示浮点数的**个数**（以 ULP 为单位）"，"个数"暗示返回值为非负计数。但函数签名为 `(x: Float32, y: Float32): Int32` / `(x: Float64, y: Float64): Int64`，使用有符号整数类型。GLM 1.0.3 的 `float_distance` 返回的是有符号差值（可正可负），当 x < y 时为负数。描述中的"个数"与实际签名的有符号语义矛盾。

**严重程度**：中等 — 编码阶段会导致 confusion，开发人员不清楚返回值的取值范围。

**改进建议**：
- 方案 A（推荐，与 GLM 一致）：将描述修正为"返回 x 和 y 之间可表示浮点数的有符号差值（以 ULP 为单位），x < y 时结果为负"，保持 Int32/Int64 签名不变。
- 方案 B：若意图为非负距离，将返回类型改为 UInt32/UInt64，并在实现中使用绝对值 `|x_bits - y_bits|`。

### P4（中等）— gtc/ulp.cj 未讨论 Float16 覆盖策略

**所在位置**：§3.3 gtc/ulp.cj，第 726–732 行

**问题描述**：
文档声明 ulp.cj 使用"具体类型重载（Float32 + Float64）"，完全不提及 Float16。仓颉支持 `Float16` 且提供了 `toBits()`/`fromBits()` 方法（与 Float32/Float64 相同），ULP 相关函数在 Float16 上同样有数学意义。当前设计未说明 Float16 是否需要 ULP 函数支持，也未给出排除理由。

**严重程度**：中等 — 编码阶段若 Float16 ULP 需求出现，需重新设计。

**改进建议**：
- 在 §3.3 ulp.cj 末尾补充 Float16 策略声明。两种选项：
  - 选项 A：新增 Float16 重载（推荐，实现路径与 Float32 一致），设计决策中补充理由；
  - 选项 B：明确声明"Float16 不提供 ULP 函数，因 Float16 主要用于存储格式而非计算格式，ULP 比较需求极低"，并标注为已知缺失。

### P5（中等）— ext/quaternion_common.cj mix 描述存在不确定性措辞

**所在位置**：§3.2 ext/quaternion_common.cj，第 503 行

**问题描述**：
描述中写道"此函数**可能**需要 common.cj 的 clamp 来确保 a 在 [0,1] 内"，其中的"可能"（might need）引入了不确定性。D10 设计决策已明确"mix 使用 clamp(a, T(0), T(1)) 而非 assert"，§3.2 的描述应与决策一致，使用确定性措辞。

**严重程度**：中等 — 编码人员无法从"可能需要"判断是否应实现 clamp 保护。

**改进建议**：
- 删除"可能"二字，改为"此函数**使用** common.cj 的 clamp 将 a 截断至 [0,1] 后执行插值（见 D10），与阶段三 lerp 的 assert 策略不同"。

### P6（一般）— gtc/noise.cj 命名约定变更未记录为设计决策

**所在位置**：§3.3 gtc/noise.cj，第 655–672 行 vs. §7 设计决策

**问题描述**：
GLM 1.0.3 使用单一函数名 `perlin`/`simplex` 通过 C++ 模板维度参数 `L` 区分 1D/2D/3D/4D，本文档改用 `perlin1D~/perlin4D`、`simplex1D~/simplex4D` 的拆分命名。此变更是仓颉不支持整数维度泛型参数的直接结果，但未在 §7 设计决策章节中记录。相比下，`gtc/ulp.cj` 的"具体类型重载替代泛型"（D27）和 `modf`/`frexp` 的"元组返回替代引用参数"（D25）均单独成条。保持设计决策记录的完整性有助于后续维护。

**严重程度**：一般 — 不影响编码可行性，但决策追溯性不足。

**改进建议**：
- 在 §7 中新增一条设计决策（如 D31），记录噪声函数命名拆分的原因：仓颉不支持整数模板参数（如 C++ 的 `length_t L`），需手动为 1D/2D/3D/4D 各写单独函数签名的重载；使用 `perlin1D~/perlin4D` 命名模式以区分维度。

### P7（一般）— fwd.cj 与 gtc/type_precision.cj 的别名关系未澄清

**所在位置**：§3.3 gtc/type_precision.cj，第 692–719 行；§2 包组织，第 166 行

**问题描述**：
`glm.fwd.cj`（"沿用阶段三，本阶段不新增核心类型别名"）与 `glm.gtc.type_precision.cj`（"高精度类型别名定义"）都涉及类型别名。文档未说明两者关系：
- `fwd.cj` 中已定义哪些别名？`type_precision.cj` 是补充 `fwd.cj` 未覆盖的精度变体，还是独立重定义全部别名？
- 若 `type_precision.cj` 定义了与 `fwd.cj` 同名的别名（如在 `glm.gtc` 包中的 `fvec2` 和在 `glm.fwd.cj` 中可能的 `fvec2`），公共导入时是否存在命名冲突？

**严重程度**：一般 — 实际编译时可能暴露，但设计文档未覆盖此集成细节。

**改进建议**：
- 在 §3.3 type_precision.cj 末尾或 §9 集成方式中补充一段说明，明确 type_precision.cj 的别名与 fwd.cj 的关系：是"在 gtc 包中新增仅 gtc 可见的精度别名"还是"补充 fwd.cj 的完整精度别名体系"。
- 建议方案：type_precision.cj 中定义的别名位于 `glm.gtc` 包中，通过 lib.cj 的 `public import` 暴露至 `glm` 命名空间；`fwd.cj` 中的别名在 `glm` 包中。两者属不同包，`public import` 使它们在调用方（`glm` 包）命名空间中汇合，需确保无同名冲突。应在设计决策或集成说明中确认此兼容性。
