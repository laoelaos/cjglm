## 当前审查结果

**发现1（中等）**：`componentAt` 缺少验收层级的强制验证标准
- 所在位置：§12.2 验收标准表（全文）
- 问题描述：§12.1 层次二已新增 componentAt const 访问验证项，但 §12.2 验收标准表未同步增加对应的强制验收项。componentAt 的正确性验证仅停留在"测试项"层级（层次二），未升格为"验收标准"层级。若实现有缺陷（越界断言行为不一致、返回值不正确），验收审查时可能被遗漏。
- 改进建议：在 §12.2 验收标准表中新增一行：

| 验收项 | 标准 | 验证方法 |
|--------|------|---------|
| componentAt const 访问正确性 | const 函数体内调用 `componentAt(i)` 返回值与相同索引的 `operator[]` 取值形式一致；越界时触发断言 | §12.1 层次二 `componentAt` 验证项通过 |

**发现2（中等）**：§11 缺少"标量在左运算的具名函数替换"迁移成本评估
- 所在位置：§11（全文）
- 问题描述：`scalar + vec`、`scalar - vec`、`scalar / vec`（标量在左、向量在右）的不可重载性在 D10 和 §4.3 中已有充分分析，替代方案为具名函数 `add(s, v)`、`sub(s, v)`、`div(s, v)` 等。但 §11 迁移成本评估的 9 个子节（§11.1~§11.9）未设立独立章节评估此替换的迁移模式。实施者在将 C++ 代码 `s + vec` 迁移为 `add(s, vec)` 时，缺乏批量搜索替换规则和受影响范围评估。
- 改进建议：在 §11 中新增子节"§11.10 标量在左运算的具名函数替换"，覆盖：受影响表达式模式表、与 vec-op-scalar 方向扩展成员函数（`v.add(s)`）的关系和选择指南、搜索替换规则、工作量评估。

**发现3（一般，经质询调整焦点）**：`@Derive[Hashable]` 在泛型 struct 上对类型参数 T 的约束传递性未验证
- 所在位置：§3.2 Hashable 实现策略、§4.1 构造函数体系、§10 验证计划
- 问题描述（质询调整后）：Vec 结构体采用 `@Derive[Hashable]` 自动派生 `hashCode()`。质询确认，原始诊断中关于"仅 `const init` 无默认构造时编译器能否正确生成 hashCode"的假设在仓颉语言规范中无对应限制依据，`@Derive[Hashable]` 不要求默认构造函数存在。真实风险点应是 **泛型类型参数 T 的 `Hashable` 约束传递性**——即当 T 未实现 `Hashable` 时派生行为如何。此外，`HashSet<Vec2<Float32>>` 等浮点 Vec 在 `@Derive[Hashable]` 下的编译通过性在 §12.2 验收标准中未覆盖。
- 改进建议（质询调整后）：将发现焦点调整为"`@Derive[Hashable]` 在泛型 struct 上对类型参数 T 的约束传递性验证"，移除关于默认构造函数的无效假设；在 §10 新增验证依赖；在 §12.2 验收标准表"哈希集合可用性"项的验证方法中补充 `HashSet<Vec2<Float32, PackedHighp>>` 编译通过。

**发现4（一般）**：模块间依赖表未标注标准库依赖
- 所在位置：§2 模块间依赖表 `scalar_vec_ops` 行
- 问题描述：`scalar_vec_ops.cj` 中的浮点 `mod` 实现依赖 `std.math.trunc`（Float64 版本），且 Float32 路径涉及 Float64 提升后调用 `trunc`（§10 浮点 mod 验证依赖②）。但 §2 模块依赖表 `scalar_vec_ops` 行的依赖仅列出"qualifier"，未标注对 `std.math` 的标准库依赖。
- 改进建议：在 §2 模块间依赖表末尾的脚注或依赖拓扑说明中，补充一行标准库依赖汇总："`scalar_vec_ops.cj` 浮点路径依赖 `std.math.trunc`，编码阶段需在文件头部添加 `import std.math.{ trunc }`。" 或在 §8.1 文件清单的"说明"列中标注此标准库依赖。
