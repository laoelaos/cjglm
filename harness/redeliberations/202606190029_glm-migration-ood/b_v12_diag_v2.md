## 发现的问题

### P1（一般）：§10 多处残留"const if"独立术语，与 D34 约定不一致

**问题描述**：D34（§7）明确约定不再使用"const if"作为独立关键字表示。v12 P4 修复了 §3.4 和 §3.6 中的残留意象，但 §10 的以下位置未被覆盖：

| 位置 | 当前表述 | 应替换为 |
|------|---------|---------|
| §10 标题（第 1267 行） | "`const if` 编译抑制行为综合验证" | "编译期 `if` 分支抑制行为综合验证" |
| §10 首段（第 1269 行） | "`const if` 编译期分支的非选择分支抑制行为" | "编译期 `if` 分支的抑制行为" |
| §10 依赖点清单表（第 1273 行）第三列 | "const if 表达式" | "`const` 函数中的 `if` 表达式" |
| §10 依赖点清单表（第 1273-1277 行）"风险等级"列 | 多处 "若 `const if` 抑制不成立" | "若编译期 `if` 分支抑制不成立" |
| §10 统一验证方法（第 1281 行） | "验证 `const if` 在 `const` 上下文中的非选择分支抑制行为" | "验证 `const` 函数中 `if` 表达式的编译期分支抑制行为" |
| §10 统一验证（第 1303 行） | "若 `const if` 抑制不成立" | "若编译期 `if` 分支抑制不成立" |
| §10 mod 验证子节（第 1315 行） | "mod 函数边界值验证（独立于 const if 抑制）" | "mod 函数边界值验证（独立于编译期 `if` 分支）" |

**严重程度**：一般  
**影响分析**：术语不一致不影响设计正确性，但可能误导实施者尝试编写不存在的 `const if` 关键字。  
**改进建议**：对 §10 全文做一次 grep 确认所有"const if"出现位置，按 D34 约定的表述统一替换。

**严重程度**：一般

### P6（严重）：`const init` 构造函数缺少显式 `public` 可见性声明，Vec1 可能无法从外部构造

**问题描述**：§4.1 构造函数清单中，**所有 Vec 类型的 `const init` 构造函数均未添加 `public` 修饰符**。以仓颉语言默认可见性规则（`internal`——包级可见）为前提，这导致以下风险：

| Vec 类型 | 有无 `public` 外部可用的构造函数 |
|---------|-------------------------------|
| **Vec1** | **无**：唯一构造函数 `const init(x: T)` 若为 `internal`，外部包（`package glm` 及下游）无法构造任何 Vec1 实例 |
| Vec2~Vec4 | **间接可用**：存在 `public init(scalar: T)` 作为备选，但 `const init(x, y, ...)` 逐分量构造形式仍不可用——无法写 `Vec2(Float32(1.0), Float32(2.0))`，只能 `Vec2(Float32(1.0))` 标量填充 |

Vec1 尤其严重——它**只有** `const init(x: T)` 一个基础构造函数，如果该构造函数不是 `public`，Vec1 类型虽然可以被 `public import` 到 `package glm`，但外部代码无法创建 Vec1 实例。唯一路径是通过跨 Vec 构造函数（如 `Vec1<Int32>(Vec2<Int32>(...))`），这在语义上是荒谬的。

即使设计者假设 `const init` 在仓颉中隐式为 `public`，文档中也**没有在任何位置明确声明此假设**——实施者无法确定。第 1712 行附近明确指出 `var` 成员无初始值且编译器不生成默认构造函数，"调用方须显式构造"，但如果 `const init` 不可见，调用方无法显式构造。

**严重程度**：严重  
**影响分析**：若 `const init` 默认可见性为 `internal`，整个 Vec 类型体系对外不可构造——设计承诺的"下游消费者只需 `import glm.*` 即可使用所有公共 API"将落空。这是一项阻塞编码启动的实现前提性假设，必须在编码前明确。  
**改进建议**：以下二选一：
1. 在 §4.1 所有 `const init` 签名前补充 `public` 修饰符，并在一处位置（如 §7 D11 或 §4.1 开头注释）明确声明"所有 `const init` 构造函数均为 `public`"。
2. 若仓颉语言规范明确 `init` 构造函数隐式为 `public`，在 §7 设计决策（D11 或新增 D36）中记录此语言特性假设，并作为 §10 验证计划的一部分编写原型测试验证。

同时，在 §12.2 验收标准表中新增验收项："从独立测试包中通过 `import glm.Vec2` 导入后，验证 `Vec2<Float32, PackedHighp>(1.0f, 2.0f)` 构造可编译通过"。

## 附录：§8.4 前置验证中 GLM_CONFIG_CLIP_CONTROL 值的交叉验证

经查阅 `references/glm-1.0.3/glm/glm/detail/setup.hpp` 第 573-585 行，文档 §7 D6 中断言的"默认值为 `RH_NO = 0x0A`"正确：

- `RH_BIT = 1 << 3 = 8`，`NO_BIT = 1 << 1 = 2`  
- `RH_NO = 8 | 2 = 10 = 0x0A`  
- 条件编译：`GLM_FORCE_DEPTH_ZERO_TO_ONE` 未定义 → 走 `#else` 分支；`GLM_FORCE_LEFT_HANDED` 未定义 → 走 `#else` 分支 → `GLM_CLIP_CONTROL_RH_NO`

文档值与参考实现一致。
