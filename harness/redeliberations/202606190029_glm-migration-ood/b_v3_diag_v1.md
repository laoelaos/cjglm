**产出**：OOD 设计方案 v3（审查迭代修订版）
**审查轮次**：第 3 轮
**审查视角**：需求响应充分度、事实正确性、深度与完整性、可落地性

## 问题 1（严重 / 事实错误）：`bitwiseNot()` 对 `VecN<Bool, Q>` 的排除策略在仓颉中不可实现

**所在位置**：§3.2 `VecN<Bool, Q>` 行为约定（第 175 行）、§7 D25（第 858-862 行）

**问题描述**：设计文档声明"若编码阶段需要与 C++ GLM 行为严格一致，应为 `VecN<Bool, Q>` 排除 `bitwiseNot()`，使其在实例化时产生编译错误"。此建议在仓颉语言中**不可实现**。原因如下：

1. 仓颉不支持偏特化（Roadmap §0.1 已确认：`❌ 不支持`），无法为特定 `T = Bool` 的 `VecN` 实例排除或重新定义 `bitwiseNot()` 方法。
2. 仓颉的 `where` 子句仅支持正向约束（要求类型实现某接口），不支持负向约束（排除特定类型）。
3. `VecN` 的泛型参数 `T` 采用 D5 宽松约束（无 `Number<T>` 约束），不存在一个"所有数值类型实现但 `Bool` 不实现"的接口可用于条件性排除。

因此，`bitwiseNot()` 对 `VecN<Bool, Q>` 将始终可编译通过（行为为逻辑否定），编码人员无法按文档建议的路径消除此差异。如果 C++ 迁移代码中使用了 `~bvec` 且被机械地翻译为 `bitwiseNot()`，将在运行时获得**相反结果且无编译错误提示**。

**改进建议**：
- 方案 A（推荐）：承认该行为差异不可消除，在 §3.2 和 §11.4 中移除"可通过排除实现严格兼容"的不可行建议，改为明确声明："此为不可消除的已知行为差异。迁移 `~bvec` 时需要人工审查：若期望"全 true"行为，替换为 `VecN<Bool, Q>(true)`；若期望逻辑否定，使用 `!` 运算符。"
- 方案 B：若仍希望在编码阶段保留检测能力，改为在 `bitwiseNot()` 函数体内添加运行时 `weak_assert` 或文档标注，但需明确这无法在编译期捕获问题。

## 问题 2（中等 / 需求响应充分度）：别名命名约定偏离 C++ GLM API，迁移成本未纳入 §11 评估

**所在位置**：§3.8 命名约定（第 317-319 行）、§11 迁移成本评估（全文）

**问题描述**：设计将 256 个别名统一采用 PascalCase 风格（如 `BVec2`、`IVec3`、`Vec4`），这与 C++ GLM 的原始小写命名（`bvec2`、`ivec3`、`vec4`）完全不一致。C++ GLM 用户代码中大量使用这些类型名作为变量声明和函数签名的一部分。迁移此类代码时需要进行系统性的全局重命名（`bvec2` → `BVec2`、`ivec3` → `IVec3`、`vec4` → `Vec4` 等），涉及所有 `import` 语句、类型注解、函数签名和泛型约束。此命名偏差的迁移工作量和影响范围在 §11 中**未被任何子节覆盖**。

**对比**：§11 覆盖了 `&&`/`||` 具名函数迁移（§11.1）、`++`/`--` 迁移（§11.2）、跨类型赋值差异（§11.3）、`bitwiseNot` 迁移（§11.4）、`Inf == Inf` 差异（§11.5）、`@OverflowWrapping` 备选方案成本（§11.6），但未包含最具影响力的 API 重命名成本。

**改进建议**：在 §11 中新增子节评估别名重命名的迁移成本，包括：
- 受影响的范围（所有使用 GLM Vec 类型名的代码）
- 重命名映射表（C++ GLM 小写名 → 仓颉 PascalCase 名）
- 机械替换的可行性（全局搜索替换的边界条件和风险）
- 建议的迁移工具策略（正则脚本 + 代码审查）

## 问题 3（中等 / 关键遗漏）：跨 Q 赋值迁移模式未纳入 §11

**所在位置**：§11（全文）

**问题描述**：C++ GLM 中 `qualifier` 是枚举参数且支持默认值（`qualifier Q = defaultp`），代码中通常不显式指定 Q。但仓颉泛型不支持类型参数默认值（§3.2 已确认），所有 Vec 类型必须显式写出 Q 参数（如 `Vec2<Float32, PackedHighp>`）。迁移 C++ 代码中的 `glm::vec2` 为仓颉 `glm::Vec2<Float32, PackedHighp>` 时，需要为每个 Vec 类型声明补全 Q 参数。此迁移负担在 §11 中未被评估，编码人员首次遇到时可能低估其影响范围。

**影响量化**：
- 每个独立的 Vec 类型使用处需要补全 Q 参数
- `using`/`typedef` 在仓颉中映射为 `type` 别名（集中在 `fwd.cj`），但函数参数和局部变量需要逐处修改
- 不影响已有别名（`type Vec2f = Vec2<Float32, PackedHighp>` 已在 `fwd.cj` 中定义）

**改进建议**：在 §11 中补充跨 Q 赋值的迁移模式说明，明确：
- 哪些场景需要补全 Q（类型声明、函数签名中的 Vec 参数等）
- 哪些场景可通过别名避免（使用 `fwd.cj` 中的预定义别名）
- 搜索替换规则和自动化工具的可行性评估

## 问题 4（一般 / 事实错误）：`epsilonOf<T>()` Option B 兼容性策略不可行

**所在位置**：§3.5 兼容性策略 Option B（第 256-257 行）

**问题描述**：设计提出的兼容性策略 Option B 建议"在 `shim_limits.cj` 中为 `NumericLimits<Bool>` 添加空壳特化（仅声明结构体，不实现 `epsilon()` 方法），使 `Bool` 类型满足 `Number<T>` 约束的类型检查但不实际调用"。此方案与以下设计约束冲突：

1. **D18 约束**：`NumericLimits<T>` 保留了 `where T <: Number<T>` 约束。`Bool` 不实现 `Number<T>`，因此 `NumericLimits<Bool>` 的声明本身就违反约束条件，无法编译通过。
2. **缺乏偏特化**：仓颉不支持 C++ 式模板特化（Roadmap §0.1），无法为 `Bool` 提供无约束的 `NumericLimits` 特化版本。
3. **文档自认不推荐**：设计已标注 Option B"不推荐"，但其存在本身会误导编码人员在遇到约束问题时尝试此路径，浪费调试时间。

**改进建议**：从 §3.5 中完全移除 Option B，仅保留 Option A（内联固定容差）作为唯一的兼容性策略。Option B 的不可行性应在编码阶段的注意事项中明确排除，避免误导。

## 问题 5（一般 / 关键遗漏）：`operator[]` const 取值形式未提供

**所在位置**：§4.2 分量访问（第 500-502 行）、§7 D28（第 876-880 行）

**问题描述**：设计定义了 `operator[]` 的两种形式——非 `mut` 取值形式和 `mut` 赋值形式。但未提供 `const` 版本的取值形式。这意味着 `operator[]` 不能在 `const` 实例成员函数（如 `const operator ==` 或其他 `const` 方法）中作为分量访问手段。当前设计通过直接访问具名成员（`this.x`、`this.y` 等）绕过了此问题，但：

1. 任何未来需要在 `const` 上下文中通过 `[]` 访问分量的代码将无法编译。
2. 使用者期望 `v[0]` 这类语法在 `const` 函数中可用是合理的，当前设计构成隐性限制。
3. 作为公开 API 的 `[]` 运算符，缺少 `const` 版本是不完整的运算符设计。

**改进建议**：补充 `const operator func [](i: Int64): T` 取值形式，与现有的非 `const` 取值形式共存。仓颉支持运算符重载的 `const` 版本——`const` 修饰符不构成重载区分依据（const README §3.2 规则 7），但 `const` 版本与现有版本参数列表相同会导致重复定义错误。因此需要验证仓颉是否支持仅通过 `const` 修饰符区分运算符版本。若不支持，需通过其他方式（如具名 const 访问函数 `componentAt(i: Int64): T`）提供 const 上下文中的分量访问能力。

## 问题 6（一般 / 深度不足）：`mod` 浮点实现的外部依赖未显式验证

**所在位置**：§4.3 `%` 运算的整数约束（第 527 行）

**问题描述**：设计声明浮点 `mod` 采用算术恒等式 `x - y * trunc(x / y)`，其中 `trunc` 取自 `std.math`。本次审查已确认 `std.math.trunc` 存在（取 `Float64` 参数返回 `Float64`），但设计文档本身未记录此验证过程。具体风险点：

1. `Float32` → `Float64` 提升→截断→转回的精度策略未经验证：`Float32` 精度约 7 位十进制数，`Float64` 约 15 位，提升后计算再转回在理论上是无损的，但不同编译器实现的浮点常量折叠规则可能在极端值（如 `Float32` 的最大/最小归一化数）下产生差异。
2. `const if` 分支中调用 `trunc`：虽然编译期选路的 `const if` 不要求分支内代码为 `const`，但此调用链（`const if` → `trunc` → `std.math`）的编译器兼容性未验证。

**改进建议**：在 §10 的设计阶段验证要求表中新增一项，验证 `Float32` 浮点 `mod` 通过算术恒等式实现的正确性，至少覆盖边界值（`Float32.Max`、`Float32.MinPositive`、`0.0`、`±Inf`、`NaN`）下的行为。

## 整体评价

### 需求响应充分度
设计完整覆盖了 Roadmap §3E/§3G 定义的首轮迁移范围（基础设施层 + Vec1~Vec4 + 标量和向量类型别名），§8 的范围可追溯性对照表有效建立了 Roadmap→设计的映射关系。**主要不足**：PascalCase 别名命名导致的可迁移性问题未被充分认识，跨 Q 参数显式指定的迁移成本未被评估。

### 事实正确性
设计整体事实正确，经过前两轮审查后已修正了多个严重错误（`@OverflowWrapping` 委托方向、`CLIP_CONTROL` 默认值、跨类型构造参数类型等）。**主要残留问题**：`bitwiseNot()` 对 Bool Vec 的排除策略（问题 1）和 `epsilonOf<T>()` Option B（问题 4）在仓颉中技术上不可行。

### 深度与完整性
设计对 Vec 类型的构造函数体系、运算符重载、溢出策略、Qualifier 体系等核心方面做了充分的设计推演，方案深入到了仓颉语法级别。§7 的 31 条设计决策和 §11 的迁移成本评估具有参考价值。**主要不足**：
- §12 验收标准完善，但部分关键假设（`const if` 表达式链、`@OverflowWrapping` 继承性）经过 3 轮迭代仍未完成编译器验证
- `operator[]` 缺少 const 取值形式（问题 5）
- 未验证浮点 `mod` 实现的边界行为（问题 6）

### 可落地性
设计可直接指导编码实现的程度较高——每个 Vec 类型提供了完整的构造函数签名清单、运算符签名、`extend` 块语法示例和错误处理策略。编码人员在理解 D5 宽松约束的前提下可以开始实现。但需注意上述未验证的依赖项可能在编码阶段触发回退路径，导致设计方案实质性的返工。
