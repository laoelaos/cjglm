# 质量审查报告

## 审查概要

- **审查产出**：`a_v6_design_v2.md`（GLM 1.0.3 仓颉迁移首轮 OOD 设计方案）
- **审查视角**：需求响应充分度、事实正确性、深度完整性、落地可行性
- **迭代轮次**：第 6 轮
- **审查范围**：重点覆盖内部审议（组件 A）未能充分覆盖的维度，避免重复已确认的技术可行性结论

---

## 发现的问题

### P1（严重）— 文档版本标识自相矛盾，实施者无法确定当前版本

- **所在位置**：文档标题（第 1 行）、文件路径名、文档末尾「修订说明（v6）」
- **问题描述**：文档标题声明为 `v5 — 审查迭代修订版`，文件路径名为 `a_v6_design_v2.md`，末尾最新修订说明标记为 `v6`。三重版本标识不一致。此外，文档末尾的修订说明序列混乱——从 `v7` 开始历经 v8→v15，然后跳回 v2→v6，实施者无法据此判断文档主体所代表的实际版本。
- **严重程度**：严重
- **改进建议**：① 将文档标题统一为当前实际版本号（v6），与文件路径名和最新修订说明一致；② 整理修订说明部分，按时间正序排列并标注当前生效版本，将历史修订移入附录或归档文件；③ 在文档顶部增加版本变更摘要行（如 `Last revised: v6 (2026-06-19)`），使实施者一眼可知当前版本。

### P2（严重）— 缺少包构建基础设施定义，无法直接指导编码启动

- **所在位置**：§2 模块划分、§8 迁移文件清单、全文
- **问题描述**：设计详细描述了文件组织和包结构（`glm.detail`/`glm`），但未定义以下编码启动的前提条件：
  - 包配置文件（`cjpm.toml`）的内容——包名、版本号、依赖声明等
  - 编译入口和构建命令（`cjpm build` 的配置前提）
  - 项目初始化的脚手架步骤（`cjpm init` → 目录调整 → 文件填充的完整流程）
  - 测试运行器配置（`@Test`/`@TestCase` 框架的使用前提）
  实施者拿到设计文档后，无法从"零"开始直接执行第一个编译步骤。
- **严重程度**：严重
- **改进建议**：在 §2 模块划分之前或 §8 迁移顺序中补充"前置准备工作"子节，提供最小 `cjpm.toml` 模板、初始目录结构示例和第一条编译验证命令。确保实施者能按文档所述从 `mkdir` 到 `cjpm build` 通过无中断路径执行。

### P3（严重）— D32 声明 40 个算术函数为 `const`，但 const 兼容性验证计划（D29）未覆盖

- **所在位置**：§7 D32（第 935-939 行）、§7 D29（第 887-901 行）、§10 验证要求表（第 1080-1088 行）
- **问题描述**：D32 将 `add`/`sub`/`mul`/`div`/`mod` 共 40 个函数（两种方向 × 四种 Vec 类型 × 五种运算）统一声明为 `const` 函数。但：
  - D29 的 const 表达式链验证仅针对 `const operator ==`（`abs(a - b) <= epsilonOf<T>()`），不覆盖算术运算的 const 兼容性
  - §10 的设计阶段验证要求仅覆盖 `isIec559Of<T>()` 的实现、`abs()`/`epsilon()` 的 const 兼容性和浮点 `mod` 的算术恒等式正确性——同样未覆盖算术运算函数体（如 `component + s`）在 `const` 上下文中的编译通过性
  - 若目标类型 `T` 的算术运算符（`+`、`-`、`*`、`/`）在 `const` 函数体内不可用，则 40 个函数的 `const` 声明将同时失效，影响范围远超 D29 回退方案所覆盖的范围
- **严重程度**：严重
- **改进建议**：① 在 §7 D32 中补充 `const` 兼容性验证要求，引用 D29 的验证模式扩展至算术运算；② 在 §10 中新增"算术运算 const 兼容性"验证依赖项，覆盖 `T + T`、`T - T`、`T * T`、`T / T`（对 `Int32`、`Float32`、`Bool` 三种代表性类型）在 `const` 函数体内的编译通过性；③ 定义 D32 的独立回退方案：若部分 `T` 不支持 `const` 算术，则应降级为仅 `const if` 的必要函数（`mod`）保留 `const`，其余函数移除 `const` 修饰符，并在 D29 或 §10 中记录此回退路径。

### P4（中等）— `compute_vector_decl.cj` 的双重角色（消费型+预留型）架构层次模糊

- **所在位置**：§2 模块划分（第 46 行）、§7 D16（第 795-802 行）
- **问题描述**：`compute_vector_decl.cj` 同时包含两类代码：(1) 首轮**被消费**的 `scalar-op-vec` 方向辅助函数（`add(s, v)`、`sub(s, v)` 等 20 个函数）；(2) 首轮**仅定义不消费**的 `compute_vec_*` 策略结构体（为后续 SIMD 轮次预留）。D16 将此列为"例外项"，但这种混合架构导致：
  - 编码人员在修改被消费的函数时可能不慎影响到预留的策略结构体
  - 后续轮次引入 SIMD 路径时需要拆分该文件，产生重构成本
  - 文件职责不单一（single responsibility principle 违反），文件级 import 可见性难以精确控制
- **严重程度**：中等
- **改进建议**：将 `scalar-op-vec` 方向辅助函数独立为单独文件（如 `scalar_vec_ops.cj`），与预留的 `compute_vec_*` 策略结构体分离。`compute_vector_decl.cj` 仅保留策略结构体定义。`lib.cj` 的 `public import` 改为引用新文件中的具名函数。此拆分在首轮实施成本低（仅增加一个文件），可消除架构模糊并为后续轮次提供清晰的扩展边界。

### P5（中等）— `Float16` 的 `T(0)` const 构造兼容性未在验证计划中覆盖

- **所在位置**：§10 设计阶段验证要求表（第 1082-1086 行）、§10 `isIec559Of` 对照表（第 1065 行）
- **问题描述**：`isIec559Of<T>()` 的实现为 `const if (T(0) is Float64 || T(0) is Float32)`。§10 验证依赖①列出了 `T(0)` const 构造的验证，但验证方法中仅指定了 `T = Int32, Float32, Bool` 三种类型，未包含 `Float16`。虽然设计声明 `Float16` 不在首轮类型系统中活跃使用，但 `isIec559Of<T>()` 是一个无约束泛型函数，理论上可被任意 `T` 实例化。如果调用方在不知情的情况下对 `Float16` 调用了该函数，或者未来的维护者对 Vec 泛型添加了 `Float16` 支持，则 `T(0)` const 构造可能失败。当前验证计划未排除此风险。
- **严重程度**：中等
- **改进建议**：在 §10 验证依赖①中将 `Float16` 也纳入验证类型列表（与 `Int32`、`Float32`、`Bool` 并列）。若验证发现 `Float16(0)` 在 const 上下文中不可用，则在 `isIec559Of` 实现前增加 `const if` 前置防护，或在函数文档注释中标注 `Float16` 的限制。

### P6（中等）— 缺少增量验证节点，迁移顺序无法作为编码质量门禁

- **所在位置**：§8.3 迁移顺序（第 994 行）、§12 实现验证与验收策略（第 1369 行起）
- **问题描述**：§8.3 定义了迁移顺序（`setup.cj` + shim 文件 → `qualifier.cj` → 辅助工具 → Vec 类型 → 别名），§12 定义了最终验收层次，但两者之间缺少**逐步骤的增量验证节点**。实施者在完成每个文件/每组文件后应执行什么验证标准才能确认可进入下一步？例如：
  - 完成 `setup.cj` 和三个 shim 文件后，应验证什么？
  - 完成 `qualifier.cj` 后，如何确认 Qualifier 体系正确？
  - 完成 `type_vec2.cj` 后，可否仅编译验证还是需要写单元测试？
  当前设计将验证集中到了最终阶段（§12），增加了早期错误被延迟发现的风险。对于 1665 行的复杂设计文档，缺少中期验证节点会导致错误级联。
- **严重程度**：中等
- **改进建议**：在 §8.3 迁移顺序表中为每个阶段/子阶段补充"增量验证"列，标注该阶段完成后应执行的验证动作（编译通过/原型测试/手动审查）。例如：
  - 阶段 1（setup + shims）：`cjpm build` 无报错
  - 阶段 2（qualifier）：编写原型测试确认 `PackedHighp <: Qualifier` 编译通过、跨 Q 构造合法
  - 阶段 3（Vec 类型）：至少一个 Vec 类型（如 `Vec2<Int32, PackedHighp>`）的编译验证通过
  - 阶段 4（别名）：首批别名编译验证、重复定义检测

### P7（一般）— §9 异常场景表引用了 `equalEpsilon` 但该函数未在首轮范围内

- **所在位置**：§9.3 NaN/Infinity 传播表（第 1034 行：`equalEpsilon` 具名函数）
- **问题描述**：§9.3 的"NaN/Infinity 传播"行为约定表中，第 1034 行写道"调用方可通过 `equalEpsilon` 具名函数获取更精确的控制，不属于首轮范围"。此描述自相矛盾——`equalEpsilon` 既被引用为 NaN/Infinity 比较的"补救措施"，又被声明为"不属于首轮范围"。实施者会困惑：究竟应如何处理 Infinity 比较场景？如果 `equalEpsilon` 不在首轮范围内，则 §9.3 对它的引用是无效指引。
- **严重程度**：一般
- **改进建议**：将 §9.3 中的 `equalEpsilon` 引用替换为 `equalExact`（已在 §4.5 中定义且确认为首轮范围），或删除该引用并明确说明"NaN/Infinity 的精确比较通过 `equalExact()` 完成，详见 §4.5"。如果 `equalEpsilon` 确属后续轮次，则将其从 §9.3 中完全移除，避免给实施者双重信号。

### P8（一般）— 256 个别名降级策略的触发条件语焉不详

- **所在位置**：§3.8 首轮实现策略（第 355-358 行）
- **问题描述**：§3.8 描述了"若首轮孵化阶段因进度或资源限制需要缩小范围"可将别名从 256 缩减至 64 个，同时声明"必须触发范围变更评审"。但未定义：
  - 谁有权决定触发缩编（主 Agent/实施者/审查者）？
  - 缩编的触发阈值是什么（如"进度落后 2 周"或"编译验证失败"）？
  - 64 个之外剩余的 192 个别名何时必须补全？"第 2 轮"的具体轮次编号是？
  缺少这些定义，降级策略在实际执行中可能被滥用为随意裁剪范围的借口，或相反——无人敢触发缩编导致进度风险。
- **严重程度**：一般
- **改进建议**：① 在 §3.8 中补充降级触发条件的量化标准（如"编译验证阶段发现别名文件编译时间超过 30 分钟"或"核心 Vec 类型编码进度落后 2 周以上"）；② 指定触发决策的审批路径和文档化要求（如"范围变更评审需由主 Agent 记录到迭代历史文件"）；③ 明确缩编后剩余别名的最迟补全期限（如"第 7 轮审议前必须全量补全"）。

### P9（一般）— `lib.cj` 的 `public import` 行在文档的多处位置不一致

- **所在位置**：§2 公共 API 面设计（第 67 行）、§8.1 文件清单（第 968 行）、§8.2 范围追溯（第 990 行）
- **问题描述**：`lib.cj` 的 `public import` 列表在三处描述略有差异：
  - §2 第 67 行的 `public import` 包括 `compute_vector_decl.{ add, sub, mul, div, mod }`
  - §8.1 第 968 行的说明为"公共 API 重导出——通过 `public import` 暴露 Vec 类型、Qualifier 类型及 `scalar-op-vec` 方向辅助函数"
  - §8.2 第 990 行的偏离说明为"新增 `lib.cj` 用于 `public import` 重导出核心类型和 `scalar-op-vec` 方向辅助函数"
  - 但均未以精确的仓颉语法完整列出 `public import` 全部内容
  对于实施者，需要知道 `lib.cj` 中每一行 `public import` 的确切内容才能编码，而当前的分散描述方式可能遗漏某些类型（如 `Defaultp`、`PackedHighp` 等 Qualifier 实现类型是否全部导出）。
- **严重程度**：一般
- **改进建议**：在 §2 公共 API 面设计（第 60-74 行）中以完整代码块形式给出 `lib.cj` 的完整内容（包含每一条 `public import`），并在 §8.1 的文件清单中标记为"见 §2 完整清单"。消除多处分散引用的不一致风险。

## 整体质量评价

本设计文档在技术深度和细节覆盖上表现出较高水准，多次迭代审议已消除了大量技术可行性问题和内部矛盾。但从"实施者可依此直接编码"的落地视角看，仍存在以下系统性不足：

1. **版本管理的马虎**：标题版本号与文件路径、修订说明不一致，直接影响实施者对新版本内容的信任度。
2. **构建上下文缺失**：缺少 `cjpm.toml` 等包基础设施定义，实施者无法从零开始编译第一个文件。
3. **验证计划覆盖面不足**：D32 大量 `const` 函数声明的验证未被 D29 覆盖，存在整体失效风险。
4. **中间节点缺失**：缺少增量验证节点设计，1665 行的复杂文档集中在最终阶段验证，风险积聚。

以上问题中 P1-P3 直接影响实施者的生产力，建议在下一轮迭代中优先修复。P4-P9 可按计划依次处理。
