### 3.2 完整性补充意见

**发现 3 — `scalar_vec_ops.cj` 的 `std.math` 外部依赖未在模块依赖表中标注（一般）**
- **位置**：§2 模块间依赖表 `scalar_vec_ops` 行
- **描述**：迭代 14 (发现 4) 已要求标注此依赖。当前方案在 §2"标准库依赖声明"自然段中补充了描述，§8.1 文件清单的说明列也标注了，但模块依赖表本身的 `scalar_vec_ops` 行仍只写到 `→ qualifier`，未显式标注 `→ std.math`。只看依赖表的实施者可能遗漏 `import std.math.{ trunc }`。
- **严重程度**：一般
- **改进建议**：在依赖表 `scalar_vec_ops` 行添加脚注 `【注】浮点 mod 路径依赖 std.math.trunc`，或修改为 `scalar_vec_ops → qualifier + std.math（浮点 mod 路径）`。

**发现 4 — T2 跨类型约束矩阵未集中归档（一般）**
- **位置**：§4.1 跨类型构造函数描述末尾
- **描述**：迭代 8 (发现 4) 已要求添加 T2 约束矩阵。当前方案以自然语言段落（"跨类型构造函数 T2 有效性范围"）描述，并引用 §9.4 类型转换边界表。虽然内容正确，但缺少一个集中的结构化矩阵表，实施者仍需交叉查阅两处。
- **严重程度**：一般
- **改进建议**：在 §4.1"跨类型构造函数 T2 有效性范围"段落末尾新增一个矩阵表，行标题为 T2 类型（Int8/Int16/.../Float32/Float64/Bool），列标题为 T 类型（同组），单元格填写"编译通过 / 编译错误 / 精度损失"等简洁行为描述。此矩阵与 §9.4 类型转换边界表内容互补——矩阵聚焦"哪些 T/T2 组合可编译"，§9.4 聚焦"编译后转换语义"。

**发现 5 — const 真空备选方案的架构影响未在 §11 中评估（一般）**
- **位置**：§4.5 const 真空警示、§11 迁移成本评估
- **描述**：§4.5 在 const 真空警示中提出了两个缓解方案，其中方案 1（将 `equalEpsilon` 移至 struct 体作为 const 函数）涉及架构变更——会引入 `equalExact`（extend, non-const）与 `equalEpsilon`（struct, const）之间的不对称性。当前 §11 迁移成本评估未覆盖此备选方案的实现工作量和架构影响评估。
- **严重程度**：一般
- **改进建议**：在 §11 末尾（或作为 §11.13 的子项）新增"`equalEpsilon` const 化备选方案成本评估"子节，评估将 `equalEpsilon` 从 extend 块迁至 struct 体的具体工作量和影响。或显式声明此方案仅在 D29 回退触发且 const 上下文中必须使用容差比较的极端场景下才考虑，正常场景不进入评估。
