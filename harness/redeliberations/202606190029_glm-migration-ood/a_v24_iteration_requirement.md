## 当前审查结果

### 发现 1（严重）：`fwd.cj` 中 `type` 别名缺少 `public` 可见性修饰符，公共 API 面不可达

**问题描述**：§3.8 提供的 Python 脚本模板（第 484-514 行）和预期文件结构（第 448-469 行）生成的形式为 `type BVec1 = Vec1<Bool, PackedHighp>`。仓颉语言中，顶层声明的默认可见性为 `internal`（包及子包可见）。`type` 别名属于"其他顶层声明"，默认即为 `internal`。`lib.cj` 通过 `public import glm.fwd.*` 重新导出 `fwd.cj` 中的声明，但 `public import` 仅重新导出标为 `public` 的声明——`internal` 声明不会被重新导出。因此下游消费者执行 `import glm.*` 后，无法访问 256 个别名。

**改进建议**：
1. 在脚本模板的生成逻辑中为所有 `type` 别名添加 `public` 修饰符
2. 同步更新 §3.8 中预期的 `fwd.cj` 文件结构示例，将 `type BVec1` 改为 `public type BVec1`
3. 在 §12.2 验收标准中新增一项："别名对外可见性——从独立测试包中通过 `import glm.BVec2` 访问别名并实例化，验证编译通过"
4. 在 §2.1 编码前验证检查清单中新增此项验证

### 发现 2（中等）：`type` 别名可见性相关章节的表述存在自我矛盾

**问题描述**：§7 D1 描述"使用 type 别名在 `glm` 包中提供常用具现化的简写形式"，§2 描述"`public import glm.fwd.*` 将 fwd.cj 中的 256 个别名一并暴露"，但 §3.8 别名定义示例中使用的是 `type`（默认 `internal`）而非 `public type`。设计意图（public API）与代码示例（internal 默认）不一致。

**改进建议**：
- 统一全文中所有 `type` 别名的示例为 `public type` 形式
- 在脚本定制指南（§3.8 末尾）中显式注明"仓颉中 `type` 别名默认可见性为 `internal`，若要对外暴露须添加 `public` 修饰符"

### 发现 3（中等）：Roadmap §3G 范围确认项缺失显式追溯

**问题描述**：§8.2 范围可追溯性对照表仅逐项映射 §3E 的内容，但 Roadmap §3G 中明确包含的依赖闭合性声明、独立可编译性声明、最简可行范围定义未在可追溯性表中体现。

**改进建议**：
- 在 §8.2 表格末尾新增一行或多行，将 §3G 的关键结论（依赖闭合性、可编译性、最小可行范围参数）显式映射到对应设计章节
- 或在 §8.2 表格前增加一段说明，明确标注"§3G 范围确认结论已在 §2 模块依赖分析、§8.3 迁移顺序和 §3.8 首轮实现策略中覆盖"

### 发现 4（一般）：验证计划缺少对生成脚本的 CI 集成验证

**问题描述**：§3.8 推荐使用外部脚本生成 `fwd.cj`，但 §10 验证要求表和 §2.1 编码前验证检查清单中均未包含对生成脚本本身的验证项。脚本生成的 `fwd.cj` 需要纳入版本控制并与脚本保持同步，但文档未定义检测机制。

**改进建议**：
- 在 §2.1 编码前验证检查清单中新增验证项："生成脚本映射表正确性验证——抽查 4 个家族的别名命名是否符合 §3.8 命名约定"
- 在 §10 中新增对生成脚本的验证依赖
- 在 §3.8 脚本生命周期说明中补充警告："手动修改 `fwd.cj` 后须同步更新脚本映射表"
