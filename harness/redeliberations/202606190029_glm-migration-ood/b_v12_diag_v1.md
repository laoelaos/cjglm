## 发现的问题

### P1（一般）：§10 多处残留"const if"独立术语，与 D34 约定不一致

**问题描述**：D34（§7）明确约定"不再使用'const if'作为独立关键字表示"，改用"编译期 `if` 分支"或"`const` 函数中的 `if` 表达式"。v12 P4 修复了 §3.4 和 §3.6 中的残留意象，但 §10 的以下位置**未被覆盖**：

| 位置 | 当前表述 | 应替换为 |
|------|---------|---------|
| §10 标题（第 1267 行） | "`const if` 编译抑制行为综合验证" | "编译期 `if` 分支抑制行为综合验证" |
| §10 首段（第 1269 行） | "`const if` 编译期分支的非选择分支抑制行为" | "编译期 `if` 分支的抑制行为" |
| §10 依赖点清单表（第 1273 行）第三列 | "const if 表达式" | "`const` 函数中的 `if` 表达式" |
| §10 依赖点清单表（第 1273-1277 行）"风险等级"列 | 多处 "若 `const if` 抑制不成立" | "若编译期 `if` 分支抑制不成立" |
| §10 统一验证方法（第 1281 行） | "验证 `const if` 在 `const` 上下文中的非选择分支抑制行为" | "验证 `const` 函数中 `if` 表达式的编译期分支抑制行为" |
| §10 统一验证（第 1303 行） | "若 `const if` 抑制不成立" | "若编译期 `if` 分支抑制不成立" |
| §10 mod 验证子节（第 1315 行） | "mod 函数边界值验证（独立于 const if 抑制）" | "mod 函数边界值验证（独立于编译期 `if` 分支）" |

**严重程度**：一般  
**影响分析**：术语不一致本身不影响设计正确性，但 D34 的修正目的之一是**消除对实施者的误导**——若实施者看到 §10 仍使用"const if"术语，可能在实际编码时尝试编写不存在的 `const if` 关键字造成语法错误。  
**改进建议**：对 §10 全文做一次 grep 确认所有"const if"出现位置，按 D34 约定的表述统一替换。建议在 `replaceAll` 操作后人工逐条审查验证描述的可读性。

### P2（一般）：§12.3 测试目录路径描述存在残余不一致

**问题描述**：v12 P2 修复了测试目录结构 `tests/glm.detail/` → `tests/glm/detail/`，该修复已正确应用于：
- §2 初始目录树形图（第 139-144 行）
- §12.3 开头（第 1717 行）："`tests/glm/detail/` 目录下的测试对应 `package glm.detail`"

但 §12.3 第 1718 行仍使用带点号的路径表示：
```
需要测试 glm.detail 中 internal 可见性类型（如 AlignedHighp/AlignedMediump/AlignedLowp）的测试代码，放置在 `tests/glm.detail/` 包中
```

此处的 `` `tests/glm.detail/` `` 混合了包路径（点号）和目录路径（斜杠）两种表示法，且与修正后的目录树形图（使用 `tests/glm/detail/` 斜杠路径）不一致。

**严重程度**：一般  
**改进建议**：将第 1718 行的 `` `tests/glm.detail/` `` 改为 `` `tests/glm/detail/` `` 目录路径，或改为只使用包名称 `glm.detail`（避免路径表示法混淆）。

### P3（中等）：编码前验证任务分散在多个章节，缺少集中清单

**问题描述**：首轮编码开始前必须完成的验证任务分布在文档的三个独立位置，实施者需交叉阅读才能获得完整清单：

| 来源章节 | 验证任务 | 触发时机 |
|---------|---------|---------|
| §2 项目初始化（第 148 行，步骤 6） | "首次编码前原型验证"——const 表达式链、T(0) const 构造、is 运算符在 const 泛型中的可用性、@OverflowWrapping 标注继承性 | 编码前 |
| §8.4 策略 0（第 1170 行） | "前置验证"——setup.hpp 中 GLM_CONFIG_CLIP_CONTROL 的条件编译逻辑、storage 模板非 SIMD 路径、type_vecN.inl 构造函数宏展开 | 编码前 |
| §10 设计阶段验证要求（第 1257-1265 行） | 依赖①②③——T(0) const 构造、is 在 const 泛型中的可用性、abs()/epsilon() const 兼容性 | 编码前 |
| §10 编译期 `if` 抑制验证（第 1281-1313 行） | testConstIfSuppression<T>() 统一原型测试——Float32/Int32/Bool 实例化 | 编码前 |
| §10 @OverflowWrapping 继承性验证（第 1350-1360 行） | @OverflowWrapping 与 const 函数的语法和语义共存性 | 编码前 |
| §10 mod 浮点实现验证（第 1326-1334 行） | std.math.trunc 可用性、Float32→Float64 提升回降正确性、负操作数符号行为 | 编码前 |

这些验证任务虽有重复（如 T(0) const 构造在 §2 和 §10 均提及），但**缺少一份集中的"编码启动检查清单"**——实施者无法通过查阅单个位置获取所有必须先完成的验证项。

**严重程度**：中等  
**影响分析**：编码阶段实施者若只阅读了 §2 的步骤 6 和 §8.4 的策略 0，可能遗漏 §10 中定义的 `@OverflowWrapping` 继承性验证和 `const if` 抑制验证等关键依赖项。这些依赖项的验证结果直接决定整体架构的可行性——若遗漏验证而直接编码，可能在投入大量工作后发现基础假设不成立。  
**改进建议**：在 §2 的"项目初始化与构建基础设施"末尾新增一个子节"§2.1 编码前验证检查清单"，以表格形式集中列出所有必须在编码前完成的验证任务、对应的验证方法和通过标准，并为每个验证项标注指向 §8.4/§10 对应详细描述的交叉引用。或者，将 §8.4 策略 0 和 §10 的验证要求整合进 §2 的检查清单中，消除分散状态。

### P4（一般）：§10 Float16 T(0) const 构造验证超出首轮范围

**问题描述**：§10 设计阶段验证表依赖①（第 1261 行）要求验证 `Float16` 类型的 `T(0)` const 构造可行性，验证方法中列出 `T = Int32, Float32, Float16, Bool`。但根据 §3.8 定义的 16 个类型家族和 §8.2 范围追溯表，Float16 **不在首轮 16 个分量类型家族范围内**。

首轮覆盖的分量类型：`Bool`、`Int8`/`Int16`/`Int32`/`Int64`、`UInt8`/`UInt16`/`UInt32`/`UInt64`、`Float32`/`Float64`。`Float16` 在 GLM 中主要用于 `hvec`/`half` 类型，属于扩展数值类型，首轮别名列表中也无对应家族。

**严重程度**：一般  
**影响分析**：要求验证一个首轮范围外的类型，额外占用编码前原型验证的测试资源和注意力。`Float16` 的 `T(0)` const 构造如果失败，会被误认为是需要解决的设计风险，实际上它在首轮中根本不会被调用。  
**改进建议**：从验证表依赖①的验证类型列表中移除 `Float16`。保留 `isIec559Of` 取值对照表（§10 第 1242 行）中对 Float16 的注释行，该注释用于澄清 `isIec559Of` 的行为边界，有文档价值。同时，在 §10 验证表末尾（或 §7 设计决策中）补充一条说明："`isIec559Of<T>()` 已在类型层面覆盖所有可能实例化，但仅首轮范围内的类型（§8.2 对照表）需要实际验证。"

### P5（一般）：§4.3 mod 浮点实现将精度假设陈述为确定性断言

**问题描述**：§4.3（第 626 行）对 `Float32` 类型的浮点取模实现策略的表述为：

> "通过先提升至 `Float64` 计算截断再转换回 `Float32` 的方式实现（`std.math.trunc` 仅提供 `Float64` 版本，但此精度提升无损取模语义的正确性）"

段落末尾使用了**确定性断言**"精度提升无损取模语义的正确性"，但 §10 第 1331 行的验证表依赖②将此项标注为**中等级别风险**并列为待验证假设，其验证方法要求覆盖边界值（`Float32.Max`、`Float32.MinPositive`、`0.0`、`±Inf`、`NaN`）。

同一处设计在两个章节中以不同置信度级别描述——§4.3 表述为已确认的事实，§10 表述为待验证的假设。实施者阅读 §4.3 时可能认为此方案已通过验证可以直接采用，而实际它还处于假设状态。

**严重程度**：一般  
**影响分析**：此不一致不会导致编码错误（验证项已在 §10 覆盖），但可能误导实施者跳过或简化验证步骤。  
**改进建议**：在 §4.3 的"精度提升无损取模语义的正确性"断言后添加标注"（此假设的验证要求见 §10 浮点 `mod` 实现正确性验证依赖项②）"，或将确定性断言改为"预期精度提升无损取模语义的正确性（需验证，详见 §10）"。

## 附录：§8.4 前置验证中 GLM_CONFIG_CLIP_CONTROL 值的交叉验证

经查阅 `references/glm-1.0.3/glm/glm/detail/setup.hpp` 第 573-585 行，文档 §7 D6 中断言的"默认值为 `RH_NO = 0x0A`"正确：
