## 问题 1（中等）— `mod` 具名函数使用 `const if` 但未声明为 `const` 函数

**问题描述**：`§4.3` 明确声明 `mod` 函数对浮点/整数双路径"在函数体内通过 `const if (isIec559Of<T>())` 实现编译期分支选择"。但该函数在文档全文中未标注 `const` 修饰符（既不是 extend 成员函数也不是包级独立函数）。Cangjie 中 `const if` 要求其所在函数为 `const` 函数（const 函数体内的表达式必须为 const 表达式，`const if` 的条件必须在编译期求值）。若编码阶段按当前签名直接实现，`const if` 在非 `const` 函数体内将导致编译错误。此外，`add`/`sub`/`mul`/`div` 虽不需要 `const if` 分支，但若保持与 `mod` 一致的 API 风格也需一并评估 const 化必要性。

**所在位置**：§4.3 `mod` 具名函数描述及浮点取模实现策略段落；§10 浮点 `mod` 实现正确性验证表中的依赖项④（`const if` 分支正确性验证）

**严重程度**：中等 — 编码阶段会直接触发的编译错误，需修改函数签名或在实现前补充设计覆盖。

**改进建议**：
1. 将 `vec-op-scalar` 方向的 `mod`（extend 成员函数）和 `scalar-op-vec` 方向的 `mod`（包级独立函数）均声明为 `const` 函数。
2. 同步评估 `add`/`sub`/`mul`/`div` 是否也需要 `const` 化（若不需要，需在文档中显式说明为何只有 `mod` 需要）。
3. 在 §7 新增一条设计决策记录此选择理由。

## 问题 2（一般）— `length()` 静态函数未声明为 `const`，无法用于编译期上下文

**问题描述**：`§3.2` 将 `public static func length(): Int64` 描述为"返回编译期常量"和"计算分量数的静态接口"，但缺少 `const` 修饰符。在 Cangjie 中，`static func` 不带 `const` 即非编译期可调用函数，无法在 `const` 函数体、`const if` 条件或 `const` 变量初始化表达式中使用。若下游泛型代码期望在编译期查询 Vec 分量数（例如 `const if (Vec2.length() < 3)` 等模式），将因 `length()` 非 `const` 而编译失败。设计声称其角色是"替代 C++ `length_t` 参数在泛型运算中的角色"——C++ `length_t` 是编译期常量，仓颉中若不能在编译期使用则无法等效替代。

**所在位置**：§3.2 Vec 结构体系，`public static func length(): Int64` 声明描述

**严重程度**：一般 — 不影响基本功能正确性，但限制了 `length()` 在泛型编译期运算中的使用范围，与设计文档声称的编译期角色自相矛盾。

**改进建议**：将签名改为 `const public static func length(): Int64`。由于函数体仅返回字面量（1/2/3/4），改为 `const` 无需额外实现变更，且零成本启用编译期上下文调用能力。
