## 发现的问题

### P1（严重）— §2 初始目录结构树形图存在事实错误

**问题描述**：`§2 项目初始化与构建基础设施` 中的目录树形图将 `src/glm/` 目录错误地表示为两个同级条目：

```
├── src/
│   ├── glm/
│   │   ├── lib.cj
│   │   └── fwd.cj
│   └── glm/
│       └── detail/
│           ├── setup.cj
```

`src/` 下出现两个名为 `glm/` 的子目录，这在任何文件系统中都是不可能的。正确的表示应为 `src/glm/` 同时包含 `lib.cj`、`fwd.cj` 以及 `detail/` 子目录。此事实错误将直接导致实施者在创建初始项目结构时感到困惑。

**所在位置**：§2 "项目初始化与构建基础设施" 小节，初始目录结构树形图（图中 `src/glm/` 出现两次为同一层级兄弟条目）。

**严重程度**：严重

**改进建议**：将树形图修正为：

```
├── src/
│   └── glm/
│       ├── lib.cj
│       ├── fwd.cj
│       └── detail/
│           ├── setup.cj
│           ├── qualifier.cj
│           ...
```

### P2（中等）— §2 测试目录树形图不完整，遗漏 `tests/glm/`

**问题描述**：同一小节中的测试目录树仅展示 `glm.detail/`：

```
├── tests/
│   └── glm.detail/
│       └── (测试文件，与 src 结构对应)
```

但根据 §8.1 测试文件清单，T13（`test_lib.cj`）和 T14（`test_fwd.cj`）属于 `package glm`，需放置于 `tests/glm/` 目录下。当前树形图遗漏了 `tests/glm/` 目录，实施者将无法确定 `package glm` 测试文件的存放位置。

**所在位置**：§2 "项目初始化与构建基础设施" 小节，初始目录结构树形图中的 `tests/` 子树。

**严重程度**：中等

**改进建议**：在测试目录树中补充 `tests/glm/` 条目，使其与源码目录结构一致：

```
├── tests/
│   ├── glm/
│   │   └── (test_lib.cj, test_fwd.cj 等 package glm 测试)
│   └── glm.detail/
│       └── (test_type_vecN.cj 等 package glm.detail 测试)
```

### P3（中等）— 缺少对上一轮次设计产出的显式追溯

**问题描述**：用户需求中明确要求引用的两个关键文件：

- 初始设计产物：`harness/redeliberations/202606170050_glm-migration-ood/a_v15_design_v1.md`
- 迭代历史记录：`harness/redeliberations/202606170050_glm-migration-ood/iteration_history.md`

在 v9 文档中均未被显式引用。文档的版本摘要仅提到 "v9 基于 v8（a_v8_design_v2.md）迭代"，而附录中的修订说明虽追溯到了 v15，但并未标注初始设计产物和上一轮迭代历史的文件路径。对于首次接触本文档的新实施者，无法从文档中找到其设计演进的上游来源。

**所在位置**：文档标题下方的 "版本变更摘要" 及 §1 概述全文。

**严重程度**：中等

**改进建议**：在 §1 概述末尾或版本变更摘要中显式引用：

- 初始设计文件路径：`harness/redeliberations/202606170050_glm-migration-ood/a_v15_design_v1.md`
- 历史迭代记录：`harness/redeliberations/202606170050_glm-migration-ood/iteration_history.md`
- 当前轮次迭代历史：`harness/redeliberations/202606190029_glm-migration-ood/iteration_history.md`

确保新接手者能一步追溯到设计源头。

### P4（一般）— `l` 与 `1` 的视觉混淆风险（排版问题）

**问题描述**：文档在多处使用字形极易混淆的字符组合，如 `Vec1`（数字1）与 `Vec` + 字母 `l` 在常规等宽字体中难以区分。此问题虽不影响内容正确性，但在长达 1794 行的文档中反复出现（数百次 `Vec1`、`Vec2` 等引用），会增加阅读负担和人工审查时的误判风险。建议在文档头部或关键处给出视觉提示。

**所在位置**：全文，尤以 §4.1 构造函数清单、§3.8 别名表、§8.1 文件清单等密集列出数字分量的章节为甚。

**严重程度**：一般

**改进建议**：在 §1 或 §3.2 首次出现 `Vec1` 处添加注释说明 "注意：`Vec1`（中间为数字1，非字母l）"，或在关键代码块中混用反色背景以增强数字辨识度。
