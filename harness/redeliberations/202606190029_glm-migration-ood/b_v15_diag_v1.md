## 问题清单

### P1（中等）— `equalEpsilon` 回退函数在行为契约中缺失正式定义

**位置**：§3.5 备选路径（第345行）、§7 D29 回退方案（第1018行）

**问题描述**：D29 回退路径中明确提及 `equalEpsilon` 作为浮点容差比较的替代函数：

- §3.5："浮点容差比较通过独立的具名函数（如 `equalEpsilon`）提供给需要容差比较的场景"
- §7 D29：回退路径描述为"浮点容差比较通过独立的非 `const` 具名函数（如 `equalEpsilon`）提供"

但 **`equalEpsilon` 未在 §4 关键行为契约中正式定义**——无函数签名、无定义位置、无行为约定。§4.5 仅定义了 `equalExact`（精确比较）和 `==`（容差比较）。如果 D29 回退被触发，实施者缺少 `equalEpsilon` 的定义规范，无法直接编码。

**注**：§9.3 在第 7 轮中已将 `equalEpsilon` 替换为 `equalExact`（见迭代第7轮P7），但 §3.5 和 §7 D29 两处残留未同步修复。

**严重程度**：中等

**改进建议**：选择以下方案之一：
A. （推荐）在 §4.5 中新增 `equalEpsilon(other: VecN<T,Q>): Bool` 的完整定义（签名、实现策略、行为约定），使其成为设计中的一等公民函数
B. 将 §3.5 和 §7 D29 中的 `equalEpsilon` 改为引用现有函数 `equalExact`，并评估容差比较替代路径的语义等价性
C. 若设计意图是回退时直接依赖内部容差比较路径而非独立函数，则删除两处对 `equalEpsilon` 的引用

### P2（中等）— Bool→数值转换方案 B 的运行时 `if` 分支抑制假设未经论证

**位置**：§9.4 Bool→数值转换行（第1241行）

**问题描述**：§9.4 为跨类型构造中的 Bool→数值转换推荐"方案 B（运行时兜底）"：

```
在 `init` 体内通过 `if (v.x is Bool)` 运行时检测分支——`true → T(1)`，`false → T(0)`
```

此方案假设：在非 `const` 的 `init` 函数中，运行时 `if` 的 else 分支（`T(v.x)`，当 `T2=Bool` 时不合法）不会被编译检查。这是错误的——**仅 `const` 函数中的 `if` 表达式具有编译期分支抑制行为**（见 §7 D34）。非 `const` 函数中，编译器会检查所有分支的语法和类型正确性。

因此，当 `T2=Bool` 且 `T` 为数值类型时，若 `Int32(Bool)` 在仓颉中不合法（当前设计未确认其合法性），则整个构造函数在实例化时会产生编译错误。

**严重程度**：中等

**改进建议**：
1. 将此假设明确标注为"待验证"状态，加入 §10 验证计划新增依赖项
2. 若验证不通过，定义明确回退方案：
   - 编译期方案：利用 `const init` 中的编译期 `if` 分支（但要求构造函数为 `const`——与当前非 `const` 的跨类型构造函数不兼容）
   - 定义分离的具名转换函数 `fromBool<T, Q>(v: VecN<Bool, Q>): VecN<T, Q>`
3. 在 §9.4 移除"方案 B 为首选推荐路径"的确定性语句，改为"待验证方案"或"假设性方案"

### P3（一般）— 附录历史修订版本序列混乱，新实施者难以追溯当前版本状态

**位置**：附录 历史修订说明（第1797-2020行）

**问题描述**：附录混用两套版本编号体系（"新审议轮次"v2~v13 与"旧审议轮次"v7~v15），三段之间虽已用分隔线区分，但：
1. 新审议轮次中 v7~v13 正序排列在 v6~v2 倒序之后，版本号不连续且排列方向不一致
2. "新审议轮次（早期，v6~v2 倒序）"与"新审议轮次（后续，v7~v13 正序）"之间存在版本跳跃（v2→v7）
3. 当前 v15 版本的修订说明仅位于附录末，但标题显示为"v15 — 审查迭代修订版"。一个不熟悉迭代过程的读者无法通过附录快速理解当前版本（v15）相对于 v14 的增量变更

**严重程度**：一般

**改进建议**：
1. 在附录起始处增加**版本关系图**或**简化版本时间线**，帮助新读者一目了然
2. 将当前版本（v15）的修订说明从附录末尾移至章节顶部（紧跟版本变更摘要之后），使其成为最显著的变更说明
3. （可选）将旧审议轮次历史移至独立文件，仅在附录中保留摘要和交叉引用链接，减少单文件信息负载

### P4（一般）— §9.4 中 `v.x is Bool` 运行时类型检测对 Vec2~Vec4 的适用性未讨论

**位置**：§9.4 Bool→数值转换行（第1241行）

**问题描述**：§9.4 的方案 B 以 Vec1 为例（`v.x is Bool`），但跨类型构造函数适用于所有 Vec1~Vec4。对于 Vec4，需要四个分量分别执行运行时类型检测（`v.x is Bool`、`v.y is Bool`、`v.z is Bool`、`v.w is Bool`），但文档仅以单分量的 Vec1 示例描述方案。实施方案者可能需要额外推断 Vec2~Vec4 的模式，增加了编码中的不确定性和出错风险。

**严重程度**：一般

**改进建议**：在 §9.4 明确说明跨类型构造中 Vec2~Vec4 的 Bool 检测模式，例如：
- Vec2：`if (v.x is Bool) { this.x = v.x ? T(1) : T(0) } else { this.x = T(v.x) }`（重复于 y 分量）
- 或以 `init` 体内循环、委托给辅助函数的方式实现（消除每个 VecN 的重复代码）
