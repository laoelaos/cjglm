## 发现 1（中等）— const 真空缓解方案存在逻辑矛盾

**所在位置**：§4.5（第 881-884 行）、§11.14（第 2174-2204 行）、§7 D29（第 1257-1271 行）

**问题描述**：文档在 §4.5 提出 D29 回退触发后 `equalEpsilon` 在 const 上下文中不可用的"const 真空"问题，并在 §11.14 给出缓解方案①——"将 `equalEpsilon` 定义从 `extend` 块移至 Vec 结构体体内，利用 `const init` 声明为 `const` 实例成员函数"。但此方案与 D29 回退触发的前提条件存在根本性矛盾。

**论证**：
1. D29 回退的触发条件是：`const` 表达式链 `abs(a - b) <= epsilonOf<T>()` 在 const 上下文中**编译失败**。
2. `equalEpsilon` 的函数体与此表达式链完全一致（§4.5 第 873 行：`abs(this.x - other.x) <= epsilonOf<T>()`）。
3. 将 `equalEpsilon` 移至 struct 体并声明为 `const` 后，其函数体仍需在 const 上下文中通过编译——恰好是 D29 已验证失败的条件。
4. 因此方案①**无法解决** const 真空——它仅改变了函数定义位置和 `const` 修饰符，不改变函数体内容，在 D29 已回退的前提下方案①必然同样编译失败。

**严重程度**：中等。此矛盾误导实施者以为存在"不解自明"的缓解路径，实际在 D29 回退场景下该路径不可行。§11.14 的成本评估（约 0.5 人天）基于不可行的前提，需要重新估计。

**改进建议**：
- 将 §4.5 const 真空警示段落中方案①的表述从缓解方案修正为"理论路径"并标注**前提条件**：仅当 D29 回退的根因**不是**表达式链中的 `abs()`/`epsilonOf<T>()` const 不可用（而是 `const operator ==` 本身的重载规则或语法约束问题）时，方案①才可能可行。
- 将 §11.14 方案①的可行性评级从"推荐优先采用"降为"需验证前提条件后判断"，成本评估标注"前提条件满足时约 0.5 人天"。
- 补充第三条缓解方案：非 const 上下文中使用 `equalEpsilon`（运行时委托），const 上下文中使用精确比较（`==`）。原建议"在 const 上下文中使用运行时委托（通过非 const 函数间接包装后调用）"不可行——const 函数不能调用非 const 函数，此处修正为 const 上下文中直接使用精确比较。

## 发现 3（一般）— `protected` 回退方案可能无法跨模块工作

**所在位置**：§12.3（第 2297 行）

**问题描述**：文档将 `protected` 作为 `internal` 测试访问回退的第 1 优先级方案，声称 "`protected` 在当前仓颉版本中仅允许同包和子包访问"。但仓颉语言规范中 `protected` 的可见域定义为**当前模块**。在 cjpm 项目中，`tests/` 目录的测试文件与 `src/` 目录的生产代码是否属于同一模块取决于 cjpm 的编译单元划分——通常是分离的测试目标。

**论证**：若测试文件被编译为独立的测试二进制（`cjpm test` 的典型行为），则测试代码和生产代码分属不同模块，`protected` 可见性**不足以**跨模块访问。文档 §12.3 本身也标注了"此方案的可行性依赖仓颉编译器对 `protected` 可见性跨包访问的支持程度"——与"当前模块"的规范定义存在张力。

**严重程度**：一般。文档已标注不确定性，但将最不确定的方案放在回退层级首位，可能导致实施者优先选择此方案尝试后才发现不可行，浪费时间。

**改进建议**：
- 在 §12.3 回退方案层级中，将第 1 优先级的 `protected` 方案标注明确的前提条件："需在目标编译器上验证 `tests/glm/detail/` 测试包与 `glm.detail` 生产包是否属于同一模块。若 `cjpm test` 将测试文件编译为独立模块，`protected` 不可行。"
- 建议将回退层级顺序调整为：首选方案（同包测试直接访问）→ 验证 `protected` 可行性 → helper 文件方案 → `public` 提升方案，而非将 `protected` 直接标为 1st。

## 发现 5（一般）— equalEpsilon 在整数 Vec 上的编译通过性断言可能不成立

**所在位置**：§4.5（第 871-873 行）、§10 依赖②（第 1654 行）

**问题描述**：文档声称 `equalEpsilon` 在整数 `T`（如 `Int32`）上编译通过，理由是 "Cangjie 整数类型支持 `-` 和 `abs`"。但 `equalEpsilon` 的函数体包含 `epsilonOf<T>()` 的调用，而 `epsilonOf<T>()` 的签名为 `const func epsilonOf<T>(): T where T <: Number<T>`（§3.5，第 362 行）。

**评估**：仓颉标准库中 `Int32` 类型实现了 `Number<Int32>` 接口（`Int8`~`Int64`、`UInt8`~`UInt64` 等数值类型均实现 `Number<T>`），因此 `Int32` 满足 `Number<Int32>` 约束。风险等级低于 v2 报告中的评估——约束满足是确定的，但 `epsilonOf<T>()` 的 `Number<T>` 约束与 D5 的宽松约束策略之间的兼容性仍然是需要验证的项目（已纳入 §3.5 兼容性策略路径 A/B）。

**严重程度**：一般。风险已在 §3.5 预覆盖，`Int32 <: Number<Int32>` 成立，但未在 §4.5 和 §10 的验证项中形成显式追踪链。

**改进建议**：
- 在 §4.5 `equalEpsilon` 代码示例的注释中，显式指向 §3.5 兼容性策略的路径 A/路径 B。
- 在 §10 依赖②的验证方法中，补充说明验证步骤与 §3.5 双路径策略的联动关系——先尝试路径 A（验证 `Int32 <: Number<Int32>` 是否满足），若失败则走路径 B。

## 发现 7（提示）— 256 个别名的编译时间影响未在架构评估中考虑

**所在位置**：§3.8（第 449-524 行）、§11 迁移成本评估

**问题描述**：文档在 §3.8 提及 256 个别名可能导致总编译时间超过阈值（如 > 30 分钟）是别名缩编的触发条件之一，但未在架构评估中预评估别名对模块整体编译时间的影响。

**分析**：`type BVec2 = Vec2<Bool, PackedHighp>` 是类型别名而非实例化——编译器处理别名时代为类型替换而非完整模板实例化，因此编译时间影响低。但在别名用于泛型方法调用时（如 `let v: BVec2; v + v`），编译器需对 `Vec2<Bool, PackedHighp>` 进行完整的类型检查和运算符查找，此时 256 个别名的间接影响不可忽略。

**严重程度**：提示级。

**改进建议**：
- 在 §3.8 或 §11 中补充一段编译时间考量说明，区分别名定义阶段（编译时别名展开，影响低）和别名使用阶段（导致 256 种 Vec 具现化组合，在泛型函数实例化时可能影响编译时间），帮助实施者评估缩编触发条件的实际适用性。

## 发现 9（中等）— 接口定义充分性评估（新增，回应质询 Q1）

**所在位置**：§2 公共 API 面设计（第 64-84 行）、§3.8 向量类型别名（第 445-607 行）、§4 关键行为契约（全部）、§12.2 验收标准（第 2246-2275 行）

**问题描述**：质询指出的"接口定义充分性"维度在 v2 中未系统性覆盖。经独立评估，发现以下情况：

### 9.1 公共 API 完备性
- **覆盖充分**：`lib.cj` 的 `public import` 列表覆盖所有公开类型（Vec1~Vec4、Qualifier 实现类型、Defaultp）和公开函数（`add`/`sub`/`mul`/`div`/`mod`、`fromBoolVec`/`fromBoolVecQ2`）。
- **扩展成员函数**（`increment`/`decrement`/`bitwiseNot`/`equalExact`/`equalEpsilon`/`logicalAnd`/`logicalOr`）作为 Vec 的扩展成员函数自动可见，无需额外导出。此设计合理——这些函数随 Vec 类型绑定，通过 `import glm.Vec2` 即可使用。
- **`componentAt`**：作为 Vec struct 体内的 `const` 实例成员函数，随 Vec 类型导入自动可见。

### 9.2 别名可用性（消费者视角）
- 256 个别名按 16 家族 × 4 分量数 × 4 精度变体的组织方式清晰，文件结构示例完整
- 命名约定强制规则（PascalCase）已在 §3.8 明确给出格式模板和映射表，消费者可通过这些规则自行推导任意类型的别名形式
- **潜在认知缺口**：消费者首次接触可能被 256 个别名的数量吓到——文档未提供"我应该用哪个别名"的快速选择指南。§3.8 的"必备/常用/可选"分级目前标记为"仅用于编码顺序规划，非交付标准"，但对消费者的实际用途有限。建议新增 2-3 行的快速选择示例（如"二维图形用 BVec2，三维图形用 Vec3"等场景指引），或在 fwd.cj 文件头注释中提供简短的使用示例。

### 9.3 泛型签名约束完整性
- 所有公开函数签名均完整标注了泛型参数约束。如 `add<T, Q>(s: T, v: Vec2<T, Q>): Vec2<T, Q> where Q <: Qualifier`——`where` 子句明确。
- **`fromBoolVec`/`fromBoolVecQ2` 的 T 参数无约束**：函数签名 `public func fromBoolVec<T, Q>(v: Vec1<Bool, Q>): Vec1<T, Q> where Q <: Qualifier` 中 `T` 无约束。这是符合 D5 延迟检查策略的有意设计——但在接口层面，消费者仅看签名无法判断 `T` 的合法取值范围。建议在 §4.8 或函数注释中补充说明："`T` 应为数值类型（`Int8`~`Int64`、`UInt8`~`UInt64`、`Float32`、`Float64`），非数值类型实例化将产生编译错误。"

### 9.4 与 C++ GLM API 的迁移对标
- 命名差异（PascalCase vs 小写）已在 §11.7 中提供完整的映射表和迁移策略

**缺口 A — 浮点 `mod` 边界行为与 §9.3 的交叉引用不完整**：§10 的浮点 `mod` 实现正确性验证（依赖②）要求验证 `±Inf`/`NaN` 边界值，但 §9 的 NaN/Infinity 传播约定中未显式提及 `mod` 运算（`mod` 实现使用 `trunc` 恒等式，而非原生 `%`，其 NaN/Infinity 传播行为需要单独约定）。文档在 §4.3 提到了 `mod` 的浮点实现策略，但未在 §9.3 中明确提及 `mod` 的 NaN/Infinity 边界行为。建议在 §9.3 或 §4.3 中补充一行说明："`mod` 使用 `x - y * trunc(x / y)` 恒等式，NaN/Infinity 的传播行为由 `trunc` 和该恒等式确定，与 IEEE 754 `fmod` 行为一致。"

**缺口 B — `@OverflowWrapping` 在无符号整数上的截断语义与 C++ 的行为差异未在异常场景中显式声明**：§5.1 声称"标注 `@OverflowWrapping` 将溢出行为从'抛异常'改为'wrapping'，与 C++ UB 行为在数值结果上等价"。但 C++ 的无符号整数溢出是定义良好的（wrapping），而有符号整数溢出是 UB。仓颉的 `@OverflowWrapping` 同时覆盖有符号和无符号。此差异在数值结果上等价（均 wrapping），但 C++ 开发者可能对有符号整数的 wrapping 感到意外——他们可能依赖于有符号溢出的 UB（某些优化假设）。建议在 §5.1 或 §11 新增一条备注，说明与此相关的迁移注意事项。

**严重程度**：一般。现有异常场景覆盖已相当完善（8 个场景组，20+ 条约定），缺口 A 和 B 为边界精度补充，非阻塞性问题。

**改进建议**：
- 在 §9.3 NaN/Infinity 传播约定中新增 `mod` 运算的说明行。
- 在 §5.1 或 §11 补充有符号整数 `@OverflowWrapping` 与 C++ UB 的迁移注意事项。

**严重程度**：一般。整体编码可指导性良好——签署完整、迁移顺序清晰、验证闭环完整。阶段 5 的粗粒度和部分运算符签名的自然语言描述是提高编码直达性的两个小缺口。
