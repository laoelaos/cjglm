## 当前审查结果

### 发现 1（中等）— `@OverflowWrapping` 标注策略未覆盖 scalar-op-vec 方向包级独立函数
- **所在位置**：§4.3 具名函数定义（第 631-646 行）、§4.6 `@OverflowWrapping` 标注分布表（第 736-748 行）
- **问题描述**：§4.6 的 `@OverflowWrapping` 标注分布表完全未提及 scalar-op-vec 方向的包级独立函数（`add(s,v)`、`sub(s,v)`、`mul(s,v)`、`div(s,v)`、`mod(s,v)`，共 5 运算 × 4 Vec = 20 个函数）。这 20 个函数使用 `T` 类型的原生算术运算符，默认采用 `@OverflowThrowing` 策略，而 vec-op-scalar 方向对应函数通过委托给已标注 `@OverflowWrapping` 的二元运算符采用 wrapping 语义，两者溢出行为不一致。
- **影响**：公共 API 面的行为不一致，下游消费者根据操作数顺序选择 `v.add(s)` 或 `add(s, v)` 预期行为一致，但实际溢出策略不同，可能导致迁移代码中的隐蔽 bug。
- **改进建议**：方案 A（推荐）：在 §4.6 标注分布表中新增条目，明确 scalar-op-vec 方向包级独立函数的 `@OverflowWrapping` 标注策略——在 `scalar_vec_ops.cj` 中的 20 个函数定义上直接标注 `@OverflowWrapping`。方案 B：若标注位置受限，在 §4.3 中明确标注溢出行为约定，在 §4.6 分布表中新增条目说明标注方式。

### 发现 2（一般）— §8.2 范围可追溯性表缺少 `scalar_vec_ops.cj` 的显式映射行
- **所在位置**：§8.2 范围可追溯性对照表（第 1195-1211 行）
- **问题描述**：`scalar_vec_ops.cj` 在 §8.1 文件清单中列为序号 8a，但在 §8.2 范围可追溯性表中没有对应的显式行。`lib.cj`（同为新增文件）在 §8.2 末尾有一个 "—" 行并附偏离说明，而 `scalar_vec_ops.cj` 完全未出现在追溯范围中。
- **影响**：实施者无法从 §8.2 直接确认 scalar_vec_ops.cj 的 roadmap 映射关系，需交叉查阅 §8.1 和 §8.2。
- **改进建议**：在 §8.2 末尾新增一行，将 `scalar_vec_ops.cj` 的 roadmap 映射关系显式记录。

- **改进建议**：可选增强：在 §4.1 末尾补充一个"重载解析期望"注释，说明当参数类型存在多重匹配时的编译器选择预期（如"具体类型优先于泛型"），以降低实施者的认知负担。

### 发现 4（提示）— `equalEpsilon` 中"内联固定容差"回退策略的语义漂移未显式标注
- **所在位置**：§3.5 `epsilonOf<T>()` 兼容性策略（第 328-329 行），§4.5 `equalEpsilon` 定义（第 701-714 行），§10 依赖④（第 1413 行）
- **问题描述**：§3.5 的兼容性回退策略描述为"将 epsilon 逻辑内联到 `ComputeEqual.callConst` 的浮点分支中"，但 §10 依赖④ 的描述将其理解为"内联到 `equalEpsilon`"。存在两层歧义：内联位置歧义（callConst 还是 equalEpsilon？）和 `epsilonOf<T>()` 函数生命周期歧义（保留并修改实现还是完全移除？）。
- **影响**：实施者在编码阶段遇到回退时需自行解读歧义，可能选择与设计意图不同的实现方式。
- **改进建议**：在 §3.5 兼容性策略中明确 `epsilonOf<T>()` 回退后的生命周期。若函数保留：修改其实现为 `const func epsilonOf<T>(): T { T(1e-6) }`（移除 `Number<T>` 约束），则 `equalEpsilon` 无需修改。若函数移除：在 §3.5 和 §4.5 中均标注联动修改要求，声明 `equalEpsilon` 须同步内联。
