# 质量审查报告 — v23 OOD 设计文档

**审查对象**：GLM 1.0.3 仓颉迁移首轮 OOD 设计方案（v23）
**审查维度**：需求响应充分度、事实正确性、深度与完整性、实施就绪度
**审查轮次**：第 1 轮（首轮审查）

---

## 发现 1（严重）：`fwd.cj` 中 `type` 别名缺少 `public` 可见性修饰符，公共 API 面不可达

**问题描述**：
§3.8 提供的 Python 脚本模板（第 484-514 行）和预期文件结构（第 448-469 行）生成的形式为：
```
type BVec1 = Vec1<Bool, PackedHighp>
```

仓颉语言中，顶层声明的默认可见性为 `internal`（包及子包可见，见 `package/README.md` §5.2）。`type` 别名属于"其他顶层声明"，默认即为 `internal`。

`lib.cj` 通过 `public import glm.fwd.*` 重新导出 `fwd.cj` 中的声明。但 `public import` 仅重新导出**标为 `public` 的声明**——`internal` 声明不会被重新导出。因此下游消费者执行 `import glm.*` 后，无法访问 `BVec2`、`IVec3` 等 256 个别名。

**影响范围**：
- 256 个别名整体对外不可见，整个别名体系失效
- `lib.cj` 的 `public import glm.fwd.*` 成为空操作
- 下游消费者被迫直接引用 `Vec2<Int32, PackedHighp>` 等完整泛型签名，与设计目标（通过别名隐藏 Q 参数细节，§3.8）背道而驰
- §12.2 验收标准中"别名覆盖"项仅验证编译无重复定义，未验证对外可见性，存在验证盲区

**改进建议**：
1. 在脚本模板的生成逻辑中为所有 `type` 别名添加 `public` 修饰符：
   ```python
   lines.append(f'public type {name} = {VEC_TYPES[dim]}<{typ}, {qual}>')
   ```
2. 同步更新 §3.8 中预期的 `fwd.cj` 文件结构示例，将 `type BVec1` 改为 `public type BVec1`
3. 在 §12.2 验收标准中新增一项："别名对外可见性——从独立测试包中通过 `import glm.BVec2` 访问别名并实例化，验证编译通过"
4. 在 §2.1 编码前验证检查清单中新增此项验证

## 发现 2（中等）：`type` 别名可见性相关章节的表述存在自我矛盾

**问题描述**：
§7 D1（设计决策）中关于 `glm` 包别名的描述：
> - 使用 type 别名在 `glm` 包中提供常用具现化的简写形式

§2 公共 API 面设计：
> `public import glm.fwd.*` 将 fwd.cj 中的 256 个别名一并暴露，使用者无需单独导入 fwd.cj。

§3.8 别名定义示例：
> `type BVec2 = Vec2<Bool, PackedHighp>`

§2 的表述明确声称别名可通过 `public import` 暴露给下游，但 §3.8 的代码示例中使用的是 `type`（默认 `internal`）而非 `public type`。此处存在设计意图（public API）与代码示例（internal 默认）的不一致。编码阶段若直接按脚本模板实施，将产生与设计意图不符的结果。

**改进建议**：
- 统一全文中所有 `type` 别名的示例为 `public type` 形式
- 在脚本定制指南（§3.8 末尾）中显式注明 "仓颉中 `type` 别名默认可见性为 `internal`，若要对外暴露须添加 `public` 修饰符"

## 发现 3（中等）：Roadmap §3G 范围确认项缺失显式追溯

**问题描述**：
§8.2 范围可追溯性对照表标题声明覆盖 "roadmap §3E/§3G 定义的首轮范围"，但表格仅逐项映射 §3E 的 11 个文件 + 3 个 shim + 2 个新增文件。Roadmap §3G 中明确包含的以下内容未在可追溯性表中体现：

- **依赖闭合性声明**（§3G 第 548-556 行）：要求 11 个核心文件 + 3 个 shim 文件的依赖链不超出首轮范围
- **独立可编译性声明**：应按子范围逐层编译验证
- **最简可行范围定义**（§3G 第 462 行）：16 家族 × 3 分量数 × 1 精度变体 = 48 个别名

虽然设计文档的 §2、§8.3、§3.8 等内容**隐式覆盖**了这些要求，但 §8.2 作为正式的追溯性断言，缺少对 §3G 内容的显式映射，读者无法快速验证 §3G 的所有要求均已响应。

**改进建议**：
- 在 §8.2 表格末尾新增一行或多行，将 §3G 的关键结论（依赖闭合性、可编译性、最小可行范围参数）显式映射到对应设计章节
- 或在 §8.2 表格前增加一段说明，明确标注"§3G 范围确认结论已在 §2 模块依赖分析、§8.3 迁移顺序和 §3.8 首轮实现策略中覆盖"

## 发现 4（一般）：验证计划缺少对生成脚本的 CI 集成验证

**问题描述**：
§3.8 推荐使用外部脚本生成 `fwd.cj`，并提供脚本模板。§11.11 评估了脚本开发工作量。但：

1. **§10 验证要求表**和 **§2.1 编码前验证检查清单**中均未包含对生成脚本本身的验证项——脚本映射表正确性、生成的 `fwd.cj` 与模板一致性等
2. 脚本生成的 `fwd.cj` 需要纳入版本控制并与脚本保持同步，但文档未定义"脚本输出与预期输出不一致"时的检测机制
3. 若某次编码阶段手动修改了 `fwd.cj` 但未同步更新脚本，后续重新生成会覆盖手动修改，文档缺少对此风险的约定

**改进建议**：
- 在 §2.1 编码前验证检查清单中新增验证项："生成脚本映射表正确性验证——抽查 4 个家族的别名命名是否符合 §3.8 命名约定"
- 在 §10 中新增对生成脚本的验证依赖："`scripts/gen_fwd_aliases.py` 的输入-输出映射正确性验证"
- 在 §3.8 脚本生命周期说明中补充警告："手动修改 `fwd.cj` 后须同步更新脚本映射表，否则重新生成会覆盖手动改动"

## 总结

v23 设计文档经过 22 轮迭代审议，在技术细节层面已非常成熟。本次审查发现的 4 个问题主要集中在**公共 API 可达性**和**可追溯性完整性**维度——这些维度在之前的技术性内部审议中可能未被充分覆盖。

| 发现 | 严重程度 | 板块 | 性质 |
|------|---------|------|------|
| 发现 1 | 严重 | §3.8 脚本模板 + §2 lib.cj | `public` 可见性遗漏，将导致整个别名体系对外不可用 |
| 发现 2 | 中等 | §2 vs §3.8 vs §7 | 设计意图与代码示例的可见性不一致 |
| 发现 3 | 中等 | §8.2 | Roadmap §3G 追溯缺失 |
| 发现 4 | 一般 | §2.1 + §10 | 生成脚本的验证和治理缺失 |

**总体评估**：设计在架构层面完备、行为契约层面精确、迁移策略层面可执行。四个发现项均可通过文档修订快速修复，不影响整体设计框架的合理性。修复后即可作为编码实施的权威依据。
