## 逐维度审查

### 1. 证据充分性

**[问题-严重]** "无遗漏项"断言缺乏可独立验证的证据支撑

"需求响应充分度"段落声明"经独立验证 §8.2 范围可追溯性对照表"确认全部 14 项已显式映射、无遗漏项。但报告未呈现验证过程、映射对照结果或方法论——读者（包括设计文档作者）无法从报告内容独立核实这一核心判断。在 25 轮迭代中多次出现追溯表与实际设计的偏差（如第 8 轮 countof 偏离遗漏、第 21 轮 scalar_vec_ops.cj 缺少映射行等），追溯表自身的历史错误使得"经独立验证"这一声称更需透明证据，而非仅陈述结论。

**[建议]** 在报告中提供以下之一的证据形式：① 以附录形式呈现 §8.2 与 roadmap §3E 的逐项映射对照表；② 说明验证方法（如"逐行读取 §8.2 追溯表的 14 行，逐项与 roadmap §3E 文件清单交叉比对"）及发现的任何偏离；③ 标注验证时发现的§8.2 中存在的任何不精确之处，而非仅断言"无遗漏项"。

---

**[问题-一般]** "事实/逻辑正确性"结论的验证基础不足

报告认定"事实层面无显著错误"，但声明此为"静态分析，实际编译器验证需编码阶段完成"。此自我限定的验证基础与"无显著错误"的确定性结论之间存在张力——若干关键设计依赖（const 表达式链、`@OverflowWrapping` 继承性等）尚为待验证假设，报告的 F5 本身也列出 12+ 个编译器行为假设。将待验证的设计状态与"无显著错误"并列表述可能淡化实际风险。

**[建议]** 将"事实/逻辑正确性"结论的措辞从"无显著错误"改为"设计层面的自洽性经静态审查确认，编译器行为依赖的验证待 §2.1 清单执行"，明确区分静态可验证与动态待验证的内容。

### 2. 逻辑完整性

**[问题-严重]** F4 严重程度标注（[提示]）与自身描述的自洽性断裂

F4 描述"若重载解析与预期不符，修改范围涉及 Vec4 构造函数体系（约 30 个构造函数签名），重构成本较高"，但最终标注为 [提示]，理由是"风险已标注，回退方案可后续补充"。此逻辑存在以下问题：

1. **影响与严重程度不匹配**：30 个构造函数的修改范围与"重构成本较高"的定性，落在多数审查框架中对应 [一般] 或 [中等] 而非 [提示]。
2. **理由与结论不匹配**："风险已标注"是文档状态描述而非风险缓释——已标注风险不降低该风险的潜在影响。"回退方案可后续补充"恰是确认其可行性定义不足（而非充分）的标志。
3. **与 F3 的横向不一致**：F3 因"功能真空"定性从 [提示] 升级为 [一般]，但 F4 的"重构成本较高"×30 构造函数却维持 [提示]，两处校准尺度不统一。

**[建议]** 将 F4 严重程度升级为 [一般]，或提供更充分的论证说明为何 30 个构造函数的高成本重构对应 [提示] 级别（如概率极低、或回退方案极简单）。

---

**[问题-一般]** "非阻塞优化项"归类与 F4 自述矛盾

报告总结处将 F1/F3/F4 统称为"非阻塞优化项"，但 F4 自身描述提及"重构成本较高"。若某风险潜在涉及 30 个构造函数的大规模重构，将其归类为"非阻塞优化"需说明判定标准——是假定该风险触发概率极低？还是假定回退方案极其简单？报告未提供此分析，造成首尾结论不一致。

**[建议]** 在 F4 或总结处补充说明 F4 被归类为"非阻塞"的具体判断依据（如概率/影响矩阵分析），或根据实际评估调整归类。

### 3. 覆盖完备性

**[问题-严重]** 参考实现使用策略的审查遗漏

需求文档明确提及"已有参考实现位于 references/glm-1.0.3/"，且迭代历史中迭代第 7 轮（问题 7："对参考实现的使用策略未作说明"）和第 10 轮（问题 4："对 references/glm-1.0.3/ 的引用基于假设其存在，但未验证关键路径的实现准确性"）已将参考实现的使用策略定位为反复未解决的问题。但报告在对设计文档的审查中未评估参考实现使用策略的充分性——设计文档是否已提供可操作的参考实现对照验证方法。此维度属"内部审议未充分覆盖的维度"（技术细节已由内部审议覆盖，但参考实现策略属于整体落地指导的范畴）。

**[建议]** 审查设计文档中是否已补充以下内容：① 参考实现的使用策略说明；② 关键设计决策与参考实现的对照验证方法；③ §8.4 或相应章节是否已包含参考实现文件路径和行号引用（迭代第24轮问题4提出的要求）。

---

**[问题-严重]** "接口定义是否足以支持下游消费者"维度的审查缺失

任务明确要求"从实际落地视角评估：接口定义是否足以支持下游消费者"。报告仅通过 F1 触及 `fromBoolVec` 的 API 风格问题，但未系统评估以下消费者视角的关键维度：

1. **`lib.cj` 公共 API 面设计**：通过 `public import` 暴露的具名函数集（add/sub/mul/div/mod）是否有合理的使用说明？命名空间中 `equalEpsilon`、`bitwiseNot` 等函数对外可见性是否正确？
2. **const/non-const API 分裂**：迭代第 24 轮已标记"const/extend 不对称导致 11 个日常操作在 const 上下文中不可用"，报告是否验证设计文档已新增"const 上下文 API 可用性速查表"来告知消费者这一限制？
3. **256 个别名的消费者体验**：大写驼峰命名（`BVec2`、`IVec3`）与 C++ GLM 小写习惯的差异是否已在文档中有足够的消费者告知（迭代第 5 轮问题 6）？
4. **类型转换的行为可预见性**：消费者能否从文档中准确判断 `Vec2<Float32, PackedHighp>(Vec2<Int32, AlignedLowp>(...))` 的编译通过性和运行时语义？

**[建议]** 补充对上述消费者视角维度的系统性审查，至少对每个维度给出"已充分覆盖/部分覆盖/未覆盖"的判定及证据引用。

---

**[问题-一般]** 异常场景/边界条件覆盖审查仅停留于引用而非独立验证

任务要求评估"异常场景和边界条件是否已考虑"。报告在"深度与完整性"表中引用了"§9 覆盖零向量/除零/NaN/Infinity/精度损失等边界条件"，但未独立验证 §9 的实际覆盖充分性——例如：迭代第 9 轮问题 3 指出 equalExact 未覆盖 NaN 返回 false 场景；迭代第 16 轮问题 1 指出 epsilonOf 与 NumericLimits 描述矛盾；迭代第 16 轮问题 4 指出 equalEpsilon 缺少验证计划覆盖。这些历史问题修复情况需独立验证，而非仅引用 §9 标题。

**[建议]** 抽样验证 §9 中 2~3 个历史问题（如 NaN 比较行为、零值除法的异常类型）是否已在当前设计文档中得到正确处理，而非仅引用章节标题。

## 质询要点（仅 CHALLENGED 时存在）

- **问题**："需求响应充分度"结论断言"无遗漏项"但未呈现独立可验证的证据支撑。
- **原因**：§8.2 追溯表自身在迭代历史中曾多次出现遗漏（第 8 轮 countof 偏离、第 21 轮 scalar_vec_ops.cj 映射缺失），其准确性需独立验证方可采信。报告仅陈述结论而未展示验证过程，降低了整体审查结论的可信度——设计文档作者和后续实施者无法据此判断"无遗漏"是否可靠。

- **建议方向**：补充 §8.2 与 roadmap §3E 的逐项对照表或验证方法说明，明确验证中发现的任何不精确之处。

---

- **问题**：参考实现使用策略的审查遗漏——任务需求明确提及"已有参考实现位于 references/glm-1.0.3/"，此问题已多次出现在迭代历史中（第 7 轮问题 7、第 10 轮问题 4），但报告未评估设计文档是否已提供可操作的参考实现对照验证方法。
- **原因**：此维度属于"内部审议（设计-验证循环）未充分覆盖的维度"，正是本质量审查应侧重之处。遗漏此维度使整体审查的针对性降低——设计文档的落地可执行性恰依赖于能否有效对照参考实现验证关键设计决策的正确性。

- **建议方向**：审查设计文档中 §1 或 §8.4 是否已补充参考实现使用策略、文件路径引用、以及验证方法。

---

- **问题**："接口定义是否足以支持下游消费者"维度的审查缺失——报告仅通过 F1 触及 API 风格问题，未系统评估公共 API 面设计、const/non-const API 分裂告知、别名消费者体验、类型转换可预见性等消费者视角维度。
- **原因**：任务明确要求从实际落地视角评估接口定义是否足以支持下游消费者。这一维度的缺失意味着报告的"实际落地视角"评估不完整——设计文档可能满足了技术正确性要求，但未提供消费者所需的决策充分性信息。

- **建议方向**：补充消费者视角的 4 个子维度（lib.cj API 面设计、const 上下文可用性告知、别名命名差异告知、类型转换行为可预见性）的系统性评估。

---

- **问题**：F4 严重程度标注（[提示]）与自身描述（"重构成本较高"×30 构造函数）之间逻辑自洽性断裂，且与 F3 的校准尺度不一致。
- **原因**：严重程度是审查报告的核心判定之一，不一致的校准尺度会削弱读者对整体优先级排序的信任。如果 F3 的"功能真空"升级为 [一般] 而 F4 的"重构成本较高"×30 仍为 [提示]，读者难以理解和信赖报告的严重程度评定标准。

- **建议方向**：将 F4 升级为 [一般] 并提供更充分的概率/影响论证，或明确解释为何高成本对应 [提示] 级别。
