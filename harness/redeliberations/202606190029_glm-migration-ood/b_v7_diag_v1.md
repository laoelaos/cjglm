# 质量审查报告 — GLM 1.0.3 仓颉迁移首轮 OOD 设计方案 v7

## 审查范围与视角

- **审查对象**：`a_v7_copy_from_v6.md`（v7 — 审查迭代修订版）
- **审查维度**：需求响应充分度、事实准确性、深度与完整性（侧重内部审议未充分覆盖的维度）
- **落地视角**：设计是否可直接指导编码实现、接口定义是否足以支持下游消费者、异常场景和边界条件是否已考虑

## 发现问题

### P1. 严重 — `const` 扩展成员函数的多处声明依据不足

**问题描述**：文档在 §4.4 和 §4.5 多处假设 extend 块中可定义 `const` 实例成员函数，但仓颉语言文档对扩展成员允许的修饰符定义为：`static`、`public`、`protected`、`internal`、`private`、`mut`，**不包含 `const`**。涉及的具体函数：

- §4.4 `bitwiseNot()` 在 extend 块中声明为 `const`（第 640 行）
- §4.5 `equalExact()` 在 extend 块中声明为 `const`（第 668-672 行代码示例）
- §4.5 `logicalAnd`/`logicalOr` 在 extend 块中定义为扩展成员函数（第 683-684 行），虽未显写出 `const` 修饰符，但 §7 D33 将其纳入与 D32 保持一致的 const 声明决策中
- §4.4 `<<`/`&`/`|`/`^` 的 extend 块位运算定义（第 624-643 行）——当前设计未对这些函数标注 `const`，但若后续轮次统一 const 化，同样面临此约束

**所在位置**：§4.4（第 624-643 行）、§4.5（第 668-672 行）、§4.5（第 683-684 行）、§7 D33（第 1043-1048 行）

**严重程度**：严重

**改进建议**：
1. 编码前原型验证中新增验证项：在 extend 块中声明 `const` 实例成员函数是否编译通过。
2. 若验证失败，将 extend 块中的所有 `const` 函数降级为非 `const`：
   - `bitwiseNot()` → 非 `const` 扩展成员函数（功能不受影响，仅失去编译期调用能力）
   - `equalExact()` → 非 `const` 扩展成员函数（已在 §4.5 中声明了此风险，并在 §7 D29 中定义了精确比较回退路径，合入此降级）
   - `logicalAnd`/`logicalOr` → 非 `const`（与 §4.5 现有描述一致，D33 的 const 化决议需撤回）
3. 同步更新 §7 D32 和 D33 中的 const 统一决策，明确"extend 块中的函数无法声明为 `const`"这项约束。

### P2. 中等 — §9.4 Bool→数值转换的 `const if (T2 is Bool)` 语法可行性依赖未纳入验证计划

**问题描述**：§9.4 方案 A 使用 `const if (T2 is Bool && T is not Bool)` 在构造函数体内实现编译期约束，但存在两个未验证的假设：

1. **对类型参数直接使用 `is` 运算符**：文档中其他所有 `is` 用法均为 `T(0) is Float64` 形式（先通过 `T(0)` 构造值再对值检测类型），从未使用过 `T2 is Bool` 这种对类型参数本身的直接 `is` 判断。仓颉 `is` 运算符的语义是值级类型检测（value is Type），而非类型级检测（TypeParameter is Type）。若语法不支持，`T2 is Bool` 将产生编译错误。

2. **`T is not Bool` 语法**：仓颉中 `is not` 是否为一元运算符的组合形式（`!(T(0) is Bool)`）而非独立语法关键字，需确认。

**所在位置**：§9.4（第 1159 行）

**严重程度**：中等

**改进建议**：
1. 将此项验证纳入 §10 的设计阶段验证要求表，新增一项依赖项，验证 `const if` 体内对类型参数使用 `is` 的语法可行性。
2. 若验证失败，默认启用方案 B（运行时分支），并更新 §9.4 的描述明确方案 A 不可行。
3. 同步审查文档中所有 `is` 运算符的用法，区分"值级检测"（已验证）和"类型参数级检测"（未验证）两种模式。

### P3. 中等 — §8.1 迁移文件清单未覆盖测试文件，编码启动指引存在缺口

**问题描述**：§8.1 文件清单（第 1055-1071 行）列出了所有 14 个迁移源文件，但未包含对应的测试文件。虽然 §12.3 描述了测试包组织结构，但作为"可直接指导编码启动"的设计文档，文件清单与测试组织结构之间存在断裂——维护者拿到文件清单后无法直接从清单推知需要创建哪些测试文件。

**所在位置**：§8.1 文件清单（第 1055-1071 行）、§12.3 测试工具（第 1553-1557 行）

**严重程度**：中等

**改进建议**：
1. 在 §8.1 文件清单中新增"对应测试文件"列，为每个迁移文件列出对应的测试文件路径。
2. 或新增单独的 §8.1.1 测试文件清单子节，列出所有测试文件及其与被测文件的对应关系。
3. 确保测试文件的目录结构与 §8.3 迁移顺序的阶段验证对应（如 type_vec1.cj 对应 test_type_vec1.cj，在阶段 5 的增量验证中引用）。

### P4. 中等 — §4.4 `<<` 默认溢出策略的事实准确性依赖编译器行为验证

**问题描述**：§4.4（第 647 行）和 §4.6（第 699 行）声明"仓颉中 `<<` 默认使用 `@OverflowThrowing` 策略（移位超出位宽抛 `ArithmeticException`）"。但 std.overflow 文档中 `ThrowingOp<T>` 接口**不包含** `throwingShl` 方法，表明标准溢出策略接口未覆盖移位运算。`WrappingOp<T>` 也未包含 `wrappingShl`。`<<` 运算符的默认溢出行为属于编译器实现定义行为，而非标准库接口覆盖。当前文档将此行为描述为已确认事实而非验证假设。

**所在位置**：§4.4（第 647 行）、§4.6（第 699 行）

**严重程度**：中等

**改进建议**：
1. 将当前"仓颉中 `<<` 默认使用 `@OverflowThrowing` 策略"的确定性陈述改写为"预期默认行为"或标注为待验证假设。
2. 在 §10 原型验证表中新增一项依赖：验证未标注 `@Overflow*` 的 `<<` 运算符在溢出时的行为，确认其是否抛出异常。
3. 若验证发现 `<<` 不使用 `@OverflowThrowing`，评估当前的 `@OverflowWrapping` 标注策略是否需要调整。

### P5. 一般 — 构造函数体系对 `Q` 参数的约束充分性存疑，跨 Q 类型构造的重载解析风险

**问题描述**：所有跨类型构造函数（`init<T2, Q2>(v: VecN<T2, Q2>)` 等）使用 `where Q2 <: Qualifier` 约束。但当 `T2 = T` 且 `Q2 = Q` 时，这些泛型构造函数可能与 `const init(x: T, y: T, ...)` 产生重载冲突。§4.1 第 461 行声明"跨类型转换构造函数"（参数为 Vec），第 468 行起定义 `const init`（参数为标量），参数类型不同故无直接冲突。但 §4.1 中 `public init<T2, Q2>(a: Vec1<T2, Q2>, b: T)` 等组合构造函数在 `T2 = T` 且 `Q2 = Q` 的场景下，与 Vec1 的 `const init(x: T)` 的参数列表不同，应无冲突。然而 D31（第 1021-1027 行）已识别了 fill-from-Vec1 场景的重载冲突，说明跨类型构造的重载解析确实存在风险。

**主要风险**：当前构造函数清单中，Vec2~Vec4 同时存在 `init(scalar: T)`（非 const）和 `const init(x: T, y: T, ...)`（const），两者参数数量和类型不同，无重载冲突。但 §3.2 第 208 行声明"同参数列表的普通 `init` 不再需要"，对应的设计决策（D11）基于"const 修饰符不构成区分依据"的约束。此约束在构造函数重载中的具体表现已在 D11 中阐述，但 D11 未评估对跨类型泛型构造函数的影响——当调用方传入 `Vec2<Int32, PackedHighp>` 作为参数时，编译器可能在多个匹配的泛型构造函数（如 `init<T2, Q2>(a: Vec2<T2, Q2>, b: T)` 和 `init<T2, Q2>(a: Vec2<T2, Q2>, b: Vec1<T2, Q2>)`）之间产生歧义。

**所在位置**：§4.1（第 529-575 行，Vec4 构造函数清单）、§7 D11（第 859-863 行）、§7 D31（第 1021-1027 行）

**严重程度**：一般

**改进建议**：
1. 在 §10 原型验证中新增重载解析验证，至少覆盖 Vec4 的 `init<T2, Q2>(a: Vec3<T2, Q2>, b: T)` 与 `init<T2, Q2>(a: Vec3<T2, Q2>, b: Vec1<T2, Q2>)` 在 `b` 为 Vec1 时的重载选择。
2. 若验证发现歧义，定义构造函数优先级策略或限制特定位数的组合构造形式。

### P6. 一般 — `const if` 编译期分支抑制依赖的综合评估未反映在验证计划中

**问题描述**：文档多处依赖 `const if` 的非选择分支编译抑制行为，但 §10 验证计划对这组依赖的管理不完整：

| 依赖 `const if` 抑制的位置 | 是否纳入 §10 验证计划 |
|---|---|
| §3.5 `ComputeEqual.callConst` 中 `const if (isIec559Of<T>())` | 部分覆盖（与 D29 合并验证） |
| §4.3 `mod` 函数整数/浮点双路径 | 已覆盖（P16 关键项） |
| §3.6 `isIec559Of<T>()` 中 `const if (T(0) is Float64)` | 无显式验证——依赖项①②③仅验证"const 兼容性"未验证"抑制" |
| §9.4 Bool→数值转换中 `const if (T2 is Bool)` | **未覆盖** |

`isIec559Of<T>()` 中的 `const if (T(0) is Float64 || T(0) is Float32)` 若编译器在非浮点 `T` 上对 `T(0) is Float64` 表达式进行类型检查失败（如 `Bool(0) is Float64` 中 `Bool(0)` 的构造是否合法），则整个 `isIec559Of` 方案不可行。§10 验证表依赖项①验证了 `T(0)` const 构造，但未将 **const if 非选择分支抑制**作为一个独立验证维度覆盖所有依赖点。

**所在位置**：§3.5（第 286-294 行）、§3.6（第 339-344 行）、§4.3（第 613 行）、§9.4（第 1159 行）、§10（第 1195-1201 行）

**严重程度**：一般

**改进建议**：
1. 在 §10 验证要求表的"通用编译期假设"区域新增一条独立验证依赖：`const if` 的非选择分支被编译器完全抑制（不执行语义检查）。此依赖作为所有 `const if` 分支选择的基础假设，以表格形式集中列出一处，各处引用。
2. 在 `isIec559Of<T>()` 的验证依赖项描述中明确标注：验证通过的标准不仅是"编译通过"，还包括"非浮点类型 `T` 的实例化不会产生编译错误"。

### P7. 一般 — 对 "参考实现" 的使用策略未说明

**问题描述**：用户需求明确指出"已有参考实现位于 references/glm-1.0.3/"，整篇设计文档大量引用 C++ GLM 源文件名和行为，但未说明在编码阶段如何利用 `references/glm-1.0.3/` 目录中已有的参考实现来验证设计正确性。例如：未定义如何通过参考实现验证 `ComputeEqual` 容差路径的数值等价性、未说明从参考实现中提取测试用例的方法、未提供参考实现到仓颉实现的交叉引用指引。

**所在位置**：全文

**严重程度**：一般

**改进建议**：
1. 在 §1 概述中新增"参考实现的使用策略"子节，说明：
   - 参考实现的目录结构和各文件对应哪些设计章节
   - 编码阶段如何从参考实现中提取测试向量（如 C++ GLM 的测试用例）
   - 关键的数值等价性验证如何借助参考实现完成
2. 或至少在 §12 验证策略中补充"与参考实现的交叉验证"方法。

## 整体评价

文档在 v7 版本中已高度成熟，对内部审议指出的问题均完成修改（附录 P1-P20），版本标识、构建基础设施、const 兼容性验证、mod const if 验证、并发设计等维度得到显著改善。当前存在的质量问题集中在：

1. **仓颉语言特性的边界验证不充分**——`const` 扩展成员函数、`const if` 非选择分支抑制适用范围、类型参数级 `is` 运算符的语法可行性等，均属于应在设计阶段验证但当前 §10 验证计划未充分覆盖的语言特性假设。
2. **编码启动指引的完整性**——文件清单与测试组织结构之间的断裂，以及对参考实现使用策略的缺失。
3. **编译器行为假设的准确性**——`<<` 默认溢出策略的描述需要从"事实陈述"改写为"待验证假设"。

以上问题均可在编码前原型验证阶段通过补充验证项解决，不影响设计的整体架构正确性。建议在启动编码前完成 P1（extend 块 const 语法）和 P6（const if 抑制依赖综合评估）的验证优先级最高，因为它们涉及面最广。
