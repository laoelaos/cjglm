# 质量审查诊断报告

> 审查对象：GLM 1.0.3 仓颉迁移首轮 OOD 设计方案（v15 — 审查迭代修订版）
> 审查轮次：第 1 轮
> 审查视角：需求响应充分度、事实错误与逻辑矛盾、深度与完整性、编码就绪度

---

## 1. 事实错误与逻辑矛盾

### 问题 1.1（严重）：`equalExact` 的 `const` 声明与 extend 块中 `const` 函数的编译器支持性未经验证

- **所在位置**：§4.5（第 543 行），§3.2（第 156 行）
- **问题描述**：`equalExact` 在 §4.5 中声明为 `const` 的扩展成员函数。该声明的两个前提均未在文档中得到验证：① extend 块能否定义 `const` 实例成员函数（仓颉 extend 语法是否允许 `extend` 内的函数加 `const` 修饰符）；② `const equalExact` 函数体内对 `Bool` 分量调用仓颉原生 `==` 是否在 const 上下文中合法。文档 §3.2 确认了 Vec 类型有 `const init`，但 extend 块中 `const` 函数的语法和约束在文档未给出任何 extend 代码示例或语法说明。
- **改进建议**：在 §4.5 或 §7 中补充 extend 块中定义 `const` 函数的语法示例（含 extend 块的结构），并标注此方案的验证状态。若 extend 块不支持 `const` 修饰符，则需将 `equalExact` 降级为非 `const` 扩展成员函数，并评估对 §12.1 层次三测试的影响。

### 问题 1.2（严重）：`Inf == Inf` 在容差路径下返 false，与 C++ GLM 行为显著偏离，且迁移成本评估未覆盖

- **所在位置**：§9.3（第 952 行），§11（迁移成本评估）
- **问题描述**：文档 §9.3 承认在容差比较路径下 `Inf == Inf` 返回 `false`（因 `Inf - Inf = NaN`，容差比较失败），而 C++ GLM 的 `operator==` 使用精确比较，`Inf == Inf` 返回 `true`。但文档只在 §9.3 标注为"已知局限性"，**未在 §11 迁移成本评估中纳入此行为差异的影响分析**。如果迁移的 GLM 代码包含 `Inf` 边界的相等性检测（如裁剪空间近平面检测、标准化后向量比较），此行为差异会导致运行时逻辑错误，且 `equalExact` 虽可解决但调用方不会自动改用。
- **改进建议**：在 §11 新增子节评估 `Inf == Inf` 及 `NaN == NaN` 的行为差异对迁移代码的影响范围，给出在 C++ GLM 代码中搜索 `Inf`/`NaN` 相关 Vec 比较的模式和替换规则（如检查点替换为 `equalExact` 或回退到精确比较路径）。同时在 §12.2 验收标准中增加与此行为差异相关的验证项。

### 问题 1.3（一般）：`epsilonOf<T>()` 的 where 约束与 D5 宽松约束在非执行路径上的编译器行为不确定性

- **所在位置**：§3.5（第 246-252 行），§7 D5（第 653-655 行）
- **问题描述**：`epsilonOf<T>()` 的签名包含 `where T <: Number<T>` 约束。文档论证称"`const if` 的分支仅在浮点 `T` 上进入，因此约束在此处合法"。但仓颉编译器在泛型实例化时是否对 `const if` 的**非执行分支**也进行约束检查，是不确定的。如果编译器在泛型实例化时对所有分支（包括 `const if` 的非执行分支）进行签名级约束检查，则 `T = Bool` 时 `epsilonOf<Bool>()` 的 `where Bool <: Number<Bool>` 会导致编译失败，进而导致 `VecN<Bool, Q>` 的 `==` 运算符无法编译。此问题在 14 轮内部审议中未被识别。
- **改进建议**：在 §7 D29 的验证优先级中增加 `epsilonOf<T>()` 约束检查的兼容性验证项（优先级 2.5）。若验证发现编译器对非执行分支也检查约束，则将 `epsilonOf<T>()` 的 `where` 约束移除，改为函数体内的 `assert` 或运行时检查，或将 epsilon 逻辑内联到 `ComputeEqual.callConst` 中以避免函数调用。

## 2. 深度与完整性不足

### 问题 2.1（严重）：`const if (T(0) is Float64)` 方案的三项核心依赖均未经过编译器验证，设计根基为未证实假设

- **所在位置**：§3.5（第 234-268 行），§10（第 991-999 行），§7 D29（第 825-846 行）
- **问题描述**：`ComputeEqual` 的浮点/整数分支选择是整个 `==` 运算符行为的设计核心，但其实现路径 `const if (T(0) is Float64 || T(0) is Float32)` 依赖三个未经验证的假设：① `T(0)` 在 const 上下文中对所有目标类型（`Int32`、`Float32`、`Bool` 等）合法；② `is` 运算符在 const 泛型函数中可参与编译期求值；③ `const if` 在泛型函数中按类型实例化的分支付择行为正确。这三项依赖中任一失败，都会导致 §3.5 的核心设计失效，进而影响 §4.5 `==` 运算符、§4.6 溢出策略依赖的 `isIec559Of` 路径、§9.3 浮点比较行为等全链章节。文档在 §10 末尾将其标记为"编码前验证"，实质上是将设计阶段的核心决策风险转移到编码阶段。编码阶段一旦验证失败，回退路径（精确比较）将导致 §3.5 `ComputeEqual` 和 §4.5 `==` 的大范围设计描述与实现不符。
- **改进建议**：将此"编码前验证"明确升级为"设计阶段验证"——在当前审议轮次中即编写原型测试代码验证 `const if (T(0) is Float64)` 在目标编译器上的可行性。如果无法立即验证，应在 §3.5、§4.5、§4.6、§9.3 等核心章节增加风险标注，明确注明"此方案基于未验证假设，编码阶段回退路径为精确比较"。同时将回退路径对 §9.3 `Inf == Inf` 行为的影响写入文档。

### 问题 2.2（严重）：`@OverflowWrapping` 自动继承假设未经验证，溢出策略核心依赖为灰色地带

- **所在位置**：§4.6（第 572-578 行），§7 D30（第 848-856 行）
- **问题描述**：§4.6 的整个溢出策略建立在"编译器自动生成的复合赋值版本继承二元运算符上的 `@OverflowWrapping` 标注"这一假设之上。§7 D30 承认此"行为未在仓颉文档中明确确认"并将其推至"编码前验证"。这与问题 2.1 性质相同——核心设计决策（20+ 处 `@OverflowWrapping` 标注的布局）依赖于编译器的未文档化行为。如果验证失败，备选方案（extend 块中的具名函数实现 Vec 算术）将导致二元运算符和复合赋值均失去运算符语法糖，§4.6 的完整标注策略需要重写。
- **改进建议**：在 §4.6 中明确标注"此策略取决于编译器对 `@OverflowWrapping` 继承性的实现行为，当前未经验证"。将备选方案（具名函数 + 运算符委托）的编码工作量纳入 §11 迁移成本评估。将原型验证提前到设计阶段完成。

### 问题 2.3（一般）：`scalar-op-vec` 方向具名函数的命名空间污染风险未评估

- **所在位置**：§4.3（第 514-517 行），§2（第 67 行）
- **问题描述**：`scalar-op-vec` 方向具名函数（`add(s, v)`、`sub(s, v)`、`mul(s, v)`、`div(s, v)`、`mod(s, v)`）被定义为包级独立函数，并通过 `lib.cj` 的 `public import` 暴露。这些函数名（`add`、`sub` 等）是极为通用的标识符。下游 `import glm.*` 后，如果其代码中已有同名函数（或在其他包中定义了同名函数），将发生命名冲突。文档未评估此风险，也未给出冲突的解决策略（如调用方使用全限定名 `glm.add()` 或选择性导入 `import glm.{ add, sub }`）。
- **改进建议**：在 §4.3 或 §7 中新增设计决策/说明段落，评估命名空间冲突风险并给出调用方的最佳实践（如建议使用 `import glm.{ add => glm_add }` 的导入重命名模式）。在 §12.2 验收标准中增加"具名函数导入无冲突"的验证项。

### 问题 2.4（一般）：§9.3 "改进方案"段落混入设计文档主体，造成编码人员混淆

- **所在位置**：§9.3（第 952 行）
- **问题描述**：§9.3 在描述 NaN/Infinity 行为约定的表格后，附带了"改进方案"段落（描述 `Float64.isNaN(a)` / `Float64.isInfinite(a)` 运行时检测方案）。该段落使用了"可""应""若需"等推测性措辞，与 §9 整体的"行为约定"定位不符。编码人员阅读此节时无法判断：当前应按照"首轮约定"实现，还是按照"改进方案"实现。此问题在历史迭代（v7 修订说明第 13 项）中已被触达但修正不彻底——描述的措辞从"通过 is 检查"改为"通过 isNaN/isInfinite 检测"，但"改进方案"段落的推测性质未改变。
- **改进建议**：将"改进方案"段落中的实现约束（仅有限值范围适用）合并到行为表格的"NaN/Infinity 参与比较运算"行中，移除段落中的推测性方案描述（如"改进方案""若需精确比较"等）。将这些备选方案的讨论转移到 §7 设计决策中记录。

### 问题 2.5（一般）：D29 验证树过于复杂，将设计不确定性实质性地推至编码阶段

- **所在位置**：§7 D29（第 820-846 行）
- **问题描述**：D29 定义了包含 5 个验证优先级、3 个触发回退条件的复杂决策树。此设计本质上将一个在编码阶段才能确认的复合条件（4 个环节的 const 兼容性）作为整个浮点容差比较路径的前提。与问题 2.1、2.2 同理，这实质上是将设计阶段的决策不确定性转移到了编码阶段。如果编码阶段验证后需要回退，涉及从 §3.5 ComputeEqual、§4.5 `==` 行为、§9.3 浮点比较到 §12.1 测试层次的全面修订，影响范围超出编码阶段的可控边界。
- **改进建议**：简化验证策略为二值决策——要么"完整 const 路径通过"（维持当前设计），要么"回退到精确比较"（统一使用精确 `==`）。移除优先级 2-4 的逐级回退方案。当前 D29 的"优先级 2 失败→移除浮点容差"、"优先级 3 失败→内联替代"等细分回退过于复杂，在编码阶段不可管理。简化后，如果完整路径验证失败，浮点容差比较通过独立具名函数（`equalApprox`）而非 `==` 提供，合同更加清晰。

## 3. 需求响应充分度

### 问题 3.1（一般）：256 个别名的首轮生成策略未明确，编码阶段缺乏可执行的生成方案

- **所在位置**：§3.8（第 352-356 行）
- **问题描述**：文档规定 256 个别名全量定义在 `fwd.cj` 中，命名约定也已规范化，但**未明确编码阶段如何生成这 256 行 `type` 定义**——是手动逐行编写、通过外部脚本生成、还是在仓颉中通过宏实现。256 行定义的规模使手动编写容易出错且难以审查，但文档未给出任何生成策略建议。"后续轮次可通过宏生成"的表述暗示首轮不做宏生成，但未说明首轮的替代方案。
- **改进建议**：在 §3.8 或 §8 迁移顺序中补充 256 个别名的生成策略——建议使用外部脚本（如 Python）将 GLM 原生名映射表转化为仓颉 `type` 别名文件。脚本的映射规则可直接从 §3.8 命名约定提取。同步更新 §12.2 验收标准中的验证方法以确认生成脚本的产物与命名约定一致。

### 问题 3.2（一般）：`add(v,s)` 扩展成员函数与 `v + s` 运算符的功能重叠未明确，调用方无选择依据

- **所在位置**：§4.3（第 505-513 行，第 514-518 行）
- **问题描述**：文档定义了 `vec + scalar` 二元运算符重载（§4.3 第 505 行），又将 `add(v, s)` 定义为扩展成员函数（§4.3 第 515 行），二者功能完全重叠（均执行逐分量加法）。文档未说明为什么需要同时保留两种路径，也未给出编码人员在何种场景下使用运算符、何种场景下使用具名函数的选择指南。
- **改进建议**：在 §4.3 明确说明二元运算符和具名函数的关系——二元运算符是首选路径（语法更简洁），`add(v, s)` 扩展成员函数与 `add(s, v)` 包级函数保持对称性（调用方可统一使用 `add` 模式而无需记忆哪些方向有运算符）。此说明可消除编码阶段的疑惑。

## 4. 编码就绪度

### 问题 4.1（一般）：§7 D29 的回退路径在文档其他章节中未体现，编码人员无法感知全量影响

- **所在位置**：§7 D29 回退路径 vs §3.5、§4.5、§9.3
- **问题描述**：§7 D29 定义了完整的回退触发条件和备选方案，但 §3.5（ComputeEqual）、§4.5（比较运算）、§9.3（浮点比较行为）等关键章节的文描述均以"完整 const 路径可行"为前提撰写，未标注"若 const 路径验证失败，本章内容将需要修正"。编码人员先阅读 §3.5/§4.5/§9.3 获得"容差比较"的心智模型，再在编码阶段通过 D29 得知这可能需要回退，造成信息获取时序错位。
- **改进建议**：在 §3.5 ComputeEqual 末尾、§4.5 `==` 运算符描述段落后增加回退路径标注："若 const 路径验证失败，`==` 对所有类型使用精确比较，浮点容差比较通过 `equalApprox` 具名函数提供"。类似标注也应在 §9.3 表格的浮点比较行添加。

### 问题 4.2（轻微）：`extend` 块中定义二元运算符的语法细节不足

- **所在位置**：§3.2（第 162-163 行）、§4.4（第 525 行）
- **问题描述**：§4.4 声明"二元位运算符通过 `extend` 块定义为扩展成员函数"，但文档未给出任何 `extend` 块中定义 `operator func` 的仓颉语法示例。对于编码人员，`extend Vec2<T,Q> where Q <: Qualifier { public operator func &(rhs: Vec2<T,Q>): Vec2<T,Q> @OverflowWrapping { ... } }` 的完整语法需要明确给出。
- **改进建议**：在 §4.3 或 §4.4 中至少提供一个 `extend` 块的完整语法示例（以 Vec2 的 `&` 运算符为例），包括类型参数约束、`@OverflowWrapping` 标注位置和函数体基本结构。

## 5. 整体质量评价

文档经过 15 个版本的迭代修订，在**广度**和**完备性**上已达到较高水平——构造函数清单已补全至与 GLSL 5.4.1 规范一致，范围可追溯性已完整映射 roadmap 子范围，异常场景和边界条件已系统化定义，迁移成本评估覆盖了主要行为差异。

**但文档存在一个结构性问题**：有三个设计核心路径（`const if + isIec559Of` 方案、`@OverflowWrapping` 自动继承、`const` 表达式链兼容性）的可行性均依赖**未经验证的编译器行为**，但文档将其归为"编码前验证"。这使得编码阶段成为这些设计假设的"真正的验证阶段"，而不是"按设计实现"的阶段。如果其中任何一项在编码阶段验证失败，文档描述的核心设计方案（容差比较型 `==`、溢出标注策略、const 函数路径）将需要大范围修订，影响从 §3 到 §12 的多个核心章节。

**建议**：在本轮审议中，将上述三项"编码前验证"升级为**原型代码验证**——在当前迭代中编写原型代码片段（或伪代码），在目标编译器上确认每项假设的可行性。验证结果直接写入文档，明确标注"已验证通过"或"回退路径启用"。
