## 发现 1（严重）— D30 回退路径存在循环委托风险

**所在位置**：§4.3 `@OverflowWrapping` 标注策略段 + §4.6 备选方案描述 + §7 D30 + §11.6 备选方案变更成本

**问题描述**：D30 回退方案的设计存在循环委托的逻辑漏洞。

当前设计中的委托方向：
- **二元运算符**（`+`、`-`、`*`、`/`、`%`）：直接逐分量计算（§4.6 "运算符内部实现逐分量直接运算"）
- **扩展成员函数**（`add(v,s)`、`sub(v,s)` 等）：委托给二元运算符（§4.3 "内部实现委托给对应 Vec 的二元算术运算符"，示例 `func add(self, s) { self + s }`）

D30 回退方案要求（§7 D30）：
1. 在 20 个扩展成员函数上补充 `@OverflowWrapping` 标注
2. **二元运算符委托目标**从"直接逐分量运算"改为"调用已标注的扩展成员函数"

**问题**：D30 回退方案仅描述了二元运算符委托目标的修改，但**未提及需要同步修改扩展成员函数的实现**。若仅反转委托方向而不解除扩展成员函数对二元运算符的原有委托，将产生：
```
+ (this, s) → this.add(s) → this + s → 无限递归
```

**严重性**：如果在编码阶段 D30 验证失败（`@OverflowWrapping` 继承性不成立）并实施回退方案，未注意到此循环依赖的实施者将产出运行时栈溢出。

**改进建议**：在 §7 D30 的备选方案描述中，将当前的三步描述（补充标注、修改委托目标、复合赋值消除）补充为四步，明确增加第 3 步：
> ③ **同步修改 20 个扩展成员函数的实现**：从"委托给二元运算符"改为"直接逐分量计算"（与原二元运算符的实现体一致，即 `Vec2(this.x + s, this.y + s)` 等）。否则二元运算符委托给扩展成员函数、扩展成员函数又委托回二元运算符将形成循环委托，产生无限递归。

同步更新 §11.6 备选方案变更成本表，在工作量评估中补充扩展成员函数实现重写的成本（20 处从委托改为直接计算，约 0.5-1 人天），合计成本从约 1 人天修正为约 1.5-2 人天。

## 发现 2（中等）— `GLM_CONFIG_PRECISION_*` 常量值存在事实错误

**所在位置**：§7 D6，GLM_CONFIG_* 常量完整清单表，最后 5 行（`GLM_CONFIG_PRECISION_BOOL` ~ `GLM_CONFIG_PRECISION_DOUBLE`）

**问题描述**：该表中 5 个精度配置常量的"对应 C++"列声明值与实际 C++ 参考实现不一致：

| 文档声明 | 实际 C++ 值（`references/glm-1.0.3/glm/glm/detail/setup.hpp:910-922`） |
|---------|----------------------------------------------|
| "对应 C++ `GLM_HIGHP`=3" | `#define GLM_HIGHP 1` |
| "对应 C++ `GLM_MEDIUMP`=3"（注：表中声明值为 `3`，标注为 HIGHP） | `#define GLM_MEDIUMP 2` |
| "对应 C++ `GLM_LOWP`=3"（注：表中声明值为 `3`，标注为 HIGHP） | `#define GLM_LOWP 3` |

文档将 `GLM_HIGHP` 的值错误标注为 `3`，实际为 `1`。若"对应"含义为"此常量值在 C++ 中的等效宏为 GLM_HIGHP（值为 3）"，则值 `3` 是错误的——GLM_HIGHP 的值为 `1`，值为 `3` 的宏是 `GLM_LOWP`。

**影响评估**：低。首轮 `setup.cj` 中的精度配置常量采用"字面量替换"策略（§7 D6 "子常量编码策略"），这些常量在首轮源码中不被功能代码引用，因此错误的值不影响编译或运行时行为。但错误的参考信息可能误导实施者在编码或后续轮次中交叉引用 C++ 参考实现时产生困惑。

**改进建议**：
1. 将各精度常量的"对应 C++"列修正为准确的宏名+值，如 `GLM_HIGHP=1`、`GLM_MEDIUMP=2`、`GLM_LOWP=3`
2. 或移除"对应 C++"列中具体数值，仅保留宏名（如"对应 C++ `GLM_HIGHP`"），避免值与 C++ 定义的对齐要求产生误解

## 发现 3（中等）— `equalEpsilon` 回退路径联动修改的依赖链未在 §12 验收标准中覆盖

**所在位置**：§3.5 `epsilonOf<T>()` 兼容性策略 + §4.5 `equalEpsilon` 定义 + §10 依赖④ + §12.2 验收标准表

**问题描述**：§10 依赖④已正确归档 `equalEpsilon` 与 `epsilonOf<T>()` 回退路径间的隐式依赖链——当 `epsilonOf<T>()` 的兼容性策略被触发（移除 `Number<T>` 约束，内联固定容差），`equalEpsilon` 体内的 `epsilonOf<T>()` 引用需同步改为内联固定容差。**但此联动修改未在 §12.2 验收标准表中体现**。实施者在编码阶段若触发了 `epsilonOf<T>()` 回退，可能遗漏 `equalEpsilon` 的同步修改。

**影响评估**：中等。若回退触发但联动修改遗漏，`equalEpsilon` 引用的 `epsilonOf<T>()` 在整数 T 上因 `Number<T>` 约束不满足而产生编译错误——编译期即可发现，不会产生静默错误。但遗漏将导致实施者需要额外时间定位和修复。

**改进建议**：在 §12.2 验收标准表末尾新增一项：
> | equalEpsilon 与 epsilonOf 回退联动 | 若 `epsilonOf<T>()` 的兼容性策略触发，`equalEpsilon` 实现同步从引用 `epsilonOf<T>()` 改为内联固定容差 `T(1e-6)` | §10 依赖④约定 + 层次三 `equalEpsilon` 验证项通过 |

## 发现 4（一般）— `public const init` 在 Vec2~Vec4 上的 const 可达性依赖未完全显式归档

**所在位置**：§4.1 构造函数签名清单（Vec2~Vec4 的 `public const init(x, y, ...)`）+ §7 D11

**问题描述**：D11 和 §4.1 已正确记录了 Vec1 的构造函数不对称性（`const init(x: T)` 与 `public init(scalar: T)` 因参数列表相同而不可共存），但 Vec2~Vec4 的 `public const init(x, y, ...)` 与各自的 `public init(scalar: T)` 之间的重载解析交互未显式验证。虽然参数数量不同（`const init` 2-4 个参数 vs `init` 1 个参数）在语法层面不会产生重载冲突，但当调用方使用命名参数或默认值时是否存在意外匹配未被分析。

此外，`public const init` 在 Vec2~Vec4 上的参数形态使得它们在非 `const` 上下文中可作为逐分量运行时构造函数（const README §3.2 规则 5），但此语义依赖是否在所有 Vec 类型（特别是带泛型约束 `Q <: Qualifier` 时）上一致可用，仅在 §10 验证①中被提及，未在 §12.2 验收标准中独立列项。

**影响评估**：一般。Vec2~Vec4 的 `const init` 与 `public init(scalar: T)` 参数数量不同，重载冲突风险低。但该验证未在验收标准中独立列项，可能在编码阶段被遗漏。

**改进建议**：在 §12.2 验收标准中新增一项：
> | const init 泛型构造可用性 | 对 Vec2~Vec4 各验证一次：在独立测试包中通过 `import glm.VecN` 后，使用 `VecN<Float32, PackedHighp>(1.0f, 2.0f, ...)` 语法构造可在非 const 上下文中编译通过 | `cjpm build` + 构造正确性验证 |
