以假设的下游消费者视角评估——通过 `import glm.*` 后可访问的 API 表面：

| API 类别 | 覆盖情况 | 评估 |
|---------|---------|------|
| **Vec 类型** | `Vec1<T,Q>`~`Vec4<T,Q>` | 通过 `lib.cj` 的 `public import` 暴露 ✓ |
| **Qualifier 类型** | `PackedHighp`/`PackedMediump`/`PackedLowp`/`Defaultp` | 通过 `lib.cj` 暴露 ✓；Aligned 系列首轮 `internal`，符合设计意图 ✓ |

### 2.2 公共 API 缺口分析

| 路径 | 评估 | 理由 |
|------|------|------|
| `<`/`<=`/`>`/`>=` | **刻意排除**，可接受 | D22 记录理由：C++ GLM 自身未定义，语义选择需慎重 |
| `&&`/`||` | **已替代**，可接受 | 替换为 `logicalAnd`/`logicalOr`，D12 记录理由 |
| `~` | **已替代**，可接受 | 替换为 `bitwiseNot()`，D12 记录理由 |
| `++`/`--` | **已替代**，可接受 | 替换为 `increment()`/`decrement()`，§3.2/§4.3 记录理由 |
| `toString()`/`Display` | **刻意排除**，可接受 | §3.2 字符串表示策略记录理由 —— 首轮依赖编译器默认表示 |
| `fromBoolVec` 工厂函数 | 已纳入 §9.4 路径 A | 首轮推荐方案，需在 Vec extend 块中定义 |
| `mod` 扩展成员函数浮点路径 | **受限**，可接受 | 因 extend 块不支持 `const`，浮点路径仅通过包级独立函数 `mod(s,v)` 提供，§4.3 已记录 |

### 2.3 下游消费者使用路径验证

```cangjie
// 预期合规的消费者代码模式（覆盖常见场景）：
import glm.*

main() {
    // 构造
    let v = Vec4<Float32, PackedHighp>(1.0f32, 2.0f32, 3.0f32, 4.0f32)
    // 分量访问
    let x = v[0]
    let y = v.componentAt(1)
    // 算术运算
    let sum = v + v
    let wrapped = v.add(1.0f32)    // 扩展成员函数
    let scalarLeft = add(1.0f32, v) // 包级独立函数
    // 比较运算
    let eq = (v == sum)
    let exact = v.equalExact(sum)
    let eps = v.equalEpsilon(sum)
    // 位运算（整数 Vec）
    let bv = IVec4(1, 2, 3, 4)
    let and = bv & IVec4(0, 1, 0, 1)
    // 递增/递减
    var mv = IVec2(1, 2)
    mv = mv.increment()
    // 类型别名
    let v2f: Vec2f = Vec2f(1.0f32, 2.0f32)
}
```

上述模式均可在 `import glm.*` 后直接使用，无需额外导入 `glm.detail` 包。**结论**：公共 API 表面完整，足以支撑下游消费者的常见向量操作场景。

## 四、发现问题

以下问题按严重程度降序排列。鉴于质询意见（§3）指出发现 1（D30 循环委托）和发现 2（精度常量值错误）更适合内部审议渠道，此处仅保留发现 3、发现 4 并补充一项新的全局性发现。

### 发现 1（中等）— `equalEpsilon` 与 `epsilonOf<T>()` 回退路径联动修改未在 §12.2 验收标准中覆盖

**所在位置**：§3.5 `epsilonOf<T>()` 兼容性策略 + §4.5 `equalEpsilon` 定义 + §10 依赖④ + §12.2 验收标准表

**问题描述**：§10 依赖④已归档 `equalEpsilon` 与 `epsilonOf<T>()` 回退路径间的隐式依赖链——当 `epsilonOf<T>()` 的兼容性策略被触发（移除 `Number<T>` 约束，内联固定容差），`equalEpsilon` 体内的 `epsilonOf<T>()` 引用需同步改为内联固定容差。但此联动修改未在 §12.2 验收标准表中体现。实施者若触发 `epsilonOf<T>()` 回退，可能遗漏 `equalEpsilon` 的同步修改。

**影响评估**：若回退触发但联动修改遗漏，`equalEpsilon` 引用的 `epsilonOf<T>()` 在整数 T 上因 `Number<T>` 约束不满足而产生编译错误——编译期即可发现，不会产生静默错误。但遗漏将导致实施者需要额外时间定位和修复。

**改进建议**：在 §12.2 验收标准表末尾新增一项：
> | equalEpsilon 与 epsilonOf 回退联动 | 若 `epsilonOf<T>()` 的兼容性策略触发，`equalEpsilon` 实现同步从引用 `epsilonOf<T>()` 改为内联固定容差 `T(1e-6)` | §10 依赖④约定 + 层次三 `equalEpsilon` 验证项通过 |

### 发现 2（一般）— `public const init` 在 Vec2~Vec4 上的 const 可达性依赖未完全显式归档

**所在位置**：§4.1 构造函数签名清单（Vec2~Vec4 的 `public const init(x, y, ...)`）+ §7 D11

**问题描述**：D11 和 §4.1 已记录 Vec1 的构造函数不对称性，但 `public const init` 在 Vec2~Vec4 上的参数形态使得它们在非 const 上下文中可作为逐分量运行时构造函数（const README §3.2 规则 5），此语义依赖是否在所有 Vec 类型（特别是带泛型约束 `Q <: Qualifier` 时）上一致可用，仅在 §10 验证①中被提及，未在 §12.2 验收标准中独立列项。

**影响评估**：Vec2~Vec4 的 `const init` 与 `public init(scalar: T)` 参数数量不同，重载冲突风险低。但该验证未在验收标准中独立列项，可能在编码阶段被遗漏。

**改进建议**：在 §12.2 验收标准中新增一项：
> | const init 泛型构造可用性 | 对 Vec2~Vec4 各验证一次：在独立测试包中通过 `import glm.VecN` 后，使用 `VecN<Float32, PackedHighp>(1.0f, 2.0f, ...)` 语法构造可在非 const 上下文中编译通过 | `cjpm build` + 构造正确性验证 |

### 发现 3（一般）— 文档缺少对 `mod` 扩展成员函数浮点路径不可用的显式 API 影响说明

**所在位置**：§4.3 `mod` 函数双路径描述 + §4.6 `@OverflowWrapping` 标注分布表

**问题描述**：§4.3 正确指出 `mod` 扩展成员函数方向（`v.mod(s)`）因定义在 `extend` 块中，不支持 `const` 修饰符，无法使用编译期 `if`，因此仅提供整数路径。浮点路径仅由包级独立函数 `mod(s, v)` 覆盖。这意味着 `v.mod(s)` 语法（扩展成员函数风格）对浮点 Vec 不可用——调用方必须使用 `mod(s, v)`（标量在左的包级函数风格）。此 API 不对称性已在 §4.3 中描述，但未在 §12 的易用性验证或公共 API 行为中显式标注。

**影响评估**：一般。对整数 Vec 用户无影响；对浮点 Vec 用户，需注意 `v.mod(s)` 不可用而 `mod(s, v)` 可用。此差异在编码阶段即可通过编译错误发现，不会产生静默错误，但缺少显式文档标注可能增加首次使用时的困惑。

**改进建议**：在 §4.3 `mod` 函数描述末尾或 §12.2 验收标准表中补充标注：
> "注意：`mod` 的扩展成员函数方向（`v.mod(s)`）仅支持整数 `T`。浮点 `T` 的取模运算必须使用包级独立函数 `mod(s, v)`（标量在左）——因 extend 块不支持 `const` 修饰符，浮点双路径无法在扩展成员函数中实现。"
