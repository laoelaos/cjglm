- **审查产出**：`a_v6_design_v2.md`（GLM 1.0.3 仓颉迁移首轮 OOD 设计方案）
- **审查视角**：需求响应充分度、事实正确性、深度完整性、落地可行性

- **审查范围**：重点覆盖内部审议（组件 A）未能充分覆盖的维度，避免重复已确认的技术可行性结论

## 发现的问题

### P1（严重）— 文档版本标识自相矛盾，实施者无法确定当前版本

- **所在位置**：文档标题（第 1 行）、文件路径名、文档末尾「修订说明（v6）」
- **问题描述**：文档标题声明为 `v5 — 审查迭代修订版`，文件路径名为 `a_v6_design_v2.md`，末尾最新修订说明标记为 `v6`。三重版本标识不一致。此外，文档末尾的修订说明序列混乱——从 `v7` 开始历经 v8→v15，然后跳回 v2→v6，实施者无法据此判断文档主体所代表的实际版本。
- **严重程度**：严重
- **改进建议**：① 将文档标题统一为当前实际版本号（v6），与文件路径名和最新修订说明一致；② 整理修订说明部分，按时间正序排列并标注当前生效版本，将历史修订移入附录或归档文件；③ 在文档顶部增加版本变更摘要行（如 `Last revised: v6 (2026-06-19)`），使实施者一眼可知当前版本。

### P2（严重）— 缺少包构建基础设施定义，无法直接指导编码启动

- **所在位置**：§2 模块划分、§8 迁移文件清单、全文
- **问题描述**：设计详细描述了文件组织和包结构（`glm.detail`/`glm`），但未定义以下编码启动的前提条件：
  - 包配置文件（`cjpm.toml`）的内容——包名、版本号、依赖声明等
  - 编译入口和构建命令（`cjpm build` 的配置前提）
  - 项目初始化的脚手架步骤（`cjpm init` → 目录调整 → 文件填充的完整流程）
  - 测试运行器配置（`@Test`/`@TestCase` 框架的使用前提）
  实施者拿到设计文档后，无法从"零"开始直接执行第一个编译步骤。
- **严重程度**：严重
- **改进建议**：在 §2 模块划分之前或 §8 迁移顺序中补充"前置准备工作"子节，提供最小 `cjpm.toml` 模板、初始目录结构示例和第一条编译验证命令。确保实施者能按文档所述从 `mkdir` 到 `cjpm build` 通过无中断路径执行。

### P3（严重）— D32 声明 40 个算术函数为 `const`，但 const 兼容性验证计划（D29）未覆盖

- **所在位置**：§7 D32（第 935-939 行）、§7 D29（第 887-901 行）、§10 验证要求表（第 1080-1088 行）
- **问题描述**：D32 将 `add`/`sub`/`mul`/`div`/`mod` 共 40 个函数（两种方向 × 四种 Vec 类型 × 五种运算）统一声明为 `const` 函数。但：
  - D29 的 const 表达式链验证仅针对 `const operator ==`（`abs(a - b) <= epsilonOf<T>()`），不覆盖算术运算的 const 兼容性
  - §10 的设计阶段验证要求仅覆盖 `isIec559Of<T>()` 的实现、`abs()`/`epsilon()` 的 const 兼容性和浮点 `mod` 的算术恒等式正确性——同样未覆盖算术运算函数体（如 `component + s`）在 `const` 上下文中的编译通过性
  - 若目标类型 `T` 的算术运算符（`+`、`-`、`*`、`/`）在 `const` 函数体内不可用，则 40 个函数的 `const` 声明将同时失效，影响范围远超 D29 回退方案所覆盖的范围
- **严重程度**：严重
- **改进建议**：① 在 §7 D32 中补充 `const` 兼容性验证要求，引用 D29 的验证模式扩展至算术运算；② 在 §10 中新增"算术运算 const 兼容性"验证依赖项，覆盖 `T + T`、`T - T`、`T * T`、`T / T`（对 `Int32`、`Float32`、`Bool` 三种代表性类型）在 `const` 函数体内的编译通过性；③ 定义 D32 的独立回退方案：若部分 `T` 不支持 `const` 算术，则应降级为仅 `const if` 的必要函数（`mod`）保留 `const`，其余函数移除 `const` 修饰符，并在 D29 或 §10 中记录此回退路径。

### P4（中等）— `compute_vector_decl.cj` 的双重角色（消费型+预留型）架构层次模糊

- **所在位置**：§2 模块划分（第 46 行）、§7 D16（第 795-802 行）
- **问题描述**：`compute_vector_decl.cj` 同时包含两类代码：(1) 首轮**被消费**的 `scalar-op-vec` 方向辅助函数（`add(s, v)`、`sub(s, v)` 等 20 个函数）；(2) 首轮**仅定义不消费**的 `compute_vec_*` 策略结构体（为后续 SIMD 轮次预留）。D16 将此列为"例外项"，但这种混合架构导致：
  - 编码人员在修改被消费的函数时可能不慎影响到预留的策略结构体
  - 后续轮次引入 SIMD 路径时需要拆分该文件，产生重构成本
  - 文件职责不单一（single responsibility principle 违反），文件级 import 可见性难以精确控制
- **严重程度**：中等
- **改进建议**：将 `scalar-op-vec` 方向辅助函数独立为单独文件（如 `scalar_vec_ops.cj`），与预留的 `compute_vec_*` 策略结构体分离。`compute_vector_decl.cj` 仅保留策略结构体定义。`lib.cj` 的 `public import` 改为引用新文件中的具名函数。此拆分在首轮实施成本低（仅增加一个文件），可消除架构模糊并为后续轮次提供清晰的扩展边界。

### P5（中等）— `Float16` 的 `T(0)` const 构造兼容性未在验证计划中覆盖

- **所在位置**：§10 设计阶段验证要求表（第 1082-1086 行）、§10 `isIec559Of` 对照表（第 1065 行）
- **问题描述**：`isIec559Of<T>()` 的实现为 `const if (T(0) is Float64 || T(0) is Float32)`。§10 验证依赖①列出了 `T(0)` const 构造的验证，但验证方法中仅指定了 `T = Int32, Float32, Bool` 三种类型，未包含 `Float16`。虽然设计声明 `Float16` 不在首轮类型系统中活跃使用，但 `isIec559Of<T>()` 是一个无约束泛型函数，理论上可被任意 `T` 实例化。如果调用方在不知情的情况下对 `Float16` 调用了该函数，或者未来的维护者对 Vec 泛型添加了 `Float16` 支持，则 `T(0)` const 构造可能失败。当前验证计划未排除此风险。
- **严重程度**：中等
- **改进建议**：在 §10 验证依赖①中将 `Float16` 也纳入验证类型列表（与 `Int32`、`Float32`、`Bool` 并列）。若验证发现 `Float16(0)` 在 const 上下文中不可用，则在 `isIec559Of` 实现前增加 `const if` 前置防护，或在函数文档注释中标注 `Float16` 的限制。

### P6（中等）— 缺少增量验证节点，迁移顺序无法作为编码质量门禁

- **所在位置**：§8.3 迁移顺序（第 994 行）、§12 实现验证与验收策略（第 1369 行起）
- **问题描述**：§8.3 定义了迁移顺序（`setup.cj` + shim 文件 → `qualifier.cj` → 辅助工具 → Vec 类型 → 别名），§12 定义了最终验收层次，但两者之间缺少**逐步骤的增量验证节点**。实施者在完成每个文件/每组文件后应执行什么验证标准才能确认可进入下一步？例如：
  - 完成 `setup.cj` 和三个 shim 文件后，应验证什么？
  - 完成 `qualifier.cj` 后，如何确认 Qualifier 体系正确？
  - 完成 `type_vec2.cj` 后，可否仅编译验证还是需要写单元测试？
  当前设计将验证集中到了最终阶段（§12），增加了早期错误被延迟发现的风险。对于 1665 行的复杂设计文档，缺少中期验证节点会导致错误级联。
- **严重程度**：中等
- **改进建议**：在 §8.3 迁移顺序表中为每个阶段/子阶段补充"增量验证"列，标注该阶段完成后应执行的验证动作（编译通过/原型测试/手动审查）。例如：
  - 阶段 1（setup + shims）：`cjpm build` 无报错
  - 阶段 2（qualifier）：编写原型测试确认 `PackedHighp <: Qualifier` 编译通过、跨 Q 构造合法
  - 阶段 3（Vec 类型）：至少一个 Vec 类型（如 `Vec2<Int32, PackedHighp>`）的编译验证通过
  - 阶段 4（别名）：首批别名编译验证、重复定义检测

### P7（一般）— §9 异常场景表引用了 `equalEpsilon` 但该函数未在首轮范围内

- **所在位置**：§9.3 NaN/Infinity 传播表（第 1034 行：`equalEpsilon` 具名函数）
- **问题描述**：§9.3 的"NaN/Infinity 传播"行为约定表中，第 1034 行写道"调用方可通过 `equalEpsilon` 具名函数获取更精确的控制，不属于首轮范围"。此描述自相矛盾——`equalEpsilon` 既被引用为 NaN/Infinity 比较的"补救措施"，又被声明为"不属于首轮范围"。实施者会困惑：究竟应如何处理 Infinity 比较场景？如果 `equalEpsilon` 不在首轮范围内，则 §9.3 对它的引用是无效指引。
- **严重程度**：一般
- **改进建议**：将 §9.3 中的 `equalEpsilon` 引用替换为 `equalExact`（已在 §4.5 中定义且确认为首轮范围），或删除该引用并明确说明"NaN/Infinity 的精确比较通过 `equalExact()` 完成，详见 §4.5"。如果 `equalEpsilon` 确属后续轮次，则将其从 §9.3 中完全移除，避免给实施者双重信号。

### P8（一般）— 256 个别名降级策略的触发条件语焉不详

- **所在位置**：§3.8 首轮实现策略（第 355-358 行）
- **问题描述**：§3.8 描述了"若首轮孵化阶段因进度或资源限制需要缩小范围"可将别名从 256 缩减至 64 个，同时声明"必须触发范围变更评审"。但未定义：
  - 谁有权决定触发缩编（主 Agent/实施者/审查者）？
  - 缩编的触发阈值是什么（如"进度落后 2 周"或"编译验证失败"）？
  - 64 个之外剩余的 192 个别名何时必须补全？"第 2 轮"的具体轮次编号是？
  缺少这些定义，降级策略在实际执行中可能被滥用为随意裁剪范围的借口，或相反——无人敢触发缩编导致进度风险。
- **严重程度**：一般
- **改进建议**：① 在 §3.8 中补充降级触发条件的量化标准（如"编译验证阶段发现别名文件编译时间超过 30 分钟"或"核心 Vec 类型编码进度落后 2 周以上"）；② 指定触发决策的审批路径和文档化要求（如"范围变更评审需由主 Agent 记录到迭代历史文件"）；③ 明确缩编后剩余别名的最迟补全期限（如"第 7 轮审议前必须全量补全"）。

### P9（一般）— `lib.cj` 的 `public import` 行在文档的多处位置不一致

- **所在位置**：§2 公共 API 面设计（第 67 行）、§8.1 文件清单（第 968 行）、§8.2 范围追溯（第 990 行）

### P10（严重）— 版本标识混乱在 v6 迭代中未得到根本性解决

### P11（中等）— `lib.cj` 未重导出 `fwd.cj` 的别名，下游消费者的导入路径不完整

- **所在位置**：§2 公共 API 面设计（第 60-74 行）、§8.1 文件清单（第 967-968 行）
- **问题描述**：`lib.cj` 的 `public import` 仅导出了 Vec 结构体和 Qualifier 类型（`Vec1`~`Vec4`、`Qualifier`、`PackedHighp`/`Mediump`/`Lowp`、`Defaultp`）以及 `scalar-op-vec` 方向辅助函数，**未导出 `fwd.cj` 中的类型别名**（如 `BVec2`、`Vec4Float32` 等 256 个别名）。下游消费者若要使用 `BVec2` 别名，需要额外执行 `import glm.fwd.BVec2`，而非 `import glm.BVec2`。此导入路径的分裂增加了公共 API 的使用复杂性——消费者需要记住哪些类型来自 `lib.cj`、哪些来自 `fwd.cj`。
- **严重程度**：中等
- **改进建议**：在 `lib.cj` 中增加 `public import glm.fwd.*`（或精确列出常用别名），使 `import glm.*` 一条语句即可获得所有公共 API 类型和别名。如果担心 `fwd.cj` 中 256 个别名污染命名空间，可选择仅导出常用别名家族的代表（如 `Vec2Float32`、`IVec3`、`BVec4` 等），其余别名仍通过 `import glm.fwd` 按需导入。同时在 §2 公共 API 面设计中明确记录此导入策略。

### P12（中等）— `logicalAnd`/`logicalOr` 的函数形态（成员函数 vs 包级函数）未明确指定

- **所在位置**：§4.5 布尔逻辑运算（第 597-601 行）
- **问题描述**：§4.5 仅说明"各 VecN 类型分别定义对应版本"和签名格式（`logicalAnd(a: VecN<Bool, Q>, b: VecN<Bool, Q>): VecN<Bool, Q>`），但未指定这些函数的具体定义形态：是 Vec 的**扩展成员函数**（通过 `extend VecN<Bool, Q>` 块定义，调用时通过 `a.logicalAnd(b)` 语法）还是**包级独立函数**（调用时通过 `logicalAnd(a, b)` 语法）？两种形态的调用语法、导入方式和重载行为存在显著差异。
- **严重程度**：中等
- **改进建议**：在 §4.5 布尔逻辑运算节中明确指定 `logicalAnd`/`logicalOr` 的函数形态：推荐定义为 Vec 的扩展成员函数（与 `add(v,s)` 的 `vec-op-scalar` 方向保持一致），支持 `a.logicalAnd(b)` 的调用语法。同时明确是否需要包级独立重载版本（如对 `Vec1<Bool,Q>` 和 `Vec4<Bool,Q>` 是否需要不同的独立函数）。补充 extend 块中的定义示例。

### P13（一般）— §6 并发设计过于简略，未覆盖 Vec 在并发容器中的行为

- **所在位置**：§6 并发设计（第 661-663 行）
- **问题描述**：§6 仅声明"Vec 结构体是值类型，所有运算不可变（产生新向量）或就地修改（复合赋值）。值类型的复制语义天然线程安全——不存在同一内存位置的并发写冲突。"此描述未覆盖以下并发场景：
  - Vec 在 `Atomic` 或 `Mutex` 保护下的使用模式
  - Vec 的 `@Derive[Hashable]` 产生的 `hashCode()` 在多线程哈希容器中的线程安全性
  - 复合赋值运算符（如 `+=`）在多线程环境下对同一 `var` 绑定的 Vec 变量的并发写——虽然值语义下每个线程有独立的副本，但如果多个线程通过 `AtomicRef` 或类似的引用共享方式访问同一 `var` 变量，则 `v = v.increment()` 模式（读-修改-写）不是原子的
- **严重程度**：一般
- **改进建议**：扩展 §6 并发设计覆盖以下场景：① 明确说明 Vec 类型的 `let` 绑定在并发场景下天然安全（不可变）；② 明确说明 `var` 绑定的 Vec 的复合赋值和 `v = v.increment()` 等读-修改-写模式在多线程下不做原子性保证；③ 如需跨线程安全修改，调用方应使用 `Mutex` 或原子类型的包装器。

### P14（一般）— §9.4 `Bool` → 数值类型转换的约定与 D14 跨类型转换策略不一致

- **所在位置**：§9.4 类型转换边界（第 1046 行）、§7 D14（第 783-787 行）
- **问题描述**：§9.4 对 `Bool` → 数值类型转换的约定为："跨 Vec 转换构造中 `Bool` 不可直接转换为数值类型（仓颉中 `Int32(Bool)` 可能不被支持）。非首轮范围——首轮 `Bool` 类型的 Vec 不参与跨类型数值转换。若需此功能，在构造函数中通过 `if` 分支显式处理（`true → T(1)`，`false → T(0)`）。"但 D14 的跨 Vec 转换构造函数使用了无约束泛型 + `T(v)` 显式转换——如果 `v.x` 是 `Bool` 类型，`T(v.x)` 在 `T` 为数值类型时是否可编译通过，取决于仓颉是否支持 `Int32(true)` 等构造。设计将此约定为"非首轮范围"，但跨类型 Vec 转换构造函数是无条件泛型的，任何调用方都可能触发 `Vec2<Int32, Q>(Vec2<Bool, Q>(...))` 这样的调用链，从而在编译时产生不可预期的错误。此约定在泛型构造函数层面是不闭合的。
- **严重程度**：一般
- **改进建议**：在跨类型 Vec 转换构造函数的实现中，对 `Bool` → 数值类型的转换路径增加明确的编译期约束或运行时分支——如果仓颉允许 `Int32(true)` 则无需处理；如果不允许，应在文档中标注跨类型构造对 `Bool` 源类型的限制，并在 §12 验收标准中增加 `Vec2<Bool, Q>` → `Vec2<Int32, Q>` 的编译行为测试。

### P15（一般）— 256 个别名的外部脚本生成策略缺少产出规范和验证要求

- **所在位置**：§3.8 外部脚本生成策略（第 358-359 行）
- **问题描述**：§3.8 推荐使用外部脚本生成 256 个别名定义，但未定义以下关键产出规范：
  - 脚本应输出到哪个文件路径（直接覆盖 `fwd.cj` 还是生成中间文件由编码人员合并）
  - 脚本生成的代码应遵循的格式约定（缩进风格、注释要求、分组顺序）
  - 脚本应纳入版本控制还是作为一次性工具
  - 对脚本生成的产物是否需要专项审查（如逐行核对 256 个别名的正确性）
  - 脚本输入数据的验证要求（家族名→分量类型映射表应作为脚本的数据输入文件，该输入表的正确性谁负责验证）
- **严重程度**：一般
- **改进建议**：① 在 §3.8 中补充脚本产出规范：输出路径约定为 `src/glm/fwd.cj` 的别名区域，格式遵循仓颉编码规范；② 指定脚本输入映射表（16 个家族名、11 个分量类型、4 个精度变体的完整映射）必须经过审查确认；③ 要求脚本生成的代码必须通过 `cjpm build` 编译验证，并通过抽样测试确认至少 3 个别名家族（如 `bvec`、`ivec`、`vec`）的 4 个精度变体均正确实例化。

## 整体质量评价

### 需求响应充分度（本轮重点补充评估）

本设计文档在**范围覆盖上完整响应了用户需求**。通过对照 roadmap §3E/§3G，设计的范围可追溯性表（§8.2）涵盖了全部 14 个条目（11 个核心 GLM 源文件 + 3 个 shim 文件 + 1 个新增 `lib.cj`），每个条目均标注了对应章节和偏离说明。用户需求的核心"把 GLM 1.0.3 用仓颉重新实现"的首轮范围在类型覆盖（Vec1-Vec4、Qualifier、标量/向量别名）和功能覆盖（构造、算术运算、位运算、比较运算）上均无明显遗漏。

**但存在以下充分度方面的不足**：

1. **范围偏离说明不够严谨**：§8.2 对 `setup.hpp` 的偏离说明为"1000+ 行平台/编译器检测宏首轮不迁移，仅迁移约 80-120 行配置常量"。此偏离直接影响到 GLM 运行时行为的平台一致性，但设计未说明如何验证"裁剪后的 setup 在目标平台上是否等价于完整 GLM setup 的首轮行为"。建议补充平台检测宏的替代策略及其对首轮行为的影响评估。

2. **迁移成本评估覆盖了语法差异但未覆盖语义等价性验证**：§11 详细列出了 `&&`/`||` 具名函数迁移、`++`/`--` 迁移、跨类型赋值差异等，但未评估"如何验证迁移后的 Vec 运算结果与 C++ GLM 一致"——§12.2 仅提到"与 C++ 等效代码逐值比较"但缺少具体的测试用例对照方案。

### 事实正确性

- 已在上轮审查基础上修正了多处事实错误（如 `GLM_CONFIG_CLIP_CONTROL` 默认值、`const` 函数支持规则等）。
- 当前版本中存在的主要事实风险集中在**编译器行为假设**上——`const if (T(0) is Float64)` 方案、`@OverflowWrapping` 继承性、`const` 泛型函数中的 `is` 运算符可用性——这些均标注为"待验证"风险项，但没有任何一项已在目标编译器上通过实际原型测试确认。这些假设的累积风险构成了文档最大的事实不确定性。

### 深度和完整性

文档深度充足（1665 行），涵盖了从类型体系设计到迁移成本评估的完整管线。以下是支撑"完整性"维度的定性和定量评估：

- **设计决策完整**：33 项设计决策（D1-D33）覆盖了从 Vec 结构体形态到 `const` 函数声明的所有关键选择，每项附有理由和备选方案。
- **异常场景覆盖良好**：§9 系统性地覆盖了零向量、除零行为、NaN/Infinity 传播、类型转换边界四类场景。
- **迁移顺序清晰**：§8.3 定义了明确的迁移顺序和依赖说明。
- **验证层次完备**：§12 定义了四层验证层次，覆盖编译验证、构造访问验证、运算正确性验证和异常场景验证。

**主要完整性缺口**：
- 「新增发现 P2」——缺少包构建基础设施定义，实施者无法直接从零开始
- 「新增发现 P6」——缺少增量验证节点，早期错误可能被延迟发现
- 「新增发现 P13」——并发设计过于简略

### 落地可行性

- **可直接指导编码** 的程度约 85%——Vec 类型定义、构造函数清单、运算符签名均为可实施的仓颉语法，实施者可直接将 §4 的签名清单转换为代码。
- **主要阻碍项**：缺少 `cjpm.toml` 模板（约 15% 的编码准备度缺口）和增量验证节点（约 20% 的质量保证缺口）。
- **最大落地风险**：三项编译器行为假设（`const if`、`@OverflowWrapping` 继承性、`const` 泛型 `is`）如果在编码阶段全部验证失败，需要同时回退 D29、D30 和 D32/D33 的设计决策，影响范围涵盖 `operator ==`、40 个算术具名函数、`equalExact`、`increment`/`decrement`/`bitwiseNot`，合计影响约 50+ 个函数签名。建议在正式编码启动前，优先完成这三项原型验证，确认核心设计假设成立。
