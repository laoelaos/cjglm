# OOD 设计方案审查报告（v24）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** Qualifier 采用 `interface Qualifier` + 空结构体实现模式是可行的——仓颉泛型系统支持 `where Q <: Qualifier` 约束，空结构体零开销，类型参数 `Q` 可实现编译期类型级区分。与 enum 方案（enum 不可作为泛型类型实参）和 sealed class 方案（引用类型语义）相比，接口+结构体是最优选择。

**[通过]** Vec1~Vec4 定义为四个独立泛型结构体（而非单模板 + 分量数参数化）是合理的——仓颉不支持 C++ 偏特化，无法通过单个 `Vec<N, T, Q>` 模板实现分量数差异化的行为。四个独立类型虽代码重复但实现模式简单直接，与仓颉语言能力匹配。

**[通过]** 运算符重载选择符合仓颉可重载运算符列表（`+`, `-`, `*`, `/`, `%`, `<<`, `>>`, `&`, `|`, `^`, `!`, `==`, `!=`, `[]`）。`&&`, `||`, `~`, `++`, `--` 不在列表中，设计正确选择具名函数替代。

**[通过]** 运算符不可为泛型的限制（§8.1 函数文档）被设计正确识别——跨类型运算符（如 `Vec2<Int32> + Vec2<Int16>`）不支持，调用方需显式转换。

**[通过]** `const` 函数在 struct 上的可用性前提正确——仓颉语言要求 struct 需定义 `const init` 才能有 `const` 实例成员函数（§3.2.9）。Vec 的 `const init` 定义满足此条件。

**[通过]** `const init` 中对 `var` 成员的赋值初始化合法——const 文档 §4.2.3 明确允许 `const init` 内使用 `=` 对实例成员变量赋值。

**[通过]** `extend` 块不可使用 `const` 修饰符——extend 文档 §4.2 列出的允许修饰符中不含 `const`，设计正确地将扩展成员函数声明为非 `const`。

**[通过]** 跨类型 fill-from-Vec1 构造函数的重载冲突问题（D31）已正确识别——同时存在 `init(v: Vec1<T, Q>)` 和 `init<T2, Q2>(v: Vec1<T2, Q2>)` 在 `T2=T` 时重载冲突。设计选择仅保留同类型版本，方案正确。

**[通过]** `operator[]` 的 const 取值形式与赋值形式冲突（D28）已正确识别——仓颉中 const 不构成重载区分依据。设计提供 `componentAt()` 具名 const 函数作为替代，方案可行。

**[通过]** 跨类型构造 `init<T2, Q2>(v: VecN<T2, Q2>)` 中 `T2` 不做紧约束，使用 `T(v.x)` 显式转换——与 D5 宽松泛型约束策略一致，在实例化处报错而非定义处报错，这是正确的仓颉泛型模式。

**[轻微]** `VecN<Bool, Q>.bitwiseNot()` 的行为差异已正确识别并记录为已知差异——C++ GLM 的 `~bvec` 因整数提升导致对任一分量始终返回 `true`，而仓颉 `!Bool` 执行逻辑否定，两种行为完全相反且无法通过类型系统消除。设计文档已明确标注此差异，并在 §11.4 提供了迁移指导。建议在实现时将 `bitwiseNot` 对 `VecN<Bool,Q>` 的行为以注释形式记录在代码中，防止后续维护者误用。

### 2. 标准库与生态覆盖

**[通过]** `std.math.trunc` 用于浮点 `mod` 实现——`std.math` 提供 `trunc(Float64) -> Float64`，设计通过 `Float32`→`Float64` 提升后调用。这是合理的标准库依赖。

**[通过]** `std.math.abs` 用于绝对值运算——`std.math` 提供 `abs` 函数，覆盖 `Float32`/`Float64`/整数类型。设计正确依赖此函数。

**[通过]** `@Derive[Hashable]` 用于自动派生 hashCode——`std.deriving` 提供 `@Derive[Hashable]`。设计正确识别此机制，并指出了浮点 Vec 的已知限制（按位精确哈希与容差 `==` 不一致）。

**[通过]** `@OverflowWrapping` 溢出注解——仓颉内置注解，设计在二元算术运算符、`<<`、`increment`/`decrement` 和 scalar-op-vec 方向包级独立函数上标注，策略清晰完整。

**[通过]** Python 脚本用于别名生成——外部脚本（非仓颉标准库依赖）用于产生 `fwd.cj` 的 256 个别名定义，生命周期和版本控制策略在文档中明确说明。这是一种合理的外围构建辅助手段。

**[通过]** 浮点 `mod` 验证依赖项②（`std.math.trunc` 的 `Float64` 到 `Float32` 降级语义正确性）已在 §10 的验证计划中覆盖。

### 3. 语言特性可行性

**[通过]** 错误处理策略与仓颉能力匹配——断言使用 `if + throw` 模式（首轮 Debug/Release 均保留），整数溢出使用 `@OverflowWrapping` 标注，除零依赖运行时行为。所有策略均在仓颉能力范围内。

**[通过]** 并发设计合理——Vec 为值类型，无共享状态，`@OverflowWrapping` 不影响并发行为。设计正确指出了复合赋值在并发场景下的非原子性，并建议调用方自行加锁。

**[通过]** 模块/包结构符合 cjpm 规范——`package glm.detail` 和 `package glm` 的二级包结构，`cjpm.toml` 配置示例（`output-type = "static"`, `src-dir = "src"`），测试文件组织方式均符合标准。

**[通过]** `public import` 重新导出机制使用正确——`public import` 仅重新导出被导入包中标记为 `public` 的声明。设计已修正 `fwd.cj` 中所有别名为 `public type`，确保 `public import` 可正确暴露。

**[通过]** `const init` 内使用赋值表达式 `=` 初始化 `var` 成员符合规范——const 文档 §4.2.3 明确允许此用法。设计正确利用此特性。

**[通过]** 编译期 `if` 分支抑制行为——设计多处依赖 `const` 函数体内 `if` 表达式的编译期分支选择行为。文档 §34 已统一术语并列出验证计划（依赖点 A~E），每项均有独立回退方案，风险可控。

**[轻微]** `@OverflowWrapping` 在编译器自动生成的复合赋值上的继承行为属于编译器未文档化行为——设计 D30 已识别此风险并提供了完整的备选方案（在已有 20 个扩展成员函数上直接标注 + 调整委托目标）。备选方案无需新增函数定义（函数已在 §4.3 中定义为首轮范围），仅需补充标注。设计采用"验证-决策"模式（原型验证通过则用主路径，否则切换备选），这是恰当的工程实践，不存在阻塞性风险。

### 4. 设计一致性

**[通过]** 各抽象的职责描述清晰——Qualifier（精度/对齐策略）、Vec（数学向量）、Functor（函数映射工具）、ComputeVec（运算策略）、shim（C++ 标准库替代）、标量别名、向量别名各司其职。

**[通过]** 协作关系形成闭环——Vec 结构体依赖 Qualifier 接口作为类型约束，依赖 `ComputeEqual` 实现 `==`，依赖 `scalar_vec_ops.cj` 中的包级函数提供 scalar-op-vec 方向运算。lib.cj 通过 `public import` 暴露公共 API，fwd.cj 通过 `public type` 别名提供便捷命名。

**[通过]** 模块间依赖方向合理——`glm` 单向依赖 `glm.detail`，反向无依赖。`glm.detail` 内部无循环依赖（依赖拓扑在 §2 中明确绘制）。

**[通过]** §8.2 范围可追溯性对照表将 roadmap §3E/§3G 逐项映射到设计章节和文件，新增的 §3G 追溯说明覆盖了依赖闭合性、独立可编译性和最小可行范围参数三项关键结论。

**[通过]** 文档交叉引用一致——§2.1 编码前验证检查清单中的验证项⑩⑪对应 §3.8/§12.2，§4.5 `equalEpsilon` 回退交叉引用 §3.5 双路径策略和 §12.2，§7 D30 备选方案引用 §4.3 已有函数定义。

**[轻微]** `VecN<Bool, Q>` 的行为约定在 §3.2 集中描述后被多个章节引用，这是一种良好的文档组织方式。但 §4.3 中关于 `increment`/`decrement` 在 `VecN<Bool>` 上的行为的描述与 §3.2 有部分重复。建议在编码阶段的实现指南中统一引用 §3.2 而非重复描述。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则——每个文件专注一个抽象（`qualifier.cj` 仅定义 Qualifier 体系，`type_vecN.cj` 仅定义 Vec 类型，`scalar_vec_ops.cj` 仅定义 scalar-op-vec 方向函数）。`lib.cj` 仅做重导出，`fwd.cj` 仅做别名定义。

**[通过]** 抽象层次恰当——设计未过度设计（无不必要的接口抽象，SIMD/矩阵/四元数功能明确推迟到后续轮次），也未设计不足（关键行为契约在 §4 完整列出，构造函数清单、运算符行为、溢出策略、NaN/Inf 边界行为均有约定）。

**[通过]** 可测试性强——§8.1 的测试文件清单（T1~T14）与源文件一一对应，§10 的验证计划覆盖了每项编译器行为依赖的验证方法、通过标准和回退方案，§2.1 检查清单将验证任务前置到编码前。

**[通过]** 可扩展性好——Functor 和 ComputeVec 首轮仅定义不消费，为后续 SIMD 路径预留扩展点。Qualifier 的空结构体+接口模式支持后续轮次引入差异化行为（对齐、SIMD、精度语义）。`lib.cj` 设计支持后续轮次新增 `public import` 行扩展公共 API 面。

**[通过]** D35（`scalar_vec_ops.cj` 的包级函数名独占权约定）是良好的命名空间管理实践，避免了 `public import` 导入歧义。D36（通用范围变更管理流程）为项目提供了变更治理框架。

**[轻微]** 设计文档非常详实（超过 2000 行），涵盖面广。但对于实施者来说，在编码时可能需要频繁在多个章节间跳转以获取完整信息（例如，一个 Vec 的 `==` 运算符行为涉及 §3.2 总体描述、§3.5 ComputeEqual、§4.5 比较运算、§7 D29 const 表达式链、§10 验证依赖等多个章节）。建议在实现阶段创建一份"实现索引表"（按实现顺序列出每个文件需参考的设计章节），减少信息查找开销。此建议不影响设计本身的质量。
