### 发现 B1（已移除）— 256 别名编译性能风险被低估

**已按 v2 质询移除**。原发现基于对 `type` 别名语义的误解（认为别名定义点触发泛型具现化代码生成）。仓颉规范明确规定类型别名"不创建新类型——仅为替代名称"，在定义点不触发代码生成。此发现的核心前提不成立，从报告中移除。

### 发现 B2（中等）— const/extend 不对称的消费者体验影响未充分评估

- **所在位置**：§3.2 Vec 结构体系（`extend` 块描述）、§4.3 算术运算 `const` 修饰符差异依据、§4.5 `equalExact`/`equalEpsilon`、§7 D32/D33
- **问题描述**：由于仓颉 `extend` 块不支持 `const` 修饰符，以下日常操作在 `const` 上下文中不可用：`v.add(s)`、`v.sub(s)`、`v.mul(s)`、`v.div(s)`、`v.mod(s)`、`v.equalExact(other)`、`v.logicalAnd(other)`、`v.logicalOr(other)`、`v.increment()`、`v.decrement()`、`v.bitwiseNot()`。一个从 C++ GLM 迁移的开发者，在看到 `==` 支持 const 上下文的容差比较后，可能自然认为 `v.add(s)` 也支持——直到编译错误才发现限制。
- **严重程度**：中等
- **改进建议**：① 在 §3.2 末尾新增"`const` 上下文中的 API 可用性速查表"，集中列出哪些操作在 `const` 上下文中可用、哪些不可用及替代方案；② 在 §11 迁移成本评估中补充此项不对称对迁移代码审查的工作量影响。

### 发现 B3（一般）— 编译器行为依赖的聚合风险高于单点评估之和

- **所在位置**：§10 设计阶段验证依赖点 A~E 表、§7 D29+D30 组合回退分析、§2.1 编码前验证检查清单
- **问题描述**：设计文档依赖至少 9 项独立编译器行为假设。虽然 D29/D30 的双回退组合分析已存在，但未对三回退或更高阶回退的组合场景做系统性分析。多个回退方案的编码实施工作量存在叠加效应（§11 未评估此叠加成本）。
- **严重程度**：一般
- **改进建议**：① 在 §2.1 编码前验证检查清单中新增"编译器依赖统一验证测试"——编写一个原型程序覆盖依赖点 A~E 及 `@OverflowWrapping` 继承性、`@Derive[Hashable]` 约束传递性，在编码启动前一次性运行；② 在 §11 末尾新增"编译器依赖验证回退成本"子节。

### 发现 B4（提示）— 首轮 C++ 代码裁剪的完整性验证策略不够明确

- **所在位置**：§8.2 范围可追溯性对照表、§8.4 参考实现使用策略
- **问题描述**：设计文档声明 `setup.hpp` 中 1000+ 行仅迁移约 80-120 行配置常量，但未对整个 1000+ 行文件中被裁剪内容的依赖进行系统性"反向验证"。
- **严重程度**：提示
- **改进建议**：在 §8.4 策略 0 前置验证项中新增一项：抽查 3-5 处被排除的 `#define`/`#if` 块，确认首轮其他文件中无间接引用。

## 二、新增设计质量诊断发现

### 发现 C1（严重）— 最小生成脚本模板包含系统性命名错误，直接影响 48/256 个别名

- **所在位置**：§3.8 最小生成脚本示例（第 507 行）
- **问题描述**：提供的 Python 最小脚本模板（`gen_fwd_aliases.py`）中，别名名称生成格式为 `f'{prec}{fam}Vec{dim}'`（第 507 行），其中 `fam` 遍历 `FAMILIES` 字典的键。此格式在所有家族名后硬编码追加"Vec"，但以下 3 个家族的键名**已经包含"Vec"**：
  - `'Vec'`（GLM `vec` 家族）→ 生成 `VecVec{dim}`（如 `VecVec2`），应为 `Vec{dim}`（如 `Vec2`）
  - `'DVec'`（GLM `dvec` 家族）→ 生成 `DVecVec{dim}`（如 `DVecVec2`），应为 `DVec{dim}`（如 `DVec2`）
  - `'FVec'`（GLM `fvec` 家族）→ 生成 `FVecVec{dim}`（如 `FVecVec2`），应为 `FVec{dim}`（如 `FVec2`）

  影响范围：3 家族 × 4 分量数 × 4 精度变体 = **48/256 个别名命名错误**（18.75%）。其余 13 个家族（单字母和数字前缀）不受影响。

  此错误的危害性在于：
  1. 这是设计文档提供的**最小可执行模板**，实施者极有可能直接复制使用而不逐行审查脚本逻辑
  2. §2.1 验证项⑪的抽查方案明确指出抽查 B/BVec、I/IVec、U/UVec、**Vec** ——但即便抽查到 `Vec` 家族，发现 `VecVec2` 错误后实施者可能误以为是命名约定的一部分而非脚本 bug，导致错误固化
  3. 错误是系统性的（非随机拼写），一次抽查发现的错误不会自动提示其他家族（如 `DVec`、`FVec`）也存在相同模式的问题
  4. 与 §3.8 命名约定表（第 409-419 行）和 §11.7 映射表（第 1729-1746 行）中显式列出的正确名称（`Vec4`、`DVec2`）直接矛盾

- **严重程度**：严重 — 影响设计文档提供的核心交付物（脚本模板）的可用性，直接导致生成产物存在系统性错误。修复者可行动——修复方法明确。
- **改进建议**：
  1. 修复脚本中的名称生成逻辑：将 `f'{prec}{fam}Vec{dim}'` 改为条件格式——对于键名包含"Vec"的家族（`Vec`, `DVec`, `FVec`），使用 `f'{prec}{fam}{dim}'`；对于其他家族，保持 `f'{prec}{fam}Vec{dim}'`。推荐方案：
     ```python
     def make_alias_name(prec, fam, dim):
         if fam.endswith('Vec'):
             return f'{prec}{fam}{dim}'
         else:
             return f'{prec}{fam}Vec{dim}'
     ```
  2. 同步更新 §3.8 中的预期 `fwd.cj` 文件结构示例（第 459-470 行），将 `DVec` 家族的正确别名形式（`DVec1`/`DVec2` 等）加入示例
  3. 在 §3.8 末尾或脚本定制指南中增加警告：脚本生成后必须对 `Vec`/`DVec`/`FVec` 三个家族的别名进行完整对比验证
  4. 将 §2.1 验证项⑪的抽查方案从"抽查 4 个家族"补充为**至少覆盖 `Vec` 家族**（因为它是唯一受影响的默认精度高使用频率家族），或者改为正则验证所有 256 个别名的命名模式

### 发现 C2（提示）— `VecN<Bool, Q>` 的模运算不可用性缺少集中声明

- **所在位置**：§3.2 `VecN<Bool, Q>` 行为约定（第 246-252 行）、§4.3 `%` 运算的整数约束（第 718 行）、§4.5 `equalEpsilon` Bool 编译错误说明（第 780 行）
- **问题描述**：`VecN<Bool, Q>` 的模运算（`%`／`v.mod(s)`／`mod(s, v)`）在所有形式下均不可用，但行为分散在三个章节中：
  - §3.2（行为约定）声明算术运算符在 `Bool` Vec 上实例化时报编译错误
  - §4.3（`%` 运算的整数约束）声明 `%` 运算符在浮点 `T` 时报编译错误，但未显式提及 `Bool`
  - §4.3（mod 扩展成员函数 API 影响标注）声明 `v.mod(s)` 仅提供整数路径，浮点 `T` 时报编译错误——`Bool` 既非整数也非浮点，属于"其他类型"落入了未显式覆盖的盲区
  
  最终行为是确定的（`Bool` Vec 上的 `%`/`mod` 均编译错误），但调用方式必须从不同章节的规则组合推断：算术运算符 §3.2 覆盖、`v.mod(s)` 由 §4.3 整数约束覆盖、`mod(s, v)` 由 `isIec559Of<Bool>()` = `false` 分支走 `%` 路径然后报错覆盖。此三处推理路径增加了消费者的认知负担。
- **严重程度**：提示 — 不影响功能性正确性（所有路径均产生正确的编译错误），但影响 API 文档的清晰度。
- **改进建议**：在 §3.2 `VecN<Bool, Q>` 行为约定表中，新增一行"模运算（`%`/`mod`）"并标注"所有形式均不可用——`Bool` 不支持算术运算（D5 延迟检查语义）"。

## 五、整体评价

设计文档在 24 轮迭代后已具备极高的技术完整性和设计一致性。本诊断侧重**落地视角**的补充审查，报告共含 10 项发现（1 项严重、3 项中等、3 项一般、3 项提示）。

**按优先级排序的处理建议**：
1. **C1（严重）**：立即修复脚本模板命名错误，48/256 别名将产生错误名称，必须在实施者使用前修复
2. **B2（中等）**：补充 const/API 可用性速查表，降低实施者认知成本
3. **A1（中等）**：补充迁移成本合计，为项目管理提供决策依据
4. **B3（一般）**：新增统一编译器依赖验证测试，编码启动前一次性运行

B1（别名编译性能风险）已按 v2 质询移除——该发现基于对 `type` 别名语义的误解，不可作为有效诊断保留。
