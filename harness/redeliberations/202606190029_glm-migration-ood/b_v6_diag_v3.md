## 发现的问题

### P1（严重）— 文档版本标识自相矛盾，实施者无法确定当前版本（合并 P1+P10）

- **所在位置**：文档标题（第 1 行）、文件路径名、修订说明序列（第 1435-1665 行）
- **问题描述**：文档标题声明为 v5，文件路径名为 a_v6_design_v2.md，末尾最新修订说明标记为 v6。三重版本标识不一致。修订说明序列混乱 — 从 v7 到 v15 跳回 v2 再到 v6，实施者无法据此确定当前版本。此问题在前轮已指出，本轮仅追加修订说明未系统性清理。
- **严重程度**：严重
- **改进建议**：① 将文档标题统一为 v6；② 整理修订说明按时间正序排列，历史内容移入附录；③ 文档顶部增加版本变更摘要行。

### P2（严重）— 缺少包构建基础设施定义，无法直接指导编码启动

- **所在位置**：§2 模块划分、§8 迁移文件清单、全文
- **问题描述**：设计详细描述了文件组织和包结构，但未定义 cjpm.toml 内容、编译入口和构建命令、项目初始化脚手架步骤、测试运行器配置。实施者无法从零开始直接执行第一个编译步骤。
- **严重程度**：严重
- **改进建议**：在 §2 或 §8 中补充前置准备工作子节，提供最小 cjpm.toml 模板和初始目录结构示例。

### P3（严重）— D32 声明 40 个算术函数为 const，但 const 兼容性验证计划（D29）未覆盖

- **所在位置**：§7 D32（第 935-939 行）、§7 D29（第 887-901 行）、§10 验证要求表
- **问题描述**：D32 将 add/sub/mul/div/mod 共 40 个函数统一声明为 const 函数。但 D29 的 const 表达式链验证仅针对 == 运算符，不覆盖算术运算的 const 兼容性。若目标类型 T 的算术运算符在 const 函数体内不可用，则 40 个函数的 const 声明将同时失效。
- **严重程度**：严重
- **改进建议**：在 §7 D32 中补充 const 兼容性验证要求；在 §10 中新增算术运算 const 兼容性验证；定义 D32 独立回退方案。

## 新增发现（本轮）

### P16（严重）— mod 函数的 const if 分支依赖编译期死代码消除，行为未经验证

- **所在位置**：§4.3 mod 函数实现描述（第 529 行）、§10 浮点 mod 验证表依赖项④（第 1098 行）
- **问题描述**：mod 的浮点/整数双路径通过 const if 选择分支 — 整数路径使用 % 运算符，浮点路径使用算术恒等式。此设计假设 const if 能完全抑制非选择分支的编译检查。若仓颉 const if 仅做运行期优化，则整数路径的 T % T 对浮点 T 将产生编译错误。§10 验证仅覆盖 isIec559Of 返回值正确性，未验证非选择分支的编译抑制行为。
- **严重程度**：严重
- **改进建议**：① 新增验证依赖，编写原型测试分别以 Int32 和 Float32 实例化验证 const if 编译抑制行为；② 定义回退方案为独立具名函数；③ 将此验证升级为与 D29/D30 同优先级的编码前原型验证。

## 整体质量评价

### 需求响应充分度（本轮系统性补充评估）

本设计的范围覆盖已通过自身的可追溯性机制（§8.2）建立了完整的 roadmap 映射 — 14 个条目每条均标注了对应章节和偏离说明。但存在以下不足：

1. **缺乏对参考实现的独立交叉校验**。由于 references/glm-1.0.3/ 路径当前为空，与 C++ 源文件的 API 映射比对无法在本轮完成。建议在参考实现就位后补充抽样验证（选取 3-5 个核心类型比对应仓颉签名）。

2. **setup.hpp 平台检测宏裁剪影响未评估**。1000+ 行宏首轮不迁移，但未说明对运行时行为的等价性影响。

3. **语义等价性验证策略不具体**。§12.2 仅说与 C++ 逐值比较，未给出测试用例对照方案和容差阈值选取依据。

### 事实正确性

主要事实风险集中在编译器行为假设上（四项）：T(0) const 构造合法性、const if 分支编译抑制行为、@OverflowWrapping 继承性、is 运算符在 const 泛型函数中的可用性。均未在目标编译器上通过原型测试确认。

### 深度和完整性

文档深度充足（1665 行），33 项设计决策覆盖所有关键选择。主要缺口：P2（构建基础设施）、P6（增量验证节点）、P13（并发设计）、P16（const if 分支抑制验证）。

### 落地可行性

可直接指导编码程度约 85%。主要阻碍项为 cjpm.toml 缺失和增量验证节点缺失。最大落地风险为 P16 — const if 编译抑制行为若验证失败，mod 双路径策略完全不可行，与 D29/D30 级联影响约 50+ 函数签名和 5-8 人天额外工作量。
