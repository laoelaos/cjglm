# 质量审查诊断报告 — 第 5 轮

**待审查产出**：`a_v5_copy_from_v4.md`（GLM 1.0.3 仓颉迁移首轮 OOD 设计方案 v5）
**审查视角**：落地就绪度 / 需求响应充分度 / 事实正确性 / 深度与完整性
**审查范围**：内部审议（组件A）未充分覆盖的维度，避免重复验证内部审议已确认的技术可行性等维度。

---

## 发现的问题

### P1（严重 / 事实错误 — 修订未应用到正文）

**§3.2 Vec 结构体系（第 154 行）中 `length()` 函数的声明缺少 `const` 修饰符。**

v5 修订说明明确记载："§3.2 签名修正为 `const public static func length(): Int64`"，但正文第 154 行仍为 `public static func length(): Int64`，修正**未实际应用**。编码人员将据此签名编写代码，导致 `length()` 无法在编译期上下文中使用，与 §3.2 声称的"返回编译期常量"角色自相矛盾。

**位置**：§3.2，第 154 行
**严重程度**：严重
**改进建议**：将第 154 行 `public static func length(): Int64` 修正为 `const public static func length(): Int64`，与 v5 修订说明一致。

---

### P2（严重 / 修订未应用到正文）

**§7 设计决策节缺少 D32（add/sub/mul/div/mod 统一声明为 const 函数）。**

v5 修订说明明确记载："§7 新增 D32 设计决策记录此选择理由"，但 §7 设计决策节（D1~D31）之后并未出现 D32。该决策是 `mod` 函数使用 `const if` 的语法前提，缺失将导致编码人员无法追溯"为何这 5 个运算要统一声明为 const"的设计依据。

**位置**：§7，D31 之后
**严重程度**：严重
**改进建议**：将 v5 修订说明中的 D32 设计决策文本移入 §7 正文，紧跟 D31 之后，并确保全文编号连续。

---

### P3（严重 / 逻辑矛盾）

**§7 D13 声明"浮点取模的具名函数方案不属于首轮范围"，但 §4.3 已明确将 `mod` 列为首轮具名函数（5 运算之一），并给出了完整的浮点取模实现策略（`x - y * trunc(x / y)`）。**

两者互相矛盾：D13 宣称"不属于首轮范围"，但 §4.3 属于首轮范围且已提供实现方案。造成编码人员无法判断 `mod(float_vec, scalar)` 到底是否为首轮应实现的功能。

**位置**：§7 D13（第 779 行） vs §4.3（第 527 行）
**严重程度**：严重
**改进建议**：将 D13 中"对于需要浮点取模的场景，使用具名函数方案（不属于首轮范围）"改写为"对于需要浮点取模的场景，使用具名函数 `mod`（属于首轮范围，实现在 `compute_vector_decl.cj` 中）"，消除自相矛盾。

---

### P4（中等 / 深度不足 — 缺失组合回退场景分析）

**§7 D29（const 路径验证）和 §7 D30（@OverflowWrapping 继承性验证）各自定义了独立的回退方案，但未考虑两者同时失败的组合场景。**

若 D29 回退到精确比较且 D30 回退到具名函数模式，`==` 运算符失去容差比较能力、复合赋值运算符的溢出语义需要通过不同路径维护，两者的组合交互可能产生不可预见的编译或运行时行为。文档未对此组合场景做任何分析或约定。

**位置**：§7 D29（第 885-908 行）、§7 D30（第 901-909 行）
**严重程度**：中等
**改进建议**：在 §7 D30 末尾或在 §10 验证要求中新增**组合回退分析**段落，评估两种情况同时发生时的交互影响，明确组合场景下的行为约定。

---

### P5（中等 / 设计深度不足 — 缺少对称性理由）

**`increment()` / `decrement()` / `bitwiseNot()` 未被声明为 `const` 函数，但与 `add()`/`sub()` 等（已声明为 `const`）一样是纯函数（不修改 `this`，返回新向量）。**

D32 给出 add/sub/mul/div/mod 统一声明为 `const` 的理由为："① 与 `mod` 的 API 签名一致；② 允许在 const 表达式和 const 变量初始化等编译期上下文中调用；③ 为后续轮次预留 const 路径扩展点"。`increment()`/`decrement()`/`bitwiseNot()` 同样满足这三条理由，但设计文档未解释为何不将它们也声明为 `const`。

**位置**：§4.3 `++`/`--` 替代方案段落、§4.4 `bitwiseNot()` 代码示例
**严重程度**：中等
**改进建议**：在 §7 中补充设计决策或说明，统一约定 `increment()`/`decrement()`/`bitwiseNot()` 是否声明为 `const`，并给出理由（无论采用或拒绝），确保与 D32 的决策一致性。

---

### P6（中等 / 需求响应充分度 — 别名命名风格长期维护成本未评估）

**§3.8 定义的大写驼峰（PascalCase）别名（`BVec2`、`IVec3` 等）虽已在 §11.7 评估了迁移成本，但未评估对**新代码开发**的影响——下游使用者需同时在 C++ GLM 原版文档（小写名）和仓颉代码（大写驼峰）之间做 mental mapping。**

当开发者查阅 C++ GLM 文档（使用 `bvec2`、`ivec3`）并对照仓颉代码（`BVec2`、`IVec3`）时，大小写差异是持续的认知负担。§11.7 仅评估了**迁移**（从 C++ 到仓颉）的一次性成本，未评估**长期使用**中 API 名称不一致的维护成本。

**位置**：§3.8 命名约定（第 315-323 行）、§11.7（第 1239-1280 行）
**严重程度**：中等
**改进建议**：在 §3.8 命名约定中新增一段简要说明该命名差异的长期影响，或在 §11 中新增"持续使用成本"子节评估。若后续轮次考虑通过宏/脚本提供小写别名的兼容层，需在此处记录备选方向。

---

### P7（一般 / 事实正确性 — 约束条件描述不准确）

**§3.5 关于 `epsilonOf<T>()` 的兼容性策略仍使用"采取以下方案之一"的表述，但仅列出方案 A（唯一策略）。**

迭代第 3 轮问题 #4 已确认方案 B（`NumericLimits<Bool>` 空壳特化）不可行，当前设计已废除。但文字残留"之一"暗示应有多个选项存在。

**位置**：§3.5（第 254-255 行）
**严重程度**：一般
**改进建议**：将"采取以下方案之一"改为"采用以下唯一策略"，移除"方案 A"编号或明确说明方案 B 已因仓颉不支持偏特化而被废弃。

---

### P8（一般 / 深度不足 — `@OverflowWrapping` 与 `const` 函数的交互未分析）

**若算术运算符（`+`、`-`、`*`、`/`、`%`）被声明为 `const` 函数（D32）并标注 `@OverflowWrapping`，这两个修饰符在仓颉编译器中的交互行为（`const` + overflow wrapping 共存于同一函数）未经验证。**

文档分别验证了 `@OverflowWrapping` 的继承性（D30）和 `const` 路径兼容性（D29），但未建立原型测试验证**两个修饰符共存**的场景——即 `const func add(v: Vec2<T,Q>, s: T)` 上同时标注 `@OverflowWrapping` 是否影响 `const` 求值能力，或 `const` 修饰符是否改变 overflow 行为。

**位置**：§10 设计阶段验证要求（第 1060-1064 行），§7 D30（第 901-909 行），§7 D32（仅存在于修订说明）
**严重程度**：一般
**改进建议**：在 §10 设计阶段验证要求中新增一项验证依赖："`@OverflowWrapping` 标注与 `const` 函数修饰符的共存兼容性"，编写原型测试确认 `const` + `@OverflowWrapping` 在同一函数上可同时通过编译且语义正确。

**D25、D26、D27 三个设计决策正文位于"新增设计决策（v8）"修订说明节中，虽已标注"移入 §7 主设计决策节"，但查看 §7 正文（D1~D31）确认它们实际已存在于正确位置。** 但 **D32 仍未纳入正文**（同 P2）。

仅 D32 未纳入正文，D25~D27 已纳入。此问题与 P2 实质相同，单独列出以准确描述范围。

**位置**：§7 D31 之后
**严重程度**：一般
**改进建议**：同 P2——将 D32 移入 §7 正文。

## 整体质量评价

本产出经过多轮审议迭代（从 v15 初始版经 v7→v15→v2→v3→v4→v5 多个修订循环），整体质量较高：架构设计合理、行为约定详尽、迁移成本评估全面、验证计划完整。绝大部分内部审议阶段发现的问题已得到妥善修复。

但仍存在以下系统性问题：
1. **修订未落地**：P1（`length()` 缺 `const`）和 P2（D32 缺失）说明存在"修订说明写了但正文未改"的执行断层，降低了文档的编码就绪可信度。
2. **遗留自相矛盾**：P3（D13 vs §4.3）是跨子节的不一致，属于多轮迭代中局部修改未全局同步的典型遗留。
3. **组合场景分析不足**：P4 和 P8 反映文档对"两个独立回退/验证机制同时触发"的交互场景缺乏系统性分析。

**建议优先级**：P1、P2、P3 必须在编码阶段启动前修复（影响源代码正确性）；P4、P5、P6 建议本轮修复（提升设计完备性）；P7、P8、P9 可在编码阶段同步修复。
