## 发现 1（中等）— §11.5 与 §11.9 存在结构重叠，迁移工作量存在重复计算风险

**问题描述**：§11.5 "`Inf == Inf` 容差比较路径行为差异" 与 §11.9 "`equalExact` 迁移模式" 描述的是同一组行为差异（容差路径下 `==` 的精度行为与 C++ GLM `==` 不同），但以两个独立子节呈现，各自定义了覆盖范围重叠的搜索替换规则和人工审查项。

**所在位置**：§11.5（第 1541-1573 行）、§11.9（第 1689-1725 行）

**重叠分析**：

| 重叠维度 | §11.5 内容 | §11.9 内容 |
|---------|-----------|-----------|
| 差异描述 | `Inf == Inf` 在容差路径下返回 `false` | 完整行为差异对照表（含有限值微差、Inf、NaN） |
| 替换目标 | `v.equalExact(VecN(Inf))` | `v1.equalExact(v2)` |
| 搜索规则 | 浮点 Vec `==` 表达式，含 Inf/NaN | 含 Inf/NaN 的 `==` + 精确相等语义要求 |
| 工作量评估 | **低-中**（含人工审查） | **0.5-1 人天**（含人工审查） |

两份工作量评估都包含了"人工审查浮点 Vec `==` 使用上下文"，未说明两份评估是累加还是互补。实施者如果按顺序执行，可能对同一批代码进行两次审查。

**严重程度**：中等

**改进建议**：将 §11.5 合并为 §11.9 的一个子节（如 §11.9.1 "Inf/NaN 边界行为"），统一迁移指引和工作量评估。或在 §11.9 开头注明"§11.5 定义的 `Inf == Inf` 差异为本节所述行为差异的子集，本节迁移模式已覆盖 §11.5 场景，无需重复审查"。

## 发现 2（一般）— §4.3 残留 "const if" 术语，与 D34 约定不一致

**问题描述**：§7 D34（第 1103-1109 行）明确要求全文将 `const if (...)` 统一改写为 `if (...)`，使用 "`const` 函数体内的 `if` 表达式"或"编译期 `if` 分支"表述。但 §4.3（第 646 行）仍使用 "`const if`" 作为单独术语：

> "仓颉中 `const if` 要求所在函数必须为 `const` 函数"

经核实，`if` 表达式在 `const` 函数体内的编译期分支抑制行为已确认，但 D34 要求消除 "const if" 作为独立说法。此残留可能误导实施者编写不存在的 `const if` 关键字。

**所在位置**：§4.3（第 646 行）

**严重程度**：一般

**改进建议**：将 "仓颉中 `const if` 要求" 改为 "仓颉中编译期分支 `if` 要求" 或 "仓颉中 `const` 函数体内的 `if` 表达式要求"，与 D34 一致。全文 grep 确认无其他残余。

## 发现 3（一般）— §11.4 引用了未在 §4 中定义的 `logicalNot` 函数

**问题描述**：§11.4 迁移影响表（第 1529 行）给出了 `~bvec` 的仓颉替换选项：

> "若需保持 C++ 行为，应替换为 `VecN<Bool, Q>(true)`（全 true）或按需使用 `logicalNot`"

但 §4.5 布尔逻辑运算中仅定义了 `logicalAnd`、`logicalOr` 和一元 `!` 运算符，未定义 `logicalNot` 具名函数。`!` 运算符可重载（已在 §4.5 中确认），因此 `!bvec` 是可行的，但迁移表中的 "`logicalNot`" 在没有定义的情况下无法使用。

**所在位置**：§11.4（第 1529-1530 行）

**严重程度**：一般

**改进建议**：将 "按需使用 `logicalNot`" 改为 "直接使用 `!` 运算符（已重载）"，或同时在 §4.5 中增加 `logicalNot` 具名函数的定义以保持与 `logicalAnd`/`logicalOr` 的 API 对称性。

## 发现 4（一般）— D31 中 Vec1 跨类型 fill-from-Vec1 的豁免理由未明确

**问题描述**：§7 D31（第 1073-1079 行）声明 "Vec2~Vec4 的 fill-from-Vec1 构造器仅提供同类型版本"，但未明确解释 Vec1 为什么可以保留跨类型版本（`public init<T2, Q2>(v: Vec1<T2, Q2>)`）。实施者需要自行推断原因——Vec1 的跨类型 fill-from-Vec1 因只有一个分量，本质上是分量级类型转换而非广播填充，因此不受重载冲突影响——但该推断无文档支撑。

**所在位置**：§7 D31（第 1073-1079 行）、§4.1 Vec1 构造函数清单（第 495 行）

**严重程度**：一般

**改进建议**：在 D31 末尾追加一句说明，例如："Vec1 的跨类型 fill-from-Vec1（`init<T2, Q2>(v: Vec1<T2, Q2>)`）不受此限制——Vec1 仅单个分量，跨类型版本实质是分量级类型转换而非广播填充，与 `const init(x: T)` 的参数列表（`T` vs `Vec1<T2, Q2>`）不构成重载冲突。"

## 发现 5（轻微）— §8.2 S3 偏离理由中 VArray 引用的相关性存疑

**问题描述**：§8.2 范围可追溯性对照表 S3 行（第 1177 行）解释 `countof` 偏离的理由时提到：

> "仓颉中无等价的 C 数组语义——VArray 的元素类型和长度在类型中编码，通过 `size` 属性即可获取编译期长度"

VArray（`VArray<T, $N>`）是仓颉标准库的类型，GLM 的 C++ 代码中不存在 VArray。将 VArray 作为 countof 的替代方案引入解释——VArray 并非 countof 的替代品，而是仓颉中一种独立的数据结构。正确的偏离理由应更简洁："Cangjie 无 C 数组类型，因此 GLM 中基于 C 数组的 countof 宏无映射对象。"

虽然语义正确（VArray 的 `size` 确实可获取编译期长度），但将 VArray 引入偏离理由造成"GLM 中原本使用 countof 的场景应改用 VArray"的误导印象，而实际上 countof 在 Cangjie 中无任何使用场景。

**所在位置**：§8.2（第 1177 行）

**严重程度**：轻微

**改进建议**：简化偏离理由为："Cangjie 无 C 数组类型——GLM 中 countof 宏基于 `sizeof(array)/sizeof(array[0])` 计算 C 数组元素个数，此模式在 Cangjie 中无对应映射。所有原使用场景在迁移时删除。" 移除 VArray 引用。

## 整体评估

**需求响应充分度**：充分。文档覆盖了 `docs/01_roadmap.md` 中 §3E/§3G 定义的首轮全部范围（基础设施、Vec1~Vec4、标量别名、256 个向量别名），并主动标注了与 roadmap 的差异（如 GLM_CONFIG_CLIP_CONTROL 默认值修正、新增 `scalar_vec_ops.cj`/`lib.cj`）。§8.2 可追溯性对照表提供了逐项映射。

**深度和完整性**：良好。经过 17 轮迭代，设计已覆盖类型体系、行为契约、构造函数清单、运算符策略、溢出标注、验证计划、迁移成本评估和验收标准等完整维度。结构体定义可直接转换为仓颉代码，迁移顺序可指导实施流程。

**编码可指导性**：强。§2.1 编码前验证检查清单、§4 完整构造函数签名、§8.3 增量迁移顺序、§10 验证计划和 §12 验收标准共同构成一份可直接指导编码启动的方案。

**下游消费者视角**：§2 公共 API 面设计、§3.8 别名命名约定和 256 个别名全量定义提供了完整的外部接口描述。缺少少量使用示例（可后续补充）。
