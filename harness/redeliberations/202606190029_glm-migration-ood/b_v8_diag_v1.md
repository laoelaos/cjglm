# 质量审查报告 — GLM 1.0.3 仓颉迁移首轮 OOD 设计方案 v8

## 审查范围与视角

- **审查对象**：`a_v8_design_v2.md`（v8 — 审查迭代修订版 v2）
- **审查维度**：需求响应充分度、事实准确性、深度与完整性（侧重内部审议未充分覆盖的维度）
- **落地视角**：设计是否可直接指导编码实现、接口定义是否足以支持下游消费者、异常场景和边界条件是否已考虑

## 发现问题

### P1. 严重 — 扩展成员函数 add/sub/mul/div/mod（vec-op-scalar 方向）缺少 @OverflowWrapping 标注策略

**问题描述**：§4.3 声明扩展成员函数 `v.add(s)` 等功能上等价于 `v + s` 二元运算符。§4.6 定义的 @OverflowWrapping 标注分布仅覆盖二元算术运算符（20 处）、`<<`（4 处）、`increment()`/`decrement()`（8 处），**未包含**扩展成员函数 add/sub/mul/div/mod（20 个函数）。若实现者按其描述直接编码（如 `Vec2(this.x + s, this.y + s)`），这些函数将不具有 @OverflowWrapping 保护，对于整数分量 Vec 在溢出时抛异常而非 wrapping，与二元运算符行为不一致。文档未指定这些扩展成员函数的实现模式应委托给已标注的运算符（如 `this + VecN(s)`）还是独立标注 @OverflowWrapping，实施者面临歧义。

**所在位置**：§4.3 具名函数定义段落（第 605-621 行）、§4.6 @OverflowWrapping 标注分布表（第 693-704 行）

**严重程度**：严重

**改进建议**：选择以下之一并在文档中明确：
  - A. 在 §4.6 的标注分布表中将 20 个扩展成员函数（add/sub/mul/div/mod 的 vec-op-scalar 方向）纳入 @OverflowWrapping 标注范围，每个函数独立标注；
  - B. 在 §4.3 中明确扩展成员函数的实现模式——委托给已标注 @OverflowWrapping 的二元运算符（返回 `this + VecN(s)` 而非逐分量计算 `this.x + s`），从而继承 wrapping 语义。

### P2. 中等 — Vec1 构造函数接口的不对称性缺少说明

**问题描述**：§4.1 中 Vec1 构造函数清单仅列出 `const init(x: T)`，无 `public init(scalar: T)`；而 Vec2~Vec4 同时提供 `public init(scalar: T)`（标量填充）和 `const init(x: T, y: T, ...)`（逐分量）。原因是 Vec1 的 `const init(x: T)` 与 `init(scalar: T)` 参数列表均为单 `T`，`const` 不构成重载区分依据，无法共存。文档未在任何位置解释此不对称性的原因。下游消费者看到不同的构造接口模式（Vec1 无 `init(scalar: T)`），可能怀疑设计遗漏或不确定正确的标量构造调用方式。

**所在位置**：§4.1 Vec1 构造函数清单（第 466-477 行）、§4.1 Vec2 构造函数清单（第 479-497 行）、§7 D11（第 856-860 行）

**严重程度**：中等

**改进建议**：在 §4.1 Vec1 构造函数清单的注释中或在 §7 D11 的决策描述中补充："Vec1 的 `const init(x: T)` 同时承担逐分量构造和标量填充构造两种角色。由于 `public init(scalar: T)` 与 `const init(x: T)` 参数列表完全相同，`const` 修饰符不构成重载区分依据（const README §3.2 规则 7），两者无法共存，因此仅提供 `const init`。Vec1 的调用方使用 `Vec1<T, Q>(val)` 构造即同时覆盖两种语义。"

---

### P3. 中等 — `countof` 辅助函数的范围跟踪遗漏

**问题描述**：roadmap §3A.1 明确将 `countof` 辅助函数列为 `setup.hpp`（对应 `setup.cj`）的迁移内容之一。设计文档在 §3.6 决定"不单独定义 countof 函数——VArray<T, $N> 的 $N 必须是固定数值字面量"，但 §8.2 范围可追溯性对照表（序号 1, setup.cj）的偏离说明标注为"无偏离"。这是一个显式的内容完备性缺口——文档对 roadmap 期望项的排除未在范围跟踪中体现。

**所在位置**：§8.2 范围可追溯性对照表第 1 行（第 1105 行）、§3.6 shim_cstddef.cj 描述（第 347 行）

**严重程度**：中等

**改进建议**：在 §8.2 序号 1（setup.cj）的偏离说明中补充："`countof` 辅助函数因仓颉 `VArray` 约束（`$N` 须为固定字面量，不可声明为值泛型参数）而不映射为首轮范围。若下游需要等价功能，可通过引用 `VArray.size` 实现。已在 §3.6 shim_cstddef.cj 中说明理由。"

---

### P4. 中等 — 跨类型构造函数（`init<T2, Q2>(v: VecN<T2, Q2>)`）的 T2 有效性范围未集中归档

**问题描述**：§4.1 定义的所有跨类型构造函数签名（如 `public init<T2, Q2>(v: Vec2<T2, Q2>) where Q2 <: Qualifier`）对 `T2` 无任何约束。§9.4 的转换边界表中已约定 `Bool`→数值转换的行为（方案 B 运行时兜底），但 §4.1 的构造函数清单本身未做任何类型安全的边界标注。实施者在编写构造函数实现体时，需要交叉查阅 §4.1（构造函数签名）和 §9.4（类型转换边界）两处才能了解各 `T`/`T2` 组合的编译时行为。未归档的 T2 有效性范围增加了编码阶段遗漏 Bool→数值转换分支处理的风险。

**所在位置**：§4.1 跨类型构造函数签名（第 472、486、506、536 行等）、§9.4 类型转换边界（第 1196 行）

**严重程度**：中等

**改进建议**：在 §4.1 跨类型构造函数的统一描述末尾添加约束矩阵，明确列出跨类型构造函数对 `T2` 的编译时行为：
  - `T2` 为数值类型且 `T` 为数值类型 → 编译通过（数值构造函数转换）
  - `T2` 为 `Bool` 且 `T` 为 `Bool` → 编译通过（同类型精确构造）
  - `T2` 为 `Bool` 且 `T` 为数值类型 → 运行时分支（方案 B，见 §9.4）
  - `T2` 为数值类型且 `T` 为 `Bool` → 编译错误（`Bool(Int32)` 不可构造）
  
或至少在 §4.1 中全文引用 §9.4 的类型转换边界约定作为实现指引。

### P5. 一般 — `GLM_CONFIG_CLIP_CONTROL` 的默认值与 roadmap 参考值存在差异

**问题描述**：roadmap 中 `GLM_CONFIG_CLIP_CONTROL` 的推荐默认值为 `0x02`（标注为 GLM_CLIP_CONTROL_RH_NO），但设计文档 §7 D6 表格中的值为 `0x0A`（十进制 10，对应 `RH_BIT(8) | NO_BIT(2) = 0x0A`）。验证参考实现确认：`GLM_CLIP_CONTROL_RH_NO = RH_BIT(8) | NO_BIT(2) = 0x0A`，设计文档的值正确，roadmap 的值错误。此差异看似设计无误，但编码阶段使用 roadmap 作为验收参照时可能引发不必要的验证争议，且同一项目中两个口径给出不同数值造成认知歧义。

**所在位置**：§7 D6 配置常量表 GLM_CONFIG_CLIP_CONTROL 行（第 811 行）

**严重程度**：一般

**改进建议**：在 §7 D6 中标注该默认值与 roadmap 的差异及修正说明，确认设计文档的值已通过参考实现验证。同时建议同步修正 roadmap 中的对应值。

---

### P6. 一般 — D30 回退方案的工作量评估与文档已有设计不一致

**问题描述**：§11.6 评估 D30 回退方案的额外工作量为"新增 20 个具名函数，约 1-2 人天"。但 §4.3 已将 add/sub/mul/div/mod 的 vec-op-scalar 方向扩展成员函数定义为首轮范围（非回退方案新增）。D30 回退实际需要做的是：(1) 为这 20 个已有函数补充 @OverflowWrapping 标注；(2) 将 20 个二元运算符改为委托给这些具名函数。工作量评估中的"新增"表述与文档已有设计矛盾，可能导致编码阶段人力规划失准（实际工作量低于评估）。

**所在位置**：§11.6 备选方案变更成本表（第 1462-1478 行）

**严重程度**：一般

**改进建议**：将 §11.6 变更成本表中"新增 Vec 算术具名函数"改为"为已有 20 个扩展成员函数补充 @OverflowWrapping 标注 + 修改 20 个二元运算符委托目标"，工作量重新评估为 0.5-1 人天。同时增加一项备注：若回退方案选择在 extend 块中定义新的包装函数（而非复用已有函数），则评估为原 1-2 人天，此场景需额外新增函数名约定避免与现有 add/sub 等重名。

## 整体评价

文档在 v8 版本中已经过高密度迭代审查，大部分已知的严重和中等问题（const 扩展成员函数语法不可行、const if 抑制依赖验证、构造函数跨 Q 重载解析风险、<< 默认溢出策略修正等）均已完成修复。当前版本的核心质量状态：

1. **需求响应充分度**：首轮迁移范围全量映射（基础设施 + 4 个 Vec 类型 + 3 个辅助文件 + 256 个别名 + shim 层 + 公共 API 面），与 roadmap §3E/§3G 一致。编号对应工作完整。唯一缺口是 `countof` 的范围跟踪未在 scope tracking 中标注为偏离（P3），但功能上已给出替代方案说明。

2. **事实准确性与逻辑一致性**：主要失配在于扩展成员函数的 @OverflowWrapping 标注策略缺失（P1, 严重）——这是编码前必须解决的设计遗留问题。其余问题（Vec1 构造不对称性 P2、跨类型构造 T2 有效性范围 P4）为深度不足类的完整性问题，不影响架构正确性。

3. **深度与完整性**：接口定义、异常场景、边界条件、迁移成本评估等维度均达到可指导编码启动的水平。当前 P1 是影响编码启动的最关键问题——若未解决，实施者将面临 `v.add(s)` 在整数溢出时行为不符合预期的 bug。

建议优先解决 P1（扩展成员函数 @OverflowWrapping 策略裁决），P3 和 P5 含与 roadmap 的交叉引用问题，建议同时对 roadmap 做同步修订。
