## 问题 1（严重）：`const if` 在仓颉语言中并非已确认的语法结构，整个设计对其有根本性依赖

**问题描述**：文档全文在 `ComputeEqual.callConst`（§3.5）、`isIec559Of<T>()`（§3.6）、`mod` 双路径（§4.3）、`const if` 抑制验证（§10）等关键位置将 `const if` 当作已知的仓颉关键字使用（如 `const if (isIec559Of<T>()) { ... } else { ... }`）。但经查阅仓颉语言 const 特性文档（skills/cangjie-lang-features/const/README.md）、基本数据类型文档、函数文档，**全文未出现 `const if` 作为独立关键字的任何定义或示例**。const 文档 §2.8 仅确认 `if` 可作为 const 表达式出现在 `const` 函数体内，并未定义 `const if` 作为独立语法结构。

**所在位置**：§3.5 ComputeEqual.callConst（`const if (isIec559Of<T>())`）、§3.6 isIec559Of 实现（`const if (T(0) is Float64 || T(0) is Float32)`）、§4.3 mod 双路径（`const if (isIec559Of<T>())`）、§10 const if 抑制验证（全文）

**严重程度**：严重

**改进建议**：
- 若 `const if` 是文档作者对"`const` 函数中的 `if` 表达式"的简写，则全文须将 `const if (...)` 改写为 `if (...)`，避免误导实施者编写不存在的关键字
- 若 `const if` 是确实存在的仓颉语法结构，则须在 §7 新增设计决策 D34 记录此依赖的验证状态和来源依据
- 无论哪种情况，§10 的 `const if` 抑制验证原型测试中的 `const if` 关键字须与实际可编译代码保持一致
- 此问题影响面覆盖 §3.5 容差比较路径、§3.6 编译期类型检测、§4.3 mod 浮点路径等核心设计要素的可行性

## 问题 2（中等）：`public import glm.detail.{ add, sub, mul, div, mod }` 存在潜在的命名冲突——`glm.detail` 包的其他文件未来也可能导出同名函数

**问题描述**：§2 lib.cj 使用 `public import glm.detail.{ add, sub, mul, div, mod }` 从 `glm.detail` 包导入算术函数。这些函数当前定义在 `scalar_vec_ops.cj` 中（其 `package` 声明为 `glm.detail`）。但 `glm.detail` 包由 12+ 个文件组成，若未来其他文件（如后续轮次新增的 `type_vecN.cj` 扩展或 `compute_vector_decl.cj`）也在 `glm.detail` 包中定义了同名函数，`public import` 将因 `glm.detail` 范围内的重名导致导入歧义。当前范围无此问题，但设计未说明如何防范后续轮次的命名冲突。

**所在位置**：§2 lib.cj、§4.3 具名函数定义位置

**严重程度**：中等

**改进建议**：
- 在 §2 lib.cj 的 `public import` 注解中补充命名空间占用声明：约定 `glm.detail` 包中只有 `scalar_vec_ops.cj` 文件定义 `add`/`sub`/`mul`/`div`/`mod` 包级函数，其他文件不得在 `glm.detail` 包中定义同名包级函数
- 或在 §7 新增设计决策，明确约定 `scalar_vec_ops.cj` 对 `add`/`sub`/`mul`/`div`/`mod` 名称的独占权

## 问题 3（中等）：§11 迁移成本评估缺少 `equalExact` 迁移模式的系统性覆盖

**问题描述**：§11 评估了 `&&/||`（§11.1）、`++/--`（§11.2）、跨类型赋值（§11.3）、`bitwiseNot`（§11.4）、`Inf==Inf`（§11.5）、`@OverflowWrapping`（§11.6）、别名命名（§11.7）、跨 Q 赋值（§11.8）共 8 类迁移成本。但未包含 `equalExact` 的迁移模式。C++ GLM 中没有 `equalExact`——用户直接使用 `==` 进行 IEEE 754 精确比较。迁移到仓颉后，如果 `==` 默认使用容差比较（§4.5），则原 C++ 代码中依赖精确 `==` 语义的场景（如 `Inf == Inf`、`NaN != NaN` 检测、值精确匹配用于键值等）需要被识别并替换为 `equalExact()`。此迁移模式与 §11.5（仅覆盖 Inf/NaN）不同，涵盖的是有限值范围内的精确相等需求。

**所在位置**：§11（全文）

**严重程度**：中等

**改进建议**：
- 在 §11 中新增子节"§11.9 `equalExact` 迁移模式"，覆盖以下内容：
  - C++ GLM 原生 `==` 的 IEEE 754 精确语义 vs 仓颉容差 `==` 的行为差异
  - 需要替换的场景清单（键值比较、Inf/NaN 检测、位精确匹配）
  - 搜索替换规则（识别浮点 Vec 变量上的 `==`，判断是否需改为 `.equalExact()`）
  - 迁移工作量评估（自动可识别部分 vs 人工审查部分）

## 问题 4（一般）：对 `references/glm-1.0.3/` 的引用基于假设其存在，但未验证关键路径的实现准确性

**问题描述**：§7 D6 对 `GLM_CONFIG_CLIP_CONTROL` 默认值的修正依赖于对参考实现 `setup.hpp` 的解读。P2（v10）已验证前述轮次错误的 `GLM_CLIP_CONTROL_DISABLE` 引用不存在。但整个文档对参考实现的使用策略（§8.4）依赖于"参考实现存在且结构正确"的假设，而设计文档本身在组件 A 审议中未对参考实现进行完整性检查（如确认 `setup.hpp` 中条件编译分支确实如假设的那样分叉）。

**所在位置**：§7 D6、§8.4

**严重程度**：一般

**改进建议**：
- 在 §1"参考实现使用策略"或 §8.4 中补充一项前置验证步骤：在编码启动前，确认 `references/glm-1.0.3/glm/glm/detail/setup.hpp` 中的条件编译逻辑与 §7 D6 描述的决策分支一致
- 对 §8.4 策略 4（配置宏确认）增加"若参考实现中的条件分支逻辑与当前描述不符，以参考实现为准并更新 D6"的兜底条款
