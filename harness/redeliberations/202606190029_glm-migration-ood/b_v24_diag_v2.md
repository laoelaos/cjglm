## 二、新增设计质量诊断发现

### 发现 B1（中等）— 256 个别名的编译性能风险被低估

- **所在位置**：§3.8 别名缩编降级策略（第 440-443 行）、§8.1 文件清单 fwd.cj 行
- **问题描述**：设计文档要求 `fwd.cj` 中全量定义 256 个 `public type` 别名，作为"首轮强制交付标准"。每个别名都是 `VecN<T, Q>` 泛型的一次具现化，且由于 4 个 Vec 结构体共定义了约 40+ 运算符/具名函数（§4.3 仅算术运算一个 Vec 就有 10+ 个），每个别名触发编译器生成这些类型的大量泛型实例化代码。文档将编译时间阈值定为 30 分钟——这本身说明设计者已意识到此风险，但仍将其归为"低概率"的缩编触发条件。考虑到 `cjpm build` 是编码阶段的核心迭代循环（编写代码→编译→测试→修复），若单次构建时间达 10-30 分钟，将严重损害开发者体验和迭代效率。此外，静态库产物体积（阈值 5MB）对嵌入式或移动端消费者可能构成实际问题。
- **严重程度**：中等 — 不影响设计正确性，但直接影响编码阶段的生产力。风险被低估意味着实施者可能直到编码中后期才意识到此问题的严重性，且触发缩编时会影响后续轮次范围和交付工期。
- **改进建议**：① 将 256 个别名的编译性能验证从"缩编触发条件"**升级为"编码前验证项"**——在 §2.1 编码前验证检查清单新增一项：在完成 16 个必备别名（B/I/U/Vec 各一个精度变体）后立即执行 `cjpm build` 计时，根据实际编译时间规划剩余别名的实现策略（全量一次性生成 vs 分批生成 vs 增量编译优化）；② 建议在 `cjpm.toml` 中启用增量编译（`incremental = true` 若支持），减少迭代周期；③ 在 §11.11 脚本工作量评估末尾补充编译性能验证工作量（0.25 人天）。

### 发现 B2（中等）— `const`/`extend` 不对称的消费者体验影响未充分评估

- **所在位置**：§3.2 Vec 结构体系（`extend` 块描述）、§4.3 算术运算 `const` 修饰符差异依据、§4.5 `equalExact`/`equalEpsilon`、§7 D32/D33
- **问题描述**：由于仓颉 `extend` 块不支持 `const` 修饰符（仓颉扩展 §4.2），以下日常操作在 `const` 上下文中不可用：
  - `v.add(s)` / `v.sub(s)` / `v.mul(s)` / `v.div(s)` / `v.mod(s)`（扩展成员函数方向）
  - `v.equalExact(other)`（浮点精确相等）
  - `v.logicalAnd(other)` / `v.logicalOr(other)`
  - `v.increment()` / `v.decrement()` / `v.bitwiseNot()`
  
  这些操作在非 `const` 上下文中正常工作。设计文档已充分记录此限制的技术原因，但**未评估其对下游消费者的认知负担和迁移风险**：一个从 C++ GLM 迁移的开发者，在看到 `==` 支持 const 上下文的容差比较后，可能自然认为 `v.add(s)` 也支持——直到编译错误才发现限制。这种 API 的不一致性（同类的算术操作，`+` 和 `v.add(s)` 的 const 可用性不同）增加了使用者的心智模型复杂度。
- **严重程度**：中等 — 属于 API 可用性设计问题。每个单项的影响有限（有替代方案），但累积效应（6+ 个常用操作不对称）影响消费者体验。
- **改进建议**：① 在 §3.2 末尾新增"`const` 上下文中的 API 可用性速查表"，集中列出哪些操作在 `const` 上下文中可用、哪些不可用及替代方案；② 在 §11 迁移成本评估中补充此项不对称对迁移代码审查的工作量影响（实施者需要人工识别原本的 `v++.increment()` 调用是否在 const 上下文中）。

### 发现 B3（一般）— 编译器行为依赖的聚合风险高于单点评估之和

- **所在位置**：§10 设计阶段验证依赖点 A~E 表（第 1419-1425 行）、§7 D29+D30 组合回退分析（第 1123-1147 行）、§2.1 编码前验证检查清单（11 项）
- **问题描述**：设计文档依赖**至少 9 项**（§10 依赖点 A~E + §10 浮点多验证项 + `@Derive[Hashable]` + `Bool→Numeric` 路径 B + 脚本验证）独立编译器行为假设，每项均标注为"待验证"且有回退方案。虽然 D29/D30 的双回退组合分析已存在，但**未对三回退或更高阶回退的组合场景做系统性分析**。例如，以下三条路径的依赖链：
  - D29 回退（const 表达式链失败）→ `==` 降级为精确比较 → `equalEpsilon` 成为唯一容差路径
  - D30 回退（`@OverflowWrapping` 继承不成立）→ 20 个扩展成员函数需补充标注
  - 路径 B（`Bool→Numeric` `as` 方案不可行）→ 回退到路径 A 具名工厂函数
  
  这三者相互独立无冲突，但若[编译期 if 分支抑制验证]整体失败（依赖点 A~E 均失败），则 `==`、`mod`、`isIec559Of` 等多处同时回退——多个回退方案的编码实施工作量存在叠加效应（§11 未评估此叠加成本）。验证时序建议（§10 末尾"验证时序"）仅给出了串行建议，未定义当最早验证的依赖项失败时如何调整后续项的实施优先级。
- **严重程度**：一般 — 不影响设计正确性，但影响实施规划的风险管理。若多个关键依赖同时失败，编码阶段的反工成本将高于单点评估之和。
- **改进建议**：① 在 §2.1 编码前验证检查清单中新增一项"编译器依赖统一验证测试"——编写一个原型程序覆盖依赖点 A~E（`testConstIfSuppression<T>()`）及 `@OverflowWrapping` 继承性、`@Derive[Hashable]` 约束传递性，在编码启动前一次性运行，根据运行结果决定后续编码路径——而非在编码过程中逐项暴露；② 在 §11 迁移成本评估末尾新增"编译器依赖验证回退成本"子节，评估最坏情况下（5+ 依赖同时回退）的额外工作量和工期影响。

### 发现 B4（提示）— 首轮 C++ 代码裁剪的完整性验证策略不够明确

- **所在位置**：§8.2 范围可追溯性对照表（roadmap §3E 序号 1）、§8.4 参考实现使用策略
- **问题描述**：设计文档声明 `setup.hpp` 中 1000+ 行仅迁移约 80-120 行配置常量，其余 900+ 行（平台检测宏、编译器兼容性头文件、架构识别等）被判定为首轮"不需要"。此裁剪判断的根据是 roadmap §3A.1 中的预声明。但做出此判断需要确认这些被省略的宏定义在首轮范围中不存在任何依赖路径——即没有其他首轮文件通过条件编译路径间接引用它们。如果某个被省略的宏实际上通过某个跨越首轮范围的条件编译路径影响了 Vec 类型的行为，则裁剪可能引入未预见的差异。设计文档目前的验证策略（§8.4 前置验证项①~③）仅验证了参考实现中的三处关键路径，未覆盖对整个 1000+ 行文件中被裁剪内容的依赖检查。
- **严重程度**：提示 — 裁剪判断本身在 roadmap 中已预先声明，且 23 轮迭代中未发现依赖遗漏。但缺乏系统性的"反向验证"（确认被省略内容确实无依赖）。
- **改进建议**：在 §8.4 策略 0 前置验证项中新增一项：对 `references/glm-1.0.3/glm/glm/detail/setup.hpp` 中被排除（不迁移）的 `#define`/`#if` 块，抽查 3-5 处是否在首轮其他文件的头文件链中有间接引用。例如：确认 `GLM_COMPILER_*` 宏系列是否被首轮范围中 `qualifier.hpp` 的任何 `#if` 路径引用。

## 五、整体评价

设计文档在 24 轮迭代后已具备极高的技术完整性和设计一致性。前序内部审议已充分覆盖技术可行性维度。本诊断侧重**落地视角**的补充审查，发现 9 项问题（5 项从前序审查修订沿用，4 项为本轮新增），其中 3 项中等、3 项一般、3 项提示——无严重阻塞性问题。

1. **B1（中等）**：在编码启动前即验证 256 个别名的编译性能，避免编码中后期才发现瓶颈
2. **B2（中等）**：补充 const/API 可用性速查表，降低实施者认知成本
3. **A1（中等）**：补充迁移成本合计，为项目管理提供决策依据
