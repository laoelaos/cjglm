## 审查发现

### F1. [一般] `fromBoolVec` 实例方法调用模式导致非自然调用路径（维持严重程度，更新分析）

**问题描述**：
`fromBoolVec` 工厂函数（§4.8）定义为扩展成员函数，函数体不依赖 `self`——仅从参数 `v: VecN<Bool, Q>` 中读取数据。调用方须持有一个同类型 Vec 实例作为 receiver：

```cangjie
let dummy = Vec2<Float32, PackedHighp>(0.0, 0.0);
let result = dummy.fromBoolVec(boolVec);
```

与本设计其他工厂函数风格不一致（`add(s, v)` 等包级独立函数可直接调用）。v26 虽已在 §4.8 给出完整签名清单（响应了 v25 F2），但签名模式本身未优化。

**所在位置**：§4.8 `fromBoolVec` 签名清单

**向下游消费者影响**：每个调用的地方都需要一个无关的 dummy 实例，或持有同类型目标 Vec 变量方可调用。若目标 Vec 在当前作用域尚未构造（常见于初始化场景），调用方必须先构造一个无关实例，增加 API 使用摩擦。

**改进建议**：
- 将 `fromBoolVec` 改为包级独立函数（如 `fromBoolVec<T, Q>(v: VecN<Bool, Q2>): VecN<T, Q>`），调用方使用 `fromBoolVec(boolVec)` 语法
- 或定义为关联到 VecN 类型的静态函数（若仓颉支持泛型 struct 上的静态泛型函数），调用方使用 `Vec2<Float32, PackedHighp>::fromBoolVec(boolVec)` 语法
- 若维持当前扩展成员函数模式，建议在 API 文档中标注"接收者实例的值不影响输出结果"

### F2. [一般] 构造函数重载解析风险的严重程度从实际落地视角不足（升级）

**问题描述**：
§4.1 承认 Vec4 的多重重载构造函数在重载解析中存在歧义风险。"若重载解析与预期不符，修改范围涉及 Vec4 构造函数体系（约 30 个构造函数签名），重构成本较高"。v2 诊断标注为 [提示]，v2 质询指出此标注与 F3（因"功能真空"从[提示]升级为[一般]）的校准尺度不一致。

**所在位置**：§4.1 "跨 Q 构造函数重载解析风险与约束验证"段落末

**校准分析**：
- **F3（v2）校准基准**：`equalEpsilon` 在 const 上下文中不可用，影响范围是"特定路径下的功能真空"，评为 [一般]
- **F2 的实际影响**：若重载解析不符合预期，影响范围是"整个 Vec4 构造函数体系（约 30 个签名）"，重构覆盖构造函数、测试用例、调用方代码三个层面
- 30 个构造函数签名的修改范围大于"特定路径下的功能真空"，按 F3 校准基准应不低于 [一般]
- 且 v26 设计文档**未在 §10 验证要求中新增明确的重载解析验证项**（v2 诊断建议①），也未定义具体的回退方案（v2 诊断建议②），风险缓释不足

**改进建议**：
1. 在 §10 设计阶段验证要求中新增"Vec4 构造函数重载解析验证"项，覆盖已知风险场景（`Vec3 + Vec1` vs `Vec3 + T` 的歧义验证）
2. 定义具体回退方案：优先移除 4-6 个歧义分支（如 `Vec3<T2,Q2> + Vec1<T2,Q2>` 组合），转为通过具名工厂函数提供
3. 将此验证项前置到 §2.1 编码前验证检查清单

**严重程度**：[一般]

### F3. [中等] `equalEpsilon` const 真空在 D29 回退路径描述中未显式警示（维持严重程度，补充分析）

**问题描述**：
§4.5 声明若 D29 回退触发，`equalEpsilon` 成为"浮点 Vec 上唯一可用的运算符级容差比较路径"。但 `equalEpsilon` 定义于 `extend` 块中（非 `const`），无法在 const 表达式中调用。回退路径下，const 上下文浮点容差比较既不能用 `==`（已回退到精确比较），也不能用 `equalEpsilon`（非 const），形成功能真空。

**v26 现状评估**：
- §4.5 的"重要限制"段落已文档化 `equalEpsilon` 的 const 限制
- §4.5 的 D29 回退描述段落仍表述为"`equalEpsilon` 成为浮点 Vec 上唯一可用的运算符级容差比较路径"，**未在此处显式标注 const 限制**
- 编码阶段实施者阅读 D29 回退描述时，可能未同步阅读"重要限制"段落而遗漏此约束

**改进建议**：
- 在 D29 回退描述段落（§4.5 末尾 `**重要限制**` 之后或 §7 D29 回退路径说明中）添加显式警示："注意：`equalEpsilon` 为非 `const` 函数，在 `const` 表达式中不可用。若 `const` 上下文需要容差比较，需将 `equalEpsilon` 移至 struct 体内声明为 `const` 成员函数（备选方案见 §4.5 `equalExact` 的 const 声明说明段落）"
- 在 §12.2 验收标准中新增"若 D29 回退触发且代码在 const 上下文中使用容差比较，验证备选方案可行性"项

### F4. [提示] 编译器行为假设的多项回退叠加效应未评估（维持发现，更新分析）

**问题描述**：
设计依赖 12+ 个编译器行为假设。§11.13 列出了各项的独立回退成本和回退触发时的修复成本，但**未评估多项同时回退时的叠加效应**——即回退方案之间是否存在冲突或互斥。当前仅 D29+D30 的组合分析在 §7 D30 中覆盖。

**v26 现状评估**：
- v26 在 §2.1 新增了 D29/D30 验证时序建议（F5）
- 但 §11.13 中未新增"多项回退叠加效应"分析

**潜在冲突示例**（非穷举）：
| 回退组合 | 潜在冲突 |
|---------|---------|
| D29 回退 + `mod` 编译期 if 回退 | 两者均导致 `==` 和 `mod` 降级为独立函数路径，但 `mod` 降级需要拆分为 `modInt`/`modFloat`，若与 `==` 回退路径中的 `equalEpsilon` 联动修改（§3.5 路径 B）同时触发，修改范围和联动验证成本可能叠加 |
| D30 回退 + 编译期 if 抑制回退 | D30 回退使算术运算符委托给扩展成员函数，若同时编译期 if 抑制失效需移除 const-if 分支，扩展成员函数中的算术实现是否受影响？ |
| `isIec559Of` 回退（依赖①②失败） + D29 回退 | 两者均导致 `==` 退至精确比较，但回退路径描述为"独立回退"——若两者同时触发，实施者可能重复执行同一回退操作 |

**改进建议**：
- 在 §11.13 末尾新增"多项回退叠加效应"子节，识别已知的冲突组合
- 至少覆盖上表中前两种组合的交互分析

### F5. [提示] 编码前验证检查清单存在验证项目层次不清晰（新发现）

**问题描述**：
§2.1 编码前验证检查清单共 12 项，但第⑫项（编译器依赖统一验证测试）与第①③④⑤⑧项存在显著功能重叠。第⑫项的描述为"编写统一原型程序 `testConstIfSuppression<T>()` 验证编译期 `if` 非选择分支抑制机制对所有 5 个依赖点成立；同时编写 `@OverflowWrapping` 继承性原型测试；编写 `@OverflowWrapping`+`const` 语法共存性原型测试"——这实质上将第①项（const 表达式链）、③项（isIec559Of 验证）、④项（编译期 if 抑制）、⑤项（@OverflowWrapping 继承性）、⑧项（Bool→Numeric 方案 B）的验证内容统一为一个测试。

**所在位置**：§2.1 编码前验证检查清单，第①~⑤项和第⑫项

**影响分析**：
- 实施者若按序号顺序执行验证，先做①③④⑤，再做⑫项，将**重复编写同样的原型测试代码**
- 文档未说明⑫项是"取代"还是"补充"前几项——若为取代，前几项的设计阶段验证要求（§10）中的独立验证方法描述将形成误导
- 清单上方的"验证时序建议"仅提及①和⑤优先，未澄清与⑫项的关系

**改进建议**：
- 在 §2.1 清单顶部或第⑫项旁添加注释，明确⑫项与①③④⑤⑧项的关系。建议方案：**将①③④⑤⑧项标注为"独立原型验证"，⑫项标注为"统一集成验证"**，说明⑫项是前几项验证通过后的综合确认测试，而非替代
- 或：将①③④⑤⑧项从独立清单项中移除，合并为⑫项下的子子项，避免重复

### F6. [提示] `fromBoolVec` 跨 Q 转换能力受限（维持严重程度）

**问题描述**：
`fromBoolVec` 签名中 `v: VecN<Bool, Q>` 的 `Q` 与目标 Vec 的 `Q` 绑定，仅支持同 Q 转换，不支持跨 Q 转换（如 `Vec2<Bool, PackedLowp>` → `Vec2<Float32, PackedHighp>`）。

**所在位置**：§4.8 签名清单

**实际影响评估**：
- 实践中 Bool Vec 的 Qualifier 参数通常使用默认精度（`PackedHighp`），跨 Q 转换场景出现频率低
- 但若出现（如精度敏感的 Bool 中间结果传递给要求特定精度目标 Vec 的调用方），当前设计缺乏转换路径，调用方需额外构造中间 Vec 变量

**改进建议**：
- 在接口注释标注"当前仅支持同 Q 转换。若需跨 Q 转换，可先通过跨 Vec 转换构造将 Bool Vec 的 Q 转换为目标 Q 后再调用 `fromBoolVec`"
- 或：增加独立的 `fromBoolVecQ2` 泛型版本，引入独立 `Q2 <: Qualifier` 参数
