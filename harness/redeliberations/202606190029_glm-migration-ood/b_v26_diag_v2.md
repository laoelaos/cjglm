## 审查发现

### F1. [中等] `fromBoolVec` 实例方法模式导致非自然调用路径（维持 v1）

**问题描述**：
`fromBoolVec` 工厂函数（§4.8）定义为扩展成员函数（`extend` 块中的实例方法），函数体不依赖 `self`——仅从参数 `v: VecN<Bool, Q>` 中读取数据。调用方须持有一个同类型 Vec 实例作为 receiver：

```cangjie
let dummy = Vec2<Float32, PackedHighp>(0.0, 0.0);
let result = dummy.fromBoolVec(boolVec);
```

与本设计其他工厂函数风格不一致（`add(s, v)` 等包级独立函数可直接调用）。

**所在位置**：§4.8 `fromBoolVec` 签名清单

**质询回应**：
质询指出"约 0.5 人天调整接口定义"的工作量评估未给出分解依据。重新评估：修改 extend 块 4 处签名（Vec1~Vec4 各一处）约 0.25 人天；同步更新调用方代码 0.25 人天。合计约 0.5 人天的评估在修正后仍合理，但移除"约"的不精确表述，改为"约 0.5 人天（4 处分量签名定义 + 调用方同步更新）"。

**严重程度**：中等。非正确性问题，但影响编码阶段 API 使用体验。

---

### F2. [提示] `fromBoolVec` 的 Q 参数绑定限制了跨 Q Bool→Numeric 转换（维持 v1）

**问题描述**：
当前 `fromBoolVec` 签名中 `v: VecN<Bool, Q>` 的 `Q` 与 `self` 的 `Q` 绑定，仅支持同 Q 转换，不支持跨 Q 转换。

**所在位置**：§4.8

**质询回应**：
质询指出"实践中 Bool Vec 的 Qualifier 参数通常无实际意义"的论断缺乏支撑。接受此批评——删除此预言式论断，改为事实陈述：当前设计限制同 Q 转换，若后续需求跨 Q 转换可通过增加独立 Q2 类型参数解决。删除了原文中的推测性语言。

**严重程度**：提示。实践中 Bool Vec 的 Qualifier 参数影响较小。

---

### F3. [一般] const 上下文中 `equalEpsilon` 容差比较路径真空（升级严重程度）

**问题描述**：
§4.5 声明若 `==` 因 D29 回退到精确比较路径，`equalEpsilon` 成为"浮点 Vec 上唯一可用的容差比较路径"。但 `equalEpsilon` 定义于 `extend` 块中（非 `const`），无法在 const 表达式中调用。回退路径下，const 上下文浮点容差比较既不能用 `==`（已回退），也不能用 `equalEpsilon`（非 const），形成功能真空。

**所在位置**：§4.5 `equalEpsilon` 定义段落，§7 D29

**质询回应**：
质询指出严重程度标注为 [提示] 但描述为"功能真空"，两者不一致。接受批评——若描述为功能真空则严重程度至少为 [一般]。升级为 [一般]。

**改进建议**：在 §4.5 回退路径描述处显式标注此真空及备选方案（移至 struct 体内声明为 const 成员函数）。

---

### F4. [提示] 构造函数重载解析风险未经验证且无明确回退方案

**问题描述**：
§4.1 承认 Vec4 的多重重载构造函数（如 `init<T2, Q2>(v: Vec3<T2, Q2>, b: T)` 和 `init<T2, Q2>(v: Vec3<T2, Q2>, b: Vec1<T2, Q2>)`）在重载解析中存在歧义风险。当前设计以"期望编译器遵循具体类型优先于泛型类型"原则处理，但此期望未经验证，且无明确的回退方案。

**所在位置**：§4.1 "跨 Q 构造函数重载解析风险与约束验证"段落末

**影响评估**：若重载解析与预期不符，修改范围涉及 Vec4 构造函数体系（约 30 个构造函数签名），重构成本较高。

**改进建议**：
- 在 §10 验证要求中新增明确的构造函数重载解析验证项
- 定义回退方案：若歧义确认，优先移除歧义分支（约 4-6 个构造函数），转为通过具名工厂函数提供对应语义

**严重程度**：提示。因风险已标注，回退方案可后续补充。

---

### F5. [提示] 编译器行为假设的聚合风险未系统性评估

**问题描述**：
设计依赖 12+ 个编译器行为假设（const 表达式链 ×5、`@OverflowWrapping` 继承性、const-if 抑制 ×5、`as` 泛型可用性等）。各项的独立回退方案已完整定义，但未评估多项同时回退的聚合成本。

**所在位置**：§10 验证要求表、§2.1 检查清单、§11.13 回退成本表

**影响评估**：
§11.13 的"悲观路径"已给出 3.5 人天修复成本，但此评估假设各项回退独立。若 5+ 项同时回退（如 `const` 表达式链 + 编译期 if 抑制 + @OverflowWrapping 继承性 + `as` 泛型可用性 同时失败），修复工作的交互依赖可能导致额外 20-30% 的叠加成本（约 4-5 人天）。此风险未评估。

**改进建议**：
- 在 §11.13 末尾补充"多项回退叠加效应"分析
- 识别回退方案间是否存在冲突（如 D29 回退与 D30 回退的复合赋值交互）

**严重程度**：提示。属于风险量化粒度问题，不构成设计缺陷。

## 整体评价

### 需求响应充分度

经独立验证 §8.2 范围可追溯性对照表，确认：
- Roadmap §3E 序号 1~11、S1~S3 全部 14 项已显式映射到对应设计章节和仓颉文件
- Roadmap §3G 范围确认结论（依赖闭合性、独立可编译性、最简可行范围）已在 §2/§8.3/§3.8 中落实
- 新增文件（`scalar_vec_ops.cj`、`lib.cj`）在偏离说明中标注理由
- 设计覆盖用户需求中的首轮范围（基础设施层 + 向量类型系统 + 256 个别名体系），**无遗漏项**

**结论**：需求响应充分。

### 事实/逻辑正确性

历经 25 轮审查迭代，过往发现的事实错误已修正。当前版本在事实层面无显著错误：
- 脚本别名生成逻辑（`suffix = '' if fam.endswith('Vec')`）正确解决了 v24 的 48/256 命名错误
- `public type` 可见性问题已在 v24 修复并新增验收项验证
- 编译器行为假设均以"待验证"状态标识并配有回退方案

**判断：有条件的就绪**。

当前设计可直接用于指导编码实施，但建议在编码启动前完成以下前置事项：
