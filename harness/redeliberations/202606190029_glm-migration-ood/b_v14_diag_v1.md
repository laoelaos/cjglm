## 发现1（中等）：`componentAt` 缺少验收层级的强制验证标准

**所在位置**：§12.2 验收标准表（全文）

**问题描述**：
`componentAt` 是 const 上下文中唯一的分量访问路径（§7 D28），已在 v14 中于 §12.1 层次二新增验证项（"componentAt const 访问：在 const 函数体内调用 componentAt(i) 验证正确性"）。但 §12.2 验收标准表未同步增加对应的强制验收项。

当前 §12.2 包含"`public const init` 外部可构造"、"公共 API 导入"、"公共 API 重导出"等验收项，但缺少对 `componentAt` const 访问正确性的验收标准。这意味着：
1. `componentAt` 的正确性验证仅停留在"测试项"层级（层次二），未升格为"验收标准"层级
2. 若 `componentAt` 实现有缺陷（如越界断言行为不一致、返回值不正确），在验收审查时可能被遗漏

**严重程度**：中等（验收标准覆盖不完整）

**改进建议**：
在 §12.2 验收标准表中新增一行：

| 验收项 | 标准 | 验证方法 |
|--------|------|---------|
| componentAt const 访问正确性 | const 函数体内调用 `componentAt(i)` 返回值与相同索引的 `operator[]` 取值形式一致；越界时触发断言 | §12.1 层次二 `componentAt` 验证项通过 |

## 发现2（中等）：§11 缺少"标量在左运算的具名函数替换"迁移成本评估

**所在位置**：§11（全文）

**问题描述**：
设计文档对 `scalar + vec`、`scalar - vec`、`scalar / vec`（标量在左、向量在右）的不可重载性在 D10 和 §4.3 中已有充分分析，替代方案为具名函数 `add(s, v)`、`sub(s, v)`、`div(s, v)` 等。但 §11 迁移成本评估的 9 个子节（§11.1~§11.9）未设立独立章节评估此替换的迁移模式。

比对已有章节的覆盖情况：
- §11.1 —— `&&`/`||` 具名函数迁移 ✅
- §11.2 —— `++`/`--` 具名函数迁移 ✅
- §11.3 —— 跨类型赋值差异 ✅
- §11.4 —— `bitwiseNot()` 迁移影响 ✅
- §11.5 —— `Inf == Inf` 差异 ✅
- §11.6 —— `@OverflowWrapping` 标注备选方案 ✅
- §11.7 —— 别名命名约定偏离 ✅
- §11.8 —— 跨 Q 赋值迁移模式 ✅
- §11.9 —— `equalExact` 迁移模式 ✅
- **缺失** —— scalar-on-left 运算符替换（D10/§4.3 描述但无 §11 评估 ❌）

影响：实施者在将 C++ 代码 `s + vec` 迁移为 `add(s, vec)` 时，缺乏批量搜索替换规则和受影响范围评估。

**严重程度**：中等（迁移成本评估完整性缺口）

**改进建议**：
在 §11 中新增子节"§11.10 标量在左运算的具名函数替换"，覆盖以下内容：
- 受影响表达式模式表（`scalar + vec`、`scalar - vec`、`scalar * vec`、`scalar / vec`、`scalar % vec`）
- 与 vec-op-scalar 方向扩展成员函数（`v.add(s)`）的关系和选择指南（§4.3 已有但迁移章节未总结）
- 搜索替换规则：`scalar + v` → `add(scalar, v)` 等
- 工作量评估：机械可替换比例 vs 人工审查比例

## 发现3（一般）：`@Derive[Hashable]` 与无默认构造函数的交互未经验证

**所在位置**：§3.2 Hashable 实现策略、§4.1 构造函数体系、§10 验证计划

**问题描述**：
Vec 结构体采用 `@Derive[Hashable]` 自动派生 `hashCode()`。但所有 Vec 结构体：
1. 仅有 `const init` 构造函数，无默认构造函数（成员无初始值）
2. 数据成员为 `var` 声明

`@Derive[Hashable]` 宏在 struct 仅有 `const init`（无普通 `init`）且成员为 `var` 无初始值时，编译器能否正确生成 `hashCode()`——此交互在 §10 验证计划中未被覆盖。查阅 §12.2 验收标准表，"哈希集合可用性"验收项仅验证 `HashSet<Vec2<Int32>>` 等编译通过，但未验证 `HashSet<Vec2<Float32>>` 等浮点 Vec 在 `@Derive[Hashable]` 下的编译通过性。

**严重程度**：一般（未验证的编译器行为依赖）

**改进建议**：
在 §10 设计阶段验证计划中新增验证依赖：

| 依赖项 | 描述 | 风险等级 | 验证方法 |
|--------|------|---------|---------|
| `@Derive[Hashable]` 在仅有 `const init` 的 struct 上的可用性 | `@Derive[Hashable]` 宏在 Vec 结构体（仅有 `const init`、无默认构造、无初始值 `var` 成员）上能否生成 `hashCode()` | 中 | 编写原型测试：`@Derive[Hashable] struct Vec2<T, Q> where Q <: Qualifier { var x: T; var y: T; public const init(x: T, y: T) ... }` 分别以 `Int32` 和 `Float32` 实例化，确认编译通过 |

同步在 §12.2 验收标准表"哈希集合可用性"项的验证方法中补充：`HashSet<Vec2<Float32, PackedHighp>>` 编译通过。

## 发现4（一般）：模块间依赖表未标注标准库依赖

**所在位置**：§2 模块间依赖表 `scalar_vec_ops` 行

**问题描述**：
`scalar_vec_ops.cj` 中的浮点 `mod` 实现依赖 `std.math.trunc`（Float64 版本），且 Float32 路径涉及 Float64 提升后调用 `trunc`（§10 浮点 mod 验证依赖②）。但 §2 模块依赖表 `scalar_vec_ops` 行的依赖仅列出"qualifier"，未标注对 `std.math` 的标准库依赖。虽然标准库依赖属于"隐式引入"，不要求在包内依赖表中列出，但对于实施者而言，首次编码时需要注意 `import std.math` 在 `scalar_vec_ops.cj` 中。

**严重程度**：一般（实施指引精细度不足）

**改进建议**：
在 §2 模块间依赖表末尾的脚注或依赖拓扑说明中，补充一行标准库依赖的汇总："`scalar_vec_ops.cj` 浮点路径依赖 `std.math.trunc`，编码阶段需在文件头部添加 `import std.math.{ trunc }`。" 或在 §8.1 文件清单的"说明"列中标注此标准库依赖。
