## 发现 2（中等）— 编译器行为假设依赖图谱分散在四个章节，缺少集中可视化风险图

### 位置

- **§2.1**（line 170~194）：编码前验证检查清单（14 项验证任务表）
- **§7 D29/D30**（line 1348~1398）：const 表达式链验证 + @OverflowWrapping 继承性验证的设计决策与回退条件
- **§10**（line 1678~1834）：编译期 if 分支抑制综合验证、equalEpsilon 验证、mod 边界验证、Bool→Numeric 验证、Vec4 重载验证
- **§11.13**（line 2232~2273）：编译器依赖验证回退成本 + 多项回退叠加效应分析

### 问题描述

本设计依赖 12+ 个编译器行为假设，其验证依赖关系、验证方法、回退方案分布在四个相距甚远的章节中：

| 假设类别 | 分散在 |
|---------|-------|
| 验证任务清单（14 项） | §2.1 检查清单表 |
| 回退条件/设计决策 | §7 D29/D30/D32/D33 |
| 验证方法/详细方案 | §10（编译期 if 抑制/equalEpsilon/mod/Bool→Num/Vec4 重载） |
| 回退成本/叠加效应 | §11.13 |

实施者在编码前需要跨四个章节交叉引用才能回答以下问题：
- "第①项验证失败后，我应该看哪个章节来回退？" → 需跳到 §10 回退表，再跳到 §7 D29，再跳到 §11.13 看成本
- "第②项和第⑤项是否存在依赖关系？" → 文档无显式说明（实际独立）
- "验证失败后是否影响其他验证项？" → §11.13 有用表格覆盖但距离验证清单较远

### 影响评估

- **编码启动效率降低**：新实施者（非参与前 29 轮审议的开发者）需要至少 2~3 次来回翻阅才能理解完整的验证依赖图景
- **遗漏风险**：如果实施者仅阅读 §2.1 检查清单而未交叉查阅 §10 的详细验证方法，可能在原型测试中遗漏关键的边界测试条件
- **当前已有改进**：§2.1 已补充"层级说明"（line 174~176）和第⑫项统一集成验证的描述，§11.13 已补充"多项回退叠加效应"子节（line 2256~2274），部分缓解了此问题

### 改进建议

1. 在 §2 末尾（编码前验证检查清单之后）新增一个**编译器行为依赖全景图**，以 Mermaid 流程图或结构化表格集中展示：
   - 核心里程碑：12 项验证的总体依赖关系和验证顺序
   - 关键回退路径：每个验证项失败后应跳转到的章节
   - 成本参考：每项验证/回退的人天估算快速参考
2. 在 §2.1 检查清单表的"失败回退"列中直接标注回退方案的**具体章节编号**（如"§10 回退表 A"而非"详见对应章节回退方案"）

## 发现 3（一般）— 缺少"非首轮内容排除清单"，范围边界模糊

### 位置

- **§8.2** 范围可追溯性表（line 1527~1544）：表格 + 偏离说明
- **§8.4 策略 6**（line 1585）"非首轮内容不应在首轮编码中引用"
- **§7 D24**（line 1315~1319）：genTypeTrait/find_best_type 推迟说明
- **§3.9**（line 636~639）：C++ 辅助设施省略说明

### 问题描述

§8.4 策略 6 要求实施者不引用非首轮内容，但未提供完整、可执行的**非首轮内容排除清单**。实施者需要自行从以下几处总结排除范围：

1. **从 §8.2 追溯表推导**：表中仅列出 14 个映射项，但 roadmap §3E 的非首轮项未列于表中（如 `_swizzle.hpp`、矩阵类型、四元数、`type_float.hpp`、`genTypeTrait` 等）
2. **从 §3.9 推断**：虽说明了 genTypeTrait/find_best_type 推迟，但未提供完整的"所有推迟项"总表
3. **从参考实现目录结构推断**：`references/glm-1.0.3/glm/glm/detail/` 中有约 60 个文件，哪些在首轮范围、哪些不在，文档未有完整文件级别的映射

具体风险场景：
- `compute_vector_decl.hpp` 首轮仅定义不消费，但它的 `include` 链中是否包含首轮不应引用的内容？（例如它通过 `#include "_vectorize.hpp"` 引入 functor——functor 在首轮范围内，无风险）
- `type_vec1.inl`~`type_vec4.inl` 中的 `#include "_swizzle.hpp"` 已被 `GLM_CONFIG_SWIZZLE` 排除，但如果实施者在阅读 `.inl` 参考实现时意外在仓颉代码中模拟了 swizzle 逻辑，则超出范围

### 影响评估

- **认知负荷**：有经验的实施者可通过交叉引用 roadmap、§8.2 和参考实现目录自行推断排除范围，但增加了 1~2 小时的认知投入
- **遗漏风险**：新实施者或时间紧迫的编码阶段，意外引用非首轮内容的概率存在

### 改进建议

1. 在 §8.4 末尾新增"首轮内容边界声明"表格，按文件级别列出：
   - **首轮包含**：14 个核心文件（如 `setup.cj`~`fwd.cj`）
   - **首轮排除**：所有矩阵/四元数/扩展功能文件（如 `type_mat*.cj`、`type_quat.cj` 等，标注"预留给第 N 轮"）
   - **首轮定义为仅类型/仅消费**：如 `compute_vector_decl.cj`、`vectorize.cj`（仅定义不消费）
2. 每个排除项标注排除理由和恢复轮次
3. 显式声明"编码阶段若意外需要引用排除清单中未列出的内容，应标记并推迟至对应轮次设计审议"

## 发现 4（一般）— 跨 Q 赋值行为约定的未来兼容性风险缺少"影响面标记"

### 位置

- **§3.1** Qualifier 运行时行为约定（line 226~232）

### 问题描述

§3.1 约定"首轮中不同 Qualifier 之间可相互赋值和转换"并说明"后续轮次引入 Qualifier 差异化时需重新评估"。但文档未提供**当前哪些泛型代码位置隐含了"跨 Q 赋值行为等价"的设计假设**，导致后续轮次引入差异化时难以完整评估影响面：

| 隐含跨 Q 等价的代码类别 | 影响面评估 | 风险 |
|------------------------|----------|------|
| 泛型跨类型构造函数 `init<T2, Q2>(v: VecN<T2, Q2>) where Q2 <: Qualifier` | 无差别允许所有 Q2→Q 转换，Vec1~Vec4 各一个，共 4 处 | 如果后续 AlignedHighp 引入非标准布局，此构造函数不检查 Q2/Q 兼容性 |
| 泛型算术运算符（`operator +`/`-`/`*` 等） | Vec1~Vec4 各 5 个，共 20 处，均假设 Q 不影响数据表示 | 如果后续 SIMD 路径下 PackedHighp 使用 SIMD 内部类型，跨 Q 运算可能产生非法赋值 |
| 编译器自动生成的复合赋值（`+=` 等） | 自动生成，每 Vec 约 5 个，共 20 处 | 同算术运算符风险 |
| `==`/`!=` 比较运算符 | Vec1~Vec4 各 1 个，共 8 处 | 同上 |
| `componentAt`/`operator[]` 访问 | Vec1~Vec4 各 1~2 个，共约 8 处 | 数据表示改变后分量访问可能语义变化 |
| `@Derive[Hashable]` 自动生成的 `hashCode()` | Vec1~Vec4 各 1 个，共 4 处 | 对齐策略改变后，相同分量值在不同对齐下的哈希值可能不同 |
| `fromBoolVec`/`fromBoolVecQ2` 工厂函数 | 8 个包级独立函数 | 不影响（函数内不依赖 Q 数据差异） |
| alias 类型定义（`fwd.cj` 中 256 个 `public type`） | 每个绑定固定 Q 参数，无跨 Q 风险 | 无风险（别名固定 Q） |

### 影响评估

- **对首轮编码无影响**：首轮所有 Qualifier 均为空结构体，跨 Q 类型层面等价
- **对第 2+ 轮的影响**：当后续轮次为不同 Qualifier 引入不同数据布局时，上述代码位置**全部需要重新审查**。当前文档没有提供"待审查代码清单"的快速参考，后续维护者需要人工识别所有受影响的代码位置

### 改进建议

1. 在 §3.1（或 §7 D27）中新增"跨 Q 等价假设影响范围"子节，以上表形式列出受影响的代码类别和位置
2. 在 `type_vecN.cj` 的泛型构造函数和算术运算符上添加注释标记：`// ⚠ Q-agnostic: 首轮假设跨 Q 等价，后续轮次若引入 Q 差异化需重新审查`
3. 在 §11 迁移成本评估中新增"后续轮次跨 Q 审查工作量"的预估条目（即使首轮不执行）

## 发现 5（提示）— 缺少"迁移对照示例"，实操性存在缺口

### 位置

- 全文（各章节均无 C++→仓颉代码对照的完整使用示例）

### 问题描述

第 29 轮内部审议已经指出"编码直接指导性方面"的改进空间（发现 11），v30 已做补充：拆分阶段 5 为 5a/5b、补充 Vec2 运算符语法模板（line 836~882）、添加 equalEpsilon 集中参考框（line 940~947）。但在**完整迁移示例**方面仍未覆盖。

具体缺口：

1. **没有从 `#include <glm/glm.hpp>` 到 `import glm.*` 的完整使用演示**：文档描述了 lib.cj 如何通过 `public import` 导出，但没有给出一个完整的、从消费者视角的代码示例：
   ```cangjie
   // 期望看到的示例：
   import glm.*
   
   func example() {
       let v1 = BVec2(true, false)
       let v2 = Vec2<Float32, PackedHighp>(1.0f, 2.0f)
       let v3 = v1.logicalAnd(v1)  // Bool Vec 逻辑运算
       var v4 = Vec4<Float32, PackedHighp>(v2, v2)
       v4 = v4 + Vec4<Float32, PackedHighp>(1.0f, 2.0f, 3.0f, 4.0f)
       let equal = v4.x == v4.y  // 分量比较
       let result = v1[0]  // 下标访问
   }
   ```
2. **没有 C++ GLM vs 仓颉 GLM 的逐行对照**：对于正在迁移 C++ GLM 代码库的开发团队来说，缺少一个"一眼可以看出变化"的对照示例。例如：
   ```cpp
   // C++ GLM
   #include <glm/glm.hpp>
   glm::bvec2 v(true, false);
   glm::vec4 v2(1.0f);
   glm::bvec2 v3 = ~v1;  // 行为差异!
   ```
   vs
   ```cangjie
   // Cangjie GLM
   import glm.*
   let v = BVec2(true, false)
   let v2 = Vec4<Float32, PackedHighp>(1.0f)
   // ~v1 → v.bitwiseNot() [非Bool] / !v1 [Bool, 行为不同]
   ```
3. **常见迁移陷阱速查表**：虽然 §11 提供了详细的迁移成本评估，但缺少一个"1-2-3 速查"式的编码陷阱列表。例如：
   - ❌ `~bvec2` → ✅ `!bvec2`（但行为相反！需要人工审查）
   - ❌ `v++` → ✅ `v = v.increment()`
   - ❌ `s + v` → ✅ `add(s, v)`

### 影响评估

- 不会阻碍编码启动（有经验的实施者可以从 §4 的 API 契约和 §8.3 的迁移顺序中自行推导迁移方法）
- 但对于新团队或同时进行 C++ 代码迁移的场景，缺少快速参考增加了前 2~3 天的学习曲线

### 改进建议

1. 在 §1 末尾新增一个"快速迁移示例"子节（约 20~30 行 C++ 对照代码），展示完整的 `import`→`声明`→`构造`→`运算`→`比较` 链路
2. 在 §11 迁移成本评估表格之后新增"常见迁移陷阱速查表"（表格形式，覆盖 ~10 个最常见的迁移模式差异）
