# OOD 设计方案审查报告（v26）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** `interface Qualifier` + 空结构体实现模式在仓颉类型系统中可行——泛型参数 `Q <: Qualifier` 实现编译期类型级区分，与 enum 值不能作为泛型实参的限制兼容。

**[通过]** 四个独立泛型结构体（Vec1~Vec4）替代 C++ 单模板分量的选择正确——仓颉不支持偏特化，无法实现 `Vec<N,T,Q>` 单模板的分量数差异化。

**[通过]** 跨 Vec 转换构造函数使用无约束泛型 + `T(v.x)` 显式转换匹配仓颉类型系统——仓颉原始数值类型间无子类型关系，`where T2 <: T` 约束无效，故无约束泛型是正确选择。

**[通过]** `public type` 别名 + `public import` 重导出机制在仓颉中有效——别名仅创建新名称而非新类型，`public import` 实现跨包重导出。

**[通过]** `const init` 与 `var` 成员的共存性在仓颉中成立——`const init` 内可使用 `=` 对 `var` 成员赋值（const §4.2），且 `const init` 在非 const 上下文中也可用于运行时构造（const §3.2 规则 5）。

**[通过]** `@Derive[Hashable]` 在泛型 struct 上的自动派生在仓颉宏能力范围内。

**[通过]** `const` 修饰符在 `extend` 块中不可用的限制已确认（extend §4.2——允许的修饰符列表中不含 `const`），设计中对所有扩展成员函数统一声明为非 `const` 的策略正确。

**[轻微]** `NumericLimits<T>` 和 `epsilonOf<T>()` 上使用的 `where T <: Number<T>` 约束——依据可用标准库文档（`std.core` 核心接口列表），`Number<T>` 接口的存在性未被确认。设计中的兼容性路径 A/B（§3.5）已提供有效缓解措施（移除约束、返回固定容差），但主设计路径的约束依赖需编码阶段验证。

### 2. 标准库与生态覆盖

**[通过]** 所有依赖的标准库能力在仓颉 `std` 中可用：
- `std.math.trunc`（浮点 `mod` 浮点路径）
- `@OverflowWrapping` 溢出注解（反射与注解 §1.1）
- `@Derive[Hashable]` 自动派生（泛型 struct 上可用）
- `std.unittest` 测试框架
- `Hashable` / `Equatable` 等核心接口

**[通过]** 自定义 shim 层（`shim_assert.cj`、`shim_limits.cj`、`shim_cstddef.cj`）的设计合理——通过手写替代 C++ 标准库设施，属于首轮必要范围。

**[通过]** 浮点 `mod` 的 `std.math.trunc` 依赖路径（`Float32` → `Float64` 提升 → `trunc` → 降级）的设计合理，精度提升无损取模语义。

### 3. 语言特性可行性

**[通过]** 错误处理策略与仓颉能力匹配——`if + throw` 断言模式、`@OverflowWrapping` 溢出控制、依赖运行时异常处理除零和类型转换异常。

**[通过]** 并发模型兼容——Vec 为值类型，跨线程复制传递，无共享可变状态，复合赋值非原子性已声明。

**[通过]** 编译期 `if` 分支抑制行为在仓颉 `const` 函数中为固有语义——`const` 函数体内的 `if` 条件若可在编译期确定，非选择分支仅做语法检查。

**[通过]** `is` 运算符在 const 表达式中的可用性已确认（const §1.1 规则 8），`T(0)` const 构造对数值类型和 `Bool` 可用（const §1.1 规则 7 和 `Bool(0)` 构造）。

**[通过]** `as` 运算符运行时类型检测路径 B 在仓颉中可行——`e as T` 返回 `Option<T>`，对所有类型组合合法（类型系统 §4.5）。

**[通过]** 模块/包结构设计符合 cjpm 项目组织方式——`package glm.detail` + `package glm` 双层结构，`cjpm.toml` 配置骨架已提供。

**[轻微]** `@OverflowWrapping` 标注继承性（自动生成的复合赋值版本继承二元运算符上的标注）属于编译器未文档化行为（§7 D30）。设计已提供备选方案（在 20 个已有扩展成员函数上直接标注），风险可控。

### 4. 设计一致性

**[通过]** 各抽象职责描述清晰——Qualifier 体系定义精度标记契约，Vec 结构体承载向量数据和运算，Functor/ComputeVec 为后续轮次预留，shim 层替代 C++ std 设施，别名层封装命名约定。

**[通过]** 协作关系形成闭环——`Q <: Qualifier` 约束贯穿所有泛型 Vec 类型；`ComputeEqual` 被 Vec 的 `==` 委托；`scalar_vec_ops.cj` 中的方向辅助函数被 Vec 消费。无缺失环节。

**[通过]** 行为契约描述完整——构造函数签名清单（§4.1）全面覆盖 GLSL 5.4.1 语义；各运算符的行为约定（算术/位/比较/布尔逻辑）均以章节形式显式定义；异常场景（§9）覆盖零向量/除零/NaN/Infinity/类型转换边界。

**[通过]** 模块间依赖方向合理——`glm` 单向依赖 `glm.detail`，反向无依赖；`glm.detail` 内部依赖形成 DAG（`setup.cj` 无依赖 → `qualifier.cj` → `compute_vector_decl.cj` → `type_vecN.cj`），无循环依赖。

**[通过]** 审查发现 F1~F5 的修复已在 v26 中落实——合计表脚注（F1）、`fromBoolVec` 正式定义（F2）、`scripts/` 目录创建指令（F3）、D6 根因推测（F4）、D29/D30 时序建议注释（F5）。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则——Vec 结构体关注类型定义和数据承载，别名层关注命名约定，shim 层关注 C++ 等价替代，质量层关注验证策略。

**[通过]** 抽象层次恰当——未过度设计（如 `storage` 模板因仓颉无 C 数组而省略、`genTypeTrait`/`find_best_type` 推迟到矩阵轮次），也未设计不足（构造函数签名清单完整到足以指导编码）。

**[通过]** 设计便于后续实现——文件结构与包组织已明确定义，`public import` 策略确保公共 API 封装性，256 个别名脚本生成策略降低手动错误。

**[通过]** 设计便于单元测试——测试文件组织结构与源码一一对应（`tests/glm/detail/` 对应 `package glm.detail`；`tests/glm/` 对应 `package glm`）；`internal` 类型测试访问的三级回退方案（`protected` → `public` → helper 文件）已合理定义。

## 修改要求

无严重或一般问题。上述两条轻微问题不阻塞通过：
1. `Number<T>` 约束依赖——编码阶段验证 `std.core` 中是否存在 `Number<T>` 接口，若不存在则启用 §3.5 路径 A（移除约束，返回固定容差）。
2. `@OverflowWrapping` 继承性验证——编码阶段按 §7 D30 编写原型测试，若继承性不成立则启用备选方案（在 20 个已有扩展成员函数上直接标注）。
