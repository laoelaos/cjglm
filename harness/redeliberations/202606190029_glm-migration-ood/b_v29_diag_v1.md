## 发现 1（中等）— const 真空缓解方案存在逻辑矛盾

**所在位置**：§4.5（第 881-884 行）、§11.14（第 2174-2204 行）、§7 D29（第 1257-1271 行）

**问题描述**：文档在 §4.5 提出 D29 回退触发后 `equalEpsilon` 在 const 上下文中不可用的"const 真空"问题，并在 §11.14 给出缓解方案①——"将 `equalEpsilon` 定义从 `extend` 块移至 Vec 结构体体内，利用 `const init` 声明为 `const` 实例成员函数"。但此方案与 D29 回退触发的前提条件存在根本性矛盾。

**论证**：
1. D29 回退的触发条件是：`const` 表达式链 `abs(a - b) <= epsilonOf<T>()` 在 const 上下文中**编译失败**。
2. `equalEpsilon` 的函数体与此表达式链完全一致（§4.5 第 873 行：`abs(this.x - other.x) <= epsilonOf<T>()`）。
3. 将 `equalEpsilon` 移至 struct 体并声明为 `const` 后，其函数体仍需在 const 上下文中通过编译——恰好是 D29 已验证失败的条件。
4. 因此方案①**无法解决** const 真空——它仅改变了函数定义位置和 `const` 修饰符，不改变函数体内容，在 D29 已回退的前提下方案①必然同样编译失败。

**严重程度**：中等。此矛盾误导实施者以为存在"不解自明"的缓解路径，实际在 D29 回退场景下该路径不可行。§11.14 的成本评估（约 0.5 人天）基于不可行的前提，需要重新估计。

**改进建议**：
- 将 §4.5 const 真空警示段落中方案①的表述从缓解方案修正为"理论路径"并标注**前提条件**：仅当 D29 回退的根因**不是**表达式链中的 `abs()`/`epsilonOf<T>()` const 不可用（而是 `const operator ==` 本身的重载规则或语法约束问题）时，方案①才可能可行。
- 将 §11.14 方案①的可行性评级从"推荐优先采用"降为"需验证前提条件后判断"，成本评估标注"前提条件满足时约 0.5 人天"。
- 补充第三条缓解方案（实际唯一可靠路径）：在 const 上下文中使用精确比较（`==` 已在 D29 回退中降级为精确比较），若需要容差比较则需在 const 上下文中使用运行时委托（通过非 const 函数间接包装后调用）。

## 发现 2（中等）— 修订说明版本序列存在跳变

**所在位置**：§附录 修订说明（第 2310-2395 行）

**问题描述**：修订说明中当前版本标注为 "v29"，"上一轮修订说明" 标注为 "v27"，然后是 "v24"、"v23"、"v22"、"v21"、"v20"、"v19"。版本序列 v29 → v27 → v24 → v23 → ... 存在非连续跳变——缺少 v28、v26、v25 的修订说明。

**分析**：从历史记录看，v28 审查意见的 5 个问题（P1~P5）已在当前 v29 章节中逐条回应，但 v28 本身的修订说明未被纳入主文档。上一轮（v28）文档应当包含 v28 修订说明（对应 v27→v28 变更），但 v29 文档中"上一轮"直接标记为 v27，说明 v28 的修订说明在文档迁移时被遗漏。

**严重程度**：中等。新读者无法通过修订说明追溯 v27→v28 间的完整变更，影响文档的可追溯性和版本审计能力。

**改进建议**：
- 将 "版本 v27 — 上一轮修订说明" 标注为 "版本 v28 — 上一轮修订说明"，并确认修订内容反映的是 v27→v28 的变更而非 v26→v27 的变更。
- 若 v27 修订说明的内容确实来自更早的迭代，需检查是否存在 v28 修订说明被意外覆盖的可能；若 v28 期间文档无实质性变更（仅确认性审查），应在 v28 修订说明中明确标注"本轮审查未发现需修订的问题"或"本轮仅确认审查意见已修复"以避免版本断裂。

## 发现 3（一般）— `protected` 回退方案可能无法跨模块工作

**所在位置**：§12.3（第 2297 行）

**问题描述**：文档将 `protected` 作为 `internal` 测试访问回退的第 1 优先级方案，声称 "`protected` 在当前仓颉版本中仅允许同包和子包访问"。但仓颉语言规范中 `protected` 的可见域定义为**当前模块**（package 文档 §5.1："| `protected` | ✓ | ✓ | ✓ | ✗ |"列标题为"当前模块"）。在 cjpm 项目中，`tests/` 目录的测试文件与 `src/` 目录的生产代码是否属于同一模块取决于 cjpm 的编译单元划分——通常是分离的测试目标。

**论证**：若测试文件被编译为独立的测试二进制（`cjpm test` 的典型行为），则测试代码和生产代码分属不同模块，`protected` 可见性**不足以**跨模块访问。文档 §12.3 本身也标注了"此方案的可行性依赖仓颉编译器对 `protected` 可见性跨包访问的支持程度"——与"当前模块"的规范定义存在张力。

**严重程度**：一般。文档已标注不确定性，但将最不确定的方案放在回退层级首位，可能导致实施者优先选择此方案尝试后才发现不可行，浪费时间。

**改进建议**：
- 在 §12.3 回退方案层级中，将第 1 优先级的 `protected` 方案标注明确的前提条件："需在目标编译器上验证 `tests/glm/detail/` 测试包与 `glm.detail` 生产包是否属于同一模块。若 `cjpm test` 将测试文件编译为独立模块，`protected` 不可行。"
- 建议将回退层级顺序调整为：首选方案（同包测试直接访问）→ 验证 `protected` 可行性 → helper 文件方案 → `public` 提升方案，而非将 `protected` 直接标为 1st。

## 发现 4（一般）— 构建配置缺少测试运行器设置

**所在位置**：§2 项目初始化与构建基础设施（第 106-157 行）

**问题描述**：文档提供了最小 `cjpm.toml` 骨架（`output-type = "static"`），并提到 `cjpm test` 可运行 `@Test` 标注的测试。但未提供完整的、可直接拷贝使用的 `cjpm.toml` 配置。

**具体缺口**：
1. `cjpm.toml` 中未指定 `test-dir` 或 `test` 相关配置。对于静态库项目，`cjpm test` 需要知道如何将测试文件链接到被测库——部分 cjpm 版本可能需要显式 `[test]` 节或 `test-scope` 配置。
2. 测试文件清单（§8.1 表 T1~T14）明确列出 14 个测试文件，但 `cjpm.toml` 中未说明这些文件是否自动发现。
3. 对于 `@Test` 注解在静态库 `output-type = "static"` 项目中的可用性，未在 §10 验证计划中作为独立依赖项验证。

**严重程度**：一般。编码启动时实施者需自行摸索 `cjpm.toml` 的测试配置，不符合文档宣称的"可直接指导编码启动"的目标。

**改进建议**：
- 在 §2 最小 `cjpm.toml` 配置后补充一个完整的配置示例（含测试目录设置）。
- 在 §10 设计阶段验证要求中新增一项："确认 `cjpm.toml` 中 `output-type = "static"` 时，`cjpm test` 可正确发现并执行 `tests/` 目录中的 `@Test` 标注测试用例。"

## 发现 5（一般）— equalEpsilon 在整数 Vec 上的编译通过性断言可能不成立

**所在位置**：§4.5（第 871-873 行）、§10 依赖②（第 1654 行）

**问题描述**：文档声称 `equalEpsilon` 在整数 `T`（如 `Int32`）上编译通过，理由是 "Cangjie 整数类型支持 `-` 和 `abs`"。但 `equalEpsilon` 的函数体包含 `epsilonOf<T>()` 的调用，而 `epsilonOf<T>()` 的签名为 `const func epsilonOf<T>(): T where T <: Number<T>`（§3.5，第 362 行）。

**问题**：当 `T = Int32` 时，`Int32` 实现了 `Number<Int32>` 吗？仓颉标准库中 `Int32` 的接口实现未在已加载的文档中显式确认。若 `Int32` 不满足 `Number<Int32>` 约束（或仅在特定的扩展方法中满足），则 `equalEpsilon<Int32>` 可能产生编译错误。

**文档自身的行动**：文档在 §3.5 兼容性策略中已预见到此风险，提出了两条路径（路径 A：移除 `Number<T>` 约束；路径 B：完全移除 `epsilonOf<T>()`）。但 §4.5 和 §10 对 `equalEpsilon` 的描述仍基于 `epsilonOf<T>()` 在整数 `T` 上可编译的假设，未显式关联到 §3.5 的双路径策略。

**严重程度**：一般。风险已在 §3.5 预覆盖，但未在 §4.5 和 §10 的验证项中形成显式追踪链。

**改进建议**：
- 在 §4.5 `equalEpsilon` 代码示例的注释中，显式指向 §3.5 兼容性策略的路径 A/路径 B。
- 在 §10 依赖②的验证方法中，补充说明验证步骤与 §3.5 双路径策略的联动关系——先尝试路径 A（验证 `Int32 <: Number<Int32>` 是否满足），若失败则走路径 B。

## 发现 6（提示）— 依赖拓扑中 `scalar_vec_ops.cj` 对 `std.math` 的依赖声明不完整

**所在位置**：§2 模块间依赖表（第 96 行）、§8.1 文件清单（第 1403 行）

**问题描述**：§2 模块间依赖表标注 `scalar_vec_ops → qualifier + std.math（浮点 mod 路径依赖 std.math.trunc）`。§8.1 文件清单在"说明"列标注了对 `std.math` 的依赖。

但 **`scalar_vec_ops.cj` 中除 `mod` 之外的 20 个函数（`add`/`sub`/`mul`/`div`）是否也依赖 `std.math`？** 文档第 807 行说"scalar-op-vec 方向包级独立函数（`add(s, v)` 等，共 20 个，定义于 `scalar_vec_ops.cj`）因不通过 Vec 二元运算符委托（直接使用 `T` 类型原生算术运算符），须在函数定义上直接标注 `@OverflowWrapping`"。若使用 `@OverflowWrapping` 注解，不需要 import `std.math`——注解是编译器内置机制。

但 `mod` 的浮点路径使用了 `std.math.trunc`，这是正确标注的。问题在于 §2 依赖表将 `std.math` 标注为整个 `scalar_vec_ops` 模块的依赖，可能误导实施者以为所有 5×4 个函数都依赖 `std.math`，实际上仅 `mod` 的浮点路径依赖它。

**严重程度**：提示级。不影响功能正确性，但可能误导 import 管理。

**改进建议**：
- 将 §2 依赖表中的 `+ std.math` 标注修改为 `+ std.math（仅 mod 浮点路径）`，或使用脚注形式。
- 同理更新 §8.1 文件清单中的说明列。

## 发现 7（提示）— 256 个别名的编译时间影响未在架构评估中考虑

**所在位置**：§3.8（第 449-524 行）、§11 迁移成本评估

**问题描述**：文档在 §3.8 提及 256 个别名可能导致总编译时间超过阈值（如 > 30 分钟）是别名缩编的触发条件之一，但未在架构评估中预评估别名对模块整体编译时间的影响。256 个别名意味着每个别名是一个 `public type` 定义，可能引入 256 次类型实例化（尽管是别名而非完整类型定义），在复杂泛型场景中编译器可能展开多次。

**分析**：`type BVec2 = Vec2<Bool, PackedHighp>` 是类型别名而非实例化——编译器处理别名时代为类型替换而非完整模板实例化，因此编译时间影响低。但在别名用于泛型方法调用时（如 `let v: BVec2; v + v`），编译器需对 `Vec2<Bool, PackedHighp>` 进行完整的类型检查和运算符查找，此时 256 个别名的间接影响不可忽略。

**严重程度**：提示级。

**改进建议**：
- 在 §3.8 或 §11 中补充一段编译时间考量说明，区分别名定义阶段（编译时别名展开，影响低）和别名使用阶段（导致 256 种 Vec 具现化组合，在泛型函数实例化时可能影响编译时间），帮助实施者评估缩编触发条件的实际适用性。
