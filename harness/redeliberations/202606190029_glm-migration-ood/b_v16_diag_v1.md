## 二、事实错误与逻辑矛盾

### P1（中等）— `epsilonOf<T>()` 固定容差实现与 §9.3 缩放容差描述矛盾

- **位置**：§3.5 `epsilonOf<T>()` 定义（第 322-325 行） vs §9.3 `NumericLimits<T>.epsilon()` 段落（第 1243-1245 行）
- **问题描述**：
  - §3.5 定义的 `epsilonOf<T>()` 实现为：`NumericLimits<T>.epsilon()`，返回固定值（如 `Float32.epsilon ≈ 1.19e-7`）。
  - §9.3 描述为：`epsilonOf<T>()` 返回 `epsilon() * magnitude` 的**缩放容差**。
  - 两种语义在运行时产生完全不同的比较结果：固定容差对所有数量级使用相同阈值，缩放容差根据数量级动态调整。若实施者按 §3.5 实现（固定值），与 §9.3 描述的调用方预期不符。编码阶段将面临语义歧义。
- **严重程度**：中等 — 不阻塞编译但影响容差比较的数值正确性和行为可预测性。
- **改进建议**：在全文中统一 `epsilonOf<T>()` 的语义。若为固定容差策略，修正 §9.3 描述为"固定容差值 `NumericLimits<T>.epsilon()`"；若为缩放容差策略，在 §3.5 中修改 `epsilonOf<T>()` 实现为 `NumericLimits<T>.epsilon() * abs(a)`（需注意缩放容差中的零值边界，`abs(0) = 0` 时容差退化为 0，可能导致 `0 == epsilon` 的比较失效）。

### P2（中等）— `equalEpsilon` 对 VecN<Bool, Q> 的编译行为描述错误

- **位置**：§4.5 `equalEpsilon` 定义（第 699-711 行）
- **问题描述**：
  - 文档声称"整数/Bool T 上行为等价于 exact"（第 711 行）。
  - 但 `equalEpsilon` 的函数体包含 `abs(this.x - other.x) <= epsilonOf<T>()`：
    - 对 `Bool` 类型：`this.x - other.x` 无法通过编译（Bool 不支持 `-` 运算符）。
    - 即使 `-` 通过，`epsilonOf<Bool>()` 因 `Bool` 不实现 `Number<Bool>` 约束也无法编译。
  - 因此 `equalEpsilon` 在 `VecN<Bool, Q>` 上**不产生编译错误才是意外**——它应产生编译错误（与其他算术运算在 Bool Vec 上的行为一致），而非"行为等价于 exact"。
- **严重程度**：中等 — 事实性错误，误导实施者对 Bool Vec 上 `equalEpsilon` 可用性的预期。
- **改进建议**：将描述修正为"整数 T（Int32 等）上行为等价于 exact（`-` 和 `abs` 对整数类型合法）；Bool T 上因 `-` 运算符不可用而在实例化时产生编译错误，与 §4.3 中 `increment()`/`decrement()` 在 Bool Vec 上的行为一致（D5 延迟检查语义）"。

### P3（一般）— `equalEpsilon` 代码示例中的未使用变量

- **位置**：§4.5 `equalEpsilon` 定义（第 705-707 行）
- **问题描述**：
  ```cangjie
  let comp = ComputeEqual<T>()
  // 非 const 精确容差路径：直接委托给 comp.callConst 的容差实现
  abs(this.x - other.x) <= epsilonOf<T>() && abs(this.y - other.y) <= epsilonOf<T>()
  ```
  - 创建了 `comp` 变量但从未使用，注释声称委托给 `comp.callConst`，但实际代码为内联比较。
  - `comp` 变量会产生编译器告警（未使用变量），且注释与代码自相矛盾。
- **严重程度**：一般 — 不影响设计正确性，但作为直接指导编码的示例代码，存在代码质量问题。
- **改进建议**：统一实现：要么删除 `comp` 变量和内联比较（`comp` 在此处无实际用途），要么将比较逻辑委托给 `comp.callConst(a, b)`。若采用委托形式，需注意 `callConst` 为 `const` 函数，在非 `const` 扩展成员函数中调用是合法的（const 函数可在非 const 上下文中调用）。

### P4（一般）— `equalEpsilon` 在 §10/§12 中缺少系统性验证计划覆盖

- **位置**：§10 设计阶段验证（全文）、§12 验证层次（全文）
- **问题描述**：
  - `equalEpsilon` 在 v16 中作为"完整定义"新增于 §4.5，被列为 D29 回退路径下的核心容差比较 API。
  - 但 §10 的设计阶段验证要求表中未包含 `equalEpsilon` 的验证依赖项（与 `equalExact` 有完整验证覆盖不同）。
  - §12.1 层次三"运算正确性验证"也未包含 `equalEpsilon` 的验证项。
  - 实施者将不知如何测试 `equalEpsilon` 的正确性——特别是在浮点 Vec 上的容差比较行为、与 `==` 在各场景下的一致性/差异性。
- **严重程度**：一般 — 不影响编码可行性，但验证缺口可能导致 `equalEpsilon` 的实现缺陷逃逸到后续轮次。
- **改进建议**：
  1. 在 §10 新增 `equalEpsilon` 验证依赖项，覆盖：`abs()` 在非 `const` 扩展成员函数中的可用性、`epsilonOf<T>()` 对浮点/整数 T 的编译通过性、Inf/NaN 边界行为。
  2. 在 §12.1 层次三新增 `equalEpsilon` 验证项：有限值浮点 Vec 上的容差比较正确性、Inf/NaN 场景下的行为确认、与 `==` 在容差路径/回退路径下的差异验证。
  3. 在 §12.2 验收标准表中新增 `equalEpsilon` 验收项（与 `equalExact` 同级）。
