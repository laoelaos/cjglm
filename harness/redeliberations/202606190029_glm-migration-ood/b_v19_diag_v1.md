### 1.2 与参考实现的一致性验证

### 1.3 需求中"上一轮OOD"的连续性

需求引用上一轮产物路径为 `202606170050`，当前轮次为 `202606190029`。产出通过 v3~v18 的迭代历史（`iteration_history.md`）、§7 设计决策体系（D1~D35）以及附录修订说明（v19 修订 6 项发现），完整展示了从上一轮到本轮的设计演进。**连续性良好。**

### 1.4 范围变更管理

## 二、事实错误与逻辑矛盾

### 发现 1（中等）— D34 "const if" 术语清理不彻底，全文 25 处残留，实施者面临误导风险

**所在位置**：

- §3.5 第 307 行（代码注释）：`// const 版本：通过 const if 实现编译期分支`
- §3.5 第 328 行：`非执行路径（const if 的 else 分支）`
- §3.6 第 355 行：`const 配置 + const if 在 Release 模式下跳过断言检查`
- §3.6 第 359 行：`ComputeEqual.callConst 中的 const if 分支使用`
- §5 第 622 行（同 §3.6 第 355 行相同模式）
- §5 第 765 行（同 §3.6 第 355 行相同模式）
- §7 D11 第 903 行：`可以在函数体内使用 const if 实现编译期分支`
- §7 D18 第 948 行：`由 const if 在编译期已排除非浮点分支`
- §7 D19 第 952 行：`采用 const if 配合 is 运算符的值级类型检测方案`
- §7 D19 第 958 行（同 952）
- §7 D19 第 962 行：`实现模式与 const if 的现有语义一致`
- §7 D28 第 1018 行：`该函数支持 const if 编译期分支`
- §7 D29 第 1032 行：`const if 确保仅浮点类型进入此分支`
- **§7 D32 第 1087 行（最严重）**：`仓颉要求 const if 所在函数必须标注 const 修饰符，因此仅包级独立函数方向可以承载 const if 分支`——单句出现两次 `const if`，且 D32 本身是 D34 之后的设计决策，未遵守自身约定
- §7 D32 第 1089 行：`虽不直接使用 const if`
- §7 D32 验证表第 1096 行：`const + const if 兼容性`
- §9 第 1225 行：`在 const if 路径下`
- §9.4 第 1260 行：`const if 在 const 函数体中对泛型参数的检测能力`

**问题描述**：D34（§7 第 1105-1111 行）明确决策："设计文档中所有出现的 `const if (...)` 统一改写为 `if (...)`"，理由是该独立关键字不存在于仓颉语言中，可能误导实施者编写不存在的 `const if` 关键字。v19 修订说明称已修复"发现 2（一般）— §4.3 残留 const if 术语"，但 grep 结果显示全文档共 **26 处**匹配 `const if`，v19 **仅修复了 §4.3 第 646 行 1 处**，其余 25 处全部残留。

**严重程度**：中等。实施者首次阅读时可能认为 `const if` 是仓颉的独立关键字组合（尤其是在 D32 这样的关键设计决策章节中直接使用），从而在编码阶段编写不存在的 `const if (...)` 语法导致编译错误。此问题已被 D34 明确识别和决策，但仍未系统化修复。

**改进建议**：对全文档执行系统性搜索替换：
1. `const if` → `编译期 if`（用在独立术语场景）
2. 或根据上下文分别改写为 `"const" 函数体内的 "if" 表达式` 或 `编译期 "if" 分支`（D34 推荐表述）
3. D32 第 1087 行作为重点修复对象，确保设计决策章节自身遵守约定
4. 代码注释中的 "const if" 一并改写（§3.5 第 307 行等）

### 发现 2（一般）— §12.3 同一行内混用 `tests/glm/detail/` 和 `tests/glm.detail/` 两种目录路径表示法

**所在位置**：§12.3 第 1843 行

**问题描述**：
```
- **对 `internal` 类型的访问策略**：需要测试 `glm.detail` 中 `internal` 可见性类型...放置在 `tests/glm/detail/` 包中...**p>编码阶段需编写原型测试验证：在 `tests/glm.detail/` 目录下创建测试文件
```

同一行内前半句使用正确路径 `tests/glm/detail/`（斜杠分隔），后半句使用错误路径 `tests/glm.detail/`（点号分隔）。仓颉包目录路径解析要求使用系统分隔符（`/`），点号不是有效的目录分隔符。

**严重程度**：一般。此问题自 v11 发现项 2（"测试目录结构 `tests/glm.detail/` 与仓颉包路径解析规则不匹配"）和 v12 发现项 4（"§12.3 第 1718 行残留 `tests/glm.detail/` 混合表示法"）以来已多次被指出和修复，但修复不彻底再次复发。实施者可能按点号路径创建目录导致测试无法编译。

**改进建议**：将 `tests/glm.detail/` 改写为 `tests/glm/detail/`。建议清理时全文 grep 搜索 `glm\.detail/` 模式（点号后跟斜杠的路径表示法）确保无遗漏。v19 当前只有此 1 处残留。
