# OOD 设计方案审查报告（v6）

## 审查结果

REJECTED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** `interface Qualifier` + 空结构体实现模式完全符合仓颉类型系统能力——接口可用作泛型约束 `where Q <: Qualifier`，空结构体零运行时开销。

**[通过]** Vec1~Vec4 作为独立泛型结构体定义，仓颉支持泛型 struct 和 `where` 约束。四个独立类型而非单模板偏特化的选择正确反映了仓颉不支持 C++ 偏特化的约束。

**[通过]** `@Derive[Hashable]` 在泛型结构体上的统一使用是可行方案。设计正确认知到仓颉不支持条件派生，浮点 Vec 也会自动获得 Hashable 实现，并如实记录了与浮点容差 `==` 的不一致限制。

**[通过]** `const init` + `const` 实例成员函数的模式可行——仓颉 const README §3.2 规则 9 明确要求 struct 定义 `const init` 后方可定义 `const` 实例成员函数。设计正确应用了此规则。

**[通过]** 运算符重载范围识别正确：有限集 `+`、`-`、`*`、`/`、`%`、`<<`、`>>`、`&`、`|`、`^`、`!`、`==`、`!=`、`[]` 可重载；`&&`、`||`、`~`、`++`、`--` 不可重载。对应替代方案（具名函数）合理。

**[通过]** `mut operator func []` 赋值形式强制要求标记 `mut`——仓颉 struct 非 `mut` 实例函数不能修改实例成员，设计正确添加了 `mut` 修饰符。

**[通过]** `const` 不构成运算符重载区分依据（const README §3.2 规则 7），因此提供 `componentAt` 具名 const 函数替代 `const operator[]`——策略正确。

**[通过]** 跨 Vec 转换构造函数使用无约束泛型 + `T(v.x)` 显式转换，而非 `where T2 <: T`——正确反映了仓颉数值类型间不存在子类型关系的约束。

**[通过]** D31 正确识别了 fill-from-Vec1 构造器的重载冲突场景：同类型 `init(v: Vec1<T, Q>)` 与跨类型 `init<T2, Q2>(v: Vec1<T2, Q2>)` 在 `T2=T, Q2=Q` 时参数列表重合，仓颉 `where` 子句不支持否定约束无法排除。设计决策记录清晰准确。

**[通过]** `type` 别名定义 256 个向量类型别名符合仓颉类型别名语法（类型别名不引入新类型，仅为替代名称）。

**[通过]** `public import` 公共 API 重导出方案完全符合仓颉包机制。`internal` 可见性用于 Aligned 类型的设计正确。

### 2. 标准库与生态覆盖

**[通过]** 所有标准库使用均在合理范围内：`std.math` 的 `abs`、`trunc` 等函数，`NumericLimits` 虽为自定义实现（`shim_limits.cj`），但不依赖外部库。

**[通过]** `@Derive[Hashable]` 由 `std.deriving` 提供，仓颉核心数值类型（Int/Float/Bool）均已实现 `Hashable`，自动派生条件满足。

**[通过]** `@OverflowWrapping` 是仓颉内置注解（`std.reflect`），可直接标注于函数声明。

**[通过]** 无外部生态依赖假设——所有 Shim 层（`shim_assert`、`shim_limits`、`shim_cstddef`）均为自定义实现，不依赖 C++ 标准库。

**[通过]** `VArray<T, $N>` 及其 `size` 属性提供编译期长度查询——仓颉原生支持。

### 3. 语言特性可行性

**[通过]** 异常错误处理策略可行：`@OverflowWrapping` 处理整数溢出，除零依赖仓颉运行时行为（整数抛 `ArithmeticException`，浮点返回 `±Inf`），`assert` 使用 `if + throw` 模式。

**[通过]** 值类型无共享状态的设计天然线程安全——Vec 为 struct（值语义），所有算术运算产生新向量，无内部加锁需求。

**[通过]** "双层包结构"（`glm.detail` + `glm`）符合 cjpm 项目组织方式：`package glm.detail` 和 `package glm` 为子包关系，`public import` 做公共 API 重导出。

**[通过]** `const if (isIec559Of<T>())` 编译期分支方案在仓颉语言能力范围内——`const` 函数支持 `const if`（const README §3.2 规则 4），`is` 运算符是合法的 const 表达式（const README §2 条目 8），`T(0)` 对数值/Bool 类型为合法的 const 构造。

**[通过]** `@OverflowWrapping` 标注继承性——设计正确识别此为编译器未文档化行为，并提供了完整的备选方案（具名函数标注 + 运算符委托），设计层面的风险标注充分。

### 4. 设计一致性

**[通过]** 各抽象的职责描述清晰：Qualifier 体系负责精度/对齐标记，Vec 结构体系承载向量数据与运算，Functor/ComputeVec 为后续 SIMD 轮次预留，Shim 层替代 C++ 标准库设施。

**[通过]** 模块间依赖形成有向无环图：`setup.cj` 无依赖 → `qualifier.cj` 依赖 setup → `type_vecN.cj` 依赖 qualifier + compute_vector_relational → `lib.cj` 依赖 detail 包。`glm` 反向无依赖。

**[通过]** 范围可追溯性对照表（§8.2）将 roadmap 首轮范围逐项映射到设计章节和仓颉文件，偏离项附加说明。可指导编码阶段独立验证范围完整性。

**[一般]** D13（§7）与 §4.3 存在范围描述矛盾：D13 声明"对于需要浮点取模的场景，使用具名函数方案（**不属于首轮范围**）"，但 §4.3 已将 `mod` 列为 5 个首轮具名函数之一（`add/sub/mul/div/mod`），并给出了完整的浮点取模实现策略（算术恒等式 + `Float32`→`Float64` 提升）。同一个具名函数 `mod` 在 D13 中被标记为"不属于首轮范围"、在 §4.3 中被列为"属于首轮范围"，编码人员无法确定首轮实际交付范围。此矛盾自 v15 修订说明即被记录为"已修正"，但实际 D13 正文未更新。

**[轻微]** `increment()`/`decrement()`/`bitwiseNot()` 未被声明为 `const` 函数，但与已明确声明为 `const` 的 `add()`/`sub()`/`mul()`/`div()`/`mod()` 同为纯函数（不修改参数、返回新向量），缺少对称性理由。建议在 §7 补充设计决策统一约定 `const` 声明策略。

**[轻微]** D29 和 D30 分别定义了 `const` 表达式链验证路径和 `@OverflowWrapping` 继承性验证路径，但未分析两者同时失败（`const` 路径回退与 `@OverflowWrapping` 继承性不成立同时触发）的组合场景。建议新增组合回退影响段落。

**[轻微]** §3.8 定义了大写驼峰别名命名约定，但未评估下游开发者在 C++ GLM 文档（小写名，如 `bvec2`）与仓颉代码（大写驼峰，如 `BVec2`）之间的 mental mapping 长期认知成本。§11.7 仅覆盖了迁移成本，未覆盖"持续使用成本"。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：Qualifier 接口仅负责精度标记，Vec 结构体承载向量数据与运算，ComputeEqual 封装相等比较策略，shim 层封装 C++ 标准库替代。

**[通过]** 抽象层次恰当：不引入不必要的类型层次（如 storage 模板的省略理由充分），同时为后续 SIMD/对齐差异化预留了 Functor/ComputeVec 扩展点。

**[通过]** 设计便于实现：构造函数以仓颉语法完整列出签名清单，文档声明"编码阶段以本清单为准，无需参考 C++ 宏展开"。

**[通过]** 设计便于测试：验证层次四层结构覆盖编译验证、构造/访问、运算正确性和异常场景边界。测试包组织结构与源码包一一对应，internal 类型访问策略明确。

**[轻微]** §3.5 `epsilonOf<T>()` 兼容性策略段落仍使用"采取以下**方案之一**"的表述，但仅列出方案 A（内联固定容差），无方案 B。建议将"采取以下方案之一"改为"采用以下唯一策略"，移除"方案A"编号以避免编码人员误以为存在可选替代方案。

## 修改要求（REJECTED 时存在）

### 设计一致性问题

- **问题**：D13（§7）与 §4.3 对 `mod` 具名函数的首轮范围归属存在矛盾描述
- **原因**：D13 声明浮点取模具名函数"不属于首轮范围"（line 779），§4.3 将 `mod` 列为首轮 5 个具名函数之一并给出完整浮点取模实现策略（line 527），编码人员无法确定首轮实际交付范围
- **建议方向**：将 D13 中"不属于首轮范围"改写为"属于首轮范围"，消除自相矛盾
