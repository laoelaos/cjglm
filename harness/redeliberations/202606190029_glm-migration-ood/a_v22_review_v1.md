# OOD 设计方案审查报告（v22）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** `Qualifier` 接口 + 空结构体实现模式正确——仓颉接口可作为泛型约束，空结构体零开销。四个独立 Vec 结构体（Vec1~Vec4）而非单模板偏特化，符合仓颉不支持 C++ 偏特化的语言限制。

**[通过]** 泛型约束 `where Q <: Qualifier` 是仓颉标准泛型约束语法，使用正确。

**[通过]** 泛型 Vec 结构体使用 `T` 无紧约束（D5），符合仓颉的延迟检查语义，与 C++ 模板的 SFINAE 行为一致。

**[通过]** 跨 Vec 转换构造函数使用无约束 `T2` + `T(v.x)` 显式转换，绕过仓颉数值类型间无子类型关系的限制（D14），设计正确。

**[通过]** `@Derive[Hashable]` 自动派生在泛型 struct 上的约束传递性已正确解构：首轮所有目标类型均已实现 `Hashable`，非 `Hashable` 类型在实例化时产生编译错误（D5 一致）。

**[通过]** `extend` 块定义运算符和具名函数的语法格式正确（仓颉扩展 §2.3 形式 B + `where` 约束）。`extend<T, Q> VecN<T, Q>` 的泛型扩展语法已由文档确认支持。

**[轻微]** `logicalAnd`/`logicalOr` 定义为仅限 `VecN<Bool, Q>` 的扩展成员函数（`extend<Q> VecN<Bool, Q> where Q <: Qualifier`）——仓颉扩展文档仅明确展示"完全实例化"（`extend Foo<Int64>`）和"全泛型"（`extend<T> MyList<T>`）两种形式，未显式给出部分固定类型参数的混合形式。虽可通过规则推导认为 `extend<Q> Vec2<Bool, Q>` 合法（所有类型参数 Q 在被扩展类型中使用），但此语法未在文档示例中覆盖，建议在 §2.1 编码前验证检查清单中补充原型验证项，或改用通用扩展（`extend<T, Q> VecN<T, Q>` + D5 延迟检查）规避此不确定性。

### 2. 标准库与生态覆盖

**[通过]** `std.math.trunc` 浮点取模依赖已正确声明 §8.1 文件清单和 §10 验证依赖，import 路径标注清晰。

**[通过]** `std.math.abs` 容差比较路径依赖已在 §10 验证计划中覆盖。

**[通过]** `@Derive[Hashable]` 自动派生依赖 `std.deriving`，首轮目标类型均已实现 `Hashable`。

**[通过]** `std.unittest` 测试框架用于 §12 验证层次，测试目录结构设计与 `cjpm test` 自动发现机制兼容。

**[通过]** `NumericLimits<T>.epsilon()` 的 `Number<T>` 约束已在 §3.6 通过拆分 `NumericLimits` 和 `isIec559Of<T>()` 解耦，避免 `Bool` 等非数值类型的约束冲突。

**[通过]** `@OverflowWrapping` 溢出注解的正确使用已对照仓颉注解文档确认：可标注于函数声明、作用于函数内整数运算。`%` 在文档"不可溢出"列表中，`<<` 在"可溢出"列表中——设计与文档一致。

### 3. 语言特性可行性

**[通过]** 错误处理策略合理：`@OverflowWrapping` 标注控制整数溢出行为，无 `Option`/`Result` 场景。

**[通过]** 并发设计正确：Vec 为值类型，跨线程复制语义确保无共享写冲突。复合赋值非原子性的声明（§6）准确。

**[通过]** `const init` 构造函数的使用正确——仓颉 `const init` 可在非 const 上下文中作为运行时构造函数使用（const README §3.2 规则 5），且 `const init` 的存在使得 `const` 实例成员函数（如 `operator ==`）可以定义，从而支持编译期 `if` 分支。

**[通过]** 编译期 `if` 分支抑制行为已确认——`if`、`is`、数值算术表达式均在 const 表达式列表中（const README §2 第 7/8 项），设计假设成立。

**[通过]** `extend` 块不支持 `const` 修饰符已由仓颉扩展文档 §4.2 确认（允许的修饰符列表中不含 `const`）。设计正确处理此限制（D32/D33），所有扩展成员函数声明为非 `const`。

**[通过]** `operator[]` 取值形式无法区分 `const`/非 `const` 重载的问题已正确识别（const README §3.2 规则 7），通过另提供 `componentAt` const 具名函数解决。

**[通过]** 模块/包结构设计符合 `cjpm` 项目组织方式：`package glm.detail` 和 `package glm` 两个包，`lib.cj` 通过 `public import` 重导出。目录结构与 `src-dir` 配置兼容。

**[通过]** 编码前验证清单（§2.1）覆盖关键语言特性假设：const 表达式链、`is` 在 const 泛型中的可用性、`@OverflowWrapping` 继承性、编译期 `if` 分支抑制等。

### 4. 设计一致性

**[通过]** 各抽象职责清晰：Qualifier 体系（精度标记）、Vec 结构体系（向量值类型）、Functor 体系（函数映射工具）、ComputeVec 运算策略（SIMD 预留点）、ComputeEqual（相等比较策略）、Shim 层（C++ 标准库替代）。职责不重叠。

**[通过]** 协作关系形成闭环：Vec 运算符 → (依赖) ComputeEqual + 原生运算 + scalar_vec_ops → (依赖) qualifier + shim。依赖方向合理，无循环依赖（§2 依赖拓扑图正确）。

**[通过]** v22 修订完整处理了审查发现 1~4：`@OverflowWrapping` 标注分布表已补充 scalar-op-vec 方向 20 个函数（发现 1）；`scalar_vec_ops.cj` 已在 §8.2 新增显式映射行（发现 2）；§4.1 已补充重载解析期望注解（发现 3）；`epsilonOf<T>()` 的两条显式生命周期路径已在 §3.5 明确归档（发现 4）。

**[通过]** 跨章节引用一致：§4.6 标注分布表与 §4.3 具名函数定义互引；§3.5 兼容性策略与 §10 依赖④、§12.2 验收标准联动修改已标注。

**[通过]** 模块间依赖方向单一（glm → glm.detail），无反向依赖。

**[轻微]** §8.2 新增的 `scalar_vec_ops.cj` 映射行位于序号 5 与序号 6 之间，但 §8.1 文件清单中序号 8a (`scalar_vec_ops.cj`) 出现在序号 8 之后。两表序号顺位不一致，建议统一序号排列以降低实施者的交叉查阅认知负担。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：每个文件有明确的关注点（`setup.cj` 仅配置常量、`shim_cstddef.cj` 仅类型别名、`scalar_vec_ops.cj` 仅方向辅助函数等）。

**[通过]** 抽象层次恰当：Functor/ComputeVec 首轮仅定义不消费（为后续 SIMD 轮次预留），不引入不必要的实现复杂度。方向辅助函数独立为 `scalar_vec_ops.cj` 而非混杂于 `compute_vector_decl.cj`，层次清晰。

**[通过]** 设计便于后续实现：§8.3 定义了增量迁移顺序（分段 + 验证节点），§12.1 定义了从编译到边界验证的层次化测试策略。

**[通过]** 可测试性良好：测试文件与源码一一对应（§8.1 测试清单），公共 API 和内部实现分离测试。`internal` 可见性的跨包测试策略已在 §12.3 标注为待验证假设。

**[通过]** 风险管理和回退路径完备：D29（const 表达式链回退）、D30（`@OverflowWrapping` 继承性回退）、D31（fill-from-Vec1 重载冲突约束）等均包含明确回退方案。§2.1 编码前验证清单对每项验证均定义"失败回退"列。

**[通过]** 256 个别名的外部脚本生成策略（§3.8）和缩编降级路径（含触发条件、审批路径、补全期限）设计完备。
