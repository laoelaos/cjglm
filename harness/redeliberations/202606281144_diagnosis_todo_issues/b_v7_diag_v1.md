### 问题 2：G2.1 `spin: Bool` → `k: Int64` 修复方案的迁移成本未评估

- **问题描述：** §G2.1 准确识别 `slerp(x, y, a, spin: Bool)` 签名与 OOD §3.11、D22、§11.5 三处承诺的 `k: Int64` 不一致，分类为「真实存在」。但分析段未提供修复方案的迁移成本评估：
  - **现有调用点扫描缺失：** 报告未说明是否存在 `slerp(x, y, a, spin)` 的调用点。修复时若存在调用方，需同步修改调用代码（`true`/`false` 改为 `0`/`1` 或类似语义），可能引入接口级联破坏。
  - **命名歧义消除的语义验证缺失：** `spin` 与 `k` 的语义不完全等价——GLM 原版 `k` 允许任意整数（含负数、零）控制旋转圈数（每整数对应 π 弧度偏移）。`Bool` 仅支持 true（单圈）/ false（无旋转）。修复时需明确 Cangjie 实现的语义对齐方向（建议直接按 OOD 与 GLM 采用 `Int64` + GLM `phi = angle + k * pi<T>()` 公式）。
  - **stub 状态修复验证手段缺失：** 当前函数体 `throw Exception("stub")`，修复签名后无编译错误，但缺乏运行时验证用例（G5.6 已记录该测试缺失）。建议在修复签名的同时新增最小测试用例覆盖 `k = 0` / `k = 1` / `k = -1` 三种典型场景，避免阶段四补齐时签名修复正确性无法回归。
- **所在位置：** a_v7_diag_v1.md §G2.1 分析段（行 259-274）
- **严重程度：** 中（影响修复可操作性——执行者需自行评估迁移成本与测试补齐策略）
- **改进建议：** 在 G2.1 修复建议中追加：
  1. "执行 `grep -rn 'slerp.*spin\|slerp(.*,.*,.*,.*)' cjglm/` 扫描现有 4 参数调用点"
  2. "修复签名后新增测试用例 `testSlerp4ArgsK0/K1/KMinus1` 验证 `k` 参数语义对齐 GLM"
  3. "明确语义对齐：保留 GLM 原版 `phi = angle + k * pi<T>()` 公式（`k=0` 即 3 参数 lerp 等价，`k=1` 旋转一周，`k=-1` 反向旋转）"
