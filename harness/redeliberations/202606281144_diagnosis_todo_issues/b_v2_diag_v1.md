## 发现的问题

### Q1. S3 优先级依赖关系描述存在逻辑矛盾（逻辑矛盾）

**位置**：§修复优先级建议（第 532 行）

**问题描述**：优先级表中 S3 的"理由"列写"需先修复测试才能验证 S1 修复"，但依赖关系列写"测试依赖 S1 修复结果"。二者自相矛盾：

- 若 S3 理由成立（测试先修），则应标记为"测试修复 → S1 验证"的依赖方向
- 若依赖关系成立（S1 先修），则 S3 的"理由"描述方向相反

实际工程关系是：S1 是代码 bug，可独立修复；S3 是测试方法论的薄弱问题（过于依赖 round-trip）。S1 修复后 S3 的"bug 无法被捕获"诉求自然消失，剩下的是测试增强建议。当前表述既不符合逻辑，也无法指导执行者决定先改哪个。

此外，实际代码验证显示：`test_type_quat_cast.cj:56-62` 对 `(0.2, 0.3, 0.4, 0.8)` 的 round-trip 测试在 S1 bug 存在时本应失败（quatCast 返回错误值会导致 mat3Cast 结果不同），只是由于 S2（测试不被发现）导致该测试从未执行。S3 的"可能掩盖"描述偏保守——对报告自身用作例证的 `(0.2, 0.3, 0.4, 0.8)` 输入，round-trip 可直接检测该 bug。S1 修复后 S3 的优先级别应主动下调至 Low/Medium。

**严重程度**：高 — 逻辑矛盾影响修复者对依赖关系的理解，可能导致错误的工作顺序决策

**改进建议**：
- 删除 S3 的"需先修复测试才能验证 S1 修复"理由，改为"round-trip 测试方法论薄弱，对某些输入（如单位四元数）无法检测该 bug，应补充『旋转矩阵×向量 = 四元数×向量』等价性测试"
- 在 S1 修复的上下文中标注：S1 修复后 S3 主诉求消除，剩余测试增强建议优先级降为 Low

### Q2. G3.5 `fwd.cj.bak` 行号引用不精确（事实错误）

**位置**：§G3.5（第 287 行）

**问题描述**：报告写"`fwd.cj.bak:330-332` 包含 OOD §2 明确禁止的 `HighpFQuat`/`MediumpFQuat`/`LowpFQuat` 三个错误别名"。经核查 `fwd.cj.bak`：
- `HighpFQuat` 在第 331 行 ✓
- `MediumpFQuat` 在第 334 行 ✗（超出 330-332 范围）
- `LowpFQuat` 在第 337 行 ✗（超出 330-332 范围）

行号范围 `330-332` 仅能覆盖 `HighpFQuat`（331 行），其他两个不在该范围内。该引用继承自 todo.md 的原始行号错误，报告在交叉验证时未校正。

**严重程度**：低 — 行号范围偏窄但不影响对问题的理解，通过分号名称可精确定位

**改进建议**：将行号引用改为 `:331, 334, 337` 或 `:330-338`，并在交叉验证时校正从原始 todo.md 继承的引用错误

### Q3. G3.5/G3.6 分类与需求定义的 4 类分类框架不一致（分类一致性问题）

**位置**：§G3.5（分类：真实存在）、§G3.6（分类：真实存在）

**问题描述**：requirement.md 定义 4 类分类：
1. 真实存在 — 实现与设计/参考实现对比存在偏差
2. 误报
3. OOD 文档问题
4. 其他类型

G3.5（`fwd.cj.bak` 备份文件被误纳入仓库）属版本控制问题，G3.6（`gen_fwd_aliases.py` 在 Windows 上 CRLF/LF 幂等性问题）属构建脚本兼容性问题。二者均不属于"实现与设计/参考实现对比存在偏差"——没有"设计"说 `.bak` 文件不该被跟踪，也没有"参考实现"的脚本在新行上不一致。它们更适配"其他类型的问题"分类。

报告自身在 G3.6 的 deviations.md 对照中也承认"本问题属构建工具兼容性问题，非 C++ GLM 与仓颉 GLM 之间的语义偏差，不涉及 deviations.md"，证明分类者自己也意识到这不是"实现偏差"。

**严重程度**：中 — 分类不一致影响统计口径和读者对问题性质的第一印象

**改进建议**：将 G3.5、G3.6 的分类调整为"其他类型（版本控制/构建兼容性问题）"

### Q4. S2/S1 双 Blocker 优先级不直观，特殊说明与表格分离（可操作性缺陷）

**位置**：§修复优先级建议（第 529-530 行 + 第 544 行特别说明）

**问题描述**：优先级表将 S1 和 S2 均列为"Blocker"（同一级别），但在特别说明中写"S2 应为最高优先级"。表格层级与说明存在矛盾：

- 若 S2 确实是最高优先级（修复基础设施 -> 再修其他），则 S1 不应同为 Blocker 级别，应降为 High（修复后可验证性依赖 S2）
- 若 S1 和 S2 同为 Blocker（即必须全部完成后才能推进），则应明确二者并行/串行关系

当前的双 Blocker + 特殊说明模式，修复者需要阅读两个分散的位置才能理解真实优先级，降低可操作性。

**严重程度**：中 — 影响执行者第一眼对工作顺序的判断

**改进建议**：二选一方案：
- 方案 A：S2 单列为最高一级（如 Critical/Blocking），S1 降为 High，在 S2 备注中标注"S1 修复后可同步验证"
- 方案 B：保留双 Blocker，但在表格中增加"修复顺序"列，标注"Step 1"和"Step 2"

### Q5. 未讨论 S1 修复对 S3 紧迫性的影响（深度/完整性不足）

**位置**：§S3 分析 + §修复优先级建议

**问题描述**：S3 的核心诉求是"round-trip 测试无法捕获 S1 bug"。若 S1 被修复（quatCast 算法正确），该诉求自动消失——round-trip 测试将自然通过。S3 剩余价值仅为"测试方法论薄弱，建议补充等价性测试以增强回归检测能力"，优先级应从 High 降至 Low。

报告在优先级表中将 S3 列为 High，且独立于 S1 讨论，未标注"若 S1 已修复则本项价值衰减"。这导致修复者可能同时投入修复 S1 和 S3（做双份工作），或先修复 S3（改进测试），再修复 S1（bug），结果 S3 改进的测试在 S1 修复后部分失去目的。

**严重程度**：中 — 可能导致无效工作量投入

**改进建议**：
- 在 S3 分析末尾增加注："若 S1 修复完成,本条目的主诉求（测试不能捕获 bug）消除,剩余测试方法论改进建议可降为 Low 优先级"
- 优先级表中 S3 标注条件性降级标注

### Q6. "仓库中仅包含 03/04/05 阶段文档"表述精度不足（事实表述瑕疵）

**位置**：§概述（第 13 行）

**问题描述**：报告的交叉验证依据段落写"经确认仓库中仅包含 03/04/05 阶段文档"。`docs/` 目录实际包含：`01_tech_decision.md`、`02_roadmap.md`、`03_ood_phase1.md`、`04_ood_phase2.md`、`05_ood_phase3.md`。若"阶段文档"特指 OOD 阶段设计文档（ood_phase*），则陈述正确。但无修饰语的"阶段文档"可能被读者理解为"docs/ 下全部文档"，与实际情况偏差（遗漏 01 和 02）。

上一版（`a_v2_diag_v2.md:15`）的表述更精确："经确认仓库中仅包含 `docs/03_ood_phase1.md`、`docs/04_ood_phase2.md`、`docs/05_ood_phase3.md` 三份阶段设计文档"——通过列出全路径文件名消除了歧义。本版压缩为"03/04/05 阶段文档"后精度下降。

**严重程度**：低 — 不影响实质理解，但在第 2 轮审查中精度下降属于非预期退化

**改进建议**：恢复为 v2 的精确表述，逐份列举全路径文件名

## 整体评价

报告在 v1 基础上显著改进：修复了计数错误、G1.6 自相矛盾、deviations.md 交叉验证不足、汇总表不一致、缺乏优先级排序等 8 个上一轮发现的问题。分类加权计数口径统一、数学一致，优先级表新增提供了显著的操作指导价值。

存在 6 个质量问题，其中 Q1（S3 依赖关系逻辑矛盾）和 Q5（S1 修复对 S3 紧迫性影响未讨论）涉及分析深度不足和逻辑一致性——修复者拿到报告后可能做出错误的工作排序决策。Q4（双 Blocker 优先级不直观）可操作性有提升空间。Q2/Q3/Q6 为轻微问题。

建议修复者优先处理 Q1（调整 S3 优先级理由中的逻辑矛盾）和 Q5（补充 S1→S3 条件性降级标注），其次修正 Q3（G3.5/G3.6 分类）、Q4（优先级表结构优化），最后处理 Q2/Q6 的表述精度。
