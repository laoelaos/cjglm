## 2. 发现的问题

### 问题 1：Section 3C 与 Section 3D 别名计数不一致（中等严重程度）

- **问题描述**：Section 3C（第 179 行）称"约 100+ 个 type alias 定义"，而 Section 3D（第 189 行）称"约 256 个类型别名"。同一份产出中两个数字矛盾。根据 Section 3C 自身列举的 16 个别名家族 × 4 个向量维度 × 4 个精度变体（base + lowp + mediump + highp），总数应为 256。v10 修订说明承诺"Section 3D 中将'约 100+ 个'修正为'约 256 个'"，但 Section 3C 的内容未被同步更新，属残留错误。
- **所在位置**：`a_v11_output_v1.md` 第 179 行（Section 3C 末段）
- **改进建议**：将 Section 3C 中的"约 100+ 个 type alias 定义"改为"约 256 个 type alias 定义"，与 Section 3D 计数保持一致。

### 问题 2：Section 5 依赖关系图缺少首轮辅助文件节点（轻微严重程度）

- **问题描述**：首轮范围包含 `compute_vector_relational.hpp`、`compute_vector_decl.hpp`、`_vectorize.hpp` 三个辅助文件（Section 3D 表格已计入），但 Section 5 的依赖关系图中未出现这三个文件的节点。虽然图下方的文本注中有说明，但图形本身不具备独立完整性——使用者若仅阅读依赖图，无法获知这些文件在依赖拓扑中的位置及其与 vec 类型的关系。
- **所在位置**：`a_v11_output_v1.md` 第 366-381 行（Section 5 依赖关系图）
- **改进建议**：在依赖图中补充三个辅助文件的节点，标注其与 `setup.hpp` 及 vec 类型的关系，例如：
  ```
  setup.hpp
     │
     ├── qualifier.hpp
     │      ├── ...
     │      └── vec<N,T,Q>
     │             │
     │             ▼ (via .inl)
     │      compute_vector_relational.hpp ─── setup.hpp
     │      compute_vector_decl.hpp ─── setup.hpp → _vectorize.hpp
  ```

### 问题 3：Section 0 内容超出"只确定范围"边界（轻微严重程度）

- **问题描述**：需求明确约束"只确定范围，不做具体设计和实现"。Section 0（目标语言能力假设）中每项假设的"不成立时的调整方向"列给出了具体的实现策略（如"以普通方法（如 .add()、.mul()）提供"、"用结构体外包装模拟"、"需在首轮迁移中同步创建对应的 shim 类型"等），这些内容已超出范围确定的边界，进入了设计/实施领域。
- **所在位置**：`a_v11_output_v1.md` 第 1-16 行（Section 0 表格的"不成立时的调整方向"列）
- **改进建议**：将"不成立时的调整方向"列缩减为"对首轮范围的影响"的补充说明，移除具体的实现策略描述；或将实现细节移至附录，正文保持纯范围界定。例如，将"以普通方法（如 .add()、.mul()）提供"改为"首轮验收标准需调整为非运算符形式的等价操作验证"。

## 3. 深度与完整性总评

产出深度较高，覆盖了：
- 类型依赖关系的定量分析（直接引用数统计）
