## 1. 需求响应充分度

### 1.1 对 roadmap 首轮范围的显式引用充分，但缺少范围自含性

**问题描述**：§8.2 范围可追溯性对照表依赖对 roadmap §3E/§3G 的交叉引用，设计文档自身未包含首轮范围的完整自含定义。编码实施者在需要确认"某项功能是否属于首轮范围"时，须同时翻阅 roadmap 文档和本设计文档。对独立编码实施场景（如实施者不熟悉 roadmap 结构），此依赖增加认知负担和遗漏风险。

**所在位置**：§1 范围可追溯性承诺段落、§8.2 范围可追溯性对照表

**严重程度**：一般

**改进建议**：在 §1 概述中补充首轮范围的完整文字摘要（而非仅引用 roadmap 章节号），或单独以表格形式列出首轮交付的全部功能模块清单及边界。§8.2 的映射表保留不变，但增加"快速确认某功能是否为首轮范围"的独立参考段落。

## 2. 事实错误与逻辑矛盾

### 2.1 `fromBoolVecQ2` 编译器推断依赖的级联影响分析不完整

**问题描述**：§4.8 标注了 `fromBoolVecQ2` 函数体中 `Vec1(...)` 调用依赖编译器从返回值类型推断 struct 泛型实参（`[^compiler-dep]`）。但该依赖若失败，其级联影响远不止于"显式标注类型参数"的 8 处代码修改：① `fromBoolVecQ2` 的返回值类型本身 `VecN<T, Q>` 中的 `Q` 参数需要推断（函数返回类型为 `Vec1<T, Q>`，但构造函数调用未显式指定 `<T, Q>`）；② 若编译器不支持返回值类型推断，`fromBoolVecQ2` 的 API 签名可能需要重新设计（改为要求调用方显式传入目标 Vec 类型引用作为 receiver），影响 §4.8 中"包级独立函数"的设计决策基础；③ `fromBoolVec`（同 Q 版本）不受此影响，但编译阶段若发现跨 Q 版本不可行，同 Q 版本和跨 Q 版本的对称 API 面将被打破。§10 的验证依赖表中未覆盖此级联风险。

**所在位置**：§4.8 `[^compiler-dep]` 脚注段落、§10 `Bool→Numeric` 转换路径 B 验证表

**严重程度**：严重

**改进建议**：① 在 §10 验证依赖表中新增独立验证项，明确验证编译器从返回值类型推断 struct 泛型实参的行为；② 在 §11 迁移成本评估中新增本依赖失败时的回退成本评估（含 API 重新设计的连锁影响）；③ 在 §4.8 补充设计备选方案（如为 Vec1~Vec4 各增加一个辅助函数 `fn fromBoolVecOf<T, Q>(v: VecN<Bool, Q>): VecN<T, Q>` 将返回值绑定到具名变量，绕过推断依赖）。

### 2.2 P37 其余 7 文件风险等级评估缺乏系统性依据

**问题描述**：§1 P37 状态中为其余 7 个核心文件标注了风险等级（setup.hpp→中、qualifier.hpp→中、其余 5 文件→低），并给出"整体评估：无高等级风险文件"的结论。但该评估是设计者的主观判断，而非基于系统性 API 逐项对照的结构化分析。例如：
- setup.hpp 的"中"风险标注依据为"1000+ 行平台/编译器检测宏首轮不迁移，18 个 GLM_CONFIG_* 常量已显式映射，依赖已知可替代"——但"依赖已知可替代"的确认方式未说明（是阅读了所有 type_vecN.inl 中对 setup 的引用？还是抽样评估？）。
- 编码阶段实施者参考此风险评估做编码计划时，如果某文件的"低"风险评估因遗漏依赖而证伪，将打乱编码阶段的资源分配和工期估算。

**所在位置**：§1 P37 状态（第 95 行）"其余 7 个核心文件的覆盖风险等级如下"段落

**严重程度**：一般

**改进建议**：① 明确标注每个文件风险等级的方法论来源（如"经阅读该文件全部公开 API 并与当前设计中引用的依赖项交叉核对"或"经随机抽样 5 处引用确认无遗漏"），而非仅有判断结论；② 在 §12.2 新增的"7 文件编码启动前快速扫描"验收项中，要求扫描结论与当前风险等级标注进行对照验证，如发现偏差则升级风险标注。

### 2.3 文档中"已验证"与"待验证"的边界在个别配置项上不够清晰

**问题描述**：§7 D6 中对 `GLM_CONFIG_CLIP_CONTROL` 的标注为"**默认值来源已验证（参考实现确认）**"，并描述了确认过程（`setup.hpp:563-584` 三层条件编译）。但该验证是在设计阶段对参考实现源码的阅读确认，而非编码阶段的编译器行为验证或运行时验证。文档声称"已验证"但实际是"人工阅读参考实现源码后推断确认"，两者在可复现性和严谨性上存在差距。§8.4 前置验证项①中也要求"编码前必须读取 `setup.hpp:563-584` 的原始条件编译块，逐行确认"——这进一步说明此验证应在编码阶段由实施者完成而非设计阶段已确认。§1 P7 的状态标注为"✅ 已解决（v11）— roadmap 已同步，默认值一致为 `0x0A`"，但"值一致"和"值正确"是两个不同概念——两文件一致仅说明无冲突，不证明值正确。

**所在位置**：§7 D6 默认值说明段落"**默认值来源已验证（参考实现确认）**"、§1 P7 状态行、§8.4 前置验证项①

**严重程度**：一般

**改进建议**：将 §7 D6 中"已验证"的措辞统一为"经参考实现源码阅读确认，待编码前验证项②做最终确认"，与 §8.4 的验证时序对齐。在 §1 P7 状态中补充"值正确性依赖于编码前参考实现验证项②的通过"的依赖声明。

## 3. 深度与完整性

### 3.1 验证项影响范围分类的数量描述可能误导决策

**问题描述**：§2.1 末尾"验证项影响范围分类"表标注高影响项 4 项（①③④⑫）、中影响项 9 项、低影响项 11 项。其中项⑫（统一集成验证）被归类为高影响项，且项⑫本身是项①③④⑤⑧等独立验证项的集成——在"设计确定性声明"框中被用作风险评估的核心定量依据。但项⑫的失败并不意味着 4 项高影响项全部失败——集成验证失败可能是因为其中某一项独立验证已通过但在组合场景中失败。这种"4 项高影响"的表述可能让决策者误以为仅有 4 项假设存在高风险，而忽略项⑫涵盖的多个依赖点的独立性。

**所在位置**：§2.1 "验证项影响范围分类"表、"设计确定性声明"框（§1）

**严重程度**：一般

**改进建议**：① 在影响范围分类中补充说明项⑫与项①③④⑤⑧的"覆盖而非取代"关系，避免数量统计上的重复计数误导；② 在"设计确定性声明"框的量化说明中，将"24 项验证全部通过"的表述改为"全部验证通过"，弱化数字的暗示性。

### 3.2 删除的 `equalEpsilon` 路径①备选方案残留引用未被完全清除

**问题描述**：v11 修订说明（条目 7）声明将 §11.14 中路径①的成本评估完全移除。但 §4.5 仍保留了对路径①的描述（标注为"已废弃"），§11.14 中路径①的删除声明与 §4.5 的保留描述形成不一致。虽然逻辑上不矛盾（§4.5 保留描述作为设计记录、§11.14 移除成本评估作为经济决策），但这种跨章节的不一致增加了实施者的认知负担——实施者阅读 §4.5 时看到路径①的描述可能误认为它仍是可选方案，直到查阅 §11.14 才发现已被明确禁止。

**所在位置**：§4.5 "⚠ const 真空警示"中的路径①描述、§11.14 路径①废弃声明

**严重程度**：轻微

**改进建议**：在 §4.5 路径①描述末尾增加一行引用标注"此路径已在 §11.14 中被正式废弃并移除成本评估，编码阶段严禁预留工作量"；或在 §4.5 路径①标题旁标注"❌ 已废弃——详见 §11.14"。

### 3.3 验证清单三处分布消重仍不完全

**问题描述**：v11（条目 5）声称对 §10 和 §12.3 做了进一步消重，但 §10 仍保留了多个结构化验证依赖表（"equalEpsilon 设计阶段验证"表、"Bool→Numeric 转换路径 B 可行性验证"表、"Vec4 构造函数重载解析验证"表等），这些表的格式和内容与 §2.1 高度重叠。虽然各表的"验证方法"列已改为引用 §2.1 项编号，但表本身的组织方式（依赖项×风险等级×对应检查项的矩阵结构）在 §2.1 中已有等价表达，属于形式而非内容的冗余。维护者在后续轮次修改 §2.1 时，仍需同步修改 §10 中对应表的引用列。

**所在位置**：§10 "equalEpsilon 设计阶段验证"子节完整内容、"Bool→Numeric 转换路径 B 可行性验证"子节、"Vec4 构造函数重载解析验证"子节

**严重程度**：一般

**改进建议**：将 §10 中所有验证依赖表的内容进一步压缩为单行引用文本（如"equalEpsilon 所有验证依赖项均由 §2.1 项⑫⑲㉒覆盖，详见对应项的回退方案列"），彻底消除表格式冗余。仅在 §10 保留设计特有的行为描述和回退方案依赖关系图。

### 3.4 迁移成本合计表缺少 TOTAL 编码总工作量

**问题描述**：§11.12 底部明确声明"不包含基础编码工作量（Vec 结构体和运算符的仓颉实现）"和"测试编写工作量"。作为一个旨在"满足后续编码实现需要"的 OOD 设计，缺少总编码工作量估算使得项目规划者无法从本文档获得完整的工期评估。虽然增量/迁移成本数据有其价值，但没有"基准线"的工作量数据让总成本变成一个黑盒——项目规划者必须自行估算基础编码成本并累加。

**所在位置**：§11.12 合计表底部注脚

**严重程度**：一般

**改进建议**：① 在 §11.12 之前新增一个"基础编码工作量评估"子节，对 Vec 结构体定义、运算符实现、别名脚本、shim 文件、测试编写等基础编码工作给出估算范围；② 在合计表末尾增加"基础编码"行和"总计（含基础编码）"行；③ 如因不确定性不愿意给出估算，应明确标注估算责任人（如"基础编码成本由实施团队在编码阶段启动前自行评估"）而非完全留空。

## 4. 编码落地可行性

### 4.1 24 项验证中有多项编码与执行依赖关系需人工管理

**问题描述**：§2.1 的验证时序建议（优先验证第⑫→⑮→①→④→⑤项）虽给出了执行顺序建议，但实际验证项之间存在更复杂的"验证结果→策略选择→编码实施→下一项验证"的依赖链。例如：项⑲（`T(1e-6)` 对整数类型可用性）的结果直接影响 §3.5 的路径 A/B 选择，而路径选择又影响 `equalEpsilon` 的实现和测试阈值。当前文档将验证项描述为相互独立的离散任务，未显式建模这些依赖链为"验证-决策-实施"的 DAG。编码实施者（可能非文档作者）在逐项执行时容易遗漏"在验证项 A 完成后，需要先查阅决策映射表选择路径，修改对应章节的代码，然后才能进入下一项验证"这一循环。

**所在位置**：§2.1 编码前验证检查清单、§12.3 决策映射表

**严重程度**：一般

**改进建议**：在 §2.1 验证清单前新增一个"验证-决策-实施 循环"的说明段，明确每项验证的产出不仅是"通过/失败"二元结果，还包括"根据结果实施回退方案后重新验证"的环节。考虑采用 Mermaid 流程图可视化最关键的依赖链（特别是项⑫→路径选择→多项代码修改→项④⑲㉒ 等）。

## 5. 整体评价

本 v11 产出经过 11 轮迭代审议，整体质量较高：架构设计完整、API 定义精确、回退方案全面、迁移模式细致。以上指出的 8 个问题中，2 个为严重级别（2.1 `fromBoolVecQ2` 编译器推断依赖的级联影响、4.1 验证-决策-实施循环的依赖管理），其余为一般/轻微级别。不存在直接导致设计不可用的根本性问题。

**建议优先修复**：问题 2.1（`fromBoolVecQ2` 级联影响分析）——因为该问题涉及编译器行为假设，若验证失败将影响 §4.8 API 设计的完整性，且当前分析未能覆盖回退方案的连锁影响。
