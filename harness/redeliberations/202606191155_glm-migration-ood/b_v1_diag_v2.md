## 二、问题清单

### 问题 1（严重）：§7 D6 中 `GLM_CONFIG_CLIP_CONTROL` 默认值推导描述与参考实现实际位定义不一致

**所在位置**：§7 D6 `GLM_CONFIG_CLIP_CONTROL` 常量说明段落（~第 1213 行）

**问题描述**：文档声称默认值为 `0x0A`，推导为"RH_BIT(8) | NO_BIT(2) = 0x0A"。经查阅参考实现 `references/glm-1.0.3/glm/glm/detail/setup.hpp:563-566`，实际位定义为：

```c
#define GLM_CLIP_CONTROL_ZO_BIT    (1 << 0)  // = 1
#define GLM_CLIP_CONTROL_NO_BIT    (1 << 1)  // = 2
#define GLM_CLIP_CONTROL_LH_BIT    (1 << 2)  // = 4
#define GLM_CLIP_CONTROL_RH_BIT    (1 << 3)  // = 8
```

默认路径（无 `GLM_FORCE_DEPTH_ZERO_TO_ONE`、无 `GLM_FORCE_LEFT_HANDED`）走 `#else`→`#else` 分支（第 582-583 行），结果为 `GLM_CLIP_CONTROL_RH_NO = RH_BIT(0x08) | NO_BIT(0x02) = 0x0A`。**最终值 `0x0A` 正确**，但文档中的推理表达有歧义——"RH_BIT(8)"的括号内容读者可能理解为位编号而非位值，与 C++ 源码中"RH_BIT = 1 << 3 = 8"的位编号-位值对应关系不清晰。

**更关键的问题**是 roadmap §3A.1 记录该值为 `0x02`，但本文档声称"已验证正确"却未实际引用验证过的源码位定义。文档声称 `0x0A` 是经过分析得出的结论，但在编码前验证项②（§8.4 策略 0）需要重新验证之前，此断言仍属"经推导推断但未交验"状态。文档将此推导标记为"已验证正确"（第 1213 行："文档值 0x0A 已验证正确"），这在措辞上超前于实际验证状态。

**严重程度**：严重

**改进建议**：
1. 将"文档值 0x0A 已验证正确"改为"文档值 0x0A 经参考实现条件编译逻辑推导，待编码前验证项②确认"，明确区分"推导结论"与"已验证结论"。
2. 在推导说明处补充参考实现的位定义值（`RH_BIT = 1 << 3 = 8`, `NO_BIT = 1 << 1 = 2`），消除位编号与位值的歧义。
3. 在 §8.4 前置验证项①中明确：编码前必须读取 `setup.hpp:563-584` 的原始条件编译块，逐行确认分支走向与 `0x0A` 推导一致。

### 问题 2（中等）：标量别名与向量别名的精度前缀命名风格不一致，文档未做任何说明

**所在位置**：§3.7 标量类型别名表 vs §3.8 向量类型别名命名约定

**问题描述**：标量别名使用小写下划线精度前缀（如 `highp_int8`、`mediump_int8`、`lowp_int8`），而向量别名使用 PascalCase 精度前缀（如 `HighpBVec2`、`MediumpBVec2`、`LowpBVec2`）。两种命名风格在同一个包（`package glm`）中共存，但文档既未解释此不一致的原因，也未评估其导致的认知成本。

例如，`public type highp_float = Float32`（标量）和 `HighpVec2`（向量）使用了完全不同的精度前缀形式。消费者需要在"标量用小写+下划线、向量用 PascalCase 拼接"之间频繁切换。

**严重程度**：中等

**改进建议**：
1. 在 §3.7 末尾或 §3.8 开头增加"标量与向量命名风格差异"说明段落，显式声明此不一致是有意保持 GLM 源兼容性（C++ GLM 中标量别名确实使用 `highp_int8` 形式），还是设计遗漏需要统一。
2. 若需保持 GLM 源兼容性，在 §11.7 迁移映射表中补充标量别名风格差异的提示。

### 问题 3（中等）：`equalEpsilon` 对 `VecN<Bool, Q>` 的编译错误未在迁移成本中覆盖

**所在位置**：§4.5 `equalEpsilon` 定义、§11 迁移成本评估

**问题描述**：文档正确指出 `equalEpsilon` 在 `VecN<Bool, Q>` 上实例化时产生编译错误（因 `Bool` 不支持 `-` 运算符）。然而 §11 的迁移成本评估完全没有涉及以下场景：如果下游 C++ 代码使用了 BVEC 上的等价比较（或 C++ 中 `bvec` 的逐分量 `==` 迁移到仓颉后需要 `equalEpsilon` 的场景），实施者将面临无法预料的编译失败，且无迁移指引可依循。

§12.1 层次三虽然包含了 Bool Vec 的编译错误测试，但该测试仅验证"确实报错"，未提供当此错误发生时如何修复的指引。

**严重程度**：中等

**改进建议**：
1. 在 §11 迁移成本评估中新增子节（如 §11.15），专门覆盖 Bool Vec 相关 API（`equalEpsilon`、`increment`/`decrement`、`%` 等）在迁移中遇到编译错误的处理模式。
2. 在 §4.5 `equalEpsilon` 定义附近增加"Bool Vec 迁移指南"提示框，说明遇到 `equalEpsilon` 在 Bool Vec 上编译失败时，应检查原代码逻辑并替换为 `==` 或 `logicalAnd`/`logicalOr` 等。

### 问题 4（中等）：Vec4 构造函数重载解析验证覆盖范围不足

**所在位置**：§4.1 重载解析风险描述（~第 808-814 行）、§10 Vec4 重载解析验证（~第 1901-1913 行）

**问题描述**：文档将风险聚焦在"6 组对偶重载分支"（`Vec3+T` vs `Vec3+Vec1` 等），但 Vec4 共有约 30 个构造函数签名。以下两类潜在歧义场景未被验证计划覆盖：

1. **多元 Vec1 参数组合的内部歧义**：`init<T2, Q2>(a: Vec1<T2, Q2>, b: Vec1<T2, Q2>, c: Vec1<T2, Q2>, d: Vec1<T2, Q2>)` 与 `init<T2, Q2>(a: Vec1<T2, Q2>, b: Vec1<T2, Q2>, c: Vec1<T2, Q2>, d: T)` 在 `T = T2` 时，第 4 参数 `Vec1<T, Q>` 与 `T` 是否可区分？
2. **跨 Q 构造函数的 Q2=Q 特化冲突**：当用户传入的参数 Q2 恰好等于 Vec 的 Q 时，泛型构造函数与同 Q 构造函数可能重合。

验证计划仅覆盖了明确标识的 6 组对偶分支，未制定系统性扫描策略（如按参数数量分组、组内按参数类型结构相似度排序后逐一测试）。

**严重程度**：中等

**改进建议**：
1. 在 §10 Vec4 重载解析验证中，将验证方法从"覆盖 6 组对偶分支"升级为"按参数数量分组的全扫描"——对每组参数数量相同的构造函数，逐一测试其区分度。
2. 对 §4.1"已知安全场景"补充反例测试——编写原型测试确认文档声称"安全"的场景确实无歧义。
3. 考虑在编码前验证清单中新增统一的 Vec4 构造函数全量编译验证项。

### 问题 5（中等）：§2 模块依赖图中 `type_vecN → scalar_vec_ops` 的箭头方向与实际概念依赖相反

**所在位置**：§2 模块间依赖（~第 118 行）

**问题描述**：模块依赖图中标注 `type_vecN → qualifier + compute_vector_relational + scalar_vec_ops（按需）`。然而：

- `type_vecN.cj` 中的 Vec 结构体二元运算符直接在 struct 体内通过逐分量计算实现；extend 块中的 `add(v,s)` 等函数委托给 `+` 等运算符，不调用 `scalar_vec_ops.cj`。
- `scalar_vec_ops.cj` 中的 `add(s, v)`、`sub(s, v)` 等函数的签名使用了 `VecN<T, Q>` 类型作为参数，因此 `scalar_vec_ops.cj` 在概念上依赖 `type_vecN.cj`。

箭头方向应该是 `scalar_vec_ops → type_vecN`，而非图中标注的 `type_vecN → scalar_vec_ops`。由于两者同属 `package glm.detail`，编译器层面不会有编译错误（同包内直接可见），但错误的依赖方向可能误导实施者对编译顺序和包设计层次的理解。

**严重程度**：中等

**改进建议**：修正模块依赖图中的箭头方向：`scalar_vec_ops → type_vecN`（scalar_vec_ops 的函数使用 VecN 类型），并同时在文字说明中澄清此依赖是概念性的（同包内无需显式 import，编译顺序不受影响）。

### 问题 6（轻微）：`fromBoolVecQ2` 函数命名中 `Q2` 的含义不直观

**所在位置**：§4.8 fromBoolVec/fromBoolVecQ2 签名清单

**问题描述**：`fromBoolVecQ2` 中的 `Q2` 是泛型类型参数名暴露到函数命名层级。调用方初次看到此函数名时无法直观理解 `Q2` 的含义——它表示输入 Bool Vec 的 Qualifier（而输出 Vec 的 Qualifier 由泛型参数 Q 指定）。命名约定 `FunctionName + TypeParamName` 在仓颉生态中缺乏先例，增加了 API 的认知负担。

**严重程度**：轻微

**改进建议**：
1. 考虑更具描述性的命名，如 `fromBoolVec2Qualifiers` 或 `fromBoolVecWithInputQualifier`。
2. 或在函数文档注释中增加自解释性说明（如"Q2 = source Bool Vec's Qualifier"），并确保该说明在 IDE 提示中可见。

### 问题 7（轻微）：`cjpm.toml` 中 `cjc-version = "1.0.5"` 的指定缺乏版本最低要求的说明

**所在位置**：§2 项目初始化 cjpm.toml 配置

**问题描述**：`cjpm.toml` 中硬编码了 `cjc-version = "1.0.5"`。cjpm 的 `cjc-version` 字段通常要求编译器版本精确匹配。文档未说明此版本是最低要求还是精确匹配，也未声明设计所依赖的关键语言特性（`const init`、`is` 运算符在 const 泛型中的可用性、`@OverflowWrapping` 继承性等）是否在 1.0.5 以下版本中可用。

**严重程度**：轻微

**改进建议**：
1. 在文档中明确声明 `cjc-version = "1.0.5"` 为设计所依赖的编译器版本号，说明低于此版本时哪些功能可能不可用（参考 §7 D37 编译器行为假设集中一览表中的"可确认"项）。
2. 或在 §2.1 编码前验证检查清单中新增"编译器版本确认"项，验证开发环境 cjc 版本是否满足设计最低要求。

## 三、未发现问题的维度确认

以下维度经审查后确认无明显问题：

- **需求响应充分度**：文档通过 §8.2 范围可追溯性对照表全面映射了 roadmap §3E/§3G 定义的首轮范围，新增文件（`scalar_vec_ops.cj`、`lib.cj`、`type_fromBoolVec.cj`）均有清晰的偏离说明。对以往迭代中出现的循环问题有明确的跟踪和状态标注。

- **架构可落地性**：模块划分、依赖拓扑、迁移顺序明确，编码前验证检查清单（§2.1）覆盖了 14 项关键假设验证。
