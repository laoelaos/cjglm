## 发现的问题

### 问题 1（严重）：`GLM_CONFIG_CLIP_CONTROL` 基础数据冲突历经 8 轮迭代仍未解决

**问题描述**：
§1 P7 和 §7 D6/D38 记录的设计值与 roadmap §3A.1 记录值不一致（0x0A vs 0x02）。该问题自第 2 轮迭代首次发现（P7）起已跨越 8 轮迭代，当前状态为"⚠ 依赖项——需外部协调"。

经 8 轮迭代未解决的事实暴露了以下问题：
- **解决路径不明确**：文档声明"以本文档 `0x0A` 为准"且"roadmap 侧同步不阻塞本设计"，但未指定谁负责协调、协调时限、协调失败后的升级路径。P7 的状态描述"依赖项——需外部协调"缺少可行动的完成标准。
- **编码阶段风险**：实施者面对两个官方文档的矛盾值，被迫自行判断以哪个为准。若选择 `0x0A` 编码但 roadmap 后续被其他方更新为 `0x02`（有意为之而非错误），代码将与 roadmap 不一致，在集成验证时产生争议。
- **流程失效信号**：一个基础配置常量值的冲突历经 8 轮迭代仍为开放项，说明跨文档一致性问题的发现→决议→闭环机制存在缺陷。

**所在位置**：
- §1 P7（状态追踪表）
- §7 D6（配置常量表）
- §7 D38（正式设计决策注释）
- Roadmap §3A.1（外部文件，本产出内被引用）

**严重程度**：严重

**改进建议**：
1. 明确指定协调的**责任方**（如：主 Agent 或 roadmap 维护者）、**完成时限**（如：下一轮迭代结束前）、**失败升级路径**（如：若指定时限内未同步，以设计文档 `0x0A` 为准并记录偏差）。
2. 将 P7 的状态从"⚠ 依赖项——需外部协调"升级为包含可量化完成标准的跟踪项，如"待 roadmap 维护者在 2026-06-26 前确认/修正 §3A.1 值；逾期未响应则以本文档为准关闭此跟踪项"。
3. 在 §12.2 验收标准中新增一条验收项，确认 `GLM_CONFIG_CLIP_CONTROL` 的值在编码开始时已与 roadmap 一致。

### 问题 2（严重）：API 映射完整性验证范围被高估，P37 状态描述不准确

**问题描述**：
§1 P37 状态描述称"首轮 11 个核心文件其余 7 个（setup.hpp、qualifier.hpp、compute_vector_relational.hpp 等）的 API 逐项覆盖对照已在 §8.2 范围可追溯性对照表和 §8.5 排除清单中完成"。

但附录 A（API 覆盖对照表）显式声明其验证范围**仅限** `type_vec{1,2,3,4}.hpp/.inl` 四个 Vec 类型文件，其余 7 个文件"不做逐 API 级对照"。§8.2 是 roadmap 子范围粒度的映射表（非逐 API 级），§8.5 排除清单仅记录已排除的设施（不覆盖已映射的 API）。

因此"逐项覆盖对照"的描述不准确：
- 实际有逐 API 级验证的：4/11 个文件（Vec 类型）
- 仅在子范围粒度映射的：7/11 个文件
- P37 状态描述中的"已完成"措辞误导读者认为 11 个文件均已达到同一验证粒度

**所在位置**：
- §1 P37 状态描述（~第 67 行）
- 附录 A 验证范围声明提示框（~第 2752-2762 行）

**严重程度**：严重

**改进建议**：
1. 修正 P37 状态描述，精确表述为"4 个 Vec 类型文件已完成逐 API 级覆盖对照（附录 A），其余 7 个核心文件的 API 覆盖状态以 §8.2 子范围映射表和 §8.5 排除清单为依据，尚未达到逐 API 级验证粒度"。
2. 在附录 A 的验证范围声明中，补充预计何时（哪个迭代轮次或编码阶段）完成其余 7 个文件的逐 API 级对照。
3. 在 §12.2 验收标准中新增"基础设施层文件 API 覆盖完整性"验收项（非阻塞性，允许编码阶段逐步完善）。

### 问题 3（一般）：% 运算符条件性移动策略的验证依赖覆盖不完整

**问题描述**：
§4.3 和 D13 定义了 `%` 运算符的条件性移动策略，共三条路径：
1. 延迟检查成立→`%` 保留在 struct 体内（当前设计）
2. 定义处报错→`%` 移至 extend 块
3. 定义处报错且 extend 块中 `%` 也受限→`%` 完全移除

但 §2.1 编码前验证项⑰ 仅覆盖"定义处 vs 实例化处"**两种状态的区分**，未提供「路径 2 与路径 3 的进一步区分方法」——即当编译器在定义处报错后，如何判断 extend 块中 `%` 是否可定义（是否需要走路径 3）。

**所在位置**：
- §4.3 `%` 运算的整数约束段（~第 975 行）
- §7 D13 编译器行为条件性移动策略（~第 1371 行）
- §2.1 编码前验证检查清单项⑰（~第 251 行）

**严重程度**：一般

**改进建议**：
1. 扩展 §2.1 项⑰ 的验证方法，增加"若定义处报错，进一步验证 extend 块中定义 `%` 运算符是否也报错"的分支检查步骤。
2. 在 D13 的三条路径中明确标注"路径 2 与路径 3 的区分条件"（定义处报错后 → 尝试 extend 块定义 → 若 extend 块通过则路径 2，否则路径 3）。

### 问题 4（一般）：编译期 `if` 分支抑制的级联连锁风险被低估

**问题描述**：
§11.13 的叠加效应分析将 D29（const 表达式链回退）与依赖③（编译期 `if` 分支抑制失败）的关系描述为"重叠——同时触发时只计一次回退成本"，认为二者本质上是同一验证项的两个侧面。

但此分析未充分评估以下场景：当编译期 `if` 分支抑制行为（A05）因编译器实现不成熟而整体失败时，受影响的远非仅 D29 一条路径：
- `ComputeEqual.callConst` 的容差比较路径完全不可用（D29 回退）
- `mod` 包级独立函数的浮点/整数双路径分支选择不可用（需拆分为 `modInt`/`modFloat`）
- `isIec559Of<T>()` 的 `if (T(0) is Float64 || T(0) is Float32)` 分支选择在 const 函数体内是否可用也存疑
- `epsilonOf` 路径 A/B 选择的底层假设（非选择分支中的泛型约束检查）受影响
- 如果 `const` 函数内的 `if` 整体受限，则 Vec 的 `==` 运算符声明为 `const` 函数的意义也减弱（但仍可精确比较）

文档的叠加效应表将 D29 + 依赖③的成本估算为"只计 D29 一次回退成本"，但实际触发时影响范围是**多维度**的（比较语义、取模实现、泛型约束策略），每个维度的修复并非"`==` 降级到精确比较"这一单一变更所能覆盖。

**所在位置**：
- §11.13 叠加效应分析（~第 2537-2555 行）
- §2.1 项④⑫（编译期 `if` 分支抑制验证）
- §3.5（`epsilonOf` 路径选择）
- §4.3（`mod` 双路径实现）

**严重程度**：一般

**改进建议**：
1. 在 §11.13 叠加效应表中，将 D29 + 依赖③ 组合从"重叠，只计一次成本"修正为"级联重叠——根因相同（编译期 if 分支抑制），但影响面涉及比较语义、取模实现、泛型约束策略三个独立维度，总修复成本不低于各项独立回退成本之和的 60%"
2. 在 §2.1 编码前验证时序建议中，补充说明"项④（编译期 if 分支抑制验证）的结果将同时影响 D29 回退判断、mod 实现策略和 epsilonOf 路径选择，建议优先执行"。

### 问题 5（一般）：验证检查清单的三处分布重叠未完全消解，实施者仍面临信息分散风险

**问题描述**：
第 8 轮迭代问题 2（严重）指出验证清单在 §2.1、§10、§12.3 三处重复组织。v9 修订说明声称已通过"§10 验证方法列替换为对 §2.1 的引用"和"§12.3 标注以 §2.1 为唯一权威出处"解决。

但剩余重叠仍然存在：
- §10 仍保留 **"equalEpsilon 设计阶段验证"** 子节，包含行为描述、依赖项（①~④）和回退方案——这些信息与 §2.1 项③④⑲⑫⑭ 的内容高度重叠，仅验证方法改为引用 §2.1，但核心内容未精简。
- §12.3 决策映射表的"验证通过→选择路径"和"验证失败→回退路径"列，与 §2.1 各验证项的"通过标准"和"失败回退"列功能等价，且 §12.3 的"修改触发联动检查清单"列在 §2.1 中没有对应——这意味着实施者需要同时查阅两个表格才能获得完整的验证→回退→联动修改信息链。
- 三个章节的维护责任边界不清晰：编码阶段发现某个验证项行为与文档不一致时，是修改 §2.1 的通过标准、还是 §10 的回退方案、还是 §12.3 的决策路径？没有明确的归属判定准则。

**所在位置**：
- §2.1 编码前验证检查清单
- §10 equalEpsilon 设计阶段验证子节
- §12.3 编译器验证依赖→设计路径选择决策映射表

**严重程度**：一般

**改进建议**：
1. 将 §10 的 equalEpsilon 验证子节中所有非设计特有的内容（行为描述、依赖项列表）合并至 §2.1 对应的验证项（如将 equalEpsilon 验证依赖①~④ 作为 §2.1 项③的子项），§10 仅保留"设计阶段行为描述"和"回退方案的设计影响分析"。
2. 将 §12.3 决策映射表的"修改触发联动检查清单"列合并至 §2.1 各验证项的新增列"联动修改范围"中，使实施者只需查阅 §2.1 一处即可获得完整的"验证方法→通过标准→回退方案→联动修改清单"信息链。
3. 在 §2.1 检查清单前新增一条维护原则："编码阶段发现验证项相关行为与文档不一致时，优先修改 §2.1 的对应项，再根据影响范围同步更新 §10/§12.3 中引用 §2.1 的内容"。

## 整体质量评价

产出在架构完整性、异常场景覆盖、迁移成本评估和迭代治理方面表现出明显高于一般设计文档的水平。经过 8 轮内部审议和持续迭代，大多数早期识别的问题（P1~P36）已得到有效解决。

但上述 5 个问题表明产出在当前轮次（v9）仍存在以下改进空间：

1. **跨文档一致性问题未闭环**（问题 1）——展示出"识别→决议→闭环"流程中的执行断层
2. **验证范围声明与实际不符**（问题 2）——可能误导后续环节的决策者
3. **验证依赖覆盖不完整**（问题 3）——降低条件性策略的可执行性
4. **连锁风险评估不充分**（问题 4）——低估编译器行为假设失败时的实际修复成本
5. **信息组织效率待优化**（问题 5）——增加实施者的信息检索成本

建议修复上述 5 个问题后进入下一轮迭代或编码准备阶段。
