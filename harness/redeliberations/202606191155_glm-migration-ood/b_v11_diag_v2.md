## 2. 事实错误与逻辑矛盾

### 2.1 `fromBoolVecQ2` 编译器推断依赖的级联影响仍未在验证清单中覆盖（残留/加深）

**问题描述**：上轮审查指出 `fromBoolVecQ2` 函数体中 `Vec1(...)` 构造函数调用依赖编译器从返回值类型推断 struct 泛型实参（§4.8 `[^compiler-dep]`），其级联影响超出"显式标注类型参数"的局部修改——包含 API 签名重新设计和对称性打破风险。v11 修订仅在 §4.8 保留了 `[^compiler-dep]` 标注，但 §10 验证依赖表和 §2.1 检查清单中**仍未新增独立验证项**来覆盖"编译器从返回值类型推断 struct 泛型实参"的行为。此依赖在 §2.1 的 24 项检查清单中找不到对应项，编码实施者可能完全遗漏此验证。

**所在位置**：§4.8 `[^compiler-dep]` 脚注、§10 Bool→Numeric 转换路径 B 验证表、§2.1 编码前验证检查清单

**严重程度**：严重

**改进建议**：① 在 §2.1 检查清单中新增独立验证项（第㉕项），明确验证编译器是否支持从函数返回值类型推断 struct 构造函数的泛型参数；② 在 §10 验证表中补充对应引用；③ 在 §4.8 补充该验证失败时对 `fromBoolVecQ2` API 设计的影响评估（如返回值类型 `VecN<T, Q>` 中的 `Q` 参数是否需要调用方显式指定，导致 API 签名修改）。

### 2.2 P37 其余 7 文件风险等级仍缺乏系统方法论支撑（残留问题）

**问题描述**：v11 为 7 个文件标注了风险等级，但方法论基础仍不明。具体而言：
- setup.hpp→"中"风险标注的依据为"依赖已知可替代"——但此结论的得出过程未说明（是完整阅读了所有 type_vecN.inl 中对 setup 的引用？还是抽样评估？）。
- 编码实施者参考此风险评估做编码计划时，若某个"低"风险评估因遗漏依赖被证伪，将直接打乱编码阶段的资源分配和工期估算。

§1 P37 状态中的风险等级是设计者的主观判断，而非基于结构化交叉引用的系统分析。此问题在上轮审查中被标记为"一般"严重程度，v11 修订仅补充了等级标签未补充方法论。

**所在位置**：§1 P37 状态段落（其余 7 个核心文件的覆盖风险等级列表）

**严重程度**：一般

**改进建议**：① 对每个文件，明确标注风险评估方法（如"逐 API 对照了本文件公开 API 与当前设计中的引用点，共 XX 处引用，无遗漏"或"随机抽样了 5 处交叉引用点，均匹配"）；② 在 §12.2 的"7 文件编码启动前快速扫描"验收项中，要求扫描结论与当前风险等级进行对照验证。

### 2.3 §3.2 const 上下文 API 速查表存在事实错误

**问题描述**：§3.2 const 上下文 API 可用性速查表中，`v.increment()` 的 const 替代方案列为"改用 `v = v + T(1)` 或 `v = add(T(1), v)` 等展开模式"。其中 `v = v + T(1)` 包含赋值操作 `=`——赋值在 const 函数体中不可用（const 函数不可修改实例成员），因此该替代方案本身不是 const 兼容的。正确的 const 替代方案应为返回新构造的向量（如 `Vec2(this.x + T(1), this.y + T(1))`）或在调用方通过包级独立函数 `add(T(1), v)`（该函数已声明为 `const`）处理。

**所在位置**：§3.2 const 上下文中的 API 可用性速查表（`v.increment()` 行）

**严重程度**：一般

**改进建议**：修正 `v.increment()` 行的 const 替代方案，将"改用 `v = v + T(1)`"替换为"改用 `add(T(1), v)`（包级独立函数，声明为 const）或直接构造新向量"。

## 3. 深度与完整性

### 3.1 `fromBoolVecQ2` 编译器推断依赖的验证缺口增大设计风险（承接 2.1）

**问题描述**：§4.8 的 `[^compiler-dep]` 依赖描述了"编译器从返回值类型推断 struct 泛型实参"的行为假设，但该假设在 §2.1 的 24 项验证项中完全无对应项。当前 24 项覆盖了 const 表达式链、`@OverflowWrapping` 继承性、编译期 `if` 分支抑制等编译器行为，但遗漏了构造函数泛型参数推断这一独立维度。若编码阶段发现此推断不成立，影响范围包括：
1. `fromBoolVecQ2` 的 8 个函数定义（Vec1~Vec4 × 同 Q/跨 Q）均需修改——所有 `VecN(...)` 调用处显式标注 `<T, Q>`。
2. 更严重的是，此问题可能推迟到 `fromBoolVecQ2` 首次编译测试时才被发现（而非通过专项验证提前暴露），影响编码阶段的"fail fast"节奏。

**所在位置**：§4.8 `[^compiler-dep]` 脚注、§2.1 检查清单（缺少此依赖项）、§11.13 编译器依赖验证回退成本（未覆盖此场景）

**严重程度**：一般（但影响面为中——8 处代码修改 + 验证发现滞后风险）

**改进建议**：① 在 §2.1 新增验证项覆盖此行为；② 在 §11.13 回退成本表中补充此依赖的验证成本和回退修复成本（约 0.5 人天）。

### 3.2 验证清单三处分布的消重仍未彻底完成（残留问题）

**问题描述**：v11 修订（条目 5）声称对 §10 和 §12.3 做了进一步消重，将验证方法描述替换为"对应 §2.1 检查项"引用。但 §10 仍保留了多个完整的结构化验证依赖表：
- "equalEpsilon 设计阶段验证"表（4 行数据）
- "Bool→Numeric 转换路径 B 可行性验证"表（3 行数据）
- "Vec4 构造函数重载解析验证"表（3 行数据）
- "mod 函数边界值验证"表（2 行数据）
- "编译期 if 分支抑制行为综合验证"依赖点清单（5 行数据 + 回退方案表）

虽然各表的"验证方法"列已改为"对应 §2.1 检查项"引用，但表格本身作为独立文档元素存在，维护者修改 §2.1 时仍需同步更新 §10 中对应表的引用列——此即原问题的核心（跨章节同步成本），而非仅验证方法的表述位置。

**所在位置**：§10 多个验证子节（equalEpsilon、Bool→Numeric、Vec4 重载解析、mod 边界值、编译期 if 抑制）

**严重程度**：一般

**改进建议**：将 §10 中所有非设计特有的验证表替换为纯文本单行引用（如"equalEpsilon 所有验证依赖项均由 §2.1 项⑫⑲㉒覆盖"），仅在 §10 保留设计特有的行为描述和回退方案依赖关系分析。彻底消除表格式冗余。

### 3.3 迁移成本合计表缺少基础编码工作量基准线（残留问题）

**问题描述**：§11.12 底部注脚声明"不包含基础编码工作量（Vec 结构体和运算符的仓颉实现）"和"测试编写工作量"。作为一份 OOD 设计文档，缺少总编码工作量估算使项目规划者无法从本文档获得完整的工期视图。此问题在上轮审查中被指出但未修复。

**所在位置**：§11.12 合计表底部注脚

**严重程度**：一般

**改进建议**：① 在 §11.12 之前新增"基础编码工作量评估"子节，对 Vec 结构体定义、运算符实现、别名脚本、shim 文件、测试编写等基础工作给出大致范围（或明确标注由实施团队自行评估）；② 在合计表末尾增加"基础编码"行和"总计（含基础编码）"行；③ 若因不确定性无法给出估算，应明确标注估算的责任归属而非完全留空。

### 3.4 "已验证"与"待验证"的措辞边界仍不够清晰（残留/弱化）

**问题描述**：§7 D6 中对 `GLM_CONFIG_CLIP_CONTROL` 标注为"默认值来源已验证（参考实现确认）"，§8.4 前置验证项①中要求"编码前必须读取 `setup.hpp:563-584` 的原始条件编译块，逐行确认"。前者说"已验证"，后者说"需验证"——实施者不清楚设计阶段是否已完成了此项验证。D38 补充了正式设计决策记录，但 D6 首段描述中"已验证"的措辞仍未与 §8.4 的"待编码前验证"表述对齐。

虽然 §1 P7 已标注"✅ 已解决（v11）"，但 P7 的"已解决"是指跨文件值一致性冲突已消除（roadmap → 0x0A），而非指"值正确性已通过编译器验证"。"值一致"和"值正确"是两个不同概念。

**所在位置**：§7 D6 默认值说明、"**默认值来源已验证（参考实现确认）**"措辞、§8.4 前置验证项①

**严重程度**：轻微

**改进建议**：将 §7 D6 中"已验证"的措辞统一为"经参考实现源码阅读确认，编码前验证项②做最终确认"，与 §8.4 的验证时序明确对齐。

## 4. 编码落地可行性

### 4.1 `fromBoolVecQ2` 编译器推断依赖的验证缺口（与 2.1/3.1 同源，此处聚焦编码影响）

**问题描述**：此依赖的验证缺口对编码实施者的直接影响——该依赖在发现时可能已到编码中后期（`fromBoolVecQ2` 实现阶段），届时修改涉及 8 处 `VecN(...)` 调用点，且需同步更新 `type_fromBoolVec.cj`。更关键的是，如果编译器完全不支持此推断模式（而非仅需显式标注），`fromBoolVecQ2` 的 API 签名可能需要重新设计——将返回类型绑定到具名变量以绕过推断失败，这会影响所有调用方的使用模式。

当前 §11.13 回退成本表未覆盖此场景的成本评估，实施者在编码阶段发现此问题时无法参考提前定义的回退路径和工时估算。

**所在位置**：§4.8 `[^compiler-dep]`、§11.13 编译器依赖验证回退成本表

**严重程度**：一般

**改进建议**：① 在 §11.13 中新增一行覆盖 `[^compiler-dep]` 依赖的验证和回退成本；② 在 §4.8 中补充回退方案的具体代码模式（如将 `Vec1(...)` 改为 `Vec1<T, Q>(...)` 的显式标注模式，或通过 `let tmp: Vec1<T, Q> = ...` 中间变量绕过推断）。
