## 一、整体评价

该产出质量非常高，覆盖面广（2522 行），在架构设计、行为契约、迁移策略、验证计划、编译器依赖风险评估等方面均有深度论述。文档结构清晰、交叉引用完整、设计决策记录详实，整体上已具备指导编码实现的成熟度。

## 二、问题清单

### 问题 1（严重）：`GLM_CONFIG_CLIP_CONTROL` 默认值修正的断言无法被独立验证

**所在位置**：§7 D6（`GLM_CONFIG_CLIP_CONTROL` 默认值说明段落）

**问题描述**：文档声称 roadmap §3A.1 记录的 `0x02` 是错误的，修正为 `0x0A`。此断言基于对 `references/glm-1.0.3/glm/glm/detail/setup.hpp` 条件编译逻辑的分析。然而工作空间中不存在 `references/glm-1.0.3/` 目录，无法独立验证此分析的正确性。若该断言错误（例如条件编译的实际分支走向与文档分析不一致），则所有依赖 `GLM_CONFIG_CLIP_CONTROL` 默认值的逻辑均使用错误值。

**严重程度**：严重

**改进建议**：
1. （首选）在编码前验证项②（§2.1 第 ① 项/§8.4 策略 0）的验证方法中显式增加：读取参考实现中 `GLM_CONFIG_CLIP_CONTROL` 的 `#if`/`#elif`/`#else` 条件链并记录所有分支的实际取值，确认 `0x0A` 的推导过程。
2. 在文档中标注此值的推导结果为"待参考实现验证"，而非"已验证正确"。
3. 同步将 roadmap 中的 `0x02` 修正为 `0x0A` 或提交变更通知。

### 问题 2（严重）：标量别名与向量别名的精度前缀命名风格不一致，文档未做任何说明

**所在位置**：§3.7 标量类型别名表 vs §3.8 向量类型别名命名约定

**问题描述**：标量别名使用小写下划线精度前缀（如 `highp_int8`、`mediump_int8`、`lowp_int8`），而向量别名使用 PascalCase 精度前缀（如 `HighpBVec2`、`MediumpBVec2`、`LowpBVec2`）。两种命名风格在同一个包（`package glm`）中共存，但文档既未解释此不一致的原因，也未评估其导致的认知成本。使用者需要在"标量用小写 + 下划线、向量用 PascalCase 拼接"之间频繁切换，增加了概念混淆风险。

例如，`public type highp_float = Float32`（标量）和 `HighpVec2`（向量）使用了完全不同的精度前缀形式。

**严重程度**：严重

**改进建议**：
1. （推荐）将标量别名精度前缀统一为 PascalCase 风格，与向量别名一致：`highp_int8` → `HighpInt8`。若需保持 GLM 源兼容性（C++ GLM 中标量别名确实使用 `highp_int8` 形式），则在文档中显式声明此差异并说明理由。
2. 在 §11.7 映射表或 §3.7 末尾增加"标量与向量命名风格差异"说明段落，解释此不一致是有意为之（或设计遗漏需要统一）。

### 问题 3（中等）：`equalEpsilon` 对 `VecN<Bool, Q>` 的编译错误未在迁移成本中覆盖

**所在位置**：§4.5 `equalEpsilon` 定义、§11 迁移成本评估

**问题描述**：文档正确指出 `equalEpsilon` 在 `VecN<Bool, Q>` 上实例化时产生编译错误（因 `Bool` 不支持 `-` 运算符）。然而 §11 的迁移成本评估完全没有涉及以下场景：如果下游 C++ 代码使用了 BVEC 上的等价比较（或 C++ 中 `bvec` 的逐分量 `==` 迁移到仓颉后需要 `equalEpsilon` 的场景），实施者将面临无法预料的编译失败，且无迁移指引可依循。

§12.1 层次三虽然包含了 Bool Vec 的编译错误测试，但该测试仅验证"确实报错"，未提供当此错误发生时如何修复的指引。

**严重程度**：中等

**改进建议**：
1. 在 §11 迁移成本评估中新增子节（如 §11.15），专门覆盖 Bool Vec 相关 API（`equalEpsilon`、`increment`/`decrement`、`%` 等）在迁移中遇到编译错误的处理模式。
2. 在 §4.5 `equalEpsilon` 定义附近增加"Bool Vec 迁移指南"提示框，说明迁遇到 `equalEpsilon` 在 Bool Vec 上编译失败时，应检查原代码逻辑并替换为 `==` 或 `logicalAnd`/`logicalOr` 等。

### 问题 4（中等）：Vec4 构造函数重载解析验证覆盖范围不足

**所在位置**：§4.1 重载解析风险描述（line 808-814）、§10 Vec4 重载解析验证（line 1901-1913）

**问题描述**：文档将风险聚焦在 "6 组对偶重载分支"（`Vec3+T` vs `Vec3+Vec1` 等），但 Vec4 共有约 30 个构造函数签名。以下两类潜在歧义场景未被验证计划覆盖：

1. **标量与 Vec1 类型模糊**：当参数类型为 `T` 和 `Vec1<T2, Q2>` 时，若 T2 被实例化为 `Int32` 等数值类型而 `T` 同为 `Int32`，编译器是否仅凭类型结构（值 vs 结构体）就能可靠区分？例如 `init<T2, Q2>(a: Vec2<T2, Q2>, b: T)` 与 `init<T2, Q2>(a: Vec2<T2, Q2>, b: Vec1<T2, Q2>)` 当 `T = Int32` 且 `T2 = Int32` 时，`Int32` 与 `Vec1<Int32, Q>` 的结构差异是否足够编译器区分？

2. **多元 Vec1 参数组合**：`init<T2, Q2>(a: Vec1<T2, Q2>, b: Vec1<T2, Q2>, c: Vec1<T2, Q2>, d: Vec1<T2, Q2>)` 与其他接收 4 个 `Vec1` 参数的构造函数之间是否存在歧义？

验证计划仅覆盖了明确标识的 6 组对偶分支，未制定系统性扫描策略（如按参数数量分组、组内按参数类型结构相似度排序后逐一测试）。

**严重程度**：中等

**改进建议**：
1. 在 §10 Vec4 重载解析验证中，将验证方法从"覆盖 6 组对偶分支"升级为"按参数数量分组的全扫描"——对每组参数数量相同的构造函数，逐一测试其区分度。
2. 对 §4.1 风险描述中的"已知安全场景"（line 807）补充反例测试——编写原型测试确认文档声称"安全"的场景确实无歧义。
3. 考虑在编码前验证清单中新增统一的 Vec4 构造函数全量编译验证项（而非仅验证 6 组对偶分支）。

### 问题 5（中等）：模块依赖图显示 `type_vecN → scalar_vec_ops`，但实际依赖方向可能是反向的

**所在位置**：§2 模块间依赖（line 118）

**问题描述**：模块依赖图中标注 `type_vecN → qualifier + compute_vector_relational + scalar_vec_ops（按需）`。然而分析各文件内容可知：

- `type_vecN.cj` 中的 Vec 结构体二元运算符直接在 struct 体内通过逐分量计算实现；extend 块中的 `add(v,s)` 等函数委托给 `+` 等运算符，不调用 `scalar_vec_ops.cj`。
- `scalar_vec_ops.cj` 中的 `add(s, v)`、`sub(s, v)` 等函数的签名使用了 `VecN<T, Q>` 类型作为参数，因此 `scalar_vec_ops.cj` 在概念上依赖 `type_vecN.cj`。

因此箭头方向应该是 `scalar_vec_ops → type_vecN`，而非图中标注的 `type_vecN → scalar_vec_ops`。由于两者同属 `package glm.detail`，编译器层面不会有编译错误（同包内直接可见），但错误的依赖方向可能误导实施者对编译顺序和包设计层次的理解。

**严重程度**：中等

**改进建议**：修正模块依赖图中的箭头方向：`scalar_vec_ops → type_vecN`（scalar_vec_ops 的函数使用 VecN 类型），并同时在文字说明中澄清此依赖是概念性的（同包内无需显式 import）。

### 问题 6（中等）：统一的 `testConstIfSuppression` 验证可能无法覆盖所有依赖点的差异表达式

**所在位置**：§10 编译期 `if` 分支抑制行为综合验证（line 1769-1815）

**问题描述**：文档使用单一原型函数 `testConstIfSuppression<T>()` 来验证所有 5 个依赖点 (A~E) 的编译期 `if` 分支抑制行为。然而实际依赖点的表达式形态各不相同：

| 依赖点 | 实际表达式内容 | 统一测试中的等价表达式 |
|-------|-------------|------------------|
| A | `abs(a - b) <= epsilonOf<T>()` | `true` |
| C | `x - y * trunc(x / y)` | `T(0) == T(0)` |
| D | 嵌套编译期 `if` | 无嵌套 |

统一测试的浮点分支仅含 `true`，整数分支仅含 `T(0) == T(0)`，这两个表达式都非常简单，无法覆盖实际代码中更复杂的表达式形态（如浮点运算链、嵌套分支、多分支选择）。这意味着即使统一测试通过，实际依赖点仍可能在更复杂表达式中失败。

**严重程度**：中等

**改进建议**：扩充统一验证的覆盖范围，对每个依赖点分别编写独立的验证函数，使用与生产代码形态一致的表达式。例如：
- 对依赖点 A：验证 `ComputeEqual.callConst` 的完整表达式链
- 对依赖点 C：验证浮点 mod 的 `trunc` 恒等式
- 对依赖点 D：验证编译期 `if` 嵌套场景

在统一测试通过的基础上，逐项执行独立验证。

### 问题 7（轻微）：`fromBoolVecQ2` 函数命名中 `Q2` 的含义不直观

**所在位置**：§4.8 fromBoolVec/fromBoolVecQ2 签名清单

**问题描述**：`fromBoolVecQ2` 中的 `Q2` 是泛型类型参数名暴露到函数命名层级。调用方初次看到此函数名时无法直观理解 `Q2` 的含义——它表示输入 Bool Vec 的 Qualifier（而输出 Vec 的 Qualifier 由泛型参数 Q 指定）。命名约定 `FunctionName + TypeParamName` 在仓颉生态中缺乏先例，增加了 API 的认知负担。

**严重程度**：轻微

**改进建议**：
1. （推荐）考虑更具描述性的命名，如 `fromBoolVecQualifier` 或 `fromBoolVecWithQualifier`。
2. 或在文档注释中为 `fromBoolVecQ2` 增加自解释性说明（如 "Q2 = source Bool Vec's Qualifier"），并确保该说明在 IDE 提示中可见。

### 问题 8（轻微）：cjpm.toml 中 `cjc-version = "1.0.5"` 的指定可能过于刚性

**所在位置**：§2 项目初始化 cjpm.toml 配置

**问题描述**：`cjpm.toml` 中硬编码了 `cjc-version = "1.0.5"`。如果开发环境中的 cjc 版本低于 1.0.5，`cjpm build` 可能因版本不匹配直接失败。文档未说明是否所有设计假设均已针对 1.0.5 以下版本验证（如 `const init` 行为、`is` 运算符在 const 泛型中的可用性等可能随编译器版本变化）。

**严重程度**：轻微

**改进建议**：
1. 将 `cjc-version` 改为 `>= "1.0.5"` 以允许补丁版本兼容。
2. 或在文档中明确声明设计所依赖的编译器版本号，并说明低于此版本时哪些功能可能不可用。

## 三、未发现问题的维度确认

以下维度经审查后确认无明显问题（仅作客观陈述，不构成改进建议）：

- **需求响应充分度**：文档通过 §8.2 范围可追溯性对照表全面映射了 roadmap §3E/§3G 定义的首轮范围，新增文件（`scalar_vec_ops.cj`、`lib.cj`、`type_fromBoolVec.cj`）均有清晰的偏离说明。对以往迭代中出现的循环问题有明确的跟踪和状态标注。

- **异常场景和边界条件**：§5/§9 对下标越界、整数溢出、除零、NaN/Infinity 传播、类型转换边界均有明确的约定。§6 并发设计覆盖了值类型线程安全、复合赋值非原子性等场景。
