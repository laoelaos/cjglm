## 问题 1

**问题描述**：C++ GLM 中二元运算符和复合赋值运算符同样支持跨类型参数 `U`（`template<typename U> operator+=(U scalar)` 和 `template<typename U> operator+=(vec<2, U, Q> const& v)`），但设计文档对二元运算符的跨类型支持限制（D15）与复合赋值的跨类型限制（§4.3）分别独立陈述，未做对称性分析。关键遗漏：C++ GLM 的**二元**运算符同样使用了跨类型 `U`（如 `operator+(vec<2, T, Q> const& v, U scalar)` 和 `operator+(vec<1, U, Q> const& v1, vec<2, T, Q> const& v2)` 是独立模板函数），而设计文档仅将二元运算符的右操作数限制为同类型 `T`。§4.3 中将二元运算符签名为 `operator func +(rhs: Vec2<T, Q>): Vec2<T, Q>` 和 `operator func +(rhs: T): Vec2<T, Q>`，这意味着 `T` 为 `Int32` 时 `Vec2<Int32, Q> + Int16(1)` 将需要显式类型转换，而 C++ GLM 中此场景由 `operator+(vec<2, T, Q>, U)` 自动处理。设计文档正确分析了复合赋值的跨类型损失，但**对二元运算符跨类型 `U` 参数损失的评估不对称**——二元运算符限制与复合赋值限制是同一语言限制（运算符重载不能为泛型）的两种表现，但文档仅将二元运算符限制简单归因于 D15 而未评估其迁移影响。

**所在位置**：§4.3 算术运算（二元运算符签名清单 + D15 引用段）、§4.6 标注策略、§11 迁移成本评估

**严重程度**：一般

**改进建议**：在 §4.3 二元运算符设计段落补充"二元运算符跨类型 `U` 参数限制"子段，与复合赋值的跨类型限制做对称分析；明确声明仓颉中 `Vec2<Int32> + Int16(1)` 需要显式 `Int32(1)` 转换或 `Vec2<Int32>(Int16(1))` 构造转换，评估此限制与复合赋值跨类型限制的同源关系；在 §8.5 排除清单中补充"二元运算符跨类型 `U` 参数不可用"条目；在 §11 中补充迁移模式评估。

## 问题 2

**问题描述**：设计文档 §4.3 将 `vec-op-scalar` 方向的 `add(v, s)`/`sub(v, s)`/`mul(v, s)`/`div(v, s)`/`mod(v, s)` 定义为 Vec 的**扩展成员函数**，同时将 `scalar-op-vec` 方向定义为包级独立函数。但 C++ GLM 中 `scalar + vec` 形式的运算是通过独立模板函数声明的（`template<typename T, qualifier Q> GLM_FUNC_DECL vec<2, T, Q> operator+(T scalar, vec<2, T, Q> const& v)`），其中的 `T` 与 Vec 分量类型相同——即 C++ GLM 中标量在左的运算也仅支持与 Vec 分量类型相同类型的标量，**同样不支持跨类型标量 `U`**。设计文档未区分这一点——当声明"scalar + vec 不可重载"时，应对 C++ GLM 原生 `operator+(T scalar, Vec2<T,Q>)` 与仓颉 `add(s: T, v: Vec2<T,Q>)` 的**功能等价性**做出明确声明，否则迁移者可能误以为 `add(s, v)` 也需要支持跨类型 `U` 参数。

**所在位置**：§4.3 标量在左的运算替代方案段落、§7 D10

**严重程度**：轻微

**改进建议**：在 §4.3 或 §7 D10 中补充声明：`add(s, v)` 的标量参数 `s` 类型为 `T`（与 Vec 分量类型相同），与 C++ GLM `operator+(T scalar, vec<N,T,Q>)` 功能等价。跨类型标量在左运算（`U + VecN<T,Q>`）在 C++ GLM 中同样不可用（因模板参数 `T` 统一约束），无需额外适配。

## 问题 3

**问题描述**：C++ GLM 中 Vec2~Vec4 的所有复合赋值运算符均支持 `vec<1, U, Q>` 作为右操作数（如 `operator+=(vec<1, U, Q> const& v)`），这是跨类型 Vec1 右操作数的复合赋值。设计文档 §4.3 中"Vec1 广播运算模式"段落分析了 `VecN += Vec1`（同类型）复合赋值由编译器自动生成，但声明"跨类型 Vec1 广播复合赋值不可用"时，未区分此限制的两个维度：(a) **同类型 `U = T` 的 Vec1 复合赋值**（`Vec2<T,Q> += Vec1<T,Q>`，由广播运算符自动生成，可用）；(b) **跨类型 `U ≠ T` 的 Vec1 复合赋值**（`Vec2<T,Q> += Vec1<U,Q>`，不可用）。而在 C++ GLM 中，`(b)` 是通过 `operator+=(vec<1, U, Q>)` 显式模板化实现的，是一个重要的跨类型操作入口。设计文档虽在 §8.5 排除清单中包含了"跨类型 Vec1 广播复合赋值"条目，但在 §4.3 广播运算模式段落中**未将 `(b)` 场景的不可用性显式声明为设计限制**，仅将 `(a)` 声明为"自动生成可用"，而 `(b)` 仅在排除清单中一笔带过。编码实施者阅读 §4.3 时可能忽略 `(b)` 的限制。

**所在位置**：§4.3 Vec1 广播运算模式段落（复合赋值子段）

**严重程度**：一般

**改进建议**：在 §4.3 Vec1 广播运算模式段落的复合赋值子段中，显式声明"Vec1 广播复合赋值仅支持同类型 `T` 右操作数（`VecN<T,Q> += Vec1<T,Q>`），跨类型 `U` 版本（`VecN<T,Q> += Vec1<U,Q>`）不可用，详见 §8.5 排除清单"。

## 问题 4

**问题描述**：设计文档中 `compute_vector_relational.cj` 的文件清单序号为 6，`vectorize.cj` 为 7，`compute_vector_decl.cj` 为 8，`scalar_vec_ops.cj` 为 8a。在 §8.3 迁移顺序表中，阶段 3 为 `compute_vector_relational.cj + vectorize.cj`（可并行），阶段 4 为 `compute_vector_decl.cj`。但 `compute_vector_decl.cj` 依赖 `vectorize.cj`（§8.1 文件清单中标注"依赖：`vectorize.cj`"），而迁移阶段 3 和 4 是分阶段而非可并行的。这与 §2 模块间依赖描述一致，但与 §8.3"阶段 3 辅助工具（可并行）"的标签不一致——`compute_vector_decl.cj` 有 `vectorize.cj` 的编译前置依赖，不能与 `vectorize.cj` 并行编译。§8.3 中阶段 3 仅列出 `6. compute_vector_relational.cj + 7. vectorize.cj`（不含 `8. compute_vector_decl.cj`），阶段 4 单独列出 `8. compute_vector_decl.cj`——这种分阶段是正确的。但**阶段 3 标注为"可并行"时，未说明 `compute_vector_relational.cj` 与 `vectorize.cj` 之间无编译依赖（前者依赖 `shim_limits.cj`，后者依赖 `qualifier.cj`），容易让实施者误以为所有辅助工具文件均可并行**。

**所在位置**：§8.3 迁移顺序表阶段 3

**严重程度**：轻微

**改进建议**：在 §8.3 阶段 3 注释中补充"`compute_vector_relational.cj` 与 `vectorize.cj` 无相互编译依赖，可并行编译；`compute_vector_decl.cj` 依赖 `vectorize.cj`，须在阶段 4 编译"。

## 问题 5

**问题描述**：设计文档 §3.5 `ComputeEqual.callConst` 中浮点路径使用 `abs(a - b) <= epsilonOf<T>()` 进行容差比较，但此处的 `epsilonOf<T>()` 返回 `NumericLimits<T>.epsilon()`（机器 epsilon，Float32 下约 1.19e-7，Float64 下约 2.22e-16）。机器 epsilon 代表 1.0 附近的最小可区分差值，而非全值域的通用容差——对于大值（如 `a = 1e10, b = 1e10 + 1`）容差比较可能误判为相等，而对于极小值（如 `a = 1e-20, b = 0.0`）可能误判为不等。C++ GLM 中 `compute_equal` 的容差比较**同样使用 `NumericLimits<T>::epsilon()`**，因此此行为与 C++ GLM 一致。设计文档在 §3.5 和 §10 处描述此方案时，没有在任何地方**显式声明容差比较的"仅适用于1.0附近值域"的固有局限性**，也未声明此行为与 C++ GLM 一致。§9.3 中讨论了 Inf/NaN 的边界行为，但未讨论大值和极小值场景下机器 epsilon 容差的行为特征。

**所在位置**：§3.5 ComputeEqual.callConst 容差比较路径描述、§10 依赖③验证描述

**严重程度**：一般

**改进建议**：在 §3.5 或 §9.3 中补充"机器 epsilon 容差比较的值域局限性"说明段：声明容差比较使用 `NumericLimits<T>.epsilon()` 作为绝对容差阈值，此值仅适用于接近 1.0 量级的浮点值比较；对于大值（|x| >> 1）容差过于宽松，对于极小值（|x| << 1）容差过于严格。此行为与 C++ GLM 的 `compute_equal` 一致——C++ GLM 同样使用 `std::numeric_limits<T>::epsilon()` 作为容差标准。需要相对容差比较的下游消费者应自行实现基于 `max(abs(a), abs(b)) * epsilon` 的比较逻辑。

## 问题 6

**问题描述**：§4.3 算术运算的 `@OverflowWrapping` 标注策略中，文档声明二元算术运算符在 Vec 结构体内部定义并标注 `@OverflowWrapping`，复合赋值由编译器自动生成并继承标注。但在 §4.6 标注策略总结段落中，声称标注了"30 个 Vec1 广播算术运算符 + 30 个 Vec1 广播位运算符"，这些广播运算符在 `extend` 块中定义。问题在于：广播位运算符中 `&`、`|`、`^`、`>>` 不产生溢出（文档自身也声明无需标注），但总结中将其与算术广播运算符合并在"30+30"计数中，未区分哪些广播位运算符实际需要 `@OverflowWrapping` 标注（仅 `<<` 相关需要），哪些不需要。实施者按此总结计数标注时，可能对不需要标注的 `&`/`|`/`^`/`>>` 广播运算符也做标注，虽无害但增加冗余。

**所在位置**：§4.6 标注策略总结段落（含括号内计数）

**严重程度**：轻微

**改进建议**：在 §4.6 总结段落中将"30 个 Vec1 广播位运算符"细化为"5 个 `<<` 广播位运算符（需标注）+ 25 个 `&`/`|`/`^`/`>>` 广播位运算符（无需标注）"，或直接写"30 个 Vec1 广播算术运算符 + `<<` 相关广播位运算符（约 5×3=15 个需标注）"。

## 问题 7

**问题描述**：§3.8 向量类型别名中，`FVec` 家族的默认分量类型映射为 `Float32`（脚本 `VEC_FAMILIES` 中 `'FVec': 'Float32'`），而 C++ GLM 中 `fvec` 在 `fwd.hpp` 中为 `typedef vec<1, float, highp>	fvec1;`——`fvec` 确实映射到 `float`（即 `Float32`），此映射正确。但 `f32vec` 和 `f64vec` 两个家族分别映射 `Float32` 和 `Float64`，与 `Vec`（也是 `Float32`）和 `DVec`（`Float64`）存在功能冗余——`F32Vec2` 等价于 `Vec2`，`F64Vec2` 等价于 `DVec2`。设计文档未声明此冗余关系和调用方选择指引——当 `F32Vec2` 与 `Vec2` 映射到同一泛型具现化时，调用方应优先使用哪个别名？这与 Vec1 运算符专节（§4.3 末）中讨论的 Vec1/标量语义重叠属于同类问题，但未在别名层面给出选择指引。

**所在位置**：§3.8 向量类型别名（命名约定 + 消费者快速选择指引）

**严重程度**：轻微

**改进建议**：在 §3.8 消费者快速选择指引表中增加注释行或脚注，声明 `FVec`/`F32Vec`/`F64Vec` 与 `Vec`/`Vec`/`DVec` 分别映射到相同泛型具现化（`F32Vec2 = Vec2<Float32, PackedHighp> = Vec2`，`F64Vec2 = Vec2<Float64, PackedHighp> = DVec2`），推荐优先使用 `Vec`/`DVec` 系列别名（更简洁），`F32Vec`/`F64Vec` 系列留给需要显式标注位宽的代码风格场景。

## 问题 8

**问题描述**：§2.1 编码前验证检查清单将验证任务混合了独立原型验证和构建工具验证两种不同类型的任务，层级说明区分了"独立原型验证"和"统一集成验证"，但⑭（测试运行器配置验证）和⑮（编译器版本确认）标注为"构建工具验证"但未列入单独层级说明。这两个验证项的性质与原型验证确实不同，但缺乏与①~⑬和⑯~㉓一致的层级归类标注（如"构建工具验证"层级说明仅在⑭⑮隐含体现，未在层级说明段落显式声明）。

**所在位置**：§2.1 编码前验证检查清单层级说明段落

**严重程度**：轻微

**改进建议**：在 §2.1 的层级说明段落末尾补充"第⑭⑮项为 **构建工具验证**——验证构建工具配置和编译器版本，与原型验证和集成验证并行但性质不同，不依赖其他验证项结果"的层级声明。

## 整体质量评价

本产出是一份高度详尽的 OOD 设计文档，经过 4 轮外部再审议迭代，大部分历史问题已闭环解决。设计决策体系完整，编译器行为假设集中管理（§7 D37），验证清单系统化（§2.1 共 23 项），排除清单详尽（§8.5），迁移成本评估覆盖乐观/最可能/悲观三档。核心架构（Qualifier 接口 + Vec 泛型结构体 + 双层包结构）清晰可行，构造函数体系、运算符覆盖、类型别名系统、错误处理策略均达到可供编码实施者直接参照的成熟度。本轮识别的 8 个问题中无严重等级，均为一般/轻微，表明产出整体质量已达到较高水准，剩余问题主要集中在跨类型参数损失的不对称分析、边界行为说明和实施细节的精确性上。
