# 再审议判定报告（v11）

## 判定结果

RETRY

## 判定理由

组件B诊断报告（b_v11_diag_v3.md）经质询确认（LOCATED，实际轮次3 < 最大轮次12，提前终止）后，仍包含以下超出轻微等级的问题：

- **严重级别**问题1个：§3.1 `fromBoolVecQ2` 编译器推断依赖的验证缺口（跨多轮迭代未被修复，§2.1 验证清单持续缺失对应项）
- **一般/中等等级**问题2个：§2.1 const 上下文 API 可用性速查表事实错误（前轮已指出但未修复）、§4.1 `fromBoolVecQ2` 编译器推断依赖对编码实施阶段的直接影响

根据判定标准，审查报告包含严重或一般等级问题，判定为 RETRY。

## 需要解决的问题

- **问题描述**：§2.1 编码前验证清单中 `fromBoolVecQ2` 编译器推断依赖的验证项持续缺失。自第2轮审查起已被连续指出，跨 v8→v9→v10→v11 多轮修订均未修复。该依赖在 §4.8 以 `[^compiler-dep]` 脚注形式标注，但 §2.1 24 项验证清单无对应项，编码实施者将在实现阶段面临需自行判断编译器行为的风险。
- **所在位置**：§4.8 `[^compiler-dep]` 脚注、§2.1 编码前验证检查清单（缺少此项）、§11.13 回退成本表
- **严重程度**：严重
- **改进建议**：在 §2.1 新增第㉕项验证编译器是否支持从函数返回值类型推断 struct 泛型参数；同步更新 §11.13 回退成本表；在 §4.8 补充显式标注 `<T, Q>` 后的完整函数体代码示例

- **问题描述**：§3.2 const 上下文 API 可用性速查表中 `v.increment()` 行的 const 替代方案仍标注为 `v = v + T(1)` 或 `add(T(1), v)`，其中 `v = v + T(1)` 含赋值操作，在 const 函数体内不可用。此问题在上一轮审查（b_v11_diag_v2.md §2.3）中已被指出但未修复。
- **所在位置**：§3.2 const 上下文 API 可用性速查表（`v.increment()` 行）
- **严重程度**：一般
- **改进建议**：移除 `v = v + T(1)`，仅保留 `add(T(1), v)`（包级独立函数，声明为 const）；同步检查同一速查表中 `v.decrement()` 等行的替代方案是否存在类似问题

- **问题描述**：§4.8 `fromBoolVecQ2` 函数定义中以 `[^compiler-dep]` 脚注标注的编译器推断依赖，未指定回退方案的具体代码模式。实施者首次编写时需自行判断是否需显式标注 `<T, Q>`，且跨 Q 版本中泛型参数顺序可能导致笔误。
- **所在位置**：§4.8 `[^compiler-dep]`、§11.13 回退成本表
- **严重程度**：一般
- **改进建议**：在 §4.8 补充显式标注 `<T, Q>` 后的完整函数体代码模板；在 §11.13 回退成本表中新增一行覆盖此依赖的验证和回退成本估算
