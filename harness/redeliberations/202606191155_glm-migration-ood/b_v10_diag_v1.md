## 问题汇总

### 问题 1（严重）：`GLM_CONFIG_CLIP_CONTROL` 冲突声明基于过时信息，P7 跟踪前提不成立

**问题描述**：设计文档在以下多处声称 roadmap §3A.1 仍记录为 `0x02`，与本文档的 `0x0A` 存在冲突，并基于此将 P7 升级为"🔴 跟踪项——需外部协调"：

- §1 P7：`"roadmap §3A.1 仍记录为 0x02，两文件未同步"`
- §7 D38：`"roadmap §3A.1 记录该配置默认值为 0x02，...此差异需显式协调"`
- §12.2 CLIP_CONTROL 验收项：要求"编码启动前确认 roadmap 与本文档一致"

经实际查阅 `docs/01_roadmap.md:349`，roadmap 中的 `GLM_CONFIG_CLIP_CONTROL` 默认值**已为 `0x0A`**（与设计文档一致），跨文件冲突事实上**已不存在**。

**所在位置**：§1 P7（第 37 行）、§7 D38（第 1686 行）、§12.2 CLIP_CONTROL 验收项（第 2689 行）

**严重程度**：严重

**影响分析**：
- P7 被升级为"需外部协调"的外部跟踪项，但协调的前提已不成立，实施者被误导认为存在一个实际不存在的阻塞项
- 编码阶段实施者将花费时间验证一个已解决的不存在的冲突
- §12.2 的 CLIP_CONTROL 验收项的"一致性确认"已被 roadmap 实际内容自动满足

**改进建议**：
1. 将 P7 状态从"🔴 跟踪项——需外部协调"降级为"✅ 已解决（roadmap 已同步）"或直接关闭
2. 删除 §7 D38 中关于 roadmap 冲突的协调声明段落（冲突已不存在）
3. 将 §12.2 CLIP_CONTROL 验收项从"编码前一致性确认"降级为"已确认一致"或移除
4. 全文搜索所有仍引用"roadmap 记录为 0x02"的措辞并同步修正

### 问题 2（严重）：P37 API 映射覆盖度声明与设计完整性声称存在实质性差距

**问题描述**：首轮 11 个核心文件中仅 4 个 Vec 类型文件（`type_vec{1,2,3,4}.hpp/.inl`）完成了逐 API 级对照（附录 A）。其余 7 个文件（`setup.hpp`、`qualifier.hpp`、`compute_vector_relational.hpp`、`compute_vector_decl.hpp`、`_vectorize.hpp`、`fwd.hpp` 及 shim 依赖）仅以"子范围粒度"映射至 §8.2/§8.5，**未做逐 API 级覆盖验证**。

P37 状态声明已从"11 个完成"修正为"4/11 个完成"，但设计文档同时声称"产出应包含完整的架构设计，满足后续编码实现的需要"（需求 §4）。对于以下实际场景，当前的设计完整性不足以直接指导编码：

- `setup.hpp` 中约 1000+ 行平台/编译器检测宏的首轮裁剪范围是否能完整闭合，仅通过"约 80-120 行配置常量"的估算，缺乏逐宏级别的确认
- `qualifier.hpp` 中的辅助设施（`genTypeTrait`、`init_gentype` 等）虽已声明推迟，但未必穷尽了所有被间接引用的设施
- 附录 A 声明的"其余 7 个文件的逐 API 级对照规划在第 2 轮编码阶段"意味着**首轮编码阶段开始时**这些文件的 API 覆盖完整性仍处于未验证状态

**所在位置**：§1 P37（第 67 行）、附录 A 验证范围声明（第 2757 行）

**严重程度**：严重

**改进建议**：
1. 评估其余 7 个文件中是否存在当前设计依赖于其某个 API/宏但尚未显式映射的设施——至少在启动编码前完成交叉引用审计
2. 若无法在编码前完成 7 个文件的逐 API 级对照，应在 §8.2 范围表中精确标注每个文件的风险等级（"高——依赖已知可替代"、"中——未发现未覆盖依赖"、"低——无直接引用"）
3. 在验收标准中增加一条：编码启动前，对 7 个未逐 API 对照的文件做一次快速扫描，确认无遗漏的移植依赖项
4. P37 状态中补充每个文件的覆盖风险等级，而非仅标注文件数量比例

### 问题 3（严重）：编译器行为假设密度过高导致设计确定性不足

**问题描述**：设计文档依赖 12+ 个编译器行为假设（D29/D30/A01-A19 共 20+ 项假设标注，编码前验证检查清单 24 项）。其中多项假设（如 D13 `%` 运算符定义处检查、项⑰ extend 块行为、项㉑ `@Derive[Hashable]` 约束检查策略）的验证失败将触发不同程度的设计修改。

设计虽为每项假设提供了回退方案，但以下结构性风险未被充分评估：

- **回退方案的组合爆炸**：假设并非独立——D29 和依赖③（编译期 if 分支抑制）同时回退时，影响面涉及比较语义、取模、约束策略三个独立维度。§11.13 叠加效应分析仅覆盖了 6 组已知组合，但 12+ 假设的可能失败组合数远超此范围。
- **验证时序依赖的脆弱性**：§2.1 建议优先验证 D29/D30/D29-④，但代码本身上百处引用了这些假设的预期行为。若多项同时失败，设计文档约 30-40% 的内容（涉及比较语义、取模、溢出策略、构造语义等）需要同步修订。
- **"满足后续编码需要"的风险**：编码阶段实施者面临的情况可能是"设计文档描述了 A 路径，但验证发现 A 路径不可行，需要改为 B 路径，而 B 路径的联动修改分布在文档 6 个章节"。设计文档提供回退方案，但未提供回退后的**完整修订后设计**。

**所在位置**：§7 D37 编译器假设一览表（第 1656-1677 行）、§2.1 编码前验证检查清单（24 项）、§10、§11.13 叠加效应分析（第 2539-2557 行）

**严重程度**：严重

**改进建议**：
1. 对 24 项验证项进行**影响范围分类**（高/中/低），标注每项验证失败时设计文档中需要修改的章节数，帮助实施者估算风险暴露面
2. 对 D29 + 依赖③ + D30 的"三重回退"场景（三项同时失败），在 §11.13 中补充完整的修订后设计快照，而非仅描述各维度独立回退
3. 在 §1 新增"设计确定性声明"框，量化说明：当前设计在 X 项假设全部通过时可直接编码，Y 项部分回退时可基于回退方案指导编码，Z 项回退时需要停工重评估
4. 将验证项⑫（统一集成验证）的范围进一步提前——建议在编码前验证清单中将其优先级提升至第②位（次于编译器版本确认），以尽早暴露多项假设的同时失效

### 问题 4（一般）：§11.12 迁移成本合计表乐观场景算术不一致

**问题描述**：总计表中乐观（optimistic）场景标注为 **6.0 人天**，但逐项累加各项乐观值（0.3+0.5+0.3+0.2+0+0+0.5+0+0.3+0.3+1.0+0.5+1.0+0.5+0+0.1+0.1+0.3）结果为 **5.9 人天**，偏差 0.1 人天。

最可能（12.15）和悲观（22.5）合计经验证正确。

**所在位置**：§11.12 合计表（第 2505 行）

**严重程度**：一般

**影响分析**：差距极小（0.1 人天），在估算精度范围内，但对合计表的算术准确性产生质疑，可能削弱读者对成本评估整体可靠性的信心。

**改进建议**：将乐观合计值修正为 **5.9 人天**，或复核各分项乐观值确认是否有 0.1 人天的遗漏项（如 §11.15 的 0.1 人天是否被重复计算或某分项值是否需要微调）。

- §10 "equalEpsilon 设计阶段验证"子节（第 1975-1986 行）仍保留了完整的三列验证表（依赖项、风险等级、对应 §2.1 检查项），与 §2.1 中的相关内容有信息冗余
- §10 "mod 函数边界值验证"表（第 1954-1961 行）同样保留了独立的验证方法描述
- §12.3 决策映射表（第 2697-2710 行）各行仍包含了验证方法和路径描述，部分与 §2.1 重复

实施者编码时仍需在三个章节间交叉引用才能获得完整的验证信息全景。

**所在位置**：§2.1 维护原则（第 227 行）、§10 equalEpsilon 验证表（第 1975-1986 行）、§10 mod 验证表（第 1954-1961 行）、§12.3 决策映射表（第 2697-2710 行）

3. 在最终状态中实现：实施者只需阅读 §2.1 即可获得所有验证任务的完整描述，§10 仅包含设计特有上下文行为描述，§12.3 仅包含路径选择映射

### 问题 6（一般）：缺少面向编码实施者的集中式快速参考

**问题描述**：设计文档的 2934 行以内部实现视角组织，关键信息分散于多个章节。以下信息实施者在编码时需要频繁查询，但目前分散在文档各处：

- **构造函数签名**: §4.1（完整清单）——但有部分行为约定散布在 §3.2（const init 分析）和 §9.4（类型转换边界）
- **运算符行为**: §4.3（算术）、§4.4（位运算）、§4.5（比较运算）——但 Vec1 广播运算规则在 §4.3 的子节中

- **异常行为**: §5.2（标准）、§9 系列（补充）——分别位于文档前部和后部

**所在位置**：全篇

**严重程度**：一般

**改进建议**：
1. 在文档开头新增"编码实施快速参考"框，列出编码阶段最常查阅的章节及其定位
2. 或独立生成一份精简版"编码参考手册"（1-2 页），从实施者视角（而非设计审议视角）组织信息：类型定义→构造→运算→别名→测试

## 整体质量评价

设计文档在第 10 轮迭代后，在**技术细节的完整性和结构化方面达到了较高水准**——构造函数全量签名清单、运算符设计模式模板、异常行为边界表、迁移成本分项评估等均为编码实施提供了充分依据。

但存在以下**系统性风险**：

1. **数据陈旧风险**（问题 1）：GLM_CONFIG_CLIP_CONTROL 冲突声明基于过时的 roadmap 信息，说明设计文档对依赖项的实时状态跟踪存在盲区。需建立机制确保文档中引用的外部状态（如 roadmap 内容）与实际文件一致。

2. **验证驱动的设计不确定性**（问题 3）：文档依赖的 12+ 个编译器行为假设使得设计的实际可行性严重依赖未经验证的编译器行为。编码前验证阶段的结果可能在文档中引发 30-40% 内容范围的联动修改。

3. **完整性声称与实际覆盖的差距**（问题 2）：文档声称"满足后续编码实现需要"，但 7/11 个核心文件未做逐 API 级对照，附录 A 的验证范围声明与完整性声称之间存在可量化的差距。

4. **信息组织可改进**（问题 5、6）：验证清单消重未彻底、缺少面向实施者的快速参考，增加了编码阶段的信息查找成本。

以上问题的核心整改方向：**修正过时外部引用、量化设计不确定性、缩小完整性声称与实际覆盖的差距、消重验证信息、补充编码快速参考**。
