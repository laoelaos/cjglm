## 接口完备性分析（本轮新增）

**下游消费者视角**：Vec 类型通过 `public import` 从 `glm.detail` 重导出到 `package glm`，下游 `import glm.*` 即可使用所有公共类型和别名。Qualifier 实现类型（Aligned 系列为 `internal`）的可见性策略合理——首轮只需 Packed 系列。

**API 签名完备性**：§4.1 给出 Vec1~Vec4 的完整构造函数签名清单。§4.3 给出算术运算符和具名函数的完整语法模板。§4.4 给出位运算符 `extend` 块模板。§4.5 给出比较运算符和容差比较函数的完整定义。每个 API 的仓颉语法形态均已给出。

**重载决议透明性**：§4.1 明确标注了 Vec1 构造函数不对称性（`const init` 与 `init(scalar)` 重载冲突）、Vec4 约 30 个重载版本的歧义风险分析和回退方案。调用方可通过文档预见可能的重载解析行为。

**潜在问题**：§4.3 中 `scalar-op-vec` 方向具名函数（`add(s, v)` 等）的导入路径依赖 `import glm.detail.{ add, sub, mul, div, mod }`，但 §2 公共 API 面设计的 `public import` 示例代码（第 163-170 行）使用的导入语法为 `import glm.detail.{ add, sub, mul, div, mod }`——此导入模式在仓颉中是否将包级函数正确重导向 `package glm` 命名空间，依赖编译器行为验证。此依赖在 §2.1 验证项⑱中已覆盖，但若验证失败需切换到 `public type` 别名替代方案，这将改变下游消费者的 API 调用语法。此风险在设计中已识别但未在 §12.2 验收标准中单独列为验收项。

### 问题 2（严重）：附录 A 及 §8.5 中"构造函数不能为泛型"的排除理由存在事实错误

**描述**：附录 A 未覆盖 API 汇总表（第 2805 行、第 2791 行）及 §8.5 排除清单（第 1896 行）将完全泛型多元构造函数的排除理由写为"仓颉构造函数不能为泛型（运算符重载限制）"。此陈述与仓颉语言事实不符——本文档 §4.1 中已大量使用 `public init<T2, Q2>(v: VecN<T2, Q2>) where Q2 <: Qualifier` 形式的泛型构造函数，证明仓颉构造函数完全可以拥有泛型类型参数。实际原因是：各分量独立推导的多元泛型参数（A/B/C/D 各可不同）在重载解析层面与已有 `const init(x: T, y: T)` 等构造函数产生歧义，且各分量需统一为目标类型 `T`。此问题在上一轮诊断（b_v12_diag_v3.md 问题 2）中已被识别且未修复。跨轮次计数：此问题首次识别于 b_v12_diag_v2.md，经 v3 诊断重申，至今未被修复——已跨越至少 3 轮诊断周期。

**所在位置**：
- 附录 A Vec1 对照表（第 2791 行）— "完全泛型双参数构造"备注
- 附录 A Vec2 对照表（第 2805 行）
- §8.5 排除清单（第 1896 行）

**严重程度**：严重（事实性错误，且跨 3 轮未修复）

**改进建议**：将上述三处排除理由统一修正为以下准确表述："各分量类型可独立推导的完全泛型多元构造函数（如 C++ `template<A,B> vec2(A,B)` 允许 A≠B）在仓颉中无法直接映射——此泛型构造函数与已有 `const init(x: T, y: T)` 在调用方传入同类型参数时产生重载歧义，且各分量需通过 `T(v)` 转换为统一的目标类型 `T`。C++ 该模式已由标量填充构造、逐分量 const init、跨 Vec 转换构造等已有构造函数形态覆盖，无需单独映射。"
