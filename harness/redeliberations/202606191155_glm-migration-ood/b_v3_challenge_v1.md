# 质量质询报告（v3）

## 质询结果

CHALLENGED

## 逐维度审查

### 1. 证据充分性

**[通过]** 问题 1（Vec1 广播运算遗漏）：诊断报告主张 C++ GLM 的 `type_vecN.inl` 中为 Vec2~Vec4 均提供了 `vecN op vec1` 和 `vec1 op vecN` 广播运算符。经对照 C++ GLM 源码确认，`type_vec2.inl`/`type_vec3.inl`/`type_vec4.inl` 中确实存在大量 `vec<N,T,Q> op vec<1,T,Q>` 非成员运算符重载（涵盖 `+`、`-`、`*`、`/`、`%`、`&`、`|`、`^`、`<<`、`>>` 共 10 种运算符，各 VecN 各 2 方向），且 OOD 设计文档 §4.3 的运算符清单和语法模板中仅列出 `VecN op VecN` 和 `VecN op T` 两种右操作数类型，§8.5 排除清单中也未提及此模式，证据充分。

**[通过]** 问题 4（`Vec` 家族别名同名问题）：诊断报告指出 `fwd.cj` 中 `public type Vec2 = Vec2<Float32, PackedHighp>` 的左侧别名 `Vec2` 与右侧被导入的 `glm.detail.Vec2` 泛型结构体同名可能产生自引用歧义。经对照设计文档 §3.8 脚本模板确认，`Vec` 家族确实以 `suffix = ''` 生成无后缀别名（如 `Vec2`），而 `fwd.cj` 通过 `import glm.detail.{ Vec2 }` 引入右侧类型。此同名问题未在设计文档中讨论，证据充分。

**[通过]** 问题 5（`abs()` 来源未指定）：经对照 §4.5 `equalEpsilon` 代码示例和 §3.5 `ComputeEqual.callConst` 代码，确认 `abs(a - b)` 调用未指定 `abs` 的来源包或类型。证据充分。

**[问题-严重]** 问题 2（Vec2 算术运算符语法模板签名缺漏）与问题 1 核心事实主张完全相同——均指出设计遗漏了 `VecN op Vec1` 运算符。问题 2 称其侧重"语法模板自身签名缺漏"，但所提供的证据与问题 1 无增量差异。诊断报告未说明为何需要两个独立问题项来描述同一事实遗漏，也未分析问题 2 描述的错误是否可在问题 1 的改进中自然消解。

**[问题-严重]** 问题 3（复合赋值运算符接受 Vec1 广播参数）标注为"严重"，但其存在完全依赖问题 1/2 的修复决策——仅当设计决定提供 `VecN op Vec1` 运算符时，编译器才会自动生成 `+=Vec1` 版本；若决定不提供，`VecN += Vec1` 不可用是问题 1/2 的自然推论而非独立问题。诊断报告未评估在问题 1/2 已识别的前提下，将此条件性问题单独标注为"严重"是否夸大了独立严重程度。

**[建议]** 1. 将问题 2 合并至问题 1，或改为问题 1 的子项（说明语法模板层面的具体缺漏作为问题 1 改进建议的一部分）；2. 将问题 3 从"严重"降级为"一般"或合并至问题 1 的改进建议中，显式标注其依赖关系——"若问题 1 决定提供 Vec1 广播运算符，则须同步处理复合赋值"。

### 2. 逻辑完整性

**[通过]** 问题 1/3 之间的依赖关系诊断报告已有描述（问题 3 明确声明"若问题 1/2 的改进方案决定提供 … 则复合赋值 … 将由编译器自动生成"），逻辑关系表达清楚。

**[通过]** 各问题的改进建议与问题本身一致且可行。

**[问题-严重]** 诊断报告在"整体质量评价"中称"历史迭代的 23 个问题中全部标注为已解决"，但未实际验证 v4 文档中这些问题是否真正被充分解决——仅引用了设计文档自身的 P1~P23 状态追踪表（该表由设计文档作者填写）。作为质量审查报告，应至少抽查 2~3 项历史问题的实际修复状态，而非直接采信设计文档的自我声明。此疏忽降低了"整体质量评价"段落的可信度。

**[建议]** 在整体评价中对至少 2~3 项历史严重问题（如 P7 CLIP_CONTROL 措辞、P14 const init 共存性、P15 % 运算符编译器行为）进行抽查验证，确认其修复状态与设计文档声明一致。

### 3. 覆盖完备性

**[通过]** 任务描述要求识别"需求响应充分度"、"事实错误或逻辑矛盾"、"深度和完整性"三个维度。诊断报告的问题 1~6 覆盖了 API 完整性（需求响应）、事实错误/设计遗漏（逻辑矛盾）、排除清单缺失（完整性），维度覆盖充分。

**[问题-严重]** 诊断报告识别了 Vec1 广播运算的遗漏（问题 1），但未进一步评估此遗漏对下游矩阵类型迁移的连锁影响。设计文档 §3.2 明确声明 Vec2~Vec4 是矩阵类型的内部列类型，矩阵运算（如 `mat * vec`）广泛使用 `vec1` 广播模式。C++ GLM 中 `type_mat2x2.inl` 等矩阵实现文件的运算符大量通过 `vec + vec1` 形式进行分量级操作。诊断报告虽提及"下游代码（特别是矩阵运算 `mat * vec` 等）广泛使用"，但未将此连锁影响作为独立评估项——即在矩阵类型迁移轮次中，Vec1 广播缺失将迫使矩阵运算实现者逐一将 `vec + vec1` 改写为 `vec + vec1.x` 形式，迁移成本显著上升。此连锁影响分析对问题 1 的严重程度判定有直接支撑作用，但诊断报告仅一笔带过。

**[问题-一般]** 路线图 §3E 首轮迁移文件清单中 `type_vec1.hpp` + `type_vec1.inl` 作为子范围 4 被明确纳入首轮。但 OOD 设计文档 §4.3 中 Vec1 的运算符设计相对薄弱——Vec1 的二元运算符 `Vec1 op Vec1` 本质等价于标量运算，Vec1 与标量 `T` 的运算关系未单独分析（如 `Vec1<Int32> + Int32` 的 `operator func +(rhs: T)` 在 Vec1 上与 `Vec1 + Vec1` 语义重叠是否导致调用方困惑）。诊断报告未覆盖 Vec1 自身的运算符设计完整性。

**[问题-一般]** C++ GLM 中 `vec<1, T, Q>` 的 `operator+=(vec<1, U, Q> const& v)` 等复合赋值允许跨类型 U（类型参数独立于 struct 的 T）。仓颉复合赋值由编译器自动生成，仅接受同类型右操作数——即 `Vec1<Int32> += Vec1<Float32>` 在仓颉中不可用，而 C++ 中可用。诊断报告仅覆盖了 `VecN<Bool> += VecN<Bool>` 等布尔分支问题（历史迭代），未覆盖整数/浮点跨类型复合赋值的损失。

**[建议]** 1. 在问题 1 中补充矩阵类型迁移连锁影响的独立分析片段，将"矩阵运算 `mat * vec` 广泛使用 `vec + vec1`"扩展为对矩阵迁移成本的定量评估；2. 补充 Vec1 运算符设计完整性审查；3. 补充跨类型复合赋值的可用性损失评估。

### 4. 报告必要性

**[问题-一般]** 问题 7（验证清单编号体系）属于呈现格式细节——Unicode CJK 带圈数字的可读性/兼容性问题是文档编辑体验层面的问题，与 OOD 设计文档的可直接指导编码实现的核心职责关联度低。此问题即使不修复也不影响编码实施者对验证清单的理解和使用。同理，问题 9（验证清单退出规则）属于验证流程管理的次要细节，对"设计是否可直接指导编码实现"这一核心审查目标影响微弱。

**[建议]** 将问题 7 和问题 9 从诊断报告中移除或合并为"验证清单呈现优化"次要建议条目，避免分散审查报告对核心问题的聚焦。

## 质询要点

### 要点 1

- **问题**：问题 1 与问题 2 描述的是同一事实遗漏（Vec1 广播运算符缺失），诊断报告将其拆分为两个"严重"级独立问题，导致严重程度膨胀
- **原因**：两个问题共享同一证据基础（C++ GLM `type_vecN.inl` 中的 `vecN op vec1` 运算符），仅侧重点描述角度不同。拆分后产出作者可能对同一修复点执行两次独立改进，造成工作量估算偏差
- **建议方向**：合并问题 1 和问题 2，将"语法模板签名缺漏"作为问题 1 改进建议的子项；问题 3 降级为"一般"或作为问题 1 的条件性补充建议

### 要点 2

- **问题**：整体质量评价段直接采信设计文档的自我声明（P1~P23 已解决），未进行任何抽查验证
- **原因**：质量审查报告的可信度依赖于独立验证，直接引用被审查对象的自我声明构成循环论证，降低了审查结论的可靠性
- **建议方向**：对 2~3 项历史严重问题进行抽查（如 P7、P14、P15），确认实际修复状态与声明一致

### 要点 3

- **问题**：Vec1 广播运算遗漏对矩阵类型迁移的连锁影响未被充分分析
- **原因**：诊断报告仅提及"下游代码广泛使用"但未展开定量评估，使问题 1 的严重程度判定缺乏充分的下游影响支撑
- **建议方向**：在问题 1 中补充矩阵迁移连锁影响分析，评估 Vec1 广播缺失在矩阵轮次中的具体迁移成本增量
