### 问题 2（一般）：`fromBoolVec`/`fromBoolVecQ2` 的 Vec1 构造函数类型参数依赖编译器推断行为

§4.8 的 `fromBoolVecQ2` 实现代码中（以 Vec1 为例）：
```
public func fromBoolVecQ2<T, Q, Q2>(v: Vec1<Bool, Q2>): Vec1<T, Q> where Q <: Qualifier, Q2 <: Qualifier {
    Vec1(if (v.x) { T(1) } else { T(0) })
}
```
函数体内 `Vec1(...)` 调用未显式标注类型参数 `<T, Q>`，依赖仓颉编译器从函数返回值类型 `Vec1<T, Q>` 推断 `Vec1` 构造函数的泛型实参。此推断行为在当前设计文档中**未被验证覆盖**——§2.1 编码前验证清单无对应项。若编译器不支持从返回值类型推断 struct 构造函数的泛型实参，上述代码将编译失败。

**改进建议**：在 §4.8 中标注此项编译器推断依赖（如脚注"仓颉编译器是否支持从返回值类型推断 struct 泛型参数，编码阶段须验证"），或在 §2.1 编码前验证检查清单中补充一项原型验证。

### 问题 3（一般）：`equalEpsilon` 的 Path A/Path B 决策依赖链仍未全量可视化

§3.5 定义的 `equalEpsilon`/`epsilonOf<T>()` 双路径（路径 A：保留 `epsilonOf<T>()` → `T(1e-6)`；路径 B：完全移除，内联固定容差）是跨章节影响的关键决策点，涉及 §3.5、§4.5、§10（依赖④）、§12.2（回退联动验收项）、§12.3（决策映射表第 11 行）共 5 处分布。虽然 §12.3 第 11 行新增了"修改触发联动检查清单"列，清晰列出了路径 B 的 6 处联动修改，但**路径 A 触发的联动修改清单缺失**——仅描述了路径 A 修改一处 `epsilonOf<T>()`，未说明 `epsilonOf<T>()` 的 `Number<T>` 约束移除后，`ComputeEqual.callConst` 中的 `if (isIec559Of<T>())` 分支内 `abs(a - b) <= epsilonOf<T>()` 是否仍需要处理整数 `T` 的兼容性问题。

**改进建议**：在 §12.3 第 11 行的联动清单列中补充路径 A 的触发条件和检查项："路径 A 触发时：修改 `shim_limits.cj` 中 `epsilonOf<T>()` 定义（移除 `Number<T>` 约束，改为 `T(1e-6)`）→ 验证编译期 `if` 分支抑制保护有效（§2.1 项④⑫）→ 确认整数 `T` 不进入 `epsilonOf<T>()` 调用路径 → 若抑制保护无效，回退至路径 B"。

### 问题 4（轻微）：`fromBoolVec` 对 `T = Bool` 的语义描述措辞可能误导

§4.8 第 1198 行："当 `T = Bool` 时，`fromBoolVec` 生成 `VecN<Bool, Q>`，语义为'Bool 值保持不变'（`true→T(1)=Bool(1)=true`，`false→T(0)=Bool(0)=false`），即此函数等价于恒等映射。"此表述的隐含推论是 `fromBoolVec` 对 `Bool` 输入返回相同值的 `VecN<Bool, Q>`——但在跨 Q 场景中（`fromBoolVecQ2`），若输入 `Bool` Vec 的 Q2 与输出 Q 不同，返回值并**不等价于**输入（Qualifier 类型改变），"恒等映射"的陈述不准确。

**改进建议**：将"即此函数等价于恒等映射"修改为"即此函数的值语义等价于恒等映射（分量值保持不变，但 Qualifier 类型可能变化）"。
