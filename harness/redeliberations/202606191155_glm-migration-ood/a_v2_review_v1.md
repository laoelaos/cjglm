# OOD 设计方案审查报告（v2）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** Qualifier 体系采用 `interface Qualifier` + 空结构体实现的模式与仓颉泛型系统能力匹配——空结构体可作为泛型类型实参传递，`where Q <: Qualifier` 约束在仓颉泛型约束语法中合法。接口的多实现（PackedHighp/Mediump/Lowp/Aligned 系列）在仓颉单继承多接口实现约束内可行。

**[通过]** 四个独立 Vec 结构体（Vec1~Vec4）各自定义泛型 `<T, Q> where Q <: Qualifier`，避免仓颉不支持偏特化的限制，设计合理。

**[通过]** 跨类型构造函数使用无约束泛型参数 `T2`（不使用 `where T2 <: T`），符合仓颉数值类型间无子类型关系的事实。D14 的选择理由充分。

**[通过]** Functor 体系按分量数拆分为 Vec1~Vec4 四组独立结构体，回避了仓颉不支持模板模板参数的限制。ComputeVec* 运算策略同样按分量数独立定义，设计模式一致。

**[通过]** `@Derive[Hashable]` 在泛型 struct `VecN<T, Q>` 上的使用在仓颉中可行——首轮目标类型（Int8~Int64、UInt8~UInt64、Float32/Float64、Bool）均已实现 `Hashable`，_script_文档确认 `@Derive` 对泛型 struct 可用，约束传递性已有验证计划。

**[通过]** 类型别名 `public type` 用于 `fwd.cj` 中的 256 个别名定义，仓颉 `type` 关键字支持泛型类型别名且不支持 `where` 约束（符合设计要求——别名直接映射到已具现化类型），可见性通过 `public` 修饰符控制，与包组织一致。

### 2. 标准库与生态覆盖

**[通过]** 浮点取模使用 `std.math.trunc`（Float64 版本），Float32 提升至 Float64 后调用。`std.math` 文档确认 `trunc` 函数可用。精度策略依赖项已在 §10 中识别并安排验证。

**[通过]** `shim_limits.cj` 中 `NumericLimits<T>` 替代 C++ `std::numeric_limits`，`isIec559Of<T>()` 使用编译期 `if` + `is` 运算符实现类型检测，不依赖标准库。

**[通过]** `shim_assert.cj` 使用 `if + throw` 模式替代 C++ `assert` 宏，仓颉异常体系支持此模式。

**[通过]** `shim_cstddef.cj` 定义 `SizeT = UInt64`、`LengthT = Int64`，VArray 的 `size` 属性替代 `countof` 宏，均在仓颉标准库能力范围内。

**[通过]** 测试使用 `std.unittest` 框架（`@Test`/`@TestCase`），cjpm `test` 子命令支持自动发现，标准库覆盖充足。

**[通过]** `@OverflowWrapping` 注解为仓颉内置注解，仓颉注解文档确认三种溢出策略注解可用且只能标记于函数声明上。

**[轻微]** `std.math.trunc` 仅提供 Float64 版本，Float32 路径需提升-截断-降级处理。此精度策略的数值正确性已安排在 §10 验证依赖项②中覆盖，设计合理但存在边界值风险（如 Float32.Max 取模）。建议编码阶段优先验证 Float32 边界场景。

### 3. 语言特性可行性

**[通过]** 错误处理策略：`shim_assert.cj` 使用 `if + throw` 模式替代 C++ assert 宏，整数溢出通过 `@OverflowWrapping` 控制，除零依赖仓颉运行时默认行为。各策略与仓颉错误处理能力（异常体系 + 溢出注解）匹配。

**[通过]** 首轮无共享可变状态并发场景（Vec 为值类型，运算产生新副本），值语义线程安全，无需同步原语。设计声明了复合赋值非原子性，调用方需自行加锁——此约定合理。

**[通过]** `const init` + `const` 实例成员函数的模式在仓颉中可行——const 文档确认 struct 定义 `const init` 后可定义 `const` 实例成员函数，`const init` 在非 const 上下文中也可用于运行时构造。

**[通过]** `extend` 块不支持 `const` 修饰符的限制已由 D32/D33 统一处理——所有扩展成员函数声明为非 `const`，包级独立函数声明为 `const`。`componentAt` const 函数定义在 struct 体内，避免了 extend 块限制。

**[通过]** cjpm 项目组织方式：静态库项目（`output-type = "static"`）、包声明（`package glm` / `package glm.detail`）、`public import` 重导出、测试目录配置（`[test] src-dir = "tests"`）均符合 cjpm 文档描述的项目结构规范。

**[通过]** 编译期 `if` 分支抑制行为是 `const` 函数的固有语义。const 表达式列表确认 `is`、`if` 均可参与 const 表达式。D29/D30 组合回退场景分析完整，无不可调和冲突。

**[轻微]** `@OverflowWrapping` 标注继承性（编译器自动生成的复合赋值是否继承标注）依赖编译器未文档化行为。设计已正确识别此风险（D30），提供了完整回退方案（补充 20 处已有函数标注），并在 §2.1 编码前验证清单中安排验证。

**[轻微]** `equalEpsilon` 的 const 真空（D29 回退后 extend 块中非 const 函数在 const 上下文不可用）是已知限制。§4.5 已提供三条缓解路径，路径②（使用精确比较 `==`）零成本可行。

### 4. 设计一致性

**[通过]** 各抽象的职责描述清晰——Qualifier 负责编译期精度/对齐标记，Vec 负责向量数据与运算，Shim 层替代 C++ 标准库，Functor/ComputeVec* 为后续轮次预留。职责划分无重叠。

**[通过]** 协作关系形成闭环：Vec → Qualifier（类型约束）→ Setup（配置常量）→ Shim（运行时支持）→ ComputeEqual（比较策略）→ Vec（== 运算符委托）。`fromBoolVec` 工厂函数作为 Bool→Numeric 转换的闭环补充。`scalar_vec_ops` 辅助函数作为标量在左运算的闭环补充。

**[通过]** 构造函数体系（§4.1 完整清单）、运算符重载（§4.3~§4.6）、比较策略（§4.5/§3.5）、错误处理（§5）的行为契约已描述到足以指导后续实现的程度——每项行为约定包含签名、语义和编译/运行时预期结果。

**[通过]** 模块间依赖方向合理：`glm.detail` 包内部依赖形成有向无环图（setup → qualifier → compute_vector_relational → type_vecN → scalar_vec_ops），`glm` 包单向依赖 `glm.detail`，反向无依赖。§2 模块依赖图已修正箭头方向（scalar_vec_ops → type_vecN）。

**[通过]** D5/D14/D15/D27 等设计决策对仓颉语言限制（不支持隐式转换、偏特化、类型参数默认值）的处理方式一致——通过显式构造、独立类型定义、类型别名等方式弥补，无矛盾。

**[通过]** 历史问题追踪表（P1~P13）标记了前两轮循环问题的解决状态，迭代需求中的 7 个审查意见均已对应解决（修订说明 v3 表格逐一列出了修改措施），闭环追踪有效。

**[轻微]** §7 D6 中 `GLM_CONFIG_CLIP_CONTROL` 默认值 `0x0A` 与 roadmap §3A.1 记录值 `0x02` 的冲突已标记为跟踪项且要求编码前必须读取 `setup.hpp` 原始条件编译块确认，但两处文档值不一致本身可能给实施者造成困惑。建议在 v3 修订说明或 D6 表格中更醒目地标注此差异为"待验证"而非"已推导"。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：Setup 管理配置、Qualifier 管理精度标记、Vec 管理向量数据与运算、Shim 替代 C++ 标准库、scalar_vec_ops 管理标量在左辅助函数、fromBoolVec 管理 Bool→Numeric 转换。每类关注点独立成文件。

**[通过]** 抽象层次恰当：架构级设计覆盖了类型形态选择、包组织、关键行为契约、错误处理策略、编译器假设管理，不深入具体字段实现细节。§2.1 编码前验证清单提供了从设计到编码的过渡桥梁。

**[通过]** 设计便于后续详细设计和实现：§8.3 迁移顺序定义了 7 个阶段含增量验证节点，§4.1 构造函数清单提供了可直接编码的签名列表，§3.8 Python 脚本模板可自动生成 256 个别名，§4.3 运算符语法模板可直接参考。

**[通过]** 设计便于单元测试：各 Vec 类型可独立实例化测试，Shim 层可独立验证，包级独立函数可独立调用，`internal` 类型测试访问策略提供了 4 级回退方案（§12.3）。`fromBoolVec` 工厂函数不依赖 receiver 实例，调用模式自然利于测试。

**[通过]** 编译器行为假设集中管理（D37 表格 19 项假设）+ §2.1 编码前验证检查清单 15 项验证任务 + 各假设的独立回退方案，确保了设计在不确定编译器行为前提下的可控性。

**[轻微]** Vec4 构造函数约 30 个重载版本的验证覆盖策略从"6 组对偶分支"升级为"参数数量全扫描"，但仍依赖编码阶段原型测试确认——设计层面的风险已识别和标注，回退方案（移除 4-6 个歧义分支改为具名工厂函数）可行。建议在编码阶段尽早验证 Vec4 重载解析。
