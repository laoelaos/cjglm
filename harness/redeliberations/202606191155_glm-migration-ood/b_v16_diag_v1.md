## 发现问题

### 问题 1（严重）：P37 "已解决（v16）"状态声明与实际解决程度存在实质性差距

**问题描述**：§1 P37 状态追踪表标注为 "✅ 已解决（v16）"，声称问题已闭环。但实际解决措施是：为 7 个非 Vec 核心文件（setup.hpp、qualifier.hpp、compute_vector_relational.hpp、compute_vector_decl.hpp、_vectorize.hpp、fwd.hpp、shim 文件）新增了一个"依赖项核查清单"（§1 第 97-117 行），并将逐 API 级对照规划推迟到"第 2 轮编码阶段"。

核查清单本质是高级别扫描清单（每个文件 3~5 个核查要点），远未达到"API 映射完整性"所需的逐函数行号级对照粒度。附录 A 对 4 个 Vec 文件做到了逐 API 对照，但对 7 个非 Vec 文件的映射深度与 Vec 文件差距显著——附录 A 验证范围声明（第 2877 行）明确"本附录不做其余 7 个文件的逐 API 级对照"，§1 的解决声明却未同步体现此粒度差异。

**所在位置**：§1（第 95 行）问题状态追踪表 P37、"7 文件快速扫描依赖项核查清单"、附录 A 验证范围声明（第 2877 行）

**严重程度**：严重

**改进建议**：P37 状态描述应从 "✅ 已解决（v16）" 修正为更精确的表述，如 "✅ 部分解决——4/11 文件（type_vec1~4）已完成逐 API 级对照；7 个非 Vec 文件已提供核查清单，逐 API 级对照推迟至编码阶段"。或将 P37 拆分为两个子问题，分别追踪：P37a（Vec 4 文件——已解决）、P37b（非 Vec 7 文件——推迟至编码阶段）。

### 问题 2（一般）：§3.2 const 上下文 API 可用性速查表中 `decrement()` 的 const 替代方案存在语义错误

**问题描述**：`v.decrement()` 行的 const 替代方案标注为"改用 `sub(T(1), v)`（包级独立函数，声明为 `const`）"。但 `sub(s: T, v: VecN<T,Q>)` 的语义定义为标量减向量（`s - v`），因此 `sub(T(1), v)` 计算的是 `T(1) - v_i`（`1 - 分量值`），而非 decrement 所需的 `v_i - 1`。减法的不可交换性使此替代方案产生错误的计算结果。此问题在 v12 修订说明中曾被认为已修复（第 3093 行），但当时仅修复了 `increment()` 行（加法可交换，`add(T(1), v)` 等价于 `v + 1` 正确），未发现 `decrement()` 行的语义不对称性。

**所在位置**：§3.2 const 上下文 API 可用性速查表（第 477 行）

**严重程度**：一般

**改进建议**：将 `decrement()` 的 const 替代方案修正为能正确表达 `v - 1` 语义的形式：
- 若仓颉二元运算符 `-` 在 struct 体内可声明为 `const operator func`，则改为"改用 `v - T(1)`（二元运算符 `-`）"
- 否则改用"在 struct 体内定义 const 版本的 decrement 函数，或使用 `v + (-T(1))` 通过加法实现递减效果"

同步检查 `increment()` 行是否同样存在此问题——加法可交换故不影响。

### 问题 3（一般）：`@OverflowWrapping` 标注对浮点运算符代码生成的潜在影响缺乏验证覆盖

**问题描述**：§4.6 在浮点类型 `@OverflowWrapping` 说明段（第 1229 行）明确承认标注可能"抑制编译器对 NaN/Infinity 检测的优化路径"，并指出"若编码阶段发现非预期的编译优化行为"需采取应对措施。但 §2.1 编码前验证清单的 25 项验证中，没有任何一项针对此风险设计验证方法。项⑤和项⑫仅验证 `@OverflowWrapping` 的继承性和语法共存性，不覆盖浮点标注对编译器代码生成语义的副作用。

此风险若在编码阶段才暴露（如发现带标注的浮点运算出现错误的 NaN/Infinity 传播），将导致 §4.3 全部 20 个浮点算术运算符的标注策略需要从"统一标注"改为"条件性标注"（仅整数类型标注），触发大规模设计修订。

**所在位置**：§4.6（第 1229 行）浮点 `@OverflowWrapping` 编译期行为影响段落、§2.1 编码前验证清单

**严重程度**：一般

**改进建议**：在 §2.1 编码前验证清单中新增验证项，或扩展项⑤/⑫，加入对 `@OverflowWrapping` 浮点标注副作用的验证步骤：
- 编写原型测试：对 `Vec4<Float32>` 执行含 NaN/Infinity 分量的算术运算，对比标注 `@OverflowWrapping` 与不标注时的浮点运算结果
- 若发现语义差异，则制定条件性标注策略（仅整数 T 标注）并评估修改范围

### 问题 4（轻微）：§7 D6 `GLM_CONFIG_CLIP_CONTROL` 常量的 "已验证结论" 措辞与 §8.4 验证项②的依赖关系未显式关联

**问题描述**：§7 D6（第 1407-1417 行）中 `GLM_CONFIG_CLIP_CONTROL` 的默认值推导结论标注为"已验证正确"，但该验证涉及的具体方法（读取 `setup.hpp:563-584` 的原始条件编译块、逐行确认分支走向）记录在 §8.4 参考实现使用策略的前置验证项①中。两处没有交叉引用，编码实施者在阅读 §7 D6 时无法直接定位到验证方法说明，需要自行搜索发现 §8.4 的验证步骤。

**所在位置**：§7 D6（第 1417 行）默认值推导结论段落、§8.4（第 1894 行）前置验证项①

**严重程度**：轻微

**改进建议**：在 §7 D6 "已验证正确" 措辞后添加指向 §8.4 的交叉引用脚注，如 `[^verify] 验证方法详见 §8.4 前置验证项①——编码前必须读取 setup.hpp:563-584 的原始条件编译块逐行确认分支走向`。使实施者从设计决策处可直接跳转到验证方法说明。

### 问题 5（轻微）：§8.3 阶段 0 "非 Vec 文件 API 快速扫描" 与 P37 的延迟对照存在执行时序矛盾

**问题描述**：§8.3 阶段 0 被定义为"编码启动前置"步骤，要求"对 7 个非 Vec 核心文件逐项确认无遗漏移植依赖项"。但 P37 明确声明"其余 7 个文件的逐 API 级对照规划在第 2 轮编码阶段以本附录为模板补充完成"。这意味着阶段 0 的扫描粒度（核查清单级）与第 2 轮规定的逐 API 级对照不同——阶段 0 声称"确认无遗漏"但仅使用了高级别清单，与 §1 第 117 行"不做'无遗漏'的绝对声称"直接矛盾。实施者无法确定阶段 0 的退出标准究竟是"高级别扫描通过"还是"API 级覆盖确认"。

**所在位置**：§8.3 阶段 0（第 1871 行）退出标准列、§1（第 117 行）闭环原则段落、附录 A（第 2877 行）验证范围声明

**严重程度**：轻微

**改进建议**：统一阶段 0 的退出标准措辞，与 P37 的粒度声明一致。建议将阶段 0 的"通过标准"列从"7 项全部确认通过"修改为"7 项核查清单全部确认通过；逐 API 级对照延迟至第 2 轮编码阶段以附录 A 为模板补充"。消除阶段 0 退出标准与 P37 范围声明之间的措辞矛盾。

## 整体质量评价

本产出经历了 16 轮迭代，在架构完整性、设计决策追溯、验证依赖分析、迁移成本评估等方面的深度显著高于一般 OOD 设计文档。多数此前迭代中指出的严重问题（如运算器事实错误、编译器假设分散、排除清单缺失、CLIP_CONTROL 跨文件冲突等）在本版本中已得到有效修正。

主要剩余问题集中在：（1）P37 "已解决"声明的精度问题——7 个非 Vec 文件仍缺少逐 API 级映射，当前方案是核查清单 + 推迟编码阶段，并非完整解决；（2）§3.2 const API 速查表中 `decrement()` 的替代方案存在语义错误（`sub(T(1), v)` 计算 `1 - v` 而非 `v - 1`）；

上述问题不构成编码启动的阻塞条件，但应在编码实施前修正，以确保实施者获得正确的指引。
