## 问题 1（撤回修正）

**问题描述**：上一轮审查报告问题 1 声称"C++ GLM 的二元运算符同样使用了跨类型 `U`"，经质询方验证 C++ GLM 源码 `type_vec2.hpp:237-249`，所有二元运算符声明均统一使用 `T` 而非独立的 `U` 模板参数（如 `operator+(vec<2, T, Q> const& v, T scalar)`、`operator+(vec<1, T, Q> const& v1, vec<2, T, Q> const& v2)`）。仅复合赋值运算符使用跨类型 `U` 参数。因此"二元运算符跨类型 `U` 参数损失"的问题不成立——仓颉设计中二元运算符限制为同类型 `T` 与 C++ GLM 二元运算符签名功能等价，不存在不对称分析的需求。此问题与问题 2 的正确判断直接矛盾。

**所在位置**：上一轮报告问题 1（§4.3 算术运算审查段）

**严重程度**：不适用（已撤回）

**改进建议**：撤回问题 1。确认仓颉 `operator func +(rhs: T): Vec2<T, Q>` 与 C++ GLM `operator+(vec<2, T, Q> const& v, T scalar)` 功能等价。跨类型 `U` 参数损失仅存在于复合赋值运算符，§4.3 和 §8.5 已充分覆盖。

## 问题 2（保留修正）

**问题描述**：设计文档 §4.3 中将 `add(s, v)` 仓颉具名函数与 C++ GLM `operator+(T scalar, vec<N,T,Q>)` 的**功能等价性**未做显式声明。此信息对迁移者判断 `add(s, v)` 是否需要支持跨类型标量 `U` 参数至关重要——迁移者可能误以为 C++ GLM 的标量在左运算也支持跨类型 `U` 参数，从而对仓颉 `add(s: T, v: VecN<T,Q>)` 的同类型约束感到困惑。此问题与上一轮报告问题 2 一致，质询方确认通过。

**所在位置**：§4.3 标量在左的运算替代方案段落、§7 D10

**严重程度**：轻微

**改进建议**：在 §4.3 标量在左运算段落或 §7 D10 中补充声明：`add(s, v)` 的标量参数 `s` 类型为 `T`（与 Vec 分量类型相同），与 C++ GLM `operator+(T scalar, vec<N,T,Q>)` 功能等价。跨类型标量在左运算（`U + VecN<T,Q>`）在 C++ GLM 中同样不可用（因模板参数 `T` 统一约束），无需额外适配。

## 问题 5（保留）

**问题描述**：设计文档 §3.5 和 §10 中描述 `ComputeEqual` 容差比较使用 `NumericLimits<T>.epsilon()`（机器 epsilon）作为绝对容差阈值。机器 epsilon 仅适用于接近 1.0 量级的浮点值比较——对大值（如 `|a|, |b| >> 1`）容差过于宽松，对极小值（如 `|a|, |b| << 1`）容差过于严格。C++ GLM 的 `compute_equal` 同样使用 `std::numeric_limits<T>::epsilon()` 作为容差，行为一致。但设计文档未在任何显式位置声明此值域局限性及与 C++ GLM 的一致性。

**所在位置**：§3.5 ComputeEqual.callConst 容差比较路径描述、§10 依赖③验证描述

**严重程度**：一般

**改进建议**：在 §3.5 或 §9.3 中补充"机器 epsilon 容差比较的值域局限性"说明段：声明容差比较使用 `NumericLimits<T>.epsilon()` 作为绝对容差阈值，此值仅适用于接近 1.0 量级的浮点值比较；对于大值容差过于宽松，对于极小值容差过于严格。此行为与 C++ GLM 的 `compute_equal` 一致。需要相对容差比较的下游消费者应自行实现基于 `max(abs(a), abs(b)) * epsilon` 的比较逻辑。

## 问题 7（保留修正）

**问题描述**：§3.8 中 `FVec` 家族（`Float32`）与 `Vec` 家族（`Float32`）、`F32Vec` 家族与 `Vec` 家族、`F64Vec` 家族与 `DVec` 家族分别映射到相同的泛型具现化（如 `F32Vec2 = Vec2<Float32, PackedHighp> = Vec2`），存在功能冗余。设计文档未声明此冗余关系和调用方选择指引。**补充说明**：此冗余继承自 C++ GLM 源码的别名体系——C++ GLM 中 `fvec2` 与 `vec2`、`f32vec2` 与 `vec2` 同样映射到 `vec<2, float, highp>`，属于忠实迁移行为而非仓颉迁移引入的新问题。

**所在位置**：§3.8 向量类型别名（消费者快速选择指引表）

**严重程度**：轻微

**改进建议**：在 §3.8 消费者快速选择指引表中增加脚注，声明 `FVec`/`F32Vec`/`F64Vec` 与 `Vec`/`Vec`/`DVec` 分别映射到相同泛型具现化，此冗余继承自 C++ GLM 别名体系。推荐优先使用 `Vec`/`DVec` 系列别名（更简洁），`F32Vec`/`F64Vec` 系列留给需要显式标注位宽的代码风格场景。
