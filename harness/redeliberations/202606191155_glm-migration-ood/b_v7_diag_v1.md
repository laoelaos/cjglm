## 发现的问题

### 问题 1（严重）：`GLM_CONFIG_CLIP_CONTROL` 跨文件值协调已跨 5 轮迭代仍未闭环，成为阻碍实施的持续阻塞项

**问题描述**：产出 §7 D6 配置常量表声明 `GLM_CONFIG_CLIP_CONTROL` 默认值为 `0x0A`，但 roadmap §3A.1 仍记录为 `0x02`。此差异自第 2 轮迭代被发现（P7）起，已历经第 3、4、5、6 轮共 5 轮迭代修复仍未解决。v7 修订说明将 P7 状态从"✅ 已解决"降级为"🔄 持续问题"，并在 D6 中标注"编码阶段实施者须同步更新 roadmap"，但产出本身未给出任何解决此协调的确定性路径或时间线。实施者面对两个互相矛盾的文件无法确定应参考哪个值——设计文档说"已验证 0x0A 正确"但 roadmap 说 0x02，协调状态为"持续问题"意味着至今未实际解决。

**所在位置**：
- §1 问题状态追踪表 P7（第 37 行）
- §7 D6 `GLM_CONFIG_CLIP_CONTROL` 常量表（第 1306 行）
- roadmap §3A.1（01_roadmap.md 第 349 行）

**严重程度**：严重

**改进建议**：这是**设计的边界外问题**——设计文档本身的值 `0x0A` 已通过参考实现验证为正确，问题在于 roadmap 未同步更新。OOD 设计文档不应将"待实施者更新 roadmap"作为协调状态的答案。建议：① 在本产出的 §8.2 范围可追溯性对照表或 §7 末尾新增一条正式的设计决策注释，声明"经参考实现 `setup.hpp:563-584` 条件编译链验证，`GLM_CONFIG_CLIP_CONTROL` 默认值确认为 `0x0A`，此值以本设计文档为准"；② 将 P7 状态从"持续问题"升级为"依赖项——需外部协调"，明确依赖方和解决时限，而不是让实施者在编码时自行发现矛盾。

### 问题 2（严重）：验证清单在 §2.1、§10、§12.3 三处重复且组织方式不一致，实施者存在遗漏关键验证项的风险

**问题描述**：编码前验证项分布在三个不同位置，且组织逻辑不同：
- §2.1 编码前验证检查清单：24 项，按"独立原型验证/统一集成验证/构建工具验证"分类，编号 ①~㉔
- §10 设计阶段验证要求：按功能主题分组（`isIec559Of`、编译期 `if` 分支抑制、浮点 `mod`、`equalEpsilon`、`equalExact`、`@OverflowWrapping`+`const` 共存、`@Derive[Hashable]` 约束、Bool→Numeric 转换、Vec4 构造函数重载、生成脚本等）
- §12.3 编译器验证依赖→设计路径选择决策映射表：14 行，按"验证依赖→§2.1 检查项→验证通过路径→回退路径"组织

同一验证项（如 `@Derive[Hashable]` 约束检查）在三处各有位置：§2.1 项㉑、§7 D37 A13、§10 "`@Derive[Hashable]` 在泛型 struct 上的约束传递性验证"、§12.3 第 9 行。部分验证项存在交叉引用但多数没有。编码实施者在启动阶段需要同时翻阅三个章节才能确定完整的验证范围，极易遗漏。

**所在位置**：
- §2.1 编码前验证检查清单（第 225-258 行）
- §10 多处分节（第 1904-2068 行）
- §12.3 决策映射表（第 2698-2713 行）

**严重程度**：严重

**改进建议**：将 §2.1 设为验证清单的**唯一权威出处**，§10 和 §12.3 中的重复验证项改为仅引用 §2.1 项目编号（如"详见 §2.1 项㉑"），不再在 §10 和 §12.3 中重复验证方法、通过标准、回退方案等完整内容。或者干脆反向：将 §2.1 精简为清单索引表，每项仅一行+编号，详细内容归入 §10。当前三处独立维护的方式在后续轮次中必然出现不同步。

### 问题 3（一般）：P37（C++ API 映射完整性）声称"已处理"但验证范围不完整

**问题描述**：§1 P37 状态为"🔄 持续问题"，列为解决手段的是"新增附录 A `API 覆盖对照表`，将未覆盖 API 纳入 §8.5 排除清单"。但附录 A 的验证范围仅限 `type_vec1.hpp/.inl`~`type_vec4.hpp/.inl` 四个 Vec 类型文件的 API，未覆盖首轮范围内的其他核心文件：`setup.hpp`、`qualifier.hpp`、`compute_vector_relational.hpp`、`compute_vector_decl.hpp`、`_vectorize.hpp`、`fwd.hpp`（标量别名部分和向量别名部分）。§8.2 范围可追溯性对照表虽提供了文件级映射，但未达到附录 A 的逐函数/逐运算符粒度。

对于实施者而言，仅靠附录 A 只能确认 Vec 类型的 API 映射完整性，无法确认基础设施层（setup + qualifier）和辅助文件层的 API 在仓颉设计中是否完整覆盖或明确排除。例如 `qualifier.hpp` 中 `storage<L,T,Q>` 的 6+ 个 SIMD/非 SIMD 特化在设计中如何处置已在 D23 声明，但缺乏类似附录 A 的逐项对照。

**所在位置**：
- §1 P37 状态表（第 67 行）
- 附录 A 标题和验证范围说明（第 2757-2765 行）
- §8.2 范围可追溯性对照表（第 1717-1743 行）

**严重程度**：一般

**改进建议**：将附录 A 的范围从"type_vec1.hpp/.inl~type_vec4.hpp/.inl 交叉验证"扩展为首轮 11 个核心文件的全量 API 覆盖对照表，或至少为每个非 Vec 核心文件（setup.hpp、qualifier.hpp、compute_vector_relational.hpp、compute_vector_decl.hpp、_vectorize.hpp、fwd.hpp）各增加一个 API 覆盖子表。如果出于文档长度考虑不便扩展，应在 P37 状态中精确标注当前验证范围仅限 Vec 类型文件，降低实施者的预期。

### 问题 4（一般）：验收标准中缺少 Vec1 广播运算符的独立验收项

**问题描述**：Vec1 广播运算是从第 3 轮开始引入的重大设计新增（P24，v5 解决），涉及 §4.3 的 30 个算术广播运算符和 §4.4 的 30 个位广播运算符，共计 60 个运算符定义。但 §12.2 验收标准的全部 20+ 项中，没有一项专门针对 Vec1 广播运算符的正确性。§12.1 层次三的运算正确性验证虽覆盖"算术运算"和"位运算"，但未明确说明测试用例应涵盖 Vec1 广播模式（如 `Vec2 + Vec1`、`Vec1 + Vec2`、`Vec2 += Vec1`）。实施者在编码阶段验收时可能遗漏广播运算的正确性验证。

**所在位置**：§12.2 验收标准（第 2668-2694 行）

**严重程度**：一般

**改进建议**：在 §12.2 新增一项验收标准："Vec1 广播运算符正确性——验证 VecN op Vec1 方向（5 算术 × 3 VecN = 15 个运算符）和 Vec1 op VecN 方向（5 算术 × 3 VecN = 15 个运算符）的编译通过性和运算正确性；验证 VecN += Vec1 复合赋值自动生成，Vec1 += VecN 不自动生成（需两步模式）"。同步在 §12.1 层次三中明列广播运算的测试用例说明。

### 问题 5（一般）：`scalar-op-vec` 方向函数在 `import glm.*` 路径下的端到端可访问性未经验收

**问题描述**：§2 公共 API 面设计声明 `lib.cj` 通过 `public import glm.detail.{ add, sub, mul, div, mod }` 将这些函数暴露到 `package glm` 公共命名空间，下游消费者只需 `import glm.*` 即可使用。但 §12.2 验收标准中的"公共 API 导入"验收项仅验证 Vec 类型（`import glm.Vec2`）的可访问性，未验证 `add(s, v)`、`sub(s, v)` 等 scalar-op-vec 方向函数通过 `import glm.*` 的可访问性。实施者无法通过验收标准确认此导入路径的实际有效性——这是一个编译器行为依赖（§7 D37 A06 未覆盖此场景），验收标准未反映此风险。

**所在位置**：
- §12.2 验收标准"公共 API 导入"项（第 2681 行）
- §2 公共 API 面设计（第 120-138 行）
- §8.1 文件清单 `lib.cj` 行（第 1698 行）

**严重程度**：一般

**改进建议**：在 §12.2 验收标准中新增"scalar-op-vec 方向函数通过 `import glm.*` 可访问"验收项：从独立测试包中仅 `import glm.*`，调用 `add(1.0f, someVec)`、`sub(1.0f, someVec)` 等函数，验证编译通过且运行时正确。若此项验证失败（`public import` 重导出行 nilability 问题），应有回退方案——改为在 `glm` 包中定义包装函数或另存 `public type` 重导出策略（已在 §2.1 验证项⑱中覆盖）。

### 问题 6（轻微）：`fromBoolVec` 函数族的泛型参数推导可能引入调用方歧义

**问题描述**：§4.8 定义了 `fromBoolVec` 和 `fromBoolVecQ2` 包级独立函数，其中 `fromBoolVec<T, Q>(v: VecN<Bool, Q>)` 的 T 类型参数仅通过返回值类型推导（无显式使用于参数列表）。在仓颉中，泛型函数的类型参数若仅在返回类型中出现，调用方可能无法依赖类型推导自动确定 T——需要显式写出 `fromBoolVec<Int32, PackedHighp>(boolVec)`。此调用模式的纤长性（特别是跨 Q 版本需要写出三个类型参数 `fromBoolVecQ2<Int32, PackedLowp, PackedHighp>`）可能影响下游采用率，但设计文档未评估此 API 体验风险。

**所在位置**：§4.8 `fromBoolVec`/`fromBoolVecQ2` 签名清单和调用示例（第 1158-1207 行）

**严重程度**：轻微

**改进建议**：在 §4.8 末尾或 §11.7c 迁移模式中新增"类型参数显式指定"的调用体验说明，评估是否需要提供辅助函数（如 `fromBoolVecToDefault<T>` 自动绑定 `PackedHighp`），减少调用方需显式写出 Q 参数的场景。

## 需求响应充分度总结

产出对 roadmap 首轮范围的全部核心文件（11 个 GLM 源文件 + 3 个 shim 文件）均有关映射，新增的 3 个文件（`scalar_vec_ops.cj`、`type_fromBoolVec.cj`、`lib.cj`）已通过 SCO-001/002/003 范围变更日志记录，范围管理符合 D36 约定。产出以接口定义粒度覆盖了 Vec 类型体系的所有关键行为契约，从编码可落地角度看，实施者可根据 §4.1~§4.8 的签名清单和 §8.3 迁移顺序直接进入编码。

未充分响应的方面集中在：① 跨文件协调状态未闭环（P7 `GLM_CONFIG_CLIP_CONTROL`）；② 验证清单的冗余组织增加了实施遗漏风险；③ P37 API 映射完整性的验证范围不足以支撑其声明的覆盖度。

## 实现可落地性评价

**强项**：
- §8.3 迁移顺序提供了按阶段增量验证的详细步骤
- §2.1 编码前验证清单覆盖了所有编译器行为依赖的验证
- §3.8 的 Python 脚本模板可以直接用于别名生成
- §4.1 构造函数清单和 §4.3 运算符模板可直接作为编码参考

**弱项**：
- 实施者需要同时参照 §2.1、§10、§12.3 三个位置才能获得完整的验证-决策映射（问题 2）
- 部分关键行为（如 Vec1 广播运算符）在验收标准中无显式覆盖（问题 4）
- 公共 API 的端到端导入路径依赖未经验证的编译器行为且验收标准未覆盖（问题 5）
