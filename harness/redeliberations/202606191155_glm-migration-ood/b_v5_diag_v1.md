## 发现的问题

### 问题 1（严重）：C++ GLM API 映射完整性未经验证，历史问题被遗漏追踪

**问题描述**：迭代第 2 轮的历史审查曾提出"设计文档未审查与 C++ GLM 源码 API 的映射完整性——C++ GLM type_vecN.inl 的公开 API 是否在仓颉设计中完整覆盖未被评估"（见 iteration_history.md 第 66-69 行）。此项问题在问题状态追踪表（P1~P36）中**无对应编号**，也未出现在任何修订说明中，表明该问题被遗漏追踪，至今未闭环。产出 §4.1~§4.8 虽然提供了详细的 API 清单，但这些清单是设计师从 GLSL 规范和设计经验推导得出的，未经系统性交叉验证（如逐行对照 `type_vecN.inl` 确认每个公开运算符/构造函数/成员函数均已覆盖）。例如，C++ `type_vec2.inl:53-57` 定义的完全泛型双参数构造函数 `template<typename A, typename B> vec(A _x, B _y)`（接受任意类型对 A、B）在产出的 Vec2 构造函数清单中无对应版本——产出仅枚举了 `Vec1+T`、`T+Vec1`、`Vec1+Vec1` 等组合，缺少 `A+B` 任意类型对的泛型版本。

**所在位置**：整体产出（§4.1~§4.8 API 清单）、问题状态追踪表（§1 历史参考）
**严重程度**：严重
**改进建议**：
- 将此项补入问题状态追踪表，分配 P 编号
- 对 `type_vec1.inl`~`type_vec4.inl` 逐文件、逐函数对比，确认所有公开 API 在仓颉设计中的覆盖状态（覆盖/排除/降级）
- 将对比结果结构化记录（如补充附录"API 覆盖对照表"）
- 对已识别但未覆盖的 API（如缺失的泛型双参构造函数）明确处理策略——若因仓颉语言限制不可实现，纳入 §8.5 排除清单并记录理由

### 问题 2（严重）：`GLM_CONFIG_CLIP_CONTROL` 默认值与 roadmap 存在 0x0A vs 0x02 冲突，未正式解决

**问题描述**：Roadmap §3A.1（第 349 行）记录的默认值为 `0x02`（`GLM_CLIP_CONTROL_RH_NO`），而产出 §7 D6（第 1299 行）记录的默认值为 `0x0A`（十进制 10）。经参考实现 `setup.hpp:563-584` 验证，`RH_BIT = 1 << 3 = 8`，`NO_BIT = 1 << 1 = 2`，`0x0A = 8 | 2` 是正确的默认值。Roadmap 值 `0x02` 确实遗漏了 `RH_BIT = 8` 的组合位。此冲突自第 1 轮迭代即被发现（P1/P7），但一直以"待编码前验证项②确认"方式推迟，未将冲突解决前置到设计阶段——roadmap 是输入需求文档，输入本身存在错误意味着设计产出的需求响应基础存在缺陷，不能仅靠"编码前验证"来解决。

**所在位置**：§7 D6（~第 1299 行）
**严重程度**：严重
**改进建议**：
- 立即协调 roadmap 的维护者确认并修正 roadmap §3A.1 中的 `GLM_CONFIG_CLIP_CONTROL` 默认值
- 在 roadmap 修正完成后，在产出中将"推导结论"升级为"已验证结论（参考实现确认）"并在 D6 中移除"待编码前验证"措辞
- 将此冲突的解决状态从"待编码前验证"升级为"设计阶段已解决"，确保编码启动前此值已定

### 问题 3（一般）：`@OverflowWrapping` 标注策略与 roadmap §0.5.1 推荐的模式相反，但偏离声明不够醒目

**问题描述**：Roadmap §0.5.1 推荐在**复合赋值运算符**上标注 `@OverflowWrapping`，二元算术运算符委托给复合赋值实现。产出 §4.6/D3 采用相反策略——在**二元算术运算符**上直接标注，复合赋值由编译器自动生成并继承标注语义。偏离的技术原因（仓颉可重载运算符列表不包含复合赋值）已在 D3 中解释。但是，从需求响应的视角看，此偏离属于对 roadmap 架构指导的实质性偏离，应更加醒目地标注为"与 roadmap 的已知偏离"而非仅在 D3 中作为设计决策记录。下游消费者和审核者可能在未阅读 D3 的情况下，依据 roadmap 的标注策略编写代码，导致不一致。

**所在位置**：§4.6（~第 1109-1112 行）、§7 D3
**严重程度**：一般
**改进建议**：
- 在 §4.6 顶部新增醒目的"与 roadmap §0.5.1 的偏离声明"提示框（如 callout block），除说明技术原因外，还需说明对下游使用者的具体影响（二元运算符和复合赋值均可获得 wrapping 语义，行为等价）
- 在 §8.2 范围可追溯性对照表中新增偏离说明行，将此项偏离正式记录

---

### 问题 4（一般）：产出新增 3 个 roadmap 范围外文件，范围变更未按 D36 流程正式记录

**问题描述**：产出的文件清单在 roadmap §3G 定义的 11+3=14 个文件基础上新增了 3 个文件：`scalar_vec_ops.cj`、`type_fromBoolVec.cj`、`lib.cj`。这些新增在 §8.2 偏离说明列中有标注，但未按照 §7 D36 定义的通用范围变更管理流程填写变更日志条目（缺少变更编号、审批路径、影响评估等）。虽然所有新增文件均为"新增仓颉适配文件"而非从 C++ 源码新增迁移，但文件结构的扩展仍属于范围变更，应正式记录以保持范围可追溯性。

**所在位置**：§8.2 范围可追溯性对照表、§7 D36 范围变更日志
**严重程度**：一般
**改进建议**：
- 在范围变更日志中为三项新增文件各创建一个条目，填写类型（新增）、触发条件、影响评估和实施状态
- 明确这三个新增文件是否属于首轮交付范围的永久扩展，还是仅编码阶段的临时辅助文件

### 问题 5（一般）：`equalEpsilon`/`epsilonOf<T>()` 条件化设计依赖未在 §12 验收标准中建立明确的验证→决策映射

**问题描述**：`epsilonOf<T>()` 的路径 A/路径 B 选择（§3.5）以及 `equalEpsilon` 的实现策略取决于编译期 `if` 分支抑制验证结果（§10 依赖点 A~E）。此条件化设计虽然在 §3.5 中详细描述，但 §12.2 验收标准中对应的验收项（`equalEpsilon-epsilonOf` 回退联动验收项）仅验证"行为正确性"，未建立明确的"验证结果→设计路径选择"的决策映射表。编码阶段实施者面对 §10 的验证结果时，需要自行查阅 §3.5 判断应走哪条路径，缺乏集中、结构化的决策矩阵。

**所在位置**：§3.5 双路径策略、§12.2 验收标准
**严重程度**：一般
**改进建议**：
- 在 §12.2 或新增 §12.3 中补充"编译器验证依赖→设计路径选择"决策映射表，列明每个验证项（依赖点 A~E、D29、D30 等）的通过/失败结果对应哪个设计路径选择
- 将此决策表与 §2.1 编码前验证检查清单形成交叉引用，确保验证项与决策路径一一对应

**所在位置**：文件头部（第 1-6 行）
**严重程度**：轻微

## 整体质量评价

产出在技术可行性和设计完备性方面已通过多轮内部审议，核心抽象（Qualifier 接口体系、Vec 结构体系、ComputeEqual 策略）定义清晰，关键行为契约（构造、运算、比较、位运算）描述详尽，编译器行为假设已集中跟踪（§7 D37），验证策略（编码前验证清单、设计阶段验证要求、层次化验收标准）完整。

待改进的主要方面集中在**需求响应充分性**和**整体完整性**的某些维度：产出与 roadmap 之间存在两处实质性不一致（CLIP_CONTROL 默认值冲突、@OverflowWrapping 标注策略偏离），且范围变更管理流程未严格执行。从实际落地视角看，设计文件体量巨大（2738 行），部分设计分支的条件化（路径 A/B）使编码启动前有多个未决决策，这将增加实施者的认知负担和决策成本。

## 综合建议优先序

1. **编码前必须解决**：问题 2（CLIP_CONTROL 冲突）— 协调 roadmap 维护者确认值后修复 roadmap
2. **编码前必须解决**：问题 1（API 完整性验证）— 补充 C++ 源文件 API 对比检查
3. **编码前建议解决**：问题 3（偏离声明）— 增强 @OverflowWrapping 标注策略偏离的醒目度
4. **编码前建议解决**：问题 5（条件化决策映射）— 补充验证→决策映射表
5. **编码阶段解决**：问题 4（范围变更记录）
6. **修复后验证**：问题 6（版本号统一）
