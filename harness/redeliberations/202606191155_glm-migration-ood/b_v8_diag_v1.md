## 二、发现的问题

### 问题 1（严重）：§11.12 迁移成本合计表存在数学矛盾

**问题描述**：§11.12 合计表的 `[^1]` 注脚明确声明"§11.5 与 §11.9 非累加关系，实际以 §11.9 为准。两项评估不可简单求和"，但合计表在求和时同时计入了这两行的成本值。具体而言：

| 场景 | §11.5 计值 | §11.9 计值 | 声明应计 | 当前合计错误 |
|------|-----------|-----------|---------|------------|
| 最可能 | 0.5 人天 | 1.0 人天 | 1.0（仅 §11.9） | 两者都计，合计偏高 0.5 人天 |
| 悲观 | 1.0 人天 | 2.0 人天 | 2.0（仅 §11.9） | 两者都计，合计偏高 1.0 人天 |

**所在位置**：§11.12 第 2488 行（§11.5 行）与第 2495 行（§11.9 行），以及底部合计值。
**严重程度**：严重 — 影响项目管理决策的准确性，实施者或主 Agent 依据此表做资源规划时会得到偏高 0.5~1.0 人天的预估。
**改进建议**：从合计表中移除 §11.5 行的成本值（保留行本身但标记为"0—已纳入 §11.9"），或将 §11.5 行标注为"已纳入 §11.9，不重复计入"，同步修正底部合计值：
  - 最可能：12.65 → 12.15
  - 悲观：23.5 → 22.5

### 问题 2（一般）：`fromBoolVec`/`fromBoolVecQ2` 的 Vec1 构造函数类型参数依赖编译器推断行为

**问题描述**：§4.8 的 `fromBoolVecQ2` 实现代码中（以 Vec1 为例）：
```
public func fromBoolVecQ2<T, Q, Q2>(v: Vec1<Bool, Q2>): Vec1<T, Q> where Q <: Qualifier, Q2 <: Qualifier {
    Vec1(if (v.x) { T(1) } else { T(0) })
}
```
函数体内 `Vec1(...)` 调用未显式标注类型参数 `<T, Q>`，依赖仓颉编译器从函数返回值类型 `Vec1<T, Q>` 推断 `Vec1` 构造函数的泛型实参。此推断行为在当前设计文档中**未被验证覆盖**——§2.1 编码前验证清单无对应项。若编译器不支持从返回值类型推断 struct 构造函数的泛型实参，上述代码将编译失败（`Vec1` 的 `T`/`Q` 无法推断）。

**所在位置**：§4.8 第 1158~1207 行，所有 `fromBoolVec`/`fromBoolVecQ2` 函数的实现代码。
**严重程度**：一般 — 回退方案简单（只需在 Vec1 构造函数调用处显式标注 `<T, Q>`），但若编码阶段才发现，可能打乱编码节奏。
**改进建议**：在 §4.8 中标注此项编译器推断依赖（如脚注"仓颉编译器是否支持从返回值类型推断 struct 泛型参数，编码阶段须验证"），或在 §2.1 编码前验证检查清单中补充一项原型验证。

### 问题 3（一般）：`equalEpsilon` 的 Path A/Path B 决策依赖链仍未全量可视化

**问题描述**：§3.5 定义的 `equalEpsilon`/`epsilonOf<T>()` 双路径（路径 A：保留 `epsilonOf<T>(` → `T(1e-6)`；路径 B：完全移除，内联固定容差）是跨章节影响的关键决策点，涉及 §3.5、§4.5、§10（依赖④）、§12.2（回退联动验收项）、§12.3（决策映射表第 11 行）共 5 处分布。虽然 §12.3 第 11 行新增了"修改触发联动检查清单"列，清晰列出了路径 B 的 6 处联动修改，但**路径 A 触发的联动修改清单缺失**——仅描述了路径 A 修改一处 `epsilonOf<T>()`，未说明 `epsilonOf<T>()` 的 `Number<T>` 约束移除后，`ComputeEqual.callConst` 中的 `if (isIec559Of<T>())` 分支内 `abs(a - b) <= epsilonOf<T>()` 是否仍需要处理整数 `T` 的兼容性问题（路径 A 中 `epsilonOf<T>()` 返回 `T(1e-6)`，整数 `T` 的 `T(1e-6)` 不可用，依赖分支抑制保护）。

**所在位置**：§3.5 第 456~458 行（路径 A 描述）、§12.3 第 2704 行（决策映射表第 11 行）。
**严重程度**：一般 — 路径 A 的风险已在 §3.5 正文中分析（整数 `T(1e-6)` 不可用依赖分支抑制保护），但决策映射表的联动清单中缺少路径 A 的对应说明，实施者按表操作时可能遗漏路径 A 的分支抑制依赖检查。
**改进建议**：在 §12.3 第 11 行的联动清单列中补充路径 A 的触发条件和检查项："路径 A 触发时：修改 `shim_limits.cj` 中 `epsilonOf<T>()` 定义（移除 `Number<T>` 约束，改为 `T(1e-6)`）→ 验证编译期 `if` 分支抑制保护有效（§2.1 项④⑫）→ 确认整数 `T` 不进入 `epsilonOf<T>()` 调用路径 → 若抑制保护无效，回退至路径 B"。

### 问题 4（轻微）：`fromBoolVec` 对 `T = Bool` 的语义描述措辞可能误导

**问题描述**：§4.8 第 1198 行："当 `T = Bool` 时，`fromBoolVec` 生成 `VecN<Bool, Q>`，语义为'Bool 值保持不变'（`true→T(1)=Bool(1)=true`，`false→T(0)=Bool(0)=false`），即此函数等价于恒等映射。"此表述的隐含推论是 `fromBoolVec` 对 `Bool` 输入返回相同值的 `VecN<Bool, Q>`——但在跨 Q 场景中（`fromBoolVecQ2`），若输入 `Bool` Vec 的 Q2 与输出 Q 不同，返回值并**不等价于**输入（Qualifier 类型改变），"恒等映射"的陈述不准确。

**所在位置**：§4.8 第 1198 行。
**严重程度**：轻微 — 语义上正确（值不变），但"恒等映射"一词在跨 Q 版本下不严谨。
**改进建议**：将"即此函数等价于恒等映射"修改为"即此函数的值语义等价于恒等映射（分量值保持不变，但 Qualifier 类型可能变化）"。

## 三、深度和完整性评估

### 总体评价：深度充分，完整性良好

产出文档共 2909 行，覆盖了从架构决策（§7）到文件清单（§8.1）、从迁移顺序（§8.3）到验收策略（§12）的完整设计轮次。具体而言：

- **编码指导性**：§4 给出了完整的构造函数签名清单（含 Vec1~Vec4 各 30+ 个签名）、运算符实现模板（Vec2 算术运算符完整语法模板）、`extend` 块语法示例（§4.4）。编码实施者可直接照此编码。
- **接口完备性**：§2 公共 API 面设计明确了 `lib.cj` 的 `public import` 清单，§8.1 文件依赖表标注了每个文件的依赖关系和 import 路径，§3.8 脚本模板可直接生成 `fwd.cj`。
- **异常/边界覆盖**：§5 错误处理策略覆盖了下标越界、整数溢出、除零、位运算溢出等场景；§9 补充了零向量、NaN/Infinity 传播、类型转换边界行为；§9.4 给出了完整的类型转换边界表格。

### 未覆盖的潜在风险区域（非区块问题，仅作为完整性说明）

1. **编译器版本升/降级的影响未评估**：§2 要求 `cjc-version = "1.0.5"`，但未说明若目标环境提供更高版本编译器时是否需要调整设计（如 1.1.0 是否向后兼容所有依赖行为）。
2. **多文件编译顺序的物理依赖未完全显式化**：§8.3 迁移顺序表列出了编译阶段，但未明确 `cjpm.toml` 中是否需额外配置编译顺序（仓颉同包内通常按文件名称顺序或依赖分析自动处理，不需显式指定）。

以上两点均不影响当前设计落地，仅作为辅助信息记录。

## 四、整体结论

产出质量良好，通过 7 轮（内部审议+再审议）迭代已解决绝大多数结构性问题和事实性错误。本报告识别出 1 项严重问题（成本合计表数学矛盾）、2 项一般问题（`fromBoolVec` 类型推断依赖未验证、路径 A 联动清单缺失）、1 项轻微问题（`fromBoolVec` Bool 语义措辞）。建议在下一版本中优先修复问题 1（影响资源决策），其余问题按优先级依次处理。
