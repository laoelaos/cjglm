### 问题 2（严重）：附录 A 及 §8.5 中完全泛型多元构造排除理由存在事实错误（上一轮未修复）

**描述**：附录 A 未覆盖 API 汇总表（第 2805 行）及 §8.5 排除清单（第 1896 行）将完全泛型多元构造函数 `template<A,B,...> vecN(A, B, ...)` 的排除理由写为"仓颉构造函数不能为泛型（运算符重载限制）"。此陈述与仓颉语言事实不符——本文档 §4.1 中已大量使用 `public init<T2, Q2>(v: VecN<T2, Q2>) where Q2 <: Qualifier` 形式的泛型构造函数，证明仓颉构造函数完全可以拥有泛型类型参数。实际原因是：各分量独立推导的多元泛型参数（A/B/C/D 各可不同）在仓颉构造函数签名层面无法避免与已有同参数个数构造函数的歧义，且各分量需统一为目标类型 T。此问题在上一轮审查中已被指出（b_v12_diag_v1.md 问题 2），v12 修订未修复。

**所在位置**：
- 附录 A Vec2 对照表（第 2805 行）
- §8.5 排除清单（第 1896 行）

**严重程度**：严重

**改进建议**：将排除理由修正为准确的表述。第 2805 行和第 1896 行均需修正。建议改为："各分量类型可独立推导的完全泛型多元构造函数（如 C++ `template<A,B> vec2(A,B)` 允许 A≠B）在仓颉中无法直接映射——仓颉构造函数虽可声明泛型参数（`init<A, B>(a: A, b: B)`），但此泛型构造与 `const init(x: T, y: T)` 在调用方传入同类型参数时产生重载歧义，且各分量需通过 `T(v)` 转换为统一的目标类型 `T`，此转换对 A≠B 场景增加了不必要的复杂度。C++ 该模式已由已有构造函数形态覆盖（标量填充构造 + 逐分量 const init + 跨 Vec 转换构造），无需单独映射。"

### 问题 4（一般）：§2.1 验证时序建议与层级说明存在逻辑矛盾（上一轮未修复）

**描述**：§2.1 编码前验证检查清单包含两段指引：
- **验证时序建议**（第 271 行）：建议优先执行第⑫项（统一集成验证），再按编号顺序验证①→②→③→...→㉕。
- **层级说明**（第 273 行）：声明第⑫项"**取代**①③④⑤⑧项的独立验证结果——若独立验证已通过但集成验证失败，以集成验证为准"。

若遵循时序建议先执行⑫验证：
- 若⑫通过 → 直接跳过①③④⑤⑧，无需逐项验证——这与时序中的"再按编号顺序验证①→②→③→..."矛盾。
- 若⑫失败 → 仍需通过①③④⑤⑧逐项识别哪些假设不成立——但层级说明声称⑫"取代"了这些项的结果。

逻辑歧义核心：时序建议暗示⑫前置（替代后续逐项验证），层级说明暗示⑫后置（覆盖已有独立验证结果）。实施者无法确定应该先执行⑫还是先执行①③④⑤⑧。此问题在上一轮审查中被指出（b_v12_diag_v1.md 问题 4），v12 修订未修复。

**所在位置**：§2.1 验证时序建议（第 271 行）与层级说明（第 273 行）

**严重程度**：一般

**改进建议**：统一建议为两种路径之一：
- **路径 A（推荐）**：先执行①③④⑤⑧独立验证，再执行⑫集成验证。若⑫失败则按集成验证结果为准（取代独立验证结论）。此路径降低单次验证的复杂度，且独立验证失败时无需继续执行⑫。
- **路径 B**：先执行⑫集成验证。若⑫通过则整体跳过①③④⑤⑧；若⑫失败再回退执行①③④⑤⑧逐项隔离。此路径验证效率高。

在时序建议中明确说明选择其中一种路径（推荐 A），同步修改层级说明中的"取代"语义描述以匹配所选路径。
