## 问题 1（严重）：Vec 与 Vec1 之间的广播运算模式完全遗漏

**问题描述**：C++ GLM 的 `type_vecN.inl` 中为 Vec2~Vec4 均提供了 `vecN op vec1` 和 `vec1 op vecN` 两种广播运算符重载（涵盖 `+`、`-`、`*`、`/`、`%`、`&`、`|`、`^`、`<<`、`>>` 共 10 种运算符），以及 `vecN op+= vec1` 形式的复合赋值（Vec2.inl 行 165/192/219/246/309/336/363/390/417/444 共 10 种）。经源码确认，type_vec2.inl 中涉及 vec1 的运算符匹配约 30 处。当前 OOD 设计的 §4.3 算术运算符仅列出 `VecN op VecN` 和 `VecN op T` 两种右操作数类型，§4.4 位运算符同样仅有 `VecN op VecN` 和 `VecN op T`，完全遗漏了 `VecN op Vec1` 广播模式。§8.5 排除清单中未提及此模式，§11 迁移成本也未评估。

此遗漏对后续矩阵类型迁移有直接连锁影响：C++ GLM 的 `type_mat2x2.inl` 等矩阵实现广泛使用 `vec + vec1`、`vec * vec1` 形式进行分量级操作，矩阵运算中 Vec1 广播是核心运算模式之一。若仓颉 Vec 缺少 Vec1 广播运算符，矩阵类型迁移时需将所有 `vec op vec1` 改写为 `vec op vec1.x` 形式，迁移成本显著上升。

**所在位置**：§4.3 算术运算（~第 876-948 行）、§4.4 位运算（~第 950-989 行）、§4.6 标注策略、§8.5 排除清单、§11 迁移成本

**严重程度**：严重

**改进建议**：
1. 在 §4.3 算术运算节新增 `VecN op Vec1` 广播运算模式设计——可在 Vec 结构体的 `extend` 块中定义 `operator func +(rhs: Vec1<T, Q>): VecN<T, Q>`，将 Vec1 的 x 分量广播到所有分量参与运算。注意 Vec1 与 VecN 的 T 和 Q 相同，因此运算符参数类型为具体的 `Vec1<T, Q>` 而非泛型 `Vec1<T2, Q2>`，不违反"运算符重载不能为泛型"的约束。
2. 位运算符（`&`/`|`/`^`/`<<`/`>>`）同样需补充 `VecN op Vec1` 广播模式。
3. 在 §4.6 `@OverflowWrapping` 标注策略中补充广播运算符的标注方案（广播运算符的标注策略与现有二元运算符一致——算术广播运算符标注 `@OverflowWrapping`，`<<` 广播标注，位运算不标注）。
4. 复合赋值 `VecN += Vec1` 若广播运算符返回类型与左操作数类型匹配，将由编译器自动生成，需在 §4.3/§4.6 中显式声明此能力。
5. 在 §8.5 排除清单中若明确不提供此模式，需显式记录排除理由及对矩阵迁移的影响；若决定提供，则无需列入排除清单。
6. 在 §11 迁移成本中补充此模式的迁移成本评估（含矩阵轮次的连锁影响）。

## 问题 2（严重）：`abs()` 调用形式在代码示例中与仓颉标准库 API 不匹配

**问题描述**：§4.5 `equalEpsilon` 代码示例（~第 1019 行）和 §3.5 `ComputeEqual.callConst` 代码中使用了 `abs(a - b)` 形式的调用。经查阅仓颉标准库文档（`std.math`），仓颉中 `abs` 是 `std.math` 包中的包级函数，签名为 `abs(x: T): T`，需通过 `import std.math.abs` 导入。代码示例中使用了类 C 的裸函数调用形式 `abs(expr)`，此形式在仓颉中如果导入正确实际可用（`import std.math.*`），但设计文档未在任何文件依赖说明中标注 `shim_limits.cj`、`type_vecN.cj` 等文件对 `std.math.abs` 的 import 依赖。

更关键的是，`ComputeEqual.callConst` 中使用 `abs(a - b)` 的上下文为 `const` 函数——需要确认 `std.math.abs` 是否为 `const` 函数。设计文档 §7 D29 的 A03 验证项标注 `Float32.abs()`/`Float64.abs()` 为静态方法须为 `const` 函数，但实际仓颉 `abs` 是 `std.math` 的包级函数而非类型的静态方法。此描述与仓颉实际 API 形式不符。

**所在位置**：§3.5 `ComputeEqual.callConst` 代码（~第 407-419 行）、§4.5 `equalEpsilon` 代码示例（~第 1019 行）、§7 D29 A03 验证项（~第 1551 行）、§8.1 文件依赖说明

**严重程度**：严重

**改进建议**：
1. 将代码示例中 `abs(a - b)` 的调用形式修正为 `import std.math.abs` 后的包级函数调用形式（或确认仓颉环境下 `abs` 需要显式 import 的路径）。
2. 修正 §7 D29 A03 中"Float32.abs()/Float64.abs() 静态方法"的描述，改为"std.math.abs 包级函数"，与仓颉实际 API 一致。
3. 在 §8.1 文件清单中为 `compute_vector_relational.cj`、`type_vecN.cj`（含 equalEpsilon）等使用 `abs` 的文件标注 `import std.math.{ abs }` 依赖。
4. 在 §2.1 编码前验证清单中补充验证项：确认 `std.math.abs` 在 `const` 函数体内可调用（当前 A03 验证项的描述已覆盖此验证目的，但描述不准确需修正）。

## 问题 3（严重）：C++ GLM 复合赋值运算符接受跨类型 `U` 参数（`vec<1, U, Q>` 和 `vec<N, U, Q>`）的可用性损失未被评估

**问题描述**：C++ GLM 中 `operator+=(vec<1, U, Q> const& v)` 和 `operator+=(vec<N, U, Q> const& v)` 允许跨类型复合赋值——`U` 是独立于 struct 的类型参数（不同于当前 Vec 的 `T`），如 `vec<2, float> += vec<2, int>` 在 C++ 中合法。仓颉编译器自动生成的复合赋值仅接受同类型右操作数——即 `VecN<Int32> += VecN<Float32>` 在仓颉中不可用。

当前设计已分析了跨类型二元运算的不可用性（D15），但跨类型复合赋值的损失未在任何章节被正式评估。对于 `VecN op= VecN<T2, Q>` 路径，仓颉中只有 "先跨类型构造再同类型 op=" 两步模式可用，增加了调用方的迁移负担。同理，`VecN op= Vec1<T, Q>`（Vec1 广播复合赋值）在 C++ 中存在（`type_vec2.inl:165` 行确认为 `operator+=(vec<1, U, Q> const& v)`），但在仓颉中若问题 1 的广播运算符未提供，相应复合赋值也不可用。

此问题与问题 1 有关联但侧重不同——问题 1 侧重广播运算符模式遗漏，本问题侧重跨类型复合赋值的可用性损失评估（含 broadcast 和同维度两种路径）。

**所在位置**：§4.3 复合赋值说明（~第 879 行）、§4.6 复合赋值段落、§8.5 排除清单、§11 迁移成本

**严重程度**：严重

**改进建议**：
1. 在 §4.3 复合赋值说明段落新增"跨类型复合赋值不可用"的显式声明，区分两种子场景：(a) `VecN<T,Q> op= VecN<T2,Q>`（同维度跨类型）——仓颉编译器自动生成的 op= 不接受跨类型右操作数；(b) `VecN<T,Q> op= Vec1<T,Q>`（Vec1 广播复合赋值）——仅在问题 1 的广播运算符提供后由编译器自动生成。
2. 在 §8.5 排除清单中新增条目"跨类型复合赋值（`VecN<T,Q> op= VecN<T2,Q>` 和 `VecN<T,Q> op= Vec1<U,Q>`）不可用"，记录排除理由和替代方案（先构造后赋值的两步模式）。
3. 在 §11 迁移成本中补充跨类型复合赋值的迁移模式评估（与 §11.3 跨类型赋值差异类似但涉及复合赋值语义）。

## 问题 4（一般）：`Vec` 家族别名与 `glm.detail.VecN` 泛型结构体同名的自引用风险未解决

**问题描述**：§3.8 的 Python 脚本模板中，`Vec` 家族（`VEC_FAMILIES` 键 `'Vec'`）因 `fam.endswith('Vec')` 为 `true` 而得到 `suffix = ''`，生成的别名形式为 `public type Vec2 = Vec2<Float32, PackedHighp>`。左侧别名 `Vec2` 与右侧引用的 `glm.detail.Vec2` 泛型结构体同名。在 `fwd.cj` 的 `package glm` 上下文中，右侧的 `Vec2` 需通过 `import glm.detail.{ Vec2 }` 引入——同一包中 `public type Vec2 = Vec2<Float32, PackedHighp>` 是否合法取决于仓颉 `type` 别名机制对同名 alias 的处理。

设计文档在第一轮诊断（v1 诊断问题 4）中已识别此风险，但 v4 版本未采取任何行动：既未在 §2.1 编码前验证清单中补充同名别名的编译器行为验证项，也未在 §3.8 中讨论风险或排除方案。此问题从第 1 轮迭代遗留至第 3 轮仍未闭环。

**所在位置**：§3.8 向量类型别名脚本模板（~第 636-665 行）、§3.8 别名格式示例表（~第 528-538 行）、§2.1 编码前验证清单

**严重程度**：一般

**改进建议**：
1. 在 §2.1 编码前验证清单中补充验证项：在 `package glm` 中声明 `import glm.detail.{ Vec2 }` 后定义 `public type Vec2 = Vec2<Float32, PackedHighp>`，验证编译是否通过。若通过则无问题；若产生自引用歧义编译错误，则需修正 `Vec` 家族的别名策略。
2. 若验证结果为自引用不合法，修正脚本中 `Vec` 家族的处理逻辑——最简修复为排除 `Vec` 家族的别名生成（因 `Vec2<Float32, PackedHighp>` 可直接通过结构体名+类型参数使用），或为 `Vec` 家族使用不同命名规则。
3. 在 §3.8 中增加脚注显式说明此同名风险及验证依赖。

## 问题 5（一般）：Vec1 的运算符设计完整性未做独立分析——Vec1 与标量 `T` 的运算语义重叠未被评估

**问题描述**：Roadmap §3E 首轮迁移文件清单中 `type_vec1.hpp` + `type_vec1.inl` 作为子范围 4 被明确纳入。C++ GLM 中 Vec1 的运算符模式与 Vec2~Vec4 有重要差异：Vec1 本质上可视为标量的向量包装，其 `operator+(vec<1, T, Q>, T scalar)` 和 `operator+(T scalar, vec<1, T, Q>)` 在语义上与 `T + T` 重叠。经源码确认，`type_vec1.inl` 中每个运算符均提供了三组重载：Vec1 op T、T op Vec1、Vec1 op Vec1。

在当前设计中，Vec1 的二元运算符在 struct 内部定义为 `Vec1 op Vec1` 和 `Vec1 op T`，`T op Vec1` 由 `scalar_vec_ops.cj` 中的包级函数 `add(s, v)` 等覆盖。但设计文档未独立分析 Vec1 运算符的语义特殊性——例如 `Vec1<Int32, Q>(5) + Int32(3)` 与 `5 + 3` 在数学结果上等价，Vec1 op Vec1 与标量运算的完全重叠是否导致调用方混淆？Vec1 的标量填充构造 `Vec1(scalar: T)` 在 v4 中被 `const init(x: T)` 兼任（因功能等价），但 `const init(x: T)` 与 `Vec1 + T`（运算符 `op T`）中的 `T` 参数在调用方的区分度如何？这些 Vec1 独有的设计问题未被讨论。

**所在位置**：§4.3 算术运算（Vec1 运算符缺少独立分析）、§3.2 Vec 结构体系

**严重程度**：一般

**改进建议**：在 §4.3 中新增 Vec1 运算符设计专节（或子段），独立分析：(1) Vec1 op Vec1 与标量运算的语义重叠关系及调用方指引；(2) `const init(x: T)` 兼任标量填充构造后，`Vec1(5) + 3` 和 `5 + 3` 的调用方选择指引；(3) Vec1 在矩阵运算中的角色（作为广播因子）及其运算符覆盖的完备性。

## 问题 6（一般）：Bool→Numeric 隐式转换构造被降级为显式工厂函数但未在排除清单和迁移成本中完整覆盖

**问题描述**：§4.1 T2 跨类型约束矩阵中 `Bool → Numeric` 路径标注为"编译错误，改用 `fromBoolVec` 工厂函数"。但 C++ GLM 中 `vec<2, float>(bvec2(true, false))` 的跨类型构造是隐式转换（通过构造函数直接调用），当前设计将此模式从隐式构造降级为显式工厂函数调用（`fromBoolVec<Float32, PackedHighp>(bvec)`），增加了调用方的迁移负担。

此降级虽属合理的语言限制应对（仓颉不支持 `Bool(Int32)` 构造），但 §8.5 排除清单中未记录此降级——排除清单仅记录了"不迁移"的设施，未记录"迁移但语义降级"的设施。§11 迁移成本中也未评估此降级对调用方代码的迁移模式影响（类似 §11.3 跨类型赋值差异的模式，但针对 Bool→Numeric 构造路径）。

**所在位置**：§4.1 T2 跨类型约束矩阵 `Bool→Numeric` 单元格、§8.5 排除清单、§11 迁移成本

**严重程度**：一般

**改进建议**：
1. 在 §8.5 排除清单中新增条目"Bool→Numeric 隐式转换构造被降级为显式工厂函数"，记录排除理由（仓颉不支持 `Bool→Numeric` 隐式构造）和替代方案（`fromBoolVec`/`fromBoolVecQ2`）。
2. 在 §11 迁移成本中补充 Bool→Numeric 构造降级的代码迁移模式评估：C++ 代码 `vec2 v(bvec2(true, false))` → 仓颉代码 `var v = fromBoolVec<Float32, PackedHighp>(BVec2(true, false))`，评估搜索替换可行性和工作量。

## 问题 8（轻微）：Vec1 的 `operator+=`/`operator-=` 等复合赋值运算符接受不同类型 `U` 的标量参数——此跨类型复合赋值场景在 Vec1 上的特殊性未被覆盖

**问题描述**：C++ GLM `type_vec1.inl:99` 的 `operator+=(U scalar)` 中 `U` 是独立类型参数（不同于 struct 的 `T`），允许 `vec1<int> += 0.5f` 形式——即接受不同数值类型的标量进行复合赋值。仓颉编译器自动生成的 `+=` 仅接受同类型右操作数，`Vec1<Int32> += Float32(0.5)` 不可用。此差异在 Vec1 上尤为显著——Vec1 因仅有单一分量，`Vec1<T> op= U` 语义上等价于 `T = T + U(T_value)`，在 C++ 中广泛使用。此问题是问题 3（跨类型复合赋值）的子集，但 Vec1 场景出现频率更高（Vec1 常作为标量的向量包装使用）。

**所在位置**：§4.3 复合赋值说明、§11 迁移成本

**严重程度**：轻微

**改进建议**：在问题 3 的改进方案中列为子场景一并评估，或在 §11 迁移成本中补充 Vec1 跨类型标量复合赋值的迁移模式。

## 整体质量评价

本方案（v4）在架构层面（Qualifier 体系、Vec 结构体系、包组织、别名体系、编译器行为假设全景图、排除清单、跨 Q 等价假设影响范围表等）表现出较高的完整性和深度。内部审议已覆盖 const init 共存性、% 运算符条件性移动策略、路径 A/B 兼容性等关键技术风险维度，历史迭代问题修复状态经抽查确认与声明一致。

主要遗留风险集中在 **C++ GLM API 覆盖完整性** 维度——Vec 与 Vec1 之间的广播运算模式（`vec2 + vec1`、`vec1 + vec2` 等）是完全遗漏的严重问题，对下游矩阵类型迁移的等价性构成直接障碍。跨类型复合赋值的可用性损失和 `abs()` 调用形式与实际仓颉 API 的不匹配也是需要补正的显著问题。其余问题为中等或轻微级别，可在编码阶段通过补充验证和设计微调解决。
