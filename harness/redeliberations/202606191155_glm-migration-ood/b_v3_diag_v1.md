## 问题 1（严重）：Vec 与 Vec1 之间的逐分量广播运算模式完全遗漏

**问题描述**：C++ GLM 的 `type_vecN.inl` 中为每个 Vec 类型（Vec2~Vec4）均提供了 `vecN op vec1` 和 `vec1 op vecN` 两种广播运算符重载（含 `+`、`-`、`*`、`/`、`%`、`&`、`|`、`^`、`<<`、`>>`）。例如 `type_vec2.inl:487` 的 `operator+(vec<2, T, Q> const& v1, vec<1, T, Q> const& v2)` 将 Vec1 的单一分量广播到 Vec2 的所有分量参与运算。这是 GLM 的核心运算模式之一，下游代码（特别是矩阵运算 `mat * vec` 等）广泛使用。当前 OOD 设计中二元算术运算符仅支持 `VecN op VecN` 和 `VecN op T`，完全遗漏了 `VecN op Vec1` 和 `Vec1 op VecN` 两种广播运算模式，也未在排除清单（§8.5）或迁移成本评估（§11）中提及。

**所在位置**：§4.3 算术运算、§4.4 位运算、§4.6 标注策略、§8.5 排除清单、§11 迁移成本

**严重程度**：严重

**改进建议**：
1. 在 §4.3 算术运算节新增 `VecN op Vec1` 和 `Vec1 op VecN` 广播运算模式设计——可通过在 Vec 结构体的 `extend` 块中定义 `operator func +(rhs: Vec1<T, Q>): VecN<T, Q>` 实现（Vec1 的单一分量 x 广播到所有分量），或在 `scalar_vec_ops.cj` 中定义包级函数 `add(v1: Vec1<T, Q>, v2: VecN<T, Q>): VecN<T, Q>` 等。
2. 位运算符（`&`/`|`/`^`/`<<`/`>>`）同样需要补充 `VecN op Vec1` 广播模式。
3. 在 §4.6 `@OverflowWrapping` 标注策略中补充广播运算符的标注方案。
4. 在 §8.5 排除清单中若明确不提供此模式，需显式记录排除理由；若决定提供，则无需列入排除清单。
5. 在 §11 迁移成本中补充此模式的迁移成本评估。

## 问题 2（严重）：Vec 结构体内部定义的二元算术运算符签名与 C++ GLM 源码存在显著缺漏——缺少 `VecN op Vec1` 同型运算符

**问题描述**：§4.3 提供的"Vec2 算术运算符完整语法模板"中，二元运算符仅列出 `VecN op VecN` 和 `VecN op T`（标量）两种右操作数类型。但 C++ GLM `type_vec2.inl:487-492` 中的 `operator+(vec<2,T,Q>, vec<1,T,Q>)` 是同类型参数 T 下的 Vec1 广播运算——Vec1 的单一分量值 x 被广播到 Vec2 的所有分量与 Vec2 对应分量相加。此运算符在 C++ 中声明在 struct 外部（非成员运算符），在仓颉中可作为 Vec 结构体的 operator func 接收 `Vec1<T, Q>` 右操作数。当前设计完全未覆盖。这一问题与问题 1 有关联但侧重不同——问题 1 侧重广播模式的整体遗漏，本问题侧重设计模板自身的签名缺漏，使编码实施者无法从语法模板直接获得完整运算符列表。

**所在位置**：§4.3 Vec2 算术运算符完整语法模板（~第 896-943 行）

**严重程度**：严重

**改进建议**：在 §4.3 语法模板中为每个二元算术运算符补充 `operator func +(rhs: Vec1<T, Q>): VecN<T, Q>` 形式的运算符定义，语义为将 Vec1 的 x 分量广播到所有分量参与运算。同步更新 §4.6 标注策略中的标注计数。若因仓颉运算符重载规则不支持（运算符不能为泛型，但 Vec1<T, Q> 中 T 与当前 Vec 的 T 相同、Q 与当前 Vec 的 Q 相同，可视为非泛型同类型运算符），需在排除清单中记录并补充替代方案。

## 问题 3（严重）：C++ GLM 复合赋值运算符接受 `Vec1` 广播参数的模式完全遗漏

**问题描述**：C++ GLM `type_vec2.inl:165` 的 `operator+=(vec<1, U, Q> const& v)` 允许 `vec2 += vec1` 形式的复合赋值（Vec1 广播到所有分量后执行分量级操作）。同样 `operator-=(vec<1, U, Q>)`、`operator*=(vec<1, U, Q>)`、`operator/=(vec<1, U, Q>)`、`operator%=(vec<1, U, Q>)` 均存在。当前设计仅声称"复合赋值由编译器自动生成"，但编译器自动生成的 `+=` 仅在二元 `+` 的右操作数类型与左操作数返回类型匹配时才生成——若设计新增 `operator func +(rhs: Vec1<T, Q>): VecN<T, Q>`，其返回类型为 `VecN<T, Q>`（与 `+=` 左操作数类型匹配），则编译器可自动生成 `+=Vec1` 版本。但若不新增此二元运算符，`v += vec1` 场景不可用，且该功能缺失未被设计文档提及。

**所在位置**：§4.3 算术运算、§4.6 复合赋值说明

**严重程度**：严重

**改进建议**：若问题 1/2 的改进方案决定提供 `VecN op Vec1` 运算符，则复合赋值 `VecN += Vec1` 将由编译器自动生成，需在本设计中显式声明此能力。若决定不提供，需在 §8.5 排除清单中记录，并在 §11 迁移成本中评估替代方案。

## 问题 4（一般）：`fwd.cj` 别名脚本 `Vec`/`DVec`/`FVec` 家族名称生成存在事实错误

**问题描述**：§3.8 的 Python 脚本模板（~第 656 行）中使用 `suffix = '' if fam.endswith('Vec') else 'Vec'` 逻辑为家族名拼接后缀。对于 `Vec` 家族（`VEC_FAMILIES` 键 `'Vec'`），`fam.endswith('Vec')` 为 `true`，`suffix = ''`，因此生成的别名为 `Vec2`/`Vec3`/`Vec4`（精度前缀+家族名+维度），例如 `HighpVec2`。然而，`Vec` 家族的目标名称应为 `Vec2`（绑定 `Vec2<Float32, PackedHighp>`），而脚本生成的 `Vec2` 实际上是 `public type Vec2 = Vec2<Float32, PackedHighp>`——这与 `glm.detail` 包中直接定义的 Vec2 泛型结构体同名，将产生类型遮蔽（shadowing）问题。`public type Vec2 = Vec2<Float32, PackedHighp>` 中的 `Vec2` 在 `fwd.cj` 的 `package glm` 上下文中，右侧的 `Vec2` 引用需通过 `import glm.detail.{ Vec2 }` 引入——同一包中 `type Vec2 = Vec2` 会产生循环引用或自引用歧义。同样，`DVec` 家族生成的 `DVec2` 对应 `Vec2<Float64, PackedHighp>` 是合法的（因 DVec ≠ Vec2），但 `Vec` 家族的别名与泛型结构体同名。

**所在位置**：§3.8 向量类型别名脚本模板（~第 636-665 行）、§3.8 别名格式示例表（~第 528-538 行）

**严重程度**：一般

**改进建议**：
1. 验证仓颉 `type` 别名机制中，`public type Vec2 = Vec2<Float32, PackedHighp>` 是否合法——若右侧 `Vec2` 通过 `import glm.detail.Vec2` 引入且与左侧别名同名的 `type` 声明不冲突（alias 不遮蔽同 package 中 import 的类型），则当前脚本正确；若产生编译错误，则 `Vec` 家族的别名名称需改为其他命名（如 `F32Vec2` 形式或保持 `Vec2` 但改变生成策略）。
2. 在 §2.1 编码前验证清单中补充此编译器行为验证项：在 `package glm` 中声明 `import glm.detail.{ Vec2 }` 后定义 `public type Vec2 = Vec2<Float32, PackedHighp>`，验证编译是否通过。
3. 若验证结果为自引用不合法，修正脚本中 `Vec` 家族的处理逻辑——最简修复为排除 `Vec` 家族的别名生成（因 `Vec2<Float32, PackedHighp>` 可直接通过结构体名 `Vec2` + 类型参数使用），或为 `Vec` 家族使用不同命名规则。

## 问题 5（一般）：`equalEpsilon` 中使用 `abs()` 但未指定 `abs` 的来源和导入策略

**问题描述**：§4.5 `equalEpsilon` 代码示例中使用 `abs(this.x - other.x)` 调用，但设计文档未指定此 `abs` 函数的来源。仓颉标准库中 `abs` 是多种类型的静态成员函数（如 `Float32.abs()`、`Int32.abs()`）还是包级函数，需要根据实际 API 确定。若 `abs` 调用形式不正确（如需要 `T.abs(this.x - other.x)` 或 `import std.math.abs`），则代码示例将误导编码实施者。同样的问题影响 `ComputeEqual.callConst` 中的 `abs(a - b)` 调用。

**所在位置**：§4.5 `equalEpsilon` 代码示例（~第 1019 行）、§3.5 `ComputeEqual.callConst` 代码

**严重程度**：一般

**改进建议**：1. 查阅仓颉标准库文档确认 `abs` 的调用形式 —— 是 `(a - b).abs()` / `Float32.abs(a - b)` 还是独立函数 `abs(a - b)`；2. 在代码示例中修正 `abs` 调用形式；3. 在 §8.1 文件清单中标注 `shim_limits.cj` 或 `compute_vector_relational.cj` 对 `abs` 的依赖关系（如需添加 `import std.math.abs` 或内联实现）。

## 问题 6（一般）：§4.1 T2 跨类型约束矩阵中 `Bool` → 数值路径标注与 `fromBoolVec` 存在语义分化但未被评估

**问题描述**：§4.1 跨类型约束矩阵中 `Bool → Numeric` 路径标注为"编译错误，改用 `fromBoolVec` 工厂函数"。但 `fromBoolVec` 的语义是 `true→T(1), false→T(0)`（§4.8），而 C++ GLM 中 `vec<2, float>(bvec2(true, false))` 的跨类型构造语义是 `Bool`→数值类型的语言级转换——C++ 中 `float(true)=1.0f`，`float(false)=0.0f`。两种语义在此场景下等价。然而，若未来 C++ 代码使用 `vec<2, int>(bvec2)` 并依赖 `bool→int` 的整数提升语义（C++ 中 `int(true)=1`），`fromBoolVec` 也能覆盖。但 C++ GLM 中 `vec<2, bool, highp> b; vec<2, float, highp> f(b)` 走的是**隐式转换构造**，而非显式工厂函数调用。当前设计将此模式从隐式构造降级为显式工厂函数调用，增加了调用方的迁移负担，且此降级的替代方案在 §8.5 排除清单和 §11 迁移成本中均未完整覆盖。

**所在位置**：§4.1 T2 跨类型约束矩阵 `Bool→Numeric` 单元格、§8.5 排除清单、§11 迁移成本

**严重程度**：一般

**改进建议**：1. 在 §8.5 排除清单中新增条目"Bool→Numeric 隐式转换构造被降级为显式工厂函数"，记录排除理由（仓颉不支持 Bool→Numeric 隐式构造）和替代方案（`fromBoolVec`/`fromBoolVecQ2`）；2. 在 §11 迁移成本中补充 Bool→Numeric 隐式构造→显式工厂函数的代码迁移模式评估（类似 §11.3 跨类型赋值迁移模式）；3. 考虑在 §4.1 的 T2 矩阵 `Bool→Numeric` 单元格中补充 `fromBoolVec` 的调用示例 `fromBoolVec<Int32, PackedHighp>(bvec)` 以提升编码实施者的可用性。

## 问题 7（一般）：§2.1 编码前验证清单的验证项编号体系扩展后不够稳健——超出 21 后使用了非标准编号符号

**问题描述**：§2.1 检查清单从第 ① 项扩展到第 ㉑ 项，使用了带圆圈的 CJK 数字序号。当项数超过 20 时，`㉑`（U+3251）等字符是较少见的 Unicode 字符，在部分编辑器/工具中可能显示为方框或无法正确渲染。随着后续迭代可能继续扩展，编号体系将面临可读性和兼容性问题。

**所在位置**：§2.1 编码前验证检查清单（~第 218-239 行）

**严重程度**：一般

**改进建议**：将编号体系改为更稳健的方案，如数字编号 + 方括号形式 `[1]`/`[2]`/.../`[21]`，或使用字母编号 `A1`/`A2`/.../`A21`。同时更新所有跨章节引用此编号的地方（§7 D37、§10 等）。

## 问题 8（一般）：§3.8 别名数量声明 256 与 `fwd.cj` 文件结构中实际标注的别名族数量不一致

**问题描述**：§3.8 明确声明"16 个别名家族"，但给出的家族列表中 `Vec` 和 `DVec` 分别对应 `float` 和 `double` 的 C++ GLM 默认类型，而 `FVec` 也对应 `Float32` —— 也就是说 `Vec` 和 `FVec` 两个家族映射到相同的 `VecN<Float32, Qualifier>` 类型。别名名称固然不同（`Vec2` vs `FVec2`），但脚本模板中 `VEC_FAMILIES` 字典的 `Vec` 家族生成 `Vec2`/`Vec3`/`Vec4` 会与 `glm.detail.Vec2` 泛型结构体同名（见问题 4）。此重叠是否构成问题取决于仓颉 type alias 机制对同名的处理——但目前设计中未讨论此风险。

**所在位置**：§3.8 别名数量声明、VEC_FAMILIES 映射表

**严重程度**：一般

**改进建议**：在 §3.8 中增加一个脚注或段落，显式说明 `Vec` 家族的别名名称（如 `Vec2`）与 `glm.detail.Vec2` 泛型结构体同名这一事实，并在编码前验证中确认此类别名是否合法。若不合法，修正家族名映射规则或排除清单。

## 问题 9（轻微）：验证清单中"独立原型验证"与"统一集成验证"存在测试覆盖重叠但无退出规则

**问题描述**：§2.1 层级说明中声明"第⑫项**取代**①③④⑤⑧项的独立验证结果——若独立验证已通过但集成验证失败，以集成验证为准"。但未说明：若部分独立验证通过、部分失败（如①通过但③失败），集成验证应如何处理。集成验证失败时"以集成验证为准"含义不清——是否意味着已通过的独立验证也被推翻？此外，若①③④⑤⑧全部独立验证通过，⑫集成验证是否仍需执行（即独立验证通过不能豁免集成验证），这一点需明确。

**所在位置**：§2.1 层级说明段落（~第 215 行）

**严重程度**：轻微

**改进建议**：补充明确规则：(1) 独立验证通过后仍须执行集成验证——集成验证覆盖组合场景，独立验证仅覆盖单一假设；(2) 若独立验证部分通过部分失败，集成验证可能验证通过的子集在组合场景下的行为，仍须执行；(3) 若独立验证全部通过但集成验证失败，以集成验证为准，触发对应回退方案。

## 整体质量评价

本方案（v4）作为 GLM 1.0.3 仓颉迁移的 OOD 设计文档，在架构层面（Qualifier 体系、Vec 结构体系、包组织、别名体系）表现出较高的完整性和深度。内部审议已覆盖了编译器行为假设验证（21 项验证清单）、`@OverflowWrapping` 标注策略、`const init` 共存性等关键技术风险维度。历史迭代的 23 个问题中全部标注为已解决，且修订说明章节详细记录了每个问题到修改的映射关系，展现了良好的可追溯性。

主要遗留风险集中在**C++ GLM API 覆盖完整性**维度——Vec 与 Vec1 之间的广播运算模式（`vec2 + vec1`、`vec1 + vec2` 等）是完全遗漏的严重问题，对下游代码的等价性迁移构成直接障碍。其余问题为中等或轻微级别，可在编码阶段通过补充验证和设计微调解决。
