## 问题 1（严重）

**问题描述**：设计文档多处将仓颉浮点 `==` 的容差比较路径描述为与 C++ GLM 行为一致，但 C++ GLM 1.0.3 的 `compute_equal` 对所有类型（含浮点）均使用精确比较 `a == b`——浮点特化版 `compute_equal<T, true>` 在参考实现 `compute_vector_relational.hpp` 第 18-28 行中被整段注释掉。因此，仓颉的容差比较是一项**对 C++ GLM 的全量行为偏离**（覆盖所有浮点比较场景，而非仅 Inf/NaN 边界），但设计文档将其定性为与 C++ GLM 行为一致或仅在 Inf/NaN 边界存在偏离。具体受影响段落：

- §3.5 `ComputeEqual.callConst` 容差比较路径默认启用，未声明此为对 C++ GLM 的行为偏离
- §9.3 异常场景表中"与 C++ GLM 的 `==` 行为显著偏离"仅限 Inf/NaN 边界场景（第 1790 行），未声明在所有浮点比较场景中均存在偏离
- §11.5 `Inf == Inf` 差异分析（第 2120 行）将偏离限定为"Inf == Inf"场景，未涵盖全量浮点比较
- §11.9 `equalExact` 迁移模式（第 2318 行）将 C++ GLM 的 `==` 描述为"IEEE 754 精确相等语义"，但将容差路径描述为仅在 Inf/NaN 边界产生差异
- §4.5 `equalExact` 定义说明（第 1037 行）仅提及 Inf/NaN 边界差异，未声明容差路径在所有浮点比较场景中均与 C++ GLM 不一致

**所在位置**：§3.5、§4.5、§9.3、§11.5、§11.9 多处散布

**严重程度**：严重——此事实错误影响：(1) 迁移成本评估——§11.9 将 `equalExact` 迁移影响限定为"约 20% 涉及 Inf/NaN"，实际上所有浮点 Vec 比较代码均受影响；(2) 下游消费者行为预期——若消费者认为容差路径与 C++ GLM 一致，可能忽略对全量浮点比较场景的审查；(3) `==` 运算符语义文档的准确性——容差比较是仓颉迁移引入的新增行为，C++ GLM 无此路径

**改进建议**：(1) 在 §3.5 `ComputeEqual` 描述中新增"C++ GLM 行为偏离声明"段：明确 C++ GLM 1.0.3 的 `compute_equal` 对所有类型使用精确比较（`a == b`），浮点容差路径是仓颉迁移引入的新增行为，非继承自 C++ GLM；(2) 修正 §9.3 异常场景表——将"与 C++ GLM 的 `==` 行为显著偏离"的范围从"Inf/NaN 边界"扩展为"所有浮点比较场景"；(3) 修正 §11.9 迁移工作量评估——受影响范围从"约 20%（Inf/NaN）"上调为覆盖所有使用浮点 Vec `==` 的代码，重新评估人工审查工作量；(4) 修正 §11.5 标题和范围——从"Inf == Inf"差异扩展为"浮点容差比较与 C++ GLM 精确比较的全量行为偏离"，Inf == Inf 只是其中一个子场景；(5) 修正 §4.5 `equalExact` 说明——补充"容差路径在所有浮点比较场景中均与 C++ GLM 行为不同（包括有限值微差、Inf、NaN），`equalExact` 是恢复 C++ GLM 精确比较语义的唯一路径"

## 问题 2（一般）

**问题描述**：§4.6 标注策略总结段落中 `<<` 广播位运算符需标注 `@OverflowWrapping` 的计数存在错误。设计文档原文为"约 9 个 `<<` 广播位运算符"（3×3），但实际 `<<` 广播位运算符数量为 6 个：VecN<<Vec1 方向 3 个（Vec2<<Vec1, Vec3<<Vec1, Vec4<<Vec1）+ Vec1<<VecN 方向 3 个（Vec1<<Vec2, Vec1<<Vec3, Vec1<<Vec4）。"3×3=9"的乘数来源不明——广播运算一侧为 Vec1 另一侧为 Vec2/Vec3/Vec4 之一，不存在 3×3 维度组合。同时，设计文档 §4.6 总结段落中的"3 个 `<<` 广播"也仅覆盖了 VecN-op-Vec1 方向，遗漏了 Vec1-op-VecN 方向的 3 个。

**所在位置**：§4.6 标注策略总结段落（约第 1117 行）

**严重程度**：一般——计数错误虽不影响编码正确性（标注策略本身正确——所有 `<<` 均需标注），但会导致实施者对标注数量产生混淆，且设计文档自身的"3 个 `<<` 广播"计数遗漏了 Vec1-op-VecN 方向

**改进建议**：(1) 修正 `<<` 广播位运算符需标注数为 6 个（VecN<<Vec1 方向 3 个 + Vec1<<VecN 方向 3 个）；(2) 同步修正总结段落中"3 个 `<<` 广播"为"6 个 `<<` 广播（VecN<<Vec1 方向 3 个 + Vec1<<VecN 方向 3 个）"

## 问题 5（一般）

**问题描述**：设计文档 §3.5 和 §10 中描述 `ComputeEqual` 容差比较使用 `NumericLimits<T>.epsilon()`（机器 epsilon）作为绝对容差阈值。机器 epsilon 仅适用于接近 1.0 量级的浮点值比较——对大值（如 `|a|, |b| >> 1`）容差过于宽松，对极小值（如 `|a|, |b| << 1`）容差过于严格。设计文档未在任何显式位置声明此值域局限性。**注意**：此局限性本身不需要修正——C++ GLM 被注释掉的浮点特化版同样使用了 `epsilon`（虽未生效），且仓颉容差路径为新增行为，设计者有权选择此容差策略。但文档应在显式位置声明此局限性，使下游消费者对容差路径的适用范围有准确认知。

**所在位置**：§3.5 `ComputeEqual.callConst` 容差比较路径描述、§10 依赖③验证描述

**严重程度**：一般——值域局限性本身不是错误，但未声明可能导致下游消费者在非适用范围误用

**改进建议**：在 §3.5 或 §9.3 中补充"机器 epsilon 容差比较的值域局限性"说明段：声明容差比较使用 `NumericLimits<T>.epsilon()` 作为绝对容差阈值，此值仅适用于接近 1.0 量级的浮点值比较；对于大值容差过于宽松，对于极小值容差过于严格。需要相对容差比较的下游消费者应自行实现基于 `max(abs(a), abs(b)) * epsilon` 的比较逻辑。同时注明此为仓颉迁移引入的新增行为（C++ GLM 使用精确比较，见问题 1），设计者选择保持与 C++ GLM 废弃容差路径一致的策略。
