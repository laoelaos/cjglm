### 问题 2（严重）：附录 A 及 §8.5 中完全泛型多元构造排除理由存在事实错误

**描述**：附录 A 未覆盖 API 汇总表（第 2837-2844 行）及 §8.5 排除清单最后一项（第 1896 行）将完全泛型多元构造函数 `template<A,B,...> vecN(A, B, ...)` 的排除理由写为"仓颉构造函数不能为泛型（运算符重载限制）"。此陈述**与仓颉语言事实不符**——仓颉构造函数可以拥有泛型类型参数（本文档 §4.1 中已大量使用 `public init<T2, Q2>(v: VecN<T2, Q2>) where Q2 <: Qualifier` 形式的泛型构造函数）。实际原因应是：各分量独立推导的多元泛型参数（A/B/C/D 各可不同）在仓颉构造函数签名层面无法避免与已有同参数个数构造函数的歧义，且各分量需统一为目标类型 T，而非"构造函数不能为泛型"。

**所在位置**：
- 附录 A 第 2837-2844 行（未覆盖 API 汇总表）
- §8.5 排除清单第 1896 行

**严重程度**：严重

**改进建议**：将排除理由修正为准确的表述。建议改为："各分量类型可独立推导的完全泛型多元构造函数（如 C++ `template<A,B> vec2(A,B)` 允许 A≠B）在仓颉中无法直接映射——仓颉构造函数虽可声明泛型参数（`init<A, B>(a: A, b: B)`），但后构造函数与 `const init(x: T, y: T)` 在调用方传入同类型参数时产生重载歧义，且各分量需通过 `T(v)` 转换为统一的目标类型 `T`，此转换对 A≠B 场景增加了不必要的复杂度。C++ 该模式已由已有构造函数形态覆盖（标量填充构造 + 逐分量 const init + 跨 Vec 转换构造），无需单独映射。"

### 问题 3（严重）：§11.12 合计表说明中引用了不存在的版本号和章节

**描述**：§11.12 合计表底部说明（第 2506 行）写道"此项在 v24 中未纳入评估，v25 新增"。然而本文档的版本谱系为 v4→v5→v6→v7→v8→v9→v10→v11→v12，**不存在 v24 或 v25**。此引用明显是前序文档的遗留文本未被清理，编码实施者读到"v24/v25"时无法定位到任何对应内容。

此外，同行的"B2"行标签无对应章节——正文中不存在 §B2 或任何以 "B2" 编号的章节。实施者无法通过章节引用理解"B2 const/extend 不对称审查工作量"的含义。

**所在位置**：§11.12 合计表说明段（第 2506 行）

**严重程度**：严重

**改进建议**：
1. 将"v24/v25"修正为当前文档中首次引入该成本项的实际版本号。追溯修订说明，B2 项首现于 v11 合计表（第 2497 行）——应标注为"此项在 v11 中新增纳入评估，本轮（v12）维持不变"或直接删除版本号引用。
2. 将"B2"行标签改为清晰的章节引用，如"§11.16 const/extend 不对称审查工作量"，并在正文中新增对应成本评估子节或明确指向已有相关段落。

### 问题 4（一般）：§2.1 验证时序建议与层级说明存在逻辑矛盾

**描述**：§2.1 编码前验证检查清单包含两段指引：
- **验证时序建议**（第 271 行）：建议优先执行第⑫项（统一集成验证），再按编号顺序验证①→②→③→...。
- **层级说明**（第 273 行）：声明第⑫项"**取代**①③④⑤⑧项的独立验证结果——若独立验证已通过但集成验证失败，以集成验证为准"。

若遵循时序建议先执行⑫验证：
- 若⑫通过 → 直接跳过①③④⑤⑧，无需逐项验证——这与时序中的"再按编号顺序验证①→②→③→..."矛盾。
- 若⑫失败 → 仍需通过①③④⑤⑧逐项识别哪些假设不成立——但层级说明声称⑫"取代"了这些项的结果。

**逻辑歧义核心**：时序建议暗示⑫前置（替代后续逐项验证），层级说明暗示⑫后置（覆盖已有独立验证结果）。实施者无法确定应该先执行⑫还是先执行①③④⑤⑧。

**所在位置**：§2.1 验证时序建议（第 271 行）与层级说明（第 273 行）

**严重程度**：一般

**改进建议**：统一建议为两种路径之一：
- **路径 A（推荐）**：先执行①③④⑤⑧独立验证，再执行⑫集成验证。若⑫失败则按集成验证结果为准（取代独立验证结论）。此路径降低单次验证的复杂度，且独立验证失败时无需继续执行⑫。
- **路径 B**：先执行⑫集成验证。若⑫通过则整体跳过①③④⑤⑧；若⑫失败再回退执行①③④⑤⑧逐项隔离。此路径验证效率高。
  
在时序建议中明确说明选择其中一种路径（推荐 A），同步修改层级说明中的"取代"语义描述以匹配所选路径。

### 问题 7（轻微）：§4.8 `fromBoolVecQ2` 跨 Q 版本中 Vec2/Vec3/Vec4 代码有潜在混淆点

**描述**：§4.8 中 Vec2~Vec4 的 `fromBoolVecQ2` 函数体（第 1228-1249 行）使用 `if (v.x) { T(1) } else { T(0) }` 等模式，但未在代码注释中说明 Vec2 应处理 x/y 两个分量、Vec3 处理 x/y/z、Vec4 处理 x/y/z/w。虽然函数名清晰表达了 Vec 类型，但编码实施者在复制-修改模板时可能因疏忽而遗漏某个分量（如将 Vec3 函数体的 z 分量条件遗漏导致编译不匹配）。对比 v11 审查中指出的 `fromBoolVecQ2` 依赖编译器推断问题已修复（新增了第㉕项验证和显式标注代码模板），但分量数与成员数的一致性作为编码阶段的笔误风险尚未被标注。

**所在位置**：§4.8 `fromBoolVecQ2` 函数体（第 1228-1251 行）

**严重程度**：轻微

**改进建议**：在各 VecN 的 `fromBoolVecQ2` 代码示例后添加一行注释标注当前 Vec 类型的分量数（如 `// Vec2: handles x, y`），降低编码阶段因复制-粘贴遗漏分量的风险。

## 综合质量评价

**需求响应充分度**：✅ 产出充分回应了用户需求——首轮 OOD 设计覆盖了 roadmap §3E/3G 的全部子范围，包含完整的模块划分、核心抽象、API 契约、异常场景、迁移成本评估和验收标准。

**事实正确性**：⚠ 发现两处严重的事实错误——① 附录 A/§8.5 中"仓颉构造函数不能为泛型"的表述与仓颉语言事实不符；② §11.12 说明中引用了不存在的版本号 v24/v25。此外，§1 设计确定性声明中验证项数量自相矛盾（24 vs 25）。

**逻辑一致性**：⚠ §2.1 验证时序与层级说明存在逻辑矛盾，实施者可能对执行顺序产生混淆。其余章节间逻辑无明显冲突。

**深度与完整性**：✅ 整体深度充分，接口定义（§4.1 构造函数清单、§4.3~§4.5 运算符签名、§4.8 工厂函数）足以支撑编码实现。异常场景（§9）、类型转换边界（§9.4）、Inf/NaN 传播（§9.3）等边界条件已覆盖。25 项编码前验证项为编译器假设提供了结构化验证依赖。

**编码可实施性**：✅ 编码实施快速参考框（§1）、迁移顺序（§8.3）、文件清单与依赖拓扑（§8.1）、排除清单（§8.5）为编码阶段提供了直接可用的指引。各 API 签名的仓颉语法形态已完整给出。

**主要改进方向**：上述 7 个问题中，问题 1/2/3 需要优先修复（事实错误），问题 4/5/6 为完整性改进，问题 7 为编码辅助优化。
