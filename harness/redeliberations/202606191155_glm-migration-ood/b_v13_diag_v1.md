## 问题 1（严重）：7 个非 Vec 核心文件的 API 映射盲区未设明确闭环节点，编码阶段存在"扫到责任人不明"的风险

**所在位置**：§1 P37 状态描述（第 95-96 行）、§12.2 验收项"7 文件编码启动前快速扫描"（第 2739 行）、附录 A 验证范围声明（第 2805 行）

**问题描述**：产出明确声明 11 个首轮核心文件中仅 4 个 Vec 类型文件（type_vec{1,2,3,4}.hpp/.inl）已完成逐 API 级对照，其余 7 个文件（setup.hpp、qualifier.hpp、compute_vector_relational.hpp、compute_vector_decl.hpp、_vectorize.hpp、fwd.hpp、shim 文件）仅在 §8.2/§8.5 中以子范围粒度映射。§12.2 虽设置了"7 文件编码启动前快速扫描"验收项，但其验证方法仅描述为"逐文件按风险等级（§1 P37 已标注）审查，确认每个文件的 API 映射状态与当前设计一致"——未定义以下关键要素：

- **扫描执行人**未指定（由编码实施者自主判断？还是设计者补充？）
- **扫描通过标准**缺失（发现遗漏依赖后如何判断是否阻断编码启动？）
- **扫描超时升级路径**缺失（若扫描发现多项遗漏依赖，工期影响如何评估？谁做决策？）
- **发现≠解决**：验收项仅提到"更新对应设计章节并记录于范围变更日志"，未规定修复的优先级和责任人

**严重程度**：严重。编码实施者面对此模糊要求时，可能跳过扫描直接编码，将 7 个文件中的隐式依赖问题推迟到集成阶段才发现。

**改进建议**：
1. 为"7 文件快速扫描"设置明确的执行责任归属（建议：设计者或主 Agent 在编码启动前完成，而非由编码实施者自行判断）
2. 补充扫描通过标准（例："7 文件中任一文件发现 >3 项未映射 API 或 >1 项被引用但未定义的依赖类型 → 触发范围变更评审；≤3 项且无被引用依赖 → 标记后进入编码"）
3. 在 §8.3 迁移顺序的阶段 1 前增加一个前置步骤"阶段 0：非 Vec 文件 API 快速扫描"，明确此阶段输出的产物（扫描报告）和审批人

## 问题 2（严重）：密集的编译器行为假设与"25 项全部通过即可编码"的乐观路径之间缺少可靠的置信度锚点

**所在位置**：§1 设计确定性声明（第 116-124 行）、§2.1 验证清单（第 275-301 行）、§11.12 成本合计表（第 2498 行）

**问题描述**：产出依赖 25 项编译器行为假设，其中 4 项标注为"高影响"（①③④⑫）、10 项"中影响"、11 项"低影响"。然而：

- 乐观路径合计（5.9 人天）的前提是"所有 25 项验证全部通过"——此前提在当前仓颉编译器生态中成立的概率未知。产出未提供任何历史数据或类似项目的先例证据来支撑"大概率全部通过"的判断。
- 最可能路径（12.15 人天）与乐观路径的差距为 6.25 人天（>100%），悲观路径（22.5 人天）差距达 16.6 人天（~280%）——如此大的方差反映设计本身对编译器行为的依赖程度极高，但产出未给出在编码启动前即可快速评估置信度的"快速预筛"方案。
- 设计确定性声明（第 116 行）的 Z 类别触发阈值"①②③⑫⑮ 五项中任意三项同时失败"中的 ⑮（编译器版本确认，低影响）被纳入停工条件，而同等重要的 ④（编译期 `if` 分支抑制验证，高影响）却在阈值条件之外——此选择缺乏显式理由，可能误导实施者对风险优先级的判断。

**严重程度**：严重。编码实施者面临的核心问题是：投入 1.4 人天完成全部 25 项验证后，若发现多项假设不成立，之前的"完整设计"实际上需要重评估——此风险在设计文档中虽已识别但未给出早期预警机制。

**改进建议**：
1. 在 §2.1 验证时序建议中新增"快速预筛"步骤：在执行全部 25 项验证前，先执行 3-4 项最低成本但最具诊断价值的验证（如⑬ Vec4 重载解析、⑮ 编译器版本、⑰ `%` 运算符行为），输出一份快速置信度报告。
2. 重新审视 Z 类别阈值条件：明确阈值所依赖的 5 项选择的理由，或修正为仅包含高影响项（①③④⑫），删除低影响项 ⑮，避免歧义。
3. 在 §11.12 合计表旁增加一列"置信度评估"或脚注，说明乐观/最可能/悲观三档在当前编译器生态下的概率估计依据（即使只是经验估计）。

## 问题 3（一般）：Vec 运算符的重复编码缺乏降本指引，实施者面临约 2000+ 行手动编写无模板可循

**所在位置**：§3.2 Vec 结构体系（第 369-394 行）、§4.3 算术运算符模板（第 981-1028 行）、§7 D1（第 1324-1328 行）

**问题描述**：每个 type_vecN.cj 需要手动实现：
- 约 15-30 个构造函数重载（Vec4 约 30 个，Vec1 约 5 个）
- 10 个二元算术运算符（5 运算符 × 2 方向：VecN op VecN + VecN op T）
- 5 个一元运算符
- 10 个位运算符及 Vec1 广播（双向 × 5 运算）
- 30 个 Vec1 广播算术运算符
- 约 10 个具名函数（increment/decrement/bitwiseNot/logicalAnd/logicalOr/equalEpsilon/equalExact）
- 下标运算符及 const 替代

合计每个 Vec 类型约 70-80 个函数/运算符定义，4 个 Vec 类型合计约 280-320 个。§4.3 提供了 Vec2 的算术运算符作为"编码参考模板"（第 981 行），但也仅覆盖 Vec2 的 10 个算术运算符——其他 30 个广播运算符、20 个扩展成员函数、以及 Vec1/Vec3/Vec4 各自的适配均无模板。

§7 D1 声明"首轮四个 Vec 结构体的定义模式高度一致且数量可控（4 个），独立编写可提供更好的 IDE 支持"，但未给出如何高效完成这 280+ 个定义的实操方案。与 §3.8 别名脚本的"零遗漏"策略相比，Vec 运算符实现完全依赖人工编写，遗漏和笔误的风险更高。

**严重程度**：一般。不构成设计层面的阻塞性问题，但延长编码阶段、增加缺陷密度。

**改进建议**：
1. 在 §7 D1 下方新增"编码实现效率建议"提示框，推荐编码实施者使用多光标编辑、片段模板或简单脚本生成 Vec 运算符的重复模式代码——可参考 §3.8 别名生成脚本的思路，为 type_vecN.cj 的主要代码块（构造函数列表、算术运算符、位运算符）提供结构化模板，标注占位符位置。
2. 在 §8.3 迁移顺序的阶段 5a/5b 中为每个子步骤增加预期代码量的标注（如"此步骤约需编写 30 行：Vec2 二元算术运算符"），帮助实施者预估工时。

## 问题 4（一般）：P37 的整体风险评级与后续轮次依赖关系存在隐性矛盾

**所在位置**：§1 P37 状态描述（第 95-96 行）、附录 A 验证范围声明（第 2805 行）

**问题描述**：产出评估其余 7 个文件的风险等级为 2 个"中"、5 个"低"，整体评估为"无高等级风险文件"。但同时声明"其余 7 个文件的逐 API 级对照规划在第 2 轮编码阶段以本附录为模板补充完成"——这意味着第 2 轮编码阶段实施者的首要任务之一是补充本应在首轮设计阶段完成的 API 映射工作。这在实践中可能导致：

- 第 2 轮编码启动时发现 setup.hpp 或 qualifier.hpp 中存在被首轮遗漏的 API，需要回溯修改首轮代码。
- 首轮"无高等级风险"的评估结论因其未经逐 API 验证而本质上是推测性的——后续可能被证实为误判，届时设计变更成本更高（涉及已交付代码的回归）。

**严重程度**：一般。当前的 4/11 覆盖度声明是透明的，但"无高等级风险"的推测性结论与"第 2 轮逐 API 对照"的安排之间的时间差风险未被显式管理。

**改进建议**：
1. 将 §1 P37 状态描述中的"整体评估：无高等级风险文件"措辞修正为"当前评估：基于首轮范围定义，7 个文件中未发现需在首轮设计阶段阻断的已知风险项。此评估的确认依赖编码启动前的快速扫描（详见 §12.2 验收项）。"——消除"结论已确认"的暗示。
2. 在 §8.3 迁移顺序的阶段 1 备注列补充"此阶段的 3 个 shim 文件的 API 覆盖评估为依赖项级别，非逐 API 级别——编码时若发现新的依赖 API，即时补充设计记录并更新 §8.5。"

## 问题 5（轻微）：§1 确定性声明的"核心假设"清单与 §2.1 验证项的高影响项不完全一致

**所在位置**：§1 设计确定性声明（第 116-124 行，特别是第 122 行 Z 类别阈值）

**问题描述**：§1 设计确定性声明的 Z 类别将停工阈值设为"①②③⑫⑮ 五项中任意三项同时失败"。但：
- §2.1 后续的影响范围分类（第 309 行）将"高影响"项标注为 ①③④⑫，而 Z 阈值中的 ② 是"中影响"、⑮ 是"低影响"
- 高影响项 ④（编译期 `if` 分支抑制验证）不在 Z 阈值中，而低影响项 ⑮（编译器版本确认）却在

无论选择哪一组，至少有两种选择可与分类体系自洽：（A）仅高影响项：①③④⑫四项中任意两项失败→停工；（B）核心依赖项：①③④⑫⑮ 五项中任意三项失败→停工。当前混合选择（①②③⑫⑮）无自洽的分类依据，实施者无法判断为何 ④（高影响）被排除、②（中影响）和 ⑮（低影响）被包含。

**严重程度**：轻微。不影响编码实施，但体现设计确定性框架与验证分类体系之间的脱节。

**改进建议**：将 Z 类别阈值修正为与 §2.1 影响范围分类一致：建议采用"①③④⑫ 四项高影响项中任意两项同时失败 → 触发停工评估"，并在脚注中说明选择 ⑤ 项（含低影响项 ⑮）的原因（如"编译器版本低于 1.0.5 可能导致后续验证项无法执行"），或移除 ⑮ 使阈值仅含高影响项。

## 问题 6（轻微）：§4.6 标注策略中 `@OverflowWrapping` 对浮点类型的行为描述存在可优化空间

**所在位置**：§4.6（第 1187 行）

**问题描述**：产出声明"浮点类型（`Float32`、`Float64`）在仓颉中不会触发溢出异常（浮点溢出产生 `±Infinity` 而非异常），因此 `@OverflowWrapping` 在浮点类型的 Vec 上语义无影响（标注与否行为一致）。首轮采用统一标注的简化策略——在所有 Vec 二元算术运算符上标注，无论 `T` 为何类型。"

此描述本身正确，但忽略了 `@OverflowWrapping` 的**编译期行为影响**：标注 `@OverflowWrapping` 后，编译器可能关闭对该函数的溢出检测优化（如移除了不必要的边界检查）。对于浮点运算——虽然不会触发整数异常——标注的存在可能影响编译器的代码生成选择（如是否插入额外的 NaN/Infinity 检测）。此影响虽小，但若编码阶段发现 `@OverflowWrapping` 在浮点类型上导致非预期编译错误或性能降级，当前设计未提供排除策略。

**严重程度**：轻微。

**改进建议**：在 §4.6 浮点类型标注段的末尾补充一条脚注，标注可选的编译期优化路径："若编码阶段发现在浮点类型上标注 `@OverflowWrapping` 产生非预期编译问题，可改为在二元算术运算符上按类型条件性地标注——对浮点 Vec 的运算符定义不移除标注（保持代码统一性），但可通过编译期 `if` 在浮点路径中跳过 `@OverflowWrapping` 的影响。此方案未在首轮设计中显式推荐，仅在编码阶段发现限制时作为参考。"
