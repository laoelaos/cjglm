## 问题清单

### 问题 1：`const init` 内不可使用复合赋值表达式，但设计声称 `const init` 可初始化 `var` 成员——跨类型构造函数的语义依赖链存在断裂风险

**问题描述**：§3.2 声称 `const init` 内可使用 `=` 赋值初始化 `var` 成员。仓颉 const 文档 §4.2 明确规定："`const init` 内可以使用赋值表达式 `=` 对实例成员变量赋值，除此以外不能有其他赋值表达式（如 `+=`、`-=`）"。然而，Vec 的跨类型构造函数 `init<T2, Q2>(v: VecN<T2, Q2>)` 内部需要对各分量逐个执行 `this.x = T(v.x)` 等赋值——当 `T2 ≠ T` 时，`T(v.x)` 是类型构造函数调用，不是简单的 `=` 赋值初始化。设计未分析以下关键场景：当 `const init` 与跨类型构造函数 `init<T2, Q2>` 同签名时（Vec1 的情况除外），`const init` 内部的 `T(v.x)` 类型构造表达式是否作为"赋值表达式 `=`"的右侧被 const 上下文接受？若编译器拒绝 `T(v.x)` 在 `const init` 中的使用（因为 `T(v.x)` 依赖运行时类型转换），则所有跨类型构造函数将无法声明为 `const init`，导致 `${==}$` 运算符无法在 `const` 上下文中使用。

更具体地：Vec1 的 `const init(x: T)` 天然合法（`T` 类型值构造），但 Vec2~Vec4 的 `const init(x: T, y: T, ...)` 也只涉及同类型参数，因此也没问题。**真正的问题在于**：非 `const` 的跨类型构造函数 `init<T2, Q2>` 与 `const init(x, y, ...)` 共存时，如果跨类型构造中的 `T(v.x)` 在某些 `T2→T` 组合下不被 `const init` 支持（如 `Bool→Int32` 不合法），这是否导致整个 `const init` 实例化路径受影响？设计对此未做分层分析。

**所在位置**：§3.2 `const init` 相关段落（~第 291 行）、§4.1 构造函数清单（~第 697-810 行）

**严重程度**：严重

**改进建议**：(1) 明确将构造函数分为两类：`const init`（逐分量初始化，同类型参数，无需类型转换）和非 `const` `init`（跨类型转换），并在构造函数清单中标注哪些是 `const init`、哪些是普通 `init`；(2) 分析跨类型构造函数是否可与 `const init` 安全共存——如果 `const init` 的参数列表与跨类型 `init` 参数列表永远不同（因参数数量或类型结构差异），则共存无问题；(3) 在 §2.1 编码前验证清单中补充验证项：确认同一结构体上 `const init` 与非 `const init` 共存时的编译器行为。

### 问题 2：版本号标识混乱——文件声称"v3"但被审查为"第 2 次迭代"的产出

**问题描述**：产出文件头部标注"v3"且迭代轮次标注"第 3 轮"（第 4-5 行），但任务描述明确当前为"第 2 次"迭代。此版本号不一致意味着：(1) 产出方可能混淆了外部迭代编号与内部版本编号；(2) 迭代历史中的修订说明（v2/v3）是针对产出自身的内部迭代，而非外部再审议循环的轮次编号。这种混用会导致跟踪混乱——外部方按"第 2 次"理解，但产出内部已自标注为"第 3 轮"。

**所在位置**：文件头部第 4-5 行

**严重程度**：一般

**改进建议**：明确区分"外部再审议迭代轮次"和"产出内部版本号"两个维度，在文件头部分别标注。例如：`外部迭代轮次：第 2 次` / `产出内部版本：v3`。避免使用单一的"迭代轮次"字段承载两种不同含义。

### 问题 3：`%` 运算符对浮点类型 Vec 的编译错误缺乏对下游影响的充分分析

**问题描述**：§3.2 和 §4.3 声明 `%` 运算符在 `VecN<Float32, Q>` 实例化时因 `Float32` 不支持 `%` 而报编译错误，将此行为归类为 D5（宽松泛型约束）的延迟检查语义。然而，设计未分析以下关键场景：当用户编写泛型函数 `func foo<T, Q>(v: Vec2<T, Q>, s: T) where Q <: Qualifier { v % s }` 时，如果 `T` 的实例化包括浮点类型，编译器将在调用处而非定义处报错。**关键遗漏**：仓颉编译器对泛型函数体中"对所有合法 `T` 均非法"的表达式（如 `Float32 % Float32`）是否在**定义处**就报错，还是在**实例化处**才报错？这与 C++ SFINAE/延迟检查的行为不同——仓颉编译器可能选择在定义处做"至少一组实例化须合法"的检查（类似 Rust 的 well-formedness 检查），此时 `%` 运算符在 `Vec2<T, Q>` 定义处就因"对所有 T 不合法"而编译错误，而非延迟到实例化处。设计对此编译器行为未做验证。

**所在位置**：§3.2 `%` 运算符段落（~第 301 行）、§4.3（~第 927-930 行）、§7 D13

**严重程度**：严重

**改进建议**：(1) 在 §2.1 编码前验证清单中补充验证项：定义包含 `%` 运算符的泛型 struct（`struct Foo<T> { public operator func %(rhs: Foo<T>): Foo<T> { Foo(this.value % rhs.value) } }`），验证编译器是在定义处还是实例化处报错；(2) 若编译器在定义处报错，则 `%` 运算符不能定义在泛型 Vec 结构体内部，需要改用其他策略（如改为仅在整数特化的 extend 块中定义、或完全移除 `%` 运算符改用 `%` 的具名函数 `rem()` 替代）。

### 问题 4：`lib.cj` 的 `public import` 重导出策略依赖仓颉 `public import` 的实测行为，但未纳入编码前验证清单

**问题描述**：§2 公共 API 面设计（~第 94-112 行）声称 `lib.cj` 通过 `public import glm.detail.{ Vec1, Vec2, ... }` 将核心类型重导出到 `package glm`。然而，仓颉包文档中 `public import` 的重导出行为是否真正使得 `import glm.*` 的下游消费者通过 `glm.Vec2` 路径访问到类型，仍有细微模糊——`public import` 重导出的类型在目标包中是否保持原始的访问修饰符（`public`）？重导出后类型的名义路径是否变为 `glm.Vec2` 还是保持 `glm.detail.Vec2`？这些差异影响下游代码中的类型名匹配和 `is` 运算符行为。

**所在位置**：§2 公共 API 面设计（~第 94-112 行）

**严重程度**：一般

**改进建议**：在 §2.1 编码前验证清单中补充验证项：创建两个包（`pkg_a` 定义 `public struct S`，`pkg_b` 执行 `public import pkg_a.{S}`），验证从第三个包通过 `import pkg_b.*` 是否可访问 `S`，以及 `S` 的类型名义路径（`is` 运算符、`TypeInfo` 等场景下的行为）。

### 问题 5：`equalEpsilon` 在整数类型 `T` 上的编译可用性假设缺乏验证

**问题描述**：§4.5 `equalEpsilon` 的实现（~第 993-1004 行）引用 `epsilonOf<T>()`。§3.5 兼容性策略讨论了路径 A（保留 `epsilonOf<T>()` 并移除 `Number<T>` 约束）和路径 B（完全移除 `epsilonOf<T>()`）。然而，若选择路径 A 并移除 `Number<T>` 约束，`epsilonOf<T>()` 的实现变更为 `const func epsilonOf<T>(): T { T(1e-6) }`——此时对整数类型 `T = Int32`，`T(1e-6)` 将产生编译错误（`Int32` 构造函数不接受 `Float64` 字面量）。路径 A 中移除 `Number<T>` 约束后，`epsilonOf<Int32>()` 的可用性并未被验证。设计声称路径 A"修改范围最小"，但如果 `T(1e-6)` 对整数类型不合法，路径 A 实际上需要额外的分支逻辑（如 `const` 函数体内的编译期 `if` 检测 `T` 是否为浮点类型），复杂度与路径 B 相当。

**所在位置**：§3.5 `epsilonOf<T>()` 兼容性策略（~第 412-419 行）、§4.5 `equalEpsilon` 实现（~第 993-1004 行）

**严重程度**：严重

**改进建议**：(1) 在路径 A 的描述中补充分析：移除 `Number<T>` 约束后 `epsilonOf<T>()` 返回 `T(1e-6)` 对整数的可用性问题；(2) 若 `T(1e-6)` 对整数不合法，路径 A 需增加编译期 `if` 分支（仅在浮点 `T` 时返回 epsilon），这使得路径 A 的修改范围不再"最小"；(3) 重新评估路径 A/B 的优劣排序，考虑路径 B（内联 `T(1e-6)` 并仅在浮点分支中存在）可能更简单直接；(4) 在 §2.1 编码前验证清单中补充验证：`Int32(1e-6)` 是否编译通过。

### 问题 6：Gen 脚本模板中标量别名部分缺少 `uint` 别名和高精度前缀别名，生成的 `fwd.cj` 将不完整

**问题描述**：§3.8 的 Python 生成脚本模板（~第 593-645 行）中标量别名部分的循环仅覆盖 `SCALAR_MAP` 中的 10 个类型与 `SCALAR_PRECISIONS` 的 4 个精度变体，生成 40 个标量别名。但 §3.7 的完整别名列表明实际需要：10 基础类型 × 4 精度变体 = 40 个 + `uint` 别名 1 个 + `highp_uint`/`mediump_uint`/`lowp_uint` 3 个 = **44 个**。脚本中 `uint` 别名在标量部分开头硬编码输出一行（~第 624 行），但缺少 `highp_uint`/`mediump_uint`/`lowp_uint` 三个精度变体（因为 `uint` 未出现在 `SCALAR_MAP` 中，不参与精度变体循环）。此外，`SCALAR_PRECISIONS` 的格式是 `(前缀, 废弃字段)` 对，但循环中 `alias = f'{prec_prefix}{glm_name}'` 将为无前缀变体生成如 `int8`、为 `highp_` 变体生成 `highp_int8` 等——这些别名名与 §3.7 表格一致。但 `uint` 的精度变体（`highp_uint` 等）需要单独处理或在 `SCALAR_MAP` 中增加 `uint` 条目。

**所在位置**：§3.8 Python 脚本模板（~第 593-645 行）

**严重程度**：一般

**改进建议**：(1) 在 `SCALAR_MAP` 中增加 `'uint': 'UInt32'` 条目，使 `uint` 也参与精度变体循环；(2) 移除脚本中单独硬编码的 `lines.append(f'public type uint = UInt32')` 行（避免与循环重复生成无前缀 `uint`）；(3) 验证修复后脚本输出 44 个标量别名，与 §3.7 表格完全匹配。

### 问题 7：设计文档对 Vec2~Vec4 的 `scalar: T` 标量填充构造函数是否与 `Vec1<T, Q>` 参数构造函数产生重载歧义未做充分分析

**问题描述**：§4.1 中 Vec2 的构造函数清单同时包含 `public init(scalar: T)` 和 `public init(v: Vec1<T, Q>)`（`fill-from-Vec1`）。当 `T` 的值为 `Vec1<T, Q>` 类型时（虽然实际不会出现，因为 `T` 是数值类型），这两个构造函数不会冲突。但设计未分析以下场景：仓颉编译器在解析 `Vec2(x)`（其中 `x` 是 `Vec1<Int32, PackedHighp>` 类型变量）时，是否优先选择 `init(v: Vec1<T, Q>)` 而非尝试将 `Vec1` 实例作为 `T` 类型参数传入 `init(scalar: T)`？"具体类型优先于泛型类型"规则仅在第 §4.1 和 §10 中针对 Vec4 的 `Vec1<T,Q>` vs `Vec1<T2,Q2>` 讨论中提及，但未对 `scalar: T` vs `Vec1<T,Q>` 的重载歧义做显式分析。

虽然 `T` 是数值类型而 `Vec1<T,Q>` 是结构体类型，两者类型结构显著不同，编译器应能区分，但设计未在 §2.1 中新增对应的验证项予以确认。

**所在位置**：§4.1 构造函数清单（~第 714-810 行）、§2.1 编码前验证清单

**严重程度**：轻微

**改进建议**：在 §2.1 编码前验证清单中补充验证项：对 Vec2~Vec4，编写原型测试验证 `Vec2(Vec1<T,Q>)`（fill-from-Vec1 版本）在 `Vec1` 和 `init(scalar: T)` 共存时优先选择 fill-from-Vec1 版本。

### 问题 8：`Inf == Inf` 容差路径行为差异的被消解路径缺乏条件追溯

**问题描述**：§9.3 和 §11.5 详细描述了浮点 `Inf == Inf` 在容差路径下返回 `false` 的行为差异，并声称"若 `const operator ==` 的容差路径因验证失败而整体回退到精确比较（§7 D29），此差异自动消除"。这一声称隐含了一个条件：差异消解**仅当 D29 回退被触发时**生效。意味着首轮交付的 `==` 行为**取决于 D29 验证结果**——如果 D29 通过（容差路径可行），`Inf == Inf` 返回 `false` 的行为差异将作为首轮已知限制交付；如果 D29 回退，差异自动消解。设计未在验收标准（§12.2）中明确标注这一条件依赖——验收标准应声明：`Inf == Inf` 的行为取决于 D29 验证结果，两种路径均为首轮合法交付。

**所在位置**：§9.3（~第 1722 行）、§12.2 验收标准

**严重程度**：一般

**改进建议**：在 §12.2 验收标准中新增验收项：`Inf == Inf` 行为的两种合法路径——(1) 容差路径可行时 `Inf == Inf` 返回 `false` 以 `equalExact` 补救；(2) D29 回退时 `Inf == Inf` 返回 `true` 与 C++ GLM 一致。验收时应根据 D29 验证结果选择对应路径的测试用例。

### 问题 9：`@Derive[Hashable]` 对泛型 struct 的约束传递性描述与仓颉文档不一致

**问题描述**：§3.2 声称 `@Derive[Hashable]` 在泛型 struct `VecN<T, Q>` 上使用时，"仓颉编译器要求所有实例化路径中 `T` 必须实现 `Hashable` 接口"。然而，查看仓颉语言文档中关于 `@Derive` 的描述，自动派生的语义是"为所有字段类型已实现 `Hashable` 的实例化生成 `hashCode()`"——即编译器在**具体实例化时**检查字段类型是否满足 `Hashable`，而非在**泛型定义处**要求 `T <: Hashable` 约束。如果编译器采用延迟检查策略（在实例化处报错而非定义处），则 §3.2 中"编译器要求所有实例化路径中 T 必须实现 Hashable 接口"的描述就不准确——实际情况是仅在实例化时才检查。这个区别对设计的影响：如果编译器在定义处就要求 `T <: Hashable`，则需要在 Vec 的泛型参数上增加 `where T <: Hashable` 约束，与 D5（宽松约束）冲突。

**所在位置**：§3.2 Hashable 实现策略（~第 314-316 行）

**严重程度**：一般

**改进建议**：(1) 在 §2.1 编码前验证清单中对 `@Derive[Hashable]` 补充验证：定义 `@Derive[Hashable] struct Foo<T> { var x: T }`（无 `where T <: Hashable` 约束），验证编译器是在定义处还是实例化处报错；(2) 根据验证结果调整 §3.2 的描述——如果编译器采用延迟检查，将描述从"要求所有实例化路径中 T 必须实现 Hashable"改为"仅在实例化时检查字段类型是否满足 Hashable"。

## 总体评价

产出在架构完整性、决策可追溯性和回退方案完备性方面表现优秀——12 个设计决策（D1-D37）均有明确理由和回退路径，编码前验证清单覆盖 15 个验证项，历史迭代问题 P1-P13 全部标记为已解决。上述问题中，问题 1（`const init` 与跨类型构造共存性）、问题 3（`%` 运算符延迟检查假设）、问题 5（路径 A 的 `epsilonOf<T>()` 对整数可用性）为严重级别，均涉及核心编译器行为假设未被纳入验证清单。建议编码前优先补充这三项的原型验证。
