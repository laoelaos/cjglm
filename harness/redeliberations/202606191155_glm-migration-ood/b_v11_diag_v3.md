## 2. 事实错误与逻辑矛盾

### 2.1 §3.2 const 上下文 API 可用性速查表的事实错误未被修复

**问题描述**：`v.increment()` 行的 const 替代方案仍标注为"改用 `v = v + T(1)` 或 `v = add(T(1), v)` 等展开模式"。其中 `v = v + T(1)` 包含赋值操作 `=`——赋值在 const 函数体内不可用（const 函数不可修改实例成员），因此该替代方案本身不是 const 兼容的。正确的 const 替代方案应为 `add(T(1), v)`（包级独立函数，声明为 const）或直接构造新向量。此问题在上一轮审查（b_v11_diag_v2.md §2.3）中已被指出，但 v11 修订未修复。

**所在位置**：§3.2 const 上下文 API 可用性速查表（`v.increment()` 行）

**严重程度**：中等（已在上一轮指出但未修复，持续影响编码实施者的参考准确性）

**改进建议**：将 `v.increment()` 行的 const 替代方案中的 `v = v + T(1)` 移除，仅保留 `add(T(1), v)`（包级独立函数，声明为 const）作为正确的 const 兼容替代方案。同步检查该速查表中其他行的替代方案是否也存在类似问题（如 `v.decrement()` 行）。

## 3. 深度与完整性

### 3.1 `fromBoolVecQ2` 编译器推断依赖的验证缺口在第 11 轮仍未闭环

**问题描述**：§4.8 的 `[^compiler-dep]` 脚注描述了"编译器从返回值类型推断 struct 泛型实参"的行为假设，但该假设在 §2.1 的 24 项验证清单中完全无对应项。此问题自第 2 轮审查（b_v11_diag_v2.md §2.1，标记为"严重"）起已被连续指出，但历经多轮修订（v8→v9→v10→v11）均未在 §2.1 清单中新增独立验证项。

第 11 轮修订（修订说明 v11 的 7 个问题）中不包含与此相关的修复。24 项验证清单保持为 ①~㉔，未新增第㉕项。这意味着编码实施者在进入 `fromBoolVecQ2` 的实现阶段时，该编译器行为假设仍处于"零验证覆盖"状态。

此问题的影响面按 §2.1 验证项影响范围分类标准（高/中/低），应归类为"中"——失败时影响 8 处函数定义（Vec1~Vec4 × 同 Q/跨 Q）的泛型参数标注方式，且发现时机可能推迟到 `type_fromBoolVec.cj` 编码阶段。

**所在位置**：§4.8 `[^compiler-dep]` 脚注、§2.1 编码前验证检查清单（缺少此项）、§11.13 编译器依赖验证回退成本表（未覆盖此场景）

**严重程度**：严重（验证缺口持续存在，且跨多轮迭代未被修复）

**改进建议**：① 在 §2.1 验证清单中新增第㉕项，验证编译器是否支持从函数返回值类型推断 struct 构造函数的泛型参数；② 在 §11.13 回退成本表中补充此依赖的验证和回退成本；③ 在 §4.8 的回退方案说明中补充更具体的代码模式（如显式标注 `<T, Q>` 后的完整函数体示例）。

### 3.2 §10 验证表消重工作仍不彻底

**问题描述**：v11 修订（问题 5）声称对 §10 和 §12.3 做了进一步消重，但 §10 中仍保留了多个完整的结构化验证表（equalEpsilon 验证、Bool→Numeric 转换验证、Vec4 重载解析验证、mod 边界值验证、编译期 if 抑制依赖点清单）。虽然各表的"验证方法"列已改为引用 §2.1 检查项，但表格本身作为独立文档元素存在——维护者修改 §2.1 时仍需同步更新 §10 对应表的引用列。跨章节同步成本的本质问题未解决。

**所在位置**：§10 多个验证子节

**严重程度**：轻微（维护成本问题，不影响编码实施）

**改进建议**：将 §10 中所有非设计特有的结构化验证表替换为纯文本单行引用（如"equalEpsilon 所有验证依赖项均由 §2.1 项⑫⑲㉒覆盖"），仅在 §10 保留设计特有的行为描述和回退方案依赖关系分析。

## 4. 编码落地可行性

### 4.1 `fromBoolVecQ2` 编译器推断依赖对编码实施阶段的直接影响

**问题描述**：承接 §3.1 的验证缺口，此依赖对编码实施者的具体影响如下：

该依赖在 §4.8 的函数定义中以 `[^compiler-dep]` 脚注形式标注，但未指定回退方案的具体代码模式。实施者首次编写 `fromBoolVecQ2` 时面临两个需要自行判断的问题：
1. 是否需要为 `Vec1(...)` 调用显式标注 `<T, Q>`（取决于编译器是否支持推断）——此判断本应在设计阶段通过原型验证提前回答，现被推迟到编码阶段。
2. 若编译器不支持推断，8 处 `VecN(...)` 调用均需修改，且跨 Q 版本中 Q 与 Q2 两个泛型参数在 `VecN<T, Q>(...)` 标注中的顺序可能导致笔误（混淆 Q 与 Q2 的位置）。

**所在位置**：§4.8 `[^compiler-dep]`、§11.13 回退成本表（未覆盖此场景）

**严重程度**：一般

**改进建议**：① 在 §4.8 中补充回退方案的具体代码模板（显式标注 `<T, Q>` 后的完整函数体示例）；② 在 §11.13 回退成本表中新增一行覆盖此依赖的验证和回退成本估算。

## 5. 整体评价

本 v11 产出经过 11 轮迭代审议，在架构正确性、回退方案全面性、编译器依赖分析和迁移模式设计方面已达到较高的成熟度。从需求响应充分度看，产出已充分覆盖 roadmap 首轮范围；从异常/边界条件覆盖看，§9 的文档化已足够指导编码实现；从 API 消费者视角看，导入路径设计和别名体系已具有良好的可用性。

**本轮报告指出的问题**：
- **1 个中等严重程度**：§3.2 const 上下文速查表的事实错误（已被前轮指出但未修复）
- **1 个严重级别**：§2.1 编码前验证清单中 `fromBoolVecQ2` 编译器推断依赖的验证缺口（跨多轮迭代未被修复）
- **1 个轻微级别**：§10 验证表消重不彻底（维护成本问题）

**建议优先修复**：
1. §2.1 新增第㉕项验证 `fromBoolVecQ2` 编译器推断依赖——此缺口是唯一跨多轮迭代未被修复的"严重"级问题
2. §3.2 const 上下文速查表的 `v.increment()` 替代方案修正——修复成本极低（修改一行文字），但上一轮已被指出后仍未修复，修复后有助于恢复对文档修订跟踪机制的信心
