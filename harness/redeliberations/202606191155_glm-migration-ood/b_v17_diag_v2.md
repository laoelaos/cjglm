## 问题详情

### Q1（严重）：D30 回退方案中循环委托遗漏未被修复

**问题描述**：v1 审查报告（Q1）指出了 D30 回退方案中的循环委托问题，但本产出 v17 未做任何修改——v17 的 §7 D30 回退方案描述与 v16 完全一致，循环风险仍然存在。

循环产生的机理（与 v1 报告一致）：
- §4.3 第 1015 行声明扩展成员函数（`add(v,s)`/`sub(v,s)`/`mul(v,s)`/`div(v,s)`/`mod(v,s)`，共 20 个）的内部实现**委托给对应 Vec 的二元算术运算符**（`{ self + s }`）。
- §7 D30 回退方案（第 1609 行）仅要求"将运算符委托目标从'直接逐分量运算'改为'调用已标注的扩展成员函数'"。
- 若运算符改为调用扩展成员函数，而扩展成员函数仍保持 `{ self + s }` 模式（委托给运算符），则形成：`operator + → add(v,s) → self + s → operator + → ...` 循环。

§4.6 第 1226 行声明"vec-op-scalar 方向扩展成员函数...无需直接标注。这些函数的内部实现委托给对应的二元算术运算符。委托实现模式：`func add(self, s) { self + s }`"。D30 回退方案未提到需将此实现从`{ self + s }`改造为直接逐分量运算（`{ Vec2(this.x + s, this.y + s) }`）。

**所在位置**：
- §4.3 第 1015 行：当前扩展成员函数的委托实现声明
- §7 D30 回退方案（第 1609 行）：仅修改运算符委托目标，未提及扩展成员函数改造
- §4.6 第 1226 行：委托实现模式 `{ self + s }` 仍被标注为当前设计
- §11.6 变更成本表（第 2255-2261 行）：仅列出"补充标注"和"修改委托目标"，遗漏扩展成员函数实现的同步修改

**严重程度**：严重（编码阶段若触发 D30 回退，实施者将面临循环调用导致编译栈溢出或运行时无限递归，排查时间约 1-2 天）

**改进建议**：
1. 在 D30 回退方案中显式增加改造步骤：**扩展成员函数的内部实现从"委托给二元运算符"改为"直接逐分量运算"**（即从 `{ self + s }` 改为 `{ Vec2(this.x + s, this.y + s) }`）。
2. 在 §11.6 变更成本表中增加"扩展成员函数实现改造"行项（约 20 处，预计追加 0.3-0.5 人天），修正总工作量评估。
3. 同步修正 §11.13 三重回退场景设计快照中的算术溢出策略行，增补扩展成员函数实现改造步骤。
4. 在 §4.6 第 1226 行的委托实现模式示例中增加脚注，提示 D30 回退场景下此模式须反转。

### Q2（一般）：扩展成员函数可见性依赖未纳入验收前置条件

**问题描述**：产出在 §2 公共 API 面设计中声明通过 `public import` 重导出 Vec 类型（含扩展成员函数）。但 D37 A20 明确将"扩展成员函数在 `public import` 重导出后保持对外可见"标注为"待验证"（未经验证的编译器行为）。若此行为不成立，则 `v.add(s)`、`v.sub(s)`、`v.mod(s)`、`v.increment()`、`v.decrement()`、`v.bitwiseNot()`、`v.logicalAnd()`、`v.logicalOr()`、`v.equalEpsilon()`、`v.equalExact()` 等约 30+ 个扩展成员函数将在 `import glm.*` 后对下游不可见。

此风险的回退方案（D37 A20 回退列："改为 `public type` 类型别名方案，扩展成员函数改为包级独立函数形式"）涉及显著的公共 API 面变更——部分函数名称将被改变（从 `v.add(s)` 改为 `add(v, s)` 或更冗余的形式）、部分函数将因无法在包级独立函数中保持 `v.method()` 调用模式而丢失方法链能力。

然而，§12.2 验收标准的 20+ 项验收项中，没有任何一项将"D37 A20 验证通过"或"扩展成员函数可见性确认"列为验收前置条件。若编码阶段此项验证失败，验收标准中涉及扩展成员函数的所有验收项（如 `equalEpsilon` 正确性、`equalExact` 正确性、Vec1 广播运算符正确性等）均需大幅修改。

**所在位置**：
- §2 公共 API 面设计（第 210-214 行）：`public import` 策略声明
- §7 D37 A20（第 1785 行）：扩展成员函数可见性标注为"待验证"
- §12.2 验收标准（第 2777-2812 行）：缺少对应验收前置项
- §8.5 排除清单：未将此潜在不可用性记录为排除项

**严重程度**：一般（影响面广但回退方案明确，编码阶段早期即可发现）

**改进建议**：
1. 在 §12.2 验收标准中新增一项**非阻塞验收前置项**：**"扩展成员函数 `public import` 可见性验证"**——要求在阶段 0（§8.3）编码前验证 D37 A20，若验证失败则按回退方案调整公共 API 面设计，并重新评估所有涉及扩展成员函数的验收条件的可达成性。
2. 在 §8.3 迁移顺序的阶段 0 的"退出标准"列中补充此验证项的通过要求。
3. 在 §8.5 排除清单中新增预估条目："若 D37 A20 验证失败，扩展成员函数对下游不可用——回退方案见 D37 A20"，使实施者提前知晓此风险。

### Q3（一般）：`fromBoolVecQ2` 显式标注模板不完整

**问题描述**：§4.8 为回退场景（编译器不支持从返回值类型推断 struct 泛型实参，即验证项㉕失败）提供了显式标注 `<T, Q>` 后的完整函数体代码模板（第 1308-1318 行）。但该模板仅覆盖 Vec2 和 Vec4 两个示例，缺少 Vec1 和 Vec3 的对应模板。虽然实施者可推断 Vec1/Vec3 的模式（分别调整分量数和分量名），但 §4.8 的脚注声明覆盖 "Vec1~Vec4 各一个版本...所有 4 个重载版本"，而实际提供的模板仅为 2/4，与声明不完全一致。

编码实施者在回退场景下需自行推断 Vec1 的单分向量和 Vec3 的三分向量对应模式。对于已处于编译器假设验证失败回退压力下的实施者，每增加一次推断步骤即增加一次笔误风险（如 Vec1 的 `T(1)` vs `T(0)` 单条件表达式、Vec3 的 `z` 分量遗漏等）。

**所在位置**：
- §4.8 `[^compiler-dep]` 编译器推断依赖说明段（第 1307-1319 行）：仅提供 Vec2 和 Vec4 的完整模板
- 同段脚注声明："此修改需覆盖 `fromBoolVecQ2` 的所有 4 个重载版本（Vec1~Vec4）"

**严重程度**：一般（影响局限在回退场景，且模式简单可推断，但增加了不必要的实施风险）

**改进建议**：
1. 在 §4.8 中补充 Vec1 和 Vec3 的完整函数体模板，与 Vec2/Vec4 的模板并列展示。
2. 或者，提供一个统一的泛型函数体模板，标注 `// VecN 模式：{分量名}...` 注释，将四个版本简化为一个通用公式描述，减少模板数量和维护成本。

- `lib.cj` 明确列出 `public import` 内容（§2 第 199-207 行）

**质询点 2：需求响应充分度的验证**
