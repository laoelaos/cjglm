## 需要解决的问题

- **问题描述**：C++ GLM `compute_equal` 行为一致性的系统性误述——设计文档多处将仓颉浮点容差比较路径描述为与 C++ GLM 行为一致或仅在 Inf/NaN 边界偏离，但 C++ GLM 1.0.3 的 `compute_equal` 对所有类型使用精确比较 `a == b`，浮点容差路径是仓颉迁移引入的全量行为偏离
- **所在位置**：§3.5、§4.5、§9.3、§11.5、§11.9 多处散布
- **严重程度**：严重
- **改进建议**：在 §3.5 新增"C++ GLM 行为偏离声明"段；修正 §9.3 异常场景表偏离范围从"Inf/NaN 边界"扩展为"所有浮点比较场景"；修正 §11.9 迁移工作量评估受影响范围；修正 §11.5 标题和范围；修正 §4.5 `equalExact` 说明补充全量行为偏离声明

- **问题描述**：§4.6 标注策略总结段落中 `<<` 广播位运算符计数错误——"约 9 个"应为 6 个（VecN<<Vec1 方向 3 个 + Vec1<<VecN 方向 3 个），"3 个 `<<` 广播"遗漏了 Vec1-op-VecN 方向
- **所在位置**：§4.6 标注策略总结段落（约第 1117 行）
- **严重程度**：一般
- **改进建议**：修正 `<<` 广播位运算符需标注数为 6 个；同步修正总结段落为"6 个 `<<` 广播（VecN<<Vec1 方向 3 个 + Vec1<<VecN 方向 3 个）"

- **问题描述**：§3.5 和 §10 中描述 `ComputeEqual` 容差比较使用 `NumericLimits<T>.epsilon()` 作为绝对容差阈值，但未在显式位置声明此值域局限性——机器 epsilon 仅适用于接近 1.0 量级的浮点值比较，对大值容差过于宽松，对极小值容差过于严格
- **所在位置**：§3.5 `ComputeEqual.callConst` 容差比较路径描述、§10 依赖③验证描述
- **严重程度**：一般
- **改进建议**：在 §3.5 或 §9.3 中补充"机器 epsilon 容差比较的值域局限性"说明段，声明适用范围和局限性，并注明此为仓颉迁移新增行为
