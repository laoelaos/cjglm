# OOD 设计方案审查报告（v5）

## 审查结果

APPROVED

## 逐维度审查

### 1. 类型系统可行性

**[通过]** Qualifier 体系采用 `interface Qualifier` + 空结构体实现，泛型参数 `Q <: Qualifier` 约束与仓颉泛型系统能力匹配。空结构体作为泛型实参传递在仓颉中合法，接口约束机制正确使用。

**[通过]** Vec1~Vec4 四个独立泛型结构体的设计决策合理——仓颉不支持偏特化，无法通过分量数参数化的单一泛型模板实现差异化行为。`struct VecN<T, Q> where Q <: Qualifier` 的泛型约束语法正确。

**[通过]** `extend<T, Q> VecN<T, Q> where Q <: Qualifier` 泛型扩展语法与仓颉扩展文档§2.3形式B一致，类型参数均在使用处出现，满足规则。

**[通过]** 运算符重载不能为泛型（`operator func +<U>(rhs: U)` 不可行）——与仓颉函数文档§8.1"不能为泛型"限制一致。跨类型运算通过构造函数显式转换的替代方案可行。

**[通过]** `type` 别名（`public type Vec2 = Vec2<Float32, PackedHighp>`）与仓颉类型系统文档§5一致——类型别名不创建新类型，仅为替代名称。§3.8 同名自引用风险已识别并纳入验证项㉓，修正策略（添加完整包路径 `glm.detail.Vec2`）在仓颉中可行。

**[通过]** `const init` 与非 `const init` 共存性分析正确——§3.2 的两类构造函数（A: `const init(x: T, y: T)` 和 B: `init<T2, Q2>(v: VecN<T2, Q2>)`）参数列表类型结构显著不同，仓颉重载规则可正确区分。

**[通过的进一步确认]** 复合赋值运算符自动生成——仓颉函数文档§8.5确认"重载二元运算符时自动启用对应复合赋值版本，前提是返回类型与左操作数类型匹配"，与设计§4.3的断言一致。

**[轻微]** Vec1 广播运算符的 extend 块中同时定义 VecN op Vec1（Vec2~Vec4 的 extend 块）和 Vec1 op VecN（Vec1 的 extend 块），两种方向的广播运算符分别在 3 个不同的 extend 块中定义。虽然语法可行，但广播位运算符（`&`/`|`/`^`/`<<`/`>>`）中 `<<` 的 Vec1 广播版本标注 `@OverflowWrapping` 的语义——Vec1.x 作为移位量广播时，`<<` 的 `@OverflowWrapping` 实际控制的是分量值的移位溢出而非移位量本身的溢出——这一点在文档中未显式说明，但对正确性无影响。

### 2. 标准库与生态覆盖

**[通过]** `std.math.abs` 为包级函数（确认自仓颉 `std.math` 文档："`abs(x: T): T` 支持多种数值类型"），设计文档§3.5 和§7 D29 A03 已正确标注为"包级函数（非 `Float32.abs()`/`Float64.abs()` 静态方法）"。代码示例中 `abs(a - b)` 的调用在 `import std.math.{ abs }` 后可用。

**[通过]** `std.math.trunc` 为 `Float64 → Float64` 函数（确认自 `std.math` 文档），用于浮点 mod 的恒等式 `x - y * trunc(x / y)` 中。Float32→Float64 提升后调用 trunc 再转回 Float32 的精度策略合理。

**[通过]** `std.deriving` 的 `@Derive[Hashable]` 支持自动派生（确认自文档），适用于 struct 类型。设计文档已正确识别泛型 struct 上 `@Derive[Hashable]` 对类型参数 `T` 的约束检查策略尚未确认，并纳入验证项㉑。

**[通过]** `std.overflow` 提供五种溢出策略，`WrappingOp<T>` 的 `wrappingAdd` 等与 `@OverflowWrapping` 注解语义等价。标准库覆盖足以支持设计中的溢出策略需求。

**[通过]** `std.unittest` 框架提供 `@Test`/`@TestCase` 注解，`cjpm test` 自动发现测试用例，与§12.3 测试工具设计一致。

**[轻微]** `std.math.abs` 是否为 `const` 函数在标准库文档中未明确声明。设计已识别此风险（验证项㉒），回退方案（内联 `x >= T(0) ? x : -x`）可行。

### 3. 语言特性可行性

**[通过]** 错误处理策略与仓颉能力匹配：整数除零抛 `ArithmeticException`（仓颉运行时默认行为）、浮点除零返回 `±Inf`（IEEE 754 默认行为）、跨类型构造溢出行为由仓颉类型构造函数决定。

**[通过]** `@OverflowWrapping` 注解"只能标记于函数声明上"（确认自注解文档§1.1），作用于函数内的整数运算和整型转换。设计在二元算术运算符上直接标注的策略是唯一可行路径（复合赋值不可显式定义）。

**[通过]** 编译期 `if` 分支抑制：仓颉 const 文档§2 规则8确认 `if`、`is`、`as` 均属于 const 表达式。`const` 函数体内的 `if` 在编译期求值时仅编译被选择分支——此行为已在 const README 中确认（§3.1"const 函数在 const 上下文中编译时求值"），但非选择分支的代码抑制程度（语法检查 vs 完全抑制）需验证，设计已纳入验证项④⑫。

**[通过]** `const init` 在非 const 上下文中可作为运行时构造函数使用（const README §3.2 规则5），设计文档的断言与语言文档一致。`const` 修饰符不构成函数重载区分依据（const README §3.2 规则7），Vec1 构造函数不对称性分析正确。

**[通过]** `extend` 块不支持 `const` 修饰符——确认自扩展文档§4.2，允许的扩展成员修饰符列表为 `static`、`public`、`protected`、`internal`、`private`、`mut`，不含 `const`。设计文档中所有 extend 块成员声明为非 `const` 的策略与语言限制一致。

**[通过]** 运算符重载限制已正确识别：% 不支持 Float（仓颉原生限制）、++/-- 不可重载、&&/||/~ 不可重载、<< 默认 @OverflowThrowing。所有不可重载运算符的替代方案（具名函数）设计合理。

**[通过]** `cjpm` 项目组织方式与设计一致——`package glm.detail` + `package glm` 双层包结构、`cjpm.toml` 的 `[test] src-dir = "tests"` 配置、`cjpm test` 自动发现 `@Test` 标注函数。

**[通过]** `@Derive[Hashable]` 在泛型 struct 上的字段类型约束——派生文档§4确认"参与派生的字段类型必须已实现对应接口"。对于泛型 struct `VecN<T, Q>`，编译器对 `T` 的 Hashable 约束检查策略（延迟检查 vs 定义处检查）尚未确认，设计提供了两种回退方案。

### 4. 设计一致性

**[通过]** 各抽象的职责描述清晰：Qualifier 管理精度/对齐策略、VecN 承载数据与运算、ComputeEqual 管理比较策略、Functor/ComputeVec* 预留后续轮次、shim 替代 C++ 标准库。模块间职责边界明确。

**[通过]** 协作关系形成闭环：VecN → Qualifier（泛型约束）、VecN → ComputeEqual（`==` 委托）、VecN → scalar_vec_ops（标量+向量运算）、lib.cj → glm.detail（公共 API 重导出）、fwd.cj → glm.detail（别名引用）。无缺失环节。

**[通过]** 行为契约描述完整：§4.1 构造函数完整签名清单覆盖 GLSL 5.4.1 规范、§4.3 算术运算含 Vec1 广播模式、§4.4 位运算含 Vec1 广播模式、§4.5 比较运算含容差/精确两条路径、§4.6 标注策略含所有标注计数、§4.8 fromBoolVec 工厂函数签名完整。契约足以指导后续实现。

**[通过]** 模块间依赖方向合理：glm.detail 内部按依赖拓扑排列（setup→qualifier→shim→compute→vectorize→type_vecN），glm 单向依赖 glm.detail。无循环依赖。

**[通过]** 排除清单（§8.5）与迁移成本评估（§11）形成闭环——每个排除项的替代方案和迁移模式均已记录。Bool→Numeric 降级已纳入排除清单（§8.5 新增条目）和迁移成本（§11.7c）。

**[轻微]** §4.3 Vec1 运算符设计专节中"Vec1 在矩阵运算中的角色"段落描述了矩阵迁移的连锁影响，但实际矩阵运算中 Vec1 广播的具体语义（如 `mat * vec1` 中 Vec1 是作为 1×N 行向量还是标量广播）留待矩阵轮次确定。当前描述作为首轮预留基础是合理的。

### 5. 设计质量

**[通过]** 职责划分遵循单一职责原则：Qualifier 仅管理精度标记、VecN 仅承载向量数据与运算、ComputeEqual 仅管理比较策略、scalar_vec_ops.cj 仅管理标量-向量运算辅助函数、type_fromBoolVec.cj 仅管理 Bool→Numeric 转换。

**[通过]** 抽象层次恰当：Qualifer 接口+空结构体实现开销为零、VecN 使用 `var` 成员+`const init` 满足 const 函数需求、Functor/ComputeVec* 仅定义不消费避免过度设计、256个别名由脚本生成避免手工维护负担。

**[通过]** 设计便于后续详细设计和实现：§8.3 迁移顺序提供增量验证节点、§2.1 编码前验证清单覆盖所有编译器行为假设、§4.3 提供完整的运算符语法模板（Vec2 算术运算符示例）、§3.8 提供最小生成脚本示例。

**[通过]** 设计便于单元测试：测试包与源码包一一对应（§12.3）、`internal` 类型测试访问策略提供了4级回退方案、验证层次从编译→构造访问→运算正确性→异常场景逐步深入。

**[通过]** 历史问题闭环状态经抽查确认：P24~P31（v5 新增8项）均有对应的修改措施和问题状态追踪表记录。P14/P15（v4 修复）经独立抽查确认修复有效。Vec 别名同名自引用风险（P27，从第1轮遗留至第3轮）本轮通过验证项㉓+§3.8脚注闭环。
