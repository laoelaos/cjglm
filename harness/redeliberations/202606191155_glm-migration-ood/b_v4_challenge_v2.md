## 逐维度审查

### 1. 证据充分性

**[问题-严重]** 问题 5 声称"C++ GLM 的 `compute_equal` 同样使用 `std::numeric_limits<T>::epsilon()` 作为容差，行为一致"——这一核心事实认定存在严重错误。经查验 C++ GLM 1.0.3 参考实现 `references/glm-1.0.3/glm/glm/detail/compute_vector_relational.hpp`，浮点特化版 `compute_equal<T, true>` 的定义位于第 20-27 行，**整段被注释掉**（`/* ... */`）。实际生效的实现仅为通用版 `compute_equal<T, false>`（第 11-17 行），其 `call` 函数体为 `return a == b;`——对所有类型（含浮点）均使用精确比较，不涉及任何 epsilon 容差。因此：
- (a) C++ GLM 1.0.3 的 `==` 运算符对所有类型均使用精确比较，仓颉设计的容差比较是一项**对 C++ GLM 行为的显式偏离**，而非"与 C++ GLM 行为一致"
- (b) 问题 5 基于"行为一致"的判断将容差路径的值域局限性定性为"一般"严重度，但考虑到容差路径本身即是对 C++ GLM 的行为偏离（C++ GLM 并无此路径），值域局限性实际上是**仓颉迁移引入的新增行为偏离**，对迁移者的指导和影响评估应更为审慎
- (c) 改进建议中"此行为与 C++ GLM 的 `compute_equal` 一致"的措辞需修正，下游消费者应对此偏离有准确认知

此事实错误直接影响问题 5 核心论断的可信度，也影响设计文档 §3.5 和 §4.5 中关于容差比较路径的"与 C++ GLM 一致性"声明的准确性评估。

**[问题-严重]** 问题 6 声称"仅 `<<` 相关广播位运算符（约 3×3=9 个）需标注"——此计数存在事实错误。`<<` 广播位运算符的实际数量为：
- VecN op Vec1 方向（Vec2<<Vec1, Vec3<<Vec1, Vec4<<Vec1）：3 个
- Vec1 op VecN 方向（Vec1<<Vec2, Vec1<<Vec3, Vec1<<Vec4）：3 个
- 合计 6 个

"3×3=9"的乘数来源不明——Vec1 广播运算符中不存在 3×3 的维度组合（广播运算的一侧始终为 Vec1，另一侧为 Vec2/Vec3/Vec4 之一）。同时，设计文档 §4.6 总结段落中标注的"3 个 `<<` 广播"也仅覆盖了 VecN-op-Vec1 方向，遗漏了 Vec1-op-VecN 方向的 3 个。问题 6 的改进建议采纳了错误的 9 个计数，将导致实施者对标注数量产生混淆。

**[通过]** 问题 1（撤回修正）基于实际源码验证 C++ GLM 二元运算符使用 `T` 而非 `U`，证据充分。

**[通过]** 问题 2 关于 `add(s, v)` 等价性声明的判断，基于设计文档 §4.3 标量在左运算段落的实际内容确认，证据充分。

**[通过]** 问题 3 对 §4.3 复合赋值子段同类型/跨类型描述不足的判断，经与设计文档第 896 行正文对比确认，证据充分。

**[通过]** 问题 4 关于 §8.3 阶段 3 "可并行"标注精确性的判断，经与设计文档第 1691 行对比确认，证据充分。

**[通过]** 问题 7 关于 FVec/F32Vec/F64Vec 与 Vec/DVec 别名冗余的判断，经与 §3.8 消费者选择指引表对比确认，证据充分。

### 3. 覆盖完备性

**[问题-严重]** 审查报告未识别设计文档的一个重大事实错误：§3.5 和 §4.5 中关于 C++ GLM `==` 行为的描述声称仓颉容差路径与 C++ GLM `==` "行为一致"（参见设计文档 §9.3 和 §11.5），但 C++ GLM 1.0.3 的 `compute_equal` 浮点特化已被注释掉，`==` 使用精确比较。设计文档多处将容差路径描述为与 C++ GLM 行为一致（如 §9.3 异常场景表、§11.5 Inf==Inf 差异分析中"与 C++ GLM 的 `==` 行为显著偏离"仅限于 Inf/NaN 边界场景的论述），这些描述均建立在 C++ GLM 使用容差比较的错误前提上。仓颉设计的容差比较不仅在 Inf/NaN 边界与 C++ GLM 不一致，而是**在所有浮点比较场景中均与 C++ GLM 不一致**。此事实错误影响设计文档核心行为语义的准确性，属于审查应覆盖但遗漏的严重问题。

**[问题-一般]** 审查报告的历史问题抽查段仅抽查 P14/P15 两项，未覆盖更多历史严重问题（如 P7-CLI_CONTROL 默认值、P17-epsilonOf 整数类型可用性等），覆盖面较窄。考虑到第 2 轮迭代中问题 1 首次发现了 C++ GLM 二元运算符签名的关键事实错误，本轮抽查应覆盖更多涉及 C++ GLM 源码对比验证的条目，以增强可信度。

## 质询要点

### 质询要点 1：问题 5 对 C++ GLM `compute_equal` 行为的核心事实认定错误

- **问题**：问题 5 声称"C++ GLM 的 compute_equal 同样使用 std::numeric_limits<T>::epsilon() 作为容差"，但 C++ GLM 1.0.3 实际源码中 compute_equal 的浮点特化版本已被注释掉，所有类型均使用 `a == b` 精确比较
- **原因**：此错误导致问题 5 将仓颉容差路径定性为"与 C++ GLM 一致的行为偏离"，实际上容差路径本身就是对 C++ GLM 的行为偏离，而非继承自 C++ GLM 的已知限制。定性偏差直接影响下游消费者对容差路径行为差异的认知和迁移决策
- **建议方向**：(a) 修正问题 5 的事实基础——C++ GLM 1.0.3 的 `==` 对所有类型使用精确比较，仓颉的容差比较路径是迁移引入的新增行为偏离；(b) 重新评估严重度——容差路径在所有浮点比较场景（而非仅 Inf/NaN 边界场景）与 C++ GLM 不一致，影响范围更广；(c) 在审查报告中增加独立问题，指出设计文档多处关于"与 C++ GLM 一致"的声明需修正

### 质询要点 2：问题 6 的 `<<` 广播位运算符计数错误

- **问题**：问题 6 声称约 3×3=9 个 `<<` 广播位运算符需标注，实际应为 6 个（VecN<<Vec1 方向 3 个 + Vec1<<VecN 方向 3 个）
- **原因**：计数错误导致改进建议中的数字不可信。同时设计文档自身的"3 个 `<<` 广播"也存在同样遗漏（仅计 VecN-op-Vec1 方向）。问题 6 未能发现设计文档的计数遗漏，反以错误的 9 补之
- **建议方向**：修正 `<<` 广播位运算符需标注数为 6 个（3+3），同时在改进建议中指出设计文档 §4.6 总结中"3 个 `<<` 广播"遗漏了 Vec1-op-VecN 方向的 3 个

### 质询要点 3：重大事实错误遗漏——设计文档关于 C++ GLM `==` 行为一致性的系统性误述

- **问题**：审查报告未识别设计文档中将容差比较路径描述为与 C++ GLM 一致的系统性事实错误。设计文档 §9.3 异常场景表、§11.5 `Inf == Inf` 差异分析、§3.5 容差比较路径说明等多处隐含或显式声称容差路径与 C++ GLM 行为一致或在已知边界场景有偏，而实际上 C++ GLM 1.0.3 的 `==` 对所有类型使用精确比较，容差路径是仓颉迁移引入的全量行为偏离
- **原因**：此遗漏意味着修订后的审查报告仍无法让产出作者准确理解仓颉设计与 C++ GLM 的行为差异全貌——设计文档中关于"C++ GLM `==` 使用精确比较"这一基础事实的描述错误将直接影响迁移成本评估（§11 中基于"与 C++ GLM 行为一致"前提的迁移成本需全部重新评估）和下游消费者的行为预期
- **建议方向**：新增独立问题，指出设计文档关于 C++ GLM `compute_equal` 行为的描述与参考实现不符，要求设计文档：(a) 修正所有隐含/显式声称"C++ GLM `==` 使用容差比较"的段落为"C++ GLM `==` 使用精确比较，仓颉的容差比较路径是新增行为"；(b) 重新评估 §11 中容差路径相关迁移成本（该成本应从"边界场景微调"升级为"全量浮点比较场景迁移"等级）；(c) 重新评估 §9.3 异常场景表和 §4.5 equalEpsilon/equalExact 设计决策的前提一致性
