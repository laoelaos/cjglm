## 发现的问题

### 问题 1（严重）：\`type_vecN.cj\` 缺失 \`std.math.abs\` 依赖标注

**问题描述**：\`equalEpsilon\` 函数定义在各 \`type_vecN.cj\` 的 extend 块中（§4.5），函数体内调用 \`abs(a - b)\`，该 \`abs\` 为 \`std.math.abs\` 包级函数（已在 §3.5/§4.5 代码注释中明确标注）。但 §8.1 文件清单中第 9~12 行（\`type_vec1.cj\`~\`type_vec4.cj\`）的"依赖"列仅列出 \`qualifier.cj + compute_vector_relational.cj + scalar_vec_ops.cj\`，未标注对 \`std.math\` 的依赖。编码阶段的实施者如仅参考 §8.1 文件清单进行 import 管理，将遗漏 \`import std.math.{ abs }\`，导致编译错误。

**所在位置**：§8.1 文件清单（第 1692 行），\`type_vec1.cj\`~\`type_vec4.cj\` 依赖列

**严重程度**：严重——直接导致编译失败，且依赖 column 是实施者的主要参考入口

**改进建议**：在 §8.1 文件清单第 9~12 行的依赖列中补充 \`std.math（abs 包级函数，用于 equalEpsilon）\`；或新增一处集中标注说明，声明 \`type_vecN.cj\` 中 extend 块内的 \`equalEpsilon\` / \`equalExact\` 等函数依赖 \`std.math.abs\`

**所在位置**：文件头部第 4 行"外部再审议迭代轮次：第 4 次"

### 问题 3（一般）：\`@OverflowWrapping\` 标注策略总结段中广播运算符计数存在矛盾

**问题描述**：§4.6 策略总结（第 1134 行）开列"30 个 Vec1 广播算术运算符 + 30 个 Vec1 广播位运算符"作为标注目标，随即括号内修正为"实际需标注约 15 个算术广播 + 6 个 << 广播"。但 §4.3 Vec1 广播运算模式段（第 906 行）明确声明所有算术广播运算符（VecN op Vec1 方向 15 个 + Vec1 op VecN 方向 15 个 = 30 个）均在 extend 块中**直接标注 \`@OverflowWrapping\`**，而非仅标注 15 个。总结段中的"约 15 个算术广播"与 §4.3 的确定性标注声明矛盾。此计数不一致使得编码阶段预估工作量产生偏差。

**所在位置**：§4.6 最后总结段（第 1134 行）

**严重程度**：一般——不影响设计正确性，但混淆工作量估算和标注分布理解

**改进建议**：
- 方案 A：若所有 30 个算术广播运算符均标注，总结段应写为"30 个算术广播运算符"而非"约 15 个"
- 方案 B：若确有仅标注 15 个的理由（如按方向区分标注策略），需在 §4.3 明确声明哪一侧不标注及理由
- 结论：总结段与 §4.3 的标注声明必须一致

### 问题 4（一般）：\`<<\` 溢出策略标注与编译器默认行为的依赖性缺乏针对性验证项

**问题描述**：§4.4（第 1033 行）承认"仓颉中 \`<<\` 的默认溢出策略**属于编译器实现定义行为**"，并说明"编码阶段须通过原型测试确认目标编译器的实际行为"。但当前设计直接标注 \`@OverflowWrapping\` 并作为确定方案进入 §4.6 标注策略总结（计为 1 个运算符 × 4 Vec = 4 处标注）。若编译器默认已 wrapping（高位自动丢弃），则标注冗余但无害；若编译器默认 throwing（抛异常），标注正确需要。风险不在于标注本身，而在于设计文档将其作为"确定性标注策略"描述，未在 §2.1 编码前验证清单中为 \`<<\` 设置独立的编译器行为确认项（当前 §2.1 项⑤仅覆盖复合赋值继承性，不覆盖 \`<<\` 默认策略）。编码阶段若未特意验证 \`<<\` 默认策略，可能遗漏。

**所在位置**：§4.4（第 1033 行）、§4.6 标注策略（第 1123 行）、§2.1 编码前验证清单

**严重程度**：一般——标注本身无害，但缺失独立验证项可能导致编码阶段遗漏确认

**改进建议**：在 §2.1 编码前验证清单中新增一项："确认仓颉 \`<<\` 运算符的默认溢出策略——编写原型测试对 \`UInt8(1) << 10\`（超出位宽）验证是否抛异常；若默认 wrapping 则标注 \`@OverflowWrapping\` 冗余保留；若默认 throwing 则标注必要"

### 问题 5（一般）：附录 A API 覆盖对照表缺少对 \`.hpp\` 文件中声明（而非 \`.inl\` 中实现）的 API 验证

**问题描述**：附录 A 的标题明确限定为 "\`type_vec1.inl\`~\`type_vec4.inl\` 交叉验证"，覆盖构造函数、运算符重载、成员函数等。但 C++ GLM 的公开 API 面部分由 \`.hpp\` 文件声明（如 \`length()\` 静态函数——§3.2 定义于 Vec 结构体体内，在 GLM 中通过 \`type_vecN.hpp\` 的宏 \`GLM_FUNC_QUALIFIER\` 声明）。\`length()\` 等 \`.hpp\` 级 API 未被附录 A 验证覆盖。此外，\`GLM_FUNC_DECL\` 与 \`GLM_FUNC_QUALIFIER\` 条件编译的函数声明模式也未在附录中对照。P37 的修复声称"逐文件逐函数对照验证"，但对照范围限定于 \`.inl\` 文件，未覆盖 \`.hpp\` 声明的公开 API。虽然核心 API 已在 §4.1~§4.8 覆盖，但附录 A 作为"API 覆盖完整性的证据"其验证范围存在 gap。

**所在位置**：附录 A 标题（第 2756 行）及验证范围说明（第 2762 行）

**严重程度**：一般——核心 API 已被正文覆盖，但附录 A 作为 P37 的修复证据其声称的验证完整性低于实际

**改进建议**：将附录 A 标题从 "\`type_vec1.inl\`~\`type_vec4.inl\` 交叉验证"扩展为 "\`type_vec1.hpp/.inl\`~\`type_vec4.hpp/.inl\` 交叉验证"，补充验证各 \`type_vecN.hpp\` 中 \`length()\`、\`GLM_FUNC_DECL\` 标注函数等的覆盖状态。或在验证范围说明中显式声明"\`.hpp\` 级 API 已在 §3.2（length 函数）/§4.1~§4.8（核心 API）中覆盖，附录 A 仅验证 \`.inl\` 中宏展开的具体实现"

### 问题 6（一般）：\`equalEpsilon\` 双路径依赖链跨三处章节，编码阶段联动遗漏风险仍被低估

**问题描述**：§3.5 定义 \`epsilonOf<T>()\` 的生命周期路径 A/B（第 455~462 行），§4.5 \`equalEpsilon\` 实现引用该函数（第 1076 行），§10 依赖④归档联动要求（第 1991 行）。文档虽已在三处新增交叉引用和注释，但在 §12.2 验收标准中仅有一条"equalEpsilon-epsilonOf 回退联动验收项"（第 2688 行）。当编码实施者需要实际修改时，以下场景仍可能遗漏：场景①——路径 A 实施者修改了 \`epsilonOf<T>()\` 签名但未更新 \`equalEpsilon\` 的注释（编译通过，但代码注释与行为不一致）；场景②——路径 B 实施者移除了 \`epsilonOf<T>()\` 但遗漏了 \`ComputeEqual.callConst\` 中的引用（编译失败，可通过编译发现）。风险等级较低但跨章节联动仍需更具体的编码检查机制。

**所在位置**：§3.5 路径 A/B（第 455~462 行）、§4.5 equalEpsilon（第 1076 行）、§12.2 验收标准（第 2688 行）

**严重程度**：一般——已有交叉引用但联动验证粒度不够细

**改进建议**：在 §12.3 决策映射表的第 11 行（epsilonOf 兼容性策略）中增加"修改触发联动检查清单"列，列出路径 A/B 各需同步修改的文件列表（\`shim_limits.cj\`、\`type_vec1~4.cj\` 的 \`equalEpsilon\`、\`compute_vector_relational.cj\` 的 \`ComputeEqual.callConst\`），作为编码阶段实施者的 checklist

### 问题 7（轻微）：\`fromBoolVec\` 对 \`Bool\` 目标类型的语义描述不够精确

**问题描述**：§4.8（第 1197 行）声明当 \`T = Bool\` 时 \`fromBoolVec\` "Bool 值保持不变"（\`true→T(1)=Bool(1)=true\`，\`false→T(0)=Bool(0)=false\`）。但仓颉中 \`Bool(1)\` 返回 \`true\`，\`Bool(0)\` 返回 \`false\`，因此 \`fromBoolVec<Bool, Q>\` 确实是恒等映射。问题在于：此路径虽在语法上合法，但语义上"Bool 类型通过 fromBoolVec 转换为 Bool 类型"是一个无操作，调用方可能会误认为这是一个有实际转换行为的函数。虽然行为正确，但缺少一句说明"当 T = Bool 时，此函数等价于恒等映射，调用方可直接使用原 Vec 变量"

**所在位置**：§4.8（第 1197 行）\`T\` 参数类型范围说明段

**严重程度**：轻微——不影响正确性，仅影响调用方理解和使用文档的效率

**改进建议**：在 \`T\` 参数类型范围说明段末尾补充"当 \`T = Bool\` 时，此函数为恒等映射（\`fromBoolVec<Bool, Q>(boolVec)\` 返回与输入相同值的 Vec），调用方可直接使用原变量"
