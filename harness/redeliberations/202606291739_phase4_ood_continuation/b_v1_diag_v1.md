# 质量审查诊断报告（第 1 轮）

> 审查对象：`a_v1_imported.md`（OOD 设计文档最终版，v16）
> 审查日期：2026-06-29
> 审查视角：需求响应充分度、整体深度和完整性、实际落地可行性

---

## 审查发现

### 事实错误

#### P1（严重）— `shear` 函数参数类型错误且与 GLM 1.0.3 不符

**描述**：§3.2 `ext/matrix_transform.cj` 中 `shear` 函数参数 `p`、`l_x`、`l_y`、`l_z` 仅列举参数名，未指定类型。GLM 1.0.3 实际签名为 `shear(mat4x4 const& m, vec3 const& p, vec2 const& l_x, vec2 const& l_y, vec2 const& l_z)`——`p` 为 `Vec3<T, Q>`，`l_x/l_y/l_z` 各为 `Vec2<T, Q>`。§3.3 `shear_slow` 签名（第 552 行）更将参数错误标注为标量 `T`（`p: T, l_x: T, ...`）。按此签名编码将产生与 GLM 1.0.3 不兼容的 API。

**位置**：§3.2 ext/matrix_transform.cj 第 441 行；§3.3 gtc/matrix_transform.cj _slow 变体第 552 行

**严重程度**：严重

**改进建议**：两处均修正为：
- `shear<T, Q>(m: Mat4x4<T, Q>, p: Vec3<T, Q>, l_x: Vec2<T, Q>, l_y: Vec2<T, Q>, l_z: Vec2<T, Q>): Mat4x4<T, Q>`
- `shear_slow` 同名参数使用相同类型。同时删除第 552 行中"`tan` 计算剪切因子"的描述（已验证 GLM 1.0.3 中 `shear_slow` 未使用 `tan`）。

---

#### P2（严重）— `slerp` 第四参数 `k` 类型未指定

**描述**：§3.2 `ext/quaternion_common.cj` 中 `slerp(x, y, a, k)`（第 506 行）仅描述为"扩展球面插值"，未给出参数 `k` 的类型。GLM 1.0.3 中 `k` 为浮点类型 `T`（与 `a` 类型相同），控制插值曲线的形状。无类型声明则编码人员无法确定 API。

**位置**：§3.2 ext/quaternion_common.cj 第 506 行

**严重程度**：严重

**改进建议**：补充完整签名为 `slerp<T, Q>(x: Quat<T, Q>, y: Quat<T, Q>, a: T, k: T): Quat<T, Q> where T <: FloatingPoint<T>, Q <: Qualifier`。同步更新 §8 lib.cj 导出代码块中 `slerp` 相关的重载计数。

---

#### P3（中等）— `shear_slow` 实现路径描述与 GLM 1.0.3 源码不符

**描述**：§3.3（第 552 行）声称 `shear_slow` 使用 `tan` 计算剪切因子。经查验 GLM 1.0.3 `ext/matrix_transform.inl`，`shear_slow` 内部仅从 `vec2` 参数提取分量构造剪切矩阵，未使用 `tan` 函数。

**位置**：§3.3 gtc/matrix_transform.cj _slow 变体说明第 552 行

**严重程度**：中等

**改进建议**：删除"使用 `tan` 计算剪切因子"的说法，改为与 GLM 一致的"从 `Vec2` 参数提取分量构造剪切矩阵"的描述。

---

### 关键遗漏

#### P4（严重）— `stdmath_shim.cj` 未给出完整包装函数清单

**描述**：`stdmath_shim.cj` 是本阶段所有 core 函数库（`exponential.cj`、`trigonometric.cj`、`geometric.cj`、`common.cj`）的底层数学运算基础设施，设计仅描述了模式（`sqrtT`/`powT` 等）但未列出所有需要的包装函数。编码人员无法据此实现。需要列出的函数至少包括：`sqrtT`、`expT`、`logT`、`log2T`、`exp2T`、`powT`、`sinT`、`cosT`、`tanT`、`asinT`、`acosT`、`atanT`、`atan2T`、`sinhT`、`coshT`、`tanhT`、`asinhT`、`acoshT`、`atanhT`、`roundT`、`floorT`（若有用）、`ceilT`（若有用）。

**位置**：§1.4 第 50 行；§2 模块划分第 99-102 行；§3.1 多处引用

**严重程度**：严重

**改进建议**：在 §1.4 或 §2 的 `stdmath_shim.cj` 描述中补充完整包装函数列表（至少 20 个），标注每个包装函数的签名和委托路径。或用代码块形式给出完整 `stdmath_shim.cj` 的伪实现。

---

#### P5（中等）— 阶段四未包含任何测试/验证策略

**描述**：本文档经过 16 轮迭代，内容涵盖架构、模块划分、接口设计、数据流、错误处理、并发设计和实施计划，但完全没有涉及功能验证策略。对于 200+ 个数学函数（包含 NaN/Inf/零值/越界等边缘场景），缺乏验证策略意味着编码完成后无法系统性地确认实现正确性。缺乏对以下问题的回答：测试按模块独立组织还是统一组织？是否与 GLM 1.0.3 交叉验证？边缘场景测试覆盖率要求？已有阶段（如阶段三）的测试模式是否沿用？

**位置**：全文

**严重程度**：中等

**改进建议**：参考已有阶段的测试模式，在 §8 实施计划中为每批增加"测试"子项，或在新增 §10 章节中描述测试策略。至少应包括：每个函数族的单元测试组织方式、与 GLM 1.0.3 参考实现的交叉验证方法、已知行为差异的测试标注。

---

### 深度与完整性不足

#### P6（中等）— `round.cj` NaN/Inf/零值边界行为未定义

**描述**：§3.3 `gtc/round.cj` 定义了 6 个函数的命名和约束，但未讨论任何函数在 NaN/Inf/零值输入时的行为。例如 `floorPowerOfTwo(+Inf)` 应返回什么、`ceilPowerOfTwo(NaN)` 的行为、`roundPowerOfTwo(0)` 的输出等均未定义。虽在其他验证报告中标记为"编码阶段需关注"，但 OOD 设计文档中对本阶段实现的新函数缺少边界行为定义是完整性缺陷。

**位置**：§3.3 gtc/round.cj 第 740-751 行

**严重程度**：中等

**改进建议**：参照 GLM 1.0.3 `gtc/round.inl` 源码，为全部 6 个 round 函数补充 NaN/Inf/零值/负数的输入行为定义，或补充设计决策记录选择理由（如接受 IEEE 754 自然传播，则需在 §5 错误表中新增一行）。

---

#### P7（中等）— `noise.cj` 实现复杂度被明显低估

**描述**：§3.3 将噪声算法迁移评估为"约 250~300 行有效代码"（第 678 行），但 GLM 1.0.3 的 `gtc/noise.inl` 约 807 行 C++ 代码，`_noise.hpp` 辅助函数约 81 行。仓颉实现即使采用数学公式直译，考虑到为绕过维度泛型限制而拆分为 12 个独立函数（`perlin1D~4D` + `simplex1D~4D` + `perlin1D~4D` 周期版），每函数的索引计算和梯度计算逻辑均需逐行翻译，实际代码量至少为 500-800 行而非 250-300 行。低估复杂度可能导致编码阶段进度偏差。

**位置**：§3.3 gtc/noise.cj 工作量评估第 678 行

**严重程度**：中等

**改进建议**：重新评估代码量，建议以"500-800 行"作为估算基准，并标注噪声模块为阶段四中最复杂的算法迁移模块，建议安排充分的编码和 code review 时间。

---

#### P8（中等）— `lib.cj` 第 23 行修改缺少精确描述

**描述**：§8 和 §9.4 多次引用"修改现有 lib.cj 第 23 行"，但未提供该行的当前内容或精确的修改后代码块。编码人员在没有当前 `lib.cj` 上下文的情况下需要自行定位并理解修改意图。虽然 D30 给出了设计理由，但缺少可直接参照的精确代码变更。

**位置**：§8 lib.cj 更新第 956 行；§9.4 第 1051 行

**严重程度**：中等

**改进建议**：在 §8 或 D30 中补充 lib.cj 第 23 行的当前内容（引用形式）和修改后的完整行，格式如下：
```cangjie
// 当前第 23 行（阶段三）：public import glm.gtc.{...}
// 修改后（阶段四）：从 gtc import 中删除 translate, rotate, scale, shear, lookAt, lookAtRH, lookAtLH
```

---

#### P9（轻微）— D16 向后不兼容影响的波及范围未分析

**描述**：D16 声明 `geometric.cj` 约束从 `Number<T>`（阶段三）收紧为 `FloatingPoint<T>`（阶段四），标注为向后不兼容变更。但设计未分析此变更是否影响阶段一~三中的已有调用代码（如是否已有整数类型的 `normalize`/`length` 调用）。若不存在受影响代码，应注明"经验证无影响"；若存在，应为受影响部分给出迁移指引。

**位置**：§7 D16 第 890 行

**严重程度**：轻微

**改进建议**：在 D16 或 §9 中补充对阶段一~三已有代码的搜索结论（"经过搜索，阶段一~三中无 `Number<T>` 实例化 geometric.cj 函数的调用点"或列出受影响文件并提供迁移方案）。

## 总体评价

产出文档质量较高，经过 16 轮迭代后在架构一致性、阶段衔接、仓颉适配等方面已相当成熟。三个独立验证子 agent 的审查覆盖了大多数技术维度。本轮审查发现的主要问题集中在：

1. **残留事实错误**（P1-P3）：`shear`/`shear_slow` 参数类型和实现描述仍与 GLM 不一致，`slerp k` 类型未指定——这些是直接影响编码正确性的错误
2. **基础设施完整性**（P4）：`stdmath_shim.cj` 作为所有 core 函数库的底层依赖，其函数清单缺失使得编码无法启动
3. **验证策略缺失**（P5）：200+ 数学函数的实现缺乏验证策略指导
4. **边界行为不完整**（P6）：`round.cj` 的 NaN/Inf/零值行为未定义
5. **工作评估偏差**（P7）：`noise.cj` 代码量明显低估，可能影响项目进度规划

以上问题中 P1-P4 属于直接影响编码启动和正确性的障碍，建议在启动编码前优先修复。
