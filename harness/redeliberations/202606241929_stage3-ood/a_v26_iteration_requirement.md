## 当前审查结果

### 问题 1（中度）
§1 API 可用率声明声称「约 77 个函数（约 47%）为真完整实现」，但 §3.13.2 审计结论明确「本阶段可调用的真完整实现函数（无 stub 依赖）共 18 个函数」。77 与 18 的差值（59 个）主要由 gtc/constants 28 个常量函数、ext/vector_relational epsilon 16 个重载、以及构造函数/运算符重载等高度同构的变体组成。首次阅读者在 §1 看到"47% 真完整实现"时，容易形成"阶段三可交付近半数 API 功能"的印象，而实际独立可用的函数范式仅 18 个。这种口径差异属于沟通透明度问题。

改进建议：在 §1 API 可用率声明表格下方补充注记，明确说明 77 个 ✅ 包含 28 个 gtc/constants 常量函数 + 16 个 vector_relational epsilon 重载 + 构造函数/运算符重载等变体，实际独立函数签名数约 18 个。或将 §1 表格拆分为两行——「独立函数范式」与「含重载变体展开」——分别给出计数。

### 问题 2（中度）
`normalize` 函数的整数 T 实例化时的边界行为在 §3.7 和 §5.3 中存在契约缺失。§3.7 明确 `normalize` 使用 `where T <: Number<T>` 约束（未收紧至 `FloatingPoint<T>`），允许整数四元数调用 `normalize`。但 §5.3 边界条件表对 `normalize` 的两行条目均假设浮点行为（Inf/NaN 传播）。对于整数四元数，非零分支采用整型除法（截断），可能导致非单位四元数结果，与浮点行为有本质差异。

改进建议：在 §5.3 边界条件表新增「整数 T 实例化」行，明确 `normalize(q)` 在整数四元数上的行为契约；或收紧 `normalize` 的 T 约束为 `FloatingPoint<T>`，在编译期拒绝整数 T 实例化。

### 问题 3（轻度）
§3.4 运算符体系表中 `Quat/T` 二元除法运算符的整数除零行为未明确，与 §3.11 `inverse` 的整数除零行为描述（§5.3 第 1060 行「触发仓颉 ArithmeticException」）形成不对称。`@OverflowWrapping` 注解覆盖的是整数溢出语义，不覆盖除零语义。

改进建议：在 §3.4 `Quat/T` 行的「备注」列补充「整数 T 除零时触发仓颉 ArithmeticException，浮点 T 除零时产生 Inf/NaN 分量」。或新增 §5.3 边界条件表条目统一声明所有除法运算符的整数除零行为。

### 问题 4（轻度）
`slerp` 4 参数版本的 `k: Int64` 类型决策与 GLM 原始模式之间的潜在兼容性问题未在 §9 差异声明或 §7 D22 中记录。该决策意味着阶段四完整实现时，如果下游用户需要 `Int32` 或 `Int16` 类型的 `k`，当前的 `Int64` 签名将强制要求调用方做类型转换。

改进建议：在 §9 差异声明表中新增条目「`slerp` 4 参数版本 `k` 类型固定为 `Int64`（GLM 原始版本使用独立泛型整型参数 `S`）」，注明该偏差的设计理由和阶段四可选的泛化路径。

### 问题 5（轻度）
§8 编码启动前验证项 29（glm.detail 导入 glm.ext 验证）的描述与 §3.4 最新实施的内联 conjugate/dot 路径存在脱节。验证项 29 描述为「先引入跨包 import 再验证报错」，但当前设计已经没有跨包 import 可引入来触发循环依赖错误。

改进建议：将验证项 29 的验证方式修订为「确认 type_quat.cj 无 import glm.ext.* 声明 + cjpm build 通过」。
