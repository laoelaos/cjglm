## 当前审查结果

### 问题 1（严重）— 需求响应充分度：缺少对 GLM 1.0.3 stage 3 API 范围的系统性覆盖声明确认

设计文档虽然提供了 §10 覆盖矩阵（按 GLM 头文件分组），但缺少一份从「GLM 1.0.3 stage 3 完整 API 范围 → 本设计覆盖状态」的逐项映射表。当前 §10 覆盖矩阵仅列出设计选择覆盖的函数，**未声明 GLM 1.0.3 stage 3「到底包含哪些头文件的哪些函数」这一基准**。下游编码者无法从文档中验证是否所有 GLM 1.0.3 stage 3 范围内函数均已覆盖（无遗漏）、哪些函数被显式推迟至 stage 4 且推迟理由充分、设计范围是否对标了正确的 GLM 版本基准（GLM 1.0.3）。文档在 §3.16 中承认了与路线图之间的三处不一致（D35-D37），但从未给出类似「GLM 1.0.3 stage 3 API 完整清单」的基准定义。

**所在位置**：§10 覆盖矩阵全文、§3.16 需求对齐说明段
**改进建议**：在 §10 覆盖矩阵前方新增「GLM 1.0.3 stage 3 API 基准声明」段——列出 GLM 1.0.3 中与 stage 3 直接相关的所有头文件及每个头文件的函数总数，然后与本设计的覆盖状态做逐项对比。对确认为「本阶段不覆盖」的函数，标注推迟至 stage 4 的理由。该基准声明应明确版本号（GLM 1.0.3）以避免版本偏差。

### 问题 2（严重）— §8.3 与 §11.5 权威声明矛盾历经 8 个迭代版本未闭环修复

b_v20_diag_v1 已识别此问题（问题 3），v20 修订说明声称已修复，但实际仅修正了 §11.5 首段（第 1633 行，从「单一权威来源」降级为「主要参考来源」），§8.3 D 节第 1362 行仍原样保留「`§11.5 函数可用性对照表` 是阶段三验证的**权威基准**」。两处对同一表格的定位描述自相矛盾。该问题自 v12 首次引入以来，历经 v13~v20 共 8 个迭代版本，虽两次被检出但始终未彻底修复。

**所在位置**：§8.3 D 节第 1362 行（「权威基准」）vs §11.5 首段第 1633 行（「主要参考来源」）
**改进建议**：二选一——(a) 将 §8.3 第 1362 行同步降级为「主要参考基准」，与 §11.5 措辞一致，并注明补齐后升级计划；或 (b) 若 v20 已补齐 5 个缺失条目后认为完整性已达权威标准，则 §8.3 和 §11.5 措辞应统一升级，但需在 §11.5 删除「尚不具备完整性要求」的自我降级声明。推荐方案 (a)。

### 问题 3（严重）— gen_fwd_aliases.py 验证命令模板存在事实错误，直接导致验证步骤不可执行

§2 实施路径段（第 213-225 行）提供了 4 条 grep 验证命令模板，但其中至少两条命令的匹配模式与生成代码的实际格式不匹配：(a) `grep -c "type Quat = Quat<" cjglm/src/fwd.cj` 要求等号右侧紧跟 `Quat<`，但实际为 `detail.Quat<`，匹配数为 0，而非文档预期的 9；(b) 第 225 行 `grep "type.*Quat.*Vec" cjglm/src/fwd.cj` 预期「无匹配」，但该命令同时也检查了「不包含 Vec 关键词」——即使代码生成了错误的 `Quat1`/`Quat2`/`Quat3`（不含 "Vec"），该 grep 也无法检出。该验证命令自 v13 引入以来，历经至少 7 个迭代版本未被检出。

**所在位置**：§2 lib.cj/fwd.cj 段落「实施路径」验证命令模板（第 213-225 行）
**改进建议**：将 (a) 修正为 `grep -c "detail.Quat<" cjglm/src/fwd.cj`（匹配 12 行，包含 3 个 FQuat 精度冗余）或精确匹配 `grep -c '^public type.*= detail.Quat<' cjglm/src/fwd.cj`；将 (b) 修正为直接检查不应生成的模式——`grep -c "Quat1\|Quat2\|Quat3" cjglm/src/fwd.cj`（预期 0）；同时补充一条可检出 FQuat 精度前缀冗余的命令：`grep "HighpFQuat\|MediumpFQuat\|LowpFQuat" cjglm/src/fwd.cj`，预期输出为 0。

### 问题 4（严重）— 落地可实施性：关键信息碎片化严重，下游实施需要考古式跨节阅读

设计文档经过 20 轮迭代，在正文中散布了大量 `v{n} 修订`/`v{n} 关键修订`/`v{n} 新增` 标注（全文粗略计数约 150+ 处）。一个典型函数的完整状态信息碎片化分布在多个章节。以 `normalize` 为例，实施者需交叉阅读 §3.7 函数描述、§1 系统性设计约束、§5.3 边界条件表、§3.13.2 审计表、§11.5 可用性对照表、§8 验证项 25 这 6 个相距较远的位置。虽然文档在 §3.13.2 审计表和 §5.3 边界条件表中引入了交叉引用机制，但 §3.7 `normalize` 函数描述本身并未提供到其他 5 个位置的集中导航。

**所在位置**：全文散布的修订标注（§1/§3.7/§3.9/§3.11 等）均可能遗漏被修改函数的集中参考清单
**改进建议**：对 §3.7~§3.15 中每个「完整实现」函数，在其函数描述末尾新增「**集中参考清单**」子节，列出该函数涉及的所有跨节位置（§1 约束、§5.3 边界条件、§3.13.2 审计表行、§11.5 可用性行、§8 验证项号）。建议优先为 P0 约束相关的函数（`normalize`/`axis`/`mat3Cast`/`mat4Cast`）补齐。

### 问题 5（一般）— 需求响应充分度：编码启动前验证项缺少独立验证报告模板，baseline 假设管理纪律缺失

### 问题 6（一般）— 落地可实施性：§3.3 item 7 fromMat4 手动提取策略依赖未验证的字段名假设

§3.3 item 7 的 `fromMat4` 实现策略描述为「直接读取 `m.c0`/`m.c1`/`m.c2` 三列构建 Mat3x3」。该假设在验证项 27 中被承认「需验证 Mat4x4 列字段名是否为 `c0`/`c1`/`c2`/`c3`」。若阶段二 Mat4x4 的列字段名实际为 `col0`/`col1`/`col2`/`col3`（或其他命名），则 §3.3 item 7 的实现描述将直接导致编译错误。设计文档将修复责任标注为「编码者——验证后按实际字段名修改实现注释」，但设计文档本身应保证描述基于已验证的事实。
