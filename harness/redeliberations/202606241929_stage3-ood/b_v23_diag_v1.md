## 问题 1（严重）：设计建立于未经实证的两项 P0 核心假设之上，经过 23 次迭代仍未验证

**问题描述**：全文设计系统性依赖两项未经编译验证的 P0 核心假设——(1) `T(Float64(n))` 语法对所有受影响的类型变体（`Float32`/`Float64`/`Int8~Int64`/`UInt8~UInt64`）编译可行；(2) `FloatingPoint<T>` 接口实际提供 `getMinDenormal()`/`getInf()`/`getNaN()`/`isInf()`/`isNaN()` 等 9 个方法。这两项假设自迭代第 10 轮首次检出以来，历经 v10→v23 共 13 个迭代版本、至少 6 轮审查检出，始终停留在「编码启动前验证项」状态，从未被实际编译验证。若任一假设被证伪，影响范围覆盖 normalize/axis/mat3Cast/mat4Cast/Quat×Vec3/Quat×Vec4/pow/log/quatCast 共 10+ 个函数的核心实现路径，导致整份设计文档的大面积失效。该问题非「技术可行性」维度——内部审议无法验证未经实际编译的假设，属深度和完整性维度的根本性缺陷。

**所在位置**：§1「系统性设计约束」「回退方案」子节；§3.7 normalize；§3.9 axis；§3.2.1 mat3Cast/mat4Cast；§3.4 Quat×Vec3/Vec4；§3.10 pow/log；§8 编码启动前验证项 20/25（P0）；§1.3 验证结果记录表模板

**严重程度**：严重

**改进建议**：在设计文档被交付编码阶段之前，必须完成两项 P0 验证项（20、25）的实际编译测试，并在文档中记录验证结果与编译器版本。若验证失败，§1 回退方案决策树中的路径 A/B/C 必须升级为正式设计方案，当前以 `T(Float64(n))` 为默认路径的所有函数描述需同步修改。在此之前，文档不应声称「可指导编码实现」——其核心假设未经实证。

## 问题 2（严重）：数据冗余且不一致的覆盖矩阵降低了文档作为实施依据的可信度

**问题描述**：§9a「GLM 1.0.3 Stage 3 API 基准声明」与 §10「GLM 1.0.3 API 阶段覆盖矩阵」是两份表述同一组数据但结构不同的表格。两表的函数总数/覆盖状态/合计行数据虽在 v19-v23 迭代中趋于一致，但文档自身在 §8.3 I 节 O-02 承认「§9a 与 §10 两张覆盖矩阵存在数据冗余，需合并为单一来源」，并计划推迟至 v24 修复。这意味着当前 v23 版本在实际交付时仍然保有未同步的数据双重来源——下游编码者以哪张表为准？两表若后续修订不同步（如阶段四扩展时只更新了其中一张）将产生无法自动检测的不一致。

**所在位置**：§9a GLM 1.0.3 Stage 3 API 基准声明（行 1535-1573）；§10 GLM 1.0.3 API 阶段覆盖矩阵（行 1577-1716）；§8.3 I 节 O-02

**严重程度**：严重

**改进建议**：在 v23 阶段即完成合并，而非推迟至 v24。v23 作为「可指导编码」的版本不应携带明确的「计划在下一版本修复」的数据冲突。建议将 §9a 的基准计数列直接融入 §10 表头，删除 §9a 独立表格，仅保留「推迟至 Stage 4 的函数及理由」清单（该清单在 §10 中无等价呈现）。

## 问题 3（一般）：文献体量过大且历史痕迹过重，严重损害实际落地可操作性

**问题描述**：全文 2247 行，其中 §修订说明(v12)~§修订说明(v23) 共 12 轮修订说明占据了约 800 行（~36%）。每轮修订说明以表格形式逐条记录数百项问题与修改措施，本质上是一份「设计迭代日志」而非「当前设计文档」。下游编码者若要确定一个函数的当前设计意图（如 `normalize` 的边界行为），必须在 §3.7 正文 + §5.3 边界条件表 + §3.13.2 审计表 + §11.5 可用性表 + §11.7 索引中交叉查阅，同时还需防范修订说明中已废弃的历史描述（如 v11 修订前的「仅支持 Float64」错误描述）的误导。这种「考古式阅读」模式使得设计文档无法作为快速参考指南使用。

**所在位置**：全文结构性问题——§修订说明(v12)~§修订说明(v23) 共 12 个章节（行 1961-2247）；§3.7/§3.9/§3.11 等正文中的版本标记（如「v11 修订」「v14 关键修订」）

**严重程度**：一般

**改进建议**：(1) 将全部修订说明剥离至独立历史文件（如 `a_v23_revision_history.md`），正文中完全不保留修订说明章节；(2) 正文中所有版本前缀（「v11 修订」「v14 关键修订」等）替换为最终结论——下游不需要知道某个结论是在哪个版本修订的，只需要知道当前结论是什么；(3) §11.7 全函数索引应提升为核心导航工具，使下游可通过单一入口获取每个函数的全部设计信息。

## 问题 4（一般）：阶段三交付范围与阶段四的职能边界定义模糊，「53% 推迟」策略未评估对需求响应充分度的影响

**问题描述**：GLM 1.0.3 Stage 3 相关 API 共 165 个函数（§9a 声明），本设计仅完成 77 个 ✅（47%），88 个 ⚠️/❌ 推迟至阶段四（53%）。推迟理由为「依赖 trigonometric.cj/common.cj/geometric.cj 等 stub 函数库」，但阶段四本身也需实现这些函数库——而阶段四的实现范围、时间计划、资源投入在当前文档中完全没有定义。这意味着阶段三交付后，下游消费者可使用的四元数 API 不到一半，而剩下的 53% 何时可用完全悬而未决。用户需求「迁移阶段三」是否接受超过半数 API 不可用？设计文档未对此做任何说明或取得共识。这不是技术可行性问题（内部审议已验证是合理的）——这是需求响应充分度的问题：设计选择是否与用户期望对齐。

**所在位置**：§9a 基准声明表（行 1556 ✅ vs ❌ 合计）；§3.16 需求对齐说明（「阶段三→阶段四演进指南」承诺签名不变但不承诺交付时间）

**严重程度**：一般

**改进建议**：(1) 在 §3.16 或新增章节中明确阶段四的时间预期（预计人月/人日）和前提条件（哪些 stub 必须就绪），并标注为「设计假设——需与项目方确认」；(2) 对 88 个推迟函数按是否存在替代路径分类——哪些函数可通过纯算术或内联方式提前实现以避免推迟；(3) 如果 53% 推迟是项目方已认可的决策，需在文档中记录此共识及理由。

## 问题 5（一般）：演化承诺尚缺与下游消费者的接口契约

**问题描述**：§3.16「阶段三→阶段四演进指南」宣称「所有 stub 函数签名在阶段四实现时保持不变」，但缺乏与下游消费者的接口契约机制。具体地：(1) 没有版本兼容性声明（semver 策略）——阶段三交付的 API 是否构成公共契约？(2) 没有废弃（deprecation）策略——若阶段四发现某 stub 函数需要新增重载，现有签名如何处理？(3) 签名冻结例外审批流程（§3.16 新增）的评审标准和时长未定义——一个「需经设计评审」的审批流程在实践中可能耗时数天到数周不等，而下游无法预判。

**所在位置**：§3.16「阶段三→阶段四演进指南」「签名冻结例外审批流程」节（行 931-934）；§8.3 E 节签名冻结核验项

**严重程度**：一般

**改进建议**：(1) 明确阶段三 API 的版本化策略——建议声明为 `0.x`（不稳定）或 `1.0-stage3`（准稳定）以管理下游期望；(2) 为签名冻结例外审批流程补充时间约束——「设计评审应在 3 个工作日内完成」；(3) 若某签名必须修改，定义迁移路径——旧签名标记为 `@Deprecated` 并保留至少一个次要版本。

## 问题 6（一般）：函数可用状态的「单一权威来源」声明仍存在自指矛盾

**问题描述**：§11.5 在 v23 中从「主要参考来源」升级回「单一权威来源」，理由是 v22 已补齐全部 5 个缺失 API 条目。但 §8.3 D 节同时引用 §11.5 为基准，形成线性引用链（§8.3 D → §11.5 → 正文）。这使得 §11.5 既是被引用的「权威来源」又自身需要外部验证其完整性。当 §11.5 出现遗漏（如 v20 缺失 5 个条目），该遗漏不会被自动发现——因为依赖 §11.5 的 §8.3 D 节只能引用已列出的条目，无法检出「未被列出的条目」。这是一个系统性缺陷——单一来源的假设不成立时，所有依赖该来源的位置同时失效。

**所在位置**：§11.5 表头声明（行 1772-1774）；§8.3 D 节（行 1441-1443）；§3.13.2 审计结论（行 792-796）；§8 产出物清单（行 1106-1108）

**严重程度**：一般

**改进建议**：将 §11.5 的定位声明从「单一权威来源」改为「主要参考来源，§10 覆盖矩阵作为补充」，与 §9a 合并计划一致。在 §8.3 H 节新增「§11.5 完整性自动化核验」核验项——通过生成脚本将 §10 的函数清单与 §11.5 的行数逐项比对，确保无遗漏。

## 问题 7（轻微）：编码启动前验证的 P0 项缺乏验证通过的退出标准

**问题描述**：§8「验证结果记录表模板」要求 3 项 P0 验证（20、25、29）全部标注「通过」后方可启动 stage 3 编码。但文档未定义「通过」的具体标准——对于验证项 25（`T(Float64(n))` 语法），是所有 8 种类型变体全部编译通过才算通过？还是部分通过即可？验证项 20（`FloatingPoint<T>` 接口方法），是 9 个方法全部存在才算通过？还是核心的 3-4 个即可？如果不明确退出标准，验证结果的可重复性无法保证——不同验证人可能得出不同结论。

**所在位置**：§8 验证结果记录表模板 P0 通过条件行（行 1395）；§8 高优先级编译前验证项子节（行 1351-1359）

**严重程度**：轻微

**改进建议**：对每项 P0 验证项补充显式的「通过条件」——例如「验证项 25 通过条件：`Float32(Float64(0))`/`Float32(Float64(1))`/`Float32(Float64(2))`/`Float64(Float64(0))`/`Float64(Float64(1))`/`Int32(Float64(0))`/`Int64(Float64(1))` 共 7 个表达式全部编译通过」。
