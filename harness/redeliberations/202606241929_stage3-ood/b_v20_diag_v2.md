**审查范围**：需求响应充分度、事实正确性/逻辑一致性、深度与完整性/落地可实施性
**审查定位**：第 2 轮审查，处理第 1 轮质询意见（覆盖完备性不足），新增侧重需求响应充分度与落地可实施性评估

## 问题 1（严重）— 需求响应充分度：缺少对 GLM 1.0.3 stage 3 API 范围的系统性覆盖声明确认

**问题描述**：用户需求为「将 references/glm-1.0.3 阶段三用仓颉重新实现」。设计文档虽然提供了 §10 覆盖矩阵（按 GLM 头文件分组），但缺少一份从「GLM 1.0.3 stage 3 完整 API 范围 → 本设计覆盖状态」的逐项映射表。当前 §10 覆盖矩阵仅列出设计选择覆盖的函数，**未声明 GLM 1.0.3 stage 3「到底包含哪些头文件的哪些函数」这一基准**。下游编码者无法从文档中验证：
- 是否所有 GLM 1.0.3 stage 3 范围内函数均已覆盖（无遗漏）
- 哪些函数被显式推迟至 stage 4 且推迟理由充分
- 设计范围是否对标了正确的 GLM 版本基准（GLM 1.0.3）

文档在 §3.16 中承认了与路线图之间的三处不一致（D35-D37），但从未给出类似「GLM 1.0.3 stage 3 API 完整清单」的基准定义。当未来版本迭代时，缺乏基准会导致无法判断「设计覆盖了百分之多少的 GLM stage 3」。

**所在位置**：§10 覆盖矩阵全文、§3.16 需求对齐说明段

**严重程度**：严重

**改进建议**：在 §10 覆盖矩阵前方新增「GLM 1.0.3 stage 3 API 基准声明」段——列出 GLM 1.0.3 中与 stage 3 直接相关的所有头文件及每个头文件的函数总数（如 `detail/type_quat.hpp: XX 个函数`），然后与本设计的覆盖状态做逐项对比。对确认为「本阶段不覆盖」的函数，标注推迟至 stage 4 的理由。该基准声明应明确版本号（GLM 1.0.3）以避免版本偏差。

## 问题 2（严重）— §8.3 与 §11.5 权威声明矛盾历经 8 个迭代版本未闭环修复

**问题描述**：b_v20_diag_v1 已识别此问题（问题 3），v20 修订说明声称已修复，但实际仅修正了 §11.5 首段（第 1633 行，从「单一权威来源」降级为「主要参考来源」），§8.3 D 节第 1362 行仍原样保留「`§11.5 函数可用性对照表` 是阶段三验证的**权威基准**」。两处对同一表格的定位描述自相矛盾：下游编码者在 §8.3 读到「权威基准」，以为 §11.5 是最终仲裁依据；切换到 §11.5 又读到「承认当前版本因缺失部分 API 条目尚不具备完整性要求」——产生信任困惑。该问题自 v12 首次引入以来，历经 v13~v20 共 8 个迭代版本，虽两次被检出（b_v19_diag 和 b_v20_diag_v1）但始终未彻底修复。

**所在位置**：§8.3 D 节第 1362 行（「权威基准」）vs §11.5 首段第 1633 行（「主要参考来源」）

**严重程度**：严重

**改进建议**：二选一——(a) 将 §8.3 第 1362 行同步降级为「主要参考基准」，与 §11.5 措辞一致，并注明补齐后升级计划；或 (b) 若 v20 已补齐 5 个缺失条目后认为完整性已达权威标准，则 §8.3 和 §11.5 措辞应统一升级，但需在 §11.5 删除「尚不具备完整性要求」的自我降级声明。推荐方案 (a)，因 §11.5 的自我降级声明应视为当前版本的诚实表述，§8.3 不应与它矛盾。

## 问题 3（严重）— gen_fwd_aliases.py 验证命令模板存在事实错误，直接导致验证步骤不可执行

**问题描述**：§2 实施路径段（第 213-225 行）提供了 4 条 grep 验证命令模板，但其中至少两条命令的匹配模式与生成代码的实际格式不匹配，按字面执行将输出错误结果：

(a) `grep -c "type Quat = Quat<" cjglm/src/fwd.cj` — 生成脚本模板为 `public type {alias_name} = detail.Quat<{family_type}, {prec_type}>`，实际输出行如 `public type Quat = detail.Quat<Float32, PackedHighp>`。该 grep 模式要求等号右侧紧跟 `Quat<`，但实际为 `detail.Quat<`，因此匹配数为 **0**，而非文档预期的 **9**。该验证命令自 v13 引入以来，历经至少 7 个迭代版本未被检出。

(b) 第 225 行 `grep "type.*Quat.*Vec" cjglm/src/fwd.cj` 预期「无匹配」，用于验证不生成 `Quat1/Quat2/Quat3`。但该命令同时也检查了「不包含 Vec 关键词」——即使代码生成了错误的 `Quat1`/`Quat2`/`Quat3`（不含 "Vec"），该 grep 也无法检出（因无 "Vec" 关键词），所以该验证形同虚设。

**所在位置**：§2 lib.cj/fwd.cj 段落「实施路径」验证命令模板（第 213-225 行）

**严重程度**：严重

**改进建议**：将 (a) 修正为 `grep -c "detail.Quat<" cjglm/src/fwd.cj`（匹配 12 行，包含 3 个 FQuat 精度冗余）或精确匹配 `grep -c '^public type.*= detail.Quat<' cjglm/src/fwd.cj`；将 (b) 修正为直接检查不应生成的模式——`grep -c "Quat1\|Quat2\|Quat3" cjglm/src/fwd.cj`（预期 0）；同时补充一条可检出 FQuat 精度前缀冗余的命令：`grep "HighpFQuat\|MediumpFQuat\|LowpFQuat" cjglm/src/fwd.cj`，预期输出为 0（若设计目标为 9 个别名且冗余不应存在）。

## 问题 4（严重）— 落地可实施性：关键信息碎片化严重，下游实施需要考古式跨节阅读

**问题描述**：设计文档经过 20 轮迭代，在正文中散布了大量 `v{n} 修订`/`v{n} 关键修订`/`v{n} 新增` 标注（全文粗略计数约 150+ 处）。一个典型函数的完整状态信息碎片化分布在多个章节：

以 `normalize` 为例，实施者需交叉阅读：
- §3.7 函数描述（v11/v13/v14 多处保留历史修订标记）
- §1 系统性设计约束（T(0)/T(1) 字面量替代策略，v8/v9/v14 多层迭代）
- §5.3 边界条件表（零四元数 + length 极小值两行）
- §3.13.2 审计表（运行时行为契约行）
- §11.5 可用性对照表（✅ 可用标记）
- §8 验证项 25（P0 验证 T(Float64(n)) 语法）

这 6 个章节分布在文档的 6 个相距较远的位置。虽然文档在 §3.13.2 审计表和 §5.3 边界条件表中引入了交叉引用机制（「参见 §X」标注），但 §3.7 `normalize` 函数描述本身并未提供到其他 5 个位置的集中导航。下游编码者在实现该函数时可能遗漏关键约束（如 T(0) 字面量替代路径的 v14 修订），尤其是当该修订标注在 §1 而非 §3.7 中时。

**所在位置**：全文散布的修订标注（§1/§3.7/§3.9/§3.11 等）均可能遗漏被修改函数的集中参考清单

**严重程度**：严重

**改进建议**：对 §3.7~§3.15 中每个「完整实现」函数，在其函数描述末尾新增「**集中参考清单**」子节，列出该函数涉及的所有跨节位置（§1 约束、§5.3 边界条件、§3.13.2 审计表行、§11.5 可用性行、§8 验证项号），格式如：「**cross-ref**: §1 T(n) 字面量策略 / §5.3 边界表 `zero quaternion` 行 / §3.13.2 审计表 `normalize` 行 / §11.5 `normalize` 行 / §8 验证项 25」。该清单可作为文档重构的渐进式步骤，无需短期内全部补齐，但建议优先为 P0 约束相关的函数（`normalize`/`axis`/`mat3Cast`/`mat4Cast`）补齐。

## 问题 5（一般）— 需求响应充分度：编码启动前验证项缺少独立验证报告模板，baseline 假设管理纪律缺失

**问题描述**：§8 列出 29 项编码启动前验证项（含 3 项 P0），但文档未提供任何「验证结果记录表」模板。从 v10 到 v11 的教训可知（`FloatingPoint<T>` 接口方法从「不存在」被修正为「存在」，v10 错误持续多个版本），同样的假设验证错误可能再次发生。缺乏标准化验证报告模板意味着：

- 验证结果不可追溯（谁何时验证了什么，结果如何）
- 验证项 20 虽然已声明「要求独立重新核验」，但未定义核验通过标准（如「编写 ~20 行最小仓颉测试文件并通过 `cjpm build`」）
- 验证失败时无标准化的反馈渠道通知设计修订

**所在位置**：§8 编码启动前验证项（第 1242-1270 行）

**严重程度**：一般

**改进建议**：在 §8 编码启动前验证项末尾新增「验证结果记录表」模板，包含列：验证项编号、假设描述、验证通过/失败、验证日期、验证人、失败后处置路径（引用 §1 回退方案或 §8 受影响章节清单）。对 3 项 P0 验证项（20、25、29），要求验证通过后才可启动编码。该模板不增加新的验证负担——仅结构化已有验证流程。

## 问题 6（一般）— 落地可实施性：§3.3 item 7 fromMat4 手动提取策略依赖未验证的字段名假设

**问题描述**：§3.3 item 7 的 `fromMat4` 实现策略描述为「直接读取 `m.c0`/`m.c1`/`m.c2` 三列构建 Mat3x3」。该假设在验证项 27（§8 第 1269 行）中被承认「需验证 Mat4x4 列字段名是否为 `c0`/`c1`/`c2`/`c3`」。若阶段二 Mat4x4 的列字段名实际为 `col0`/`col1`/`col2`/`col3`（或其他命名），则 §3.3 item 7 的实现描述将直接导致编译错误。设计文档将修复责任标注为「编码者——验证后按实际字段名修改实现注释」，但设计文档本身应保证描述基于已验证的事实，而非「假设字段名为 X」——因字段名是已存在的设计事实（阶段二已完成），应在设计阶段确认而非留给编码者。

**所在位置**：§3.3 item 7（第 444 行 fromMat4 实现描述）、§8 验证项 27（第 1269 行）

**严重程度**：一般

**改进建议**：在发布本设计文档前，**直接查阅阶段二 `cjglm/src/detail/type_mat4x4.cj` 确认列字段实际命名**，然后将 §3.3 item 7 描述中的 `m.c0`/`m.c1`/`m.c2` 修正为实际字段名。此验证可在 1 分钟内完成（仅需读一行文件），不应留给编码者。同时可将验证项 27 的状态从「验证假设」升级为「已核验确认」。

## 问题 7（一般）— 落地可实施性：§11.5 表缺少恒等性映射标注，下游按需查找定位成本高

**问题描述**：§11.5 函数可用性对照表按函数名排列，但未标注每个函数在设计文档正文中的定位章节。对于熟悉 GLM 1.0.3 但不熟悉本设计文档结构的开发者，需要在 §11.5、§10 覆盖矩阵和 §3.x 函数描述间反复切换才能定位一个函数的完整信息。例如，函数 `pow` 在 §11.5 中标为 `❌ stub`，但无法直接跳转到 §3.10（函数详细依赖描述）、§8 验证项 16（次正规数边界验证）或 §1 回退方案 H2（`FloatingPoint<T>.getMinDenormal()` fallback）。相比之下，§10 覆盖矩阵已有「对应设计方案章节」列，§11.5 作为文档声明的「主要参考来源」反而缺少此列，形成可用性缺口。

**所在位置**：§11.5 函数可用性对照表（第 1639-1705 行）

**严重程度**：一般

**改进建议**：在 §11.5 表中新增「设计章节」列，为每个函数列出其在 §3.x 中的主要描述位置。对 stub 函数同时标注其依赖的 §1 约束位置。该列可以简化格式（如「§3.10」或「§3.11+§1」），与 §10 覆盖矩阵的「对应设计方案章节」列保持一致。

## 整体质量评价

v20 产出在技术深度和覆盖完整性方面表现出色——完整覆盖了阶段三所需的四元数类型、运算符体系、4 个 ext 函数库、gtc 函数库、空桩文件、测试设计、编码验证项、覆盖矩阵和消费者迁移指南，对 GLM 1.0.3 的实际依赖链分析深入且基于源码核实。经过 20 轮迭代和累计 50+ 项审查意见的落实，文档在事实正确性方面已相当可靠。

**针对质询意见的回应**：

质询报告指出 v1 审查报告「几乎全部集中于维度 2（事实错误/逻辑矛盾），对维度 1（需求响应充分度）和维度 3（整体深度和完整性的落地视角评估）覆盖严重不足」。本 v2 审查报告新增了维度 1 的系统性评估（问题 1：缺少 GLM API 范围基准声明；问题 5：验证项模板缺失）和维度 3 的落地视角评估（问题 4：信息碎片化影响可实施性；问题 6：字段名假设未提前核验；问题 7：§11.5 缺少章节映射），同时保留了对维度 2 已有但未修复的遗留问题（问题 2：权威声明矛盾；问题 3：验证命令事实错误）的重新确认。

**当前文档的改进优先级建议**：

- **P0（阻塞编码启动）**：修复 §8.3 与 §11.5 的权威声明矛盾（问题 2）——该矛盾影响下游对文档的信任，且跨越 8 个版本未修复；修正 gen_fwd_aliases.py 验证命令（问题 3）——否则编码者无法验证 fwd.cj 生成正确性
- **P1（高优先级）**：为 normalize/mat3Cast/mat4Cast/axis 等 P0 约束相关函数补充集中参考清单（问题 4）；查阅阶段二 Mat4x4 源码确认字段名并更新 fromMat4 描述（问题 6）
- **P2（中等优先级）**：新增 GLM 1.0.3 stage 3 API 基准声明（问题 1）；在 §11.5 新增映射章节列（问题 7）；提供验证结果记录表模板（问题 5）
