## 问题 1（严重）：Vec4(Vec3, T) 双参数构造函数在阶段一不存在，Quat×Vec4 实现路径中断

**问题描述**：§3.4 运算符体系表 `Quat×Vec4` 行（第 635 行附近）的实现公式为 `Vec4(q * Vec3(v), v.w)`，并明确声明：
> "该双参数主构造函数在阶段一 `type_vec4.cj` 中已实现——阶段一 Vec4 主构造函数（v12 核验，Issue 9 响应）已声明 `init(vec3: Vec3<T, Q>, w: T)` 两参数版本作为 Vec4 标准构造入口之一"

**验证结果**：经查阅 `cjglm/src/detail/type_vec4.cj` 全文（194 行），Vec4 结构体仅声明 3 个构造函数：
- `init(scalar: T)` — 标量填充（第 13 行）
- `const init(x: T, y: T, z: T, w: T)` — 逐分量构造（第 20 行）
- `init(v: Vec1<T, Q>)` — 从 Vec1 填充（第 27 行）

**不存在** `init(vec3: Vec3<T, Q>, w: T)` 双参数构造函数。这意味着 `Quat×Vec4` 运算符的公式 `Vec4(q * Vec3(v), v.w)` **在现有代码库中不可编译**。阶段一的设计文档可能做过此设计，但实现中从未完成。

- **所在位置**：§3.4 运算符体系表 `Quat×Vec4` 行（第 635 行附近）
- **严重程度**：严重
- **影响分析**：
  - 直接阻塞 `Quat×Vec4` 运算符的编码实现
  - `Vec4×Quat` 运算符通过 `Vec3` 中间路径间接依赖此构造函数，同样受影响
  - §11.5 中标注 ⚠️（可编译但运行时抛 stub）的 `Quat×Vec4`/`Vec4×Quat` 实际上连编译都无法通过
  - §3.13.2 审计表中标注 ⚠️ 的函数状态不准确——`Quat×Vec4`/`Vec4×Quat` 在本阶段状态应为 ❌（编译失败）而非 ⚠️，除非在阶段三补充该构造函数
- **改进建议**：
  1. **方案 A（推荐）**：在阶段三 `detail/type_quat.cj` 的 Quat extend 块中实现 `Quat×Vec4` 为手动 Vec4 逐分量构造：`let rotated_vec3 = q * Vec3(v.x, v.y, v.z); Vec4(rotated_vec3.x, rotated_vec3.y, rotated_vec3.z, v.w)`，消除对 Vec4(Vec3, T) 构造函数的依赖。这是最简单的修复，不涉及其他文件修改。
  2. **方案 B**：在阶段三的 `type_vec4.cj` 中补充 `init(v: Vec3<T,Q>, w: T)` 构造函数。但需注意 `init`（非 `const init`）不能用于 `const` 上下文，且涉及对阶段一已完成文件的修改，需跨阶段协调。
  3. 无论选择哪种方案，需同步修正 §11.5 中 `Quat×Vec4`/`Vec4×Quat` 的状态标注（从 ⚠️ 修正为正确的分类）以及 §3.13.2、§8 产出物清单等处的状态引用。同时修正 §3.4 中"阶段一已实现"的错误声明。

## 问题 2（一般）：「阶段一 Vec3 已成功声明 27 个 const func」的实践依据不成立

**问题描述**：§3.11 `conjugate` 函数的 const 适用性论证中宣称：
> "阶段一 `cjglm/src/detail/type_vec3.cj:54-80` 已成功声明 27 个逐分量运算符与函数为 `const func`，包括 `negative(vec3): Vec3`（逐分量取反，等价于 `-v`）……——`conjugate` 函数体实现模式……与阶段一 `negative`/`negate` 的实现模式完全一致"

§5.4 const 上下文约束段也引用同样依据。

**验证结果**：经查阅 `cjglm/src/detail/type_vec3.cj` 全文（185 行），该文件的 extend 块中**没有任何函数或运算符声明为 `const func`**。从第 63 行开始的 `extend<T, Q> Vec3<T, Q> where T <: Number<T>, Q <: Qualifier` 块中所有运算符和函数（`+`/`-`/`*`/`/`/一元 `-`/`add`/`sub`/`mul`/`div`/`negative`/`negate` 等 27 个函数）均未带 `const` 关键字。`type_mat2x2.cj` 等其他阶段二文件也确认无 `const func`。

这意味着"27 个 const func 已在阶段一成功实践"这一论断**是虚构的**，`conjugate` 可声明为 `const func` 的推论失去了实践依据。项目实际模式是**所有 extend 块函数均非 const**——这可能是因为仓颉 extend 块本身不支持 const 声明（属于语言限制），而非设计选择。

- **所在位置**：§3.11 `conjugate` 描述段（第 782 行附近）、§5.4 const 上下文约束段（第 1655 行附近）
- **严重程度**：一般
- **影响分析**：
  - `conjugate` 能否声明为 `const func` 需要独立的编译验证（验证项 24），不能凭虚构的阶段一实践推断
  - 若 extend 块不支持 const 声明，`conjugate` 应按照项目实际模式声明为普通 `func`
  - §5.4 的 const 推理链条需重新评估
- **改进建议**：
  1. 删除 §3.11 中引用阶段一"27 个 const func"的段落，替换为诚实的声明：**"项目当前实践——所有 extend 块函数均不声明为 const（经查阅 type_vec3.cj 确认）。`conjugate` 在编码阶段由验证项 24 确认 extend 块是否支持 const 声明"**
  2. 同步修订 §5.4 中对应的实践依据引用
  3. 验证项 24 的验证方式应从"参照阶段一 const func 实践"修订为"独立验证 extend 块中 const func 声明的编译可行性"

## 问题 3（一般）：§9a 与 §11.5 的 ❌ 计数存在未解释的 2 函数差异

**问题描述**：§9a 覆盖矩阵合计行声明 "86 ❌ + 4 ⚠️ = 90"，而 §11.5 合计行声明 "⚠️ 4 个…❌ ~84 个" = 88。两者在 ❌ 计数上相差 2 个函数。

**验证**：
- §9a 的 ✅ ~75 + ❌ 86 + ⚠️ 4 = 165（总 GLM API 数）
- §11.5 的 ✅ ~18 + ⚠️ 4 + ❌ ~84 = 106（独立函数范式计数）

§11.5 的"§11.5 与 §9a 计数口径映射注记"解释了 ✅ 类别的计数差异（106 vs 165，差 59），也解释了 59 的来源（28 常量 + 16 重载 + 15 运算符变体 = 59）。**但该注记未解释为何 ❌ 计数在两种口径下也相差 2 个**（86 vs 84）。下游读者无法判断这 2 个函数的去向——是 §9a 的"86 ❌"包含了 §11.5 中不计入独立函数范式的 2 个运算符重载变体？还是两者存在实质性数据矛盾？

- **所在位置**：§9a 覆盖矩阵合计行（第 2220 行）、§11.5 合计行（第 2378 行）及计数口径映射注记（第 2381 行）
- **严重程度**：一般
- **改进建议**：在 §11.5 的"§11.5 与 §9a 计数口径映射注记"中**显式说明 ❌ 类别的口径差异来源**。例如：**"§9a 的 86 ❌ 包含 §11.5 中未独立计数的运算符重载变体和某些具有 GLM 映射但未进入独立范式计数的函数变体；在独立范式口径下这 2 个 ❌ 函数计为已含在其他条目中，非实质性缺失。"** 确保 H2b 核验项的交叉核对口径明确包含 ❌ 类别的精度说明。

## 问题 4（一般）：`fromMat4` 与 `Mat3x3(Vec3, Vec3, Vec3)` 主构造函数的可行性依赖未经验证

**问题描述**：§3.3 item 7（第 614 行）的 `fromMat4` 实现路径描述为：
> "读取 `m.c0`/`m.c1`/`m.c2` 三列构建 `Mat3x3<T,Q>(Vec3<T,Q>(m.c0.x, m.c0.y, m.c0.z), Vec3<T,Q>(m.c1.x, m.c1.y, m.c1.z), Vec3<T,Q>(m.c2.x, m.c2.y, m.c2.z))`"

这假设 `Mat3x3<T,Q>` 存在一个接受 3 个 `Vec3<T,Q>` 参数的主构造函数。经查 `cjglm/src/detail/type_mat3x3.cj` 确认存在 `const init(x: T, y: T, z: T)` 逐分量构造，但**三参数 `Vec3` 版本构造函数是否存在、签名是否匹配，未在文档中验证或引用**。

文档声称"无需 `one: T` 形参"的论证链条中，Vec3 和 Mat3x3 的主构造函数无 `one` 形参是核心论据，但 Mat3x3 的三 Vec3 参数主构造函数签名（`init(a: Vec3<T,Q>, b: Vec3<T,Q>, c: Vec3<T,Q>)`）在阶段二中是否存在，文档未做引用或验证。

- **所在位置**：§3.3 item 7（第 614 行附近）
- **严重程度**：一般
- **改进建议**：
  1. 查阅阶段二 `type_mat3x3.cj` 确认三 Vec3 参数主构造函数的存在性和签名（若不存在，需要求补充或改用逐元素填充替代路径）
  2. 在 §3.3 item 7 中增加明确的引用行号（如"经查阅 `type_mat3x3.cj:xx` 确认存在 `init(a: Vec3<T,Q>, b: Vec3<T,Q>, c: Vec3<T,Q>)` 构造函数"）
  3. 或者，更保守的做法：将 `fromMat4` 的实现路径从构造中间 Mat3x3 改为**逐元素提取 9 个标量**直接构造 Quat（与 GLM `gtc/quaternion.inl:128` 一致），完全避免对 Mat3x3 多参数构造函数的依赖。该方法已在问题 1 的 C 轮迭代中被建议过（历史迭代第 4 轮问题 1），但设计方最终选择了手动提取列 + Mat3x3 构造函数路径——如果该构造函数不存在，需回退至此方案。
