## 问题 3（严重）：v30 轮次才修正 Vec3×Quat/Vec4×Quat 状态，反映根本原因分析流程仍存在缺口

**问题描述**：v30 修订将 Vec3×Quat/Vec4×Quat 从 ✅ 修正为 ⚠️，新增的"实现链路注释"详细说明原因——内联 conjugate/dot 路径消除的是包间循环依赖（glm.detail → glm.ext），但并未消除对 geometric.cj 向量 cross stub 的运行时调用。

此修正暴露出的问题是：在 v18 做出"内联 conjugate/dot 路径消除包间循环依赖"的决策时（这是一个正确的决策），并未同时评估该路径是否消除了对 Vec3 cross stub 的运行时依赖——实质上，当时只评估了编译时依赖（import 关系），未评估运行时依赖（调用链传播）。

这意味着从 v18 到 v30（跨越 12 轮迭代），该函数的状态一直被错误标注，而所有依赖该标注的测试覆盖、用户能力视图、计数统计、验收标准也同步错误。追溯原因：§3.4 的"实现链路注释"此时才首次完整描述从 Vec3×Quat 到 geometric.cj cross stub 的完整异常传播路径——在此之前，这个运行时调用链从未在文档中被完整追踪过。

**所在位置**：§3.4 Vec extend 块运算符表、"实现链路注释"段
**严重程度**：严重
**改进建议**：
1. 对所有标注为 ⚠️/✅ 但依赖 stub 的函数，在其函数描述段中新增"运行时调用链依赖分析"子段——显式画出从函数入口到所有可能抛 stub 异常的调用路径
2. 将此分析纳入编码启动前验证项的覆盖范围（验证项 29 仅检查 `import` 关系，未检查运行时调用链）
3. 在 §8.3 H 节新增"运行时调用链无 stub 依赖"验证门禁，对标记为 ✅ 的函数强制执行

## 问题 4（一般）：§11.5 合计行算术仍然存在误导性表述

**问题描述**：§11.5 合计行声明为：

```
✅ ~18 个 | ⚠️ 4 个（Quat×Vec3 + Quat×Vec4 + Vec3×Quat + Vec4×Quat 分别计数） | ❌ ~84 个（含 3 个整数 T 抛异常条目）
⚠️+❌ 小计 = 88 个
```

算术上 `4 + 84 = 88`，但 `18 + 88 = 106 ≠ 165`。虽然文档在 "§11.5 与 §9a 计数口径映射注记" 中解释了差异来源于计数规则不同（独立函数范式 vs 含重载/常量展开），但 §11.5 表本身作为一个"权威来源"，其合计行缺少一行"总函数数"来让读者一眼确认 165 个 GLM API 的去向。

读者需要手动计算 `18 (✅) + 4 (⚠️) + 84 (❌) = 106`，再对照 §9a 覆盖矩阵的 `~77 (含展开口径) + 88 = 165`，再通过口径映射注记才能理解 59 个函数的差异来源。这个认知链条过长。

**所在位置**：§11.5 合计行
**严重程度**：一般
**改进建议**：在合计行中新增一行"总函数数（独立函数范式口径）= 106"，并列出与 165 的差值 59 的来源分类：(a) gtc/constants 28 个常量不计入独立函数；(b) vector_relational epsilon 4 个重载族不计入；(c) 运算符/构造函数/复合赋值变体展开不计入。使读者无需跳转到口径注记即可理解计数缺口。

## 问题 5（一般）：§5.3 边界条件表仍然缺少跨函数组合场景的边界契约

**问题描述**：§5.3 边界条件表在 v24 新增了 NaN/Inf 输入传播、lerp a=NaN、单位矩阵/零矩阵输入、axis NaN w 分量、inverse 非单位四元数等条目。但仍有以下组合场景未覆盖：

1. **normalize 对 NaN 输入**：当 `length(q)` 返回 NaN 时（如 q 的分量中包含 NaN），`normalize(q)` 的保护分支 `tmp1 <= T(0)` 判定结果为 false（NaN 与 0 比较返回 false），进入 else 分支 `q / tmp1`，结果全为 NaN——此行为是否与预期一致？GLM `quaternion_geometric.inl:17-24` 的 `if (len <= T(0))` 同样无法捕获 NaN 输入。

2. **mat3Cast 对 NaN 四元数输入**：当输入四元数包含 NaN 分量时，`mat3Cast` 公式中 `q.x * q.x` 等运算产生 NaN，结果矩阵是否全为 NaN？§3.2.1 边界行为契约仅描述"非单位四元数"行为，未涉及 NaN 四元数。

3. **inverse 对 NaN 四元数输入**：`conjugate(q)` 对 NaN 分量取反后仍为 NaN，`dot(q,q)` 中的 NaN 运算产生 NaN，最终 `NaN / NaN = NaN`——结果是 NaN 四元数。§5.3 仅描述零四元数场景。

4. **lerp 对无穷大 a**：当 `a = +Inf` 时，`x * (1 - Inf) + y * Inf = x * (-Inf) + y * Inf`，产生 NaN——是否应纳入边界条件表？

这些组合场景在阶段三的实现中具有实际意义——阶段三的 ✅ 函数（conjugate、inverse、lerp、normalize、mat3Cast 等）不会抛 stub，因此这些 NaN/Inf 传播行为会实际执行，下游消费者需要知晓。

**所在位置**：§5.3 边界条件表
**严重程度**：一般
**改进建议**：新增 4 行逐条覆盖上述 4 类 NaN/Inf 输入组合场景的行为契约，统一引用 GLM 的浮点传播语义作为最终行为基准。

## 问题 6（一般）：三角函数标量签名的 `where T <: FloatingPoint<T>` 约束实际限制了下游使用场景，但与 §1 中 T(0)/T(1) 字面量替代策略中的泛型假设存在张力

**问题描述**：§3.13.1 trigonometric.cj 中所有 75 个函数签名统一标注 `where T <: FloatingPoint<T>`。§1 中"常量型 T(n) 字面量替代"策略的 H1 验证结论为 `T(Float64(n))` 对所有 `Number<T>` 类型（包括 Int8~Int64）均编译通过并运行时值正确。

如果 trigonometric.cj 的标量 `sin`/`cos` 等函数收紧到 `FloatingPoint<T>`，而阶段四 `pow` 函数内部调用的 `std.math.asin`/`std.math.acos` 也仅支持 `Float32`/`Float64`（仓颉 std.math 限制），那么这个约束是有必要的——但文档没有明确说明"如果某用户确实需要整型 T 实例化 trig 函数，应通过何种路径实现"。

此外，v11 附注（质询报告 B 项响应）声明："本约束的实施不阻塞阶段三本身的完整实现路径，仅在阶段四实现 trigonometric.cj 函数体时生效"——但在阶段三，trigonometric.cj 所有函数都仅为 stub（`throw Exception("stub")`），因此约束收紧在阶段三并不影响运行行为。但约束收紧**影响编译行为**——整型 T 实例化 trig 函数在阶段三会编译失败（而非运行时报 stub）。文档的"不阻塞阶段三"声明只考虑了运行时行为，未考虑编译时行为。

**所在位置**：§3.13.1 函数表 + §1 系统性设计约束段
**严重程度**：一般
**改进建议**：在 §3.13.1 表头或下方补充说明："本约束收紧在阶段三仅影响编译行为——整型 T 实例化 trig 函数的调用方在编译期报类型不匹配错误。阶段三所有 trig 函数体为 `throw Exception("stub")`，若某调用方需要整型 T 实例化 trig 函数进行编译期类型验证，可先自行通过 `Float64` 类型降级调用。阶段四实现函数体后，该约束确保 `std.math` 函数的编译正确性。"
