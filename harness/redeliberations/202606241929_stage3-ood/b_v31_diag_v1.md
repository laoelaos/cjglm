## 二、存在的问题

### 问题 1（严重）：版本标注清理承诺未闭环，文档可实施性仍受损害

**问题描述**：
v31 文档头部声称「v31 已对全部非决策性版本标注执行统一清理」。经 `grep` 全文扫描，正文中仍广泛分布大量非决策性版本标注，包括但不限于：
- `（v{N} 关键修订）`、`（v{N} 修订）`、`（v{N} 新增）`
- `（Issue {N} 响应）`
- `（v{N} 澄清）`、`（v{N} 明确）`
- `***v{N}` 模式（如 `**v3 关键变更**`、`**v5 修订**`）
- 上述非决策性标记广泛分布于 §3.2、§3.2.1、§3.3、§3.4、§3.5、§3.7、§3.9、§3.10、§3.11、§3.12、§3.13.1、§3.15、§3.16、§5.3、§11.5 等关键章节

这导致以下问题：
(a) 下游实施者无法区分「当前有效的设计内容」与「历史修订追溯」，被迫考古式阅读。
(b) 「v31 已清理」的声称为虚假声明。迭代历史第 26/27/29/30 轮反复出现同类问题，结构性预防措施（H3 核验门禁）未生效。
(c) 版本标注噪音此前已在迭代第 7/15/22/26/29/30 轮被反复识别和承诺修复，横跨 14 个迭代版本未根治，说明文档的自我核验机制存在系统性缺口。

**所在位置**：全文 §3.2、§3.2.1、§3.3、§3.4、§3.5、§3.7、§3.9、§3.10、§3.11、§3.12、§3.13.1、§3.15、§3.16、§5.3、§11.5 等章节正文中的 `（v{N} 修订）`、`（v{N} 新增）`、`（Issue {N} 响应）` 模式。具体 100+ 处，详见 grep 结果。

**严重程度**：严重

**改进建议**：
- 在版本发布流程中增加 `grep -n "（v[0-9]\|（Issue [0-9]\|\\*\\*v[0-9]" ` 自动化门禁，确保零残留。
- 或采用「版本标注全部剥离至独立文件（如 `a_v27_revision_history.md`），正文仅保留一条变更摘要行」策略（迭代第 24 轮曾提出此建议但未落实），从根源消除散布式修订追溯。
- 在 v31 认为已清理的基础上做第二次全文人工扫描 + 机器辅助核查，非决策性标注全部移除，保留的设计内容直接写入正文。

### 问题 2（严重）：需求响应充分度的结构性不足——设计未回答"这是否满足用户需求"

**问题描述**：

文档虽然在 §1.2 新增了「用户能力视图」来诚实展示能力边界，但**没有明确回答以下根本问题**：
- 这个设计是否仍然满足阶段三的路线图目标？还是路线图与设计的偏差已经大到需要路线图本身做出修订？
- 如果 11% 的 API 可用率对于阶段三的预期交付目标已是重大偏离，文档是否应主动升级此发现？
- 三项已决策偏差（D35-D37）仅记录为「已闭环」，但「slerp 不可用」本身是否影响路线图的阶段三验证标准——文档未做自我评估。

**所在位置**：§1 设计目标段、§1.2 用户能力视图、§3.16 需求对齐说明（D35-D37 决策段）

**严重程度**：严重

**改进建议**：
- 在 §3.16 末尾新增「阶段三交付能力与路线图预期差距评估」子节，量化回答「本设计是否满足路线图阶段三目标」这一根本问题。
- 若结论是「不满足」，应在文档中主动提出路线图修订建议，而非仅以设计决策记录偏差。
- 明确 18 个 ✅ 函数（11%）是否构成阶段三的可用基线的判断依据——从 GLM 迁移者的视角评估这些函数是否能支撑「四元数类型的基本数学运算」这一定位。

### 问题 5（一般）：trigonometric.cj 75 个 stub 函数的「内部依赖」标注存在误导风险

**问题描述**：
§3.13.1 三角函数表中，75 个 stub 函数的「内部依赖」列标注了 `std.math.sin（阶段四依赖）`、`std.math.cos（阶段四依赖）` 等依赖。v23 修订已添加「标注为阶段四依赖」前缀以避免误导。但：

- 表前说明虽然澄清「本阶段所有函数体为 `throw Exception("stub")`，编译时不实际触发这些依赖」，但表行中仍有 `radians`/`degrees` 行标注「**例外（stdlib 不存在）**」和「硬编码 π 字面量」等实现细节——这些细节与 stub 函数的 `throw Exception("stub")` 行为无关，仅对阶段四实现有意义。
- 下游编码者按 §3.13.1 表实现 stub 函数时，需要反复对照 table 中的「内部依赖」列和表前的「依赖是阶段四依赖」说明，才能确认编译时不需要实际实现这些依赖——认知负荷仍偏高。
- `radians`/`degrees` 行的「交叉引用」子句建议引入 `PIVAL<T>` 共享常量，但该共享常量在 §3.12 `pi<T>()` 实现中是否存在尚未确认——形成跨节的假设链。

**所在位置**：§3.13.1 三角函数表（共 15 行 + 表前说明 + `radians`/`degrees` 行附注）

**严重程度**：一般

**改进建议**：
- 三角函数表的「内部依赖」列，对所有 75 个函数统一为仅标注「（阶段四依赖）」，不区分 `std.math.sin`、`std.math.cos` 等具体函数名——这些细节在阶段四设计文档中展开即可。
- `radians`/`degrees` 行的实现细节（硬编码 π 字面量、`PIVAL<T>` 交叉引用）移至表下「实现策略」段，表行只保留「（阶段四依赖，硬编码 π 字面量）」简短标注。
- 在 §8 编码启动前验证项中新增「`PIVAL<T>` 共享常量可行性验证」（如果采用此建议），或者在 §3.12 中明确声明 `pi<T>()` 的 π 值来源，消除跨节假设。

### 问题 6（轻微）：§3.9 `axis` 公式中的冗余类型转换未消除

**问题描述**：
§3.9 `axis` 函数实现公式写为：
`tmp2 = T(Float64(1)) / T(std.math.sqrt(tmp1))`

但 `std.math.sqrt` 提供 Float16/Float32/Float64 重载（§1 v11 已明确），`T=Float32` 时返回 Float32，`T=Float64` 时返回 Float64。此处 `T(std.math.sqrt(tmp1))` 的 `T(...)` 外层包裹：
- `T=Float32` 时：`Float32(std.math.sqrt(Float32(tmp1)))` → 冗余，`std.math.sqrt` 已返回 Float32
- `T=Float64` 时：`Float64(std.math.sqrt(Float64(tmp1)))` → 冗余，已返回 Float64

该冗余在第 5 轮已被指出（问题 11），v11 修订已确认删除。但在 v31 中该公式仍然保留冗余的 `T(...)` 外层包裹。

**所在位置**：§3.9 `axis` 函数描述（`tmp2 = T(Float64(1)) / T(std.math.sqrt(tmp1))`）

**严重程度**：轻微

**改进建议**：
- 将公式修订为 `tmp2 = T(Float64(1)) / std.math.sqrt(tmp1)`，删除冗余的 `T(...)` 外层包裹。
- 如果保留 `T(...)` 是为了防御性转换，应在公式旁添加注释说明意图，而非让下游猜测。

## 三、整体质量评价

v31 文档经过 30 轮迭代修订，在技术方案正确性、边界契约完备性、依赖方向合理性等维度已达到较高成熟度。模块间依赖 DAG 无循环依赖、P0 假设已通过编译验证、§5.3 边界条件表覆盖全面、§3.16 演进兼容性承诺完整——这些是内部审议反复覆盖且已解决的维度。

从需求响应充分度、文档可实施性、整体深度完整性的角度看，**两个严重问题需要优先处理**：
1. **版本标注清理不彻底**（问题 1）直接影响文档的可实施性——下游无法快速定位当前有效设计。该问题横跨 14 个迭代版本反复出现，根因是修订历史与正文内容混合存储的结构性问题，局部清理无法根治。
2. **需求响应充分度的自我评估缺失**（问题 2）——文档未主动回答「11% 可用率是否满足路线图预期」这一根本问题。这不是设计正确性问题，而是设计定位问题：一份 OOD 文档应明确其产出与需求的差距，而非仅仅记录差距而不评估影响。

其余 5 个问题（问题 3-7）属于可实施性优化范畴，不阻塞编码启动，但建议在编码前完成问题 3（§11.5 权威声明统一 + 函数状态去重）和问题 4（验证时序声明审计）的修复。
