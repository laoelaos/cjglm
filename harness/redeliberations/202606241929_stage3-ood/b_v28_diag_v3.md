## 问题 1（严重）：设计未从用户需求响应充分度角度评估阶段三的实际交付能力

**问题描述**：用户需求为「把 references/glm-1.0.3 用仓颉重新实现」及「迁移阶段三」。v28 文档从技术规格角度完成了对 165 个 GLM API 的逐函数状态标注和 88 个 ⚠️/❌ 函数的量化说明，但从未从最终用户视角回答一个根本问题：**「阶段三完成后，一个 GLM 四元数使用者能否将代码迁移到 cjglm？如果能，哪些场景能用、哪些不能用？」**

具体缺失：

(a) **无用户场景级交付能力声明**：§1 设计目标段声明「为库提供基础的旋转表达和插值运算框架」，§1 前置风险声明（v28 新增）列出了受影响函数，但始终未将 API 状态转化为用户可理解的场景化评估。例如：
- 一个游戏开发者想要 `q * v`（四元数旋转向量）：**运行时抛 stub 异常**（§3.4 标注⚠️）
- 一个动画系统开发者想要 `slerp(q1, q2, t)`: **❌ stub**（§3.11 标注❌）
- 一个相机系统开发者想要 `quatLookAt(direction, up)`: **❌ stub**（§3.15 标注❌）

这些信息散布在文档各处，用户需自行对照 4+ 个章节才能拼出完整图景。§3.16 的「临时降级路径一览表」虽列出了变通方案，但其描述方式（如「自行构造旋转矩阵 + fromMat3 转换」）假定用户具备四元数数学专业背景，未评估普通 GLM 使用者能否独立实施。

(b) **无「用户能做 vs 不能做」的能力边界声明**：文档中 18 个 ✅ 可用函数以 `conjugate`/`inverse`/`lerp`/`dot`/`length`/`normalize`/`axis` 等底层运算为主，但缺少一句诸如：「阶段三完成后，用户可以：构造四元数 → 取共轭/逆/归一化 → 四元数间比较和加法 → 矩阵↔四元数互转；但无法直接执行：`q * v` 四元数旋转 → slerp 球面插值 → exp/log/pow/sqrt 指数运算 → 欧拉角提取 → 看向函数」的用户导向总结。

(c) **§3.16 降级路径的实际可操作性不足**：「临时降级路径一览表」（§3.16 line 1102-1108）中，「摄像机看向」的降级方案描述为「自行构造旋转矩阵 + fromMat3 转换，约 10 行代码，1-2 人日」。该描述缺少：(i) 对应的 GLM 源函数名（`glm::quatLookAtRH`）；(ii) 仓颉代码骨架示例；(iii) 与原生 `quatLookAt` 的行为差异（如坐标系约定）；(iv) 1-2 人日估算的分解依据。下游决策者无法据此判断降级路径对自身项目是否可接受。

**所在位置**：§1 设计目标段（line 12-14）、§1 前置风险声明（line 14-16）、§3.16 临时降级路径一览表（line 1102-1108）

**严重程度**：严重

**改进建议**：
1. 在 §1 设计目标段后新增「阶段三用户能力视图」小节，将 165 个 API 按常见使用场景分组（如「构造/析构」「旋转」「插值」「互转」「比较」「常量」），每组以表格形式标注本阶段 ✅/⚠️/❌ 状态，让用户一目了然知道自己的使用场景是否受支持
2. 「用户能力视图」应包含约 5-8 个典型 GLM 使用场景的端到端代码示例，标注本阶段运行结论
3. §3.16 降级路径表为每条路径补充：(a) 对应的 GLM 源函数名；(b) 仓颉代码骨架（非伪代码）；(c) 与原生 API 的行为差异量化评估；(d) 成本分解依据
4. 在 §3.16 末尾新增「阶段三交付后用户预期管理」段落，明确告知使用者「能用什么、不能用什么、降级路径是什么」

## 问题 2（严重）：§11.5 合计行算术表述存在误导，且与 §9a 合计行的计数口径映射关系缺失

**问题描述**：v28 §11.5 合计行（line 2029）表述为「**✅ ~18 个（独立函数）+ ⚠️ 1 个 + ❌ ~87 个 = 合计 88 个 ⚠️/❌**」。该表述存在以下问题：

(a) **算术结构误导**：将 ✅ 计数与 ⚠️/❌ 计数同列求和后标注为「合计 88 个 ⚠️/❌」，但 18 + 1 + 87 = 106，≠ 88。实际「88」仅为 ⚠️ + ❌ = 1 + 87 的计数。该表述将不同维度的数字拼在一行，读者无法区分「106（全口径）」与「88（⚠️/❌）」哪个是合计值。对比 §9a（line 1874）的表述「约 77 ✅ + 87 ❌ + 1 ⚠️ = 88」——§9a 明确标注「约 77 ✅」作为加数之一但不混入 ⚠️/❌ 小计，表述更清晰。

(b) **口径映射缺失**：§11.5 采用「独立函数范式」口径（~18 ✅），§9a 采用「含重载/常量展开」口径（~77 ✅），两者差距 59 个。文档虽在 §1 表格注记中解释了两种口径的差异（line 25-26），但 §11.5 与 §9a 之间无任何交叉引用或映射说明——读者需自行从 §1 跳转到 §9a 再跳转到 §3.13.2，才能理解「18」与「77」如何对应。合计行本身也未标注「独立函数范式」字样，仅靠表头「权威声明」段中的「独立函数范式」隐式表明口径。

(c) **与 §9a 的交叉核验缺少闭环**：§11.5 合计行声明「与 §9a 交叉核验」，但两个合计行使用不同口径（106 vs 165），读者无法直接核验。理论上 18（独立函数）+ 87（❌）+ 1（⚠️）= 106 个 GLM API 的独立函数名覆盖，与 GLM 165 个 API 的差距 59 个（即重载/常量展开的增量）没有在 §11.5 中展示。

**所在位置**：§11.5 合计行（line 2029）、§9a 合计行（line 1874）

**严重程度**：严重

**改进建议**：
1. 将 §11.5 合计行拆分为两行：
   - 「**独立函数口径**：✅ 18 + ⚠️ 1 + ❌ 87 = 106（占 165 个对标 API 的 64%，差距 59 个为重载/常量展开）」
   - 「**⚠️/❌ 小计（独立函数口径）**：1 ⚠️ + 87 ❌ = 88」
2. 在合计行下方新增与 §9a 的映射注记：「§9a（含重载/常量展开口径）：✅ ~77 + ⚠️ 1 + ❌ 87 = 165。两口径差距 59 个，由 28 个 gtc/constants 常量 + 16 个 epsilon 重载 + 构造函数/运算符变体等组成，详见 §1 计数口径说明。」
3. §11.5 表头或合计行旁标注本表使用的计数口径（「独立函数范式」），与 §9a 的「含重载/常量展开」口径形成显式对比
4. 将 §11.5 合计行的交叉核验指向改为明确口径说明：「本行使用独立函数范式口径，与 §9a 合计行（含重载/常量展开口径）形成 18→77 的展开映射」

## 问题 3（一般）：`inverse` 对整数 T 的非零除截断行为未在契约中声明，与 `normalize` 的约束收紧策略不一致

**问题描述**：v26 已将 `normalize` 的 T 约束收紧为 `where T <: FloatingPoint<T>`（§3.7 normalize 描述、v26 修订），理由为整数除法截断会产生无意义的非单位四元数。但 `inverse`（`conjugate(q) / dot(q, q)`）以相同的整数除法截断问题保持 `Number<T>` 宽约束，两处约束策略不一致且未解释。

具体风险场景：`Quat<Int32, PackedHighp>(1, 0, 0, 2)` 调用 `inverse(q)` 时：
- `dot(q, q)` = 1² + 0² + 0² + 2² = 5
- `conjugate(q)` = Quat(-1, 0, 0, 2)（整数取反）
- `inverse(q)` = conjugate(q) / 5 = Quat(0, 0, 0, 0)（整数除法截断，正确应为近似 Quat(-0.2, 0, 0, 0.4)）
- **无声**返回错误结果，不触发异常，不产生 Inf/NaN 等预警信号

§5.3 边界条件表（line 1274-1275）仅覆盖了 `dot(q, q) == 0` 时浮点产生 Inf/NaN、整数抛 `ArithmeticException` 的零除场景，对 `dot(q, q) > 0` 但结果因截断而语义错误的情形无任何说明。这是计算正确性问题，不仅是文档精度问题。

与 `normalize` 对比：`normalize` 的 v26 收紧理由（消除整数除法截断导致的语义歧义）对 `inverse` 同等成立，文档未解释为何两处采用不同策略。若理由为「inverse 在整数 T 下仍然有实用价值」，则应在契约中明确整数截断行为；若理由为「无整数四元数实际用例」，则应将约束收紧至 `FloatingPoint<T>` 与 normalize 一致。

**所在位置**：§3.11 inverse 描述（line 654-656）、§5.3 边界条件表 inverse 行（line 1274-1275）、§3.7 normalize 描述（line 591 v26 约束收紧标注）

**严重程度**：一般

**改进建议**：
1. 在 §3.11 inverse 描述末尾新增约束策略决策说明，二选一并由 D38 明确：
   - **方案 A**：将 `inverse` 约束收紧为 `where T <: FloatingPoint<T>`（与 normalize 一致），消除整数除法截断风险。理由：整数四元数无逆运算的实际用例，收紧约束在编译期暴露问题
   - **方案 B**：保留 `Number<T>` 宽约束，但在 §3.11 和 §5.3 中显式声明整数 T 非零除时的截断行为契约（如「整数 T 的 `inverse` 返回逐分量整数除法截断结果，与浮点 T 的精确逆四元数不同」）
2. 同步在 §7 新增设计决策 D38 记录选定方案及理由

## 问题 4（一般）：trigonometric.cj 75 个函数签名仍依赖下游手工推断，与 gtc/matrix_transform 64 个签名的完全展开形成不对称处理

**问题描述**：v28 已将 gtc/matrix_transform.cj 的 64 个函数从 6 个代表性模板升级为 9 类全部 64 个可直接复制的签名代码块（§3.13 line 733-833，合计 11+10+9+9+9+7+2+6+1 = 64 行签名）。但 trigonometric.cj 的 75 个展开函数仍以「sin + atan2 两个示例 + 其余 13 个按相同模式展开」的方式处理（§3.13.1 line 890-915）。

下游编码者实现 trigonometric.cj 时，需：
1. 将 13 个单参数三角函数（cos/tan/asin/acos/atan/sinh/cosh/tanh/asinh/acosh/atanh/radians/degrees）逐一按「标量 1 个 + Vec1~Vec4 各 1 个」模式展开 = 13 × 5 = 65 个签名
2. 将 atan2 按「标量 1 个 + Vec1~Vec4 各 1 个」模式展开 = 5 个签名
3. 合计 65 + 10（sin 示例已给出 5 个签名 + atan2 示例 5 个签名已给）= 75 个签名

虽然展开规则已在 §3.13.1 line 846-848 明确说明，但与 gtc/matrix_transform 的完全展开处理仍存在实质差距：任何一处模式应用错误（where 子句格式、泛型参数命名、约束声明的细微偏差）只能在编码阶段通过编译错误发现。两种同类 stub 函数组的处理方式不对称。

这延续了上一轮审查（b_v28_diag_v2 Problem 1）的判定。需澄清的是：本问题不否定 v28 在 gtc/matrix_transform 签名模板上「58/64 → 64/64」的改进，仅指出 trigonometric.cj 的 75 个签名仍处于「参考模式自行展开」级别，与 gtc/matrix_transform 的「直接复制粘贴」级别存在梯度差异。

**所在位置**：§3.13.1 完整签名模板示例段（line 890-915）、§3.13.1 三角函数表（line 863-884）

**严重程度**：一般

**改进建议**：
1. 仿照 gtc/matrix_transform 的处理方式，新增附录文件（如 `a_v28_trigonometric_signatures.md`）承载 trigonometric.cj 全部 75 个函数的完整签名模板，在 §3.13.1 正文中引用该附录文件
2. 若因篇幅约束不愿扩展正文，至少为「双曲类」（sinh/cosh/tanh/asinh/acosh/atanh 6 个）和「反三角类」（asin/acos/atan 3 个）各提供 1 个完整展开示例（当前仅 sin 示例 + atan2 示例，只覆盖了 2/15 的函数类别），减少下游推断跨度

## 问题 5（一般）：跨 Qualifier 行为契约在除 `lerp` 外的所有函数中未声明

**问题描述**：跨 Qualifier 调用（如 `dot(highp_q, mediump_q)`）因泛型参数 Q 不匹配将触发编译错误。除 §3.11 `lerp`（D34 讨论）外，§3 中 40+ 个泛型函数和运算符（§3.4 运算符表、§3.7 dot/length/normalize/cross、§3.9 axis、§3.11 inverse/conjugate/isnan/isinf、§3.6 equal/notEqual 等）均无任何跨 Qualifier 行为的前置说明。

这与 §3.11 `lerp` 行（line 658）明确讨论「跨 Qualifier 行为」并引用 D34 决策形成显式对比——同节内 `lerp` 有跨 Qualifier 说明而 `inverse`/`conjugate`/`isnan`/`isinf` 无，下游无法判断是遗漏还是设计意图。

具体证据（v28 正文摘录）：
- §3.4 运算符表（line 499-534）：8 种运算符全部标注 `Number<T>` 约束，0 处提及跨 Qualifier
- §3.7 dot/length/normalize/cross 描述（line 589-601）：4 个函数描述中 0 处提及跨 Qualifier
- §3.9 axis 描述（line 620-630）：公式说明和边界条件中 0 处提及跨 Qualifier
- §3.11 inverse/conjugate 描述（line 653-656）：0 处提及跨 Qualifier
- §3.11 isnan/isinf 描述（line 662-663）：0 处提及跨 Qualifier

上述确认了前一版报告（b_v28_diag_v2 Problem 3）的判断——v28 未对 lerp 之外的任何函数补充跨 Qualifier 行为说明。

**所在位置**：§3.4 运算符表、§3.7（dot/length/normalize/cross）、§3.9（axis）、§3.11（inverse/conjugate/isnan/isinf）

**严重程度**：一般

**改进建议**：
1. 在 §1 或 §5 末尾新增跨 Qualifier 行为的统一契约声明：「本阶段所有泛型函数和运算符（包括 dot/length/normalize/cross/axis/inverse/conjugate/isnan/isinf/算术运算符/比较运算符）仅支持相同 Qualifier 的操作数。跨 Qualifier 调用因泛型参数 Q 不匹配触发编译错误，此为预期行为。跨 Q 兼容性扩展（双 Q 重载）计划在阶段四按需补充（参见 D34）。」
2. 在 §5.3 边界条件表新增一行「跨 Qualifier 调用 → 编译期拒绝（预期行为）」条目
3. §3.11 lerp 的 D34 讨论（line 658）应扩展为统驭全篇的跨 Qualifier 策略声明，而非仅针对 lerp 的独立注记

## 总结

v28 在以下方面有显著改进：前置风险声明（§1）、路线图偏差量化与降级路径一览表（§3.16）、gtc/matrix_transform 64 个签名全面展开（§3.13）、P0 验证项 20/25/29 全部闭环（§8）、O-06 可执行落地计划（§8.3 I 节）、版本标注再清理。验证项 29 完成验证是 v28 的重要突破。

本审查识别出 **5 项问题**（2 严重 + 3 一般）。核心发现：

1. **设计的技术完备度与用户视角之间存在根本性落差**（问题 1、严重）。文档可以精确回答「每个函数的编译/运行时状态是什么」但无法回答「我作为一个 GLM 用户，能在阶段三完成后迁移我的代码吗」。这与用户需求「把 references/glm-1.0.3 用仓颉重新实现」的直接映射是缺失的。

2. **计数体系已修复但在展示逻辑上仍有缺陷**（问题 2、严重）。§11.5 与 §9a 两套合计口径（106 vs 165）的映射关系未在引用点展示，读者需要三处跳转才能理解差距来源。

3. **约束策略一致性持续作为遗留问题**（问题 3、4、5）。`inverse` vs `normalize` 的整数 T 行为差异（问题 3）、trigonometric.cj 与 gtc/matrix_transform 的签名模板展开粒度不对称（问题 4）、跨 Qualifier 行为覆盖率悬殊（问题 5）——三处均为「已在部分范围内解决但未系统性推广」的模式。

**编码启动前置条件整体评估**：
- ✅ P0 假设验证（H1/H2/H3）：三项全部通过（v28 验证闭环）
