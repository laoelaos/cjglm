## 问题 1（中度）

**问题描述**：§1 API 可用率声明与 §3.13.2 审计结论的计数口径差异未做充分说明，可能误导读者对阶段三实际交付能力的判断。§1 声称「约 77 个函数（约 47%）为真完整实现」，但 §3.13.2 审计结论明确「本阶段可调用的真完整实现函数（无 stub 依赖）共 18 个函数」。77 与 18 的差值（59 个）主要由 gtc/constants 28 个常量函数、ext/vector_relational epsilon 16 个重载、以及构造函数/运算符重载等高度同构的变体组成。虽然文档在 §3.13.2 说明了"按函数名计数（不含重载）"并在 §1 表格底部说明"函数级状态以 §11.5 为准"，但首次阅读者在 §1 看到"47% 真完整实现"时，容易形成"阶段三可交付近半数 API 功能"的印象，而实际独立可用的函数范式仅 18 个（其余为常量/比较/重载变体）。这种口径差异在需求响应充分度的视角下属于**沟通透明度问题**——读者从 §1 顶层声明获得的交付预期与 §3.13.2 的精确审计结论存在显著量级差异。

**所在位置**：§1「阶段三 API 可用率声明」段（第 14-20 行）、§3.13.2「审计结论」段（第 822-825 行）

**严重程度**：中度

**改进建议**：在 §1 API 可用率声明表格下方补充注记，明确说明 77 个 ✅ 包含 28 个 gtc/constants 常量函数 + 16 个 vector_relational epsilon 重载 + 构造函数/运算符重载等变体，实际独立函数签名数（不计相同语义的重载变体）约 18 个。或将 §1 表格拆分为两行——「独立函数范式」与「含重载变体展开」——分别给出计数，让读者在同一位置即可获得两种口径的理解。

## 问题 2（中度）

**问题描述**：`normalize` 函数的整数 T 实例化时的边界行为在 §3.7 和 §5.3 中存在契约缺失。§3.7 明确 `normalize` 使用 `where T <: Number<T>` 约束（未收紧至 `FloatingPoint<T>`），允许整数四元数调用 `normalize`。但 §5.3 边界条件表对 `normalize` 的两行条目——「零四元数保护」和「length 极小值」——均假设浮点行为（Inf/NaN 传播）。对于整数四元数 `normalize`：
- 零保护分支：`T(Float64(0))` 字面量替代路径在 §1 H1 验证中已确认对 `Int8~Int64` 编译通过，这意味着 `Int32(Float64(0))` 返回 `Int32(0)`，行为已定义，不触发除零。
- 非零分支：`q / tmp1` 中的除法是整型除法（截断）。当 `tmp1 > 1` 时，整型除法可能导致零分量（如 `Int32(1)/Int32(2) = 0`），这意味着 `normalize(Quat<Int32, ...>(1, 0, 0, 1))` 可能返回 `Quat(1, 0, 0, 0)` 而非单位四元数，与浮点行为有本质差异。
§5.3 对这两种整数场景的行为契约均为空白——既未声明整型除法截断的预期行为，也未说是否应通过约束收紧在编译期拒绝整数 T。

**所在位置**：§3.7 `normalize` 函数描述（第 574 行附近）、§5.3 边界条件表「零四元数」行（第 1057 行）、「length 极小值」行（第 1058 行）

**严重程度**：中度

**改进建议**：在 §5.3 边界条件表新增「整数 T 实例化」行，明确 `normalize(q)` 在整数四元数上的行为契约——(a) 零保护分支行为与浮点一致（返回单位四元数）；(b) 非零分支采用整数除法，结果因截断可能不满足 `|q_normalized| = 1`，语义上为"与浮点行为最接近的整数分解"。或收紧 `normalize` 的 T 约束为 `FloatingPoint<T>`（与 `mat3Cast`/`mat4Cast` D32 策略一致），在编译期拒绝整数 T 实例化，消除整型除法的语义歧义。若选择收紧，需同步修订 §3.7 的约束描述和 §5.3 边界条件表。

## 问题 3（轻度）

**问题描述**：§3.4 运算符体系表中 `Quat/T` 二元除法运算符的整数除零行为未明确，与 §3.11 `inverse` 的整数除零行为描述（§5.3 第 1060 行「触发仓颉 ArithmeticException」）形成不对称。`inverse` 在 §3.11 和 §5.3 中明确区分了浮点除零（Inf/NaN）和整数除零（ArithmeticException），但 `Quat/T` 运算符作为成员运算符，仅在表行标注了 `@OverflowWrapping`。当 `T = Int32` 且 `divisor = 0` 时，`q / 0` 的行为无契约说明——`@OverflowWrapping` 注解覆盖的是整数溢出语义（模 2^n 回绕），不覆盖除零语义（ArithmeticException）。

**所在位置**：§3.4 运算符体系表「二元 / (Quat/T)」行（第 491 行）、§5.3 边界条件表

**严重程度**：轻度

**改进建议**：在 §3.4 `Quat/T` 行的「备注」列补充「整数 T 除零时触发仓颉 ArithmeticException，浮点 T 除零时产生 Inf/NaN 分量（与 `inverse` 行为一致）」。或新增 §5.3 边界条件表条目统一声明所有除法运算符（`Quat/T`、`T/Quat`、`inverse`）的整数除零行为。

## 问题 4（轻度）

**问题描述**：`slerp` 4 参数版本的 `k: Int64` 类型决策与 GLM 原始模式之间的潜在兼容性问题未在 §9 差异声明或 §7 D22 中记录。GLM 1.0.3 的 4 参数 `slerp` 使用独立泛型参数 `S`（通过 `static_assert(is_integer)` 约束整型），仓颉版本固定为 `Int64`。虽然文档在 §3.11 和 D22 中说明了此简化决策，但该决策意味着阶段四完整实现时，如果下游用户需要 `Int32` 或 `Int16` 类型的 `k`，当前的 `Int64` 签名将强制要求调用方做类型转换。这构成了一个与 GLM API 形态的偏差，应记录于 §9「差异声明」中以保持 GLM 参考实现的可追溯性。

**所在位置**：§3.11 `slerp` 4 参数版本描述（第 644 行）、§7 D22 设计决策（第 1123 行）、§9 差异声明表

**严重程度**：轻度

**改进建议**：在 §9 差异声明表中新增条目「`slerp` 4 参数版本 `k` 类型固定为 `Int64`（GLM 原始版本使用独立泛型整型参数 `S`）」，注明该偏差的设计理由（"与 DV-03 风格一致"）和阶段四可选的泛化路径（新增 `k: K where K <: Integer<K>` 重载，保留 `Int64` 重载向后兼容）。

## 问题 5（轻度）

**问题描述**：§8 编码启动前验证项列表（§8「编码启动前验证项」，行 1321-1395）中，验证项 29（glm.detail 导入 glm.ext 验证）的描述和验证目标与 §3.4 最新实施的内联 conjugate/dot 路径存在细微脱节。验证项 29 描述为「验证 detail/type_quat.cj 是否引入 glm.detail → glm.ext 跨包 import」，验证方式为"在 type_quat.cj 中 import glm.ext.quaternion_geometric.*，确认 cjpm build 报循环依赖错误"。但 §3.4 已在 v18 中确认 Vec3×Quat/Vec4×Quat 采用内联 conjugate/dot 计算路径（不依赖任何 glm.ext 符号），因此验证项 29 描述的"先引入跨包 import 再验证报错"的步骤与当前设计状态直接矛盾——当前设计已经没有跨包 import 可引入来触发循环依赖错误。

**所在位置**：§8 编码启动前验证项 29（第 1351 行）、§3.4 Vec extend 块描述（第 503-506 行）

**严重程度**：轻度

**改进建议**：将验证项 29 的验证方式从"先引入错误 import 再确认报错"修订为"确认 type_quat.cj/grep 验证无 import glm.ext.* 声明 + cjpm build 通过"。同步更新「验证失败后受影响章节」映射表中验证项 29 的条目。

## 整体质量评价

该文档经过 25 轮迭代，覆盖了阶段三 OOD 所需的完整维度：包组织结构、核心抽象定义、运算符体系、扩展函数库、依赖分析、边界条件契约、错误处理策略、测试设计、覆盖矩阵、编码启动前验证项等。大量依赖关系经过了 GLM 1.0.3 实际源码的交叉验证，多处公式和契约与 GLM 实现逐行对照。v25 对 H1/H2 核心假设的实际编译验证是对此前 13 轮迭代始终未关闭的技术风险敞口的正式关闭，具有实质性的质量改进意义。

主要的质量短板集中于两层之间：**顶层声明（§1）的宏观口径与中层审计（§3.13.2）的精确口径之间的沟通一致性**，以及**少数边界条件下（整数 T 的 normalize/除法行为）的契约完整性**。这两类问题不影响设计的技术正确性，但影响下游消费者对交付范围的准确理解和整数四元数用例的实现可靠性。
