# 质量审查诊断报告

**审查对象**：a_v17_copy_from_v16.md（阶段三 OOD 设计方案第 17 轮迭代）
**审查视角**：需求响应充分度、整体深度和完整性、实际落地可操作性
**审查日期**：2026-06-25

## 发现的问题

### 问题 1：修订说明章节排序严重混乱，破坏可追溯性

**问题描述**：文档末尾的修订说明按 v15 → v13 → v12 → v16 → v17 的顺序排列，完全不符合时间顺序（正确顺序应为 v12 → v13 → v15 → v16 → v17）。v15 的修订说明出现在 v13 和 v12 之前，读者按顺序阅读时会先看到 v15 的内容，然后倒退到 v13 和 v12，最后才是 v16 和 v17。该方法破坏了版本演变的时间线完整性，下游实施者若需追溯特定决策的演变历程（如 T(1) 获取策略从 v8 到 v14 的演变），在当前顺序下需要在前翻后跳，极易遗漏关键上下文。

**所在位置**：§修订说明（v15）行 1639 → §修订说明（v13）行 1659 → §修订说明（v12）行 1686 → §修订说明（v16）行 1713 → §修订说明（v17）行 1730

**严重程度**：严重

**改进建议**：
- 将修订说明按版本号升序重排：v12 → v13 → v15 → v16 → v17
- 或统一以倒序排列（最新在前）：v17 → v16 → v15 → v13 → v12，并确保一致性
- 在修订说明区域顶部增加索引行，标注各版本的页码位置

### 问题 2：§8.3 验收项 B 测试用例总数未按 v17 修订声明修正，修复未闭环

**问题描述**：v17 修订说明（行 1732）明确声明「§8.3 验收项 B 测试用例数计算行修正为 192（下限）」，但 §8.3 验收项 B 表（行 1240）的「数量」列仍写为「≥199」。虽然备注中有「192（下限）；通过增补 7 个边界用例可达 ≥199」的解释，但验收标准的数字列与 v17 修订说明的修正意图不一致。下游验收者以 §8.3 表的「数量」列为准时，会误以为验收底线是 ≥199 而非 ≥192，导致对测试数量的判断偏差。

**所在位置**：§8.3 验收项 B（行 1240）

**严重程度**：严重

**改进建议**：将 §8.3 验收项 B 的数量列从「≥199」修正为「≥192」，备注列保持「通过增补 7 个边界用例可达 ≥199」说明。确保与 §8.2 文件清单合计（行 1072）的 ≥192、分配表合计（行 1133）的 192 三者统一。

### 问题 3：`glm.detail.type_quat.cj` 对 `glm.ext.quaternion_geometric` 的依赖声明可能构成包间循环依赖

**问题描述**：§2 模块间依赖图（行 208）声明 `type_quat.cj → ext/quaternion_geometric（dot/length/normalize/cross）`。同时声明（行 217）`quaternion_geometric.cj → glm.detail（Quat）`。若前者成立，则形成 `glm.detail → glm.ext → glm.detail` 的包间循环依赖，违反仓颉 cjpm 构建系统对循环依赖的禁止约束。经 17 轮迭代仍未核实此依赖的事实可行性——`type_quat.cj` 中定义的 Quat 运算符（§3.4）是否真的需要调用 `glm.ext.quaternion_geometric` 中的函数，设计文档未明确说明。如果 `type_quat.cj` 确实需要跨包导入 `glm.ext`，则项目无法通过 `cjpm build`，整个阶段三的包结构设计存在根本性错误。

**所在位置**：§2 模块间依赖图（行 208 与行 217）

**严重程度**：严重

**改进建议**：
- 逐项核查 `type_quat.cj` 中每个函数/运算符是否实际需要调用 `glm.ext.quaternion_geometric` 中的函数
- 若不需要，删除行 208 中的 `type_quat.cj → ext/quaternion_geometric` 依赖声明
- 若确实需要（如 Vec3×Quat 运算符定义在 `type_vec3.cj` 中需要调用 `inverse`），则需重新设计包结构——选项包括：①将 Vec3×Quat 运算符移至 `glm.ext` 层定义；②在 `glm.detail` 层为 Quat 内联实现所需函数的简化版本；③将 `glm.detail` 和 `glm.ext` 合并为同一包
- 在 §8 编码启动前验证项中新增「`glm.detail` 包对 `glm.ext` 的导入可行性验证」（当前 28 项验证中无此项）

### 问题 4：§8.2 测试用例数三表之间仍存在口径不一致

**问题描述**：§8.2 中三处测试用例数的表述存在口径矛盾：
1. 文件清单表（行 1072）：合计「≥192」
2. 用例分配表（行 1133）：全表总计「≥192」
3. 各文件 ≥ 下限值之和（行 1072 每行 ≥值相加）：40+8+8+12+13+4+4+2+16+13+9+28+27+8 = **192**
但文件清单中 `test_ext_scalar_constants.cj` 标注「≥13」（行 1067），`test_ext_quaternion_aliases.cj` 标注「≥9」（行 1068），`test_ext_quaternion_trigonometric.cj` 标注「≥4」（行 1063）——这三项在分配表中对应的合计值分别为 13、9、4，与文件清单一致。所以目前三表口径在数字层面上已一致。问题在于：「≥192」中的「≥」意味着实际值可以大于 192，但 §8.3 验收项 B（问题 2）的「≥199」暗示至少需要 199。两处 ≥ 值的下限不同（192 vs 199），造成混淆。

**所在位置**：§8.2 行 1072、行 1133；§8.3 行 1240

**严重程度**：一般

**改进建议**：
- 统一全文档对测试用例基线的表述为「192（下限）」，将「≥199」改为「≥192」或「≥199（含增补）」
- 在 §8.2 末尾新增一行显式说明：「基础下限 192，可通过增补 7 个边界用例达 ≥199」

### 问题 5：Vec3×Quat/Vec4×Quat 运算符的定义位置未明确，影响实施者判断包依赖

**问题描述**：§3.4（行 415-420）列出了 Vec3×Quat 和 Vec4×Quat 运算符，标注为「定义在 Vec3/Vec4 extend 块中」，但未说明这些 extend 块位于哪个文件中。它们可能在：① `type_vec3.cj`/`type_vec4.cj`（`glm.detail` 包）；② `ext/quaternion_common.cj`（`glm.ext` 包）；③ 新增的独立文件。不同选择对应不同的包依赖结构——若在 `glm.detail` 中定义，则 `type_vec3.cj` 需要 import `glm.ext.quaternion_common.inverse`，与问题 3 的循环依赖风险相关。设计文档在迭代 17 轮后仍未明确此归属，实施者无法确定正确的包依赖声明。

**所在位置**：§3.4（行 415-420）、§2 模块间依赖图

**严重程度**：一般

**改进建议**：
- 在 §3.4「Vec extend 块成员运算符」表后新增一行说明：「定义位置：[明确文件名]，package [明确包名]」
- 在 §2 模块间依赖图中为 Vec3×Quat/Vec4×Quat 运算符新增依赖方向标注
- 若定义在 `glm.detail` 层，需同步在 §8 验证项中新增跨包导入可行性验证

### 问题 6：§11.5 函数可用性对照表的逐函数拆分不彻底，`gtc/matrix_transform` 行仍有隐含的层级结构

**问题描述**：v17 修订声明已将 §11.5 表所有分组行拆分为逐函数行（共 52 行）。但 `gtc/matrix_transform` 行（行 1598）仍以「全部 64 个函数」的聚合形式出现，而非逐函数列举。作为「单一权威来源」，此聚合行导致下游消费者无法在 §11.5 中逐项核验 64 个 stub 函数的个体可用性，仍需查阅 §3.13 的详细清单。这削弱了「本表是函数阶段状态追踪的单一权威来源」的宣称可信度。

**所在位置**：§11.5（行 1598）

**严重程度**：一般

**改进建议**：
- 将 gtc/matrix_transform 的 64 个函数拆分为逐函数行（或至少按 6 大类分组 + 每类内逐函数行）
- 或在 §11.5 表头添加覆盖范围声明：「本表不逐行列示 gtc/matrix_transform 的 64 个 stub 函数，其状态统一标记为 ❌ stub，详细清单见 §3.13 表格」

### 问题 7：编码启动前验证项缺少综合验证失败的快速修复指引

**问题描述**：§8 编码启动前验证项 1-28（行 1184-1211）逐项列出了需要验证的编译假设和可行性检查，但缺少「验证失败 → 需修改哪些章节」的快速映射表。例如验证项 25（`T(Float64(n))` 语法编译可行性）失败时，§1 回退方案子节虽提供了三条回退路径（A/B/C），但未列出这些路径分别需要修改的章节清单。实施者验证失败后必须手动翻查全文中所有受影响的函数描述（散布于 §3.2.1、§3.4、§3.7、§3.9、§5.1、§5.3、§8.3 等多处），逐一修订，极易遗漏。

**所在位置**：§8 编码启动前验证项 1-28、§1 回退方案子节

**严重程度**：一般

**改进建议**：
- 在 §1 回退方案子节的每条回退路径后，新增「影响章节清单」列，列出需同步修订的所有章节编号
- 或在 §8 编码启动前验证项 25/20 的末尾，补充标准格式的「验证失败后受影响章节清单」

## 总体评价

本设计文档经过 17 轮审查迭代，在技术细节的完整性、GLM 函数覆盖的全面性、边界条件的考量方面已达到较高水准。修订说明中对每一轮审查意见的逐项响应和落实位置标注体现出严谨的迭代治理过程。

然而，文档在以下三个方面仍存在不足：
1. **文档组织**：修订说明的排序混乱（问题 1）严重影响可追溯性，与「可直接指导编码实现」的要求有差距
2. **修复闭环完整性**：v17 已声明修复但正文未实际修改（问题 2），测试用例计数口径仍未完全统一（问题 4）
3. **核心依赖假设的可验证性**：`glm.detail → glm.ext → glm.detail` 的潜在循环依赖（问题 3）作为包结构的基础假设，在 17 轮迭代中始终未被单独验证，是本次审查发现的最严重风险

建议优先修复问题 1（文档组织）和问题 3（循环依赖风险），再处理其余一般问题。
