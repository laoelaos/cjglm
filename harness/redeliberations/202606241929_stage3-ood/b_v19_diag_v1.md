## 问题 1（严重 — 关键遗漏）

**问题描述**：§11.5 函数可用性对照表自称"单一权威来源"（行 1631-1633：「本表是函数阶段状态追踪的**单一权威来源**」），但缺少以下阶段三可用的关键 API 条目：

- `wxyz(w, x, y, z)` 工厂函数（§3.3 item 10，§10 覆盖矩阵行 1455 标注 ✅ 可用）
- `fromQuat<U, P>(conv, q)` 跨类型构造工厂函数（§3.3 item 4，§10 覆盖矩阵行 1449 标注 ✅ 可用）
- `[]` 下标运算符（§3.1，§10 覆盖矩阵行 1445 标注 ✅ 可用）
- `public static func length(): Int64` 编译期查询函数（§3.1，行 340）
- 复合赋值运算符 `+=`/`-=`/`*=`/`/=`（§3.4 "由编译器自动生成"）

这些函数/运算符均属于阶段三可用的公共 API，下游按 §11.5 表核验函数可用性时会遗漏覆盖。§10 覆盖矩阵虽然部分收录了这些条目，但 §11.5 的"单一权威来源"定位与事实收录范围之间存在显著差距。

**所在位置**：§11.5 函数可用性对照表（行 1637-1699）

**严重程度**：严重

**改进建议**：在下轮修订中将缺失条目补齐至 §11.5 表。具体方式：为 `wxyz`、`fromQuat`、`[]` 各新增一行并标注本阶段状态；`static func length(): Int64` 作为静态成员函数单独行标注；复合赋值运算符以一行聚合标注（"编译器自动生成，本阶段可用"）。确保 §11.5 真正达到"单一权威来源"覆盖所有阶段三可用 API 的定位。

## 问题 2（一般 — 逻辑矛盾）

**问题描述**：§3.13.2 审计结论与 §10 覆盖矩阵采用完全不同的函数分组计数口径，且两处之间不存在直接映射关系：

- §3.13.2 审计结论（行 776-778）将函数分为 3 类：18 个"真完整实现" + 2 个 ⚠️ + 78 个 ❌ = 98 个函数组
- §10 覆盖矩阵（行 1343-1358）按 GLM 头文件统计：约 77 个 ✅ + 87 ❌ + 1 ⚠️ = 165 个 GLM 函数
- 两处口径的 78 vs 87 ❌ 差值达 9 个，无法直接从 §3.13.2 溯源到 §10 的哪个头文件对应哪些遗漏

下游在核对测试覆盖完整性时，无法同时查阅两处口径确定"78 个 stub 签注是否完全覆盖了 87 个 ❌ 函数"。该问题在迭代过程中已部分被意识到（§3.13.2 行 747 声明"全文档函数可用状态以 §11.5 为最终基准"），但 §3.13.2 自身的计数与 §11.5 的按函数逐行计数仍存在口径差异（§3.13.2 的 18 个"真完整实现"在 §11.5 中被拆分为 30+ 行）。

**所在位置**：§3.13.2 审计结论（行 776-778）与 §10 覆盖矩阵合计行（行 1358）

**严重程度**：一般

**改进建议**：在 §3.13.2 审计结论末尾或 §8.3 验收项 C 下方新增一个"计数口径映射表"，将 §10 的 12 个 GLM 头文件分别映射到 §3.13.2 的三类分类，使下游能一眼看出"这 87 个 ❌ 分别落在哪些头文件"。同时修订 §3.13.2 中 78 这个数字，改为与 §11.5/§10 对齐的 87 ❌（或明确标注"78 是 Cangjie 文件级的函数组计数，87 是 GLM 源码级的函数签名计数，两处口径不同"）。

## 问题 3（一般 — 可维护性与信息纵贯度）

**问题描述**：§3.10 `pow` 函数的依赖描述（行 594-598）包含过量的 GLM 实现细节，包括：

- 具体的 C++ 代码翻译（GLM `wxyz` 仓库翻译为 Cangjie 等价代码）
- 具体的 GLM 源码行号引用（line 52/63/65/68/78）
- 逐行的 C++ → Cangjie 变量名对照说明

这些内容属于**编码实现笔记**而非 OOD 设计规范。当阶段四实际实现 `pow` 时，实际 Cangjie 实现路径可能与当前的 GLM 翻译路径存在偏差（例如选择不同的浮点类型转换路径），届时设计文档中的 GLM 逐行分析细节将成为过时信息。

此问题已在迭代 15（v15）中被部分解决——D21 设计决策的执行描述被精简为"完整依赖清单及仓颉等价翻译路径见 §3.10"（行 1041-1042），但 §3.10 正文中的实现细节未被精简。§3.10 的定位似乎是"设计规范"与"编码笔记本"的混合体。

**所在位置**：§3.10 `pow` 函数依赖描述段（行 594-598）

**严重程度**：一般

**改进建议**：将 §3.10 `pow` 函数体内部依赖于 GLM 的实现细节（如 line 65/78 的 C++ 翻译、具体的 Cangjie 等价代码片段）剥离至独立文件（如 `notes/quaternion_exponential_implementation_notes.md`），§3.10 仅保留依赖函数清单（哪些需 stub、哪些来自 scalar_constants.cj）、约束标注和策略决策（stub 占位）。若保留这些实现笔记作为参考，应在段落前标注"以下为阶段四实施参考笔记，非阶段三设计规范"。

## 问题 4（一般 — 假设未重新核验）

**问题描述**：§8 编码启动前验证项 20（P0）对 `FloatingPoint<T>` 接口方法的依赖描述引用了 `cangjie-original-docs/std/math/math_package_api/math_package_interfaces.md`，但 v19 修订本身在行 1261 声明："v13 中对该文档的引用内容未纳入本轮 v14 审查范围的独立复核……下游编码者在实际调用 `FloatingPoint<T>.getMinDenormal()`/`getInf()` 等方法前应以仓颉编译器实际签名确认。"

这意味着自 v11 修订（该轮对 `FloatingPoint<T>` 接口进行了独立核实）以来，后续 7 个迭代版本（v12-v19，跨越近大半个设计周期）均未重新核验该假设的持续正确性。若该版本使用的 Cangjie stdlib 版本与文档引用的文档版本不一致，`FloatingPoint<T>.getMinDenormal()` 或 `FloatingPoint<T>.getInf()` 的 API 形态可能已变化。

该假设若被证伪，影响范围覆盖 §3.10 `pow`/`log`、§3.11 `isnan`/`isinf`、§10 覆盖矩阵等多个章节。

**所在位置**：§8 编码启动前验证项 20（行 1261）、§1 回退方案 H2 子节（行 101-108）

**严重程度**：一般

**改进建议**：在编码启动前验证项 20 中补充"Cangjie stdlib 版本号记录"，说明当前参考的 stdlib 文档对应哪个具体的 cjpm 工具链版本，并在执行 P0 验证前先用一个最小测试文件确认当前 Cangjie 编译器环境下 `FloatingPoint<T>` 的实际 API 形态。

## 问题 5（轻微 — 标注歧义）

**问题描述**：§11.5 函数可用性对照表中，`fromMat3`/`fromMat4` 行标注了"约束：`where T <: FloatingPoint<T>`"（行 1640-1641），`mat3Cast`/`mat4Cast`/`quatCast` 行也标注了相同的约束（行 1687-1690）。但前者的约束是**传递性继承**自内部 callee `quatCast`（即 `fromMat3` 的签名显式声明了 `FloatingPoint<T>` 约束，因为其函数体调用 `quatCast`），而后者是**直接声明**在签名上。

两种约束来源在 §11.5 中使用完全相同的标注形式（"约束：`where T <: FloatingPoint<T>`"，且均引用 D32），下游读者无法区分哪一行是直接约束、哪一行是传递性约束。这种差异在 `fromMat3`/`fromMat4` 的约束传递性注记在 v13 才被补充（§13 问题 2），但标注的呈现形式未做区分。

**所在位置**：§11.5 `fromMat3`/`fromMat4` 行（行 1640-1641）与 `mat3Cast` 等行（行 1687-1690）

**严重程度**：轻微

**改进建议**：将 `fromMat3`/`fromMat4` 的约束注记修订为"约束：`where T <: FloatingPoint<T>`（**传递性**——签名显式声明，继承自内部 callee `quatCast` 的 D32 约束）"，与 `mat3Cast` 的"约束：`where T <: FloatingPoint<T>`（D32，直接声明于函数签名上）"形成区分。
