## 问题 1（严重）：trigonometric.cj 75 个 stub 函数签名展开仍需下游手工推断

**问题描述**：v28 已为 gtc/matrix_transform.cj 的 64 个函数提供完整可直接复制的签名模板列表（§3.13 代码块 1-9，合计 11+10+9+9+9+7+2+6+1 = 64 行签名），但 trigonometric.cj 的 75 个展开函数仍以「sin + atan2 两个示例 + 其余 13 个按相同模式展开」的方式处理。下游实现 trigonometric.cj 时，需手工将 13 个单参数三角函数（cos/tan/asin/acos/atan/sinh/cosh/tanh/asinh/acosh/atanh/radians/degrees）逐一按标量 + Vec1~Vec4 展开。任何一处模式应用错误（where 子句格式、约束声明、泛型参数命名的细微偏差）只能在编码阶段通过编译错误发现。

两种同类 stub 函数组的处理方式不一致：gtc/matrix_transform 提供了「直接复制粘贴」级别的完整签名列表，trigonometric.cj 仍处于「参考模式自行展开」级别。

**所在位置**：§3.13.1 完整签名模板示例段（第 890-915 行）；§3.13.1 三角函数表（第 863-884 行）

**严重程度**：严重

**改进建议**：
1. 仿照 gtc/matrix_transform 的处理方式，为 14 个单参数函数 + 1 个双参数函数提供展开后全部 75 个签名模板（可直接复制粘贴到文件中），至少为每个函数类别（双曲类、反三角类等）提供 1 个完整展开示例
2. 若因篇幅约束不宜展开全部 75 行，可新增附录文件（如 `a_v28_trigonometric_signatures.md`）承载完整签名列表，在正文中引用

## 问题 2（严重）：§11.5 的"权威来源"定位与实际覆盖范围不匹配

**问题描述**：§11.5 自称"函数级状态追踪的权威来源"（第 1959 行），实际仅覆盖约 70+ 个 API 的逐函数行。以下 139 个 API 在 §11.5 中仅有聚合行而无逐函数条目：

- trigonometric.cj 75 个函数（聚合行 `trigonometric.cj 全部 75 函数 ❌`）
- gtc/matrix_transform 64 个函数（聚合行 `gtc/matrix_transform 全部 64 函数 ❌`）

合计 139 个函数仅以 2 行聚合行表示，占全部 165 个对标 API 的 84%。下游消费者无法从 §11.5 获取这些函数的逐函数状态，必须跳转到 §3.13.1 或 §9a 补充查阅，违背"权威来源"的设计目标。

此外，§11.5 合计行的计数口径（✅ ~18 个独立函数 + ⚠️ 1 + ❌ ~87 = 88）与 §9a 合计行（约 77 ✅ + 88 ⚠️/❌ = 165）使用不同口径——前者按独立函数名计数，后者按含重载/常量的签名行数计数。两处口径差异不是简单数字差（18 vs 77 差距很大），且 §11.5 未说明如何映射。

**所在位置**：§11.5 函数可用性对照表（第 1961-2029 行）、§11.5 合计行（第 2029 行）

**严重程度**：严重

**改进建议**：
1. 在 §11.5 表头新增覆盖范围声明及与 §9a 的分工说明，明确「本表提供核心 26 个函数的逐函数状态；trigonometric.cj 和 gtc/matrix_transform 的逐函数状态以 §9a 和 §3.13.x 为权威来源」
2. 在合计行旁标注口径差异：「§11.5（独立函数口径）：18 ✅ + 1 ⚠️ + 87 ❌ = 106；§9a（含重载/常量口径）：77 ✅ + 88 ⚠️/❌ = 165」
3. 或将 §11.5 扩展为完整 165 个 API 的逐函数表（可参照 gtc/matrix_transform 的聚合处理方式为 trigonometric.cj 新增代表性签名行的引用）

## 问题 3（一般）：跨 Qualifier 行为契约仍未系统性声明

**问题描述**：上一轮审查报告（b_v28_diag_v1 Problem 2）已指出 `dot`/`length`/`normalize`/`cross`/`axis`/`inverse`/`conjugate`/运算符等函数的跨 Qualifier 行为未声明。v28 未做任何补充。仅 §3.11 `lerp`/`inverse` 的 D34 涉及跨 Qualifier 讨论（且 D34 明确将双 Q 重载推迟至阶段四），其他所有泛型函数和运算符的跨 Qualifier 行为均无任何前置说明。

下游编码者实现 `dot(highp_q, mediump_q)` 或 `axis(highp_q)` 等调用时将遭遇编译错误（Q 泛型参数不匹配），但文档未声明此为预期行为。编码者将花费不必要的时间排查不存在的实现 bug。

**所在位置**：§3.7（dot/length/normalize/cross）、§3.9（axis）、§3.11（inverse/conjugate）、§3.4（运算符）

**严重程度**：一般

**改进建议**：
1. 在 §1 或 §5 末尾新增跨 Qualifier 行为的统一契约声明：「本阶段所有泛型函数和运算符（包括 dot/length/normalize/cross/axis/inverse/conjugate/算术运算符/比较运算符）仅支持相同 Qualifier 的操作数。跨 Qualifier 调用因泛型参数 Q 不匹配触发编译错误。跨 Q 兼容性扩展（双 Q 重载）计划在阶段四按需补充（参见 D34）。」
2. 在 §5.3 边界条件表新增一行「跨 Qualifier 调用 → 编译期拒绝（预期行为）」条目

## 问题 4（一般）：`axis` 函数的类型约束（where 子句）未显式声明

**问题描述**：§3.9 描述 `axis` 的实现公式调用 `std.math.sqrt(tmp1)`，后者仅对浮点类型提供重载。但 §3.9 未显式列出 `where T <: FloatingPoint<T>` 约束，与同等依赖 std.math 的函数（§3.2.1 mat3Cast/mat4Cast 显式标注、§3.7 normalize 在 v26 收紧为 FloatingPoint<T>）形成对比。

下游编码者若按"axis 是完整实现"的结论开始编码，且依据命名推断约束为 `Number<T>`，则 `Quat<Int64, Q>.axis()` 实例化时在 `std.math.sqrt(tmp1)` 处出现编译错误，错误位置不直接指向约束缺失。

**所在位置**：§3.9 axis 函数描述（第 620-630 行附近）

**严重程度**：一般

**改进建议**：在 §3.9 axis 函数签名描述中显式标注 `where T <: FloatingPoint<T>, Q <: Qualifier`，与 §3.2.1/§3.7/§3.11 同类函数一致。同步在 §11.5 axis 行补充约束注记。

## 问题 5（一般/风险备忘）：`fromMat4` 对 stage 2 Mat4x4 字段名的依赖缺乏解耦机制

**问题描述**：§3.3 item 7 的 `fromMat4` 实现直接引用 `m.c0`/`m.c1`/`m.c2`（Mat4x4 列字段名）。验证项 27 已确认当前字段名正确。但若 stage 2 在后续迭代中重构字段名（如改为 `col0`/`col1`/`col2`），`fromMat4` 将编译失败且错误信息不直接指向字段名假设。

**对质询的回应**：本问题从上一轮的「一般」降级为「风险备忘」——质询报告指出其建立在假设性场景上，stage 2 当前无重构计划。保留为风险备忘以标记该依赖关系，供阶段四演进时参考。

**所在位置**：§3.3 item 7（第 485 行）；验证项 27

**严重程度**：一般（风险备忘）

**改进建议**：在 §3.3 item 7 末尾追加显式声明：「本实现依赖 Mat4x4 的列字段名为 `c0`/`c1`/`c2`/`c3`。若 stage 2 重构此命名，需同步修改本函数，或改为通过 `[]` 下标运算符访问以解耦。」同时在验证项 27 末尾标注该字段名依赖为"阶段三→阶段四的持续契约"。

## 问题 6（一般）：`inverse` 对整数 T 的行为契约不完整

**问题描述**：§3.11 标记 `inverse` 为「完整实现」，签名未显式标注约束（推断为 `Number<T>` 宽约束）。§5.3 边界条件表分别定义了浮点（产生 Inf/NaN）和整数（触发 ArithmeticException）两种零除行为。但 `inverse` 中 `conjugate(q) / dot(q, q)` 的除法在整数 T 下是整数除法——非零除场景下（如 `dot(q,q)=2`），整数除法截断会导致语义上不正确的逆四元数（如 `(1,0,0,2)/2 = (0,0,0,1)` 而非 `(0.5,0,0,1)`）。

与 `normalize`（v26 已收紧为 `FloatingPoint<T>`）比较：两者均涉及除法且对整数 T 产生无意义结果，但 `inverse` 保持 `Number<T>` 宽约束，`normalize` 已收紧。不一致性文档未解释。

**所在位置**：§3.11 inverse 描述（第 654-656 行）、§3.7 normalize 描述（第 591 行附近）、§5.3 边界条件表 inverse 行

**严重程度**：一般

**改进建议**：
1. 明确 `inverse` 的约束策略——要么收紧为 `FloatingPoint<T>` 与 normalize 一致（消除整数除法截断问题），要么保留 `Number<T>` 并在 §5.3 补充整数非零除时的截断行为契约
2. 在 §3.11 inverse 描述末尾补充与 normalize 的约束差异说明及理由

## 总结

v28 在以下方面有显著改进：前置风险声明（§1）、路线图偏差量化（§3.16 D35-D37 量化影响 + 降级路径一览表）、gtc/matrix_transform 64 个签名模板全覆盖（§3.13）、编译期签名验证覆盖维度（§8.2 第 9 类）、O-06 可执行落地计划（§8.3 I 节）、版本标注清理（约 50+ 处）。验证项 20/25/29 标记为已完成。

本审查识别出 **6 项问题**（2 严重 + 4 一般/风险备忘）。核心发现：v28 修复了多数历史严重的质量和计数问题，但文档的"可信权威来源"体系仍存在结构性问题——§11.5 的覆盖缺口（139/165 个 API 仅有聚合行而非逐函数行）和 trigonometric.cj 签名模板的不完整展开，使下游编码者仍需在多处跳转后才能获得编码所需的完整信息。
