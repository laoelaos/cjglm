# 质量审查报告：a_v24_copy_from_v23.md

## 审查范围

审查视角：需求响应充分度、整体深度和完整性、从实际落地角度看设计是否可直接指导编码实现。重点覆盖内部审议（组件A设计-验证循环）可能未充分覆盖的维度。

## 发现的问题

### 问题 1 — 核心约束尚未验证：H1/H2 两大基础假设历经 24 轮迭代仍未闭环

**问题描述**：文档 §1「回退方案」明确声明两项 P0 基础假设（H1：`T(Float64(n))` 语法编译可行性；H2：`FloatingPoint<T>` 接口方法可用性）"尚待编码启动前验证"。这两项假设是整个设计的技术地基——H1 影响 `normalize`/`axis`/`mat3Cast`/`mat4Cast`/`Quat×Vec3`/`Quat×Vec4` 共 6 个函数的零值/单位值获取策略；H2 影响 `pow`/`log`/`isnan`/`isinf`/`mat3Cast`/`mat4Cast`/`quatCast`/`trigonometric.cj` 全体的实现路径。若其中任一项验证失败，需要触发的**回退路径波及范围可能覆盖设计文档中 15 个以上的章节**（参见 §8.3 反馈结果记录表模板中的「失败后处置路径」列），且回退方案本身（如追加 `one: T` 形参）将导致 API 签名偏离 GLM 标准形态。

**风险放大因素**：
- 文档自承该假设自 v11 轮独立核实以来「历经 v12-v19 共 8 个迭代版本未重新核验」（§8 验证项 20 注记），v24 版本仍然仅添加了"验证时限承诺"（5 个工作日验证）而未实际执行验证。这意味着该风险敞口自 v11 版本以来延续了 13 个迭代轮次。
- H1/H2 的验证在编码阶段启动后才能执行，但设计文档本身依赖这些假设来推导 API 签名、约束策略和函数体实现路径——若验证失败，重新设计将触发「编码→发现基础假设证伪→回溯修订设计」的逆向成本，而这种逆向成本在 24 轮迭代的体量下将显著高于「先在设计阶段建造最小仓颉测试文件验证」的前置验证方法。

**所在位置**：§1「回退方案」节、§8.3 H 节验证项 20/25、§8 验证结果记录表模板

**严重程度**：严重

**改进建议**：在输出最终设计版本前，**先建造最小仓颉测试文件（2 个文件，各 ~20 行）**，将 H1 和 H2 从「待验证假设」升级为「已验证事实」。具体来说：
- 测试文件 A：实例化 `Float32(Float64(0))`/`Float64(Float64(0))`/`Int32(Float64(0))`/`Int64(Float64(1))`——直接验证 H1 的全类型编译可行性
- 测试文件 B：声明 `where T <: FloatingPoint<T>` 并调用 `FloatingPoint<Float32>.getInf()`/`.getMinDenormal()`/`.getNaN()`——直接验证 H2 的接口方法可用性
- 将验证结果（通过/失败及失败详情）记录于 §8 验证结果记录表中，替代当前的"待验证"占位符状态。若验证失败，编码前即可触发回退方案决策树，规避编码阶段的逆向修复成本。

### 问题 2 — 需求响应充分度：约 53% 的 API 以 stub 占位，但未作为"重大范围假设"突出标注

**问题描述**：根据 §9a 基准声明，GLM 1.0.3 Stage 3 共涉及 13 个头文件、165 个 API 函数。本设计在本阶段完成的「真完整实现」仅约 77 个（✅），其余 88 个（约 53%）以 stub（❌）或「抛 stub 异常」（⚠️）状态占位。从用户需求（"把 references/glm-1.0.3 用仓颉重新实现" + "迁移阶段三"）的预期来看，**超过一半的 API 不能调用**是一个超出常规预期的范围削减，但该信息散布在 §9a、§10、§11.5 三张表中，缺乏显眼的**前置风险声明**。首次阅读者在 §1 概述看到"四元数类型…使库具备旋转/插值表达能力"（第 12 行）时，不会预期到 slerp、mix、pow、exp、log、sqrt、angle、angleAxis、rotate、欧拉角、看向函数等核心旋转/插值 API 全部为 stub。

**所在位置**：§1 概述（第 12 行）、§9a 合计行、§10 各文件覆盖矩阵

**严重程度**：一般

**改进建议**：在 §1「设计目标」段末尾（第 13 行之后）插入显式的**阶段三 API 可用率声明**段落，例如：
> **阶段三 API 可用率**：165 个对标 API 中 77 个（约 47%）本阶段可完整调用（✅），88 个（约 53%）因依赖 trigonometric.cj/common.cj/geometric.cj 等 stub 文件需推迟至阶段四。核心旋转/插值 API（slerp、mix、pow、exp、log、sqrt、angle、angleAxis、rotate、欧拉角提取、看向函数）均为 stub 状态，阶段三不可用。详细逐函数状态见 §11.5。

同时，§1 概述第 12 行的"使库具备旋转/插值表达能力"措辞需要同步修订——当前阶段三不提供 slerp/mix/pow 等关键插值能力，声称具备"插值表达能力"在交付时会形成过度承诺。

### 问题 3 — 三表冗余（§9a / §10 / §11.5）的根本问题尚未解决

**问题描述**：v24 修订（Issue 8 响应）将 §11.5 的定位从「单一权威来源」改为「函数级权威来源，与 §9a/§10 互补」，表面上解决了自相矛盾，但**根本的冗余问题未触及**：
- §9a「GLM 1.0.3 Stage 3 API 基准声明」（第 1569-1603 行）与 §10「GLM 1.0.3 API 阶段覆盖矩阵」（第 1613-1745 行）覆盖的是同一组 API 函数，以不同的维度交叉呈现（§9a 按头文件聚合计数 + 推迟理由；§10 按文件分组 + 设计章节映射）。两表的合计计数（77 ✅、88 ⚠️/❌）完全一致，数据冗余度接近 100%。
- §10 本身还包含逐文件的状态行，与 §11.5 的逐函数行也存在粒度重叠（§10 的部分文件逐行列出每函数的状态，直接与 §11.5 的逐函数表撞车）。
- 三表共存的情况下，下游消费者维护一致性需要三处同步更新，而 §10 的合计计数在 v13→v19 间经历了 8 轮迭代的持续性偏差（实际 88 但声明 80），恰恰说明**多表冗余直接导致了维护负担和不一致风险**。

**所在位置**：§9a 全节、§10 全节、§11.5 首段定位声明

**严重程度**：一般

**改进建议**：
- 方案 A（推荐）：将 §9a 与 §10 合并为一张表——以 §9a 的「头文件→函数总数→推迟理由」为主体，以 §10 的「设计章节映射」为附加列，形成单一覆盖矩阵。删除冗余的 §10 独立表格。
- 方案 B（最小改动）：至少将 §10 中的重复状态列移除，改为「参见 §9a」引用；§10 仅保留「设计章节映射」功能，不做状态统计的二次声明。同时删除 §10 顶部的合计行，该合计行已由 §9a 提供。
- 无论哪个方案，同步更新 §8.3 H 节的核验项 H1/H2，简化核验内容。

### 问题 4 — 修订历史占比过高，影响文档可用性

**问题描述**：v12~v24 共 13 个修订说明章节占据约 500 行（占全文 2310 行的约 ~22%）。这些修订记录详细记录了每个问题的历史修复过程，但产生了以下可用性问题：
- 首次阅读此文档的下游编码者需要在 2310 行中辨别"哪些是当前有效设计"和"哪些是历史修订"。尤其是修订说明中的「问题描述」列描述的是**前序版本的错误**，新读者可能误读为当前文档仍存在的问题。
- §修订说明(v12)~§修订说明(v24) 的读者价值极低——下游编码者不需要知道 v12 的 `mat3_cast` 命名如何演变为 v24 的 `mat3Cast`，只需要知道当前是 `mat3Cast` 即可。将修订历史"当面展示"给所有读者相当于把整个设计过程的 24 轮审计追踪绑在交付物上强迫每个人阅读。
- 类似问题在成熟工程文档实践中通常的做法是维护独立的变更日志（CHANGELOG.md 或 git log），而非将逐轮修订细节内嵌在文档正文中。

**所在位置**：§修订说明(v12) 至 §修订说明(v24) 共 13 个子节，约第 1991-2310 行

**严重程度**：一般

**改进建议**：将所有修订说明（v12-v24）从设计文档正文剥离至独立文件（如 `a_v24_revision_history.md`），设计文档正文仅保留一条简洁的变更摘要行：
> 修订历史：详见 `a_v24_revision_history.md`（v12-v24 共 24 轮迭代，累计修复约 85 项问题列表）。

这样做的好处：
- 设计文档正文从 2310 行缩减至约 1800 行，核心设计内容占比从 78% 提升至近 100%
- 下游编码者"即开即用"，无需穿越历史信息寻找当前设计状态
- 审计追踪完整保留在独立文件中，不影响文档主体的可读性

### 问题 5 — 系统性问题的循环复发记录表明设计质量流程存在根本缺陷

**问题描述**：从迭代历史可以识别出以下问题类别在多个迭代轮次中反复出现：

| 问题类别 | 检出轮次 |
|---------|---------|
| 计数偏差（函数数/别名数/用例数） | 轮 3、5、10、16、17、19、20 |
| 版本号混乱 | 轮 10、14、15、17、18、22、23 |
| §11.5 缺失 API 条目 | 轮 1、13、17、20 |
| 公式与 §1 约束策略不一致 | 轮 3、6、11、12、22 |
| 修订说明排列顺序错误 | 轮 13、15、16、17、18 |
| 虚假修复率声称（后已删除） | 轮 15~23 共 8 轮 |

其中计数偏差问题在轮 19 被标注为"终极修复"后，轮 20 又检出同类问题（§3.13.2 审计计数 78 vs §10 合计 88 的差值）。「版本号混乱」问题横跨轮 10→24 共 14 个迭代轮次才在 v23 引入 H3 核验项。这些反复模式表明：

- 当前设计迭代流程是"问题驱动"而非"质量内置"的——每个问题被修复后，同类问题在后续轮次中再次因相同根因产生，说明修复动作没有转化为预防性质量控制措施。
- H1/H2 核验项要求在 v11 引入后经历了 13 轮均"未重新核验"，说明缺乏"必须执行核验才能推进"的流程纪律。
- 虚假 100% 修复率声称持续 8 轮，说明设计团队的自我评估机制在多个轮次中未能被审查者有效质疑。

**所在位置**：全文档分散；系统性问题模式体现在迭代历史各轮次的重复问题描述中

**严重程度**：严重

**改进建议**：引入以下三项结构性预防措施（非针对某个具体问题，而是针对质量流程本身）：
1. **自动化一致性检查**：将 H1/H2/H3 核验项脚本化并纳入文档输出流程，在版本发布前自动执行。包括但不限于：版本号 grep 检查（H3）、合计计数交叉核验（H1）、全文 snake_case 残留检查。
2. **核验前置化**：将核心假设（H1/H2）的验证从"编码阶段开始后 5 个工作日内"提前至"设计版本固化前"。要求在 OOD 文档输出前，先输出最小仓颉测试文件的编译通过结果作为附件。此项改动可阻断 H1/H2 从 v11→v24 共 13 轮的未验证状态。
3. **质量门禁**：设计文档输出前设置「已知未解决问题列表」的门禁条件——只有当 I 节列表为空时，文档版本号才可以标记为"final"或"ready for coding"。当前 v24 的 I 节仍有 O-03/O-04 未闭环，不应被视为"可交付编码"的状态。

## 整体评价

本设计文档在**技术深度**和**跨节一致性**方面达到了优秀水平——24 轮迭代中累计修复约 85 项审查发现的问题，在函数依赖分析、边界条件定义、包间循环依赖消除等核心工程维度上建立了系统性的方法论。文档的交叉引用网络（§3.x ↔ §5.3 ↔ §3.13.2 ↔ §11.5 ↔ §8）为下游提供了充分的追溯能力。

然而在**需求响应透明度**和**基础假设验证**两个维度上存在未闭环的结构性风险：约 53% 的 API 在阶段三不可用但未前置声明，两个 P0 基础假设历经 13 轮未实际验证。这两个风险的共同根因是——设计过程高度依赖"先设计后验证"范式，核心假设的验证被推迟到编码阶段，导致设计文档的技术精确性与实际可执行性之间存在断层。

建议在主 Agent 推进编码前，优先关闭问题 1（基础假设验证）和问题 5 中提到的质量门禁要求。
