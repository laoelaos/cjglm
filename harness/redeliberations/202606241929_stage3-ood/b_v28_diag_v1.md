## 问题 1（严重）：P0 验证声明的可证实性与自洽性不足

**问题描述**：文档在 §1 回退方案段（验证执行记录）和 §8 验证项 29 处声明所有 P0 核心假设（H1 `T(Float64(n))` 语法可行性、H2 `FloatingPoint<T>` 接口方法可用性、Item 29 `glm.detail` 对 `glm.ext` 无反向依赖）已在 v25/v28 通过编译验证，但验证方法描述存在两处自洽性问题：

(a) **验证项 29 的验证方法出现循环引用**：第 135 行描述验证方式 (a) 为「经审核 `type_quat.cj` 中 Vec3×Quat/Vec4×Quat 已采用 v18 内联 conjugate/dot 计算路径，无 `glm.ext` 跨包 import」——此处「已采用 v18 内联计算路径」是**设计文档自身的决策描述**，而非对已有代码的独立检查。OOD 阶段 `type_quat.cj` 尚未编写阶段三的 Vec3×Quat/Vec4×Quat 运算符（stage 2 的 `type_quat.cj` 不存在四元数相关内容），所谓「grep `type_quat.cj` 确认无 import」核查的是 stage 2 既有代码而非 stage 3 新增内容，无法验证 stage 3 设计是否引入了 `glm.detail → glm.ext` 跨包调用。

(b) **验证项 29(c) 的 `cjpm build` 编译对象不明确**：第 135 行描述「执行 `cjpm build` 确认编译通过」——未指明是编译已有 stage 2 项目（验证回归）还是编译包含 stage 3 原型的项目。若为前者，对 stage 3 循环依赖风险的验证力为零；若为后者，则意味着 stage 3 已在 OOD 完成前开始编码实现，与 OOD 文档的定位矛盾。

(c) **迭代历史矛盾未解释**：迭代历史第 27 轮（倒数第二份记录）明确记载「编码启动前验证项 29（P0）仍未完成，P0 通过条件不满足即输出 v27」，而 v28 声称已通过验证。文档未说明从「v27 未完成」到「v28 已完成」之间发生了什么具体验证活动（测试文件内容、编译命令输出、验证负责人等），缺乏独立可复现的验证证据。

**影响**：如果 P0 假设在编码阶段被证伪，整个设计的大面积函数（normalize/axis/mat3Cast/mat4Cast/Quat×Vec3/Quat×Vec4/isnan/isinf/epsilon/pi 等所有使用 `T(Float64(n))` 或 `FloatingPoint<T>` 约束的函数）的实现策略失效。设计声称「已验证」但验证过程不可独立复现，下游实现者无法确认风险是否已真正关闭。

**所在位置**：§1 回退方案段「验证执行记录」（第 132-135 行）；§8 验证结果记录表模板

**严重程度**：严重

**改进建议**：
1. 提供验证项 29 的独立验证证据——要么提供包含 stage 3 修改的最小原型项目路径及 `cjpm build` 输出，要么将验证方式从「设计审查」降级为「待编码启动前验证」并移入待验证清单
2. 对验证项 20/25，将验证用例文件（`test_h1_t_float64_n.cj`、`test_h2_floatingpoint.cj`、`test_h2b_direct.cj`）的内容直接附在文档中或提供可访问路径，使验证可独立复现
3. 明确标注「H2 验证发现 `FloatingPoint<Float32>.getInf()` 直接接口调用失败，仅 `T.getInf()` 泛型路径可用」这一重要发现的后续影响——是否对 `where T <: FloatingPoint<T>` 约束下的静态方法调用模式在所有函数上均成立，还是可能存在某些函数（如 `epsilon<T>()` 的无参调用模式）会受此限制

## 问题 2（一般）：横向 Qualifier 兼容性未做系统性契约声明

**问题描述**：文档仅在 §3.11 `lerp` 描述和 D34 决策中讨论了跨 Qualifier 行为（提供同 Q 签名模板，跨 Q 重载延至阶段四），但以下函数的跨 Qualifier 行为未做任何声明：

- `dot`/`length`/`normalize`/`cross`（§3.7 quaternion_geometric.cj）
- `inverse`/`conjugate`（§3.11 quaternion_common.cj）
- `axis`（§3.9 quaternion_trigonometric.cj）
- `mat3Cast`/`mat4Cast`/`quatCast`（§3.2.1 type_quat_cast.cj）
- 四元数算术运算符（§3.4）
- 4 个 gtc 比较函数（§3.15）

下游消费者对 `dot(q_highp, q_mediump)` 调用会产生编译错误（因 Q 泛型参数不匹配），但文档没有在任何位置解释这种行为。对比之下，`lerp` 有 D34 解释跨 Q 方案，而其他函数同样需要跨 Q 场景但完全未提及。

**影响**：下游编码者按设计实现时，调用 `dot(highp_q, mediump_q)` 将遇到编译错误。若文档不提前声明，开发者可能误以为是实现有 bug，花时间排查不存在的实现问题。

**所在位置**：§3.7（dot/length/normalize/cross）、§3.9（axis）、§3.11（inverse/conjugate，对 cross-Qualifier 无声明）、§3.4（运算符）

**严重程度**：一般

**改进建议**：
1. 在 §1「系统性设计约束」段末尾或 §2 模块间依赖图总览处新增一条统一的跨 Qualifier 契约声明，例如：「本阶段所有泛型函数（运算符/转换/几何/公共函数）仅支持相同 Qualifier 的两个操作数。跨 Qualifier 调用（如 `dot(Quat<T,Highp>, Quat<T,Mediump>)`）因泛型参数 Q 不匹配而触发编译错误。跨 Qualifier 兼容性扩展（双 Q 重载）将在阶段四按需补充（参见 D34）。」——该声明可作为全文档所有函数的默认行为，无需逐函数重复。
2. 同时在 §5.3 边界条件表末尾新增一行「跨 Qualifier 调用」说明。

## 问题 3（一般）：`fromMat4` 的字段名依赖未提供阶段二重构时的保护机制

**问题描述**：§3.3 item 7 的 `fromMat4` 实现直接引用 `m.c0`/`m.c1`/`m.c2`（即 Mat4x4 结构的列字段名）。验证项 27 已将此依赖标记为「已核验确认」（基于对 stage 2 `type_mat4x4.cj:8-11` 的检查）。然而，stage 2 和 stage 3 可能在同一个项目迭代周期内并行开发或被不同开发者维护，如果 stage 2 的 Mat4x4 列字段名在后续重构中发生变更（如改为 `col0`/`col1`/`col2` 以对齐 GLM 命名惯用），`fromMat4` 将编译失败且失败信息不会直接指向字段名假设——而是显示模糊的类型成员不存在错误。

文档当前虽已识别该依赖（验证项 27），但未提供任何 fallback 路径或解耦策略将此依赖设计为「显式契约」而非「隐藏假设」。

**影响**：stage 2 与 stage 3 的耦合性超出必要范围。字段名变更将导致 `fromMat4` 静默失效，失效定位成本高（编译错误发生在表达式层面而非 import 层面）。

**所在位置**：§3.3 item 7（第 485 行）；验证项 27（第 1613 行）

**严重程度**：一般

**改进建议**：
1. 在 §3.3 item 7 末尾新增一条显式契约声明：「本实现依赖 Mat4x4 的列字段名为 `c0`/`c1`/`c2`/`c3`（验证项 27 已确认）。若 stage 2 重构此命名，需同步修改本函数的字段引用路径，或改为通过 `[]` 下标运算符访问（`m[0]`/`m[1]`/`m[2]`）以解耦。」
2. 在 §8.4 实施批次表或 §8 编码启动前验证项中，将验证项 27 的状态从「已核验确认」升级为「需在 stage 3 编码前重新确认 stage 2 Mat4x4 字段名是否发生变动的回归核验」。

## 问题 4（一般）：`normalize` 函数的 T 约束收紧决策未与 `length` 的 T 约束策略保持联动

**问题描述**：§3.7 `normalize` 函数的实现策略段（v26 修订）明确收紧 T 约束为 `where T <: FloatingPoint<T>`，理由是「整数 T 实例化时编译期报类型不匹配错误，消除整数除法截断导致的语义歧义」。但 `normalize` 的依赖函数 `length(q)`（同节，§3.7）的签名采用 `Number<T>` 宽约束（因 `dot` 采用 `Number<T>` 约束，`sqrt` 由 std.math 提供浮点重载）。这意味着：

- 当 `T = Int64` 时，`length(q)` 可编译（`dot` 在 `Number<T>` 下可行，`sqrt` 以 Float64 重载调用），但是 `normalize(q)` 因 `FloatingPoint<T>` 约束编译期拒绝
- `normalize` 的约束比其依赖函数 `length` 更窄，这种「被调用方约束比调用方更宽」的模式在泛型系统中是合法但棘手的——它意味着下游开发者必须先调用 `length` 判断是否是零四元数，然后发现不能使用 `normalize` 处理零四元数，被迫自行实现零保护分支

文档没有对 `normalize` 和 `length` 约束不一致的设计后果做明确说明。

**影响**：整数四元数可以调用 `length(q)` 但不能调用 `normalize(q)`，这种不对称行为未被设计文档显式捕获。下游开发者可能设计 `fn process<T>(q: Quat<T, Q>) where T <: Number<T> { let len = length(q); if (len > 0) { normalize(q) } }` 而遭遇编译失败。

**所在位置**：§3.7 `normalize` 实现策略段（第 591 行附近）、§3.7 `length` 签名描述（第 590 行附近）

**严重程度**：一般

**改进建议**：
1. 在 §3.7 `normalize` 函数描述末尾补充一条明确的约束差异性说明：「注意：`normalize` 的 T 约束（`FloatingPoint<T>`）比其内部依赖函数 `length`（`Number<T>`）更窄。整数四元数（`Quat<Int64, Q>`）可调用 `length(q)` 获取长度，但调用 `normalize(q)` 将因 `FloatingPoint<T>` 约束不满足而编译失败——整数四元数不存在归一化语义。」
2. 在 §5.3 边界条件表新增一行「整型 T 调用 `normalize(q)`」的编译期行为契约。

## 问题 5（一般）：需求响应充分度——阶段三能力画像不直观

**问题描述**：用户需求为「接下来计划迁移阶段三，你来做本次迁移的OOD」。文档以 2236 行的规模提供了详尽的架构设计，但阶段三「实际能做什么」和「不能做什么」的信息散布在 §1 概述、§3.13.2 审计表、§11.5 对照表、§5.3 边界条件表等多处。一个新接入的下游消费者（如使用 cjglm 的应用程序开发者）需要翻阅 2000+ 行才能形成关于阶段三能力边界的完整画像。

对比文档的自评价：§11.5 自称「主要参考基准」并期望下游以它为权威入口，但 §11.5 本身不是一个能力摘要——它是逐函数的状态标注表，任何读完此表的读者仍然需要到 §3.x 寻找函数的具体签名和约束。这种「先读聚合表→再读具体章节→再核对约束声明→再确认边界条件」的多跳阅读模式在经济性上不够理想。

**影响**：下游消费者评估是否可以采用 stage 3 时，信息获取成本高。不确定的读者可能跳过关键的约束信息（如 `isnan` 要求 `FloatingPoint<T>`、比较函数用 `Comparable<T>`），导致编码时遭遇意外编译错误。

**所在位置**：§1 概述（第 16-23 行 API 可用率声明）、§11.5 函数可用性对照表

**严重程度**：一般

**改进建议**：
1. 在 §1 概述末尾（API 可用率声明后方）新增一个「阶段三能力分类摘要」段落，将 18 个真完整实现函数按**用户场景**分组（而非按 GLM 头文件分组），例如：「旋转表达：Quat 类型 + 构造/identity/x/wxyz/mat3Cast/mat4Cast/quatCast + conjugate/inverse；插值：lerp；几何：dot/length/normalize/cross/axis；常量：epsilon/pi/cos_one_over_two + 28 gtc/constants...；判断：equal/notEqual(Quat)/isnan/isinf/lessThan/greaterThan 系族；不可用：slerp/mix/exp/log/pow/sqrt/angle/angleAxis/rotate/eulerAngles/quatLookAt 等 88 个→见 §3.16 降级路径」
2. 在 §8.4 实施批次表前补充一段文字说明，帮助开发者理解「先做什么后做什么」背后的业务理由：「第一批完成后即可构建和运行单位四元数；第二批完成后支持旋转表达核心功能；第三批完成后可进行四元数比较和 gtc 命名空间调用」

## 问题 6（轻微）：`axis` 函数对 `w = 1`（单位四元数）和 `w = -1`（等价单位四元数）的触发条件描述不够精确

**问题描述**：§5.3 边界条件表第 1270 行描述 `axis` 函数在 `1 - x.w*x.w <= 0` 时返回 `Vec3(T(0), T(0), T(1))`，并标注「典型场景：单位四元数 `(0, 0, 0, 1)` 即 `|w| >= 1`」。数学上 `(0, 0, 0, 1)` 的 `w=1` 时 `1-1=0` 触发条件；但 `w = -1` 时同样 `1-(-1)²=0` 触发，且 `w = 1` 和 `w = -1` 表示相同的旋转（双覆盖）。文档仅显式写了 `(0,0,0,1)` 而未提及 `(0,0,0,-1)`，虽然数学上行为一致，但开发者按字面实现时可能只测试 `(0,0,0,1)` 而未测试 `(0,0,0,-1)`，错过负数 w 的边界覆盖。

**影响**：测试可能遗漏 `w = -1` 场景，虽然功能不影响正确性但留下了测试盲区。

**所在位置**：§5.3 边界条件表 `axis(q)` 行（第 1270 行）

**严重程度**：轻微

**改进建议**：将 §5.3 `axis(q)` 行「典型场景」表述扩充为「|w| >= 1（包括 `w=1` 和 `w=-1`，两者表示同一旋转的双覆盖）」，并在 §8.2 `test_ext_quaternion_trigonometric.cj` 的 axis 测试用例说明中补充 `w = -1` 边界的覆盖要求。

## 总结

本审查识别出 **6 项问题**（1 严重 + 4 一般 + 1 轻微）。核心发现：文档在技术细节的完备性上经过 28 轮迭代已达到较高水准，但存在两类结构性问题需修复者关注：

1. **验证可信度问题（问题 1）**——P0 基础假设的验证缺乏可独立复现的证据，且验证方法描述存在循环引用。这是唯一被标记为「严重」的问题，因其直接影响整个设计的基础假设是否真正被确认。

2. **设计信息获取效率问题（问题 2-5）**——跨 Qualifier 行为契约缺失、约束联动性未声明、依赖耦合未设计保护机制、能力画像不直观。这些问题不影响设计正确性，但增加下游开发者的理解成本和编码风险。
