## 二、严重问题（4 项）

### 问题 1（严重）：`gtc/quaternion.cj` snake_case 函数名与 `public import` 机制不兼容，下游调用示例将编译失败

**问题描述**：设计 §3.15 / §10 / §11.4 一致要求 gtc 命名空间下暴露 `mat3_cast`/`mat4_cast`/`quat_cast`（snake_case，GLM 习惯命名）；§3.2.1 定义 detail 端函数名为 `mat3Cast`/`mat4Cast`/`quatCast`（camelCase，仓颉惯用命名）；§2 lib.cj 段落 / §3.15 跨包引用段 / D11 决策均以 `public import glm.detail.{mat3Cast, mat4Cast, quatCast}` 描述重导出机制。

但根据 `cangjie-lang-features/package/README.md` 第 156-166 行明确说明，`public import a.b.f` 仅「将 a.b 中的 f 重新导出」，**原函数名 f 在新包中仍以 f 名称对外可见，不进行命名转换**。仓颉语言规范未提供类似 Python `as` 别名重命名的能力用于 `public import`（`import ... as` 仅解决冲突，原名不可用）。

实测验证：`cjglm/src/lib.cj:8` 的 `public import glm.detail.{transpose, matrixCompMult, outerProduct}` 三者在 `glm` 顶层命名空间下仍以 `transpose`/`matrixCompMult`/`outerProduct`（原始 camelCase 名）暴露，未发生 snake_case 转换。

下游按 §11.4 迁移示例调用 `glm.gtc.quaternion.mat3_cast(q)` / `glm.gtc.quaternion.quat_cast(m)` 时将编译失败（`mat3_cast`/`quat_cast` 在 `glm.gtc.quaternion` 命名空间下不存在，实际存在的是 `mat3Cast`/`mat4Cast`/`quatCast`）。§4.4 行为契约示例也存在同样问题——虽然 v11 修订统一为 `glm.mat3Cast(q)`（顶层 glm 命名空间），但 §11.4 gtc 端示例仍使用 snake_case。

**所在位置**：
- §2 lib.cj/fwd.cj 段落（约第 132 行）
- §3.2.1 `mat3Cast`/`mat4Cast`/`quatCast` 函数签名定义（第 285-292 行）
- §3.15 gtc/quaternion.cj 重导出函数段（第 708-712 行）
- §10 覆盖矩阵 gtc/quaternion 表（第 1291-1294 行）
- §11.4 迁移示例（第 1354-1355 行）

**严重程度**：严重（直接阻塞下游编码落地）

**改进建议**（任选其一）：
1. **方案 A**：在 `gtc/quaternion.cj` 中显式定义 wrapper 函数（无需 `public import`），如：
   ```cangjie
   public func mat3_cast<T, Q>(q: Quat<T, Q>): Mat3x3<T, Q> where T <: FloatingPoint<T>, Q <: Qualifier {
       mat3Cast(q)  // 调用同包 detail 端的 camelCase 版本
   }
   ```
   该方案保留 GLM snake_case 习惯，下游消费者可同时使用 `glm.gtc.quaternion.mat3_cast(q)` 与 `glm.detail.mat3Cast(q)` 两条调用路径。
2. **方案 B**：detail 端函数名直接采用 snake_case（`mat3_cast`/`mat4_cast`/`quat_cast`），与 GLM 完全对齐，`public import` 直接生效。代价是 detail 端 API 偏离仓颉 camelCase 惯用命名。
3. **方案 C**：gtc 端采用 camelCase 命名（`mat3Cast`/`mat4Cast`/`quatCast`），与 detail 端和仓颉习惯一致，§11.4 迁移示例与 §10 覆盖矩阵同步修订。

## 五、总结

本轮（v7 迭代第 7 轮）新识别质量问题共 **14 项**（4 严重 + 5 一般 + 5 轻微）。

**严重问题分布特征**：本轮 4 项严重问题均聚焦于「**设计可落地性**」维度——前序 6 轮（v2~v11）侧重事实正确性、GLM 对齐、约束收紧、依赖闭环等设计内容质量维度，但对「设计文档能否被下游直接按字面实施编码」这一维度覆盖不足。具体表现为：
- 问题 1：`public import` 机制与 snake_case 命名约定的兼容性未验证
- 问题 2：自动生成文件的手动修改策略未说明
- 问题 3：v11 多轮修订的内容冗余未闭环
- 问题 4：空桩文件的具体函数签名清单未明确

**建议优先级**：
1. **必须修复**（问题 1、2、3、4 严重）：直接影响下游编码落地。
2. **建议修复**（问题 5、6、7、8、9 一般）：提升设计文档的内部一致性与下游查阅效率。
3. **可选修复**（问题 10、11、12、13、14 轻微）：属于文档组织偏好，可在后续迭代中酌情处理。

**整体设计质量评价**：经过 7 轮迭代，v11 设计文档在事实正确性、设计完整性、GLM 对齐、约束一致性等维度已较为完善，可作为下游编码的主要参考；本轮识别的 4 项严重问题均集中于「从设计到代码的最后一公里」，建议下一轮迭代（v8）重点关注设计可落地性维度的验证（如通过原型实现验证 §3.13.1 trigonometric.cj 函数签名的编译可行性、§3.2.1 type_quat_cast 函数的 public import 行为等）。
