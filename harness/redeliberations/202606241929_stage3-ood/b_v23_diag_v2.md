## 问题 1（严重）：§5.4 对 `inverse` 不可声明为 const 的推理存在逻辑错误

**问题描述**：§5.4（行 1045-1046）声明 `inverse` 不可在 const 上下文调用，理由是「整数 T 在 `dot(q,q) == 0` 时触发仓颉 `ArithmeticException`」。该推理存在概念混淆——**const 函数约束是关于编译器能否在编译期对函数求值**，而非关于函数运行时是否可能抛异常。仓颉的 `const func` 在运行期执行时同样可以抛异常（如 `assert` 在 const 函数中同样可抛异常），但 `assert` 在 const 上下文中被拒绝的原因并非「运行时抛异常」而是「编译期求值时无法执行 assert」。`inverse` 函数体内的 `/` 运算符对于整数类型在编译期求值时同样存在除零可能性，其被 const 拒绝的根本原因是**整数除法在仓颉中不是 constexpr 友好的操作**（涉及可能的硬件异常路径），而非「运行时抛异常」。§5.4 以「运行时抛异常」作为 const 拒绝理由，是因果倒置——如果 const 函数因「运行时可能抛异常」就被拒绝，那么几乎所有函数（包括文档声称可 const 的 `conjugate`）在输入非法时都可能抛异常，此标准无法自洽。

**所在位置**：§5.4（行 1045-1046）

**严重程度**：严重

**改进建议**：将 §5.4 `inverse` const 拒绝理由修订为：「`inverse` 函数体内部 `conjugate(q) / dot(q, q)` 的 `/` 运算符在仓颉整数类型上非 constexpr 兼容，故不可声明为 `const func`」，删除「运行时抛算术异常」的错误归因。同时检查全文档其他依赖相似推理的位置（如 `lerp` 的 `assert` 拒绝理由），确保推理逻辑链一致。

## 问题 2（严重）：§1 `T(Float64(n))` 语法假设与 scalar_constants 整型拒绝策略逻辑不相容

**问题描述**：§1（行 46-53）声明 `T(Float64(n))` 语法对 `T = Int8~Int64/UInt8~UInt64` 均编译可行，并以此作为 normalize/axis/mat3Cast/mat4Cast/Quat×Vec3/Quat×Vec4 共 6 个函数变体中 T(0)/T(1) 的默认获取路径。同时，§3.12（行 640-646）声明 `epsilon<T>()`/`pi<T>()`/`cos_one_over_two<T>()` 在整型 T 实例化时抛 `Exception("not defined for non-floating-point types")`，理由是「GLM 使用 `static_assert(is_iec559, ...)` 静态断言非浮点类型编译失败」。两处策略逻辑不相容：如果 `T(Float64(1))` 对 `Int32` 确实编译可行并返回 `1`，那么 `epsilon<Int32>()` 完全可以通过 `T(Float64(1.1920929e-7))` 返回整型 `0`（截断）——不存在必须抛异常的技术原因。文档声称整型 epsilon 必须抛异常的理由是「与 GLM 行为一致」，但 GLM 通过编译期静态断言拒绝，而仓颉版本通过运行时异常拒绝——运行时异常路径的存在本身就说明编译期约束收紧未被采用，恰与 `T(Float64(n))` 语法可行的假设矛盾。若 `T(Float64(n))` 对整型编译可行，则 §3.12 的整型拒绝策略应改为编译期约束收紧（与 `FloatingPoint<T>` 一起），而非保留运行时异常作为 fallback。

**所在位置**：§1（行 46-53）vs §3.12（行 640-646）vs D25（行 1090）

**严重程度**：严重

**改进建议**：二选一——(a) 承认 `T(Float64(n))` 对整型 T 可能不可行（将 §1 中的支持声明限定为浮点类型，整型降级为 match/one 形参路径），此时 scalar_constants 的整型拒绝策略保持运行时异常；(b) 确认 `T(Float64(n))` 对整型 T 确实可行（保留 §1 声明），但将 scalar_constants 的 epsilon/pi/cos_one_over_two 的 T 约束收紧为 `FloatingPoint<T>`（编译期拒绝整型）或允许整型返回截断值（与 §1 策略一贯）。两种路径选一并在全文档统一。

## 问题 3（一般）：§3.13.1 trigonometric.cj 75 个 stub 函数的「内部依赖」标注存在语义矛盾

**问题描述**：§3.13.1 三角函数表（行 724-743）对所有 14 个单参数三角函数和 1 个双参数 atan2 的「内部依赖」列标注 `std.math.sin`/`std.math.cos` 等依赖。但 §3.13.1 表前说明（行 757）明确「所有函数本阶段均为 stub（仅签名空壳 + `throw Exception("stub")」）。当函数体为 `throw Exception("stub")` 时，`std.math.sin`/`std.math.cos` 等并非编译时依赖——throw 语句不触发类型参数到 std.math 函数的实例化。标注这些依赖作为「阶段四实现后将依赖」的预期信息是合理的，但当前标注的语法（`std.math.sin`）会误导下游编码者认为这些 stub 函数当前编译时需要这些依赖可用。这与 §1 v11 修订版「T=Float32 实例化时直接调用 std.math 对应重载」的讨论形成矛盾——如果所有 75 个函数都是 stub，那么 Float32 重载的可用性与当前实现无关。

**所在位置**：§3.13.1 三角函数表（行 724-743）与表前说明（行 757）

**严重程度**：一般

**改进建议**：修订三角函数表「内部依赖」列标注规则——(a) 表头添加说明：「标注的依赖为阶段四完整的实现后依赖；本阶段 stub 函数体为 `throw Exception("stub")`，编译时不实际触发这些依赖」；(b) 将 `std.math.sin` 等标注改为 `std.math.sin（阶段四依赖）` 格式，明确标注其为未来依赖而非当前依赖。同时检查 §3.10 `pow`/`log`/`exp` 的依赖描述是否也有类似混淆——这些函数也是 stub 但依赖描述使用现在时「依赖……」。

## 问题 4（一般）：§3.13.2 审计表计数「18 个真完整实现」的函数名清单与 §11.7.1 的行数不一致

**问题描述**：§3.13.2 审计结论（行 792）列出 18 个真完整实现函数：`epsilon/pi/cos_one_over_two/axis/length/normalize/inverse/isnan/isinf/mat3Cast/mat4Cast/quatCast/conjugate/lerp/dot/cross(Quat)/equal(Quat)/notEqual(Quat)`。但按照函数名粒度统计：
- `isnan` + `isinf` = 2 个独立函数
- `mat3Cast` + `mat4Cast` + `quatCast`(Mat3) + `quatCast`(Mat4) = 4 个独立函数
- `equal(Quat)` + `equal(Quat, epsilon)` + `notEqual(Quat)` + `notEqual(Quat, epsilon)` = 按 §11.5 表为 4 个独立行
- 其余：`epsilon`/`pi`/`cos_one_over_two`/`axis`/`length`/`normalize`/`inverse`/`conjugate`/`lerp`/`dot`/`cross(Quat)` = 11 个

合计按行粒度：2 + 4 + 4 + 11 = **21 行**。按函数名粒度（不区分重载）：isnan/isinf 算 2 个，mat3Cast/mat4Cast/quatCast 算 3 个，equal/notEqual 算 2 个，其余 11 个 = **18 个**。文档选择「函数名」粒度（18 个）是对的，但 §11.7.1 索引表与 §11.5 可用性表均使用「行」粒度呈现，而 §3.13.2 使用「名」粒度，两处粒度不一致，下游在 §3.13.2 和 §11.5 之间查阅时会产生计数错觉（以为 §3.13.2 遗漏了某些重载）。此外，§8.3 A 表（行 1405）声称「18 函数 ✅ 可用」，与 §3.13.2 一致，但与 §11.5 的 20+ 行之间存在视觉偏差。

**所在位置**：§3.13.2 审计结论（行 792）；§11.7.1（行 1886-1916）；§8.3 A 表（行 1405）

**严重程度**：一般

**改进建议**：在 §3.13.2 审计结论的「18 个函数」后添加附注：「按函数名计数（不含重载）；按 §11.5 表行粒度展开为 21 行」。同 §8.3 A 表「18 函数」后添加相同附注。建立全文档统一的计数粒度声明——建议统一使用「函数名」粒度计数，在必要时标注重载展开数。

## 问题 5（一般）：关于 P0 核心假设未经验证的维度归属澄清（回应质询问题 B）

**问题描述**：上一轮报告的 Issue 1 声称两项 P0 核心假设（`T(Float64(n))` 语法可行性和 `FloatingPoint<T>` 接口方法可用性）未经编译验证属「深度和完整性维度的根本性缺陷，非技术可行性维度」。质询问题 B 指出该维度归属论证不严谨。经重新审视：该问题的本质属于**技术可行性与设计完整性交叉维度**——从技术可行性角度，「编译器是否接受这种语法」是一个技术可行性问题；从设计完整性角度，「在未经编译验证的情况下将其作为全文档的默认实现路径」属于设计过程的完整性缺陷。v23 版本的 §8 验证结果记录表模板、§1 回退方案决策树的引入已部分改善了设计完整性（提供了假设被证伪的处置路径），但两项假设仍停留在「待验证」状态。v23 版文档的核心改进在于：不再声称「可直接指导编码实现」——§8 明确标注 P0 3 项验证通过后方可启动编码。**本轮审查结论**：该问题在维度归属上确属于跨维度问题，但 v23 版本相比 v10 版本已从「完全依赖未验证假设」改善为「标记为待验证并提供了回退路径」，严重程度可维持「严重」但性质从「根本性缺陷」调整为「风险管理缺口」——即设计本身逻辑是自洽的，但风险敞口未关闭。

**所在位置**：§1 回退方案（行 89-115）；§8 编码启动前验证项 20/25/29（行 1353-1359）；§8 验证记录表模板（行 1361-1395）

**严重程度**：一般（从严重降级——v23 已引入回退方案决策树，不再是无 B 计划的状态）

**改进建议**：在 §1「回退方案」决策树末尾增加「验证时限承诺」——例如「验证项 20/25/29 应在编码启动后 5 个工作日内完成最小测试文件验证，验证结果反馈至 §8 验证记录表模板」。同时将 §1 受这两项假设影响的函数清单明确标注「依赖假设：H1/H2，详见 §1 回退方案」。

## 问题 6（一般）：接口定义在大批量高同构函数（trigonometric.cj 75 个/gtc/matrix_transform 64 个）上缺乏展开到可编码粒度的模板（回应质询问题 C 的接口定义维度）

**问题描述**：§3.13（行 681-694）列出 gtc/matrix_transform.cj 的 64 个函数仅按类别分组列函数名，未提供每个函数的完整签名模板（泛型参数、参数类型、返回值类型、where 子句）。§8 产出物清单（行 1136）将这些函数归为 stub，但 §8.3 E 节同时要求「所有 stub 函数签名已记录于 §11.5 且阶段四不得单方面修改」——但 §11.5 表（行 1847）仅列出一行「gtc/matrix_transform 全部 64 个函数 ❌ stub」，没有逐函数签名。这形成一个实际矛盾：声称「所有 stub 函数签名已记录于 §11.5」但 §11.5 实际没有记录这 64 个函数的任何逐函数签名。如果阶段四需要修改某个具体函数的签名，验证者无法通过 §11.5 定位该函数——因为签名根本不在表中。同样的问题也影响 trigonometric.cj 的 75 个函数（§11.5 无对应行）。下游编码者打开 `detail/trigonometric.cj` 和 `gtc/matrix_transform.cj` 进行实现时，只能回到 §3.13 的类别分组清单手动逐一编写签名，且签名参数类型/约束/返回类型完全依赖编码者对 GLM 1.0.3 源码的自行解读——这本质上是设计文档将签名设计责任下放给了编码者。

**所在位置**：§3.13 表格（行 681-694）；§11.5 可用性表（行 1847）；§8.3 E 节（行 1447-1457）

**严重程度**：一般

**改进建议**：(a) 在 §3.13 表格下为 gtc/matrix_transform.cj 的 6 大类函数各提供一个代表性函数的完整签名模板（含 where 子句），并说明其他同类函数按此模板类推的展开规则；(b) 在 §11.5 中为 gtc/matrix_transform.cj 和 trigonometric.cj 各补充一行聚合状态行，明确标注「签名序号见 §3.13 表格，当前未逐函数枚举于本表」——替代当前空洞的「已记录于 §11.5」声称；(c) 在 §8.3 E 节的「所有 stub 函数签名已记录于 §11.5」措辞前补充范围限定——适用于四元数相关函数，gtc/matrix_transform 和 trigonometric 的签名的记录位置为 §3.13。

## 问题 7（一般）：边界条件表的场景覆盖存在 5 处未覆盖的边界条件（回应质询问题 C 的边界条件维度）

**问题描述**：§5.3 边界条件表（行 1022-1041）覆盖了 19 类边界场景，但经逐一核对该表与 §3.7/§3.9/§3.11/§3.12 的函数行为描述，以下 5 类边界条件未覆盖：

(1) **NaN/Inf 输入传播**：§3.7 `length`/`normalize`/`dot` 对包含 NaN 或 Inf 分量的四元数输入的行为未定义。GLM 中 NaN/Inf 通过 std.math 传播（如 `sqrt(NaN)` = NaN），但仓颉侧 std.math 的 NaN 行为是否一致未声明。示例：`normalize(Quat(NaN, 0, 0, 0))` 触发 `length(NaN)` → `sqrt(NaN)` → 结果未定义。

(2) **`lerp` 参数 `a` 为 NaN/Inf 或负数无穷**：§5.3 仅覆盖 `a` 超出 [0,1] 触发 assert，但未覆盖 `a = NaN` 时 `assert(NaN >= 0)` 行为（仓颉 `NaN >= 0` 返回 `false`，进入 assert 失败路径）。若 assert 在 release 构建中被移除，`a = NaN` 的 `lerp` 结果行为未定义。

(3) **`fromMat3`/`fromMat4` 对单位矩阵/零矩阵输入**：§3.3 item 6/7 声明「仅对纯旋转矩阵产生有意义的四元数」，但未明确单位矩阵（纯旋转矩阵的特例）和零矩阵输入的行为。单位矩阵输入应返回单位四元数——这是 GLM 的实际行为，但设计文档未声明。

(4) **`axis` 对 NaN w 分量**：§3.9 公式 `tmp1 = T(1) - x.w * x.w`，当 `x.w = NaN` 时 `tmp1 = NaN`，`NaN <= T(0)` = false，进入 else 分支：`tmp2 = T(1) / T(sqrt(NaN))` = `T(1) / T(NaN)` = `NaN`，返回 `Vec3(NaN*x.x, NaN*x.y, NaN*x.z)`。文档未声明此行为——是已定义（传播 NaN）还是未定义？

(5) **`inverse` 对非单位四元数输入**：§3.11 声明 `inverse(q) = conjugate(q) / dot(q, q)`。当输入为非单位四元数时，`dot(q, q)` 为标量值，除法按标量除法进行，结果的行为是已定义的（返回 `conjugate/norm²`）。但文档未声明该行为是否对非单位四元数输入有意义（结果仍然是数学上的逆四元数，不受输入是否为单位的影响）。§5.3 中 `inverse` 的两行（浮点零除 / 整数零除）覆盖了零四元数边界，但未声明非零非单位输入的行为。

**所在位置**：§5.3 边界条件表（行 1022-1041）

**严重程度**：一般

**改进建议**：(1) 在 §5.3 表内新增「NaN/Inf 输入传播」行，声明为「NaN/Inf 通过 std.math 传播；`sqrt(NaN)`/`NaN <= T(0)` 等行为与 std.math NaN 传播语义一致」；(2) 新增 `lerp` 参数 NaN 行，声明 `a = NaN` 时 `assert` 失败（debug）/结果未定义（release）；(3) 新增 `fromMat3`/`fromMat4` 单位矩阵行，声明「单位矩阵输入返回单位四元数」；(4) 新增 `axis` NaN w 行，声明 NaN 传播行为；(5) 新增 `inverse` 非单位四元数输入行，声明「行为已定义：数学上的四元数逆 `conjugate/‖q‖²`，对非单位四元数有效」。

## 问题 8（轻微）：§8.3 I 节 O-02 与正文「§11.5 为单一权威来源」形成自相矛盾

**问题描述**：§8.3 I 节 O-02（行 1488）声明「§9a 与 §10 两张覆盖矩阵存在数据冗余，需合并为单一来源」，并计划在 v24 修复。但 §11.5（行 1774）声称「v22 已补齐全部 5 个缺失 API 条目，当前版本已具备完整性要求」并升级为「单一权威来源」。如果 §11.5 确实是单一权威来源，则 §9a 和 §10 的存在本身就是冗余的——因为下游应只查阅 §11.5。但实际上 §9a 提供了 §11.5 没有的「GLM 1.0.3 基准函数总数」信息，§10 提供了按 GLM 头文件分组的覆盖状态视图——这些信息 §11.5 并未提供。因此「单一权威来源」声称与「§9a/§10 需合并」的推迟修复计划之间存在逻辑矛盾：如果 §11.5 真的是单一权威来源，§9a 和 §10 根本不需要存在（即使作为冗余视图）。O-02 承认了 §9a/§10 作为独立表格的持续存在，实际上否定了 §11.5 的「单一权威」地位。

**所在位置**：§11.5 表头声明（行 1772-1774）；§8.3 I 节 O-02（行 1488）

**严重程度**：轻微

**改进建议**：将 §11.5 的定位声明从「单一权威来源」修订为「函数级状态追踪的权威来源（每函数一行，标注本阶段/阶段四状态）。§9a 和 §10 提供按头文件分组的聚合统计视图与 GLM 1.0.3 基准声明，与 §11.5 构成互补关系」。这样可以逻辑自洽：§11.5 管函数级状态，§9a/§10 管基准视图和按头文件分组统计。
