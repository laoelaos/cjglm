## 2. 逻辑矛盾

### 2.1 [一般] T17 插入位置"可并行"建议与 §6 串行要求自相矛盾

- **位置**：§4 T17「插入位置建议」段落——"推荐追加到文件末尾以避免行号冲突，且便于与 T5（修正在 :330-344）并行实施"
- **问题描述**：诊断报告 §6 依赖图正确标注 `T5→T17` 共享文件关系（均操作 `test_integration_matrix.cj`），并在推荐执行顺序中明确要求"须串行：T5 先于 T17"。但 §4 T17 末段却说"便于与 T5 并行实施"，与 §6 建议自相矛盾。即使行号不冲突（T5 在行 330-344，T17 追加至文件末尾），在 git 中同时对同一文件做两处修改仍会产生文件级合并冲突，**不应标注为"可并行"**。
- **改进建议**：将 §4 T17 末句改为"推荐追加到文件末尾，建议先完成 T5 修正确保文件基线一致后再实施 T17"——与 §6 的串行建议保持一致。

### 2.2 [一般] §6 组 C 串行约束缺少 T7 与 T9 的相对顺序指引

- **位置**：§6 Phase 4 组 C
- **问题描述**：组 C 标注两条串行约束：(1) T9→T10（共享三个测试文件）；(2) T7→T10（共享 `test_matrix.cj`）。但 T7 与 T9 之间无文件共享和依赖关系，报告中未给出二者相对顺序的建议。若执行者试图将 T7 与 T9 并行执行（而分别串行到 T10），则 T10 需要同步处理来自两个来源的行号偏移，复杂度显著增加。
- **改进建议**：在组 C 中明确标注"T7 和 T9 无跨文件依赖，可按任意顺序执行，但两者均须在 T10 之前完成"。

## 3. 深度与完整性不足

### 3.1 [一般] T15 未按用户需求选取 2~3 个代表性矩阵类型

- **位置**：§4 T15
- **问题描述**：用户需求明确要求"选取 2~3 个代表性矩阵尺寸"进行 NaN 传播测试（来源：`requirement.md` T15 描述第 4 条）。诊断报告给出了 GLM 参考代码位置和 NaN 传播的通用说明，但未具体推荐应选择哪 2~3 个矩阵类型，也未说明选择理由。执行者无从知晓哪些矩阵尺寸最具代表性，可能选取极端情况导致测试不充分，或选取全部 9 个类型导致测试膨胀。
- **改进建议**：在 T15 中补充"推荐矩阵类型"子段落。例如推荐 Mat4x4（全尺寸方阵，覆盖最多列/行组合）、Mat2x3（最小非方阵，与 Mat4x4 形成尺寸极端对比）、Mat3x2（转置非方阵，覆盖不同列/行比例），并说明选择理由。

### 3.2 [一般] T6/T7 Float 变体膨胀量化未考虑整数类型的额外覆盖成本

- **位置**：§1 T6/T7「范围膨胀量化」段落
- **问题描述**：T6 测试文件中已存在 Int32 和 Float 变体测试（如 `testScalarDivMat2x2Int32Truncation`），表明执行惯例倾向于覆盖多种数值类型。T6/T7 的膨胀量化（T6: 24→72, T7: 12→36）仅按 Int64 + Float32 + Float64 三种类型计算，未考虑 Int32/Bool 等离散类型的测试可能性。若 Phase 2 执行者按 T16 标准同时覆盖这些类型，实际测试膨胀将超出报告给出的量化估计。
- **改进建议**：在膨胀量化后追加脚注，说明"此处仅按 T16 最小覆盖标准（Int64/Float32/Float64）计算。若需对齐现有测试文件中的 Int32 等额外类型，实际测试数将进一步膨胀。建议与 T16 的执行者协商明确覆盖边界。"

### 3.3 [轻微] T6 缺失集成测试层面的影响分析

- **位置**：§1 T6
- **问题描述**：T6 覆盖了标量-矩阵运算的单元测试缺口，但 OOD §3.5 定义的 4 运算 × 9 类型 = 36 个全局函数不仅在 `test_scalar_mat_ops.cj` 中缺失测试，在 `test_integration_matrix.cj` 中也没有非方阵标量运算的集成测试覆盖。诊断报告仅从单元测试层面分析缺口，未讨论集成测试层面的覆盖情况——执行者完成 T6 后，集成测试仍可能存在相同的非方阵标量运算覆盖缺口。
- **改进建议**：在 T6 风险分析中追加一句，提示"集成测试层面是否存在同类缺口需另案审视，建议 T6 完成后检查 test_integration_matrix.cj 是否包含非方阵标量运算场景"。

## 4. 可操作性与优先级

### 4.1 [一般] Phase 2 阻塞性质说明可进一步加强

- **位置**：§6 推荐执行顺序表 Phase 1
- **问题描述**：第 7 轮将 Phase 1 从"解除 CI 阻塞"改为"解除 CI 运行时失败"，并区分了 T5（运行时失败）和 T6/T7（覆盖率评审阻塞），修正了此前阻塞类型混淆的问题。但 T6/T7 同样会在评审阶段被拒绝合并——如果评审者严格执行覆盖标准，T6/T7 的阻塞性质不亚于 T5。仅标注两类阻塞的"紧急程度和执行顺序不同"未给出实践指引。
- **改进建议**：在 Phase 1 说明中明确标注"若执行者希望一次性通过评审，建议在 Phase 1 后立即执行 Phase 2；否则即使 T5 修复通过 CI，Phase 2 未完成前仍无法合并"。

### 4.2 [一般] T13 三种模式的切换条件可操作性仍有改善空间

- **位置**：§5 T13「验证步骤指南」
- **问题描述**：模式 A→B 的触发条件是"cjpm build 报错，错误信息涉及 `package glm.ext` 未识别或子包路由失败"。但 cjpm 的报错信息格式未知——执行者面对一个实际错误时，可能无法判断"这属于需要切换到模式 B 的错误"还是"属于模式 A 下可解决的其他错误"。同理，模式 B→C 的触发条件"package-configuration 语法不被 cjpm 当前版本支持"——若模式 B 因其他原因失败（如配置项名称拼写错误），执行者可能错误地切换到模式 C。
- **改进建议**：为每个触发条件补充"验证确认步骤"。例如："收到报错后，先在 cjpm 文档中搜索对应报错信息确认确属子包路由问题，而非语法错误或导入路径问题"。
