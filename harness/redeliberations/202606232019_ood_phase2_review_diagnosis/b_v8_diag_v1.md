## 发现的问题

### 问题 1：T7 缺失测试数据构造建议，可操作性不足

- **所在位置**：§1 T7「具体缺口」
- **严重程度**：一般
- **问题描述**：T7 列出了 matrixCompMult（6 个缺失重载）和 outerProduct（6 个缺失重载）的函数签名，但**未提供任何测试数据构造建议或参考测试模式**。对比同节其他问题：T6 给出了缺口表格和风险说明（列/行索引写反的检测），T15 给出了推荐的矩阵类型和选取理由，T17 给出了完整的源数据矩阵和预期值构造。T7 仅列出类型名称，执行者需自行设计全部 12 个测试函数的数据和预期值，增加了测试数据选择不当的风险（例如所选数据无法区分特定的行列索引错误）。
- **改进建议**：至少为 matrixCompMult 和 outerProduct 各提供一个代表性测试重载的数据示例和预期值构造模式（可引用现有已覆盖测试 `test_matrix.cj:164-172` 或 `:211-219` 的样式），执行者可据此复制扩展到其余缺失重载。

### 问题 2：T10 expectStubMessage 辅助函数未应用 @Fail/catch 互斥风险的规避方案

- **所在位置**：§4 T10「可复用性优化建议」
- **严重程度**：一般
- **问题描述**：报告中 inline try-catch 示例（第 312-323 行）已标注 `@Fail` 与 `catch(e: Exception)` 互斥风险，并给出规避方案（在 catch 块顶部检查异常类型区分 `AssertFailException` 与业务异常）。但随后提出的 `expectStubMessage` 辅助函数（第 336-344 行）**没有应用此规避方案**——它同样在 `try` 块中使用 `@Fail`，`catch` 块仅检查 `e.message.contains("stub")`，未区分 `AssertFailException`。执行者可能误以为辅助函数已完全规避此风险，直接复制使用导致调试时产生误导性错误消息。此外，`expectStubMessage` 的参数类型 `funcBody: () => Unit` 可能需按仓颉实际语法调整（`Unit` 类型名和函数类型箭头写法需确认）。
- **改进建议**：在辅助函数示例中添加类型区分逻辑，或追加明确的警告说明"此辅助函数同样存在 @Fail/catch 互斥风险，需在 catch 块首行添加 `if (Type(e) == AssertFailException) { throw e }` 类型检查后使用"。
