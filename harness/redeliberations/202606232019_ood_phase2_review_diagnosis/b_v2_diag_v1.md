## 2. 发现的问题

### 2.1 缺乏全局修复执行优先级排序

- **位置**：§6 跨问题依赖分析
- **严重程度**：一般
- **问题描述**：§6 给出了依赖图与串行约束分析（如 T5→T17 共享文件、T1→T19 共享文件），但**没有给出明确的"先做什么后做什么"的全局执行优先级建议**。T5/T6/T7 被归类为"阻塞合并"，但报告未将这些阻塞项置于一个统合的 fix sequence 中。执行者拿到报告后需要自行从依赖分析中推导执行顺序。
- **改进建议**：在 §6 开头或末尾追加明确的推荐执行顺序（例如：Phase 1=修复 T5 解除 CI 阻塞 → Phase 2=补充 T6/T7 严重覆盖缺口 → Phase 3=T1/T19 偏差文档 → Phase 4=其余一般/轻微项），并标注各阶段是否可以并行。

### 2.2 T15 未显式限定 NaN 测试作用于浮点类型的范围约束

- **位置**：§4 T15
- **严重程度**：一般
- **问题描述**：T15 讨论了 OOD §5 要求的 NaN 传播行为测试（矩阵含 NaN 列、向量含 NaN 分量等），但**未在自身段落中说明 NaN 测试仅适用于 Float32/Float64 类型矩阵**。当前 9 个矩阵测试文件几乎全部使用 Int64 作为底层类型（T16 已确认这一事实），而 Int64 等整数类型不存在 NaN。若执行者先读 T15 着手编写测试，可能会尝试在 Int64 矩阵上构造 NaN 场景，这是不可能的。§6 的依赖分析虽将 T15 与 T16 关联，但 T15 段落自身缺乏范围约束说明。
- **改进建议**：在 T15 段落末尾追加显式约束说明，例如："注意：NaN 行为测试仅适用于 Float32/Float64 浮点类型矩阵；Int64 等整数类型不存在 NaN 概念。应在 T16（浮点类型基础覆盖）完成后执行本项。"

### 2.3 T13 缺乏具体可操作的验证方法指南

- **位置**：§5 T13
- **严重程度**：一般
- **问题描述**：T13 准确识别了原型验证计划 5 个步骤均未执行，也区分了"测试域内 alias 验证"与"外部导入验证"两个层面。但**报告只指出了"什么没做"，没有给出"具体怎么做"的操作指南**。例如：
  1. 如何在 `cjpm.toml` 中配置 `package-configuration` 或 `sub-package` 以实现 `src/ext/` 子包自动发现？
  2. 如何创建最小外部项目来验证 `import glm.ext.*` 可解析？
  3. 若 cjpm 不支持自动发现，回退方案的具体实现方式是什么？
  执行者（尤其是对 cjpm 包配置不熟悉的人）在读完 T13 后仍不知道从何下手。
- **改进建议**：补充一个"验证步骤"子章节，至少给出三个最常见的 cjpm 子包配置模式（显式 package-configuration、sub-package 声明、或在 `src/` 下放置 `ext/` 目录结构）以及验证命令，或明确标注"需先查阅 cjpm 文档确认子包路由机制"。

### 2.4 T5 未建议先验证 OOD DEVIATION 本身的正确性

- **位置**：§1 T5
- **严重程度**：一般
- **问题描述**：T5 的根因结论为"集成测试的预期值是错误的"，直接要求将预期值修改为符合 OOD DEVIATION 的结果。但报告**未要求执行者在修改前独立验证 OOD DEVIATION（Mat4x4←Mat4x2 丢弃源列 2/3 数据）是否与 GLM 1.0.3 真实行为一致**。虽然 OOD 明确声明此偏差，并标注了引用行（`type_mat4x4.inl:246`），但未要求执行者实际打开 GLM 参考代码确认。若 OOD 文档记录本身有误（偏差方向判断错误），则直接修改测试预期值会引入新的错误。
- **改进建议**：在"影响范围"后追加验证步骤要求，例如："修复前应打开 `references/glm-1.0.3/glm/glm/detail/type_mat4x4.inl` 第 246 行附近确认 GLM `mat4x4(colon, mat4x2 const& m)` 实际行为，确认 OOD DEVIATION 的描述正确后再修改预期值。"

### 2.5 T18 给出多个平行选项降低可操作性

- **位置**：§4 T18
- **严重程度**：轻微
- **问题描述**：T18 的结论为"建议保留 ZeroVec 测试，但可将 9 个合并为 1~2 个代表性场景，或选择保持现状（18 个测试的维护成本可接受）"。**三个选项并列为执行者提供了"做不做、做多少"的多种选择，本质上将决策责任推给了执行者**。一份诊断报告应给出明确的推荐方案，让执行者直接"执行"而非"决策"。
- **改进建议**：选定其中一个方案并给出理由，例如"建议保留全部 9 个 ZeroVec 测试——它们的维护成本极低（模板化代码），且作为零向量×任意矩阵=零矩阵的数学恒等式验证，每个矩阵类型均有独立的验证价值。"

### 2.6 T16 覆盖标准估算未讨论测试模式共享优化

- **位置**：§4 T16
- **严重程度**：轻微
- **问题描述**：T16 提出的最小覆盖标准为"9 类型 × 3 场景 × 2 精度 = 54 个新增测试"，但**未评估不同类型的测试模式是否可共享**。例如，Mat2x3 和 Mat2x4 的行数相同（均为 2 行），其列访问和标量运算的测试展开模式相同，可通过参数化测试或辅助函数减少重复。54 的估算可能高估实际需要的新增测试数量，执行者可能因"54 个太多"而推迟行动。
- **改进建议**：补充说明可利用仓颉的 `@Types[T in <Float32, Float64>]` 类型参数化测试机制减少测试代码重复，或将覆盖标准简化为"每个矩阵族系（按行数分 3 组：C=2/3/4）至少一组场景 × 2 精度"，约需 3×3×2=18 个新增测试。

## 3. 整体评价

**优势**：诊断报告对 T1~T19 全部问题的根因定位清晰，证据（文件路径、行号、代码引用）充分，修订说明完整回应了上一轮质询的全部 8 项意见。§6 跨问题依赖分析是新增的高价值内容。

**主要不足**：报告在"可操作性"维度存在系统性短板——若干问题的分析止步于"问题是什么/为什么"，未充分延伸到"修复者如何执行"。具体到 T5（缺少偏差验证步骤）、T13（缺少操作指南）、T18（平行选项未定一）和全局（缺少统一 fix sequence），执行者需要额外推导才能进入编码阶段。

**建议**：在下一轮迭代中，每个问题增加一个"修复指导"子章节（或注明"需补充执行者才可开始"），并追加全局执行顺序建议。
