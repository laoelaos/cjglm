### §2.1 [一般] T17 插入位置"可并行"建议与 §6 串行要求自相矛盾

### §2.2 [一般] §6 组 C 串行约束缺少 T7 与 T9 的相对顺序指引

**位置**：§6 Phase 4 组 C
**问题描述**：组 C 标注两条串行约束：(1) T9→T10（共享三个测试文件）；(2) T7→T10（共享 `test_matrix.cj`）。但 T7 与 T9 之间无文件共享和依赖关系，报告中未给出二者相对顺序的建议。

### §3.1 [一般] T15 未按用户需求选取 2~3 个代表性矩阵类型

**位置**：§4 T15
**问题描述**：用户需求明确要求"选取 2~3 个代表性矩阵尺寸"进行 NaN 传播测试。诊断报告给出了 GLM 参考代码位置和 NaN 传播的通用说明，但未具体推荐应选择哪 2~3 个矩阵类型，也未说明选择理由。
**改进建议**：在 T15 中补充"推荐矩阵类型"子段落。例如推荐 Mat4x4（全尺寸方阵）、Mat2x3（最小非方阵）、Mat3x2（转置非方阵），并说明选择理由。

### §3.2 [一般] T6/T7 Float 变体膨胀量化未考虑整数类型的额外覆盖成本

**位置**：§1 T6/T7「范围膨胀量化」段落
**问题描述**：T6 测试文件中已存在 Int32 和 Float 变体测试，表明执行惯例倾向于覆盖多种数值类型。T6/T7 的膨胀量化（T6: 24→72, T7: 12→36）仅按 Int64 + Float32 + Float64 三种类型计算，未考虑 Int32/Bool 等离散类型的测试可能性。
**改进建议**：在膨胀量化后追加脚注，说明"此处仅按 T16 最小覆盖标准（Int64/Float32/Float64）计算。若需对齐现有测试文件中的 Int32 等额外类型，实际测试数将进一步膨胀。建议与 T16 的执行者协商明确覆盖边界。"

### §3.3 [轻微] T6 缺失集成测试层面的影响分析

**位置**：§1 T6
**问题描述**：T6 覆盖了标量-矩阵运算的单元测试缺口，但 OOD §3.5 定义的 36 个全局函数不仅在 `test_scalar_mat_ops.cj` 中缺失测试，在 `test_integration_matrix.cj` 中也没有非方阵标量运算的集成测试覆盖。诊断报告仅从单元测试层面分析缺口，未讨论集成测试层面的覆盖情况。
**改进建议**：在 T6 风险分析中追加一句，提示"集成测试层面是否存在同类缺口需另案审视，建议 T6 完成后检查 test_integration_matrix.cj 是否包含非方阵标量运算场景"。

### §4.1 [一般] Phase 2 阻塞性质说明可进一步加强

**位置**：§6 推荐执行顺序表 Phase 1
**问题描述**：第 7 轮将 Phase 1 从"解除 CI 阻塞"改为"解除 CI 运行时失败"，并区分了 T5（运行时失败）和 T6/T7（覆盖率评审阻塞），修正了此前阻塞类型混淆的问题。但 T6/T7 同样会在评审阶段被拒绝合并，仅标注两类阻塞的"紧急程度和执行顺序不同"未给出实践指引。
**改进建议**：在 Phase 1 说明中明确标注"若执行者希望一次性通过评审，建议在 Phase 1 后立即执行 Phase 2；否则即使 T5 修复通过 CI，Phase 2 未完成前仍无法合并"。

### §4.2 [一般] T13 三种模式的切换条件可操作性仍有改善空间

**位置**：§5 T13「验证步骤指南」
**问题描述**：模式 A→B 的触发条件描述为"cjpm build 报错，错误信息涉及 `package glm.ext` 未识别或子包路由失败"，但执行者面对实际错误时可能无法判断属于哪种类型的错误。同理，模式 B→C 的触发条件也存在同样问题。
