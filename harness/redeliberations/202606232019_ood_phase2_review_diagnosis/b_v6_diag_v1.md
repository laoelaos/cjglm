## 2. 事实错误与逻辑矛盾

### 2.1 [严重] T8: 纯收缩方向计数显著偏差

- **位置**：§3 T8，第 226 行"纯收缩方向共 30 个 6a + 30 个 6b = 60 个重载"
- **错误描述**：声称纯收缩方向（C_dst ≤ C_src 且 R_dst ≤ R_src）有 60 个重载
- **事实核查**：枚举所有 9 目标 × 8 源 × 2 变体的 fromMat 方向，满足纯收缩条件的共：
  - Mat2x2←{2x3,2x4,3x2,3x3,3x4,4x2,4x3,4x4}=8
  - Mat2x3←{2x4,3x3,3x4,4x3,4x4}=5
  - Mat2x4←{3x4,4x4}=2
  - Mat3x2←{3x3,3x4,4x2,4x3,4x4}=5
  - Mat3x3←{3x4,4x3,4x4}=3
  - Mat3x4←{4x4}=1
  - Mat4x2←{4x3,4x4}=2
  - Mat4x3←{4x4}=1
  - Mat4x4←{}=0
  - **合计：27 个方向 × 2 变体 = 54 个重载**，而非 60
- **影响**：虽然此计数不影响 T8 的核心判断（存在无用零值声明）也不影响修复路径（因决策路径指引不依赖计数），但数字偏差降低了专业性，可能被审查质疑
- **改进建议**：修正计数为 27+27=54，或附上枚举清单以证明计数准确性

### 2.2 [一般] T10: try-catch 模式中 `@Fail` 与 `catch (e: Exception)` 的互斥风险

- **位置**：§4 T10，第 302-312 行示例代码
- **问题描述**：示例中 `@Fail("Expected Exception")` 在 try 块内，如果 `@Fail` 宏展开为 `throw Exception(...)`，则会被 `catch (e: Exception)` 捕获，导致异常消息验证逻辑错误地执行（检查的是 `@Fail` 的消息而非 stub 函数的异常消息）
- **事实核查**：根据 cangjie-std/unittest 文档，`@Fail` 宏展开为 `throw AssertFailException(msg)`，该异常类型继承自 `Exception`。因此上述 try-catch 结构**会捕获 `@Fail` 抛出的异常**，导致：
  1. stubFunc() 未抛出异常 → @Fail 执行 → catch 捕获 → @Expect 检查 `@Fail` 的消息（"Expected Exception"）是否含 "stub" → 失败
  2. 测试确实失败，但失败原因错误（报告消息不匹配而非"未抛出异常"），且 `@Expect` 的失败信息会混淆真正的测试意图
- **影响**：此模式在实际运行时可能产生误导性测试失败消息。但考虑到产出是指南性建议而非最终代码，执行者实现时需自行适配正确的捕获模式
- **改进建议**：在示例代码下方追加注释警告此风险，或改用 `@ExpectThrows` 与独立消息验证函数分离的方式

### 2.3 [一般] T13: 模式 C 的副作用遗漏——`test_ext.cj` 编译中断

- **位置**：§5 T13 模式 C 描述（第 540-544 行）
- **问题描述**：模式 C（回退方案）建议将 `src/ext/` 下文件移至 `src/` 并改 `package glm`，但未分析此操作对 `tests/glm/test_ext.cj` 的影响
- **事实核查**：`test_ext.cj`（第 8 行）使用 `ext.Mat4x4` 引用子包别名。若 `src/ext/` 文件改为 `package glm` 并移至 `src/`，则 `ext.*` 命名空间消失，该文件全部 16 个 @Test 用例将编译失败
- **影响**：实施模式 C 后，现有的通过了 16 个测试的验证产物将失效。跑 `cjpm test` 会报告大量编译错误，干扰构建验证
- **改进建议**：在模式 C 中追加副作用分析，标注受影响的测试文件及应对措施（如同步修改 `tests/glm/test_ext.cj` 的导入路径）

### 2.4 [一般] T16: 族系内共享模板的可行性论证不足

- **位置**：§4 T16，第 415-417 行
- **问题描述**：产出提出按族系（C=2/3/4 共 3 组）共享测试模板将 54 个测试压缩至 18 个，但未提供具体语言层面的可行性验证
- **事实核查**：
  - 族系内不同类型（如 Mat2x2/Mat2x3/Mat2x4）列数相同但行数不同，`col(i)` 返回的向量维度（Vec2/Vec3/Vec4）不同，断言格式和预期值不同
  - CangJie 测试框架是否支持跨不同向量维度的参数化测试夹具（如 `@TestCase` 的泛型化参数化）需要验证
  - 产出仅以"族系内共享模板的具体实现复杂度需进一步评估"一笔带过，未评估复杂度等级
- **影响**：执行者可能误以为 18 个测试是低成本的优化方案而选择尝试，实际实现中可能发现此方案不可行或成本高于逐类型独立编写的 54 个方案
- **改进建议**：补充 CangJie 测试框架中实现参数化测试模板的具体机制或约束说明；或明确标注"18 个测试仅为理论最大值，实际最小单位应按矩阵类型独立编写"并给出 54 个方案的独立成本评估

## 3. 深度与完整性

### 3.1 [一般] §6 依赖分析的执行顺序建议中缺少对风险评估

- **位置**：§6 推荐执行顺序表格（第 618-650 行）
- **问题描述**：Phase 2 建议"建议直接覆盖 Float 变体"以消除 T6/T7→T16 前向依赖，但未评估此建议的风险：Phase 2 实施者的 T16 覆盖标准与后续 Phase 4 执行者实际采用的 T16 标准不一致的可能性
- **影响**：如果 Phase 2 执行者按自己的理解覆盖 Float 变体，而 Phase 4 T16 执行者有更严格的标准，则 T6/T7 的"提前覆盖"仍然需要返工
- **改进建议**：在 Phase 2 建议后追加约束条件——"仅当 Phase 2 执行者同时是 Phase 4 T16 的执行者，或 T16 覆盖标准在 Phase 2 前已冻结，此方案才有效"

### 3.2 [轻微] T15/T16 的依赖方向与时间逻辑矛盾

- **位置**：§6 组 B（第 639 行）
- **问题描述**：组 B 标注"先 T16 → 后 T15"，但 §4 T15 的类型约束说明（第 381 行）也陈述"应在 T16 完成后执行本项"。然而 Phase 4 的大分组中将 T15 和 T16 放在不同分组，未明确谁先执行
- **影响**：执行者看到 Phase 4 组 B 虽有依赖标注，但 T15 和 T16 分散在不同分组中，不易直观识别串行要求
- **改进建议**：在 Phase 4 分组中将 T15/T16 明确从不同分组中标注为"须串行：T16 先于 T15"，或将两者放入同一子组
