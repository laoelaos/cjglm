## 发现的问题

### 问题 1：Stage 6 范围描述存在内部逻辑矛盾

| 属性 | 内容 |
|------|------|
| **严重程度** | 严重 |
| **所在位置** | Stage 6 "范围"（第 202 行）与 "关键依赖"（第 207 行） |
| **问题描述** | "范围"中声明 `gtx/* 全部实验性扩展文件`（强调"全部"），而"关键依赖"中声明 `按需驱动：无强制迁移要求，根据用户需求选择性实施`（强调"选择性"）。同一阶段内，**"全部迁移"与"选择性迁移"直接矛盾**，使用者无法判断应按哪个指令执行——若按"全部"执行则过度投入，若按"选择性"执行则可能与"全部"声明的阶段承诺不符。 |
| **改进建议** | 统一表述。若意图为按需迁移，则将"范围"中的"全部"改为"选定的"或"按需选定的"；若意图为全部迁移，则修改"关键依赖"中的"按需驱动"描述，但后者与基线文档中 `gtx/` 按需迁移的建议（`01_roadmap.md` "第 5 轮及以后"节）冲突，建议采用前者。 |

### 问题 2：Stage 3 验证标准未区分 stub 影响下的可验证范围

| 属性 | 内容 |
|------|------|
| **严重程度** | 一般 |
| **所在位置** | Stage 3 "验证标准"（第 126-127 行），相关上下文在"范围"第 88 行 |
| **问题描述** | Stage 3 中 `ext/quaternion_transform.cj` 和 `ext/quaternion_common.cj` 标注为完整实现，但在第 88 行已说明"部分依赖 `geometric.cj` 的函数如 `lookRotate` 使用 stub 占位或延迟实现"。然而验证标准（第 126-127 行）列出了这些文件的核心功能验证项（如 `angleAxis`、`rotate`、`conjugate`、`inverse`、`normalize`），**未标注哪些验证项因底层 stub 占位而不可执行或仅部分可执行**。实施者拿到该文档后，可能花费时间调试因 stub 导致的预期失败（如 `lookRotate` 调用 `geometric.cj` stub 返回无意义值），无法区分"测试未通过"与"该功能被刻意延迟实现"。 |
| **改进建议** | 在验证标准中为每个依赖 stub 的验证项添加状态标注，例如标记 `[可验证]` 或 `[待 Stage 4]`；或在范围描述中明确声明哪些函数属于"延迟实现"并统一在验证标准中跳过。 |

### 问题 3：Stage 5 SIMD 可行性风险描述与基线文档已知限制不一致

| 属性 | 内容 |
|------|------|
| **严重程度** | 一般 |
| **所在位置** | Stage 5 "关键依赖"（第 183-184 行） |
| **问题描述** | 第 184 行将 SIMD 支持描述为"仓颉编译器对 SIMD 基础类型的支持程度（需在实施时确认）"，措辞暗示这只是一个"待确认"事项。然而基线文档（`01_roadmap.md`，第 364 行及 4.5.5 节前文关于 `storage` 的 SIMD 依赖说明）已明确记载：**CangJie 当前版本不提供 SIMD 基础类型**。这不是"待确认"的问题，而是一个已知的缺失能力。当前表述过于乐观，可能误导规划者假设 SIMD 在工作启动时即可获得支持。 |
| **改进建议** | 将"需在实施时确认"改为明确的风险标注，引用基线文档结论。例如："已知限制：CangJie 当前版本不提供 SIMD 基础类型（基线文档已确认）。需在实施时确认仓颉工具链是否已新增 SIMD 支持——若无则本阶段范围不可行，需重新评估 SIMD 特化的替代路径或推迟本阶段。" |

### 问题 4：Stage 1 产出物中新增非基线文件未说明来源

| 属性 | 内容 |
|------|------|
| **严重程度** | 轻微 |
| **所在位置** | Stage 1 "范围"（第 20-21 行）及"产出物"（第 34 行） |
| **问题描述** | Stage 1 的产出物清单中包含 `type_fromBoolVec.cj`、`type_cast.cj`、`scalar_vec_ops.cj` 等文件，这些文件**未出现在基线文档的首轮关键文件清单中**（`01_roadmap.md` 第 3G 节 11 个核心文件 + 3 个 shim 文件）。读者交叉引用基线文档时无法找到这些文件的对应关系，可能产生困惑：它们是 CangJie 适配新增文件？还是对应某个未列出的 GLM 源文件？当前文档未做任何说明。 |
| **改进建议** | 在产出物清单或范围描述中对这些**非 GLM 源文件**（即 CangJie 特有适配文件）添加简短标注，如注明"新增 CangJie 适配"或指出其对应的 GLM 功能领域，以便读者理解其来源和必要性。 |
