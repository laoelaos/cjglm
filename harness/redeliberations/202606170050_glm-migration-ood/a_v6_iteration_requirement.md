## 当前审查结果

1. **（严重）§9 异常场景与边界条件内容完全省略**：§9 标题下仅保留小节标题，正文写"（与 v3 一致，略去具体内容以节省篇幅）"。编码人员无法独立从 v5 文档获取零向量行为、除零、NaN/Infinity 传播、类型转换边界等关键边界约定，需求明确要求"设计产物需可供后续编码阶段直接使用"。改进建议：恢复 §9 的全部具体内容，或以条文形式直接将关键边界约定写回本节，确保 v5 自洽完整。

2. **（严重）`isIec559Of<T>()` 的编译期类型分发实现路径未验证**：设计多处依赖 `const if (isIec559Of<T>())` 实现编译期分支（§3.5、§3.6、§4.5）。但 §3.6 描述的"为每类数值类型提供重载函数"在仓颉泛型系统中不可行——仓颉泛型函数不允许按类型参数约束进行重载，亦不支持 C++ 式泛型函数全特化。主设计路径的可行前提尚未验证。改进建议：明确 `isIec559Of<T>()` 的仓颉实现方案——① 在函数体内使用 `const if` 配合类型比较（需验证仓颉是否支持 `T == Float32` 语法）；② 放弃编译期分支统一使用 `==` 精确比较；③ 采用反射或注解方式。

3. **（高）构造函数签名中 Vec1 类型参数不完整**：§4.1 示例性构造函数签名中多处引用 `Vec1<T2>` 而非 `Vec1<T2, Q2>`，且文档未明确 Vec1~Vec4 的 Q 参数是否有默认值。改进建议：在 §3.2 或 §3.1 中明确声明默认 Qualifier 策略，同步修正 §4.1 中所有 `Vec1<T2>` 引用为 `Vec1<T2, Q2>`。

4. **（高）`bitwiseNot()` 在浮点类型 Vec 上的编译错误未说明**：§4.4 描述 `bitwiseNot()` 通过 `!` 运算符逐分量实现，但未说明当 `T = Float32` 或 `Float64` 时仓颉浮点类型不支持 `!` 一元运算符，`VecN<Float32, Q>.bitwiseNot()` 将在实例化时产生编译错误。改进建议：补充说明 `bitwiseNot()` 对浮点类型 T 的 Vec 实例产生编译错误，与 D5（宽松泛型约束，延迟检查）一致。

5. **（中）`@OverflowWrapping` 标注计数内部不一致**：§4.6 总结语句称"共约 28 处标注"，但前文详细清单列出了 `%`、`+`、`-`、`*`、`/`（5 个）、`<<`（1 个）、`increment()`/`decrement()`（2 个），合计应为 (5+1+2)×4 = 32 处。若 `%` 未计入，则与 §4.3 和 §4.6 中明确将 `%` 列为标注对象相矛盾。改进建议：统一计数口径——若 `%` 计入则修正为"共约 32 处标注"；若不计入则说明理由。

6. **（中）256 个别名缺乏首轮必选/按需的实施指引**：§3.8 按使用频率将 256 个别名分为三级（必备~32、常用~64、可选~160），但未明确首轮编码阶段实际需要实现哪些级别的别名，也未与 roadmap 中"首轮 256 个别名"和"最简可行范围 48 个别名"对齐。改进建议：明确首轮别名层的实际实现范围，按分级实施策略在 §3.8 或 §8 中说明。

7. **（中）缺少实现验证/验收策略**：设计文档对 Vec 结构体的构造、运算、边界行为做了详尽约定，但未提及编码完成后如何验证实现的正确性，未引用 roadmap §4.4 的验证标准。改进建议：在 §5 后新增一节，引用 roadmap §4.4 的验证标准摘要，或自行定义一组最小验收用例。
