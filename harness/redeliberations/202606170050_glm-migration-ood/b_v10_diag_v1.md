# 质量审查报告 — v10 设计文档

## 审查结论

经系统审查，产出存在 **3 个严重**、**2 个一般**、**2 个轻微**质量问题。设计文档在需求响应充分度（roadmap §3E/§3G 完整追溯、范围可追溯性对照表）和整体深度方面经过 9 轮迭代已较为成熟，但在**下游消费者完整使用场景覆盖**和**与 C++ GLM 源文件的构造语法对齐**方面仍存在遗漏。

---

## 问题清单

### P1（严重）— 缺少 `Hashable` 实现，Vec 类型无法作为哈希键正常使用

**位置**：§3.2（Vec 结构体系）、§4（关键行为契约）、§12（验收标准）

**问题描述**：设计文档为 Vec 结构体自定义了 `operator==`（通过 `ComputeEqual<T>` 实现逐分量比较，对浮点类型使用 Epsilon 容差比较），但完全未提及 `Hashable` 接口的实现策略。仓颉中 `Hashable` 并非自动实现的核心接口——需通过 `@Derive[Hashable]` 或手动定义 `hashCode(): Int64` 来提供。缺少 `Hashable` 实现导致：
1. Vec 类型无法用作 `HashMap`/`HashSet` 的键类型（若底层使用默认 fallback 则行为不确定）
2. 即使通过 `@Derive[Hashable]` 自动派生，其按位哈希策略与浮点 Epsilon 容差 `==` 语义存在根本性不一致——两个 epsilon-相等的 `Vec4<Float32>` 实例的 `hashCode()` 可能不同，违反 `a == b ⇒ a.hashCode() == b.hashCode()` 契约

C++ GLM 中 vec 类型也不提供 `std::hash` 特化（需用户自行提供），但仓颉中 `Hashable` 的缺失影响更直接——无法通过编译的 `HashMap<Vec2<Int32>, T>` 用例。

**改进建议**：
1. 在 §3.2 或 §7 中新增设计决策，明确 Vec 类型的 `Hashable` 实现策略。推荐方案：对整数/Bool 分量 Vec 使用 `@Derive[Hashable]`（自动按成员 `x,y,z,w` 计算 hash）；对浮点分量 Vec 的记录为已知限制——不实现 `Hashable`（或提供精确按位的 `hashCode()` 并标注行为差异）。
2. 若采用方案，在 §4 中补充 `hashCode()` 的签名和行为约定。
3. 在 §12.2 验收标准中新增"Vec 类型在哈希集合中的可用性"验收项。

### P2（严重）— Vec2 缺少 `Vec1+Vec1` 构造函数，与 C++ GLM `type_vec2.inl` 行为不一致

**位置**：§4.1（向量构造 — Vec2 构造函数清单）

**问题描述**：C++ GLM `type_vec2.inl:73-78` 定义了 `vec<2, T, Q>(vec<1, A, Q>, vec<1, B, Q>)` 构造函数——从两个 `vec1` 构造 `vec2`。当前设计 §4.1 的 Vec2 构造函数清单包含 `Vec1+T` 和 `T+Vec1`，但**不包含 `Vec1+Vec1`**。此构造函数在 GLM 中有实际使用场景（从两个标量向量组合为二维向量），且与 Vec4 的 `Vec2+Vec2`、`Vec1+Vec1+...` 等组合构造模式一致，属于逻辑上的一致性缺口。

**改进建议**：在 Vec2 构造函数清单中补充：
```
public init<T2, Q2>(a: Vec1<T2, Q2>, b: Vec1<T2, Q2>) where Q2 <: Qualifier
```
注意需评估与 D31（跨类型 fill-from-Vec1 的限制）是否产生交互影响——此构造函数参数为两个 Vec1，不会与单一 `Vec1<T,Q>` 的 fill 构造函数构成重载冲突。

### P3（严重）— Vec3 缺少 4 个 `Vec1` 参与的组合构造函数，与 C++ GLM `type_vec3.inl` 行为不一致

**位置**：§4.1（向量构造 — Vec3 构造函数清单）

**问题描述**：C++ GLM `type_vec3.inl` 定义了以下包含 `vec1` 的组合构造函数，当前设计均缺失：

| GLM C++ 源文件行 | 构造函数 | 设计是否包含 |
|-----------------|---------|------------|
| `type_vec3.inl:80-84` | `Vec3(Vec1, Vec1, T)` | ✗ 缺失 |
| `type_vec3.inl:94-100` | `Vec3(Vec1, T, Vec1)` | ✗ 缺失 |
| `type_vec3.inl:102-108` | `Vec3(T, Vec1, Vec1)` | ✗ 缺失 |
| `type_vec3.inl:110-116` | `Vec3(Vec1, Vec1, Vec1)` | ✗ 缺失 |

设计当前仅覆盖 `Vec1+T+T`、`T+Vec1+T`、`T+T+Vec1` 三种模式，缺失 Vec1 在多个位置参与的组合构造。Vec4 的构造函数清单（§4.1）已覆盖 `Vec1` 在各位置的完整组合模式（约 20+ 个重载），Vec3 的覆盖却少于此模式，存在不一致性。

**改进建议**：在 §4.1 Vec3 构造函数清单中补充上述 4 个构造函数，保持与 Vec4 的 `Vec1` 组合构造覆盖模式一致。各构造函数标量参数位使用 `T` 类型（遵循 D26），Vec1 参数使用 `Vec1<T2, Q2>` 泛型。

### P4（一般）— `ComputeEqual` 被 `public import` 暴露为公共 API，泄漏实现细节

**位置**：§2（公共 API 面设计），`lib.cj` 的 `public import` 列表

**问题描述**：`lib.cj` 通过 `public import glm.detail.{ ComputeEqual }` 将 `ComputeEqual<T>` 暴露为 `glm` 包的公共 API。`ComputeEqual` 是 Vec 的 `operator==` 的内部比较策略实现，其公开签名（带 `const if` 浮点分支）和类型参数（单一 `T`）并非面向下游消费者的设计。下游消费者只需使用 `vec1 == vec2`，无需直接接触 `ComputeEqual`。将其纳入 `public import` 增加了不必要的 API 面，且未来如需修改 `ComputeEqual` 的实现细节（如改用不同的容差策略）时受到兼容性约束。

**改进建议**：从 `lib.cj` 的 `public import` 列表中移除 `ComputeEqual`。若 `ComputeEqual` 确实需要在 `operator==` 的 `const` 函数体中被引用且仓颉的类型可访问性规则要求其为 public，则保留但标注 `@Doc("internal")` 或类似标记，说明其非消费者的直接使用目标。否则（推荐），将其降级为 `internal` 包级可见类型，不在 `glm` 包中重新导出。

### P5（一般）— `isIec559Of<T>()` 实现中包含 `Float16` 类型增加不必要的验证风险

**位置**：§10，`isIec559Of<T>()` 实现代码（第 893-896 行）

**问题描述**：`isIec559Of<T>()` 的 const 实现代码中包含了 `T(0) is Float16` 分支。虽然 CFloat16 是仓颉合法的基本类型，但：
1. 首轮迁移范围不包含 `VecN<Float16>` 的别名或运算（256 个别名中不包含 `f16vec*`）
2. `T(0) is Float16` 在 const 上下文中对泛型 `T` 的合法性需额外验证——此验证对首轮无实际收益（因为 `Float16` Vec 不被使用），反而增加验证成本和潜在的编译器兼容性问题
3. 若 `T(0)` 对某些 `T` 类型在 const 上下文中不合法，整个函数定义失败，即使只关注 `Float32`/`Float64` 场景

当前实现将 `Float16` 与 `Float32`/`Float64` 并列作为 IEC 559 判定条件，使验证矩阵从 2 种类型扩至 3 种，且验证收益为零。

**改进建议**：
1. 将实现简化为仅检查 `Float32`/`Float64`：
   ```cangjie
   const func isIec559Of<T>(): Bool {
       const if (T(0) is Float64 || T(0) is Float32) { true } else { false }
   }
   ```
2. 在 §3.5 或 §10 的注释中注明"当后续轮次引入 `Float16` Vec 支持时，在此处追加 `T(0) is Float16` 判断"
3. 将 §10 取值对照表中的 `Float16` 行标注为"首轮不激活，待后续轮次"

### P6（轻微）— `bitwiseNot()` 对 `VecN<Bool,Q>` 的迁移影响未纳入 §11 迁移成本评估

**位置**：§3.2（`VecN<Bool,Q>` 行为约定）、§4.4（位运算）、§11（迁移成本评估）

**问题描述**：§3.2 和 §4.4 正确标注了 `bitwiseNot()` 对 `VecN<Bool,Q>` 的行为与 C++ GLM `~bvec` 的差异（逻辑否定 vs 整数提升取反），并提供了"编码阶段排除此函数"的建议。但 §11 迁移成本评估中未包含此差异的影响分析。当迁移代码包含 `~bvecExpr` 表达式时，编码人员需要知道：
1. 替换为 `!bvecExpr`（仓颉 `!` 可重载）但不能使用 `bitwiseNot()`
2. 或替换为 `!bvecExpr.logicalAnd(!bvecExpr)` 等复杂模式以匹配 C++ 的"全 true"行为
3. 这些替换模式的搜索规则和工作量未评估

§11.1 评估了 `&&`/`||` 的迁移成本，§11.2 评估了 `++`/`--` 的迁移成本，但 `~` 的迁移影响在 §11 中无对应章节。

**改进建议**：在 §11 中新增子节（如 §11.3），评估 `~bvec` 迁移为 `bitwiseNot()` 或 `!` 的影响，包括受影响的表达式模式、迁移规则和工作量。此评估应与 §3.2 的已知行为差异声明同步。

---

### P7（轻微）— `Format`/`Display`/`ToString` 输出未覆盖

**位置**：§3.2（Vec 结构体系）、§4（关键行为契约）、§7（设计决策）

**问题描述**：设计文档未提及 Vec 类型的字符串表示（`toString()` 或 `@Derive[ToString]`）。在调试、日志输出和测试断言失败信息中，Vec 类型的可读字符串表示是基本需求。仓颉 `@Derive[ToString]` 可自动生成基于 `"Vec2(1, 2)"` 格式的字符串，但设计文档未做任何约定，可能在编码阶段产生不一致的 `toString()` 实现。

**改进建议**：在 §3.2 或 §7 中新增设计决策，约定 Vec 类型的字符串表示策略。推荐：使用 `@Derive[ToString]` 获取默认格式，或自定义 `toString(): String` 输出 `"Vec2<Int32, PackedHighp>(1, 2)"` 格式。在 §12.2 验收标准中可选补充对应的验证项。

## 整体质量评价

设计文档经过 9 轮迭代审查后，在**范围覆盖**（roadmap 完全追溯）、**技术可行性**（`@OverflowWrapping` 策略、`const` 表达式链、`isIec559Of<T>()` 实现路径等高风险项均已识别和备选方案化）和**编码就绪度**（完整构造函数签名清单、设计决策记录、迁移成本评估）方面已达到较高成熟度。

主要薄弱环节在于**下游消费者完整使用场景**的覆盖——`Hashable` 缺失影响标准库集合类型的可用性，`ToString` 缺失影响可调试性。此外，与 C++ GLM 源文件的构造函数签名对齐方面（Vec2 的 Vec1+Vec1、Vec3 的多个 Vec1 组合）存在遗漏，可直接导致编码阶段出现"GLM C++ 代码可编译但仓颉对应代码无法编译"的迁移问题。
