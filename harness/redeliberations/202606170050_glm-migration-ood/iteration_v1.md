## 需要解决的问题（仅 RETRY 时存在）

- **问题描述**：Vec 结构体数据成员未指定 `let`/`var`，无法指导编码
- **所在位置**：Section 3.2（Vec 结构体系）
- **严重程度**：严重
- **改进建议**：明确指定 Vec 结构体的数据成员为 `var`，并在 Section 4.3 中补充说明 `var` 成员对 `let` 绑定变量的影响

- **问题描述**：`@OverflowWrapping` 标注策略与仓颉复合赋值自动生成机制存在矛盾
- **所在位置**：Section 4.6 + D3
- **严重程度**：严重
- **改进建议**：明确分析仓颉复合赋值自动生成对设计方案的影响；可能的替代路径：① 直接在二元运算符上标注 `@OverflowWrapping`；② 验证显式 `mut operator func +=`（带注解）的优先级高于自动生成版本

- **问题描述**：缺少 `length()` 成员函数定义
- **所在位置**：Section 3.2 + Section 4（行为契约）
- **严重程度**：严重
- **改进建议**：在 Section 4 中为每个 Vec 类型补充 `length()` 静态成员函数，统一签名 `public static func length(): Int64`

- **问题描述**：`!=` 运算符语义描述使用了仓颉中不合法的 `!==` 概念
- **所在位置**：Section 4.5（比较运算）
- **严重程度**：一般
- **改进建议**：将 `!==` 语义改为 `!(a == b)` 或直接描述为"对 `==` 的结果取反"

- **问题描述**：构造函数体系缺少完整枚举，不足以支撑编码
- **所在位置**：Section 4.1（向量构造）
- **严重程度**：一般
- **改进建议**：在 Section 4.1 中为每个 Vec 类型列出完整的构造函数重载签名清单

- **问题描述**：缺少 `operator` 重载所属位置的精确描述
- **所在位置**：Section 3.2（Vec 结构体系，运算符位置部分）
- **严重程度**：一般
- **改进建议**：将此处改为"复合赋值运算符在 Vec 结构体内定义为 `mut operator func`；二元运算符通过 `extend VecN<T,Q>` 块定义为扩展成员函数"

- **问题描述**：`scalar + vec` 不可行的结论未在设计文本中体现
- **所在位置**：Section 4.3（算术运算）
- **严重程度**：一般
- **改进建议**：明确标注 `scalar + vec` 方向不支持运算符重载；若需要保留该语义，提供具名函数替代方案

- **问题描述**：文件迁移表中 `setup.cj` 的依赖声明与 Shim 职责重叠
- **所在位置**：Section 8（迁移文件清单与依赖拓扑），文件 1 `setup.cj`
- **严重程度**：一般
- **改进建议**：明确 `LengthT`/`SizeT` 的归属，建议归入 `shim_cstddef.cj` 且 `setup.cj` 通过 `import` 引用

- **问题描述**：缺少 `countof` 辅助函数的说明
- **所在位置**：Section 8 文件 1 `setup.cj` 的说明列 / Section 3.6 Shim 层
- **严重程度**：一般
- **改进建议**：在 Section 8 和 Section 3.6 中补充 `countof` 函数

- **问题描述**：`ComputeEqual` 的 `IsFloat` 编译期判定方案存在不确定性
- **所在位置**：Section 3.5（ComputeEqual）
- **严重程度**：一般

- **问题描述**：缺少 `Vec1` 数据成员命名
