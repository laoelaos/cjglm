## 问题清单

### P1（严重）— `operator[]` 赋值形式缺少 `mut` 修饰符

- **位置**：§4.2（分量访问），第 424 行
- **问题描述**：`operator[]` 赋值形式签名声明为：
  ```
  public operator func [](i: Int64, value!: T): Unit
  ```
  缺少 `mut` 修饰符。根据仓颉语言规则（struct README §3），struct 的普通实例成员函数**不能**修改实例的成员变量——只有 `mut` 实例成员函数可以就地修改。`operator[]` 赋值形式的语义是需要修改 Vec 实例的 `x`/`y`/`z`/`w` 成员，因此必须声明为 `mut` 函数：
  ```
  public mut operator func [](i: Int64, value!: T): Unit
  ```
  若不添加 `mut`，即使通过 `var` 绑定的 Vec 变量调用 `v[i] = val`，也无法成功修改成员——函数体内的 `this.x = value` 等赋值操作将产生编译错误，因为普通实例函数中的 `this` 不可修改（在 struct 中 `this` 不可作为表达式使用），且无法对成员变量赋值。
- **严重程度**：严重 — 编码阶段若按当前签名实现将无法编译通过或赋值不生效
- **改进建议**：在 `operator[]` 赋值形式的签名中添加 `mut` 修饰符，更新 §4.2 中的签名声明。同时检查 extend 块中是否定义了赋值形式的 `[]`，若在 extend 块中定义也需 `mut`。

### P2（中高）— `const` 函数中复合表达式兼容性风险系统性识别不足

- **位置**：§3.5（ComputeEqual）、§3.6（shim_limits.cj）、§4.5（比较运算）
- **问题描述**：设计中的 `operator ==` 被声明为 `const` 函数（§3.2 第 148 行），函数体内使用 `const if (isIec559Of<T>())` 分支，在浮点路径中执行：
  ```
  abs(a - b) <= epsilonOf<T>()
  ```
  根据仓颉 const README §3.2 规则 3：**"`const` 函数中的表达式都必须是 const 表达式"**（`const init` 函数除外）。这意味着上述表达式中的所有操作——`abs()`、`-`、`<=`、`epsilonOf<T>()`——都必须是合法的 const 表达式。
  设计文档在第 10 章（第 857 行）仅标注了 `T(0)` 在 const 上下文中的合法性待验证，但**未评估** `abs()` 函数在 const 上下文中的可用性（`abs` 是否声明为 `const` 函数）以及 `epsilonOf<T>()` 的 const 兼容性（`NumericLimits<T>.epsilon()` 是否可在 const 泛型函数中求值）。
  即使上述表达式在 const 上下文中合法，当 `==` 在**运行时**调用时，`const if` 需要条件为 const 表达式，`isIec559Of<T>()` 的 `const if (T(0) is Float64)` 方案已在第 10 章标注为需验证。
- **严重程度**：中高 — 若任意环节不满足 const 表达式约束，`==` 的 const 函数路径将无法在 const 上下文中使用；虽提供了"精确比较"回退方案（第 245 行），但 const 函数整体的可行前提链条需更系统的评估
- **改进建议**：在 §3.5 或 §7 中新增一个设计决策，系统性罗列 `const operator ==` 完整表达式链的 const 兼容性要求（`abs`, `epsilonOf<T>()`, `T(0)` 构造, `is` 运算符, `-`/`<=` 运算符），标注每项的验证状态和回退方案。将"编码前验证要求"从第 10 章扩展覆盖至全链条。

### P3（中高）— `@OverflowWrapping` 标注继承性未经验证

- **位置**：§4.6（`@OverflowWrapping` 标注策略），第 485 行
- **问题描述**：设计断言"编译器自动生成的复合赋值版本继承二元运算符上的 `@OverflowWrapping` 标注"（第 485 行），且将此作为标注策略的核心前提。查阅仓颉语言特性文档（function README §8.5），明确了"重载二元运算符时自动启用对应复合赋值版本"的规则，但**文档中未述及**标注继承行为——即自动生成的 `+=` 是否继承了二元 `+` 上的 `@OverflowWrapping`。这是设计对编译器行为的推测，若无编译期验证，此策略的基础假设可能不成立。
  若编译器自动生成的复合赋值**未继承** `@OverflowWrapping` 标注，则复合赋值使用默认的 `@OverflowThrowing` 策略——整数溢出时抛 `ArithmeticException`，与设计期望的 wrapping 行为完全相反。而且此问题在编译阶段不会报错（仅在运行时产生不同的溢出行为），属于静默语义偏差。
- **严重程度**：中高 — 标注继承性是整个溢出标注策略的基础，若假设不成立则 §4.6 的策略失效
- **改进建议**：在 `setup.cj` 中编写原型测试代码验证 `@OverflowWrapping` 是否被自动生成的复合赋值继承。具体：编写一个简单的 struct，定义带 `@OverflowWrapping` 的二元 `+`，调用 `+=` 验证溢出行为。在 §4.6 中增加验证记录的引用，或附备选方案（如所有 Vec 运算通过 `extend` 块中的具名函数 `addAssign` 等实现）。

### P4（中等）— `Vec<Bool, Q>` 抽象类型引用未彻底消除

- **位置**：§3.2（Vec 结构体系），第 161-167 行
- **问题描述**：设计持续使用 `Vec<Bool, Q>` 作为集体指称——如"`Vec<Bool, Q>` 完整行为约定（集中说明）"及随后的"Bool 分量 Vec 支持..."等描述。然而实际设计中不存在 `Vec<Bool, Q>` 这个类型，只有 `Vec1<Bool, Q>`、`Vec2<Bool, Q>`、`Vec3<Bool, Q>`、`Vec4<Bool, Q>` 四个独立结构体。
  第 3 轮迭代中此问题已被标记为**严重**（迭代历史第 3 轮，问题 2），要求"将所有 `Vec<Bool, Q>` 引用替换为每个 VecN 类型的具体版本，或统一表述为`各 VecN 类型分别定义对应版本`"。v4~v9 的多轮修订中，部分位置已改为 `VecN<Bool, Q>`（如第 474-475 行的 `logicalAnd`/`logicalOr` 签名），但 §3.2 的集中说明段落仍使用 `Vec<Bool, Q>` 格式。编码阶段开发者若按此签名搜索类型，将找不到匹配的类型定义。
- **严重程度**：中等 — 历时多轮迭代未彻底修复，影响文档精确度和编码就绪度
- **改进建议**：将 §3.2 第 161-167 行的所有 `Vec<Bool, Q>` 替换为 `VecN<Bool, Q>`（其中 N=1,2,3,4）或明确说明"以下约定适用于 Vec1~Vec4 分别以 Bool 为 T 参数的实例"。同步检查全文是否还有遗漏的 `Vec<Bool, Q>` 或 `Vec<bool, Q>` 引用。

### P5（中等）— 跨类型 fill-from-Vec1 构造器缺失限制未记录为设计决策

- **位置**：§4.1（向量构造），Vec2~Vec4 的从 Vec1 构造部分；§7（设计决策）
- **问题描述**：Vec2~Vec4 的"从 Vec1 填充全部分量"构造器仅支持同一类型的 Vec1：
  - Vec2: `public init(v: Vec1<T, Q>)`（第 348 行）
  - Vec3: `public init(v: Vec1<T, Q>)`（第 367 行）
  - Vec4: `public init(v: Vec1<T, Q>)`（第 415 行）
  不支持跨类型 `Vec1<T2, Q2>` 的填充。若编码阶段需要将 `Vec1<Float32, PackedHighp>` 填充到 `Vec3<Int32, PackedLowp>`，必须通过组合构造器（如 Vec3 的 `init<T2, Q2>(a: Vec1<T2, Q2>, b: T, c: T)` 配合三个参数调用）实现，无法直接使用简洁的 `init(v: Vec1)` 形式。
  此限制源于仓颉函数重载规则——泛型构造器 `init<T2, Q2>(v: Vec1<T2, Q2>)` 与非泛型 `init(v: Vec1<T, Q>)` 在参数类型对齐后无法构成有效重载（类型变量约束不参与重载判断，function README §7.1），因此二者不能共存。但此限制**未在 §7 中以设计决策形式记录**，编码人员无法从设计文档理解其约束来源，可能误判为遗漏。
- **严重程度**：中等 — 影响构造便利性和编码阶段对构造器清单完整性的信任
- **改进建议**：在 §7 中新增一个设计决策（D28），记录"跨类型 fill-from-Vec1 构造器不可与同类型版本共存"的约束及理由，并标注可行的替代构造路径。在 §4.1 对应的构造器签名旁添加注释引用该决策。

### P6（轻微）— `increment()`/`decrement()` 语义差异未充分评估

- **位置**：§4.3（算术运算，第 443 行）、§11.2（迁移成本评估）
- **问题描述**：设计将 `++`/`--` 替代为 `increment()`/`decrement()` 具名函数，声明其与 C++ GLM 的 `operator++` "回新副本的行为一致"。但根据仓颉基本数据类型文档（basic_data_type README §11.7）：**"`expr++` / `expr--` — 仅后缀，仅整数类型"** 且 **"结果类型为 `Unit`（不能用作值）"**。
  这意味着仓颉原生的 `++` 在值语义和返回值上都与 C++ 的 `operator++` 有本质差异——C++ 前缀返回引用、后缀返回旧值副本，而仓颉原生 `++` 返回 `Unit` 且不可用作表达式值。虽然设计选择**不重载** `++`/`--`（因不在可重载列表），而是提供具名函数，但 §11.2 的迁移模式描述中将 C++ `v++` 映射为 `v = v.increment()` 时，忽略了原 C++ 代码中可能存在的 `auto old = v++` 模式（依赖后缀返回旧值语义）。§11.2 第 904 行虽提到了此情况需拆分，但整体迁移成本评估对"依赖 `++`/`--` 返回值的模式"的覆盖不完整。
- **严重程度**：轻微（但若迁移场景复杂可能升级）
- **改进建议**：在 §11.2 中补充典型 C++ `++`/`--` 使用模式的分类与映射表——特别包括 `for (auto& v : vecs) { ++v; }`（纯副作用）、`auto old = v++`（依赖旧值）、`func(v++)`（表达式内嵌）等模式，明确各类别的迁移工作量和正确映射方式。

### P7（轻微）— 配置常量表中 `Int` 类型在仓颉中非标准类型

- **位置**：§7 D6，第 604 行
- **问题描述**：配置表中 `GLM_CONFIG_CLIP_CONTROL` 和 `GLM_CONFIG_SWIZZLE` 等的仓颉类型声明为 `Int`。仓颉基本数据类型中的整型为 `Int8`/`Int16`/`Int32`/`Int64`/`IntNative` 等，**无未修饰的 `Int` 类型**。`func main() { let x: Int = 0 }` 会产生编译错误——正确的类型应为 `Int64`（仓颉默认整数类型）。
  此问题影响 `setup.cj` 的编译通过性。
- **严重程度**：轻微（机械可修复，不影响整体设计）
- **改进建议**：将 D6 配置常量表中的所有 `Int` 类型声明改为 `Int64`，同步修正 §8 迁移文件清单中 `setup.cj` 的对应说明。

## 整体评价

经过 9 轮迭代和前述 25+ 项设计决策（D1~D27）的累积，本设计方案在整体质量、一致性和完整性上已达到较高水平。存在的主要问题集中在对编译器行为的假设验证层面（P2、P3、P1）和少量残留的符号/类型不一致问题（P4、P7）。P1（缺失 `mut`）和 P7（`Int` 类型）如不修复将直接产生编译错误；P2 和 P3 的假设风险应在编码启动前通过原型测试验证。

**编码就绪度评级：有条件就绪** — 在关闭 P1（mut）和 P7（Int→Int64）两个直接阻断性问题，并对 P2（const 表达式链）和 P3（标注继承性）完成原型验证后，可正式进入编码阶段。
