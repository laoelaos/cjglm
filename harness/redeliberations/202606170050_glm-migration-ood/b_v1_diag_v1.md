# 质量审查报告（v1）

审查目标：`a_v1_design_v1.md` — GLM 1.0.3 仓颉迁移首轮 OOD 设计方案
审查轮次：第 1 次（首轮审查）
审查范围：需求响应充分度、事实/逻辑正确性、深度与完整性（侧重内部审议未覆盖维度）

---

## 问题清单

### P1 [严重] Vec 结构体数据成员未指定 `let`/`var`，无法指导编码

**所在位置**：Section 3.2（Vec 结构体系）

**问题描述**：设计方案描述 Vec 数据成员为 `T x, y, z, w` 等，但未指定是 `let`（不可变）还是 `var`（可变）。仓颉 struct 中 `let` 成员不可修改，而复合赋值运算符（`+=`、`-=` 等）需要 `mut operator func` 修改 `this` 的成员变量。若采用 `let`，则 `mut operator func +=` 中无法通过 `this.x = this.x + rhs` 修改成员；若采用 `var`，则 Vec 实例的赋值/传参语义需要额外关注。此选择是 Vec 类型完整实现的先决条件，不明确将直接阻塞编码阶段。

**改进建议**：明确指定 Vec 结构体的数据成员为 `var`（以支持复合赋值运算符的原地修改），并在 Section 4.3 中补充说明 `var` 成员对 `let` 绑定变量的影响（`let` 声明的 Vec 变量上无法调用 `mut` 运算符）。

---

### P2 [严重] `@OverflowWrapping` 标注策略与仓颉复合赋值自动生成机制存在矛盾

**所在位置**：Section 4.6 + D3

**问题描述**：设计方案采用"拷贝 + 复合赋值委托"模式——二元运算符委托给复合赋值运算符，仅在复合赋值运算符上标注 `@OverflowWrapping`，声称"二元算术运算符通过委托自动继承 wrapping 语义，无需重复标注"。然而，仓颉语言的运算符重载规则（Section 8.5）规定：重载二元运算符时**自动启用对应复合赋值版本**（前提是返回类型与左操作数类型匹配）。这意味着：

1. 若先定义 `operator func +(rhs)` 并返回 `Vec<T>`，编译器可能自动提供 `+=` 版本（但不带 `@OverflowWrapping`），而该自动生成的版本**不含 wrapping 语义**（缺省为抛异常），与设计意图相悖。
2. 若先显式定义 `mut operator func +=`（带 `@OverflowWrapping`）再定义 `+`，两者之间的"委托"关系是否与自动生成机制冲突，设计方案未做分析。
3. 方案所需的"先有 `+=`、后有 `+` 委托 `+=`"的时序和自动生成的 `+=` 之间的关系完全未讨论。

**改进建议**：明确分析仓颉复合赋值自动生成对设计方案的影响。可能的替代路径：① 直接在二元运算符上标注 `@OverflowWrapping`，不依赖自动继承；② 验证显式 `mut operator func +=`（带注解）的优先级高于自动生成版本，仅在确认可行后保留原方案。建议在设计中加入仓颉语言规则引用作为决策依据。

---

### P3 [严重] 缺少 `length()` 成员函数定义

**所在位置**：Section 3.2 + Section 4（行为契约）

**问题描述**：GLM C++ 的 `vec` 类型提供 `length()` 静态成员函数（返回分量数，如 `vec4::length()` 返回 4）。设计方案中 Vec 结构体系（Section 3.2）和关键行为契约（Section 4）均未提及 `length()` 方法。对于下游消费者，获取向量维度是最基本的能力——用于泛型代码、循环访问、维度匹配校验等。缺少 `length()` 意味着编码阶段需要自行解决维度查询问题，且不同 Vec 类型返回不同值（1~4），若通过各自独立硬编码实现会偏离设计方案中"四个独立结构体"的初衷。

**改进建议**：在 Section 4 中为每个 Vec 类型补充 `length()` 静态成员函数（返回编译期常量 `1`/`2`/`3`/`4`），或通过仓颉编译期常量属性实现。建议统一签名：`public static func length(): Int64`。

---

### P4 [一般] `!=` 运算符语义描述使用了仓颉中不合法的 `!==` 概念

**所在位置**：Section 4.5（比较运算）

**问题描述**：原文 "`!=`：`!==` 语义——任一分量不等返回 `true`"。仓颉中 `!==` 是引用类型的**严格不等**操作符，用于判断两个引用是否指向不同对象，对于值类型（struct）不存在 `!==` 操作。Vec 是 struct（值类型），`!=` 的正确语义应是对 `==` 结果取反（仓颉中 `!=` 默认被编译为 `!(a == b)`），或显式定义 `operator func !=()`。使用 `!==` 术语会造成实现者的误解，可能写出不编译或语义错误的代码。

**改进建议**：将 `!==` 语义改为 `!(a == b)` 或直接描述为"对 `==` 的结果取反"。若考虑性能，可说明"逐分量不等比较，任一分量不等即返回 `true`"。

---

### P5 [一般] 构造函数体系缺少完整枚举，不足以支撑编码

**所在位置**：Section 4.1（向量构造）

**问题描述**：设计方案将 Vec 构造函数归类为"默认构造、标量填充、逐分量、跨 Vec 转换、跨分量数构造、从 Vec1 显式构造"，但未给出每个 Vec 类型实际支持的具体构造函数签名。以 Vec4 为例，GLM C++ 支持至少以下形式：`Vec4(T,T,T,T)`、`Vec4(T)`、`Vec4(Vec2,S,S)`、`Vec4(S,Vec2,S)`、`Vec4(S,S,Vec2)`、`Vec4(Vec2,Vec2)`、`Vec4(Vec3,S)`、`Vec4(S,Vec3)`、`Vec4(Vec1*4)`、跨类型 `Vec4<U,Q2>(Vec4<U,Q2>)` 等。编码阶段需要明确的构造函数签名清单，设计师仅给出分类无法直接指导实现。同样的问题也存在于 Vec2、Vec3。

**改进建议**：在 Section 4.1 中为每个 Vec 类型（至少为 Vec2/Vec3/Vec4）列出完整的构造函数重载签名清单，包括跨分量数的组合规则。可使用表格形式并标注哪些是 GLSL 5.4.1 规范必选、哪些是可选的 GLM 扩展。

---

### P6 [一般] 缺少 `operator` 重载所属位置的精确描述

**所在位置**：Section 3.2（Vec 结构体系，运算符位置部分）

**问题描述**：内部审议已指出设计方案原文"仓颉的运算符重载语法要求在结构体/类外部定义非成员运算符函数"为事实错误（仓颉运算符只能在 `class`/`struct`/`interface`/`enum`/`extend` 内部定义，并非包级函数），但设计方案 v1 文件尚未修正此表述。主文本仍保留"运算符均在结构体外部定义为包级函数"的表述，与仓颉语言规则冲突。

**改进建议**：按照内部审议给出的修改建议修正：将此处改为"复合赋值运算符在 Vec 结构体内定义为 `mut operator func`；二元运算符（如 `+`/`-`/`*`）通过 `extend VecN<T,Q>` 块定义为扩展成员函数，以保持关注点分离"。

### P7 [一般] `scalar + vec` 不可行的结论未在设计文本中体现

**所在位置**：Section 4.3（算术运算）

**问题描述**：内部审议已指出"二元算术运算符...支持左右交换（scalar + vec 和 vec + scalar）"在仓颉中不可行（标量为内置类型，无法在其上定义运算符重载），但设计方案 v1 文本仍保留"支持左右交换"的表述，未按审议建议修正或提供替代方案（如具名函数 `add(scalar, vec)`）。这会将不可实现的约束传递给编码阶段。

**改进建议**：① 明确标注 `scalar + vec` 方向不支持运算符重载；② 若需要保留该语义，提供具名函数替代方案并列入设计决策；③ 同时更新依赖此约束的行为契约描述。

---

### P8 [一般] 文件迁移表中 `setup.cj` 的依赖声明与 Shim 职责重叠

**所在位置**：Section 8（迁移文件清单与依赖拓扑），文件 1 `setup.cj`

**问题描述**：迁移表显示 `setup.cj` 的依赖为"无"，但"说明"列列出其提供"配置常量、`LengthT`/`SizeT`、`assert`"。而文件 3 `shim_assert.cj` 和文件 5 `shim_cstddef.cj` 也分别提供 `assert()` 和 `SizeT`/`LengthT` 别名。若 `setup.cj` 自己实现 assert 和 SizeT/LengthT，则 shim 文件重复；若 `setup.cj` 从 shim 文件引用，则依赖关系应标注为对 `shim_assert.cj` 和 `shim_cstddef.cj` 的依赖。当前"依赖=无"的标注造成了实现者无法判断"LengthT 应定义在哪里"。

**改进建议**：明确 `LengthT`/`SizeT` 的归属——建议归入 `shim_cstddef.cj` 且 `setup.cj` 通过 `import` 引用，并在依赖列更正为 `shim_cstddef.cj`。`assert` 同理。或者在 Section 2 的模块结构中移除 shim 文件中与 setup.cj 重叠的部分。

---

### P9 [一般] 缺少 `countof` 辅助函数的说明

**所在位置**：Section 8 文件 1 `setup.cj` 的说明列 / Section 3.6 Shim 层

**问题描述**：roadmap（`docs/01_roadmap.md`）明确指出 `setup.hpp` 等效实现中包含 `countof` 辅助函数（计算数组元素数），是基础设施层的必要组成部分。设计方案中 `setup.cj` 的说明未提及 `countof`，Section 3.6 的 Shim 层也未列举。该函数在 C++ GLM 中被 `compute_vector_decl.hpp` 等文件引用，首轮迁移中若有依赖则可能导致缺失。

**改进建议**：在 Section 8（文件 1 说明列）和 Section 3.6（Shim 层）中补充 `countof` 函数，并确认其是否被首轮范围内的文件引用。

---

### P10 [一般] `ComputeEqual` 的 `IsFloat` 编译期判定方案存在不确定性

**所在位置**：Section 3.5（ComputeEqual）

**问题描述**：设计方案定义 `ComputeEqual<T, IsFloat>` 并期望通过 `IsFloat` 布尔值在编译期区分浮点/整数类型，以选择容差比较或精确比较路径。但设计方案同时承认"若 shim 层无法提供 `is_iec559` 等效判断，则统一走精确比较路径"。这意味着 `ComputeEqual` 的类型参数设计可能需要在实施时变更（例如改为单一 `call` 方法 + 运行时判断）。设计方案未提供两种方案之间的切换策略，也未分析在仓颉泛型系统中如何实现"当 T 是浮点类型时激活 A 路径，否则激活 B 路径"的编译期分支（仓颉无 C++ SFINAE 或 `if constexpr`，`const + if` 仅限函数体内部）。

**改进建议**：建议在设计阶段明确 `ComputeEqual` 的最终实现路径。可能的方案：① 依靠 `shim_limits.cj` 提供 `isFloat<T>` 布尔常量，配合 `const if` 在函数体内分支（如可行）；② 若不支持编译期区分，则统一使用精确比较并记录此决策。方案确定前，当前设计存在实施风险。

---

### P11 [一般] 缺少 `Vec1` 数据成员命名

**所在位置**：Section 3.2（Vec 结构体系）

**问题描述**：设计方案描述 Vec 数据成员为 `T x, y, z, w`（数量对应分量数），但未明确 Vec1 的单数据成员名称。GLM C++ 中 `vec1` 的唯一成员名为 `x`（同时也通过 `r`/`s` 别名可访问）。命名不明确会导致编码阶段出现不一致。

**改进建议**：在 Section 3.2 中明确说明 Vec1 的数据成员名为 `x`（同时注明 `r`/`s` 作为分量名同义词的存在性——若首轮提供）。

---

### P12 [轻微] `const` 配置常量的作用域限制未在设计决策中提及

**所在位置**：Section 7 D6

**问题描述**：D6 说"`const` 常量是仓颉编译时求值的基本机制，可被 `if` 分支用于消除死代码"。但 roadmap（Section 0.4）明确指出 `const + if` 方案**仅适用于函数体内的内联条件代码**，**不适用于条件包含独立文件的场景**（如下游 SIMD 或 swizzle 的独立文件条件编译）。D6 未限定 `const + if` 的作用域边界，可能让实现者误以为 `const` 常量可以完全替代 C++ 预处理器条件包含，从而在不适当的场景中使用。

**改进建议**：在 D6 中补充一条限制说明：`const + if` 仅适用于函数体内的内联条件代码；涉及条件包含独立文件的配置（如 `GLM_CONFIG_SIMD` 控制 `type_vec_simd.inl` 的包含）必须通过其他机制（宏、构建配置）处理。

---

### P13 [轻微] 256 个别名的真正实用性未做分析

**所在位置**：Section 3.8（向量类型别名）

**问题描述**：设计方案声称 256 个别名（16 家族 × 4 分量数 × 4 精度变体），但未分析其中哪些组合实际有意义。例如 `bvec1`~`bvec4` + `PackedLowp`（低精度布尔）在语义上无意义；`i8vec1` 在实际使用中极少出现。256 个别名中有相当比例可能是"为存在而存在"的死别名。这增加了维护和测试开销，且可能因 CangJie 编译器对 `type` 别名的处理差异导致预料外的编译时间/产物膨胀。

**改进建议**：建议在 Section 3.8 或设计决策中增加一节，按使用频率对别名进行分级（必备/常用/可选），在首轮可只提供高优先级别名子集（如无精度前缀 + `highp_` 变体），其余列入手轮。这也有助于缩减首轮工作量和测试范围。

## 综合质量评价

本设计方案在架构思路、抽象层次和模块划分上具有较好的结构清晰度，核心抽象（Qualifier/Vec/Functor/ComputeEqual）的职责边界明确。但存在以下需优先处理的问题：

1. **编码阻碍项**（P1, P3）：Vec 成员的 `let`/`var` 未定和 `length()` 缺失是编码阶段的两个阻塞点，需在实施前明确。
2. **语言规则匹配度**（P2, P6, P7）：`@OverflowWrapping` 策略与仓颉自动生成机制的兼容性未做分析；已知的语言事实错误（运算符位置、`scalar+vec` 不可行）在 v1 文本中仍未修正。
3. **API 完备性**（P4, P5, P11）：构造函数和比较运算符的签名描述精度不足，编码阶段无法直接作为参考资料。
4. **内部一致性**（P8, P9）：依赖拓扑和职责分配存在模糊/冲突，需厘清。

建议在进入编码阶段前完成上述问题的逐项修复（优先级：P1~P3 > P4~P8 > P9~P13）。
