## 问题 1（严重）— 上一轮诊断指出的 bitwiseNot 对 VecN<Bool,Q> 行为等价性错误仍未修复

**所在位置**：§3.2（第 134 行）、§4.4（第 425-426 行）

**问题描述**：设计文档第 134 行和第 425-426 行仍声明 `bitwiseNot()` 在 `VecN<Bool, Q>` 上使用 `!` 运算符"语义与 C++ GLM 的 `~bvec` 一致"。但 C++ `~bool` 因整数提升机制始终返回 `true`，而仓颉 `!bool` 执行逻辑否定（`!true = false`）。两种行为完全相反。

**改进建议**：修正该声明，明确标注此为已知行为差异，或方案三——排除 `VecN<Bool, Q>` 的 `bitwiseNot()` 使其实例化时报编译错误。

**验证依据**：上一轮诊断报告已给出完整验证路径（`references/glm-1.0.3/glm/glm/detail/compute_vector_decl.hpp:178-186`），C++ 标准 `[expr.unary.op]`。

**状态**：上一轮已识别，设计文档修订说明（第 917-934 行）未覆盖此问题。

## 问题 2（严重）— 上一轮诊断指出的 GLM_CONFIG_CLIP_CONTROL 默认值错误仍未修复

**所在位置**：§7 D6 配置常量表格（第 560 行）

**问题描述**：`GLM_CONFIG_CLIP_CONTROL` 首轮推荐值仍为 `0x02`（=GLM_CLIP_CONTROL_RH_NO）。根据 GLM 源码位运算：

```
GLM_CLIP_CONTROL_RH_BIT = 8
GLM_CLIP_CONTROL_NO_BIT = 2
GLM_CLIP_CONTROL_RH_NO = RH_BIT | NO_BIT = 8 | 2 = 10 = 0x0A
```

`0x02` 仅对应 `NO_BIT`，丢失了 `RH_BIT`。此误差由 roadmap 传入设计文档（roadmap §3A 表格第 349 行同样为 `0x02`），设计文档在 7 轮审查中未检出。

**注意**：设计文档修订说明第 931 行（"D6 配置常量 `GLM_CONFIG_CLIP_CONTROL` 的类型选择"）仅涉及该常量的*类型选取*（Int 而非 enum），未修正数值。该事实错误仍然存在。

**改进建议**：将值更正为 `0x0A`（或十进制 `10`），并标注其语义为 `RH_BIT | NO_BIT`。同时反馈至 roadmap 文档以阻断误差传播。

**验证依据**：`references/glm-1.0.3/glm/glm/detail/setup.hpp:563-571`。

**状态**：上一轮已识别，设计文档修订说明未覆盖此数值修正。

## 问题 4（严重）— 跨类型 Vec 构造函数中源 Vec 与标量参数的类型一致性设计存在隐患

**所在位置**：§4.1 Vec3/Vec4 跨类型构造函数签名

**问题描述**：当前 Vec4 跨类型构造函数签名中的标量参数使用了 `T2`（源分量类型）而非 `T`（目标分量类型）。例如：

```
public init<T2, Q2>(a: Vec3<T2, Q2>, b: T2) where Q2 <: Qualifier
```

此处 `b` 的类型为 `T2`（源 Vec3 的分量类型），而非目标 Vec4 的分量类型 `T`。这导致：当调用 `Vec4<Float32, PackedHighp>(Vec3<Int32, PackedLowp>(1,2,3), 4)` 时，标量 `4` 必须为 `Int32` 以匹配 `T2`。

C++ GLM 的行为是标量隐式转换为目标类型——`vec4(ivec3(1,2,3), 4.0f)` 中的 `4.0f` 是 `float`。当前设计要求调用方在跨类型场景下对标量类型做出额外选择（`T` 还是 `T2`），增加了使用复杂度且与 GLM 行为不一致。

所有涉及标量参数的跨类型构造函数（Vec3 中 `a: Vec2<T2, Q2>, b: T2`、Vec4 中 `a: Vec2<T2, Q2>, b: T2, c: T2`、`a: Vec1<T2, Q2>, b: T2, c: T2, d: T2` 等）均存在此问题。

**改进建议**：将所有跨类型构造函数中的标量参数类型改为 `T`（目标分量类型），而非 `T2`。构造函数体内通过 `T(scalar)` 转换仍可接受。此变更可保证：
1. 标量参数使用的类型与 Vec 构造目标一致，直觉上更清晰
2. 与 C++ GLM 的隐式转换在语义上更接近（调用方在目标类型的域内构造）
3. 避免 `Vec3<Float32>(vec2_of_Int32, 3.0f32)` 中需要 `Int32(3)` 的冗余转换

## 问题 6（轻微）— `isIec559Of<T>()` const `is` 方案仍为设计假设，未经验证

**所在位置**：§3.5（第 206-210 行）、§3.6（第 224-229 行）、§7 D19

**问题描述**：主设计路径使用 `const if (T(0) is Float64 || T(0) is Float32)` 实现编译期类型判定，依赖两个假设：① `T(0)` 在 const 上下文中对所有目标类型合法；② `is` 运算符在 const 函数中可参与编译期求值。虽然 `is` 在 const 表达式列表中（仓颉语言规范确认），`const init` 可赋值 `var` 成员已确认，但 `T(0)` 在 const 上下文中对泛型 `T` 的统一合法性尚未在目标编译器上验证。

**改进建议**：将此问题归类修正为「设计假设验证风险」而非「事实正确性」（上一轮诊断的分类不精确）。建议在编码前优先编写 `isIec559Of<T>()` 的原型测试代码，针对 `T = Int32, Float32, Bool` 三种类型验证 const 上下文的 `T(0) is Float64` 表达式是否可通过编译。
