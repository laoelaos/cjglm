# 质量审查报告 — 首轮 OOD 设计方案 v8

## 审查范围

本审查聚焦内部审议（组件A）未充分覆盖的维度：需求响应充分度、产出完整性、可编码就绪度、接口定义对下游消费者的充分性。技术可行性等维度已由前序多轮审议确认，本审查不重复验证。

---

## 问题清单

### P1. 别名目标数量三处不一致，编码人员无法确定首轮实际应实现的别名数量

**问题描述**：设计文档在三个不同位置给出了首轮别名数量的不同目标值，三者互相矛盾：

| 位置 | 声称目标 | 原文 |
|------|---------|------|
| §3.8 第2段 | **256** 个 | "首轮 256 个别名全量定义：与 roadmap 中'首轮 256 个别名'的要求一致，在 fwd.cj 中一次性定义全部别名" |
| §3.8 第3段 | **48** 个 | "最简可行范围（48 个别名）：若首轮孵化阶段需要缩小范围，优先实现...16 家族 × 3 分量数 × 1 默认精度 = 48 个" |
| §12.2 验收标准（第 900 行） | **≥64** 个 | "fwd.cj 中至少必备级 64 个别名可正确定义和使用" |

三个值（256、48、≥64）在范围上无法调和——48 个的"最简范围"比 "≥64 个" 的验收下限还少 16 个；256 个的"全量"与 "≥64 个" 的下限相差 4 倍。**编码团队无法根据设计确定首轮实际交付目标**。

**所在位置**：§3.8（第 266-267 行）、§12.2 验收标准表（第 900 行）

**严重程度**：严重

**改进建议**：
1. 明确首轮别名数量的**唯一目标值**，将其从"可选范围"升级为"强制交付标准"
2. 将§3.8的"最简可行范围"段落改为**可选降级策略**，而非与主策略平级的替代方案；同时标注该降级需触发范围变更评审
3. 将§12.2 验收标准中的验证数量与§3.8 的目标值对齐，消除自相矛盾

### P2. 设计未利用 `public import` 定义公共 API 面，下游消费者导入路径不明

**问题描述**：设计将所有 Vec 核心类型定义在 `package glm.detail` 中，`package glm` 仅通过 `fwd.cj` 提供预定义的 256 个别名。设计未使用仓颉的 `public import` 机制（package doc §4.6）从 `glm` 包中重导出 `glm.detail` 的核心类型。

由此导致下游消费者的三种导入场景均存在缺陷：

| 使用场景 | 设计隐含的导入路径 | 问题 |
|---------|-----------------|------|
| 使用预定义别名（如 `Vec2Float32`） | `import glm.fwd` | 可接受 |
| 使用 Vec 类型混合自定义类型参数（如 `Vec2<Int16, PackedLowp>`） | 需 `import glm.detail.type_vec2` | **违背封装原则**——`detail` 包名暗示内部实现，下游不应直接依赖 |
| 使用 `Qualifier` 实现类型 | 需 `import glm.detail.qualifier` | 同上 |

设计未定义统一的公共 API 入口文件（如 `glm/mod.cj`），未说明下游消费者应通过何种路径导入 Vec 类型和 Qualifier 类型。这与 roadmap §3F 中 "glm 包公开所有公共类型别名" 的意图一致，但 roadmap 也未提及 `public import` 重导出的策略。

**所在位置**：§2（模块划分，第 34-51 行）、§3.8（第 244-271 行）

**严重程度**：严重

**改进建议**：
1. 在 `package glm` 中新增一个公共 API 入口文件（如 `lib.cj` 或 `mod.cj`），使用 `public import` 重导出 `glm.detail` 中的 Vec 类型和 Qualifier 类型，使下游通过 `import glm.{ Vec2, Vec3, ... }` 即可使用
2. 在 §2 模块划分中明确说明公共 API 面设计，包括 `public import` 的导入导出层次
3. 在 §12 验收标准中增加"公共 API 导入可用性"验收项

### P3. 缺少与 roadmap 首轮范围的可追溯性映射，编码阶段无法独立验证范围完整性

**问题描述**：需求要求首轮范围基于 `docs/01_roadmap.md` 定义。roadmap §3E 明确定义了首轮迁移的 11 个 GLM 源文件 + 3 个 shim 文件清单及其子范围划分。设计文档的 §8 迁移文件清单虽然与 roadmap 的文件列表大致对应，但**未提供任何显式的可追溯性机制**：

- 设计未引用 roadmap 的子范围编号（S、1、2a、2b、4-8），也未在设计自己的文件清单中添加对应关系
- 设计未逐项确认 roadmap 中的 11 个核心文件是否全部覆盖，也未标注是否有新增/移除/合并的文件
- 设计对 `compute_vector_decl.hpp` 和 `_vectorize.hpp` 的处理方式（§3.3、§3.4 中"首轮仅定义不消费"）与 roadmap 的推荐范围一致，但**未声明这一简化偏离了 roadmap 的哪些假设**

特别地，roadmap §447 中明确了"最简可行范围"的降级路径（降级 vec1、缩减别名等），而设计文档的 §3.8 直接引用了 48 个别名的降级方案，却**未说明这个降级方案对应的范围边界和前提条件**（即需要同步降级 vec1、以 stub 占位 compute_vector_decl 和 _vectorize）。

**所在位置**：§1（概述，第 9 行）、§8（迁移文件清单与依赖拓扑，第 689-715 行）

**严重程度**：严重

**改进建议**：
1. 在 §8 迁移文件清单中为每个文件添加 roadmap 子范围编号和对应 roadmap 源文件路径
2. 在 §1 或独立章节增加"范围可追溯性对照表"，逐项对照 roadmap §3E/3G 的核心文件清单、shim 文件清单、别名数量
3. 对与 roadmap 推荐范围不一致的决策（如 Functor/ComputeVec "仅定义不消费"），明确标注偏离项、理由及条件

### P4. D6 配置常量表引用未定义的子常量，编码人员无法从设计文档独立获取完整常量值

**问题描述**：§7 D6 的 18 个 GLM_CONFIG_* 常量表中，"首轮推荐值"列引用了多个子常量（或宏）的内联注释值，但这些子常量在设计中**未定义**：

| 引用的子常量 | 出现位置（行号） | 含义 |
|------------|----------------|------|
| `GLM_SWIZZLE_DISABLED（=2）` | 第 546 行 | swizzle 禁用值，注释为 `2` |
| `GLM_DISABLE` | 第 547-563 行多处 | 禁用/关闭的布尔或整数值 |
| `GLM_LENGTH_INT（=1）` | 第 558 行 | length_t 类型选择值 |
| `GLM_HIGHP（=3）` | 第 559-563 行 | 默认精度等级值 |
| `RH_BIT(8)` / `NO_BIT(2)` | 第 552 行 | 裁剪空间位掩码组成部分 |

虽然注释中给出了这些子常量的数值（如 `=2`），但设计未说明这些数值应在 `setup.cj` 中以何种形式定义——是作为独立的 `const` 命名常量，还是直接使用字面量。编码人员需要自行决策：

- 若使用字面量：`const GLM_CONFIG_SWIZZLE: Int = 2` vs `const GLM_CONFIG_SWIZZLE: Int = GLM_SWIZZLE_DISABLED`
- 若使用命名常量：`GLM_SWIZZLE_DISABLED`、`GLM_DISABLE`、`GLM_LENGTH_INT`、`GLM_HIGHP` 等是否需要在 `setup.cj` 中作为独立 `const` 声明

同样，`RH_BIT(8)` 和 `NO_BIT(2)` 的注释形式暗示它们是函数（`RH_BIT(n)`）或位掩码组合。如果编码人员试图调用 `RH_BIT(8)`，需要先定义此函数。

**所在位置**：§7 D6（第 544-565 行）

**严重程度**：一般

**改进建议**：
1. 在 §7 D6 中明确声明：子常量的编码策略——统一采用**字面量替换**（推荐，减少冗余定义）或**命名常量映射**（完整对齐 C++ 宏结构）
2. 若采用字面量方案，将注释中的 `（=2）` 等去除，直接使用数值作为推荐值
3. 若采用命名常量方案，在 D6 表格后补充一个"附录：子常量定义"清单

### P5. 设计对 `Qualifier` 的运行时行为约定未涉及——不同 Qualifier 在当前设计下功能等价，但未明确约定

**问题描述**：设计将 Qualifier 定义为无成员的标记接口，各精度/对齐策略（`PackedHighp`、`PackedMediump`、`PackedLowp`、`AlignedHighp` 等）为空结构体实现。在当前首轮配置下（`GLM_CONFIG_ALIGNED_GENTYPES = false`、`GLM_CONFIG_SIMD = false`），所有 Qualifier 类型在运行时的行为完全等价——Vec 类型的算术运算、比较运算、构造等对 Qualifier 值的差异不做任何分发。

这是一个**设计中的已有意识决定**，但设计未在任何地方明确约定：
- 不同 Q 参数的 Vec 类型之间是否可以赋值（如 `var v: Vec2<Float32, PackedHighp> = Vec2<Float32, PackedLowp>(...)`）
- 未来轮次中 Qualifier 差异将如何引入（通过 SIMD 路径或对齐要求）
- 首轮中两个使用不同 Q 参数的 Vec 实例是否应被视为"等价类型"（例如在泛型代码中 `a = b` 是否合法，当 Q 不同时）

**所在位置**：§3.1（Qualifier 体系，第 82-98 行）、§3.2（Vec 结构体系，第 117-118 行 Q 参数说明）

**严重程度**：一般

**改进建议**：
1. 在 §7 中新增一个设计决策（或补充到 D2），明确约定首轮中不同 Qualifier 类型之间是否可相互赋值/转换
2. 补充一条关于未来轮次 Qualifier 差异引入路径的预期说明，让编码团队理解当前"功能等价"是有意为之而非疏忽
3. 在验收标准中增加涉及跨 Qualifier 赋值/转换行为的测试项

---

## 总结

设计经过 8 轮审议迭代后，在技术可行性（泛型约束、运算符重载、溢出策略、const init 机制等）方面已较为成熟。本审查发现的主要问题集中在范围完整性（P1/P3）、公共 API 面设计（P2）、编码就绪性（P4）、以及行为约定完备性（P5）四个方面。其中 P1-P3 属于严重级别，需要在编码启动前修复。
