## 当前审查结果

### P1（严重）— `operator[]` 赋值形式缺少 `mut` 修饰符
- **位置**：§4.2（分量访问），第 424 行
- **问题**：`operator[]` 赋值形式签名缺少 `mut` 修饰符，struct 的普通实例成员函数不能修改实例成员。若不添加 `mut`，`v[i] = val` 将无法编译通过或赋值不生效。
- **建议**：在 `operator[]` 赋值形式的签名中添加 `mut` 修饰符；同时检查 extend 块中赋值形式的 `[]` 是否需要 `mut`。

### P2（中高）— `const` 函数中复合表达式兼容性风险系统性识别不足
- **位置**：§3.5（ComputeEqual）、§3.6（shim_limits.cj）、§4.5（比较运算）
- **问题**：`const operator ==` 函数体内 `abs(a - b) <= epsilonOf<T>()` 表达式链中，`abs()` 和 `epsilonOf<T>()` 的 const 兼容性未评估。`const if (isIec559Of<T>())` 中 `T(0) is Float64` 方案已在第 10 章标注为需验证。
- **建议**：在 §3.5 或 §7 中新增设计决策，系统性罗列 `const operator ==` 完整表达式链的 const 兼容性要求，标注每项的验证状态和回退方案。

### P3（中高）— `@OverflowWrapping` 标注继承性未经验证
- **位置**：§4.6（`@OverflowWrapping` 标注策略），第 485 行
- **问题**：设计断言编译器自动生成的复合赋值版本继承二元运算符上的 `@OverflowWrapping` 标注，此行为未在文档中确认，属于对编译器行为的推测。
- **建议**：编写原型测试代码验证标注继承性，或在 §4.6 中补充备选方案（如所有 Vec 运算通过 extend 块中的具名函数实现）。

### P4（中等）— `Vec<Bool, Q>` 抽象类型引用未彻底消除
- **位置**：§3.2（Vec 结构体系），第 161-167 行
- **问题**：实际设计中不存在 `Vec<Bool, Q>` 类型，只有 `Vec1<Bool, Q>`、`Vec2<Bool, Q>`、`Vec3<Bool, Q>`、`Vec4<Bool, Q>` 四个独立结构体。§3.2 仍使用 `Vec<Bool, Q>` 格式，跨多轮迭代未彻底修复。
- **建议**：将 §3.2 第 161-167 行的所有 `Vec<Bool, Q>` 替换为 `VecN<Bool, Q>`，同步检查全文是否还有遗漏。

### P5（中等）— 跨类型 fill-from-Vec1 构造器缺失限制未记录为设计决策
- **位置**：§4.1（向量构造），Vec2~Vec4 的从 Vec1 构造部分；§7（设计决策）
- **问题**：跨类型 `Vec1<T2, Q2>` 的 fill-from-Vec1 构造器不可与 `Vec1<T, Q>` 版本共存（仓颉函数重载规则限制），但此约束未在 §7 中以设计决策形式记录。
- **建议**：在 §7 中新增设计决策，记录此约束及理由，在 §4.1 签名旁添加注释引用该决策。

### P6（轻微）— `increment()`/`decrement()` 语义差异未充分评估
- **位置**：§4.3（算术运算，第 443 行）、§11.2（迁移成本评估）
- **问题**：仓颉原生 `++` 返回 `Unit`，与 C++ `operator++` 的前缀返回引用、后缀返回旧值副本有本质差异。§11.2 对依赖 `++`/`--` 返回值的迁移模式覆盖不完整。
- **建议**：在 §11.2 中补充典型 C++ `++`/`--` 使用模式的分类与映射表，明确各类别的迁移工作量和正确映射方式。

### P7（轻微）— 配置常量表中 `Int` 类型在仓颉中非标准类型
- **位置**：§7 D6，第 604 行
- **问题**：仓颉无未修饰的 `Int` 类型，正确类型应为 `Int64`。影响 `setup.cj` 的编译通过性。
- **建议**：将 D6 配置常量表中的所有 `Int` 类型声明改为 `Int64`，同步修正 §8 中的对应说明。
