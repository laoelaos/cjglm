## 发现的问题

### 问题 1：Vec1~Vec4 构造函数清单中缺少跨分量数截断构造（关键遗漏）

- **位置**：§4.1（向量构造 — Vec1/Vec2/Vec3 构造函数清单）
- **严重程度**：严重
- **问题描述**：
  当前 §4.1 的构造函数清单中，Vec1~Vec4 的跨类型构造仅覆盖了"同分量数跨类型转换"（如 `Vec2<T2,Q2>` 构造 `Vec2<T,Q>`），未覆盖跨分量数截断构造。对照 C++ GLM 1.0.3 源码有如下缺失：

  **Vec1 缺失**（C++ `type_vec1.hpp:107-114`，`GLM_EXPLICIT`）：
  - 从 `Vec2<T2,Q2>` 截断构造（取第一个分量 `x`）
  - 从 `Vec3<T2,Q2>` 截断构造（取第一个分量 `x`）
  - 从 `Vec4<T2,Q2>` 截断构造（取第一个分量 `x`）

  **Vec2 缺失**（C++ `type_vec2.hpp:128-137`，`GLM_EXPLICIT`）：
  - 从 `Vec3<T2,Q2>` 截断构造（取 `x,y`）
  - 从 `Vec4<T2,Q2>` 截断构造（取 `x,y`）

  **Vec3 缺失**（C++ `type_vec3.hpp:142-156`）：
  - 从 `Vec4<T2,Q2>` 截断构造（取 `x,y,z`，`GLM_EXPLICIT`）

  共 **6 个构造函数** 在首轮 `type_vecN.hpp` 文件中存在对应声明但当前设计未列出。编码人员按当前清单实现将产生与 C++ GLM 不一致的 API 面——下游迁移代码中存在的 `Vec2(vec3)`、`Vec3(vec4)` 等构造将无法编译。

  §4.1 声明本文档清单覆盖"GLSL 5.4.1 规范定义的构造语义"，而 GLSL 5.4.1 确实定义了上述截断构造语义（从高维向量取前 N 个分量构造低维向量）。文档声明与实际清单不完全一致。

- **改进建议**：在 §4.1 中为 Vec1 补充从 Vec2/Vec3/Vec4 的截断构造函数（共 3 个），为 Vec2 补充从 Vec3/Vec4 的截断构造函数（共 2 个），为 Vec3 补充从 Vec4 的截断构造函数（1 个）。所有跨分量数构造均标注 `public` 并使用 `where Q2 <: Qualifier` 约束。同时在 §4.1 的说明中补充一节集中描述截断构造的语义规则（取前 N 个分量，使用 `T(v.x)` 显式转换）。

### 问题 2：Vec3 构造函数清单中缺少 Vec2+Vec1 和 Vec1+Vec2 组合构造（关键遗漏）

- **位置**：§4.1（向量构造 — Vec3 构造函数清单）
- **严重程度**：严重
- **问题描述**：
  当前 Vec3 构造函数清单包含 `Vec2+T`（第 413 行）和 `T+Vec2`（第 414 行），但不包含 `Vec2+Vec1` 和 `Vec1+Vec2` 组合。对照 C++ GLM 1.0.3 源码 `type_vec3.hpp:142-153`，以下两个构造函数明确存在且标注为"From section 5.4.1 Conversion and scalar constructors of GLSL 1.30.08 specification"：

  - `vec(vec<2, A, P> const& _xy, vec<1, B, P> const& _z)` — Vec2(2) + Vec1(1) = Vec3(3)
  - `vec(vec<1, A, P> const& _x, vec<2, B, P> const& _yz)` — Vec1(1) + Vec2(2) = Vec3(3)

  相比之下，Vec4 的构造函数清单已包含对应的 `Vec3+Vec1`（第 439 行）和 `Vec1+Vec3`（第 441 行），Vec3 上的同名组合缺失构成不等式——编码人员按当前清单实现将发现 Vec4 的 GLSL 组合构造完整而 Vec3 不完整，与 C++ GLM 行为不一致。

- **改进建议**：
  1. 在 §4.1 Vec3 构造函数清单中补充：
     ```
     public init<T2, Q2>(a: Vec2<T2, Q2>, b: Vec1<T2, Q2>) where Q2 <: Qualifier
     public init<T2, Q2>(a: Vec1<T2, Q2>, b: Vec2<T2, Q2>) where Q2 <: Qualifier
     ```
  2. 补充后验证 Vec3 的 `Vec1+Vec2`/`Vec2+Vec1` 构造函数是否与现有 `Vec1+Vec1+Vec1` 等构造函数在重载解析中无冲突（仓颉泛型重载规则按实际参数类型区分，`Vec2` 与 `Vec1+Vec1` 在类型层面是不同的——`Vec2<T2,Q2>` 是单一类型参数，`(Vec1<T2,Q2>, Vec1<T2,Q2>)` 是两个 Vec1 类型参数，应可区分）。

### 问题 3：`mod` 具名函数的首轮可用性状态描述存在歧义（逻辑矛盾）

- **位置**：§4.3（算术运算）
- **严重程度**：一般
- **问题描述**：
  §4.3 具名函数替代列表中包含 `mod(v, s)` 和 `mod(s, v)` 双向重载（第 500 行），明确列于首轮标量-op-向量替代方案中，暗示首轮可用。然而同节第 506 行的 `%` 运算符整数约束说明中写道："对于需要浮点数取模的场景，使用具名函数（如 `mod`，不属于首轮范围）"。

  两处表述矛盾——前者将 `mod` 列入首轮函数清单，后者声称 `mod` "不属于首轮范围"。编码人员将无法判断：
  - 是否需要在首轮编码中实现 `mod` 具名函数
  - `mod` 的整数/浮点类型约束策略是否与 `%` 运算符一致（整数可编译、浮点编译报错）

- **改进建议**：
  1. 做出明确的唯一决策并修正矛盾：要么将 `mod` 从首轮清单中移除（推迟到后续轮次），要么在第 506 行修正措辞为"对于需要浮点数取模的场景，使用具名函数 `mod`（首轮中与 `%` 运算符共享相同的类型约束——整数可编译，浮点实例化时报错）"。
  2. 若选择移除 `mod`，同步更新第 495-500 行的具名函数列表和 §4.6 的第 554 行注释（涉及 `%` 运算符的标注策略说明）。

### 问题 4：§7 设计决策编号顺序混乱（结构缺陷）

- **位置**：§7（设计决策）
- **严重程度**：轻微
- **问题描述**：
  §7 的设计决策编号呈现以下非顺序排列：D1→D5→D6→D19→D20→D28→D29→D30→D31→D21→D22→D23→D24。D21~D24 出现于 D31 之后。

  此问题是多轮增量迭代的副作用——新决策追加在文件末尾而非插入到对应编号位置。虽然不影响决策的技术正确性，但：
  - 读者在查阅 §7 时无法按编号顺序阅读，需要不断跳转
  - 编码人员查找 D21（`operator[]` 索引类型）时可能在 D20 附近预期找到但实际在 §7 末尾
  - 后续轮次新增决策（D32+）时若继续追加将进一步加剧混乱

- **改进建议**：将 §7 所有决策条目重排为严格编号顺序（D1→D31），D21、D22、D23、D24 移至 D20 之后、D25 之前。修订说明章节中历史记录的编号引用保持不变（因各轮次发布时编号即如此）。同步检查全文对设计决策的交叉引用（如 §3.1 引用 D23/D24、§3.6 引用 D18 等）是否因重排后位置变化而需要更新行号引用。

## 整体评价

设计文档经 14 轮迭代审议后主体质量较高，§2（模块划分）、§3.1（Qualifier 体系）、§4.5~4.7（比较运算/Overflow 策略/哈希契约）、§7 大多数决策、§8（迁移文件清单）等章节已基本完备，可指导编码。但构造函数清单（§4.1）存在 3 类共 6 个遗漏构造函数，属于直接影响编码实现完整性和与 C++ GLM 行为一致性的关键遗漏，应在编码启动前修复。§4.3 的 `mod` 可用性歧义需要在编码前做出明确唯一决策。
