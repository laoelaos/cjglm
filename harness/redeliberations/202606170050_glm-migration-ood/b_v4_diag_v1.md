# 质量审查报告 — v4 设计文档

**审查轮次**：第 4 轮  
**审查视角**：落地编码就绪度、需求响应充分度、深度与完整性  
**审查范围**：重点覆盖内部审议（组件 A）未充分覆盖的维度

## 发现的问题

### 问题 1：`++`/`--` 自增自减运算符完全缺失（关键遗漏）

- **所在位置**：§4.3（算术运算）、§4.6（`@OverflowWrapping` 标注策略）
- **严重程度**：严重
- **问题描述**：
  C++ GLM 的 `type_vecN.hpp` 中为每个 Vec 类型定义了前缀/后缀自增（`operator++`）和自减（`operator--`）运算符（共 4 个成员函数），并在 `type_vecN.inl` 中实现。Roadmap §0.5.1 的 `@OverflowWrapping` 标注策略表中明确列举了"前置/后置自增自减"为需要标注的运算符类别。然而 v4 设计文档中完全未提及 `++`/`--` 运算符：
  - §4.3 仅覆盖 `+`、`-`、`*`、`/`、`%`，无 `++`/`--`。
  - §4.6 的标注策略仅覆盖二元算术运算符（16~20 处），未纳入 `++`/`--`。
  - §3.2 的可重载运算符列表仅列出了不可重载的 `&&`/`||`/`~`，未说明 `++`/`--` 的处理方式。
  - 仓颉可重载运算符列表中不包含 `++`/`--`，因此 C++ `v++`/`++v` 语法在仓颉中不可直接映射。

  该遗漏导致编码阶段无法获知：① `++`/`--` 是直接删除还是用具名函数替代；② 若用具名函数替代，命名和签名是什么；③ 对下游 C++ GLM 代码迁移的影响范围未评估（类似 §11 对 `&&`/`||` 的迁移成本评估）。

- **改进建议**：
  1. 在 §3.2 或 §4.3 中明确 `++`/`--` 不可作为仓颉运算符重载实现。
  2. 决定替代方案：提供具名函数（如 `increment()`/`decrement()`，返回 `Unit` 或 `VecN<T,Q>`）或直接移除。
  3. 更新 §4.6 标注策略，若保留则说明 `@OverflowWrapping` 标注方式。
  4. 参照 §11 增加一节评估自增自减移除/替换对迁移工作量的影响。

### 问题 2：`NumericLimits<T>` 泛型约束与 D5 宽松约束冲突（潜在编译错误）

- **所在位置**：§3.5（ComputeEqual）、§3.6（shim_limits.cj）、§7 D5
- **严重程度**：严重
- **问题描述**：
  §3.5 中 `ComputeEqual<T>.callConst()` 使用 `NumericLimits<T>.isIec559` 在编译期选择比较路径。§3.6 描述 `NumericLimits<T>` 为"各数值类型静态属性的泛型封装：`where T <: Number<T>` 约束"。而 §7 D5 明确 Vec 的 `T` 参数**不做紧约束**（仅约束 `Q <: Qualifier`），允许 `Bool` 作为 `T`。

  若 `NumericLimits<Bool>` 不满足 `where T <: Number<T>` 约束，则 `ComputeEqual<Bool>.callConst()` 无法编译。而 `ComputeEqual<Bool>.callConst()` 被 Vec 的 `==` 运算符调用，因此 `VecN<Bool, Q>` 的 `==` 运算符将编译失败。

  当前设计对 `Bool` 类型的处理存在缺口：§4.5 定义了 `logicalAnd`/`logicalOr` 用于 `VecN<Bool, Q>` 的布尔运算，但 `==` 运算符同样需要工作在 `VecN<Bool, Q>` 上（C++ GLM 的 `type_vecN.hpp` 中 `operator==`/`operator!=` 对所有 `T` 包括 `Bool` 可用）。

- **改进建议**：
  方案 A（推荐）：重新设计 `NumericLimits<T>` 使其对任意 `T` 可用——对于非 `Number<T>` 类型，所有字段取默认值（如 `isIec559 = false`），无需 `where T <: Number<T>` 约束。
  方案 B：`ComputeEqual` 分两个版本——`ComputeEqual<T> where T <: Number<T>` 使用 `const if` 分支，`ComputeEqual<T>`（无约束）使用精确比较。在 Vec 的 `==` 中通过约束重载选择合适版本。
  方案 C：为 `Vec<Bool, Q>` 单独特化 `==` 运算符（通过 `extend VecN<Bool, Q>` 块）。

### 问题 3：`bitwiseNot()` 对整数类型的实现路径不明确（规格不完整）

- **所在位置**：§4.4（位运算）
- **严重程度**：高
- **问题描述**：
  §4.4 描述 `bitwiseNot()` 的实现为"内部逐分量对每个分量调用 `!`（逻辑非）或按位取反逻辑"。此描述存在歧义且不完整：
  - `!` 运算符在仓颉中仅对 `Bool` 类型有效（逻辑非），对整数类型不可用。
  - `~`（按位取反）不在仓颉可重载运算符列表中，因此不能直接在分量上调用。
  - 对于整数类型的 Vec 实例（如 `Vec2<Int32>`），`bitwiseNot()` 的具体实现方式未指定。
  - §4.4 还标注了"二元位运算符通过 `extend` 块定义为扩展成员函数"，但 `bitwiseNot()` 是一元操作，应通过 `extend` 块还是内部成员函数定义也未明确。

  编码阶段的开发者无法根据当前描述决定如何实现整数分量的按位取反。

- **改进建议**：
  明确指定整数分量的 `bitwiseNot()` 实现策略，例如：
  - 使用 `T(-1) ^ x`（XOR 全 1）在包裹语义下等效于按位取反。
  - 或定义一个内部辅助函数 `bitwiseNotComponent(x: T): T`，对整数使用 `T(!0) - x` 或 `!x`（若 `!` 被扩展支持）。
  - 或说明 `bitwiseNot` 通过 `extend VecN<Int8|Int16|Int32|Int64|..., Q>` 分别提供特化实现。
  同时在 §3.2 中补充 `bitwiseNot()` 的定义位置说明（extend 块 vs 内部成员）。

### 问题 4：构造函数签名清单缺少类型参数声明（编码阻塞）

- **所在位置**：§4.1（向量构造）
- **严重程度**：高
- **问题描述**：
  §4.1 给出了"**完整**构造函数签名清单（Vec2 示例）"，其中跨 Vec 转换构造函数和跨分量数构造函数的签名**未声明类型参数**：
  ```
  public init(v: Vec2<T2, Q2>)                               // 跨转换
  public init(a: Vec1<T>, b: T)                              // 跨分量数组合
  public init(v: Vec1<T>)                                    // 从 Vec1 显式构造
  ```
  正确的仓颉语法需要类型参数声明：
  ```
  public init<T2, Q2>(v: Vec2<T2, Q2>) where Q2 <: Qualifier
  public init<T1, Q1>(a: Vec1<T1, Q1>, b: T) where Q1 <: Qualifier
  ```
  此外跨分量数构造中 `Vec1<T>` 缺少 `Q` 参数（应为 `Vec1<T, Q>`），以及 `public init(a: Vec1<T>, b: T)` 中 `T` 的类型参数应在构造函数上声明（`<T1>`）而非直接引用结构体的 `T`（因为 Vec1 的 T 可能不同）。

  该清单作为"完整签名清单"声称可直接指导编码，但缺少类型参数声明意味着开发者无法直接从文档生成代码。

- **改进建议**：
  修正 §4.1 的签名清单，为每个跨类型构造函数补充完整的类型参数声明和约束。至少以 Vec2 为例给出正确语法的完整签名。同时将"完整"的描述改为"示例性"，避免误导。

### 问题 5：事实错误——"仓颉位运算无溢出异常"不适用于 `<<` 运算符

- **所在位置**：§4.4（位运算）
- **严重程度**：高
- **问题描述**：
  §4.4 声称"**不触发溢出检查（仓颉位运算无溢出异常）**"。此陈述与仓颉语言规范不符。仓颉整数溢出注解文档的溢出策略表中明确标注 `<<`（左移）为"可溢出"运算符——`<<` 默认使用 `@OverflowThrowing` 策略，在整数移位超出位宽时抛出 `ArithmeticException`。实际上，仓颉中只有 `>>`、`!`、`&`、`|`、`^` 不触发溢出检查。

  这意味着 `extend` 块中定义的 `<<` 运算符在默认行为下会在移位溢出时抛异常，而 C++ GLM 中 `vec << scalar` 是定义良好的（高位丢弃）。此行为差异未被设计文档识别或处理。

- **改进建议**：
  1. 修正 §4.4 的错误描述，明确 `<<` 在仓颉中需标注 `@OverflowWrapping` 以匹配 C++ 的 wrapping 语义，或在小整数溢出不构成问题的场景下记录该差异。
  2. 更新 §4.6 标注策略，将 `<<` 纳入 `@OverflowWrapping` 标注范围（4 个 Vec 类型 × 3 种重载形式 = 12 处），或至少在设计决策中记录不做标注的理由。
  3. 在 §5.1 标准错误场景中补充位运算溢出场景的处理策略。

### 问题 6：`scalar op vec` 模式的具名函数替代不完整（遗漏 `mod`）

- **所在位置**：§4.3（算术运算）
- **严重程度**：一般
- **问题描述**：
  §4.3 中列举了标量在左运算的替代方案，提及 `add`、`sub`、`div`、`mul` 提供双向重载，但缺少 `mod`（对应 `%` 运算符）。C++ GLM 中 `scalar % vec` 是有效的运算符重载（如 `type_vec1.hpp` 第 237 行的 `operator%(T scalar, vec<1, T, Q> const& v)`），迁移后同样需要具名函数替代。

  虽然描述中使用了"等"字（"同样，div、sub、mul 等也需提供双向重载"），但未明确列出 `mod` 可能被编码者遗漏。

- **改进建议**：
  在 §4.3 的具名函数替代列表中明确补上 `mod(vec, scalar)` 和 `mod(scalar, vec)` 双向重载，或注明"所有支持标量在左/右两种操作数顺序的运算符均需提供双向具名函数（`add`、`sub`、`mul`、`div`、`mod`、`bitwiseAnd`、`bitwiseOr`、`bitwiseXor`、`shiftLeft`、`shiftRight` 等）"。

### 问题 7：跨类型赋值语义未定义（完整性）

- **所在位置**：§4（关键行为契约）
- **严重程度**：一般
- **问题描述**：
  C++ GLM 的 Vec 类型定义了模板化的 `operator=`，允许从同分量数但不同分量类型的 Vec 赋值（如 `vec<1, T, Q>::operator=(vec<1, U, Q> const& v)`）。设计文档 §4.1 定义了跨 Vec **构造**转换，但未提及跨类型**赋值**。对于 struct 类型，仓颉自动生成的 `=` 仅接受完全相同的类型（`Vec2<Int32, Q>` 不能直接赋值为 `Vec2<Int8, Q>`）。

  若迁移代码中存在 `v = otherVec` 模式且 `v` 与 `otherVec` 分量类型不同（在 C++ GLM 中合法），在仓颉中将编译失败。当前设计未给出处理策略。

- **改进建议**：
  在 §4 或 §7 中增加一条设计决策（D17），明确三种处理路径之一：
  ① 不为 Vec 定义跨类型 `=`，要求显式转换后赋值（`v = Vec2<Int32, Q>(otherVec8)`）。
  ② 为 Vec 定义 `operator =<U>(rhs: VecN<U, Q>): VecN<T, Q>`（检查仓颉是否允许在 struct 上定义泛型 `=`）。
  ③ 记录此限制为已知差异，纳入 §11 迁移成本评估。

## 整体评价

v4 设计文档相比 v3 在内部审议覆盖的问题上已有显著改进。上述 7 个问题中，问题 1~5 属于此前审议轮次未覆盖的新问题（其中 `++`/`--` 缺失和 `<<` 溢出事实错误为关键遗漏/错误，NumericLimits 约束冲突和 bitwiseNot 规格不完整为中等严重度），问题 6~7 属补充性完善建议。文档在模块结构、依赖拓扑、设计决策记录方面较为完整，但落地编码就绪度方面仍存在上述障碍，建议修复后再进入编码阶段。
