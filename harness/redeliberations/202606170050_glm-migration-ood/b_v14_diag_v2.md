## 发现的问题

### 问题 1（严重，上轮问题未修复）：跨分量数截断构造函数仍然缺失

- **位置**：§4.1（向量构造 — Vec1/Vec2/Vec3 构造函数清单）
- **严重程度**：严重
- **问题描述**：
  上轮审查（b_v14_diag_v1.md 问题 1）已指出 §4.1 缺少 C++ GLM 中存在的跨分量数截断构造函数，但本轮设计文档**仍未补充**。当前 §4.1 清单中：

  **Vec1 缺失从 Vec2/Vec3/Vec4 的截断构造**（C++ `type_vec1.hpp:107-114`，`GLM_EXPLICIT`）：
  - `Vec1(Vec2<T2,Q2>)` — 取第一个分量 x
  - `Vec1(Vec3<T2,Q2>)` — 取第一个分量 x
  - `Vec1(Vec4<T2,Q2>)` — 取第一个分量 x

  **Vec2 缺失从 Vec3/Vec4 的截断构造**（C++ `type_vec2.hpp:128-137`，`GLM_EXPLICIT`）：
  - `Vec2(Vec3<T2,Q2>)` — 取 x,y
  - `Vec2(Vec4<T2,Q2>)` — 取 x,y

  **Vec3 缺失从 Vec4 的截断构造**（C++ `type_vec3.hpp:142-156`，`GLM_EXPLICIT`）：
  - `Vec3(Vec4<T2,Q2>)` — 取 x,y,z

  共 6 个构造函数缺失。§4.1 声明本文档清单覆盖"GLSL 5.4.1 规范定义的构造语义"，GLSL 5.4.1 确实定义了上述截断构造语义（从高维向量取前 N 个分量构造低维向量），因此文档声明与实际清单不完全一致。

- **改进建议**：在 §4.1 中为 Vec1 补充从 Vec2/Vec3/Vec4 的截断构造函数（共 3 个），为 Vec2 补充从 Vec3/Vec4 的截断构造函数（共 2 个），为 Vec3 补充从 Vec4 的截断构造函数（1 个）。所有跨分量数构造均标注 `public` 并使用 `where Q2 <: Qualifier` 约束。

### 问题 2（严重，上轮问题未修复）：Vec3 缺少 Vec2+Vec1 和 Vec1+Vec2 组合构造

- **位置**：§4.1（向量构造 — Vec3 构造函数清单）
- **严重程度**：严重
- **问题描述**：
  上轮审查（b_v14_diag_v1.md 问题 2）已指出此问题，但本轮设计文档**仍未补充**。当前 Vec3 构造函数包含 `Vec2+T` 和 `T+Vec2`，但不包含 `Vec2+Vec1` 和 `Vec1+Vec2`。对比 C++ GLM `type_vec3.hpp:142-153` 中明确存在（标注为 GLSL 5.4.1 ）：

  - `vec(vec<2, A, P> const& _xy, vec<1, B, P> const& _z)` → `init(a: Vec2<T2,Q2>, b: Vec1<T2,Q2>)`
  - `vec(vec<1, A, P> const& _x, vec<2, B, P> const& _yz)` → `init(a: Vec1<T2,Q2>, b: Vec2<T2,Q2>)`

  相比之下，Vec4 的构造函数清单已包含对应的 `Vec3+Vec1` 和 `Vec1+Vec3`，Vec3 的缺失构成 API 行为不一致。

- **改进建议**：
  1. 在 §4.1 Vec3 构造函数清单中补充：
     ```
     public init<T2, Q2>(a: Vec2<T2, Q2>, b: Vec1<T2, Q2>) where Q2 <: Qualifier
     public init<T2, Q2>(a: Vec1<T2, Q2>, b: Vec2<T2, Q2>) where Q2 <: Qualifier
     ```
  2. 补充后验证仓颉泛型重载规则下与现有 `Vec1+Vec1+Vec1` 等构造函数的重载解析无冲突。

### 问题 3（一般，上轮问题未修复）：`mod` 具名函数首轮可用性状态描述自相矛盾

- **位置**：§4.3（算术运算）
- **严重程度**：一般
- **问题描述**：
  上轮审查（b_v14_diag_v1.md 问题 3）已指出此矛盾，但本轮设计文档**仍未修正**。

  §4.3 第 500 行将 `mod(v: VecN<T,Q>, s: T)` 和 `mod(s: T, v: VecN<T,Q>)` 列于首轮具名函数替代清单中，暗示首轮可用。同一节第 506 行却写道："对于需要浮点数取模的场景，使用具名函数（如 `mod`，不属于首轮范围）"。

  编码人员无法判断是否需在首轮中实现 `mod`。

- **改进建议**：做出明确的唯一决策并修正矛盾。若 `mod` 属于首轮，修正第 506 行措辞；若不属首轮，从第 500 行的清单中移除。同步更新 §4.6 第 554 行注释（涉及 `%` 运算符的标注策略说明）。

### 问题 4（一般，新发现）：`scalar-op-vec` 方向辅助函数未列入 `lib.cj` 的公共 API 导出清单

- **位置**：§2（公共 API 面设计）+ §4.3（具名函数的定义位置）
- **严重程度**：一般
- **问题描述**：
  §4.3 将 `scalar-op-vec` 方向辅助函数（`add(s, v)`、`sub(s, v)` 等）定义为包级独立函数，集中置于 `compute_vector_decl.cj` 中，并注明"调用方通过 `import glm.detail.compute_vector_decl.*` 或通过 `glm` 包统一引入"。但 §2 中 `lib.cj` 的 `public import` 列表仅覆盖 Vec 和 Qualifier 类型，**未列任何 `add`/`sub`/`mul`/`div`/`mod` 函数的重导出**。

  这意味着下游消费者无法通过单一的 `import glm.*` 获得这些辅助函数，必须直接依赖 `glm.detail.compute_vector_decl`——这与 §2 宣称的"下游消费者只需 `import glm.*` 即可使用所有公共 API，无需直接依赖 `glm.detail` 内部包"的封装原则相矛盾。

  具体影响的函数共 20 个（5 运算 × 4 Vec 类型，scalar-op-vec 方向）；分别使能 Vec1~Vec4 双向具名函数的消费者体验。

- **改进建议**：在 §2 `lib.cj` 的 `public import` 列表中补充 `scalar-op-vec` 方向辅助函数的导出行。例如：
  ```
  public import glm.detail.compute_vector_decl.{ add, sub, mul, div, mod }
  ```
  同时在 §8 迁移文件清单的 `lib.cj` 行中更新依赖说明和偏离项记录。

### 问题 5（轻微，上轮问题未修复）：§7 设计决策编号顺序混乱

- **位置**：§7（设计决策）
- **严重程度**：轻微
- **问题描述**：
  上轮审查（b_v14_diag_v1.md 问题 4）已指出此问题，但本轮**仍未修正**。§7 的编号呈现：D1→D5→D6→D7→...→D19→D28→D29→D30→D31→D20→D21→D22→D23→D24。D28~D31 出现在 D20~D24 之前。

  多轮增量迭代中后增的 D28~D31 追加在 D19 之后，而 D20~D24 停留在原位置。

- **改进建议**：将 D20、D21、D22、D23、D24 移至 D19 之后、D25 之前，调整为严格编号顺序。全文对设计决策的交叉引用不受影响（按编号引用），但读者可顺序阅读。

## 需求响应充分度评估

对照用户需求（首轮迁移范围 OOD）和 roadmap §3E/§3G：

| 维度 | 评估 |
|------|------|
| 需求范围覆盖 | **充分**。§8.2 范围可追溯性对照表将 roadmap 14 个条目逐项映射到设计章节和仓颉文件，偏离项（平台检测宏消减、`storage`/`genTypeTrait` 省略、`lib.cj` 新增）均标注理由。 |
| Vec 类型系统 | **充分**。Vec1~Vec4 独立结构体、Qualifier 接口体系、构造函数体系（除上述问题 1~2 缺失的 8 个构造函数外）均已覆盖。 |
| 类型别名 | **充分**。256 个别名全量定义，命名约定明确，强制交付标准清晰。 |
| Shim 层（C++ 标准库替代） | **充分**。`shim_assert`/`shim_limits`/`shim_cstddef` 覆盖首轮所需全部替代设施。 |
| 运算符行为等价性 | **基本充分**，已知差异已标注（`&&`/`||` 改具名函数、`++`/`--` 改具名函数、`bitwiseNot` Bool 行为差异、`<<` @OverflowWrapping、跨类型运算不支持等）。 |
| 异常场景与边界条件 | **充分**。§9 完整覆盖零向量、除零、NaN/Infinity 传播、类型转换边界，§5 覆盖下标越界、整数溢出、除零、位运算溢出。 |
| 下游消费者视角 | **除问题 4 外充分**。`lib.cj` 封装设计良好，但 `scalar-op-vec` 辅助函数未纳入重导出，存在封装漏洞。 |

## 整体评价

设计文档经 14 轮迭代审议后主体质量高。§2、§3.1~§3.9、§4.5~§4.7、§5、§7、§8、§9、§12 等核心章节已基本完备，可直接指导编码。但存在以下需要编码前修复的关键问题：

1. **6 个跨分量数截断构造函数缺失**（问题 1）— 将导致下游 `Vec2(v3)`、`Vec3(v4)` 等迁移代码无法编译
2. **2 个 Vec3 组合构造函数缺失**（问题 2）— 与 GLSL 5.4.1 规范不一致
3. **`scalar-op-vec` 函数未导出**（问题 4）— 违反公共 API 封装设计，消费者需直接依赖 `glm.detail` 内部包

问题 1~2 和问题 5 均已在上一轮审查中明确指出但本轮未修复，建议主 Agent 修复时优先处理，并在本次迭代完成后验证关闭清单。
