## 发现的问题

### P1（严重）：`mut operator func +=` 的显式定义可能不合法

- **位置**：§4.6（`@OverflowWrapping` 标注策略）、§3.2（Vec 结构体系 — 运算符部分）
- **问题描述**：
  设计文档依赖"在 Vec 结构体内部显式定义 `mut operator func +=` 并标注 `@OverflowWrapping`"的策略来控制溢出行为。然而，仓颉语言的可重载运算符列表为：`()`、`[]`、`!`、`-`(一元)、`**`、`*`、`/`、`%`、`+`、`-`(二元)、`<<`、`>>`、`<`、`<=`、`>`、`>=`、`==`、`!=`、`&`、`^`、`|`。
  `+=` **不在上述列表中**。仓颉文档明确说明："重载二元运算符（关系运算除外）时自动启用对应复合赋值版本"——复合赋值是**自动生成**的，而非可显式重载的独立运算符。
  若 `mut operator func +=` 无法通过编译，则：
  - §4.6 的整个溢出标注策略（复合赋值标注 + 二元运算符委托）不成立
  - 二元运算符的 "copy + op=" 委托模式（§4.6 示例代码）无法编译——自动生成的 `+=` 内部调用 `+`，而 `+` 又委托回 `+=`，产生**自递归**（无限循环）
- **改进建议**：
  1. **验证路径**：查阅仓颉编译器文档或编写最小可测试代码，确认 `mut operator func +=` 的显式定义是否合法
  2. 若不可显式定义，需重构策略：
     - 方案 A：在**二元运算符**（`+`、`-`、`*`、`/`）上直接标注 `@OverflowWrapping`，实现逐分量运算，利用自动生成机制得到同样标注的 `+=`
     - 方案 B：与 roadmap §0.5.1 的"二元运算符委托给复合赋值"方向做反向对齐——二元运算符直接实现逐分量运算（不使用委托模式），复合赋值由自动生成继承
  3. 无论验证结果如何，§4.6 的"显式定义优先于自动生成版本"的论断需要事实依据支撑，不能仅凭推测

### P2（严重）：`!` 运算符不能实现整数类型的按位取反

- **位置**：§4.4（位运算 — `bitwiseNot`）
- **问题描述**：
  原文写道："内部逐分量对每个分量调用 `!`（逻辑非）或按位取反逻辑"。仓颉中 `!` 是**逻辑非**运算符，仅对 `Bool` 类型有定义。对整数类型（`Int32`、`UInt8` 等）调用 `!` 是**非法**操作。因此 `bitwiseNot()` 无法通过调用 `!` 分量来实现整数类型 Vec 的按位取反。
- **改进建议**：
  修正 `bitwiseNot()` 的实现策略：对整数类型分量，使用 `x ^ (-1)`（与全 1 异或）或等价表达式实现按位取反；对 `Bool` 类型分量使用 `!`。需在 §4.4 中明确区分两套实现路径，或确保 `bitwiseNot` 仅对整数类型 Vec 可用（通过 `where T <: Integer<T>` 约束）。

### P3（严重）：`Vec<Bool, Q>` 类型引用与设计中的类型系统不一致

- **位置**：§4.5（比较运算 — `logicalAnd`/`logicalOr`/`!` 签名）
- **问题描述**：
  设计文档明确将 Vec 类型定义为四个独立的泛型结构体 `Vec1<T,Q>`、`Vec2<T,Q>`、`Vec3<T,Q>`、`Vec4<T,Q>`（见 §3.2），不存在名为 `Vec<T,Q>` 的类型。但 §4.5 中 `logicalAnd`、`logicalOr` 和 `!` 运算符的签名使用了 `Vec<Bool, Q>` 作为参数和返回类型——这是一个**未定义的类型**。编码阶段无法根据此签名生成代码。
- **改进建议**：
  将所有 `Vec<Bool, Q>` 引用替换为每个 VecN 类型的具体版本，例如：
  - `logicalAnd(a: Vec2<Bool, Q>, b: Vec2<Bool, Q>): Vec2<Bool, Q>`
  - 或统一表述为"各 VecN 类型分别定义对应版本（Vec1~Vec4）"
  - 对于 `!` 运算符，同理需明确是针对 `VecN<Bool, Q>` 的重载

### P4（高）：运算符函数不能引入额外类型参数，限制了跨类型复合赋值

- **位置**：§3.2（复合赋值运算符描述）、§4.3（算术运算）
- **问题描述**：
  仓颉规定"运算符重载不能为泛型"（不能引入新的类型参数）。但 GLM C++ 源代码中 `operator+=` 等接受 `template<typename U> operator+=(U scalar)`，即右操作数类型 `U` 可以与 `T` 不同。设计文档未明确承认此限制。
  以 Vec2 为例，C++ GLM 可支持 `Vec2<Int32> += Vec2<Int16>`（跨类型），但仓颉的 `mut operator func +=(rhs: Vec2<T, Q>)` 只能接受与 `T` 完全相同的类型。如果要支持 `scalar` 和 `Vec1` 版本的复合赋值（如 §4.3 所述"接收标量、Vec1 或同分量数 Vec 作为右操作数"），这些重载需要额外的类型参数 `U`，与"运算符不能为泛型"规则冲突。
- **改进建议**：
  明确审查仓颉语言规则：运算符函数是否可以使用所在类型（struct）的类型参数 T 而无需引入新参数。对于 `Vec2<T, Q>`，`operator func +=(scalar: T)` 是合法的（使用现有参数 T），但 `operator func +=(scalar: U)` 不合法。需确认是否所有操作数类型都可以用 T 表达（标量和 Vec1 的 T 与 VecN 的 T 相同？T 是否包含 Bool？）。如有冲突，需调整复合赋值操作数的类型设计。

### P5（高）：`logicalAnd`/`logicalOr` 的语义与 GLM C++ 存在差异，影响迁移等价性

- **位置**：§4.5（布尔逻辑运算）
- **问题描述**：
  设计将 `&&`/`||` 映射为具名函数 `logicalAnd`/`logicalOr`，并声明"返回 `Vec<Bool, Q>`"。但 GLM C++ 中的 `operator&&`/`operator||` 对 `vec<bool>` 逐分量执行逻辑运算（非短路求值），其语义等价性与具名函数形式是否完全一致需要额外确认。更重要的是，GLM C++ 的 `operator&&` 和 `operator||` 是**运算符**，而迁移后的版本是**具名函数**，这改变了下游代码的使用方式——任何依赖于运算符语法（如 `if (v1 && v2)`）的 GLM 代码在迁移后需要改为 `if (logicalAnd(v1, v2))`。
  设计文档未评估此语法变更对现有 C++ GLM 代码迁移成本的影响。
- **改进建议**：
  在 §4.5 或设计决策中增加一节（或扩充 D12），评估将 `&&`/`||` 改为具名函数对下游迁移工作量的影响，并给出建议的代码迁移模式（如搜索-替换规则或 wrapper 函数）。

### P6（中）：Functor 与 ComputeVec 结构体未被运算符实现实际使用

- **位置**：§3.3（Functor 体系）、§3.4（ComputeVec 运算策略体系）、§4.3（算术运算）
- **问题描述**：
  设计文档定义了完整的 Functor 体系（16 个结构体）和 ComputeVec 策略结构体（如 `ComputeVecAdd`、`ComputeVecSub` 等），称其为 "Vec 运算符函数的内部委托目标"。但在 §4.3 中，运算符的实际实现描述为"copy + op=" 委托模式——即二元运算符通过拷贝加复合赋值实现，复合赋值直接在结构体内部逐分量运算。这条路径**完全不涉及** Functor 和 ComputeVec 结构体。
  结果是：Functor 和 ComputeVec 在当前的运算符设计中没有被引用，成为"孤儿代码"。如果它们的真实用途是供后续轮次的函数库（如 `func_common.hpp`、`func_geometric.hpp`）调用，则需要：
  - 明确说明它们的消费者是谁（哪些函数库、哪个轮次）
  - 确认当前轮次是否需要保留这些定义
  - 否则应推迟到实际使用时再引入，避免首轮产生死代码
- **改进建议**：
  在 §3.3/§3.4 中明确标注 Functor 和 ComputeVec 的消费链路：① 首轮 Vec 运算符是否使用它们（若不使用，说明原因）；② 后续哪一轮次的哪些函数库会消费它们；③ 如果首轮不消费，是否应推迟定义以降低首轮代码量。

### P7（中）：`operator[]` 签名未完整指定，缺少赋值形式

- **位置**：§4.2（分量访问）
- **问题描述**：
  设计仅说"通过 `operator []` 按索引访问"，未提供完整签名。仓颉中 `[]` 运算符有两种形式（只读和赋值）：
  - 取值：`operator func [](i: Int64): T`
  - 赋值：`operator func [](i: Int64, value!: T): Unit`
  赋值形式需要命名参数 `value!`（带 `!` 后缀），且下标参数在仓颉中可以接受多个索引。设计未指定这些细节，编码时需自行推断。
- **改进建议**：
  在 §4.2 中补充 `operator[]` 的完整签名，明确只读和赋值两种形式。同时明确索引类型（`Int64` vs `LengthT`）。

### P8（中）：§3.6 与 §5.1 关于 Release 模式断言行为存在矛盾

- **位置**：§3.6（`shim_assert.cj`）、§5.1（错误处理策略 — 下标越界）
- **问题描述**：
  §3.6 说："首轮暂不做消除，均保留断言"（倾向于全部保留，未来再处理）。
  §5.1 说："首轮建议通过 GLM_CONFIG_* 配置 + const if 在 Release 模式下跳过断言"（倾向于实现条件消除）。
  两条陈述指向不同的决策方向。编码阶段若遵循 §3.6，则所有断言在所有模式下都会触发；若遵循 §5.1，则需要额外编码实现条件跳转。设计文档未给出唯一的权威结论。
- **改进建议**：
  在 §3.6 和 §5.1 中统一表述，明确首轮是否实现 Debug/Release 区分。推荐统一为"首轮均保留断言，Release 模式消除标记为后续轮次任务"（与 §3.6 的陈述一致，因为 `const + if` 的接入改变了文件结构，不是零成本的改动）。

### P9（中）：`%` 运算符无条件定义为 VecN 导致浮点类型编译错误

- **位置**：§3.2（运算符列表示 `%` 在各 VecN 类型上默认列出）、§4.3（`%` 运算的整数约束）
- **问题描述**：
  设计文档建议在 `%` 上添加 `where T <: Integer<T>` 约束。但仓颉规定"运算符重载不能为泛型"，运算符函数不能有额外的泛型约束。因此不能对同一运算符在不同 T 类型上有不同的定义——`%` 要么在所有 VecN 类型上都可用（导致 T=Float32 时编译失败），要么在所有 VecN 类型上不可用（违反 GLM 兼容性）。
  设计文档未解决此矛盾，仅提供了"编译期类型检查"的模糊建议，未给出可编译的实现方案。
- **改进建议**：
  给出确定的实现策略：
  - 方案 A：`%` 运算符仅在 `where T <: Integer<T>` 约束的 VecN 上定义——但这需要修改 VecN 结构体定义，对 T 增加约束（违反 D5）
  - 方案 B：在 `%` 运算符函数体内使用 `match` 或编译期类型分支，但运算符函数体在 T=Float32 时仍会尝试调用 `%` 导致编译错误
  - 方案 C：`%` 仅在整数`Vec1~Vec4` 的派生类型别名上以具名函数（`.mod()`）形式提供，不定义为运算符
  需选择其中一条路径并记录理由。

### P10（中）：`ComputeEqual` 的 `IsFloat` 类型参数在仓颉中缺少可行的编译期映射

- **位置**：§3.5（ComputeEqual）
- **问题描述**：
  C++ GLM 的 `compute_equal<T, isFloat>` 使用 `bool` 非类型模板参数，C++ 编译器可以在偏特化中匹配 `true`/`false`。设计文档将此映射为 `ComputeEqual<T, IsFloat>`——其中 `IsFloat` 在仓颉中必须是一个**类型参数**（仓颉非类型参数仅支持整数/Rune/Bool 等有限类型，`Bool` 类型参数在泛型约束中的行为需要确认）。
  若 `IsFloat` 是 `Bool` 类型的非类型泛型参数，仓颉是否支持 `const if (IsFloat)` 在编译期选择分支？设计文档的"首选路径"依赖 `const if (NumericLimits<T>.isIec559)`，完全绕过 `IsFloat` 参数——这意味着 `IsFloat` 形同虚设。
- **改进建议**：
  明确 `IsFloat` 参数在仓颉中的映射形态——如果通过 `const if (NumericLimits<T>.isIec559)` 实现编译期分支，则 `IsFloat` 参数根本不需要（`ComputeEqual<T, IsFloat>` 应简化为 `ComputeEqual<T>`）；如果保留 `IsFloat` 参数（用作非类型 `Bool` 参数），需验证仓颉编译器是否支持。此问题在第 2 轮历史审议（P10）中已涉及，但仍有残留。

### P11（低）：`shim_limits.cj` 中 `isIec559` 的语义不等价于 C++ `std::numeric_limits<T>::is_iec559`

- **位置**：§3.6（`shim_limits.cj`）、§3.5（ComputeEqual）
- **问题描述**：
  设计文档将 `isIec559` 描述为"判断 T 是否为 IEEE 754 浮点类型"。但 C++ 的 `std::numeric_limits<T>::is_iec559` 具有更精确的语义：判断 T 是否**完全符合** IEEE 754 标准（不仅限于浮点类型，还包括对 IEC 559 的完整符合性）。仓颉的 `Float32`/`Float64` 是否完全符合 IEC 559 需验证。更重要的是，`isIec559` 在 shim 层需要人工定义（仓颉标准库未提供等价属性），这意味着：
  - 需要为每个数值类型（`Int8`~`UInt64`、`Float32`、`Float64`）手动设定 `isIec559` 值
  - `Int8` 等整数类型的 `isIec559` 应为 `false`，`Float32`/`Float64` 应为 `true`（或根据仓颉实现确认）
  - 如果仓颉的浮点类型不完全符合 IEC 559（如缺少某些 IEEE 754 要求的特性），则应设为 `false`
  设计文档未涉及这些实现细节。
- **改进建议**：
  在 §3.6 中补充 `isIec559` 的各类型取值对照表，并标注仓颉浮点类型的 IEC 559 符合性需要验证。

### P12（低）：§8 迁移文件清单中 `setup.cj` 说明列仍包含 `assert`

- **位置**：§8（迁移文件清单，文件 1 `setup.cj`）
- **问题描述**：
  清单中 `setup.cj` 的说明列写有"配置常量、`assert`"，但 `assert()` 函数实际归属于 `shim_assert.cj`（文件 3）。`setup.cj` 如果包含 `assert` 定义会导致职责重叠。此问题在 v1 审议（P8）中已处理过 `LengthT`/`SizeT` 归属，但 `assert` 的归属问题未同步清理。
- **改进建议**：
  将 `setup.cj` 说明列的 `assert` 删除或改为"引用 `shim_assert.cj`"。

## 整体评价

v3 设计文档在历史修订的基础上已修复了前两轮发现的 22 个问题，在**需求覆盖度**、**架构清晰度**和**决策记录完整性**上达到了较高水平。但在**仓颉语言语义的精确性验证**和**下游可消费性**方面仍存在不足：

1. **最严重的问题（P1）**：`mut operator func +=` 的合法性缺乏事实支撑——如果仓颉编译器不支持显式定义复合赋值运算符，整个溢出标注策略需要重构。这是所有问题中最紧急的，建议立即编写最小测试确认。
2. **类型系统一致性（P3/P9）**：设计引用了未定义的类型 `Vec<Bool, Q>`，且 `%` 运算符对浮点类型 Vec 的编译可行性未解决。这些问题会直接阻止编码阶段的第一行代码通过编译。
3. **实现链路完整性（P6/P10）**：部分组件（Functor/ComputeVec）的消费者未明确，存在歧义或死代码风险。

建议在进入编码阶段前，优先解决 P1（实体验证）、P3（类型引用修正）、P7（签名补充）三个阻塞性问题。
