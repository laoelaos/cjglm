审查对象：`a_v2_design_v1.md`（第 2 轮迭代）
审查视角：需求响应充分度、语言事实正确性、落地完整性
参考依据：仓颉官方语言特性文档、标准库文档、`01_roadmap.md`

---

## 一、事实错误 / 语言特性误用

### P1. `&&` / `||` 运算符不可重载

- **位置**：§3.2（布尔逻辑运算符）、§4.5（比较运算）
- **严重程度**：严重
- **描述**：设计文档声明在 Vec 上提供 `&&` 和 `||` 运算符（"仅限 `Vec<Bool, Q>`"）。但仓颉语言中 `&&` 和 `||` **不在可重载运算符列表内**。可重载运算符列表仅限于 `()`、`[]`、`!`、`-`（一元）、`**`、`*`、`/`、`%`、`+`、`-`（二元）、`<<`、`>>`、`<`、`<=`、`>`、`>=`、`==`、`!=`、`&`、`^`、`|`（`function/README.md:207`）。
- **改进建议**：移除 `&&` 和 `||` 的运算符重载设计。对于逐分量布尔逻辑运算，应使用具名函数（如 `logicalAnd`/`logicalOr`）或位运算符（`&`/`|`）替代。如果下游消费者需要 GLM 兼容的 `&&`/`||` 语法，需在后续轮次通过宏或其他机制重新设计。

### P2. `~` 运算符不可重载

- **位置**：§3.2（位运算符）、§4.4（位运算）
- **严重程度**：严重
- **描述**：设计文档声明在位运算符中包含 `~`（单目按位取反）。但仓颉语言中 `~` **不在可重载运算符列表内**（同上）。
- **改进建议**：移除 `~` 运算符重载。按位取反应通过具名函数（如 `bitwiseNot()`）实现。同时检查 §3.2 中"`~` 在结构体内部定义为单目运算符"的描述需要修正。

### P3. 跨 Vec 转换构造的泛型约束不适用于数值类型

- **位置**：§4.1（构造函数签名清单），第 4 行构造器 `init(v: Vec2<T2, Q2>) where T2 <: T, Q2 <: Q`
- **严重程度**：严重
- **描述**：该构造器的约束 `where T2 <: T` 要求 `T2` 为 `T` 的子类型，但仓颉中"原始数值类型之间不存在子类型关系"（`type_system/README.md:57`）。即 `Int8` 不是 `Int32` 的子类型，`Float32` 不是 `Float64` 的子类型。因此 `Vec2<Int8, Q2> → Vec2<Int32, Q>` 的跨类型转换在该约束下无法编译通过，与需求文档中"跨 Vec 转换构造"的期望直接矛盾。
- **改进建议**：重新设计跨 Vec 转换构造的约束机制。由于仓颉数值类型间无子类型关系，无法通过子类型约束表达"可转换"语义。可行的替代路径：① 不做约束（`where T2, Q2` 无限制），函数体内通过 `T(v)` 显式转换实现，编译器在调用时检查 `T(v)` 是否合法（将检查延迟到实例化点）；② 定义一个新的 `ConvertibleTo<T>` 接口，由各数值类型实现。

### P4. `const if` 在非 `const` 函数中不可用 — `==` 运算符需要 `const init`

- **位置**：§3.5（ComputeEqual）、§3.6（shim_limits）
- **严重程度**：严重
- **描述**：设计文档声明在 `==` 运算符"函数体内通过 `const if (NumericLimits<T>.isIec559)` 在编译期选择比较路径"。但 `const if`（编译期条件分支）仅在 `const` 函数上下文生效。对于 struct 类型，定义 `const` 实例成员函数的前提是 struct 已定义 `const init` 构造函数（`const/README.md:63`）。目前 Vec 结构体只有普通 `init` 构造函数，未设计 `const init`。若不为 Vec 添加 `const init`，则 `==` 运算符体内无法使用 `const if`。
- **改进建议**：增加以下设计说明——① 为 Vec 结构体添加 `const init` 构造函数（成员初始化全为 `const` 表达式），使 `==` 等比较运算符可声明为 `const` 函数；② 如不愿引入 `const init`，则替代方案为运行时 `if` 分支（编译器可能仍可优化，但非编译期消除）；③ `NumericLimits<T>.isIec559` 必须声明为 `const static prop` 或 `const` 函数，才能在 `const` 上下文中作为编译期常量使用。

## 二、逻辑矛盾 / 设计不一致

### P5. `@OverflowWrapping` 标注策略内部矛盾

- **位置**：§4.6（`@OverflowWrapping` 标注策略）
- **严重程度**：严重
- **描述**：§4.6 存在三处相互矛盾的表述：① 开头部分将二元运算符列为"直接标注 `@OverflowWrapping`"；② 中间部分又说"二元运算符委托给复合赋值实现，复合赋值已在被委托方标注"（暗示无需额外标注）；③ 末尾的"替代方案"又说"如果自动生成版本始终优先被解析，则二元运算符也直接标注"。这种模糊性导致编码人员无法确定二元运算符到底需不需要标注。同时，主路线的"二元运算符直接标注 + 委托给复合赋值"意味着双层 `@OverflowWrapping` 作用——外层的 wrapping 会将内层已经 wrapping 的结果再 wrap 一次，虽然数值结果一致但语义冗余。这与需求文档 `01_roadmap.md` §0.5.1 中明确的"二元算术运算符——**否**（委托给复合赋值 + 拷贝，无需额外标注）"相悖，且未给出偏离 roadmap 的理由。
- **改进建议**：做出明确且唯一的决策：① 确认仓颉显式 `mut operator func` 在重载解析中的优先级是否高于自动生成版本（需编码验证），并据此选定一种策略；② 如果显式定义优先，则二元运算符**不标注**（与 roadmap 一致），标注集中在复合赋值运算符（16 处）；③ 如果自动生成优先，则二元运算符和复合赋值运算符均需标注，且需修改 Vec 结构体定义方式（不在 struct 内定义复合赋值运算符，转而在 extend 块中统一管理）。

### P6. `scalar / vec` 行为描述与非存在运算符矛盾

- **位置**：§9.2（除零行为），与 §4.3（算术运算）
- **严重程度**：一般
- **描述**：§4.3 明确"`scalar + vec` 不可作为运算符重载实现……`scalar - vec` 同理"——因此 `scalar / vec` 同样不能以运算符实现。但 §9.2 的表格中出现了"标量零除向量（`0 / vec`）"的行为约定，仿佛该运算符存在。这种前后矛盾使下游编码人员困惑：到底要不要实现 `scalar / vec` 运算符？如果通过具名函数实现，§9.2 应注明函数名而非运算符语法。
- **改进建议**：统一表述。若要通过具名函数实现，在 §9.2 中将`0 / vec`改为`div(0, vec)`的语法；若不以任何形式支持，则应从 §9.2 中删除该行。

## 三、关键遗漏 / 深度不足

### P7. 缺少 GLM_CONFIG_* 常量的完整清单与值

- **位置**：§7 D6（GLM_CONFIG_* 配置常量）
- **严重程度**：一般
- **描述**：D6 仅提及"18 个 `GLM_CONFIG_*` 常量集中在 `setup.cj` 中声明"和"首轮所有配置取最保守值"，但未给出完整的常量清单和对应的保守值。需求文档 `01_roadmap.md` §3A.1 提供了 18 个常量的完整清单、CangJie 等效常量名、首轮推荐值和用途说明。设计文档作为"可供后续编码阶段直接使用"的产出，应继承这份清单或显式引用，否则编码人员无法知道 `setup.cj` 中到底要声明哪些常量以及它们的值。
- **改进建议**：在 D6 或单独一节中列出 18 个 GLM_CONFIG_* 常量的完整清单（常量名、仓颉等效名、首轮值、简要说明），可直接从 roadmap 文档引用或复制。

### P8. `%` 运算符在浮点 Vec 上的可用性未做分析

- **位置**：§3.2（运算符列表）、§4.3（算术运算）
- **严重程度**：一般
- **描述**：设计文档将 `%` 作为 Vec 的通用算术运算符，适用于所有 `T` 类型。但仓颉中 `%` 仅对整数类型有定义（`Int8`~`Int64`、`UInt8`~`UInt64`），对浮点类型（`Float32`/`Float64`）**无 `%` 运算符**。因此 `Vec3<Float32, Q> % Vec3<Float32, Q>` 的编译会失败。GLM C++ 中 `%` 同样仅对整数类型 vec 有意义——这是一个已知的跨语言差异。
- **改进建议**：在 §4.3 中明确 `%` 运算符仅在 `T` 为整数类型时可用，或在 Vec 的 `%` 重载中添加 `where T <: Integer<T>` 约束。同时检查§4.6"`%=` 不涉及溢出"的标注策略——若 `%=` 仅对整数类型有定义，则其溢出行为取决于整数类型，需标注 `@OverflowWrapping`（与 `+=` 等一致）。

### P9. `countof<T>(arr: T): Int64` 签名无法实现其功能

- **位置**：§3.6（shim_cstddef.cj）
- **严重程度**：一般
- **描述**：设计文档给出的 `countof` 签名是 `func countof<T>(arr: T): Int64`，描述为"针对固定长度数组或 `VArray` 类型，在编译期计算元素数量"。但签名 `arr: T` 中 `T` 无约束，函数体无法知道"数组长度"——长度为类型特性（`VArray<T, N>`），不是运行时值。对于 `VArray`，正确的签名应包含长度类型参数：`func countof<T, N>(arr: VArray<T, N>): Int64 { N }`。当前签名在编码阶段无法直接映射为可编译代码。
- **改进建议**：修正签名为 `func countof<T, N>(arr: VArray<T, N>): Int64` 或提供两个重载版本（数组字面量和 `VArray`），同时在 §3.6 中说明函数体内通过类型参数 `N` 获取编译期长度。

### P10. 缺少 `uint` 标量类型别名

- **位置**：§3.7（标量类型别名）
- **严重程度**：轻微
- **描述**：GLM 的 `fwd.hpp` 中标量别名包含 `uint`（对应 C++ `unsigned int`，首轮映射为 `UInt32`），且 `setup.hpp` 中定义了 `uint` 类型别名（`typedef unsigned int uint`）。设计文档的标量类型别名映射规则中未提及 `uint`。
- **改进建议**：在 §3.7 标量类型别名中补充 `type uint = UInt32`，同 `int`（= `Int32`）一并提供。

### P11. `operator[]` 错误处理与 C++ 语义差异未说明

- **位置**：§5.1（标准错误场景）
- **严重程度**：轻微
- **描述**：设计文档说"下标越界时调用 `assert()` 等效的 `if + throw` 模式"——即始终抛出异常。但 C++ GLM 的 `GLM_ASSERT_LENGTH` 是 C 风格的 `assert` 宏，在 Release 模式下通常被编译掉（`#define assert(x) ((void)0)`），不存在运行时开销。CangJie 的 `if + throw` 模式则无法在 Release 模式下消除。这一行为差异可能导致性能偏离 C++ 预期。
- **改进建议**：在 §5.1 中补充说明这一差异，并建议在 Debug/Release 配置间做出区分：Debug 模式下 `if + throw`，Release 模式下可降级为直接索引访问（不做边界检查且无运行时开销）。在设计层面记录此决策留痕。

## 四、需求响应充分度评价

### 整体结论

该设计文档对用户需求的响应总体充分——完整覆盖了首轮迁移范围的模块结构、类/接口设计、关键数据类型定义，§8 的文件清单和依赖拓扑为编码阶段提供了清晰的实施路径。第 1 轮审查中识别的 13 个问题在 v2 中已全部处理，修订记录清晰可追溯。

### 仍存的缺口

1. **可直接编码的精确度不足**：P1（`&&`/`||` 不可重载）、P2（`~` 不可重载）、P3（泛型约束不适用于数值类型）、P4（`const if` 前提条件未满足）四个语言事实错误直接导致编码阶段的编译失败，必须在编码前修正。
2. **标注策略未收敛**：P5 涉及的 `@OverflowWrapping` 策略在 v2 中仍存在内部矛盾，未形成明确唯一的决策。
3. **常量清单缺失**：P7 指出的 18 个 `GLM_CONFIG_*` 常量清单缺失，将使 `setup.cj` 的编码工作无法独立启动。
