## 逐维度审查

### 1. 类型系统可行性

**[通过]**

- Design 对仓颉类型系统的能力边界有清晰认知：Qualifier 采用 interface + 空 struct 模式（enum 不可作泛型实参、sealed class 有堆分配开销），Vec 采用四个独立泛型 struct（仓颉不支持偏特化），类型别名使用 `type` 语法
- 泛型约束 `where Q <: Qualifier` 语法可行；`where T2 <: T` 在数值类型间不成立（仓颉无隐式数值子类型关系），设计已正确改用无约束泛型 + 显式转换
- 运算符重载边界已准确识别（可重载列表、`&&`/`||`/`~`/`++`/`--` 不可重载、复合赋值不可显式定义、运算符不能引入新类型参数）
- `const init` 用于启用 `const` 实例成员函数（`==`），配合 `const if` + `is` 运算符实现编译期分支，模式在仓颉 const 表达式规则范围内
- `extend` 块定义位运算符成员，语法可行
- 跨类型构造函数标量参数已全部修正为 `T`（目标类型），§4.1 签名清单已验证

**[一般]** `const init` 与 `init` 构造函数签名重复：§4.1 中 Vec1~Vec4 均同时声明了 `public init(x: T, y: T, ...)` 和 `const init(x: T, y: T, ...)` 两个具有相同参数列表的构造函数。仓颉函数重载规则中 `const` 修饰符不构成区分依据（const README §3.2 规则 7："const 函数与非 const 函数重载规则一致"），二者将产生重复定义编译错误。

- **建议方向**：移除冗余的非 `const` 版本 `init(x: T, y: T, ...)`，仅保留 `const init(x: T, y: T, ...)`。`const init` 在非 const 上下文中同样可用（const README §3.2 规则 5："如果该函数调用的实参不符合 const 表达式要求，那这个函数调用不能作为 const 表达式使用，但仍然可以作为普通表达式使用"）。其他非 `const` 构造函数（如跨类型转换构造函数）可保留 `init` 形式，不受影响。

### 2. 标准库与生态覆盖

**[通过]**

- 所有需要的能力均在仓颉标准库/语言内置能力范围内：原生数值类型（`Int32`/`Float32`/`Bool` 等）、`@OverflowWrapping` 溢出标注（内置）、`is` 运算符、`const` 函数
- 设计不依赖仓颉扩展标准库（stdx），无外部依赖风险
- `std.math.abs` 和 `NumericLimits<T>.epsilon()` 是否在 const 上下文中可用未在标准库文档中确认——设计已意识到此风险，在 §3.5 和 §10 中明确标注了"编码前验证要求"及备选回退路径（所有类型使用精确比较），风险可控
- `type` 别名语法符合仓颉规则（仅限顶层、无 `where` 约束、不创建新类型）

### 3. 语言特性可行性

**[通过]**

- 错误处理策略匹配仓颉能力：下标越界使用 `if + throw` 断言等效、整数溢出通过 `@OverflowWrapping` 控制、除零依赖运行时行为
- 值类型语义天然线程安全，首轮无共享状态，无需并发原语
- 包结构（`package glm.detail` 与 `package glm`）符合 cjpm 项目组织方式，依赖方向清晰（`glm` → `glm.detail`，反向无依赖）
- 资源配置常量用 `const` 声明，符合编译期常量机制
- `const if` 在 `const` 函数体内用于编译期分支，语法在 const 表达式规则内可用

**[轻微]** 设计在 §4.2 中使用 `operator func [](i: Int64, value!: T): Unit` 作为下标赋值形式的语法。`value!` 是命名参数语法，需在编码阶段确认仓颉 `operator func []` 的赋值形式是否接受此写法。若编译器不接受 `!` 后缀命名参数，可使用标准 `value` 参数名或编译器规定的特殊参数名。

### 4. 设计一致性

**[通过]**

- 各抽象职责描述清晰：Qualifier（精度/对齐标记）、Vec1~Vec4（数学向量值对象）、ComputeEqual（相等比较策略）、Functor（逐分量函数映射，预留）
- 模块间依赖方向合理：`setup` 无依赖 → `qualifier` → `compute_vector_relational`/`vectorize`/`type_vecN` → `fwd`，无循环依赖
- 构造函数体系（§4.1）以完整签名清单形式列出，覆盖 GLSL 5.4.1 构造语义，无需参考 C++ 宏展开
- §3.2 集中描述了 `Vec<Bool, Q>` 的完整行为约定，避免了分散描述的矛盾
- 所有设计决策（D1~D26）有明确理由，决策记录完整

### 5. 设计质量

**[通过]**

- 职责划分遵循单一职责：Vec 关注数据与运算、ComputeEqual 关注比较策略、Qualifier 实现类仅为类型标记
- 抽象层次恰当：Functor/ComputeVec* 定义为结构体但首轮不消费（为后续 SIMD 轮次预留），不引入不必要的复杂性
- 别名分级策略（必备/常用/可选）合理，编码顺序明确
- 验证策略完整：四层验证（编译→构造访问→运算正确性→异常边界）+ 验收标准表，可指导后续编码和测试
- `isIec559Of<T>()` 的 const 可行性已在 §10 明确标注为"编码前验证要求"，并提供了备选回退路径

## 修改要求

对每个一般问题，提供：

### 问题 1：`const init` 与 `init` 签名重复

- **问题**：Vec1~Vec4 在 §4.1 中同时声明了 `public init(x: T, y: T, ...)` 和 `const init(x: T, y: T, ...)` 两个构造函数，二者参数列表完全相同。仓颉函数重载不因 `const` 修饰符区分签名，将导致重复定义编译错误。
- **原因**：违反仓颉函数重载规则，编译不可通过，影响设计可行性。
- **建议方向**：移除所有与 `const init` 参数列表相同的非 `const` `init` 版本，仅保留 `const init`。`const init` 在运行时上下文中同样适用（const README §3.2 规则 5），不影响运行时构造的可用性。其他不同参数列表的 `init` 构造函数（如跨类型转换、标量填充、Vec1 填充等）不受影响，无需改动。
