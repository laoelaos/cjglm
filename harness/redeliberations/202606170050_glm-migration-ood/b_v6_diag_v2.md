## 保留的前序审查问题

以下问题来自 `b_v6_diag_v1.md`，经质询后确认仍成立：

### 问题 1（严重）— 事实错误：`<<` 运算符右操作数类型的语言限制描述不准确

**所在位置**：§4.3（第 311 行）、§4.4（第 323 行）

**问题描述**：
§4.3 第 311 行断言"仓颉运算符不能引入新的类型参数"——此推论正确。但 §4.4 第 323 行进一步声称"仓颉 `<<` 运算符重载仅接受 `T` 类型右操作数（与左操作数分量类型相同），因此移位量类型受限于 `T`"，这是事实错误。

仓颉的运算符重载限制是"不能引入新的类型参数"，而非"运算符右操作数类型必须与左操作数的分量类型相同"。以下写法完全合法：
```
public operator func <<(rhs: Int64): Vec2<T, Q> { ... }
```
这里的 `Int64` 是具体类型，不是新的类型参数。因此移位量完全可以定义为 `Int64`，而非必须为 `T`。

### 问题 2（严重）— 与 roadmap 策略矛盾：`@OverflowWrapping` 标注策略声明"完全对齐"但实际相反

**所在位置**：§4.6（第 336 行）

**问题描述**：
§4.6 第 336 行声明"本设计采用以下唯一确定的策略管理溢出标注，与 roadmap §0.5.1 完全对齐"，但实际策略与 roadmap 所述完全相反。

Roadmap §0.5.1（原文第 79-97 行）明确：
- 标注对象：**复合赋值运算符**（`+=`、`-=`、`*=`、`/=`、`%=`）
- 二元运算符：**否**——委托给复合赋值 + 拷贝，无需额外标注
- 标注量：约 16 处

本设计 §4.6 策略：
- 标注对象：**二元算术运算符**（`+`、`-`、`*`、`/`、`%`）
- 复合赋值：编译器自动生成
- 标注量：约 32 处

两个策略各有合理之处（设计在迭代 v3→v4 过程中已论证 roadmap 方案因复合赋值不可显式定义而不可行），但"完全对齐"的声明是事实错误。

### 问题 3（一般）— 别名分级表格数值不一致

**所在位置**：§3.8（第 247-253 行）

**问题描述**：
别名实用性分级表格中的"常用"级声称 ~64，但实际计算仅为 24；"可选"级声称 ~160，实际为 72。32 + 24 + 72 = 128 ≠ 256。

### 问题 4（一般）— 缺失 `<`/`<=`/`>`/`>=` 逐分量比较运算符的设计

**所在位置**：§4.5（比较运算）

**问题描述**：
C++ GLM 为向量类型定义了完整的逐分量比较运算符集（`<`、`<=`、`>`、`>=`），返回 `VecN<Bool, Q>`，且上述运算符均在仓颉可重载运算符列表中（见仓颉语言规范可重载运算符清单，含 `<`、`<=`、`>`、`>=`）。但设计文档仅讨论了 `==` 和 `!=`，对此四组运算符完全未提及。

### 问题 5（一般）— `<<` 移位量类型设计未评估 `Int64` 路径

**所在位置**：§4.4（第 323-324 行）

**问题描述**：
接问题 1 的事实错误。§4.4 的移位量类型仅考虑了 `T`，未评估使用 `Int64`（固定整数类型）的替代方案。使用 `T` 作为移位量在 C++ GLM 中没有对应——C++ GLM 统一使用 `int` 移位量。

**所在位置**：§3.2、§4.3、§4.4、§4.5、§4.6、D5 等多处

## 新增审查问题（v2 补充）

以下问题聚焦于前序审查未充分覆盖的维度：需求响应充分度、完整性、下游消费者就绪度。

### 问题 8（严重）— 构造函数体系未作为自包含规范提供，依赖外部 C++ 源码才能确定完整清单

**所在位置**：§4.1（向量构造，第 287 行）

**问题描述**：
§4.1 明确声明签名清单为"**示例性**"，并要求编码阶段"以 GLM C++ 源文件中的实际 `#define` 展开为准"。这使设计文档在构造函数规格上不具备独立完整性——编码人员必须交叉参考 GLM C++ 源码，且 C++ 构造函数由 `#define` 宏生成（宏展开逻辑本身需要人工翻译到仓颉），增加了编码歧义空间。

具体风险：
1. Vec3/Vec4 的跨分量数组合构造函数种类多（C++ 中由多个宏展开生成），设计文档仅给出 Vec2 的示例签名清单，未给出 Vec3/Vec4 的等价清单。
2. C++ 中 `#define` 宏展开与仓颉构造函数语法不一一对应，设计文档应承担翻译责任（提供仓颉语法的完整签名清单），而非将翻译工作推给编码人员。
3. 不同编码人员解读同一组 `#define` 宏可能产生不同的构造函数清单，导致 Vec1~Vec4 各文件之间构造接口不一致。

**改进建议**：
在 §4.1 中提供 Vec1~Vec4 各类型的完整构造函数签名清单（仓颉语法），替代"示例性 + 以 C++ 展开为准"的表述。签名清单可按 Vec 类型分组，每组列出该类型支持的全部构造函数重载。对于 C++ 宏生成的组合，逐条展开为仓颉的显式签名。

### 问题 9（一般）— `storage` 结构体未映射也未解释省略理由

**所在位置**：§3.1（qualifier 体系）、§8（迁移文件清单）

**问题描述**：
C++ GLM 的 `qualifier.hpp` 定义了 `storage<L, T, Q>` 模板结构体，该结构体是 Vec 类型数据成员类型选择的核心组件（管理对齐/打包策略下的数据布局控制）。Roadmap §3A.2 明确将 `storage` 结构列为 `qualifier.hpp` 的迁移内容。

设计文档的 `qualifier.cj` 仅描述了 `Qualifier` 接口及实现类型、前向声明，完全未提及 `storage`。§8 迁移文件清单中 `qualifier.cj` 的说明列也只写了"`Qualifier` 接口及实现类型、V 前向声明注释"，无 `storage` 条目。

由于 CangJie 的 Vec 结构体直接使用 `var x: T` 等原生字段（而非通过 `storage` 包装器），`storage` 可能在设计中被主动省略。但设计文档没有记录此省略决策（缺少一项 D20），编码人员交叉参考 GLM C++ 源码时会产生疑虑：是否有 `storage` 需要实现？如果不需要，为什么不需？

**改进建议**：
在 §3.1 末尾（或新增一项设计决策 D20）中说明 `storage` 的处理策略：
- 若确定省略：记录省略理由（如"仓颉 struct 字段定义直接管理数据布局，无需 `storage` 包装层；对齐策略由 `var` 成员和编译器布局规则保证，`storage` 在首轮设计中被简化消除"）。
- 若需要部分映射：说明首轮简化后的等效方案（如始终使用非对齐路径，对应 C++ 中 `GLM_ARCH & GLM_ARCH_SSE2_BIT == 0` 的通用分支）。
同时更新 §8 中 `qualifier.cj` 的说明列。

### 问题 10（一般）— 需求追溯：`genTypeTrait` 等辅助设施未覆盖

**所在位置**：§3.1（qualifier 体系）、§8（迁移文件清单）

**问题描述**：
Roadmap §3A.2 将 `genTypeTrait`、`find_best_type` 及其他辅助 traits 列为 `qualifier.hpp` 的首轮迁移内容。设计文档的 `qualifier.cj` 中无任何提及。

虽然在首轮 CangJie 实现中这些 C++ traits 可能不直接需要（仓颉泛型系统与 C++ 模板 trait 差异大），但作为需求覆盖审计，这是一个显式缺失项。编码/设计人员需要在以下两种路径中选择并记录：① 确定首轮不需要，记录省略理由；② 确定需要部分映射，给出简化的仓颉等效。

**改进建议**：
审查 C++ `qualifier.hpp` 中的 `genTypeTrait` / `find_best_type` / 其他 trait 模板在 Vec 结构体定义和运算中的实际使用情况。对首轮无用的 trait 记录"本轮不迁移"及理由；对仍有需要的 trait 给出仓颉等效方案。在 §3.1 或 D20 中记录审计结论。

### 问题 11（一般）— Vec 结构体的 Q 参数未指定默认值

**所在位置**：§3.2（Vec 结构体系）、§7 D2

**问题描述**：
D2（第 398 行）提到"可提供默认类型参数 `= PackedHighp`"（语气为可选），§3.2 未明确说明 Vec1~Vec4 的 `Q` 参数是否具有默认值。同时，§4.1 构造函数签名清单中使用了 `Vec2<T, Q>`（无默认值写法）。

编码人员需要明确回答：以下哪种写法是合法的？
```cangjie
// 方案 A: Q 有默认值，可直接写
let v = Vec2<Float32>(1.0, 2.0)
// 方案 B: Q 无默认值，必须写
let v = Vec2<Float32, PackedHighp>(1.0, 2.0)
```

如果 Q 有默认值，所有 256 个别名和构造函数签名中的 `VecN<T, Q>` 是否需要补全 `= PackedHighp`？如果 Q 无默认值，§4.1 的构造函数签名与 D2 的"可提供"描述存在矛盾。

**改进建议**：
在 §3.2 中明确声明 Q 参数是否具有默认类型参数（`= PackedHighp`），并在 §4.1 的签名清单中统一体现。若提供默认值，建议同步更新构造函数签名示例为 `Vec2<T, Q = PackedHighp>`。

### 问题 12（轻微）— `operator[]` 索引类型未做设计决策记录

**所在位置**：§4.2（分量访问，第 291 行）

**问题描述**：
§4.2 选择 `Int64` 作为 `operator[]` 的索引类型。C++ GLM 使用 `int`（32 位）。对于分量数仅 1~4 的向量，`Int64` 在值域上明显过宽，`Int32` 或定制的 `LengthT` 别名更合理。设计文档没有记录此选择的原因或替代方案评估。

**改进建议**：
在 §4.2 或新增设计决策中增加一行说明：选择 `Int64` 的理由（如"与 `LengthT = Int64` 保持一致，避免引入额外的整数类型转换"），或改为 `Int32`/`LengthT` 并说明理由。

### 问题 13（轻微）— §9.3 NaN/Infinity 比较描述的改进方案不可行

**所在位置**：§9.3（第 596 行）

**问题描述**：
§9.3 在描述 `Inf == Inf` 容差比较路径的局限性时，提出"改进方案：在容差比较前先通过 `is` 检查是否为无穷大或 NaN 特殊值"。但仓颉中 `is` 运算符用于**类型检测**（如 `x is Float64`），而非**值检测**（如 `isInfinity(x)`）。检查浮点数是否为无穷大需要使用运行时方法（如 `Float64.isInfinite(x)`），该方法不能在 `const if` 上下文中使用。

因此该改进方案在首轮设计的编译期分支框架（`const if` + `is` 类型检测）下不可行。设计文档描述了一个无法实现的备选路径，可能误导后续实现者。

**改进建议**：
修正 §9.3 的改进方案描述：移除"通过 `is` 检查"的说法，改为"在非 const 路径下，可在运行时通过 `Float64.isInfinite()`/`Float64.isNaN()` 方法做预处理断言"。或说明此局限性在首轮范围中接受（因为容差比较路径本身已声明适用于有限值范围内的浮点比较）。

### 问题 14（轻微）— 迁移验证标准未覆盖全部异常场景

**所在位置**：§12（实现验证与验收策略，第 739 行）

**问题描述**：
§12.2 验收标准中"边界行为"条目（第 6 行）声明验证"§9 约定的零向量、除零、NaN/Infinity 传播、类型转换边界行为与约定一致"，但 §12.1 的具体验证步骤（层次三）中缺少以下异常场景的验证用例：
- 下标越界触发断言（§4.2 约定）
- `Vec<Float32>` 上调用 `%` 产生编译错误（§4.3 约定）
- `Vec<Float32>` 上调用 `bitwiseNot()` 产生编译错误（§4.4 约定）
- `Vec<Bool>` 上调用 `increment()`/`decrement()` 产生编译错误（§4.3 约定）

验收标准与验证步骤之间的覆盖不一致，编码人员在编写测试时可能遗漏这些边界验证点。

**改进建议**：
在 §12.1 层次三补充上述四个异常场景的验证步骤，确保验收标准中"边界行为"有对应的可执行验证用例。
