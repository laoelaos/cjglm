## 发现问题清单

### 问题 1（严重）：`equalExact` 函数声明为非 `const` 但无技术理由，且与 `==` 的 const 设计策略不一致

- **所在位置**：§4.5（比较运算），第 522 行
- **问题描述**：`equalExact` 被声明为"普通成员函数（非 `const`）"。该函数仅执行只读分量比较（读取 `x`/`y`/`z`/`w` 成员并调用仓颉原生 `==`），不修改任何实例成员。仓颉语言允许 struct 的只读成员函数声明为 `const`（const README §3.2 规则 9 明确允许）。当前声明为非 `const` 意味着：
  1. 在 `const` 表达式中无法调用 `equalExact`，限制了其在编译期计算场景的可用性
  2. 与 `==` 运算符的设计策略不一致——`==` 被精心设计为 `const` 函数以支持 `const if` 编译期分支，而 `equalExact` 作为补充精确比较方案同样是纯计算函数，无理由降低其 const 级别
- **严重程度**：**一般（中等）**
- **改进建议**：将 `equalExact` 改为 `const` 扩展成员函数。理由：① 函数体纯读取无修改，满足 const 函数要求；② 与 `==` 运算符的 const 策略保持一致；③ 仓颉 `const` 函数在非 const 上下文中同样可作为运行时函数使用（const README §3.2 规则 5），因此改为 `const` 不会丢失运行时调用能力；④ 需确保 extend 块中 `const` 扩展成员函数的语法合法性（仓颉扩展支持 `const` 函数）。

### 问题 2（一般）：`epsilonOf<T>()` 函数在使用处（§3.5 ComputeEqual 代码示例）未被显式定义

- **所在位置**：§3.5（ComputeEqual），第 237 行，代码示例中引用 `epsilonOf<T>()`
- **问题描述**：`ComputeEqual.callConst` 的浮点容差比较路径使用了 `epsilonOf<T>()`：
  ```
  abs(a - b) <= epsilonOf<T>()
  ```
  但该函数的签名、所属文件、返回值类型、约束条件在文档任何位置均未显式定义。§7 D29 的回退方案讨论中提及的是 `NumericLimits<T>.epsilon()`。编码人员无法判断 `epsilonOf<T>()` 是独立函数还是 `NumericLimits<T>.epsilon()` 的封装别名，也无法确定其所属文件（`shim_limits.cj` 或 `compute_vector_relational.cj`）。
- **严重程度**：**一般**
- **改进建议**：在 §3.5 或 §3.6（shim_limits.cj）中补充 `epsilonOf<T>()` 的显式定义，包括：
  - 完整签名，如 `func epsilonOf<T>(): T where T <: Number<T>`
  - 所属文件（推荐 `shim_limits.cj`）
  - 实现策略（直接委托给 `NumericLimits<T>.epsilon()` 或使用固定容差值如 `T(1e-6)`）

### 问题 3（一般）：`equalExact` 验收标准存在但 §12.1 验证层次中无对应测试覆盖

- **所在位置**：§12.1（验证层次）vs §12.2（验收标准），第 1133 行
- **问题描述**：§12.2 验收标准表已包含"`equalExact` 正确性"验收项（第 1133 行），但 §12.1 的四个验证层次（编译验证、构造与访问验证、运算正确性验证、异常场景验证）中均未显式提及 `equalExact` 的测试用例。层次三"比较运算"部分仅覆盖 `==` 和 `!=`。编码阶段若无独立测试计划覆盖 `equalExact`，该验收项可能被遗漏。
- **严重程度**：**一般**
- **改进建议**：在 §12.1 层次三"比较运算"或层次四"异常场景与边界验证"中补充 `equalExact` 验证子项，明确：① 浮点 Vec 上 `equalExact(Inf)` 返回 `true`；② `equalExact(NaN)` 返回 `false`；③ 整数/Bool Vec 上 `equalExact` 与 `==` 行为等价。

### 问题 4（轻微）：`Float16` 的 `isIec559Of` 说明与代码实际行为不完全一致

- **所在位置**：§10（取值对照表），第 963 行
- **问题描述**：对照表将 `Float16` 标注为 `false`，附注说明"简化策略，首轮不纳入 `isIec559Of` 判断"。但实际的 `isIec559Of<T>()` 实现（`T(0) is Float64 || T(0) is Float32`）即使不添加 `Float16` 排除逻辑也会返回 `false`——因为 `Float16(0) is Float64` 为 `false`，`Float16(0) is Float32` 也为 `false`。附注的"简化策略排除"表述暗示了需要主动排除，与实际代码行为不完全一致，可能误导编码人员认为需要额外的显式排除分支。
- **严重程度**：**轻微**
- **改进建议**：将第 963 行的附注修改为"非 IEC 60559 浮点类型。`isIec559Of<Float16>()` 隐式返回 `false`（`is` 类型检测不匹配），无需显式排除"。

### 问题 5（轻微）：文档版本标识与实际增量内容不匹配

- **所在位置**：文档标题（第 1 行）vs 末尾修订说明（第 1277-1284 行）
- **问题描述**：文档标题标注为"v13 — 审查迭代修订版"，但末尾包含完整的 v14 修订说明（4 项修订）。v14 说明中的 4 个问题在主体内容中已体现修复结果（D16 与 §4.3 分工、equalExact 验收标准、版本标识、D29 回退条件），但版本号未升至 v14，修订说明与标题版本不一致。
- **严重程度**：**轻微**
- **改进建议**：将文档标题更新为 v14，或在末尾修订说明中明确标注"v14 修订说明已合并入正文"。确保版本标识准确反映文档的实际增量状态，避免版本管理混淆。
