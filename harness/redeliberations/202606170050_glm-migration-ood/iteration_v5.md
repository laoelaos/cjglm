## 需要解决的问题（仅 RETRY 时存在）

- **问题描述**：§9 标题下仅保留小节标题，正文写"（与 v3 一致，略去具体内容）"，编码人员无法独立获取零向量行为、除零、NaN/Infinity 传播、类型转换边界等关键边界约定
- **所在位置**：§9（异常场景和边界条件补充），全文第 534-543 行
- **严重程度**：严重
- **改进建议**：恢复 §9 的全部具体内容，或以条文形式直接将关键边界约定写回本节，确保 v5 自洽完整

- **问题描述**：设计多处依赖 `const if (isIec559Of<T>())` 实现编译期分支，但仓颉泛型函数不允许按类型参数约束进行重载，亦不支持 C++ 式泛型函数全特化，主设计路径的可行前提尚未验证
- **所在位置**：§3.5（ComputeEqual）、§3.6（shim_limits.cj 实现策略）
- **严重程度**：严重
- **改进建议**：明确 `isIec559Of<T>()` 的仓颉实现方案。可选路径包括：① 在函数体内使用 `const if` 配合类型比较（需验证仓颉是否支持 const 上下文的 `T == Float32` 语法）；② 放弃编译期分支，统一使用 `==` 精确比较；③ 采用反射或注解方式获取运行期类型信息

- **问题描述**：构造函数签名中多处引用 `Vec1<T2>` 而非 `Vec1<T2, Q2>`，缺少 Qualifier 参数；文档未明确 Vec1~Vec4 的 Q 参数是否具有默认值
- **所在位置**：§4.1（向量构造，第 261-263 行）
- **严重程度**：一般
- **改进建议**：在 §3.2 或 §3.1 中明确声明默认 Qualifier 策略，同步修正 §4.1 中所有 `Vec1<T2>` 引用为 `Vec1<T2, Q2>`

- **问题描述**：`bitwiseNot()` 描述未说明浮点类型 Vec 实例化时 `!` 运算符不可用将产生编译错误
- **所在位置**：§4.4（位运算，bitwiseNot 段落）
- **严重程度**：一般
- **改进建议**：补充说明 `bitwiseNot()` 对浮点类型 T 的 Vec 实例产生编译错误，与 D5（宽松泛型约束，延迟检查）一致
