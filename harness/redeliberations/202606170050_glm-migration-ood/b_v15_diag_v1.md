## 问题清单

### 问题 1（轻微）— §3.2 运算符定义位置的双选项表述残留

- **位置**: §3.2 第 162 行
- **问题描述**: "二元算术运算符在 Vec 结构体内部定义（**或通过 extend 块扩展**）" 保留了双选项表述。§4.3 第 505 行已明确决策为"在 Vec 结构体内部定义"并给出具体实现模式。此残留语言源自早期迭代，虽不影响实现但降低了表述精确性。
- **改进建议**: 将"（或通过 extend 块扩展）"改为确切表述，与 §4.3 对齐。例如改为"二元算术运算符在 Vec 结构体内部定义为 `operator func`（非 `mut`）"。

### 问题 2（轻微）— §4.3 `vec + scalar` 运算符重载签名未显式列出

- **位置**: §4.3 第 505-506 行
- **问题描述**: 设计声明支持 `vec + vec` 和 `vec + scalar` 两种形式，但 §4.3 仅给出与构造函数清单类似的显式签名列表。在 §4.1 中每个 Vec 类型的构造函数签名以完整仓颉语法列出（共计 100+ 行），但算术运算符的签名仅在正文中以叙述方式描述。读者需自行推导出存在两个重载版本：`operator func +(rhs: VecN<T,Q>): VecN<T,Q>` 和 `operator func +(rhs: T): VecN<T,Q>`。对于标为"可直接用于编码"的设计文档，签名叙述与构造函数清单的格式不一致增加了编码阶段的推导负担。
- **改进建议**: 在 §4.3 中效仿 §4.1 的格式，为 Vec1~Vec4 分别给出二元算术运算符的完整重载签名清单（`vec + vec` 和 `vec + scalar` 两个版本）。

### 问题 3（轻微）— `isIec559Of<T>()` 的 `T(0)` const 构造依赖未完全覆盖所有目标类型

- **位置**: §3.5（ComputeEqual）、§3.6（shim_limits.cj）、§10 编码前验证要求
- **问题描述**: `isIec559Of<T>()` 的实现依赖 `T(0)` 在 const 上下文中的构造合法性。§10 的编码前验证要求中列举了 `T = Int32`、`Float32`、`Bool` 三种类型作为原型验证目标。但 §9.4 类型转换边界表（第 965 行）明确指出"仓颉中 `Int32(Bool)` 可能不被支持"，虽然此说明针对跨 Vec 转换构造场景，但揭示了一个潜在风险：某些类型（尤其是 `Bool`）的数值类型构造函数在 const 上下文中的行为需要独立验证。建议在所有目标类型（含 `UInt8`/`Int64`/`Float64` 等）上逐一确认。
- **改进建议**: 扩展 §10 的编码前验证范围，将验证目标类型扩充至 `Int8/Int16/Int32/Int64/UInt8/UInt16/UInt32/UInt64/Float32/Float64/Bool`，确保 `T(0)` const 构造在全部目标类型上的合法性。若 `Bool(0)` 在 const 上下文中不可用，需按备选路径（精确比较）执行或为 `Bool` 单独提供实现。

### 问题 4（轻微）— §4.3 `mod` 的整数约束段落位置与 §13 修订历史不一致

- **位置**: §4.3 第 519 行（`%` 运算的整数约束段落）
- **问题描述**: v15 修订说明第 3 项明确将 `mod` 的"不属于首轮范围"修正为"属于首轮范围"，并声称矛盾已消除。但核查 §4.3 第 519 行末尾原文："对于需要浮点数取模的场景，使用具名函数 `mod`（属于首轮范围）"。此处虽将 `mod` 标注为首轮范围，但上下文位于"`%` 运算的整数约束"段落中，讨论重点为 `%` 运算符的整数类型限制而非 `mod` 函数的作用域。读者在快速浏览时可能因段落主题（`%` 的整数约束）而误解 `mod` 仅与整数取模相关，忽略其对浮点场景也适用。
- **改进建议**: 在 §4.3 的 `mod` 具名函数定义位置说明（第 512 行附近）增加一行说明，明确 `mod` 的设计意图：整数分量的 `mod` 通过 `%` 逐分量运算实现；浮点分量的 `mod` 不通过 `%` 实现（因仓颉浮点类型不支持 `%`），需在编码阶段确定浮点取模的算法实现（如通过 `fmod` 等效的 FFI 调用或数学库函数）。
