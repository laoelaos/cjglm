## 问题 1（严重）— 事实错误：`<<` 运算符右操作数类型的语言限制描述不准确

**所在位置**：§4.3（算术运算，第 311 行）、§4.4（位运算，第 323 行）

**问题描述**：
§4.3 第 311 行断言"仓颉运算符不能引入新的类型参数"并由此得出"不支持跨类型操作数"——此推论正确。但 §4.4 第 323 行进一步声称"仓颉 `<<` 运算符重载仅接受 `T` 类型右操作数（与左操作数分量类型相同），因此移位量类型受限于 `T`"，这是事实错误。

**事实核查**：
仓颉的运算符重载限制是"运算符重载不能为泛型"（即不能引入新的**类型参数**），而非"运算符右操作数类型必须与左操作数的分量类型相同"。以下写法完全合法：

```
struct Vec2<T, Q> <: Qualifier {
    public operator func <<(rhs: Int64): Vec2<T, Q> { ... }
}
```

这里的 `Int64` 是具体类型，不是新的类型参数 `U`。因此移位量完全可以定义为 `Int64`，而非必须为 `T`。

**严重程度**：严重

**改进建议**：
修正 §4.4 移位量类型的设计理由陈述。移除"仓颉 `<<` 运算符重载仅接受 `T` 类型右操作数"的错误描述。评估将移位量类型改为 `Int64`（与 C++ GLM 的 `vec << int` 一致）的可行性，与非整数类型 Vec（如 `Bool`、`Float32`）的交互可通过 D5 的宽松约束自然处理（移位运算在非整数 `T` 实例化时编译报错）。

## 问题 2（严重）— 与 roadmap 策略矛盾：`@OverflowWrapping` 标注策略声明"完全对齐"但实际相反

**所在位置**：§4.6（`@OverflowWrapping` 标注策略，第 336 行）

**问题描述**：
§4.6 第 336 行声明"本设计采用以下唯一确定的策略管理溢出标注，与 roadmap §0.5.1 完全对齐"，但实际策略与 roadmap 所述完全相反：

| 策略要素 | Roadmap §0.5.1 | 本设计 §4.6 |
|---------|---------------|------------|
| 标注对象 | 复合赋值运算符（`+=` 等） | 二元算术运算符（`+` 等） |
| 二元运算符实现 | 委托给复合赋值（拷贝+op=） | 直接实现逐分量运算 |
| 复合赋值来源 | 显式定义（标注） | 编译器自动生成 |

Roadmap §0.5.1 明确写道："二元算术（非复合赋值）— **否**——委托给复合赋值 + 拷贝，无需额外标注"、"**算术复合赋值**（`+=`、`-=`、`*=`、`/=`、`%=`）— **是**（仅整数分量）"。而本设计采用反转策略——二元运算符标注，复合赋值自动生成。

两个策略各有合理之处（本设计在迭代 v3→v4 过程中已论证 Roadmap 方案不可行），但"完全对齐"的声明是事实错误。Roadmap 尚未更新，后续编码人员若同时参考两篇文档将产生混淆。

**严重程度**：严重

**改进建议**：
① 将 §4.6 第 336 行的"与 roadmap §0.5.1 完全对齐"改为"与 roadmap §0.5.1 的委托模式不同——已在本设计迭代 v3→v4 中确认 roadmap 方案因复合赋值不可显式定义而不可行，目前采用反转标注策略"。② 建议同步更新 `docs/01_roadmap.md` §0.5.1 以消除文档间差异。

## 问题 3（一般）— 别名分级表格数值不一致

**所在位置**：§3.8（向量类型别名，第 247-253 行）

**问题描述**：
别名实用性分级表格中的"常用"级数量声称 ~64，但实际计算仅为 24：

| 级别 | 声称数量 | 实际计算 | 偏差 |
|------|---------|---------|------|
| 必备 ~32 | 4 家族 × 4 分量数 × 2 精度 = 32 | 32 | ✓ |
| 常用 ~64 | 3 家族（d/i8/u8）× 4 分量数 × 2 精度 = 24 | **24** | **✗ 差 40** |
| 可选 ~160 | 其余 9 家族 × 4 分量数 × 2 精度（中/高）= 72 | **72** | **✗ 差 88** |

32 + 24 + 72 = 128 ≠ 256。实际上是 32 + 96 + 128 = 256 才成立，但无论哪种分配方式，当前表格的三个数值都不能同时成立。

**严重程度**：一般

**改进建议**：
统一修正三个级别的数值使其与实际计算一致。建议：① 重新审视分级粒度（例如将"常用"扩至 6-8 个家族使其达到 ~64）；② 或修正表格中的声称数值使其与实际计算一致（必备 32 + 常用 24 + 可选 200 = 256）；③ 若"~"表示近似值，偏差不应超过 50%，建议给出精确值而非近似值。

## 问题 4（一般）— 缺失 `<`/`<=`/`>`/`>=` 逐分量比较运算符的设计

**所在位置**：§4.5（比较运算）

**问题描述**：
C++ GLM 为向量类型定义了完整的逐分量比较运算符集，包括 `<`、`<=`、`>`、`>=`，每个运算符接收 `VecN<T, Q>` 返回 `VecN<Bool, Q>`。这些运算符均在仓颉可重载运算符列表中。但本设计文档仅在 §4.5 中讨论了 `==` 和 `!=`，对 `<`、`<=`、`>`、`>=` 完全未提及。

编码阶段将无法回答以下问题：
- 是否支持 `vecA < vecB`？
- 若支持，返回类型是 `VecN<Bool, Q>`（逐分量）还是 `Bool`（归约）？
- 若不支持，是否提供具名函数替代？

**严重程度**：一般

**改进建议**：
在 §4.5 中增加 `<`、`<=`、`>`、`>=` 的设计说明。建议明确：① 逐分量比较，返回 `VecN<Bool, Q>`（与 C++ GLM 一致）；② 在 Vec 结构体 extend 块中定义为扩展成员函数；③ 使用精确比较（不适用浮点容差路径，因为比较结果是布尔向量而非布尔标量）；④ 若首轮不支持，应明确排除理由并在决策中记录。

## 问题 5（一般）— `<<` 移位量类型设计未评估 `Int64` 路径

**所在位置**：§4.4（位运算，第 323-324 行）

**问题描述**：
接问题 1 的事实错误。§4.4 的移位量类型仅考虑了 `T`（与分量类型相同），未评估使用 `Int64`（或固定整数类型）的替代方案。使用 `T` 作为移位量在 C++ GLM 中没有对应——C++ GLM 的 `operator<<(vec, int)` 统一使用 `int` 移位量。当 `T = UInt8` 时，`<<(rhs: UInt8)` 的移位量范围为 0-255，而 C++ GLM 期望 `int` 范围的移位量。

**严重程度**：一般

**改进建议**：
在 D6 或 §4.4 中增加设计决策讨论：选择 `<<(rhs: Int64)` 还是 `<<(rhs: T)`？前者与 C++ GLM 一致但与非整数 `T` 的交互有差异，后者更统一但与 C++ GLM 的行为在移位量类型上存在语义差异。明确选型理由并记录。

## 问题 6（轻微）— `Vec<Bool, Q>` 的完整行为约定分布过于分散

**所在位置**：§3.2（第 119 行）、§4.3（第 310 行）、§4.4（第 319-321 行）、§4.5（第 329 行）、§4.6（第 344 行）、D5（第 414 行）等多处

**问题描述**：
`Vec<Bool, Q>` 在 D5 中被允许作为合法的泛型实例化（宽松约束策略），但其行为约定分散在 5 个以上的独立段落中：

| 操作 | Bool 上行为 | 说明位置 |
|------|-----------|---------|
| `%` | 编译错误 | §4.3 第 310 行 |
| `increment()`/`decrement()` | 编译错误 | §4.3 第 310 行 |
| `bitwiseNot()` | 通过 `!` 逻辑非（语义正确） | §4.4 第 319 行 |
| `%` 未标注 | 编译错误（`%` 仅整数） | §4.6 第 344 行 |
| `logicalAnd`/`logicalOr`/`!` | 布尔运算 | §4.5 第 329 行 |
| `<<`/`>>` | 未明确 | §4.4 |
| 算术运算符 `+`/`-`/`*`/`/` | 未明确（预期编译错误） | 未说明 |

编码人员在编码 Vec 类型时需要频繁往返查找多个章节才能确定 `Vec<Bool, Q>` 的完整行为契约。缺少统一的行为矩阵。

**严重程度**：轻微

**改进建议**：
在 §3.2 或新增的 §3.9 中增加一个 `Vec<Bool, Q>` 行为矩阵，以表格形式列出 Bool 分量类型下各运算符/函数的可用性（✓ 正常编译 / ✗ 编译错误 / ⚠️ 语义差异），使编码人员能一次性获取完整契约。

## 问题 7（轻微）— D6 配置常量 `GLM_CONFIG_CLIP_CONTROL` 的类型选择

**所在位置**：§7 D6（第 434 行）

**问题描述**：
D6 表格中将 `GLM_CONFIG_CLIP_CONTROL` 的仓颉类型声明为 `Int`。仓颉中 `Int` 是 `Int64` 的类型别名。但 C++ GLM 中 `GLM_CLIP_CONTROL` 的值为 0x00~0x03 的位组合，使用 `Int64` 既浪费位宽又不精确。应使用 `Int32` 或 `UInt8` 以精确匹配值域。

**严重程度**：轻微

**改进建议**：
将 `GLM_CONFIG_CLIP_CONTROL` 的类型由 `Int` 改为 `Int32`（或 `UInt8`），与 C++ `int` 类型的位组合语义一致。其余值域为小整数的配置常量（如 `GLM_CONFIG_SWIZZLE`、`GLM_CONFIG_CTOR_INIT`、`GLM_CONFIG_LENGTH_TYPE`、`GLM_CONFIG_PRECISION_*`）一并审查类型选择的合理性。
