## 逐维度审查

### 1. 类型系统可行性

**[通过]** Qualifier 体系采用接口 + 空结构体实现，与仓颉泛型系统兼容（`where Q <: Qualifier`）。

**[通过]** Vec1~Vec4 作为独立泛型结构体，避开仓颉不支持偏特化的限制。

**[通过]** 运算符重载清单与仓颉可重载运算符列表一致——`&&`、`||`、`~`、`++`、`--` 标记为不可重载、改用具名函数，判断正确。

**[通过]** `const init` + `const` 实例成员函数模式在仓颉中受支持：struct 必须有 `const init` 才能定义 `const` 实例成员函数（const README §3.2 规则 9），`const init` 内可使用 `=` 对 `var` 成员赋值（const README §4.2 规则）。

**[通过]** `mut operator func []` 赋值形式正确——struct 的非 `mut` 实例成员函数不能修改实例成员。

**[通过]** `public import` 重导出机制受仓颉包系统支持，`lib.cj` 正确封装了公共 API 面。

**[严重]** Hashable 实现策略不可行。设计声称整数/Bool 分量 Vec 使用 `@Derive[Hashable]`、浮点分量 Vec 不实现 `Hashable`，且 §12.2 验收标准要求"浮点分量 Vec 在哈希上下文中产生编译错误"。但 `@Derive[Hashable]` 在泛型结构体上对字段类型已实现 `Hashable` 的所有实例化均生效——`Float32`/`Float64` 是仓颉核心类型，已实现 `Hashable` 接口（`std.core` §6 核心接口表），因此 `VecN<Float32, Q>` 和 `VecN<Float64, Q>` 也会自动获得 `Hashable` 实现，无法产生所期望的编译错误。这是本质性的设计-实施断层：仓颉的类型系统不提供"类型参数不为某类型时派生"的条件派生能力。

**[通过]** 包结构（`glm.detail` 内部包 / `glm` 公共 API 包）符合 cjpm 项目组织方式，`glm` 单向依赖 `glm.detail`。

## 修改要求

### 严重问题 1 — Hashable 实现策略不可行

- **问题**：设计在 §3.2 和 §4.7 中提出"整数/Bool 分量 Vec 使用 `@Derive[Hashable]`，浮点分量 Vec 不实现 `Hashable`"的策略，且 §12.2 验收标准要求"浮点分量 Vec 在哈希上下文中产生编译错误"。但 `@Derive[Hashable]` 在泛型结构体上对字段类型已实现 `Hashable` 的所有实例化均生效。`Float32`/`Float64` 是仓颉核心类型，已实现 `Hashable` 接口，因此 `VecN<Float32, Q>` 和 `VecN<Float64, Q>` 也会自动获得 `Hashable` 实现，不产生编译错误。

- **原因**：此问题影响设计的可行性——验收标准中明确要求的"浮点 Vec 编译错误"在当前方案下无法实现。编码阶段若按此方案实施将直接验收失败。仓颉的 `@Derive` 宏和泛型约束系统均不支持"除浮点类型外实现 Hashable"这种否定约束。

- **建议方向**（以下三选一）：
  1. **全量实现 Hashable**（推荐）：对所有 Vec 类型（含浮点）统一实现 `Hashable`。可通过 `@Derive[Hashable]` 直接标注于泛型结构体——仓颉 `@Derive` 自动为所有字段类型已实现 `Hashable` 的实例化生成 `hashCode()`（此条件对所有数值类型和 `Bool` 均满足）。修订 §12.2 相关验收标准为"整数/Bool 分量 Vec 可正常用作 `HashSet`/`HashMap` 键且哈希一致性正确；浮点分量 Vec 也可用于哈希场景，但 `NaN` 分量不满足 `Hashable` 相等性一致性要求，属于已知限制需在文档中标注"。需要在 §4.7 或 §3.2 同步更新文档措辞。
  2. **统一不实现 Hashable**：所有 Vec 类型均不实现 `Hashable`。移除 §3.2 和 §4.7 中的 Hashable 相关内容。验收标准改为"所有 Vec 类型在哈希上下文中产生编译错误"。调用方需要在哈希集合/映射中使用 Vec 时自行包装为自定义新类型。
  3. **手动条件扩展**：不使用 `@Derive[Hashable]`，改为针对非浮点 Vec 的手动扩展实现。但此方案无法利用仓颉的条件接口扩展（`extend ... <: Hashable where T <: Hashable` 无法排除 `Float32`），需逐一为每个非浮点类型组合编写扩展块，工程代价高，不推荐。
