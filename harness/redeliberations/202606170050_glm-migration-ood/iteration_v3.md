## 需要解决的问题（仅 RETRY 时存在）

- **问题描述**：`mut operator func +=` 的显式定义在仓颉语言中不可实现（复合赋值由二元运算符自动生成），导致 §4.6 溢出标注策略和"copy + op="委托模式产生自递归，编码阻塞。
- **所在位置**：§4.6、§3.2
- **严重程度**：严重
- **改进建议**：重构溢出标注策略：在二元运算符（+、-、*、/）上直接标注 @OverflowWrapping，逐分量实现运算；复合赋值由编译器自动生成，方向改为 += 自动继承自 + 而非 + 委托给 +=。

- **问题描述**：设计文档引用了未定义的类型 `Vec<Bool, Q>`，但实际设计中只有 `Vec1<T,Q>`、`Vec2<T,Q>`、`Vec3<T,Q>`、`Vec4<T,Q>` 四个独立结构体，编码阶段无法根据此签名生成代码。
- **所在位置**：§4.5
- **严重程度**：严重
- **改进建议**：将所有 `Vec<Bool, Q>` 引用替换为每个 VecN 类型的具体版本，或统一表述为"各 VecN 类型分别定义对应版本"。

- **问题描述**：泛型 Vec 的 `public init()` 默认构造函数无法编译——成员 `var x: T` 无初始值导致仓颉编译器不会自动生成无参构造函数，而显式定义的 `init() { this.x = T(0) }` 对非数值类型（如 Bool）不合法。
- **所在位置**：§3.2、§4.1
- **严重程度**：高
- **改进建议**：统一 §3.2 成员声明与 §4.1 构造函数清单：为成员添加泛型默认值并自定义 Default 接口（需评估与 D5 冲突），或移除 `public init()` 默认构造，或仅在明确支持的类型族上通过约束提供。

- **问题描述**：`logicalAnd`/`logicalOr` 从 C++ 运算符（`&&`/`||`）改为具名函数，未评估语法变更对下游 GLM 代码迁移成本的影响。
- **所在位置**：§4.5
- **严重程度**：高
- **改进建议**：增加一节评估将 `&&`/`||` 改为具名函数对迁移工作量的影响，给出代码迁移模式（搜索-替换规则或 wrapper 函数）。

- **问题描述**：P4（跨类型复合赋值未明确承认）、P6（Functor/ComputeVec 无消费者）、P7（`operator[]` 签名未完整）、P8（§3.6 与 §5.1 断言行为矛盾）、P9（`%` 运算符浮点类型编译错误）、P10（`ComputeEqual` 的 `IsFloat` 参数映射问题）、N2（`countof` 签名 `$` 前缀缺失）等中等级别问题需一并处理。
- **所在位置**：§3.2~§5.1 多处
- **严重程度**：一般
- **改进建议**：逐条按诊断报告中的改进建议进行修复。
