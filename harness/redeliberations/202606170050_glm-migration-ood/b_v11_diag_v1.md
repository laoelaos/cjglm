# 质量审查报告 — GLM 仓颉迁移首轮 OOD 设计方案 v11

审查时间：第 11 轮 | 审查视角：需求响应充分度、完整性、深度

---

## 问题清单

### P1（严重）：256 个别名的命名约定未明确定义

- **问题描述**：§3.8 列举了 16 个别名家族（`bvec/ivec/uvec/vec/dvec/i8vec/i16vec/i32vec/i64vec/u8vec/u16vec/u32vec/u64vec/fvec/f32vec/f64vec`），这些名称使用全小写 + 无分隔符的 GLM 原生风格。但同一节中的示例格式使用 `Vec2Float32`、`LowpVec2Float32`、`IVec2` 等大写驼峰风格。两套命名风格存在冲突，文档未明确约定 256 个别名的实际命名规则。编码人员将面临以下未回答的问题：
  - 最终别名是 `bvec2`（小写）还是 `BVec2`（驼峰）？
  - `i8vec2` 的精度变体是 `lowp_i8vec2` 还是 `LowpI8Vec2`？
  - 精度前缀的约定是下划线连接（`lowp_bvec2`）还是驼峰连接（`LowpBVec2`）？
  - §3.8 列出的"别名家族"描述与编码阶段需要编写的 `type` 语句之间缺乏明确的映射规则。
- **所在位置**：§3.8（向量类型别名），第 291-303 行
- **严重程度**：严重 — 直接阻塞编码阶段的第一项任务（在 `fwd.cj` 中书写 256 个 `type` 别名）
- **改进建议**：在 §3.8 中增加一个明确的命名约定子节，约定以下规则：
  1. 统一采用 GLM 原生小写风格（如 `bvec2`、`ivec3`、`dvec4`），或统一采用大写驼峰风格（如 `BVec2`、`DVec4`），必须选择其一；
  2. 精度变体前缀明确约定：使用下划线（`lowp_bvec2`/`highp_bvec2`）还是前缀拼接（`lowpBVec2`/`highpBVec2`）；
  3. 可选：提供一份 `type` 别名实际格式的示例表，覆盖所有 4 个精度变体。

### P2（高）：`scalar op vec` 具名函数的定义位置和形态未指定

- **问题描述**：§4.3 为 `scalar + vec`、`scalar - vec`、`scalar / vec` 等左标量右向量运算提供了具名函数替代方案（如 `add(s: T, v: VecN<T,Q>)`、`div(s: T, v: VecN<T,Q>)` 等），但未说明这些函数：
  1. 定义在哪个包/文件中（`glm` 包、`glm.detail` 包的某个文件、还是独立的文件）；
  2. 以什么形态定义（扩展成员函数、包级独立函数、extension 块）；
  3. 调用方需要如何导入才能使用这些函数（`import glm.*` 即可，还是需要指定具体模块）；
  4. 是否为每个 Vec 类型（Vec1~Vec4）分别定义，还是通过某种模式统一提供。
- **所在位置**：§4.3（算术运算），第 455-460 行
- **严重程度**：高 — 编码人员无法确定这些函数的实现位置和组织方式，可能导致实现结构不一致或与后续模块组织冲突
- **改进建议**：在 §4.3 或 §2（模块划分）中补充：
  1. 选择文件归属（建议在 `glm.detail` 包内增加 `scalar_ops_vecN.cj` 或归入各 `type_vecN.cj` 的 extend 块）；
  2. 明确是扩展成员函数还是独立函数（扩展成员函数支持 `v.add(1.0)` 但不支持 `add(1.0, v)`，独立函数支持两种顺序但需额外参数顺序区分）；
  3. 单向扩展成员函数+双向独立函数的组合方案需在文档中明确分层。

### P3（高）：Aligned* Qualifier 类型的可见性策略未明确

- **问题描述**：§3.1 定义了 `AlignedHighp`/`AlignedMediump`/`AlignedLowp` 三种对齐类型。§2 中 `lib.cj` 的 `public import` 列表仅导出 `PackedHighp`/`PackedMediump`/`PackedLowp`/`Defaultp`，不包含 Aligned 系列。但文档未说明在 `qualifier.cj` 中 Aligned 结构体是声明为 `public` 还是包级可见（`internal`，即无 `public` 修饰符）。两种选择的后果差异显著：
  - 若为 `public`：下游可通过 `import glm.detail.AlignedHighp` 间接访问，但设计文档未约定其行为和使用规则，存在未定义行为风险；
  - 若为 `internal`：下游无法访问，但未来轮次切换 `GLM_CONFIG_ALIGNED_GENTYPES = true` 时需要修改可见性。
- **所在位置**：§3.1（Qualifier 体系）、§2（模块划分 `lib.cj` 导出列表）
- **严重程度**：高 — 影响下游 API 边界和行为约定
- **改进建议**：在 §3.1 或 §7 中新增一条设计决策，明确：
  1. Aligned 系列在首轮的可见性（建议设为 `internal`，因为 `GLM_CONFIG_ALIGNED_GENTYPES = false`）；
  2. 后续轮次切换配置时需将相关类型改为 `public` 并补充到 `lib.cj` 的导出列表；
  3. 或者采用中间方案：所有 Qualifier 实现类型保持 `public`，但文档说明首轮仅 Packed 系列和 Defaultp 属于公共 API，Aligned 系列需等待对齐配置启用后使用。

### P4（中）：`compute_vector_relational.cj` 的依赖声明包含不必要的 `shim_cstddef.cj`

- **问题描述**：§2 模块依赖图和 §8.1 迁移文件清单均声明 `compute_vector_relational.cj` 依赖 `shim_cstddef.cj` + `shim_limits.cj`。但简化后的 `ComputeEqual<T>` 在首轮设计中仅使用 `isIec559Of<T>()`（来自 `shim_limits.cj`）和 `epsilonOf<T>()`（来自 `NumericLimits<T>`，也在 `shim_limits.cj`），未引用 `SizeT` 或 `LengthT`。该依赖声明是历史残留，编码人员若按此依赖图实现会引入不必要的 `import shim_cstddef` 语句。
- **所在位置**：§2（模块间依赖，第 85 行）、§8.1（迁移文件清单，文件 6 `compute_vector_relational.cj`）
- **严重程度**：中 — 不会阻断编码，但引入无意义的 import 会降低代码质量和维护性
- **改进建议**：将 `compute_vector_relational.cj` 的依赖修正为仅 `shim_limits.cj`，移除 `shim_cstddef.cj`。

### P5（中）：`LengthT` 类型别名在 Vec 类型中未被使用

- **问题描述**：§3.6 在 `shim_cstddef.cj` 中定义了 `type LengthT = Int64`，用途说明为"索引/长度场景"。但 Vec 类型的 `length()` 静态方法返回 `Int64`（第 145 行），`operator[]` 索引参数类型也直接使用 `Int64`（第 442-443 行），均未使用 `LengthT` 别名。若 `LengthT` 的目的是作为抽象长度类型供 Vec 使用，则 Vec 应引用它以避免未来类型变更时的连锁修改。若 `LengthT` 仅为提供命名别名（供下游代码使用），则应与 Vec 的具体关系在文档中明确。
- **所在位置**：§3.6（Shim 层，第 277 行）、§3.2（Vec 结构体系，第 145 行）、§4.2（分量访问，第 442-449 行）
- **严重程度**：中 — 不影响功能，但反映抽象一致性不足
- **改进建议**：两种解决方案择一：
  1. 将 Vec 的 `length()` 返回类型和 `operator[]` 索引类型改为 `LengthT`，使别名落到实处；
  2. 在 §3.6 中说明 `LengthT` 是通用长度类型别名，供下游消费者在泛型场景中使用，Vec 内部保持 `Int64`。

### P6（中）：浮点 Vec 的 IEEE 754 精确相等比较无可用入口

- **问题描述**：`==` 运算符通过 `ComputeEqual.callConst` 实现，对浮点分量使用 Epsilon 容差比较（`abs(a - b) <= epsilonOf<T>()`）。文档 §9.3 已正确指出该方案在 `Infinity == Infinity` 场景下的局限性（容差路径下 `Inf - Inf = NaN` → `NaN <= epsilon = false`，错误返回 `false`）。但设计未提供让调用方选择"容差比较"还是"精确比较"的机制——前者是 `==` 的固定行为，后者需要一个备用的具名函数入口。文档提到 `equalEpsilon` 具名函数但不属于首轮范围（文档原文："调用方可通过 `equalEpsilon` 具名函数获取更精确的控制，不属于首轮范围"），这意味着首轮中调用方既无法使用 `==` 获得 Infinities 的 IEEE 754 正确结果，也无替代 API。
- **所在位置**：§4.5（比较运算）、§9.3（NaN/Infinity 传播）
- **严重程度**：中 — 影响涉及浮点 Infinity 相等判断的下游代码
- **改进建议**：
  1. 在 §4.5 或 §4 中增加一个非 `const` 的 `equalExact`（或 `equalIEEE`）具名函数，使用精确比较（`==`）而非容差比较，作为 `==` 运算符的补充。该函数实现简单（逐分量 `a == b`），无需依赖 `const if` 路径，与 §3.5 备选路径一致。归入各 Vec 的 extend 块或独立文件中。
  2. 这将使 §9.3 描述的 Infinity 比较局限性从"无解决方案"变为"有明确替代入口"。

### P7（低）：默认字符串表示的行为约定过于模糊

- **问题描述**：§3.2 约定 Vec 类型"不实现自定义 `toString()` 或 `Format`/`Display`"，后续补充描述为"编译器可能自动生成调试格式输出（`Vec2(1, 2)` 形式）"。使用了"可能""预期"等推测性措辞。对于一份直接指导编码的设计文档，应明确指定字符串表示是"undefined（不可依赖）"还是"编译器的默认 struct 输出（格式不确定但稳定）"，而非给出推测性的示例。
- **所在位置**：§3.2（Vec 结构体系，第 173-176 行）
- **严重程度**：低 — 不影响核心功能，但影响 API 契约的精确性
- **改进建议**：将模糊措辞替换为精确声明："首轮依赖仓颉编译器为 struct 生成的默认字符串表示——格式和内容由编译器确定，调用方不应依赖特定格式"，或选择实现自定义 `toString()`（如 `"Vec2(1.0, 2.0)"`）以提供确定的行为。

## 需求响应充分度评估

产出对 requirement.md 定义的三个要求均有覆盖：
- **模块结构**：§2 完整定义了包组织、文件划分和依赖拓扑
- **类/接口设计**：§3 完整定义了 Qualifier 体系、Vec 结构体系、Functor 体系、ComputeVec 体系和 Shim 层
- **关键数据类型**：§3.7-3.8 定义了标量和向量类型别名
- **可直接编码**：§4 提供了完整的行为契约（构造、访问、运算、错误处理、哈希），§12 提供了验收标准

**未充分响应的方面**：命名约定的精确性不足（P1）、函数定义位置未指定（P2）、API 边界未明确（P3）。这些问题虽不影响架构覆盖度，但直接影响"设计产物需可供后续编码阶段直接使用"这一核心要求。

## 整体评价

文档经过 10 轮内部审议后，在架构合理性、技术可行性和细节覆盖度上已达到较高水平。本审查发现的 7 个问题主要集中在**编码就绪度**层面——约定了"做什么"但未精确约定"以何种具体形式写代码"。P1-P3 需要在编码启动前明确约定，P4-P7 可在编码阶段作为技术债记录处理。
