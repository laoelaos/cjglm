## 当前审查结果

### 一、事实错误 / 语言特性误用

**P1. `&&` / `||` 运算符不可重载**（严重）— §3.2、§4.5 声明在 Vec 上提供 `&&` 和 `||` 运算符重载，但仓颉中 `&&` 和 `||` 不在可重载运算符列表内。改进建议：移除 `&&` 和 `||` 的运算符重载设计，改用具名函数（如 `logicalAnd`/`logicalOr`）或位运算符（`&`/`|`）替代。

**P2. `~` 运算符不可重载**（严重）— §3.2、§4.4 声明位运算符包含 `~`（单目按位取反），但仓颉中 `~` 不在可重载运算符列表内。改进建议：移除 `~` 运算符重载，按位取反应通过具名函数（如 `bitwiseNot()`）实现。

**P3. 跨 Vec 转换构造的泛型约束 `where T2 <: T` 不适用于数值类型**（严重）— §4.1 构造器 `init(v: Vec2<T2, Q2>) where T2 <: T, Q2 <: Q` 中，仓颉原始数值类型间不存在子类型关系，`Int8` 不是 `Int32` 的子类型，因此跨类型转换无法编译通过。改进建议：不做约束通过 `T(v)` 显式转换，或定义 `ConvertibleTo<T>` 接口。

**P4. `const if` 在非 `const` 函数中不可用 — `==` 运算符需要 `const init`**（严重）— §3.5 的 `==` 运算符体内使用 `const if (NumericLimits<T>.isIec559)`，但 `const if` 仅在 `const` 函数上下文生效，而 Vec 结构体未设计 `const init` 构造函数。改进建议：为 Vec 添加 `const init` 构造函数使比较运算符可声明为 `const` 函数，或改用运行时 `if` 分支。

### 二、逻辑矛盾 / 设计不一致

**P5. `@OverflowWrapping` 标注策略内部矛盾**（严重）— §4.6 存在三处相互矛盾的表述：①开头将二元运算符列为"直接标注"；②中间又说"二元运算符委托复合赋值，复合赋值已在被委托方标注"；③末尾的"替代方案"又说"若自动生成版本优先则二元运算符也直接标注"。这与 roadmap §0.5.1 中"二元算术运算符——否（委托给复合赋值，无需额外标注）"相悖。改进建议：做出明确唯一的决策——若显式定义优先则二元运算符不标注（与 roadmap 一致），若自动生成优先则均需标注且修改 Vec 定义方式。

**P6. `scalar / vec` 行为描述与不存在运算符矛盾**（一般）— §4.3 明确 `scalar + vec` 不可作为运算符实现，但 §9.2 表格中出现 "`0 / vec`" 的运算符语法。改进建议：统一表述——若通过具名函数实现则改为 `div(0, vec)`，若不支持则删除该行。

### 三、关键遗漏 / 深度不足

**P7. 缺少 GLM_CONFIG_* 常量的完整清单与值**（一般）— §7 D6 仅提及"18 个 `GLM_CONFIG_*` 常量集中在 `setup.cj` 声明"和"首轮所有配置取最保守值"，但未给出完整的常量清单和对应的保守值，编码人员无法知道 `setup.cj` 中要声明哪些常量。改进建议：在 D6 或单独一节中列出 18 个常量的完整清单（常量名、仓颉等效名、首轮值、简要说明）。

**P8. `%` 运算符在浮点 Vec 上的可用性未做分析**（一般）— §3.2、§4.3 将 `%` 作为 Vec 通用算术运算符，但仓颉中 `%` 仅对整数类型有定义，对浮点类型无 `%` 运算符。改进建议：明确 `%` 仅在 `T` 为整数类型时可用，添加 `where T <: Integer<T>` 约束。

**P9. `countof<T>(arr: T): Int64` 签名无法实现其功能**（一般）— §3.6 的签名中 `T` 无约束，函数体无法知道"数组长度"。改进建议：修正签名为 `func countof<T, N>(arr: VArray<T, N>): Int64`，通过类型参数 `N` 获取编译期长度。

**P10. 缺少 `uint` 标量类型别名**（轻微）— §3.7 标量类型别名映射规则中未提及 `uint`。改进建议：补充 `type uint = UInt32`。

**P11. `operator[]` 错误处理与 C++ 语义差异未说明**（轻微）— §5.1 说"下标越界时调用 `assert()` 等效的 `if + throw` 模式"，但 C++ `GLM_ASSERT_LENGTH` 在 Release 模式下被编译掉，而仓颉的 `if + throw` 无法消除。改进建议：补充差异说明，建议 Debug/Release 配置区分处理。
