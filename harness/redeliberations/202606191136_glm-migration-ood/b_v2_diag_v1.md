## 问题清单

### 问题 1：标量别名计数不一致（需要澄清的轻微事实错误）

- **问题描述**：§3.7 正文（第 461 行）声称"覆盖 10 个原生类型 + uint + 4 精度变体 = **44** 个别名"，但以下三处证据显示实际数量为 **41**：
  1. §3.7 表格中 uint 行仅有 1 个别名条目，其余三个精度列（highp_/mediump_/lowp_）标记为"—"，说明该表格隐含 uint 仅有 1 个别名（public type uint = UInt32），无精度变体。
  2. §3.8 的 Python 脚本模板（第 586-618 行）通过 SCALAR_MAP（10 个类型 × 4 精度变体 = 40 个）加硬编码 public type uint = UInt32（1 个）共生成 **41** 个标量别名，未生成 highp_uint/mediump_uint/lowp_uint。
  3. 表格结构统计：10 行数据（每行 4 列 = 40）+ 1 行 uint（1 列）= **41**。
- **所在位置**：§3.7 第 461 行正文 + 第 463-475 行表格 + §3.8 第 612-618 行脚本
- **严重程度**：轻微
- **改进建议**：决议二选一：1) 若 highp_uint/mediump_uint/lowp_uint 是预期的有效别名，则修正表格将 uint 行中三个"—"替换为别名定义 public type highp_uint = UInt32 等，并更新 Python 脚本在 uint 硬编码行之后增加精度变体生成循环，使计数与 44 一致；2) 若 uint 确无精度变体，则将第 461 行的"44"修正为"41"，保持正文、表格、脚本三者一致。建议采用方案(1)保持与 GLM C++ 中 highp_uint 等命名惯例的一致性。

### 问题 2：equalEpsilon 代码示例缺少对路径前提的显式标注（改进建议）

- **问题描述**：§4.5 的 equalEpsilon 代码示例（第 984-993 行）直接引用了 epsilonOf<T>()，但未在代码注释中显式标注此实现假设的是 §3.5 的**路径 A**（保留 epsilonOf<T>()）。代码示例下方的说明文本（第 995 行及之后）确实讨论了路径 A/B 的区别，但若实施者在编码阶段只参考代码示例而忽略周围的说明段落，可能误以为此代码是唯一确定方案，从而在触发回退（路径 B）时遗漏联动修改。
- **所在位置**：§4.5 第 984-993 行代码示例
- **严重程度**：轻微
- **改进建议**：在代码示例中添加行内注释，使代码示例本身自包含路径前提信息，降低联动修改遗漏风险。

### 问题 3：编译器依赖验证清单未区分"已确认"和"待验证"的执行时序（观察建议）

- **问题描述**：§2.1 编码前验证检查清单（第 199-215 行）共 14 项验证项，虽然各项标注了类型（独立原型验证/统一集成验证/构建工具验证）并给出了验证时序建议（推荐优先执行第 1/5 项），但未显式区分 D37 中标注的"可确认"项（由语言文档已声明的行为）与"待验证"项（需编码前原型验证）。实施者可能对优先级相同、但确认成本差异很大的项一视同仁，浪费时间去验证语言文档已声明的行为。
- **所在位置**：§2.1 第 195-215 行
- **严重程度**：轻微
- **改进建议**：在 §2.1 检查清单中添加一列"确认类型"（文献确认 / 原型验证），或至少在清单头部添加说明，告知实施者 D37 中标记为"可确认"的项（A08、A09、A10、A11）无需原型验证，可跳过或仅做快速语法确认。将有限的原型验证资源集中于真正需要编译实验的"待验证"项。
