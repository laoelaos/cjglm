# 代码审查报告（v2 r1）

## 审查结果
REJECTED

## 发现

- **[严重]** `src/detail/type_vec1.cj` — 详细设计 v2 要求追加 4 个跨类型构造函数（Vec1→Vec1、Vec2→Vec1、Vec3→Vec1、Vec4→Vec1），但实际源文件中 0 个被实现。实现完全偏离设计规格。

- **[严重]** `src/detail/type_vec1.cj` (设计) — 设计 v2 指定的 `<T2, Q2>` 泛型构造函数在仓颉语言中不被支持。现有 Vec2/Vec3/Vec4 中所有跨类型构造均使用 struct 自身的 `<T, Q>` 参数（`init(v: Vec1<T, Q>)`），未引入独立类型参数。此模式不可用于非同 T 的类型转换。

- **[严重]** `src/detail/type_vec1.cj` (设计) — 设计 v2 中的 `T(v.x)` 语法将类型参数 `T` 当作构造函数调用，仓颉编译器不支持此语法。类型参数不能被用作值转换函数。

## 修改要求（REJECTED）

### 问题 1：实现完全偏离设计
- **位置**：`src/detail/type_vec1.cj:10-12`（`const init(x: T)` 之后）
- **问题**：设计要求在 `const init(x: T)` 之后、`length()` 之前插入 4 个跨类型构造函数，但实际源文件中这 4 个构造函数均不存在。
- **为什么是问题**：产出与设计要求完全不一致，代码未按规格完成。
- **期望的修正方向**：需要先修正设计本身（见问题 2、3），再基于可行方案重新实现。

### 问题 2：泛型构造函数携带独立类型参数在仓颉中非法
- **位置**：详细设计 v2 第 22-36 行（构造函数签名）
- **问题**：`public init<T2, Q2>(v: Vec1<T2, Q2>)` 为 `init` 构造函数引入了独立于 struct 自身的类型参数 `<T2, Q2>`，仓颉编译器不支持此语法（"generic constructor 'init' is not supported"）。
- **为什么是问题**：这是仓颉语言的硬性限制。所有 `init` 构造函数的参数类型必须基于 struct 已有的类型参数。现有 Vec2/Vec3/Vec4 中的 `init(v: Vec1<T, Q>)` 已充分证明此规则。
- **期望的修正方向**：废弃在 `init` 构造函数中引入独立类型参数的设计思路。改用顶层泛型函数（如 `toVec1`/`cast` 系列）或 `fromBoolVec` 模式的工厂函数。

### 问题 3：类型参数作为构造函数调用非法
- **位置**：详细设计 v2 第 23 行（`this.x = T(v.x)`）
- **问题**：`T(v.x)` 将类型参数 `T` 当作构造/转换函数调用，仓颉编译器不支持此语法（"'()' is not a static member of exposed generic parameter 'T'"）。
- **为什么是问题**：仓颉语言中类型参数不能被当作构造函数调用以执行值转换。这是编译器层面的语法限制。
- **期望的修正方向**：改用显式类型转换或对每种支持的 `T` 和 `T2` 组合提供特化实现；或使用顶层泛型函数配合 `T <: Number<T>` 约束提供统一的转换方法。
