## 设计偏差说明

详细设计 v2 存在两个仓颉语言层面不可逾越的编译错误，导致 4 个构造函数无一可产出：

### 错误 1：泛型构造函数不支持（generic constructor 'init' is not supported）

```cangjie
public init<T2, Q2>(v: Vec1<T2, Q2>) where Q2 <: Qualifier {
```

仓颉语言的 struct `init` 构造函数**不允许拥有自己独立的类型参数**（与 struct 自身的 `<T, Q>` 不同）。所有 `init` 的参数类型必须基于 struct 已有的类型参数。

验证依据：`type_vec2.cj:21`、`type_vec3.cj:24`、`type_vec4.cj:27` 中已有的 `init(v: Vec1<T, Q>)` 均使用 struct 自身的 `T` 和 `Q`，未引入新类型参数。这是仓颉语言的硬性限制。

### 错误 2：类型参数不可作为构造函数调用（'()' is not a static member of exposed generic parameter 'T'）

```cangjie
this.x = T(v.x)
```

`T(v.x)` 将类型参数 `T` 当作构造/转换函数调用，仓颉编译器不支持此语法。类型参数不能被当作构造函数来转换值。

### 影响范围

两个错误独立存在，任何一个都足以阻止构建。4 个构造函数无一幸免。

### 前置条件

**设计 v3 需要先解决两个核心问题**后方可进入编码：
1. 选择 API 形态：顶层泛型函数（如 `toVec1`/`cast`）或 extend 方法 + 转换闭包参数
2. 确定跨类型值转换策略：如何在不同数值类型间转换 `x` 分量值（例如 `Float32`→`Int64`、`Int32`→`Float64` 等）
