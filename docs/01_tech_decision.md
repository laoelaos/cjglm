# GLM 1.0.3 迁移基线分析与技术决策依据

> **文档定位**：本文档仅保留 GLM 1.0.3 向仓颉迁移所需的基线分析、依赖事实与关键技术决策，不再承担范围界定、轮次规划或执行路线图职责。
> **使用方式**：实施范围、阶段安排与任务拆分应在独立执行文档中维护；本文档仅作为后续设计与实现时的引用依据。

---

## 1. 语言适配基线结论

本节只保留对迁移设计有直接约束作用的语言结论。

### 1.1 泛型与类型参数

| 能力 | CangJie 支持情况 | 迁移约束结论 |
|------|-----------------|-------------|
| 泛型 struct/class | ✅ 支持 | 可定义 `Vec1<T, Q>` 等泛型结构体 |
| 泛型约束 | ✅ 支持 `where T <: SomeInterface` | 可用接口约束 qualifier 类型参数 |
| 偏特化 / 全特化 | ❌ 不支持 | 不能直接复用 `vec<N, T, Q>` 的 C++ 模板特化模型 |
| 模板模板参数 | ❌ 无对应机制 | `_vectorize.hpp` 一类模板工具不能按原型直接迁移 |

**结论**：

1. C++ `vec<L, T, qualifier Q>` 不应直接映射为单一带分量数参数的仓颉泛型。
2. 首选建模方式是按分量数拆分为 `Vec1<T, Q>`、`Vec2<T, Q>`、`Vec3<T, Q>`、`Vec4<T, Q>` 四个独立泛型结构体。
3. 依赖 `vec<N, T, Q>` 偏特化或模板模板参数的辅助模板，需要改写为按 `Vec1` 到 `Vec4` 分发，或使用宏生成重复结构。

### 1.2 `qualifier` 的类型映射

C++ GLM 中的 `qualifier` 是枚举值，并作为模板实参参与类型区分。CangJie 泛型参数仅接受**类型**，不接受枚举值作为泛型实参。

**基线映射方案**：

1. `Qualifier` 定义为接口。
2. 各 qualifier 变体定义为实现该接口的空结构体，例如 `PackedHighp`、`PackedMediump`、`AlignedHighp`。
3. 向量类型以 `Q <: Qualifier` 约束 qualifier 参数。
4. 默认 qualifier 使用具象类型别名表达，例如 `type Defaultp = PackedHighp`。

**结论**：全文涉及 `Q` 的类型签名，均应理解为“类型级 qualifier”，而不是“枚举值 qualifier”。

### 1.3 运算符重载

| 能力 | CangJie 支持情况 | 迁移结论 |
|------|-----------------|---------|
| 算术运算符 `+ - * /` | ✅ 支持 | 可直接承载向量算术 |
| 比较运算符 | ✅ 支持 | 可实现向量比较 |
| 下标运算符 `[]` | ✅ 支持 | 可实现分量访问 |
| 复合赋值 `+= -= *= /=` | ✅ 支持 | 可作为算术实现主路径 |
| 自定义新运算符 | ❌ 不支持 | 不影响 GLM 核心迁移 |
| 泛型运算符重载限制 | 有限制 | 需围绕标准运算符设计，而非引入额外语法技巧 |

**结论**：Vec 类型可以保留 GLM 的核心运算接口形态，但实现上应优先围绕标准运算符与显式分量访问展开。

### 1.4 类型别名

| 能力 | CangJie 支持情况 | 迁移结论 |
|------|-----------------|---------|
| 类型别名 | ✅ 支持 | 可承载 GLM 常用别名层 |
| 泛型类型别名 | ✅ 支持 | 可用于简化常见类型映射 |
| 别名重复声明约束 | 别名上不能重复 `where` 约束 | 含 `Qualifier` 约束时应优先落到具象 qualifier 类型 |

**结论**：别名层可以保留，但更适合作为公共 API 映射，不应承担新的约束表达职责。

### 1.5 编译期条件能力

| 能力 | CangJie 支持情况 | 迁移结论 |
|------|-----------------|---------|
| `const` 编译期求值 | ✅ 支持 | 可替代一部分配置常量判断 |
| 宏元编程 | ✅ 支持 | 可用于代码生成 |
| C 风格预处理器 | ❌ 不支持 | 不能直接迁移 `#if` / `#ifdef` / 条件 `#include` |

**结论**：

1. GLM 中“函数体内部条件逻辑”可考虑映射为 `const + if` 或宏展开。
2. GLM 中“条件包含独立文件”的模式不能直接保留，必须改为固定配置、重组文件结构，或直接排除该能力。
3. swizzle、SIMD 等强依赖条件包含的能力，不能视为无损平移。

### 1.6 固定位宽数值类型与溢出语义

| 能力 | CangJie 支持情况 | 迁移结论 |
|------|-----------------|---------|
| `Int8`~`Int64` / `UInt8`~`UInt64` | ✅ 原生支持 | 标量位宽可直接映射 |
| `Float16` / `Float32` / `Float64` | ✅ 原生支持 | 浮点位宽可直接映射 |
| 整数溢出语义 | 与 C++ 不同 | 需显式决定 wrapping 或异常策略 |

**关键差异**：C++ 中整数溢出通常被视为未定义行为；CangJie 默认会进行溢出检查并抛出异常。

**基线决策**：

1. 对整数向量算术，必须显式定义溢出策略，不能默认沿用 C++ 语义假设。
2. 若目标是尽量贴近 GLM 既有行为，优先在整数复合赋值算术运算符上使用 `@OverflowWrapping`。
3. 二元算术运算符可通过“拷贝 + 复合赋值”的委托模式继承 wrapping 语义，减少重复标注。

### 1.7 不可直接等价的语言/底层能力

| GLM / C++ 机制 | CangJie 等价情况 | 结论 |
|---------------|------------------|------|
| 匿名 `union` / 匿名 `struct` | ❌ 不支持 | 四元数等布局技巧不能原样迁移 |
| `reinterpret_cast` | ❌ 无直接等价 | `_swizzle.hpp` 类实现不可直接迁移 |
| 平台 SIMD 内建类型与特化分支 | ⚠️ 无现成等价 | SIMD 路径应视为单独问题 |

**结论**：凡依赖匿名联合、reinterpret、SIMD 平台特化的实现路径，都应从“语义重设计”而不是“语法翻译”角度处理。

## 2. 标准库与基础设施替代结论

GLM 核心实现依赖若干 C++ 标准库设施。以下只保留会影响仓颉方案设计的替代结论。

| C++ 设施 | 主要用途 | 仓颉侧结论 |
|---------|---------|-----------|
| `<cassert>` | 断言 | 需以 `if + throw` 或统一断言 helper 替代 |
| `<cstddef>` | `size_t`、`ptrdiff_t`、长度/索引语义 | 需显式区分无符号大小语义与有符号长度语义 |
| `<limits>` | 数值边界与浮点性质 | 需以类型静态属性或统一查询包装替代 |
| `<functional>` | `std::plus<T>` 等函数对象 | 以泛型函数参数、Lambda 或显式函数对象替代 |

### 2.1 `size_t` / `length_t` 语义结论

GLM 默认语义里，长度类型更接近“索引/计数使用的整数”，而非所有场景都保留无符号 `size_t` 语义。

**基线结论**：

1. 与索引、长度、循环相关的核心路径，采用有符号整型更符合仓颉侧实现与安全性要求。
2. 与位运算、缓冲区大小、明确无符号语义相关的路径，应保留单独的无符号别名。
3. 因此基础设施层应明确区分 `LengthT` 与 `SizeT` 两种用途，而不是混用单一别名。

### 2.2 Shim 层结论

无论最终目录如何组织，以下三类替代能力都是核心基线的一部分：

1. 断言替代层。
2. 数值限值替代层。
3. `size_t` / `ptrdiff_t` / `length_t` 语义替代层。

**结论**：这些能力属于迁移的基础设施问题，而不是具体 Vec/Mat/Quat 功能问题。后续实现文档应将它们视为前置决策项。

## 3. GLM 类型依赖基线

本节仅说明类型层次与依赖事实，用于支撑后续范围决策，但不在本文内直接给出轮次或执行安排。

### 3.1 类型层次

| 层次 | 内容 | 依赖特征 |
|------|------|---------|
| 基础设施层 | `setup.hpp`、`qualifier.hpp`、标量别名 | 无核心类型前置依赖 |
| 向量层 | `type_vec1.hpp` ~ `type_vec4.hpp` | 仅依赖基础设施层 |
| 矩阵层 | `type_mat2x2.hpp` ~ `type_mat4x4.hpp` | 依赖向量层 |
| 四元数层 | `type_quat.hpp` | 依赖向量层与部分矩阵层 |
| 函数库层 | `common.hpp`、`geometric.hpp`、`gtc/*`、`ext/*` 等 | 依赖核心类型层 |

### 3.2 关键前向声明关系

`qualifier.hpp` 中的三类前向声明构成 GLM 核心类型系统的入口：

```cpp
template<length_t L, typename T, qualifier Q = defaultp> struct vec;
template<length_t C, length_t R, typename T, qualifier Q = defaultp> struct mat;
template<typename T, qualifier Q = defaultp> struct qua;
```

**结论**：仓颉迁移时，即使不保留原模板形态，也必须保留这三类核心类型族之间的依赖顺序：先向量，再矩阵，再四元数。

### 3.3 依赖度观察

基于原始头文件的直接引用统计，可得到以下稳定结论：

1. `setup.hpp` 与 `qualifier.hpp` 是所有核心类型与大多数函数库的共同前置依赖。
2. `vec2`、`vec3`、`vec4` 是被其他核心类型引用最频繁的类型层。
3. 矩阵类型主要建立在向量类型之上，本身不是更底层的基础。
4. 四元数直接位于依赖链更上层，且额外受矩阵与扩展头文件牵引。

**结论**：在任何执行方案中，向量层都应被视为核心类型基线，而不是附属类型。

### 3.4 `type_quat.hpp` 的附加复杂度

`type_quat.hpp` 除了依赖 `vec3`、`vec4`、`mat3x3`、`mat4x4` 外，还无条件牵引部分 `ext/` 与 `gtc/` 头文件。

**结论**：

1. 四元数不是单纯的“再加一个核心类型”问题。
2. 其迁移往往会连带引入函数库、常量与矩阵变换相关依赖。
3. 若未来要拆分四元数迁移范围，应优先处理 include 结构，而不是只看类型定义本身。

## 4. 关键文件与实现风险结论

### 4.1 `setup.hpp`

`setup.hpp` 原文件大量承担平台、编译器、架构和配置宏汇总职责，其中只有一部分属于核心类型迁移的直接前提。

**结论**：

1. `length_t`、基础配置常量、基础断言/大小语义是核心迁移的必要内容。
2. 平台/编译器/架构检测不应与核心类型语义耦合处理。
3. `setup.hpp` 的仓颉映射应先抽离“核心语义必需项”，剩余检测逻辑单独处理。

### 4.2 `_vectorize.hpp` 与 `compute_vector_decl.hpp`

这两类辅助模板依赖 C++ 模板特化与模板模板参数风格，其复杂度并不来自算法，而来自模板表达方式。

**结论**：

1. 这部分不应按“机械翻译头文件”处理。
2. 推荐把问题重述为“如何为 Vec1~Vec4 提供统一的分量级运算模板”。
3. 若后续实现选择宏生成，应明确其目标是替代模板表达能力，而不是简单减少代码量。

### 4.3 `_swizzle.hpp`

`_swizzle.hpp` 依赖 `reinterpret_cast<T*>(_buffer)` 一类底层内存重解释技巧。

**结论**：

1. 该实现路径与仓颉值类型模型不兼容。
2. swizzle 不能作为“默认可迁移能力”看待。
3. 后续若需要支持 swizzle，应重新设计 API 或内存模型，而非复刻现有实现。

### 4.4 SIMD 相关路径

`qualifier.hpp`、`type_vec3.inl`、`type_vec4.inl` 等位置存在 SIMD 平台特化和条件包含路径。

**结论**：

1. SIMD 不是核心类型语义的必要前提。
2. SIMD 路径应被视作后续优化/平台特化议题。
3. 任何基础版本设计都应先保证在无 SIMD 条件下语义闭合。

## 5. 迁移设计约束汇总

以下约束可直接作为后续技术方案的输入条件：

1. `vec<N, T, Q>` 不能按原 C++ 模板特化模型原样落地，需拆分为按分量数建模的类型族。
2. `qualifier` 必须从“枚举值模板参数”改为“类型级 qualifier 参数”。
3. 所有依赖条件 `#include` 的实现路径，都需要固定配置、重组结构，或另行排除。
4. 整数向量算术必须显式定义溢出策略，不能隐含沿用 C++ 语义。
5. 断言、限值、长度/大小语义替代层属于基础设施前置问题。
6. 向量层是整个 GLM 类型体系中最关键的基础层；矩阵与四元数都建立在其上。
7. 四元数迁移会连带引入额外跨层依赖，不能简单视作“又一个核心结构体”。
8. swizzle、匿名联合、reinterpret、SIMD 特化均属于需要重设计的高风险点。

## 6. 适合作为后续文档引用的稳定结论

为避免后续文档重复分析，本文可被直接引用的稳定结论如下：

1. 仓颉可承载 GLM 核心类型迁移，但不是模板语法层面的等价翻译。
2. 向量类型族是最适合作为设计与实现起点的核心基础层。
3. `Qualifier` 类型化、Vec 按分量数拆分、Shim 层前置，是当前最稳定的技术方向。
4. `_swizzle.hpp`、`_vectorize.hpp`、四元数 ext/gtc 牵引依赖，是后续方案中需要单独评审的复杂点。
5. 本文不再给出“第一轮”“最小范围”“迁移顺序”“验收步骤”等执行性结论，相关内容应在单独执行文档中维护。
